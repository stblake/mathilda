# Slow / deficient algebraic simplification

This file catalogs the cases where Mathilda's algebraic simplification is slow,
non-terminating, or otherwise deficient. It is the prerequisite work item for
several blocked features (notably the Goursat **Section 5 / A4** integrator and
faster cyclotomic-radicand integration) — those features are correct in design
but blocked purely on the simplification primitives below being too slow.

**Headline finding.** Plain rational simplification over `Q(x)` is fast. The
deficiency is almost entirely **arithmetic over algebraic-number extensions** —
`Together` / `Cancel` / `Simplify` with `Extension -> Automatic` (and the
`PossibleZeroQ` / `zero_test` Stage-1 path that calls them) over a field
`Q(α)` generated by cyclotomic or radical constants. This blows up
super-polynomially in the number of terms and the degree of the extension.

---

## 1. Core deficiency: `Together`/`Cancel`/`Simplify` over `Q(α)`

`canonic(e) = Cancel[Together[e, Extension -> Automatic], Extension -> Automatic]`
is the workhorse of every radical/pseudo-elliptic integrator. Over the rationals
it is cheap; over an algebraic-number extension it is the bottleneck.

### Reproducers (α = `Exp[2 Pi I/3]`, a primitive cube root of unity)

The clearest, reproducible-with-hard-numbers cases are at the integration level
(§6); the underlying primitive that blows up is the algebraic-number
`Together`/`Cancel`/`Simplify` of a multi-term sum of cyclotomic-coefficient
rationals — e.g. the 4-term V4 projection of `F = t/(t^3+8)` over the three
involutions `Sa,Sb,Sc` of `t^3-1`:

```
Together[F + F/.t->Sa + F/.t->Sb + F/.t->Sc, Extension->Automatic]
Simplify[F + F/.t->Sa + F/.t->Sb + F/.t->Sc]
```

does not finish within several seconds, and the V4 trivial projection
`canonic((f0+f1+f2+f3)/4)` for `(x-1)/((x+2) Sqrt[x^3-1])` did **not finish in 3
minutes** when measured with the time budget removed. `PossibleZeroQ` of the same
uncombined sum also hangs (~20s+), because its Stage 1 runs the same `Together`
(§2). The cost grows super-polynomially with the number of terms combined and
the degree of the extension.

Contrast — the *same shapes over `Q`* (no algebraic constants) all finish in
**microseconds**: `Together[1/(E^x-1)+1/(E^x+1)]`, `Factor[x^2-E^(2x)]`,
`Cancel[...deg-30 rational...]`, `Simplify[Sqrt[3+2Sqrt[2]]-1-Sqrt[2]]`,
multi-generator `Tan` simplification — none are slow. So the cost is the
extension field, not the expression size per se.

### Why it bites integration

The trivial-character / eigenspace **zero test** in the Goursat family only needs
to know whether a projection vanishes, but the historical code formed the
projection via `canonic` first. That single `canonic` over `Q(α)` never
terminates in practice:

- `Integrate[(x-1)/((x+2) Sqrt[x^3-1]), x]` — the V4 trivial projection
  `canonic((f0+f1+f2+f3)/4)` did **not complete in 3 minutes** (measured by
  bypassing the time budget). The integrand itself is elementary and now solves
  in ~0.1s only because the criterion is decided by **numeric sampling** instead
  of `canonic` (see `sample_clearly_nonzero` in `integrate_goursat.c`).

This numeric-gate workaround is in place for the *criterion*, but the actual
*reduction* still needs real `canonic` over `Q(α)`, so the cases below remain
slow.

## 2. Where the blowup currently bites

| Site | Symptom |
|------|---------|
| `src/calculus/integrate_goursat.c` — `goursat_v4` descent (`reduce_v4_piece`) | A cyclotomic-radicand integrand that passes the V4 trivial-projection gate (e.g. the Legendre–Clausen `t/((t^3+8) Sqrt[t^3-1])`) enters the descent and the per-piece `canonic` over `Q(α)` exhausts the 6 CPU-second `TimeConstrained` budget → **~12s wall, then declines**. This is why `goursat_period3` is never even reached for that integrand. |
| `integrate_goursat.c` — `goursat_period3` reduction with **cyclotomic fixed points** (fix-α / fix-α² maps) | `(t-α)/((t+2α) Sqrt[t^3-1])` (a single basis term of the Legendre–Clausen decomposition) **hangs > 60s**. The fix-1 map has rational fixed points {1, −2} and reduces in ~0.1s; the fix-α / fix-α² maps have fixed points {α, −2α} and the downstream `canonic` over `Q(α)` blows up. |
| `integrate_goursat.c` — `goursat_cubic` / `goursat_quartic` eigenprojections (`eigenproj3`/`eigenproj4`) | `canonic(simplify_e(...))` over the splitting field; `(1-x^3)^(1/3)/(1+x)` spun for minutes before the numeric gate was added (now declines in ~0.16s, but a *genuinely reducible* cyclotomic cube-root cofactor would still be slow in the reduction). |
| `zero_test.c` — `decide_rational` (Stage 1 of `PossibleZeroQ`) | Calls `internal_together` ∘ `internal_cancel` **unconditionally and first**, before the numeric Schwartz–Zippel stage. So a bare `PossibleZeroQ[f0+f1+f2+f3]` over `Q(α)` hangs (~20s+) even though Stage 3 would settle it numerically in milliseconds. Any caller that zero-tests an algebraic-number expression pays this. |
| `simp/` `Simplify` over `Q(α)` | `Solve`'s fixed points for the cyclotomic Möbius maps come back as **un-simplified nested cube-roots-of-unity** (e.g. an expression that equals `1` printed as `-1/2((-1)^(1/3)-(-1)^(2/3)+Sqrt[...])`). `Simplify` collapses small ones quickly (~0.01s) but is the dominant cost on larger projected expressions. |

## 3. Blocked feature: Goursat Section 5 / A4 (`t/((t^3+8) Sqrt[t^3-1])`)

The Legendre–Clausen integral is provably elementary and decomposes cleanly:

```
t/(t^3+8) = (1/18) (t-1)/(t+2) + (α/18) (t-α)/(t+2α) + (α^2/18) (t-α^2)/(t+2α^2)
```

(coefficients from a 3×3 DFT-style solve — itself fast). Each summand is a pure
period-3 character that the existing `goursat_period3` would reduce **if** the
cyclotomic `canonic` were fast. Two independent blockers, both in §1:

1. The fix-α / fix-α² basis terms hang in `period3_reduce`'s `canonic` over `Q(α)`.
2. Even reaching `goursat_period3` is blocked because `goursat_v4` runs first and
   its descent blows up on the same integrand (§2, row 1).

So Section 5 is **designed but not shippable** until `Q(α)` arithmetic is fast
(or a real-arithmetic reformulation that avoids `Q(α)` is found — see §5).

## 4. Other known simplification deficiencies (from prior work)

- **`Together`/`Factor` infinite-loop on symbolic exponentials** `Power[sym, var]`
  (e.g. `a^i` inside a `Together`/`Cancel` *pre-pass* of `Together`/`Cancel`
  themselves). Worked around by routing those through `Cancel` and normalising
  only the rational coefficient; the underlying loop is not fixed. (See memory
  note *Together/Factor hang on a^i*.)
- **`Simplify` multi-generator explosion** on high-generator trigonometric
  rationals (multivariate GCD / `Factor` exponential in the number of distinct
  generators). Guard sites were added to cut a Weierstrass-derivative case from
  18.6s → 3.7s, but it is a mitigation, not a cure. (See *Simplify
  multi-generator explosion*.)
- **`Together` must not `Expand`** `Power[Plus[...], n]` in `Together`/`Cancel`
  pre-passes (catastrophic blowup); currently avoided by contract rather than by
  the engine being robust to it.

## 5. Root-cause hypotheses & directions

The pattern (fast over `Q`, exploding over `Q(α)`) points at the
algebraic-number layer, not the rational-function core:

1. **No minimal-polynomial reduction.** Arithmetic over `Q(α)` likely keeps
   `α` as an opaque transcendental (`(-1)^(1/3)`, `Exp[2 Pi I/3]`, …) and relies
   on `Simplify` to discover identities like `1+α+α^2=0` and `α^3=1` after the
   fact. Representing extension elements in a fixed `Q`-basis modulo the minimal
   polynomial of `α` (e.g. `α^2 = -1-α`) would make these reductions O(1) and
   keep intermediate expressions bounded.
2. **Common-denominator blowup before reduction.** `Together` over the splitting
   field forms a common denominator across all the cyclotomic conjugates,
   producing a giant numerator/denominator that is only then `Cancel`-ed.
   Reducing each term modulo the generator ideal *before* combining (the
   `radrat` "reduce-before-rationalise" lesson) should bound the size.
3. **Conjugate-pair / real reformulation.** Cyclotomic results are real once
   conjugate pairs are summed (`α`-term + `α^2`-term ∈ `R`). Working in the real
   subfield (`Q(α)`'s fixed field under conjugation, e.g. via the real quadratic
   `(t-α)(t-α^2)=t^2+t+1`) would avoid `Q(α)` entirely for the cases that have
   real answers — the most promising route for the Legendre–Clausen family.
4. **Zero-test ordering.** `zero_test.c::decide_rational` should *not* run the
   `Together`/`Cancel` Stage 1 when the expression contains algebraic-number
   constants and free symbols — go straight to numeric Schwartz–Zippel (Stage 3),
   which is what the Goursat code already does by hand via `sample_clearly_nonzero`.
   Generalising that into `PossibleZeroQ` itself would remove the hand-rolled
   guards.

## 6. Reproducers

**Integration level (measured, the numbers that matter in practice).** A clean
build of `Mathilda`:

```
Integrate[(x-1)/((x+2) Sqrt[x^3-1]), x]                                (* ~0.1s OK -- numeric criterion gate *)
Integrate[t/((t^3+8) Sqrt[t^3-1]), t, Method -> "GoursatAlgebraic"]    (* ~12s, then declines (budget exhausted) *)
al = Exp[2 Pi I/3];
Integrate[(t-al)/((t+2 al) Sqrt[t^3-1]), t, Method -> "GoursatAlgebraic"]   (* > 60s, killed *)
```

The first is fast only because the *criterion* is decided numerically; the
second and third hit the real `canonic` over `Q(α)` and are the blocked cases.
Both *should* be fast and produce closed forms once `Q(α)` arithmetic is
efficient.

**Primitive level.** Wrap in `TimeConstrained[Timing[...][[1]], 6, "ABORTED>6s"]`
to bound each; the cyclotomic ones do not finish within several seconds while the
rational analogues finish in microseconds:

```
al = Exp[2 Pi I/3];
Sa=(t+2)/(t-1); Sb=(t+2 al)/(al^2 t-1); Sc=(t+2 al^2)/(al t-1); F=t/(t^3+8);
Together[F + (F/.t->Sa) + (F/.t->Sb) + (F/.t->Sc), Extension->Automatic]   (* slow *)
Simplify[F + (F/.t->Sa) + (F/.t->Sb) + (F/.t->Sc)]                        (* slow *)
Together[1/(E^x-1) + 1/(E^x+1)]                                            (* microseconds *)
```
