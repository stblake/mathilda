# Changelog: week of 2026-05-11 (Mon) ‚Äì 2026-05-17 (Sun)

Feature additions and fixes recorded during this week.

## Solve ‚Äî linear-system specialist `Solve\`SolveLinearSystem` (2026-05-17)

`Solve` now solves systems of linear equations in multiple variables.  A new
specialist `Solve\`SolveLinearSystem` (src/solvelinsys.c) handles every input
that the router classifies as multi-variable -- a single `Equal` with a
multi-symbol variable list, or an `And` / `List` of equations -- and the
`Solve` router canonicalises each `lhs == rhs` to `lhs - rhs` before handing
it off.

### Algorithm

Per equation: assert affine in `vars` (degree at most 1 in each variable, no
cross terms) by extracting `Coefficient[poly, v_j, 1]` for every `v_j` and
verifying both each coefficient and the residual `poly - sum_j c_j * v_j` are
free of all variables.  Non-affine systems return `NULL`, leaving `Solve`
unevaluated.

Coefficients populate an `m x (n+1)` augmented matrix with the variable
columns in **reversed order** (column `0` is the coefficient of `vars[n-1]`,
column `n` is `-residual`).  Reversing the columns is what produces
Mathematica's `Solve::svars` convention: standard left-to-right Gauss--Jordan
elimination pivots on the right-most original variable first, leaving the
left-most variables free.

Pivot selection across symbolic entries prefers concretely-non-zero pivots
(Integer, Rational, Real) over symbolic ones.  Zero-testing of intermediate
entries goes through `Cancel[Together[.]]` so that residues like `a*x - a*x`
or `(p/q) - p*Power[q, -1]` are recognised as zero during elimination.

### Output shape

| Case                       | Output                              |
| -------------------------- | ----------------------------------- |
| Unique solution            | `{{v1 -> e1, v2 -> e2, ...}}`       |
| Inconsistent system        | `{}`                                |
| Under-determined system    | `{{pivot_vars -> exprs in free vars}}` + `Solve::svars` |
| Empty equation list (True) | `{{}}` (tautology)                  |

Free variables in the under-determined case produce no rule (matching
Mathematica's convention).  Solution rules are emitted in **original** input
variable order, regardless of the internal reversed-column pivot ordering.

### Domains

`Complexes` (default), `Reals`, and `Integers` are all accepted.  Domain
filtering is a post-pass over the computed solution:

* `Integers` drops the entire solution if any rule's RHS is not a concrete
  `EXPR_INTEGER`.
* `Reals` drops the entire solution if any RHS syntactically contains a
  `Complex[_, _]` head (the heuristic conservative test for "non-real").

### Examples

```mathematica
Solve[3 x + 2 y == 11 && x + y == 12, {x, y}]
(* {{x -> -13, y -> 25}} *)

Solve[a x + c == 1 && b x - d y == 2, {x, y}]
(* {{x -> (1 - c)/a, y -> (-2 a + b - b c)/(a d)}} *)

Solve[3 x + 2 y == 11, {x, y}]
(* Solve::svars *)
(* {{y -> 11/2 - (3 x)/2}} *)

Solve[3 x + 2 y == 11 && x + y == 12 && 3 x + y == 32, {x, y}]
(* {} *)

Solve[{x + y + z == 6, 2 x - y + z == 3, x + 2 y - z == 2}, {x, y, z}]
(* {{x -> 1, y -> 2, z -> 3}} *)

Solve[2 x + y == 0 && 4 x + y == 1, {x, y}, Integers]
(* {} (x = 1/2 is not an integer) *)
```

### Memory

All linsys paths exit through a single cleanup point (the augmented matrix,
the pivot-row map, the var-is-pivot bitmap, the rules buffer); the same path
is used by the inconsistent / domain-rejected / under-determined / unique
solution exits.  `Mathilda` runs the user-spec examples and the new edge
cases through `leaks --atExit` with zero leaks reported.

### Tests

Sixteen new unit tests in `tests/test_solve.c` exercise: integer-coefficient
2x2 and 3x3 systems, symbolic-coefficient 2x2, rational-coefficient 2x2, the
under-determined / over-determined-consistent / over-determined-inconsistent
classifications, `And` vs `List` input forms, zero-rows-in-conjunction,
`False` conjunct collapse, non-affine declines, all three domains, and
direct invocation of `Solve\`SolveLinearSystem`.

## Solve ‚Äî Integers domain and Reals fixes (2026-05-17)

`Solve` (and the underlying `Solve\`SolvePolynomialEquality` specialist)
now accepts `Integers` as a third domain, alongside the previously
supported `Complexes` (default) and `Reals`.  Two existing problems in
the `Reals` path were uncovered while wiring the integer filter up and
are fixed in the same change.

### `Integers` domain

Implementation strategy is "solve under `Reals`, then keep only the
roots that are provably concrete integers (`EXPR_INTEGER` or
`EXPR_BIGINT`)".  Anything else ‚Äî `Rational[p, q]`, `Sqrt[2]`, a held
`Root[Function[...], k]`, a symbolic `Times[-1, a, Power[b, -1]]`, etc.
‚Äî is dropped.  This matches Mathematica's `Solve[..., Integers]`: it
never invents a non-integer solution, and it never returns a value it
cannot type-prove to be an integer.

Concrete examples (all newly passing):

```mathematica
Solve[x^2 - 5 x + 6 == 0, x, Integers]      (* {{x -> 2}, {x -> 3}} *)
Solve[2 x^2 - 7 x + 3 == 0, x, Integers]    (* {{x -> 3}} (1/2 dropped) *)
Solve[x^2 - 2 == 0, x, Integers]            (* {} (Sqrt[2] dropped) *)
Solve[x^3 - 6 x^2 + 11 x - 6 == 0, x, Integers]
                                            (* {{x -> 1}, {x -> 2}, {x -> 3}} *)
Solve[x^3 + x + 1 == 0, x, Integers]        (* {} (Root[] objects dropped) *)
Solve[(x-1)^2 (x-2) == 0, x, Integers]      (* {{x -> 1}, {x -> 1}, {x -> 2}} *)
Solve[a x + b == 0, x, Integers]            (* {} (Times[-1, ...] not provably integer) *)
```

Unsupported domains (`Rationals`, `Algebraics`, `Primes`, `Booleans`,
‚Ä¶) continue to leave the call unevaluated; only `Complexes`, `Reals`,
and `Integers` are accepted as `dom`.

### `Reals` ‚Äî odd-degree binomial with negative base

`solve_binomial` previously emitted the principal complex root for
odd-degree binomials with a concrete negative base, so

```mathematica
Solve[x^3 + 1 == 0, x, Reals]
```

returned `{{x -> (-1)^(1/3)}}` ‚Äî a complex number under Mathematica's
principal-root convention ‚Äî instead of the actual real root `-1`.  The
fix adds a sign-aware selector inside the odd-`n` branch:

* `base > 0`  ‚Üí  `base^(1/n)` (unchanged),
* `base == 0` ‚Üí  `0`,
* `base < 0`  ‚Üí  `-((-base)^(1/n))` (new: real branch, not principal),
* unresolved sign ‚Üí fall back to the symbolic principal (unchanged, so
  parametric `Solve[x^3 == a, x, Reals]` still returns `a^(1/3)`).

This makes `Solve[x^5 + 32 == 0, x, Reals]` return `{{x -> -2}}`,
`Solve[x^7 + 128 == 0, x, Reals]` return `{{x -> -2}}`, and so on.

### `Reals` ‚Äî multiplicity preserved when Œî = 0

The fast quadratic path used to collapse the `D == 0` case to a single
rule under `Reals`, while the `Complexes` path always emitted two
(equal) rules.  Mathematica preserves algebraic multiplicity in both
domains, so the collapse has been removed:

```mathematica
Solve[(x-1)^2 == 0, x]           (* {{x -> 1}, {x -> 1}}     *)
Solve[(x-1)^2 == 0, x, Reals]    (* {{x -> 1}, {x -> 1}}     -- new *)
Solve[x^2 + 2 x + 1 == 0, x, Integers]
                                 (* {{x -> -1}, {x -> -1}}   -- new *)
```

## Solve ‚Äî canonical Mathematica order for root output (2026-05-17)

`Solve` now sorts its returned root list with a domain-specific
comparator before wrapping each value into `Rule[var, val]`, so the
output order matches Mathematica's.  Previously roots were emitted in
the algorithm's natural construction order (`k = 0‚Ä¶n-1` for binomials),
which gave e.g.

```
Solve[x^5 + 1 == 0, x]    (* Mathilda, before *)
  -> {{x -> (-1)^(1/5)}, {x -> (-1)^(3/5)}, {x -> -1},
      {x -> -(-1)^(2/5)}, {x -> -(-1)^(4/5)}}
```

The new canonical order (`src/solvepoly.c:sl_sort_canonical`) is:

1. **Concrete real numbers first**, sorted by numerical value
   (Integer / Rational / Real / BigInt).
2. **Other expressions** are decomposed as `v = mag ¬∑ (-1)^exp` by
   absorbing every `Integer(-1)` and `Power[-1, e]` factor into the
   exponent.  The remaining factors form `mag`.  Sort key is
   `(mag, exp mod 1, exp descending)` with `expr_compare` as the final
   tiebreak.

So for `x^5 + 1 == 0` the five exponents `1, 1/5, 7/5, 3/5, 9/5` reduce
mod 1 to `0, 1/5, 2/5, 3/5, 4/5` and the result is interleaved:

```
Solve[x^5 + 1 == 0, x]    (* Mathilda, after *)
  -> {{x -> -1}, {x -> (-1)^(1/5)}, {x -> -(-1)^(2/5)},
      {x -> (-1)^(3/5)}, {x -> -(-1)^(4/5)}}
```

Other knock-on effects:

* `Solve[x^4 - 5 x^2 + 4 == 0, x]` now returns `{-2, -1, 1, 2}` instead
  of `{1, -1, 2, -2}` (concrete reals sort numerically).
* `Solve[(x-1)(x-2)(x-3) == 0, x]` returns `{1, 2, 3}` rather than the
  reversed `{3, 2, 1}` produced by the factor-walk order.
* `Solve[x^5 - 2 == 0, x]` emits `{2^(1/5), -(-1)^(1/5) 2^(1/5),
  (-1)^(2/5) 2^(1/5), -(-1)^(3/5) 2^(1/5), (-1)^(4/5) 2^(1/5)}` ‚Äî
  sorted by canonical exponent `0, 1/5, 2/5, 3/5, 4/5`.
* `Solve[x^4 - 1 == 0, x]` emits `{-1, 1, -I, I}`.

The sort is internal to `Solve`; Mathilda's global `expr_compare` (and
therefore `Sort[]` on arbitrary lists) is unchanged.

## Solve / Power ‚Äî clean radical form for root-of-unity binomials (2026-05-17)

`Solve[x^n + c == 0, x]` no longer falls into the Euler-formula trap
that expanded each root into a `Cos[‚Ä¶] + I Sin[‚Ä¶]` sum.  The polynomial
binomial solver in `src/solvepoly.c` now constructs the `k`-th `n`-th
root as `r ¬∑ (-1)^(2k/n)` (where `r` is the principal `n`-th root)
instead of `r ¬∑ Exp[2 k œÄ I / n]`.  Because `Power[-1, p/q]` already
canonicalises rational exponents into `sign ¬∑ (-1)^(r/q)`, and `Times`
fuses two `Power[-1, _]` factors with rational exponents into a single
`(-1)^(_)` factor, each root falls out in the standard Mathematica
form.

```
In[1]:= Solve[x^5 + 1 == 0, x]
Out[1]= {{x -> (-1)^(1/5)}, {x -> (-1)^(3/5)}, {x -> -1},
         {x -> -(-1)^(2/5)}, {x -> -(-1)^(4/5)}}

In[2]:= Solve[x^11 + 1 == 0, x]
Out[2]= {{x -> (-1)^(1/11)}, {x -> (-1)^(3/11)}, ‚Ä¶, {x -> -1}, ‚Ä¶,
         {x -> -(-1)^(10/11)}}

In[3]:= Solve[x^5 - 2 == 0, x]
Out[3]= {{x -> 2^(1/5)}, {x -> (-1)^(2/5) 2^(1/5)},
         {x -> (-1)^(4/5) 2^(1/5)},
         {x -> -(-1)^(1/5) 2^(1/5)},
         {x -> -(-1)^(3/5) 2^(1/5)}}
```

To make this work `src/power.c` also gained two small canonicalisations
for `Power[-1, p/q]`:

1. `(-1)^(p/2)` for odd `p ‚âÝ 1` now reduces to the right `¬±I` factor
   via `I^p` (previously only `p == 1` was handled, so higher-power
   cases like `(-1)^(3/2)`, `(-1)^(5/2)`, `(-12)^(3/2)` stayed
   unevaluated).  The principal-branch rule is

   ```
   (-n)^(p/2) = I^p ¬∑ |n|^(p/2)
              = (p mod 4 == 1 ? I : -I) ¬∑ |n|^(p/2).
   ```

2. `(-1)^(p/q)` for **even `q ‚â• 4`** and `|p| ‚â• q` now does the same
   integer-part extraction the odd-`q` branch already did for any
   negative base.  So `(-1)^(5/4) ‚Üí -(-1)^(1/4)`, `(-1)^(7/4) ‚Üí
   -(-1)^(3/4)`, `(-1)^(7/6) ‚Üí -(-1)^(1/6)`.  Negative bases other
   than `-1` with even `q ‚â• 4` (e.g. `(-16)^(1/4)`) still leave the
   expression unevaluated ‚Äî that case has no clean canonical form
   under the principal branch yet.

Existing `Sqrt[-n]`, `(-n)^(1/2)`, and odd-`q` negative-base
canonicalisations are unchanged.  `Solve[x^4 - 1 == 0, x]` still emits
`{1, I, -1, -I}` exactly; the new path for `k = 3` flows through
`Power[-1, 3/2] ‚Üí -I`.

## Printing ‚Äî drop parens around `I`/`-I` and render `Function[body]` as `body &` (2026-05-17)

Two improvements to standard-form output (FullForm is unchanged):

1. Pure imaginary `Complex[0, 1]` and `Complex[0, -1]` now render without
   the surrounding parentheses that they used to attract when appearing
   inside `Times`. Previously a root such as `I ¬∑ 2^(1/4)` printed as
   `(I) 2^(1/4)`; it now prints as `I 2^(1/4)`. Negative pure imaginary
   has its sign pulled out as a unary minus, so `-I ¬∑ 2^(1/4)` prints as
   `-I 2^(1/4)` (previously `(-I) 2^(1/4)`). The Plus printer was
   extended in parallel so terms like `Plus[1, -I¬∑x]` render as
   `1 - I x` instead of `1 + -I x`.

   ```
   In[1]:= Solve[x^4 == 2, x]
   Out[1]= {{x -> 2^(1/4)}, {x -> I 2^(1/4)},
            {x -> -2^(1/4)}, {x -> -I 2^(1/4)}}
   ```

2. Slot-based pure functions now print using the conventional `&`/`#`
   syntax instead of their explicit `Function[...]`/`Slot[...]` form:

   - `Slot[n]` renders as `#n` (and `SlotSequence[n]` as `##n`; `##1`
     collapses to `##`).
   - 1-argument `Function[body]` renders as `body &`.

   ```
   In[1]:= Function[1 - Slot[1]^2 + Slot[1]^3]
   Out[1]= 1 - #1^2 + #1^3 &

   In[2]:= (#1^2 + #2^4) &
   Out[2]= #1^2 + #2^4 &
   ```

   Named-parameter forms ‚Äî `Function[var, body]`, `Function[{vars}, body]`,
   and `Function[{vars}, body, attrs]` ‚Äî keep the explicit `Function[...]`
   spelling. FullForm output for both pure-function and Slot expressions is
   unchanged, so callers that rely on the tree structure are unaffected.

## Solve ‚Äî factor out trivial root from quadratics with zero constant term (2026-05-17)

`Solve` now special-cases the quadratic `a x^2 + b x + 0 == 0` by factoring
out `x` instead of applying the discriminant formula.  For symbolic
coefficients the discriminant `b^2 - 4¬∑a¬∑0 = b^2` cannot collapse to `|b|`,
so the previous behaviour returned a spurious `Sqrt[b^2]` in both roots.

Example ‚Äî before:

```
In[1]:= Solve[a/x + b == c/x, x]
Out[1]= {{x -> (1/2 (-a + c - Sqrt[(a - c)^2]))/b},
         {x -> (1/2 (-a + c + Sqrt[(a - c)^2]))/b}}
```

After:

```
In[1]:= Solve[a/x + b == c/x, x]
Out[1]= {{x -> -(a - c)/b}}
```

The cross-multiplication step yields `b x^2 + (a - c) x == 0`; the new
path splits it as `x ¬∑ (b x + (a - c)) == 0`, picking up `x = 0` (dropped
later by the extraneous-root filter because `D1¬∑D2 = x¬≤` vanishes there)
and the genuine linear root `x = (c - a)/b`.

## Solve ‚Äî canonicalise rational equalities before dispatch (2026-05-17)

`Solve` now handles equations whose sides are rational (not just
polynomial) in the solving variable.  `Solve[a/x + b == 0, x, Reals]`
returns `{{x -> -a/b}}` instead of leaving the call unevaluated, and the
same applies to `Solve[1/(x-1) == 2, x]`, `Solve[(x+1)/(x-1) == 3, x]`,
`Solve[x + 1/x == 5/2, x]`, etc.

Before the polynomial specialist runs, the router now:

1. Applies `Together` to each side, producing single-fraction
   normal forms `N1/D1` and `N2/D2`.
2. Cross-multiplies to `N1*D2 - N2*D1 == 0`, the form actually
   handed to the polynomial dispatcher.
3. Calls `Collect[..., var]` so the polynomial appears in standard
   form before the fast-path classifier inspects it.

When both sides are already polynomial in `var`, the denominators are
both `1` and the canonicalisation is a no-op ‚Äî existing tests are
unchanged.

Cross-multiplication can introduce spurious roots at points where one
of the original denominators vanished, so the result list is run
through an *extraneous-root filter*: any candidate `var = r` for which
`D1*D2` evaluates to a concrete zero (integer 0, BigInt 0, real 0.0,
or `Rational[0, _]`) is dropped.  Symbolic / undetermined values are
kept, matching Mathematica's "verify when you can" default.  Concrete
examples:

- `Solve[x/(x-1) == 2/(x-1), x]` ‚Üí `{{x -> 2}}` (x = 1 dropped).
- `Solve[2/(x^2-1) == 1/(x-1), x]` ‚Üí `{}` (only root would be the
  extraneous `x = 1`).
- `Solve[a/x + b == 0, x]` ‚Üí `{{x -> -a/b}}` (denominator value `-a/b`
  is symbolic, so kept).

The new test cases (`test_rational_parametric`,
`test_rational_one_sided`, `test_rational_cross_mult`,
`test_rational_extraneous`, `test_rational_mixed_poly`,
`test_rational_denominator_only`, `test_rational_sum_of_fractions` in
`tests/test_solve.c`) cover 22 expressions across the new pipeline.

## rat: Cancel ‚Äî extract algebraic-constant atoms from Plus terms (2026-05-16)

`Cancel[Sqrt[2] / (Sqrt[2] + Sqrt[2] x^4)]` now reduces to `1/(1 + x^4)`
without needing `Extension -> Automatic`.  Previously the call returned
its input untouched, because `PolynomialGCD[Sqrt[2], Sqrt[2] + Sqrt[2]
x^4]` returns 1: the integer-content recursion in `my_number_gcd` /
`poly_content` only looks at literal integer coefficients, so the
structural `Sqrt[2]` shared across every term of both num and den was
invisible.

`cancel_recursive` now runs a pre-pass (`rat_strip_symbolic_common`)
that:

1. For each Plus summand of num and den, collects the multiplicative
   factor list (Times args or a singleton).
2. Filters that list down to *algebraic-constant atoms* ‚Äî
   `Power[integer_literal, rational_non_integer_exponent]` such as
   `Sqrt[2]`, `Sqrt[3]`, `CubeRoot[5]`, `2^(2/3)`.
3. Intersects the num and den factor multisets via `expr_eq`.
4. If the intersection is non-empty, divides both num and den by the
   common factor (distributing through Plus so the Times evaluator
   collapses `Sqrt[2] ¬∑ Power[Sqrt[2], -1] -> 1` inside every summand,
   rather than wrapping the whole Plus in an opaque `Power[..., -1]`
   that the downstream `PolynomialGCD` can't see through).

The filter is intentionally narrow: only `Power[integer, rational]` is
eligible.  Symbols, `Power[symbol, integer]`, and compound Power bases
(`Sqrt[1 + x]`, ...) are left for the existing multivariate
`PolynomialGCD` path ‚Äî extending the pass to those classes caused the
integration dispatcher's pattern-matching against rational
intermediates to drift, and the `Sqrt[(1+x)(2+3x)]` family of the CRC
corpus saw DIFF NONZERO regressions.

CRC corpus regression count after the change: unchanged at 9 (the
9-case baseline is pre-existing and tracked separately).

Verified examples:

```mathematica
Cancel[Sqrt[2] / (Sqrt[2] + Sqrt[2] x^4)]            (* -> 1/(1 + x^4) *)
Cancel[(Sqrt[2] x^2 + Sqrt[2] x) / (Sqrt[2] x + Sqrt[2])]  (* -> x   *)
Cancel[(Sqrt[2] x + Sqrt[2] y) / (Sqrt[2] + Sqrt[2] x)]    (* -> (x+y)/(1+x) *)
Cancel[(x^2 - 1) / (x - 1)]                          (* -> 1 + x      *)
```

Files touched: `src/rat.c` (new helpers `rat_factor_list_copy`,
`rat_strip_numeric_factors`, `rat_factor_intersect`,
`rat_symbolic_content`, `rat_div_distribute`,
`rat_strip_symbolic_common`; one new call from `cancel_recursive`),
`docs/spec/builtins/arithmetic-and-algebra.md` (updated `Cancel`
feature list + two new examples).

## files: FilePrint (2026-05-16)

Added the `FilePrint` builtin to `src/files.c`, mirroring Mathematica's
overload set:

- `FilePrint["file"]` ‚Äî print every line of `file` to stdout.
- `FilePrint["file", n]` / `FilePrint["file", -n]` ‚Äî print the first
  (resp. last) `n` lines; values larger than the file's line count
  clamp instead of erroring.
- `FilePrint["file", m;;n]` and `FilePrint["file", m;;n;;s]` ‚Äî
  `Span`-style selection with optional signed step; negative indices
  inside the span count from the end of the file, and `All` is accepted
  in any slot.

Bytes pass through verbatim via `fwrite`, so embedded NULs and non-UTF-8
sequences print intact.  When the file's final line lacks a trailing
`\n` and the selection actually reaches it, `FilePrint` injects a
single newline so the next REPL prompt isn't glued onto the content.
Ill-formed selectors (wrong types, wrong arity, step 0) leave the call
unevaluated rather than producing partial output, and a missing file
yields `$Failed` with a `FilePrint::noopen` diagnostic.

Files touched: `src/files.{c,h}` (`builtin_fileprint` + helpers,
`SYM_FilePrint` plumbing), `src/sym_names.{c,h}`, `src/info.c` (new
docstring), `tests/test_files.c` (18 new assertions covering each
selector form, clamping, empty/unterminated files, missing-file
diagnostics, the bad-argument contract, and the zero-step guard, all
using a `dup2`-based stdout capture so the assertions inspect the
actual bytes written).  Verified leak-free under macOS `leaks`
(12917 nodes malloced, 0 leaks).

## match: canonicalize LHS at DownValue insertion time (2026-05-16)

`add_rule` in `src/symtab.c` now runs the pattern LHS through a
`pattern_canonicalize` pass before storing it.  The pass mirrors the
structural rewrites `evaluate_step` applies to runtime expressions
(Flat flatten, Orderless sort, `Power[Times[..., n_Integer]]`
distribution, nested-`Power` collapse) while leaving every pattern-
marker wrapper (`Pattern`, `Blank*`, `Optional`, `Condition`,
`PatternTest`, `HoldPattern`, `Verbatim`, `Alternatives`, `Repeated*`,
`OptionsPattern`) structurally untouched.

### Why

Held LHSs and evaluated inputs had been living in different canonical
forms. The ¬ß3.5 dispatch filter pre-computes `dispatch_arity` and
`first_arg_head_canon` from the rule's parsed shape, while every
runtime expression flowing into `apply_down_values` arrives in fully
canonicalized form.  Concretely, every CRC-style rule of the form
`f[1/(x_ ...), x_]` parses as `f[Power[Times[x_, ...], -1], x_]` and
cached `Power` as its first-arg head, but every runtime call arrived
as `f[Times[Power[x,-1], ...], x]` (Power-over-Times integer
distribution kicks in at `src/power.c:813` during evaluation).  The
filter then rejected the rule before `match()` ran, even though
`MatchQ` of the pattern against the input returned `True`.  This was
the underlying cause of the ~80 unevaluated CRC integrals attributed
in `tasks/crc_corpus_2026-05-15.md` to a "3-term Plus matcher gap"
(misdiagnosed at the time ‚Äî the matcher itself was always fine).

### What it structurally changes

- Stored DownValue / OwnValue patterns now share canonical form with
  runtime inputs.  `Power[Times[t1,...,tn], k]` and
  `Times[Power[t1,k],...,Power[tn,k]]` are no longer two
  distinguishable LHSs when `k` is an integer literal ‚Äî the second
  form is the only one that survives storage.
- `Power[Power[base, e1], e2]` collapses to `Power[base, e1*e2]` when
  both exponents are numeric (Integer / BigInt / Rational), matching
  the evaluator's nested-Power normalization.
- Flat-headed and Orderless-headed pattern children are flattened and
  sorted with the same `eval_compare_expr_ptrs` comparator the
  evaluator uses, so the dispatch filter sees consistent shapes on
  both sides.
- Pattern-marker semantics are unchanged: Condition guards still run
  at match time with bindings in scope, Optional defaults still bind
  on slot absence, HoldPattern still forces literal matching, etc.

The pass is idempotent and runs once per rule at insertion time, so
the runtime cost is negligible.

### Tests

Three new pinning tests in `tests/test_condition_downvalue.c` under
¬ß9b cover:

- A `Power[Times[...]]` LHS rule dispatches against an evaluated
  `Times[Power[...]]` input (the CRC formula 246 shape).

## integrate: three-stage cascade, Method option, CRC table termination guards (2026-05-15)

`Integrate[f, x]` now dispatches through a three-stage cascade ‚Äî
`Integrate\`BronsteinRational` ‚Üí `Integrate\`RischNorman` ‚Üí
`Integrate\`CRCTable` ‚Äî instead of the previous two-stage chain.  The
new third stage is the CRC Standard Mathematical Tables (31st ed.)
rule package at `src/internal/CRCMathTablesIntegrals.m`, packaged as
`Integrate\`CRCTable` and `Get`-loaded lazily on first invocation so
unaffected sessions pay zero startup cost.

`Integrate` also accepts an optional third argument `Method -> "<name>"`
that bypasses the cascade and calls a single subroutine with strict
no-fallback semantics.  Accepted values: `"Automatic"` (the cascade,
default), `"Rational"`, `"RischNorman"`, `"CRCTable"`.  Unknown method
names emit `Integrate::method` and bubble back unevaluated.

The CRC table previously contained ~50 reduction rules whose only
guards excluded a single value (e.g. `n =!= 1`).  Formula 49,
`1/(x^2 - c^2)^n -> ... + ... IntegrateTable[1/(x^2 - c^2)^(n - 1), x]`,
typified the class: with `n = 0, -1, -2, ‚Ä¶` each match recurses
further into the negative integers, never reaching the base case at
`n = 1`.  Every such rule now carries an `IntegerQ[idx] && idx > base`
(or `< base`) guard sufficient to force termination from first
principles ‚Äî forward reductions need the index strictly above the
base case; backward reductions strictly below; step-2 reductions need
to clear both bases.  Formula 229 had an additional self-recursive
algebra typo (recursing on the same `X^n ‚àöX` pattern instead of
`X^(n-1) ‚àöX`), corrected in the same pass.  As defence-in-depth, the
C dispatcher caps CRC-rule recursion depth at 256 levels and emits
`Integrate\`CRCTable::depth` rather than locking up if a future rule
escapes the audit.

Files: `src/integrate.c` (cascade + Method parsing + lazy loader +
depth cap), `src/internal/CRCMathTablesIntegrals.m` (~50 guard fixes
+ formula 229 algebra typo + flat package layout), `src/internal/init.m`
(no eager load).

Known limitations:
- Mathilda's pattern matcher does not yet fully support `/;`-guarded
  multi-argument rules, so most CRC formulas don't actually fire even
  after the load.  This is a separate matcher issue, independent of
  the termination work above; the cascade structure is in place for
  when the matcher gains support.

## sort: order-independent variable collection in expr_compare (2026-05-15)

`expr_compare` (the canonical comparator that drives `Sort`,
`OrderedQ`, and every Orderless-attribute Plus/Times sort in the
evaluator) was silently truncating its variable-collection buffer at
64 distinct symbol names.  The comment claimed both sides saw the
same truncated set; in fact `collect_symbols_in` early-exited as soon
as the buffer filled, so collection from `a` could fill the buffer
and leave none of `b`'s unique symbols represented.  The result was
an asymmetric comparator: `compare(A, B)` and `compare(B, A)` could
both return `1`, violating qsort's antisymmetry contract.

### Symptom

Inputs that build large polynomial ansatzes (e.g.,
`Integrate\`RischNorman` candidates with hundreds of unknowns alongside
a small log-bearing term) hit `$IterationLimit exceeded`.  The
top-level `Plus` containing the polynomial and the log term sorted
its args one way on one iteration, the opposite way the next, and
the evaluator's fixed-point loop oscillated until the 4096-iteration
cap fired.  The eventual returned answer was still correct (the
oscillation was purely cosmetic re-ordering of Plus args), but the
warning was alarming and 4096 wasted iterations were burning real
cycles.

### Fix

`src/sort.c`: replace the fixed `const char* vars[64]` with a
growable `SymSet` (stack buffer for the common ‚â§64-symbol case, heap
fallback that doubles on demand).  `collect_symbols_in` now never
truncates; the resulting variable set is always `vars(a) ‚à™ vars(b)`
regardless of argument order, restoring antisymmetry.

## Eigenvalues / Eigenvectors

`Eigenvalues[m]`, `Eigenvalues[{m, a}]`, `Eigenvalues[m, k]`,
`Eigenvalues[m, -k]`, `Eigenvalues[m, UpTo[k]]`, and the corresponding
`Eigenvectors[...]` overloads are now available
(`src/linalg.c`).

### Algorithm

- Ordinary characteristic polynomial: Faddeev‚ÄìLeverrier‚ÄìSouriau in
  O(n^4) matrix multiplications.  Avoids the n! cost of Laplace
  expansion on a polynomial-entry matrix, so eigenvalues of
  `IdentityMatrix[12]` (and similarly large rational / diagonal
  matrices) return in milliseconds.
- Generalised case (`m - lambda a`) still uses Laplace expansion of the
  polynomial-entry matrix; acceptable in practice because the corpus
  of generalised eigenproblems is small (‚â§ 3√ó3).
- Roots of the characteristic polynomial are obtained via the public
  `Solve` builtin so inexact (Real / MPFR) input automatically gets the
  rationalise ‚Üí solve ‚Üí numericalize pipeline.
- For numerical input, tiny imaginary noise introduced by the Cardano
  formula on real cubics is chopped with a relative 1e-10 / absolute
  1e-12 threshold.
- Numerical eigenvalues are sorted in order of decreasing absolute
  value via `N[Abs[lambda]]`; symbolic eigenvalues retain Solve's
  natural ordering.
- `Eigenvectors` finds each eigenvalue's null space via `RowReduce`.
  When the null-space dimension is less than the algebraic multiplicity
  (defective eigenvalue) the shortfall is padded in-line with zero
  vectors so positional alignment with `Eigenvalues` is preserved.
- For approximate matrices, the eigenvector computation is done in the
  rationalised domain and numericalized + normalised at the end, so
  numerical matrices return unit-Norm eigenvectors.
- `Infinity` generalised eigenvalues correspond to the null space of
  the right-hand-side pencil matrix `a`; the matching eigenvectors are
  computed from that null space.
- Options: `Cubics`, `Quartics`, `Method` (`Method` reserved).  The
  defaults are `Cubics -> True` and `Quartics -> True` so the
  closed-form pipeline can numericalize results without held `Root[]`
  objects sneaking past `N`.

### Limitations

- Numerical eigenvectors of n ‚â• 3 inexact matrices substitute exact
  radical eigenvalues into the residual matrix; for n ‚â• 3 the Cardano-
  derived expressions can be too large for `RowReduce` to make progress
  in reasonable time.  A proper numerical eigenvector routine (inverse
  iteration / QR) is a follow-up.
- The 100√ó100 numerical example in the Mathematica reference docs is
  not yet supported -- it requires a tridiagonal-aware numerical
  eigenvalue method.

### Symbols

`SYM_Eigenvalues`, `SYM_Eigenvectors` added to `sym_names.{h,c}`.
Both builtins are `Protected`; docstrings registered in `info.c`.

### Verification

- New test binary `eigen_tests` (24 tests, all pass) covers exact 3√ó3
  with singular det, symbolic 2√ó2 round-trip verification, machine-
  precision 3√ó3 with chop, repeated eigenvalues, identity matrices
  (up to 12√ó12), 4√ó4 block matrices, all k-spec variants
  (`k`, `-k`, `UpTo[k]`), generalised exact / symbolic / k-spec,

## qa: coalesce algebraically-dependent integer-base Sqrt generators in extension_autodetect (2026-05-14)

`extension_autodetect` collected every integer-base `Sqrt[c]` it saw
as an independent generator, including ones algebraically dependent
on other collected generators (`Sqrt[15]` given `Sqrt[3]` and
`Sqrt[5]`, since `Sqrt[15] = Sqrt[3]¬∑Sqrt[5]`). The tower builder
doesn't detect the dependency, so it produced a degree-8 primitive

degree-4 minimum over `Q(Sqrt[3], Sqrt[5])`. `qa_cancel_with_tower`'s
cost is super-linear in `deg(Œ≥)`, so the inflation showed up directly

`GEN_INT_BASE` `Sqrt[c]` with composite squarefree `c` whose prime
factors are all present as other `Sqrt[p]` generators, marks them
`GEN_ABSORBED`, and drops them in the existing dedup-compaction sweep.

The companion `decompose_redundant_sqrts` runs at the start of
`qa_cancel_with_tower`: it walks the input and rewrites every
`Sqrt[c]` occurrence with a composite-squarefree `c` (all primes in
the smaller tower) into a raw `Times[Sqrt[p_1], ..., Sqrt[p_k]]`
constructed via `expr_new_function` ‚Äî NOT through `evaluate()`, so
Mathilda's `Times` canonicaliser does not immediately recombine the
product back to `Sqrt[c]`. The subsequent per-prime substitution
replaces each `Sqrt[p_i]` with its Œ≥-poly form, after which the
`Times` factors are Œ≥-polynomials with no `Sqrt` nodes left to
recombine when `Together` re-evaluates the tree.

Scope: positive integer bases, q=2 (Sqrt only), squarefree composites
with all prime factors as other `Sqrt[p]` generators. Higher q (cube
roots) and negative bases left for later ‚Äî their dependence patterns
are more involved and rarer in practice.

## rat: Together / Cancel skip extension_autodetect on nested-radical inputs (2026-05-14)

`Together[expr, Extension -> Automatic]` and `Cancel[expr, Extension -> Automatic]`
on inputs with a nested radical (e.g. `1/Sqrt[5 + 2 Sqrt[6]]^3 + (Sqrt[2] - Sqrt[3])^3`)
previously paid ~190 ms for `extension_autodetect` + `qa_cancel_with_tower`
substitution, only for the leaf-count gate to reject the result. The
Œ≥-substitution leaves the outer `Power[Plus[‚Ä¶], p/q]` opaque to the inner
no-extension `Together`, so the inflated result fails the gate and
fallback N-generator tries hit the same wall.

A new structural prefilter `expr_has_nested_radical_radicand`
(`src/qafactor.{c,h}`) fires when:
1. The input contains a `Sqrt[non_int]` / `Power[non_int, p/q]` (q ‚â• 2)
   whose radicand itself contains a radical, AND
2. The input has no free polynomial variable.

Both conditions matter: with free variables present (the headline
`D[Integrate[a x/(x^3+2), x], x]` case has `a` and `x`),
`qaupoly_gcd` over `Q(Œ≥)[x]` is the only thing that recovers the
closed form, so the tower path must run. Without free variables, the
qaupoly path has nothing to bite on and the nested radical prevents
useful constant arithmetic in `Q(Œ≥)`.

When the prefilter fires, both `builtin_together` and `builtin_cancel`
skip `extension_autodetect` entirely and fall through to the
no-extension path. A companion predicate `qa_tower_has_nested_radical`
serves the same purpose for callers that have already paid for tower
construction.

A second change: `builtin_together` and `builtin_cancel` are now thin
wrappers over `*_compute` variants that consult the active `FactorMemo`
(pushed by `builtin_simplify` for the duration of one Simplify call).
Repeated calls on the same input across `simp_search` rounds and
`simp_bottomup` descent return cached results.

### Performance

| Call | Before | After | Speedup |
|------|--------|-------|---------|
| `Together[diff, Extension -> Automatic]` | 190 ms | 2.5 ms | 76√ó |
| `Cancel[diff, Extension -> Automatic]`   | 187 ms | 2.4 ms | 78√ó |
| `Simplify[diff]`                          | 357 ms | 173 ms | 2.1√ó |

where `diff = 1/Sqrt[5 + 2 Sqrt[6]]^3 + (Sqrt[2] - Sqrt[3])^3`.

### Additional savings inside the tower path

Two further optimisations apply to inputs that DO go through
`qa_cancel_with_tower` (those Layer-0 doesn't skip):

**Pre-substitution gate** (`qa_cancel_with_tower` in `src/qafactor.c`).
The post-substitution leaf-count gate is now mirrored by a strict
lower-bound predictor that runs before the per-Œ± substitution loop.
For k radical occurrences and Œ≥ of degree d, each Power[c, p/q]
expands to a Œ≥-polynomial of at least 2(d-1) leaves; if `lc_in + 2k(d-1)`
already exceeds the gate, the substitution is guaranteed wasted and
we bail without running it.

**Guided fallback** (`pick_best_tower_generator` in `src/rat.c`).
When `qa_cancel_with_tower` declines, the previous N-try loop ran
`together_recursive_ext` once per tower generator (up to 4 times)
and kept the smallest result. The replacement ranks generators by
their occurrence count in the input and tries only the top one ‚Äî
the single-Œ± path's effectiveness is proportional to how often the
chosen Œ± appears, so the most-used Œ± is the natural pick. Cuts the
fallback cost from N√óper-attempt to 1√ó.

### Combined performance

| Call | Before | After Layers 0+1+3+5 + simp prefilter |
|------|--------|--------------------------------------|
| `Together[diff, Extension -> Automatic]` | 190 ms | 2.5 ms |
| `Simplify[diff]` (user's case)            | 357 ms | **32 ms** |
| `Simplify[D[Integrate[a x/(x^3+2), x], x]]` | ~190 ms | 114 ms |
| `Simplify[1/Sqrt[3 + 2 Sqrt[2]]]`         | (slow)  | 1.8 ms |
| `Simplify[Sqrt[5+2 Sqrt[6]]+Sqrt[5-2 Sqrt[6]]]` | (slow) | 17.8 ms |

The user's case improves further (181 ms ‚Üí 32 ms) after applying the
same Layer-0 prefilter inside `builtin_simplify`'s alg_top branch ‚Äî
that branch also called `extension_autodetect` to decide whether to
fire `Together[expr, Extension -> Automatic]`, and on nested-radical
inputs the autodetect call itself was ~150 ms. With the prefilter,
both the predicate and the subsequent Together call are skipped.

The headline case benefits most from Layer 1+3 because its multi-Œ±
tower (Q(2^(1/3), Sqrt[Œ±/6])) doesn't hit the Layer-0 nested-radical
gate but does benefit from the tighter inner gates.

## simp: multi-extension nested-radical denesting (2026-05-14)

Extended the Sqrt-of-Plus denester in `src/simp.c` to handle
phase-2 (multi-extension, Landau-style) radicands, closing a gap
where Mathematica simplified an expression in ~100 ms but Mathilda
left it untouched.  The motivating identity from the user report:

```
  Sqrt[16 - 2 Sqrt[29] + 2 Sqrt[55 - 10 Sqrt[29]]]
    -> Sqrt[5] + Sqrt[11 - 2 Sqrt[29]]
```

Previously only single-sqrt radicands `A + b Sqrt[C]` with `A` in
`Q` would denest; this commit lifts both constraints.

### Algorithm

The half-sum identity stays the same:

```
  P, Q = (A +/- Sqrt[A^2 - b^2 C]) / 2
  Sqrt[A + b Sqrt[C]] = Sqrt[P] + sgn(b) Sqrt[Q]
```

Phase 2 extends three pieces:

1. **Outer-sqrt selection (`split_plus_into_a_plus_b_sqrt_c`).**
   When a radicand has multiple sqrt-bearing terms, the splitter
   now ranks candidates by sqrt nesting depth (deeper outer first,
   tie-broken by leaf-count complexity).  `denest_compute_pq_s`
   iterates candidates until one yields a clean denesting.
   Phase-1 single-sqrt inputs hit candidate 0 once with
   byte-identical behaviour.
2. **Recursive discriminant (`sqrt_if_clean_square`).**  When
   `D = A^2 - b^2 C` is itself a Plus carrying a sqrt-bearing
   term, recursively denest `Sqrt[D]` and accept the result iff it
   leaves no `Sqrt[Plus]` residue.  Bounded by a file-static depth
   budget (default 4) and memoised on `D` to avoid revisiting
   subtrees across the bottom-up walker and the discriminant path.
3. **Q(Sqrt[k]) sign prover (`q_sqrt_extension_is_nonneg`).**  The
   branch-validity gate previously routed any non-literal `P` / `Q`
   through assumption-set reasoning, which couldn't decide
   `11 - 2 Sqrt[29] >= 0` (the second component of the motivating
   denesting).  Added an exact sign decider for `Q(Sqrt[k])`
   elements: collect `alpha`, `beta` from a Plus, compare
   `alpha^2` against `beta^2 gamma` over exact rationals.  No
   numeric sampling -- all arithmetic runs through the evaluator
   on rational literals.

### Files touched

- `src/simp.c`
  - New helpers: `sqrt_nesting_depth`,
    `plus_has_sqrt_bearing_term`, `contains_sqrt_of_plus`,
    `simp_denest_sqrt_recursive`, `count_outer_sqrt_candidates`,
    `is_numeric_rational`, `rational_sign`,
    `classify_q_sqrt_term`, `compute_discriminant_eval`,
    `q_sqrt_extension_is_nonneg`, `denest_memo_lookup` / `_insert`
    / `_clear`.
  - Refactored: `split_plus_into_a_plus_b_sqrt_c` accepts a
    `candidate_idx`; `denest_compute_pq_s` splits into outer
    iterator + `_at_candidate` per-attempt helper;
    `sqrt_if_clean_square` gains a phase-2 recursive case and
    reorders its polynomial-Plus path so a sqrt-bearing `D` falls
    through rather than aborting; `denest_is_nonneg` consults the
    new Q(Sqrt[k]) prover; the top-level `simp_denest_sqrt`
    entry manages the budget + memo lifetime with re-entry
    detection so an evaluator-driven sub-Simplify shares state.

### Tests

`tests/test_simp_denest_phase2.c` registered as
`simp_denest_phase2_tests`.  Nine cases, all passing:

- the user's exact reported expression collapsing to `0`;
- the radicand denesting alone;
- the inner phase-1 step
  (`Sqrt[152 - 24 Sqrt[29]] -> 2 Sqrt[29] - 6`) still firing
  under the single-extension path;
- three phase-1 sanity checks (`5+2 Sqrt[6]`, `7+4 Sqrt[3]`,
  `3+2 Sqrt[2]`);
- a non-denestable multi-sqrt input (`1 + Sqrt[2] + Sqrt[3]`)
  staying untouched;
- a single-sqrt input with negative discriminant staying
  untouched;
- idempotency of the phase-2 result under a second `Simplify`.

The pre-existing `tests/test_simp_denest_inv_sqrt.c` (60+
phase-1 cases) continues to pass unchanged.

### Implementation notes

- Re-entry on `simp_denest_sqrt`: if `g_denest_budget` is
  already nonzero an in-flight denesting is sharing state with
  an evaluator-driven sub-Simplify, so we don't reset the budget
  or memo.  Depth cap is enforced end-to-end.
- `sqrt_if_clean_square`'s polynomial path used to `return NULL`
  immediately when its input contained a non-integer Power;
  phase-2 reorganises this as a guarded skip so a sqrt-bearing
  `D` falls through to the phase-2 recursive case rather than
  aborting before it can fire.
- Memo lifetime is tied to the top-level entry; nested
  invocations share the parent's memo, freed once on exit.
- Soundness invariant: phase-2 accepts a recursive result only
  when `contains_sqrt_of_plus` is false -- structural evidence
  that the inner denester unrolled every nested radical.  No
  numeric sampling in the soundness check.

## core: TimeConstrained (2026-05-14)

Added `TimeConstrained[expr, t]` and `TimeConstrained[expr, t, failexpr]`
to `src/core.c`, with `SYM_TimeConstrained` interned in `sym_names.c`,
attributes `HoldAll | Protected` in `attr.c`, and a docstring in
`info.c`. Documentation lives in `docs/spec/builtins/time-and-date.md`.

- The two-argument form evaluates `expr` and returns its value if it
  completes within `t` seconds of CPU time. On timeout the call returns
  the symbol `$Aborted`. The three-argument form evaluates `failexpr`
  instead on abort; `failexpr` is held until the timeout fires.
  `TimeConstrained[expr, Infinity]` (or `DirectedInfinity[1]`) imposes
  no constraint.
- Implementation uses `ITIMER_PROF` + `SIGPROF` so that the kernel
  charges only this process's user+kernel CPU time toward the budget
  ("CPU time spent inside the main Mathilda kernel process"). The
  intrischnorman pmint timeout (wall-clock, `SIGALRM` / `ITIMER_REAL`)
  and the test-harness `alarm(60)` therefore remain operational.
- The previous `SIGPROF` handler and `ITIMER_PROF` state are saved on
  entry and restored on exit so nested `TimeConstrained` calls compose
  and any external profiler in the address space is left untouched.
- Abort path: a one-shot `sigsetjmp` / `siglongjmp` unwinds the
  in-flight evaluator. To repair the recursion-depth counter that
  `evaluate()` did not get a chance to decrement, a new
  `eval_reset_recursion_depth(int)` helper was added to `eval.c` /
  `eval.h`; it also clears the sticky `eval_overflow` flag so a future
  evaluation is not falsely poisoned. Memory allocated by the aborted
  computation is intentionally leaked, matching the Mathematica note
  that `TimeConstrained` may produce different results on different
  occasions.
- Time-budget parsing accepts `EXPR_INTEGER`, `EXPR_REAL`, `EXPR_BIGINT`,
  `Rational[p, q]`, the symbol `Infinity`, and `DirectedInfinity[+]`.
  Non-positive or NaN budgets abort immediately without entering the
  evaluator; sub-microsecond positive budgets are rounded up to 1 us so
  that `setitimer(0, 0)` (which would disarm rather than arm the timer)
  never occurs.
- Tests live in `tests/test_time_constrained.c` and cover the
  fast-completion path (2-arg and 3-arg), the `Infinity` short-circuit,
  zero / negative / non-numeric time inputs, bad arity, `Rational`
  budgets, the time-argument-is-evaluated guarantee, `failexpr` not
  being evaluated on success, the body being evaluated exactly once
  (`HoldAll`), the timer-driven abort (`$Aborted` and `failexpr`
  branches), nested `TimeConstrained` (inner abort + outer success),
  recovery of evaluator state after an abort, and the `Protected`
  attribute / `Attributes[TimeConstrained]` readback. CPU-burn cases
  accept either the aborted result or the (unlikely) completed result
  to stay portable across very fast hardware.

## core: TimeConstrained cooperative wall-clock backstop (2026-05-14)

Made `TimeConstrained` portable to hosts whose `ITIMER_PROF` is
unreliable -- in particular WSL 1, whose syscall-translation layer
under-counts CPU time and delivers `SIGPROF` many seconds late (a 1 s
budget there was observed to abort only after ~15 s).

- Added a second enforcement layer alongside the existing
  `ITIMER_PROF` / `SIGPROF` signal path: `builtin_time_constrained`
  now captures `clock_gettime(CLOCK_MONOTONIC) + budget` as a
  wall-clock deadline when it arms the timer, and `evaluate()`'s
  fixed-point loop calls a new `tc_check_deadline()` function once
  per rewrite step. On hosts where `ITIMER_PROF` works, the signal
  wins (sub-millisecond, CPU-time semantic preserved) and the check
  is a cheap inactive read. On hosts where it doesn't, the
  cooperative check trips at the next rewrite step and
  `siglongjmp`s exactly as the signal handler would, so the abort
  path (recursion-depth reset, `$Aborted` / `failexpr` selection,
  timer / handler restore) is shared and unchanged.
- The deadline state (`tc_deadline`, `tc_deadline_active`) is saved
  and restored around `sigsetjmp` so nested `TimeConstrained` calls
  compose: the outer-call deadline resumes correctly when the inner
  one returns.
- `CLOCK_MONOTONIC` is POSIX.1-2001 and supported on every host the
  project targets (glibc Linux, macOS 10.12+, WSL 1/2). A
  `clock_gettime` failure is treated as "no time elapsed" so the
  cooperative path can only ever cause spurious *non*-aborts, never
  spurious aborts; the signal path remains the fallback in that
  unreachable case.
- The header declaration of `tc_check_deadline` is in `src/core.h`;
  the call site is in the top of the `while` loop in `evaluate()`
  in `src/eval.c`. The new state and function definition live in
  `src/core.c` next to `tc_sigprof_handler`.
- Granularity caveat (unchanged): a single long-running C builtin
  (e.g. `FactorInteger` on a huge composite) cannot be aborted
  cooperatively, since the check fires only between evaluator
  rewrite steps. On WSL 1, those calls still wait for the late
  `SIGPROF`; on real Linux / macOS, the signal interrupts the
  builtin promptly.
- A new test `test_tc_cooperative_backstop_without_sigprof`
  simulates the WSL 1 scenario by blocking `SIGPROF` for the
  duration of the call (and pre-installing `SIG_IGN` as the saved
  disposition so any pending timer signal is harmlessly discarded
  on unblock). The test asserts that `TimeConstrained[Do[1 + 1, {i,
  1, 10000000}], 0.2]` still returns `$Aborted`, proving the
  cooperative path enforces the deadline on its own.

## simp: half-int powers, Algebraic short-circuit, nested-radical scoring (2026-05-13)

Three related improvements that together close the user-reported
follow-ups to the inv-sqrt denester landed earlier the same day:

1. **Sqrt[X]^k denesting for any odd k.** Previously
   `simp_denest_sqrt_walk` only fired on `Power[X, ¬±1/2]`, so the
   cubed-reciprocal form `1/Sqrt[5+2 Sqrt[6]]^3` (which the evaluator
   canonicalises to `Power[5+2 Sqrt[6], -3/2]`) walked past untouched.
   `Simplify[1/(5+2 Sqrt[6])^(3/2) + (Sqrt[2] - Sqrt[3])^3]` returned
   the input unchanged.

   The dispatch is now a single matcher `match_half_int_power(e, &m,
   &base)` that recognises `Power[base, m/2]` for any odd integer m,
   and a unified builder `try_denest_pow_half_int(plus_node, m, ctx)`
   that handles all four cases:
   - `m == 1`:  `Sqrt[P] + sign*Sqrt[Q]` (forward)
   - `m == -1`: `(Sqrt[P] - sign*Sqrt[Q]) / s` (reciprocal, rationalised)
   - `m > 1`:   `(Sqrt[P] + sign*Sqrt[Q])^m`
   - `m < -1`:  `(Sqrt[P] - sign*Sqrt[Q])^|m| / s^|m|`

   The three now-unused helpers from the earlier landing
   (`try_denest_sqrt_radicand`, `try_denest_recip_sqrt_radicand`,
   `is_inv_sqrt`) were removed. The shared `denest_compute_pq_s`
   helper survives unchanged; both old paths are now special cases
   of the new general builder.

2. **`Algebraic` early denester short-circuit.** Profiling the

## simp: reciprocal nested-radical denesting (2026-05-13)

Closes the user-reported identity
`Simplify[Sqrt[2] - Sqrt[3] + 1/Sqrt[5 + 2 Sqrt[6]]] -> 0`
and its multi-extension cousins
(e.g. `1/Sqrt[5+2 Sqrt[6]] + 1/Sqrt[5-2 Sqrt[6]] -> 2 Sqrt[3]`).

**The gap:** `simp_denest_sqrt_walk` (the half-sum nested-radical
denester) only matched `Power[X, 1/2]`. The reciprocal form
`Power[X, -1/2]` (i.e. `1/Sqrt[X]`) walked past untouched, so any
expression where the cancellation required `1/Sqrt[a + b Sqrt[c]]` to
be rewritten as the conjugate-rationalised `(Sqrt[P] -/+ Sqrt[Q])/s`
failed to reach zero ‚Äî the downstream `qa_substrate_general_pq` pass
never saw the linear-basis form it needed.

**The fix:**

1. Extracted a shared helper `denest_compute_pq_s(plus_node, ctx,
   &P, &Q, &s, &b_is_negative)` from `try_denest_sqrt_radicand`. On
   success this returns the half-sum data
       `(A + b Sqrt[C]) = (Sqrt[P] + sign * Sqrt[Q])^2`,
       `s = P - Q = Sqrt[A^2 - b^2 C]`.
   Both the forward and reciprocal paths reuse it, so the branch-
   soundness prover (`denest_is_nonneg`), the `D = A^2 - b^2 C`
   polynomial scaffolding, and the `Sqrt[m/n]` rationaliser all live
   in one place.

   Required fixing a latent dangling-pointer hazard in the original
   forward code path: `s` was consumed when constructing `Times[-1, s]`
   to build `Q = (A - s)/2`, then leaked afterwards. The new shared
   helper has to *return* `s`, so the consume site now takes a copy.

2. Added `is_inv_sqrt(e)` ‚Äî true on `Power[X, Rational[-1, 2]]` ‚Äî and a
   new builder `try_denest_recip_sqrt_radicand` that emits
       `(Sqrt[P] -/+ Sqrt[Q]) / s`
   using the conjugate-rationalisation identities
       `1 / (Sqrt[P] + Sqrt[Q]) = (Sqrt[P] - Sqrt[Q]) / (P - Q)`
       `1 / (Sqrt[P] - Sqrt[Q]) = (Sqrt[P] + Sqrt[Q]) / (P - Q)`.
   The sign in front of `Sqrt[Q]` flips relative to the forward path.

3. Extended `simp_denest_sqrt_walk` to dispatch on
       `is_sqrt(target) || (inv = is_inv_sqrt(target))`
   and route to the appropriate builder. Bottom-up traversal,
   branch-soundness checks, and idempotency all carry over unchanged
   from the forward path.

**Behavioural coverage:**

| Input | Output |
|-------|--------|
| `Simplify[Sqrt[2] - Sqrt[3] + 1/Sqrt[5 + 2 Sqrt[6]]]` | `0` |
| `Simplify[1/Sqrt[5+2 Sqrt[6]] + 1/Sqrt[5-2 Sqrt[6]]]` | `2 Sqrt[3]` |
| `Simplify[Sqrt[5+2 Sqrt[6]] - 1/Sqrt[5+2 Sqrt[6]]]` | `2 Sqrt[2]` |
| `Simplify[1/Sqrt[3 + 2 Sqrt[2]]]` | `-1 + Sqrt[2]` |
| `Simplify[1/Sqrt[5 - 2 Sqrt[6]]]` | `Sqrt[2] + Sqrt[3]` |
| `Simplify[1/Sqrt[2 + Sqrt[3]]]` | `(Sqrt[6] - Sqrt[2])/2` |
| `Simplify[1/Sqrt[1 + Sqrt[2]]]` | unchanged (negative discriminant) |
| `Simplify[1/Sqrt[x + Sqrt[y]]]` | unchanged (branch unverifiable) |

Standalone cases where the rationalised form does not strictly reduce
`LeafCount` (e.g. `1/Sqrt[5+2 Sqrt[6]]` alone, where both sides count
13 leaves) are added to the candidate seed set but Simplify's measure
keeps the input. Cross-term cancellation still works in these cases
because the seed reaches downstream rounds via `cs_add_or_free`.

## qa: multi-gen tower fallback to single-Œ± Together/Cancel (2026-05-13)

Closes the motivating headline case
`Together[D[Integrate[a x/(x^3 + 2), x], x], Extension -> Automatic]`,

combined-but-not-cancelled fraction in `2^(1/3)` and `2^(2/3)`.

### Problem

When `extension_autodetect` builds a tower with `n ‚â• 2` generators,
`builtin_together` and `builtin_cancel` route through

Œ±_i with its Œ≥-polynomial form; on deep multi-fraction inputs this blows
the leaf count past the safety gate (`lc_out > 100 && lc_out > 2 lc_in`,
or `lc_out > 200`). The function returns NULL and the caller drops to
the generic no-extension `together_recursive`, which cannot see the
algebraic relations between generators and produces a non-cancelled

### Fix

In both `builtin_together` and `builtin_cancel` (`src/rat.c`), when the

fold-up on each candidate to combine like-coefficient terms that the
single-Œ± path treats as polynomial-variable coefficients. Pick the
smallest result by `leaf_count_internal` that strictly beats the input.

The single-Œ± path treats the remaining generators as opaque polynomial-
variable coefficients but still cancels correctly over Q(Œ±)[x]; the
final no-extension `Together` then combines like terms (e.g. it pulls
`1/a` out of `1/(144 a) + x^3/(288 a)` to give `(2 + x^3)/(288 a)`).

## algebraic: autodetect harvests absorbed nested radicals + tower-blowup safety gate (2026-05-13)

Two coordinated `qafactor.c` fixes that keep the algebraic-extension
pipeline running cleanly when the Power canonicaliser (2026-05-12 +
2026-05-13 Power-factor patch) absorbs a nested radical into existing
integer-base generators.

### Autodetect: handle absorbed nested generators

`autodetect_canonicalise_post` previously rebuilt every GEN_NESTED
generator's render via `autodetect_canonicalise_radicand`. When the
canonical form is no longer a single atomic radical -- e.g.
`Sqrt[-1/9/2^(2/3) + 2/9¬∑2^(1/3)]` canonicalises to `Sqrt[1/3/2^(2/3)]`
which the Power canonicaliser then reduces to `1/(2^(1/3) Sqrt[3])`,
a Times of inverse powers -- the borrowed render was non-atomic and
`qa_resolve_extension` couldn't build a tower from it.
`extension_autodetect` returned NULL, costing every downstream caller
(Together / Cancel with `Extension -> Automatic`) the chance to use
algebraic-extension reasoning.

New helper `autodetect_is_atomic_radical_surface(e)` detects the
absorbed case. A new enum value `GEN_ABSORBED` marks affected
entries. When canonicalisation produces a non-atomic surface, the
post-canon pass:

1. Re-walks the simplified surface with `autodetect_walk` to harvest
   any newly-exposed integer-base components (deduping into existing
   `GEN_INT_BASE` entries via `autodetect_add_int`).
2. Marks the original entry as `GEN_ABSORBED`.
3. Step 2's dedup-and-compact sweep drops `GEN_ABSORBED` entries
   along with structurally-duplicate nested entries.

Result: `extension_autodetect` now produces a tower spanning just the
integer-base generators when the nested radical is absorbed, and
`Together` / `Cancel` route through the algebraic-extension code
path where they previously fell back to the non-extension default.

### Safety gate: Step 1 substitution blowup

`qa_cancel_with_tower`'s Step 1 substitution replaces each user-Œ±
render with its polynomial-in-Œ≥ form. For deep multi-fraction inputs
(e.g. the unintegrated Risch result for `Integrate[a x/(x^3+2), x]`)
each `Power[c, p/q]` occurrence expands to a degree-up-to-deg(Œ≥)
polynomial, and the post-Step-1 leaf count can be 3-10√ó the input.
Step 2's downstream `Together` (no-extension) then runs multivariate
polynomial GCD over `Q[Œ≥_internal, x, ...]` on the inflated form,
which is intractable.

New gate after Step 1: if the substituted leaf count exceeds 100
**and** more than doubles the original, return NULL so the caller
falls back to the non-tower path (which gives the partial
combined-but-not-cancelled result that the user already had before
the autodetect harvest fix unlocked this path). Empirically tuned to
let the small `Sqrt[A] - Sqrt[B] ‚Üí 0` and `(Sqrt[2]+Sqrt[3])(Sqrt[2]-
Sqrt[3]) ‚Üí -1` cases through while bailing on the deep
`Together[D[Integrate[a x/(x^3+2), x], x], Extension -> Automatic]`
input.

The blow-up case is the right next layer of work: a multi-generator
analogue of `together_recursive_ext` that threads the tower's
polynomial relations through `PolynomialLCM` / `PolynomialQuotient`
without pre-expanding to Œ≥-polynomial form. Deferred; tracked in
`tasks/qa_substrate_general_pq.md`.

## power: Power[Times[positives], p/q] admits Power factors that compose cleanly (2026-05-13)

The `Power[Times[positive_factors], p/q]` distribution rule
(`src/power.c`, 2026-05-12) gated distribution on at least one factor
being an integer or rational that **fully reduces** under `Power[_, p/q]`
(`factor_fully_reduces_under_q` -- residue 1 on both numerator and
denominator). This left examples like `Sqrt[1/3/2^(2/3)]` and
`Sqrt[3/2^(2/3)]` unevaluated even though the `Power[2, -2/3]` factor
*would* compose cleanly under the outer `Sqrt[_]` -- the issue was
that no factor was a perfect square.

New companion predicate `power_factor_composes_cleanly(f, p_out, q_out)`
admits a `Power[positive_base, p'/q']` factor as a distribution
trigger when composing the outer exponent `p_out/q_out` with the
inner `p'/q'` introduces **no new radical complexity**: composed
reduced denominator `<= q'`. The arithmetic reduces to a single
divisibility check: `q_out | |p' * p_out|`.

Examples now reduced:

```
Sqrt[3/2^(2/3)]       ->  Sqrt[3] / 2^(1/3)
Sqrt[1/3/2^(2/3)]     ->  1 / (2^(1/3) Sqrt[3])
Sqrt[Pi/2^(2/3)]      ->  Sqrt[Pi] / 2^(1/3)
```

Mathematica-faithful adjacency cases stay untouched (the divisibility
check rejects them):

```
Sqrt[2 Sqrt[3]]       ->  Sqrt[2 Sqrt[3]]    (q_out=2 does not divide 1)
Sqrt[2 Pi]            ->  Sqrt[2 Pi]         (no Power factor present)
(4 Pi)^(2/3)          ->  (4 Pi)^(2/3)       (4 has no cube factor)
```

Tests: `tests/test_power_corpus.c::test_power_of_times_power_factor_composes`.

### Knock-on effect: algebraic-extension headline

This patch is the substrate fix for the deeper Together motivating
example. With nested `Sqrt[1/3/2^(2/3)]` now simplifying,
`Together[D[Integrate[a x/(x^3+2), x], x], Extension -> Automatic]`
produces a clean rational expression in `x` and `2^(1/3)/2^(2/3)`
(no nested Sqrts) instead of an opaque form. Full collapse to
`a x / (x^3 + 2)` is gated on a downstream polynomial-GCD-over-Q(Œ≥)
step which is not yet wired into Together's cancel path. Tracked as
a remaining limitation in `tasks/qa_substrate_general_pq.md`.

## algebraic: nested-radical generators in extension_autodetect (2026-05-13)

`extension_autodetect` (`src/qafactor.c`) previously bailed on any
`Power[base, p/q]` whose base wasn't an integer literal ‚Äî meaning
`Sqrt[1 + Sqrt[2]]`, `Sqrt[2^(1/3)]`, and similar nested radicals
caused the entire detection to return NULL.

The walker now surfaces nested radicals as a separate `GEN_NESTED`
generator class, deferring resolution to `qa_resolve_nested_radical`
(Phase G8 in `src/qafactor.c`).  Integer-base generators (the original
`GEN_INT_BASE` class) are unchanged.

Concretely:

```
Sqrt[1 + Sqrt[2]]            ‚Üí 1 generator, deg 4 (compositum minpoly y^4 - 2 y^2 - 1)
Sqrt[2^(1/3)]                ‚Üí 1 generator, deg 6 (= 2^(1/6))
```

And in the `Simplify` pipeline:

```
Simplify[Sqrt[1 + Sqrt[2]]^2]            ‚Üí 1 + Sqrt[2]
Simplify[(Sqrt[2^(1/3)])^2]              ‚Üí 2^(1/3)
Simplify[Sqrt[2^(1/3)/6] Sqrt[2^(1/3)/6]] ‚Üí 1/(3 2^(2/3))
```

**Design note.**  The walker does NOT recurse into a nested radical's
base.  `Sqrt[1 + Sqrt[2]]` does not also surface `Sqrt[2]` as a
separate generator ‚Äî G8 builds its own sub-tower from the base's
atomic radicals.  Recursing would double-count the inner radical and
inflate the compositum degree.

**Limitations remaining.**

- Structural-equality dedup only: two nested radicals that are
  mathematically equal but written differently (e.g.
  `Sqrt[-1/9¬∑2^(-2/3) + 2/9¬∑2^(1/3)]` vs `Sqrt[1/(3¬∑2^(2/3))]`, both
  equal `Sqrt[2^(1/3)/6]`) are still treated as distinct generators.
  Mathematical-equality dedup would require a canonical-form pass
  over the radicand before generator collection.
- Multi-generator detection (n ‚â• 2) still bails at the Cancel/Together
  wiring layer to no-extension ‚Äî tier-2 work to route towers through
  the qaupoly substrate end-to-end.

**Tests.**  Extends `tests/test_extension_autodetect.c` (2 new tests:
`test_sqrt_of_sqrt`, `test_sqrt_of_cbrt`) and
`tests/test_simp_algebraic_cuberoot.c` (3 new tests:
`test_nested_sqrt_squared`, `test_sqrt_of_cbrt_squared`,
`test_sqrt_of_cbrt_quotient_squared`).

**Regression risk.**  Full test suite 94 pass / 16 fail (identical
failure set, one further new passing test from the cube-root binary
that incorporates these tests).  Zero new regressions.

## algebraic: multi-generator tower routing for Cancel / Together (2026-05-13)

`qa_cancel_with_tower` (`src/qafactor.{c,h}`) is a new public helper
that handles the n ‚â• 2 case of `extension_autodetect`.  When the
input contains multiple independent algebraic generators (e.g.
`Sqrt[2] + Sqrt[3]`), the helper:

1. Substitutes each user-side Œ±_i in the input with its polynomial-
   in-Œ≥ form (where Œ≥ is the tower's primitive element).
2. Runs the standard `Together` to combine into a single fraction.
3. Lifts numerator and denominator to QAUPoly over Q(Œ≥).
4. Divides by their `qaupoly_gcd`.
5. Renders back through `qaupoly_to_expr_alpha` with Œ≥'s surface
   form.

Wired into both `builtin_cancel` and `builtin_together` (`src/rat.c`)
under `Extension -> Automatic`.  Previously these fell through to the
no-extension path on multi-generator detection.

**Worked example** that now collapses end-to-end:

```
Cancel[(Sqrt[2] + Sqrt[3]) (Sqrt[2] - Sqrt[3]), Extension -> Automatic]
   ‚Üí  -1
```

Direct evaluation expands to `2 - 3 = -1`; the qa_cancel_with_tower
path reaches the same answer through the Q(Œ≥) lift + qaupoly_gcd.

**Limitations.**

- The render-back step uses Œ≥ as the surface "Œ±" so results sometimes
  expose `Œ≥` as `Sqrt[2] + Sqrt[3]` (or whatever the primitive element
  is) inside the rational form.  Mathematical equivalence is preserved;
  cosmetic minimisation (re-expressing Œ≥-polynomials in terms of the
  user's original Œ±_i basis) is not yet done.
- Lift failures (a coefficient that isn't in Q(Œ≥)) cleanly return
  NULL and fall back to the no-extension path.

**Tests.**  `tests/test_extension_auto_builtins.c` adds 2 new tests
(`test_multigen_sqrt2_sqrt3_collapse`, `test_multigen_together_no_op`).

**Regression risk.**  Full test suite still 94 pass / 16 fail ‚Äî
identical failure set.  Zero new regressions.

## algebraic: canonical-form dedup of nested radicals (2026-05-13)

`extension_autodetect` now runs a post-walker canonicalisation pass on
nested-radical generators (`GEN_NESTED`).  Each radicand is rewritten
via `Together[radicand, Extension -> Œ±]` against the surfaced
integer-base generators, then re-split into atomic radicals (so that
`Power[c, p/q]` with `|p| > 1` is rewritten as
`c^floor(p/q) ¬∑ Power[c, (p mod q)/q]`).  After canonicalisation,
structurally-distinct-but-mathematically-equal `Sqrt[u]` terms collapse
to a single generator.

Surfacing integer bases from inside a nested radicand's base is done
via a dedicated `autodetect_walk_intbase_only` sub-walker that ONLY
collects non-atomic `Power[c, p/q]` (p != 1) forms.  Atomic
`Power[c, 1/q]` / `Sqrt[c]` sub-expressions are deliberately skipped ‚Äî
G8's nested-radical resolver already handles them via its own
sub-tower, and surfacing them again at the top level created a
redundant degree-doubling compositum (regression test:
`Sqrt[1+Sqrt[2]]` correctly resolves to a single generator of
degree 4, not 8).

**Worked example.**  Two structurally distinct but mathematically
equal nested Sqrts now compare equal under `Simplify`:

```
Simplify[Sqrt[-1/9/2^(2/3) + 2/9 2^(1/3)] - Sqrt[1/3/2^(2/3)]]   ‚Üí  0
```

(Both radicands canonicalise to `1/3¬∑2^(-2/3)` so the two Sqrts dedup
to a single GEN_NESTED generator, and Simplify's complexity-driven
search drives the difference to zero.)

**Limitations remaining.**

- The motivating end-to-end case (the derivative of `Integrate[a x /
  (x^3 + 2), x]` collapsing to `a x^2 / (x^3 + 2)`) still doesn't
  close: even after canonicalising both radicands to identical
  `Sqrt[1/3¬∑2^(-2/3)]`, G8's `expr_is_atomic_algebraic` recogniser
  rejects `Power[2, -2/3]` inside the radicand (it only accepts
  `Power[c, 1/n]`).  The deeper reason is that Mathilda's Times
  canonicaliser (`src/times.c` around line 481) deliberately
  absorbs integer coefficient factors into Power exponents ‚Äî for
  example `(1/6)¬∑2^(1/3)` is canonicalised to `(1/3)¬∑2^(-2/3)`,
  pulling the factor of 2 in the `1/6 = 1/(2¬∑3)` denominator into
  the radical's exponent.  This is Mathilda's chosen canonical form
  and is referenced widely, so the right fix is to extend the
  qa-substrate (`qa_resolve_extension` and
  `qa_expr_to_qaupoly_with_alpha`) to accept `Power[c, p/q]` for any
  integer `p`, lifting it as the polynomial-in-Œ± form via
  `qa_inverse` for negative exponents.  Deferred as a multi-hundred-
  LoC substrate change for follow-up.

**Tests.**  `tests/test_simp_algebraic_cuberoot.c` adds 1 new test
(`test_canon_dedup_equal_radicands_subtract`).

**Regression risk.**  Full test suite still 94 pass / 16 fail ‚Äî
identical failure set.  Zero new regressions.

## qa substrate ‚Äî Power[c, p/q] for general integer p (2026-05-13)

Extends `qa_resolve_extension` (`src/qafactor.c`),
`expr_is_atomic_algebraic` (G8 atom recogniser), and the QAUPoly lift
(`qa_expr_to_qaupoly_with_alpha`) to accept `Power[c, p/q]` with any
integer `p` (previously only `p == 1`).  This is the substrate fix
that lets Mathilda's Times-canonicaliser-produced forms like
`Power[2, -2/3]` participate in algebraic-extension reasoning.

### A. Recogniser broadening

After gcd-reducing `p/q` to lowest terms `p_red/q_red`, the recogniser
and resolver now accept any `q_red >= 2` with integer base `|c| >= 2`.
`qa_resolve_extension` returns a QAExt for `y^q_red - c` with
`*render_out` set to the natural form `Power[c, 1/q_red]` (or
`Sqrt[c]` when the user originally typed `Sqrt[c]` with `p == 1` for
byte-for-byte compatibility with existing tests).

### B. Lift inputs containing `Power[c, p/q]`

New helper `qa_alpha_power_signed(ext, p)` in `qa.{c,h}`: computes Œ±^p
‚àà Q(Œ±) for any integer p, using `qa_inverse` for `p < 0` and
repeated-squaring for efficiency.

New helper `expand_radicals_to_atomic_poly` in `qafactor.c`: rewrites
every `Power[c_base, p/q]` whose reduced denominator divides
`q_natural` into the polynomial-in-`alpha_sym` form representing Œ±^k,
where Œ± = c_base^(1/q_natural).  Raw structural rewrite ‚Äî does NOT
call `evaluate()` so Mathilda's Times canonicaliser doesn't undo it.

`qa_expr_to_qaupoly_with_alpha` infers `(c_base, q_natural)` from the
QAExt's `y^q - c` shape and runs `expand_radicals_to_atomic_poly`
before the existing `alpha_render ‚Üí alpha_sym` substitution.  G8's
`qa_resolve_nested_radical` and `qa_cancel_with_tower` both also run
the expand pass per-Œ±_i before substituting Œ±_i into Œ≥.

### C. Complexity gate in `qa_cancel_with_tower`

When the compositum's primitive element Œ≥ is a sum of generators,
rendering Q(Œ≥)-polynomials in user-Œ± form can produce expressions
much larger than the input.  Added a `leaf_count_internal`-based gate
at both the variable-bearing and no-variable return paths: return
NULL when the candidate output exceeds the input by leaf count, so
the caller falls back.  `leaf_count_internal` is now exposed via
`core.h` for this purpose.

### D. Proper Q(Œ≥)-arithmetic in the no-variable path

For Q(Œ≥)-constant inputs, the previous implementation substituted
`Œ≥_internal ‚Üí Œ≥_render` into the intermediate polynomial form,
producing high-degree user-form polynomials.  The new path reduces
num/den modulo Œ≥'s minimal polynomial via `PolynomialRemainder`,
lifts each to a QAUPoly with a dummy variable, extracts the QANum
constant, divides via `qa_div` (real Q(Œ≥)-arithmetic), and renders
the canonical QANum's coefficient array.  Result:
`Cancel[(Sqrt[2]+Sqrt[3])(Sqrt[2]-Sqrt[3]), Extension -> Automatic]`
now produces `-1` instead of the degree-16 Œ≥-polynomial expansion.

### Worked examples

```
Cancel[Power[2, -2/3] x^2 - x/Power[2, 1/3], Extension -> Power[2, 1/3]]
   ‚Üí 1/2 (-2^(2/3) x + 2^(1/3) x^2)

Cancel[(Sqrt[2] + Sqrt[3])(Sqrt[2] - Sqrt[3]), Extension -> Automatic]
   ‚Üí -1
```

### E. Input-side radicand canonicalisation

`canonicalise_nested_radicands(e, c_base, q_natural)` walks `e` and
rewrites every `Sqrt[u]` / `Power[u, p/q]` whose base `u` is a
non-integer expression mentioning `c_base` by canonicalising `u` via
`Together[u, Extension -> Power[c_base, 1/q_natural]]`.  This is the
input-side counterpart of Phase F's `autodetect_canonicalise_post`
(which canonicalised the generator-set surface forms).  Without it,
`Together[Sqrt[A] - Sqrt[B], Extension -> Automatic]` with `A == B`
mathematically (but structurally distinct) missed one of the Sqrts at
substitution time.

Step 0 of `qa_cancel_with_tower` runs this pass against each
integer-base generator in the tower (`Power[c, 1/q]` or `Sqrt[c]` with
`|c| >= 2`) before the user-Œ± ‚Üí polynomial-in-Œ≥ substitution loop.

`autodetect_canonicalise_post` was also generalised: it now handles
both `Sqrt[radicand]` (1-arg) and `Power[radicand, p/q]` surface forms
and rebuilds every nested generator's render with a positive `1/q`
exponent.  This collapses the `Power[X, -1/q]` shape that Mathilda's
Power canonicaliser produces (e.g. from `1/Sqrt[X]`) into the same
canonical generator as `Sqrt[X]` / `Power[X, 1/q]`, so the autodetect
dedup pass merges them.

A zero-numerator shortcut was added at the Q(Œ≥)-constant return path:
when the QAUPoly numerator is zero (regardless of denominator
representation), return `0` directly instead of running it through
`qa_div`.  Required because the input-side canonicalisation can now
turn `Sqrt[A] - Sqrt[B]` into a literal `0` before the tower lift, and
the previous path bailed when the polynomial form was zero.

### Worked example

```
Together[Sqrt[-1/9/2^(2/3) + 2/9 2^(1/3)] - Sqrt[1/3/2^(2/3)],
         Extension -> Automatic]
   ‚Üí 0
```

### Limitations remaining

- `Together[D[Integrate[a x/(x^3+2), x], x], Extension -> Automatic]`
  still produces a non-collapsed form containing `Sqrt[1/3/2^(2/3)]`
  embedded in arithmetic.  Closing this requires factoring the integer
  power out of nested radicands so that `Sqrt[1/3/2^(2/3)]` is
  recognised as `Sqrt[3]/(3¬∑2^(1/3))`, putting `Sqrt[3]` and `2^(1/3)`
  into the autodetected generator set rather than the opaque
  `Sqrt[1/3/2^(2/3)]`.

## power: rational-base and rational-exponent canonicalisation (2026-05-12)

`Power` (`src/power.c`) now reduces fractional powers of rational
bases that previously stayed unevaluated, and composes Powers of
positive bases under non-integer outer exponents. The motivating
failures were:

```
In[1]:= (1/54)^(2/3)
Out[1]= (1/54)^(2/3)         (* expected: 1/(9 * 2^(2/3)) *)

In[2]:= Sqrt[(1/54)^(2/3)]
Out[2]= Sqrt[(1/54)^(2/3)]   (* expected: 1/(3 * 2^(1/3))   *)
```

Three coordinated additions to `builtin_power`:

1. **Rational-base + rational-exponent distribution.** For
   `Power[Rational[n, d], p/q]` with `q > 1`, distribute as
   `Power[n, p/q] * Power[d, -p/q]` and recurse. Each piece routes
   through the existing integer-base + rational-exponent path which
   extracts the q-th-power coefficient and leaves a residue
   `Power[r, b/q]`. Triggering is gated on at least one piece
   simplifying: `|n| == 1`, or `factor_out_kth_power(|n|, q)` /
   `factor_out_kth_power(d, q)` extracts a factor > 1. This keeps
   `Sqrt[2/3]` and `(4/9)^(2/3)` unevaluated (Mathematica-faithful)
   while reducing `(1/54)^(2/3)`, `(8/27)^(2/3) -> 4/9`,
   `Sqrt[4/9] -> 2/3`, `Sqrt[1/4] -> 1/2`, and
   `(-1/8)^(1/3) -> (-1)^(1/3)/2`.

2. **`Power[Times[positive_factors], p/q]` distribution.** Distributes
   the outer Power when every factor is known positive and at least
   one factor cleanly reduces to a rational coefficient under
   `Power[_, p/q]` (factor_out residue == 1 on both numerator and
   denominator). Examples: `Sqrt[4 Pi] -> 2 Sqrt[Pi]`,
   `(Pi/8)^(2/3) -> Pi^(2/3)/4`, and -- crucially for chaining --
   `Sqrt[(1/9) * 2^(-2/3)] -> (1/3) * 2^(-1/3)`. Stays conservative
   for `(4 Pi)^(2/3)` (no clean cube extraction) and
   `Sqrt[2 Sqrt[3]]` (the Power factor cannot fully reduce).

3. **Power-of-Power composition for positive base + rational outer.**
   Generalises the existing integer-outer-exponent rule to
   `Power[Power[B, E_inner], p/q] -> Power[B, E_inner * p/q]` when
   `B` is known positive. Fixes `Sqrt[2^a] -> 2^(a/2)` and
   `Sqrt[Pi^2] -> Pi`; carefully leaves `Sqrt[x^4]` unevaluated since
   `x` has unknown sign. This is what reduces the inner
   `Power[Power[2, -2/3], 1/2]` to `Power[2, -1/3]` during evaluation
   of `Sqrt[(1/54)^(2/3)]`.

The cascade for the user's case:

```
Sqrt[(1/54)^(2/3)]
  -> Sqrt[(1/9) * 2^(-2/3)]                  (rational-base distribution)
  -> Sqrt[1/9] * Sqrt[Power[2, -2/3]]        (Times-of-positives distribution)
  -> (1/3) * Power[2, -1/3]                  (rational-base + Power-of-Power)
```

Soundness: every distribution is gated on positive sub-bases or on
integer outer exponents, so no principal branch cut is crossed.

## simp: trig-rational fast path for large derivative-of-Risch-Norman inputs (2026-05-12)

`Simplify` now ships a polynomial-time algebraic normal-form
pre-check (`src/trigrat.c`, `simp_trig_rational`) that handles
rational functions of trigonometric and hyperbolic functions
without entering the heuristic round-loop. It runs from
`builtin_simplify` (`src/simp.c`) before `simp_bottomup` and is also
hooked into `simp_dispatch` for the `SIMP_SHAPE_TRIG` case, with a
strict leaf-count gate so it cannot regress any existing case.

### Motivation

Two concrete failures on the previous build (both hung
indefinitely):

- `Simplify[D[Integrate`RischNorman[Tan[x]^2 + Tan[x] + 1, x], x]]`
- `Simplify[D[Integrate`RischNorman[x^4 Sin[x] Cos[x], x], x]]`

These are derivatives of Risch-Norman integration outputs and
should simplify back to the original integrand (or an equivalent
compact form). `simp_search` was slow because its
`Together`/`Cancel`/`Factor`/`TrigFactor` pipeline treats every
`Log[trig(x)]` subtree as an algebraic generator, and the
resulting multivariate polynomial GCDs blow up.

### Algorithm

Pure rational-function reduction in the polynomial quotient ring

    K[ s_i, c_i, sigma_j, chi_j, other_vars ]
        / < s_i^2 + c_i^2 - 1,  chi_j^2 - sigma_j^2 - 1 >

for distinct trig arguments {a_i} and hyperbolic arguments {b_j}.

1. **Preprocess + substitute** (single top-down walk in
   `tr_walk_subst`). Walks transparently through `Plus`/`Times`/
   `Power[_, integer_exp]`/`Rational`/`Complex`. Rewrites
   `Tan`/`Cot`/`Sec`/`Csc`/`Tanh`/`Coth`/`Sech`/`Csch` to their
   `Sin`/`Cos` (or `Sinh`/`Cosh`) form using *non-evaluating*
   constructors (the evaluator otherwise canonicalises
   `Sin[x]/Cos[x] -> Tan[x]` and infinite-loops the preprocess).
   `Sin`/`Cos`/`Sinh`/`Cosh` with a single arg get a fresh `(s, c)`
   or `(sigma, chi)` generator. Every other function call -- `Log`,
   `Exp`, `Sqrt`, `f[...]` -- becomes an opaque atom substituted by
   a fresh ground-field symbol, with the original subtree stored
   verbatim for back-substitution.
2. **Together** on the substituted form.
3. **Ideal reduction**: rewrite `s_i^n -> s_i^(n-2)(1 - c_i^2)` and
   `sigma_j^n -> sigma_j^(n-2)(chi_j^2 - 1)` for `n >= 2`,
   `Expand`-ing between passes until each odd generator has
   degree <= 1.
4. **Denominator rationalisation**: for every odd generator still
   in the denominator, multiply numerator and denominator by the
   conjugate (negate that generator in the denominator) and
   re-reduce. After this loop the denominator is in `c_i`/`chi_j`/
   other-vars only.
5. **Cancel** the resulting rational function: the existing
   multivariate-GCD `Cancel` builtin handles the trailing
   simplification in `K[c_i, chi_j, ...]`.
6. **Back-substitute** the fresh symbols and re-evaluate. The
   evaluator's auto-canonicalisation handles the `Sin/Cos -> Tan`
   and `1/Cos -> Sec` fold-back implicitly.
7. **Strict leaf-count gate** (`simp_default_complexity`): adopt
   the result only if it is strictly smaller than the input.

### Configuration

- **Budget**: up to 4 distinct trig+hyp arguments,
  24 distinct opaque atoms.
- **Leaf-count floor**: 60. Below this, defer to `simp_bottomup` --
  `simp_search`'s TanAddition / HalfAngle / TrigReduce produce
  more compact double/half-angle / angle-addition forms on small
  inputs, and the strict-win gate would otherwise miss them by a
  tie.

### Opaque-monomial grouping (the speed bump from 10 s ‚Üí 0.2 s)

After substitution, the substituted form is a rational function in
the trig generators `(s_i, c_i)` plus the opaque-atom generators
`(L_1, ..., L_k)`. A naive `Together / Cancel` runs multivariate
polynomial GCD in all `k + 2` variables -- exponential in `k` and the
chief cost on inputs that integrators produce (where every distinct
`Log[trig(x)]` becomes one opaque generator).

The grouping path (`tr_simp_l_grouped` in `src/trigrat.c`) partitions
each Plus addend into `(opaque_monomial, trig_rational_rest)`, groups
addends with structurally equal opaque monomials, and applies the
pure (s,c)-only pipeline (`tr_simp_pure_sc`) to each group's sum --
so every per-group Cancel is bivariate. On the `Tan` integration the
expression splits into 5 groups (`{1, L_1, L_2, L_3, L_4}`) and runs
in ~160 ms instead of ~10 s. The fallback single-fraction pipeline
fires when any addend has an opaque atom in a denominator
(`Power[L_k, -n]`) or fused with trig inside a non-distributable
structure (`Power[Plus[L_k, ...], _]`); the first pass is tried
without `Expand` (cheap) and one `Expand` retry is allowed for cases
where products of sums hide the structure.

### Compound-argument canonicalisation

When any collected trig argument is non-atomic (e.g. `Sin[x + Pi/6]`
or `Sin[2 x]`), the algorithm cannot recognise angle-addition or
multiple-angle relationships between distinct trig generators -- it
would otherwise produce a strict leaf-count win on a partially-
simplified form and lock simp_search out of the canonical answer
(typically `0` via TanAddition). To handle this, the entry point
calls `TrigExpand` on the input and redoes the walk; TrigExpand
unfolds `Sin[x + Pi/6]` to `(Sqrt[3]/2) Sin[x] + (1/2) Cos[x]` and
`Sin[2 x]` to `2 Sin[x] Cos[x]`, leaving only atomic-argument trig
calls. TrigExpand is cheap when arguments are already atomic, so
checking before calling it (via a single pass through the bindings
table) avoids paying its cost on the user's derivative-of-Risch-
Norman inputs where every Sin/Cos already has argument `x`. If
TrigExpand can't reduce a particular compound argument (e.g.
`Sin[f[x]]`), the entry point bails to simp_search rather than
risk a partial normal form.

### Results

| Input                                                                          | Before      | After    | Final form                              |
| -------------------------------------------------------------------------------| ------------|----------|-----------------------------------------|
| `Simplify[D[Integrate`RischNorman[Tan[x]^2 + Tan[x] + 1, x], x]]`              | hangs       | ~0.17 s  | `Sec[x]^2 (1 + Cos[x] Sin[x])`          |
| `Simplify[D[Integrate`RischNorman[x^4 Sin[x] Cos[x], x], x]]`                  | hangs       | ~1.2 s   | `x^4 Cos[x] Sin[x]`                     |

The first case reduces the 234-leaf derivative back to the
integrand-equivalent `Sec[x]^2 + Tan[x]` (printed in factored form
as `Sec[x]^2 (1 + Cos[x] Sin[x])`); the second reduces the
992-leaf derivative all the way back to the original integrand.
The Tan case timing is within ~2x of Mathematica's reported ~0.1 s.

## intrischnorman: Bronstein parallel-Risch / Risch-Norman heuristic (2026-05-12)

New transcendental integrator `Integrate`RischNorman[f, x]` wired
into `Integrate` as the Stage-2 fall-through behind the rational
integrator.  Implements Bronstein's pmint algorithm (see
`parallel_risch/pmint.maple` and `Bronstein - Parallel Risch.pdf`)
in `src/intrischnorman.c` (~2900 LOC).  Plan in `plans/RISCH_NORMAN_PLAN.md`;
phase tracker in `tasks/risch_norman.md`.

Pipeline:

- **Weierstrass rewrite** (`convert_to_tan`): full half-angle
  substitution `T = Tan[u/2]`, plus the closed-form inverses
  `Tan[u/2]^2 -> (1-Cos[u])/(1+Cos[u])`, etc., on the way back out.
  `decot_rec` undoes Mathilda's automatic `1/Tan -> Cot` collapse so
  the indeterminate set stays canonical.
- **Indeterminate collection + total-derivation construction**
  (`collect_indets_closed`, `build_vector_field`): treats every
  transcendental atom as a fresh polynomial variable; the
  derivation acts as a polynomial vector field on the resulting
  ring.  Pythagorean rewrite `Sec^2 -> 1+Tan^2`, etc., for
  closure.
- **Darboux + log candidate enumeration** (`get_special_all`,
  `my_factors`): includes the standard Darboux polynomials
  (`1+T^2` for Tan, `1¬±T` for Tanh, `T` for LambertW) plus all
  irreducible factors of numerator/denominator and total
  derivatives.
- **Linear-system solve** (`solve_linear_undet`): polynomial-equation
  form `E := ff_n * cden^2 * q - ff_d * (apply_d(num)*cden -
  num*apply_d(cden)) - ff_d*cden^2 * Œ£ B_j * apply_d(g_j) *
  ŒÝ_{k‚âÝj}(g_k)`; multivariate `CoefficientList` extracts the
  linear system, Bareiss-style `RowReduce` solves it.  Avoids the
  Together-blow-up that plagues the naive rational-candidate form.
- **K = Q ‚Üí Q(I) retry**: on failure, factors the denominator
  over `Q(I)` via `Factor[_, Extension -> I]` (Trager algebraic
  factoring) to unlock log candidates needing imaginary roots
  (e.g. `1/(1+x^2)`-style integrands surfaced through the
  Weierstrass substitution).
- **SIGALRM + sigsetjmp wall-clock budget** so a runaway candidate
  cannot stall the dispatcher.

Output cleanup (algorithmic, no `Simplify` ‚Äî per the rule that
`Simplify` is too slow on the hot path):

1. `ReplaceAll` with eight half-angle inverse rules (Tan, Cot,
   Tanh, Coth √ó power 1 / power 2)
2. `Log[a_] :> Log[Together[a]]` to flatten fractional Log args
3. `Log[a_/b_] :> Log[a] - Log[b]` expansion
4. `Together` to combine
5. Fresh-symbol substitution (`Cos[x] -> pmint$cc`, `Sin[x] ->

## intrat: parametric closure for quadratics, biquadratics, and bi-quadratics-in-t (2026-05-11)

Three closely-related fixes in `src/intrat.c` widen the parametric
reach of the LRT-then-real-form pipeline so that the families

  Integrate[1/(a x^2 + b x + c), x]                  (parametric quadratic)
  Integrate[1/(b + a x^4), x]                        (parametric Sophie-Germain)
  Integrate[1/(x^4 + a^4), x]
  Integrate[1/(a x^4 + b x^2 + c), x]                (parametric biquadratic)
  Integrate[x^2/(a x^4 + b x^2 + c), x]

now close to a real `Plus` of `ArcTan`/`Log` instead of falling through
to NaiveLogPart's held `RootSum`.

1. **`logtoreal_quadratic` parametric branch**.  When the discriminant
   sign is indeterminate under the positive-symbol assumption (the
   canonical case `disc = b^2 - 4 a c` with `a, b, c` independent
   symbols), the dispatcher now defaults to the negative-disc / ArcTan
   branch with a held `Sqrt[-disc]`.  That formula is the principal
   antiderivative for both signs of the discriminant ‚Äî `ArcTan`
   continues analytically through imaginary argument ‚Äî and matches
   Mathematica's parametric-quadratic output.

2. **Sqrt-simplifier negation distributes**.  The negation step
   `neg_disc_raw = -1 * disc` now passes through `expr_expand`, so the
   downstream `intrat_simp_pos_sqrt` sees a flat Plus rather than
   `Times[-1, Plus[...]]`.  Without this, the Sqrt simplifier walked
   the Times factors and pulled `Sqrt[-1] = I` out of the leading
   `-1`, producing an imaginary `v_root` that collapsed
   logtoreal_quadratic's output to 0.  The bug was masked on rational
   inputs (no Plus survived the constant fold) and only surfaced for
   the parametric quadratic now reachable via fix (1).

3. **Sophie-Germain `ratio_pos` widens to positive-symbol assumption**.
   The deg-4 dispatch's `ratio_pos = c0/c4 > 0` check accepted only
   literal positive integers/rationals.  It now accepts any expression
   provably positive under the positive-symbol assumption (e.g.
   `1/(256 a^12)` for the parametric `1/(x^4 + a^4)` family).

4. **Biquadratic-in-`t` branch (NEW)**.  The deg-4 dispatch gains a
   `c1 = 0, c3 = 0, c2 != 0` path that splits
   `c4 t^4 + c2 t^2 + c0` via the substitution `u = t^2` into
   `(t^2 - u_+)(t^2 - u_-)` with
   `u_¬± = (-c2 ¬± Sqrt[c2^2 - 4 c0 c4]) / (2 c4)`, then dispatches each
   factor through `logtoreal_quadratic` (whose new parametric branch
   from fix (1) handles the unknown-sign discriminant
   `4 u_¬±` cleanly).  This is what catches the
   `1/(a x^4 + b x^2 + c)` family.

(`D[result, x] - integrand /. {a -> 2, b -> 5, c -> 7, x -> 3}`
evaluates to ‚â§ 1e-17 in `N[]`).  All `intrat` and `simplify` test

## intrat: biquadratic-in-t gate to preserve correctness on Sqrt[3]-laden inputs (2026-05-11)

Refined the biquadratic-in-`t` branch (added earlier today) with an
inner-discriminant soundness gate.  The branch substitutes `u = t^2`
and splits Q(t) into two quadratics `(t^2 - u_¬±)`, where
`u_¬± = (-c2 ¬± Sqrt[c2^2 - 4 c0 c4]) / (2 c4)`.  When the inner
discriminant `c2^2 - 4 c0 c4` is negative or sign-indeterminate, `u_¬±`
form a complex conjugate pair, the sub-factors have complex
coefficients, and `logtoreal_quadratic` (which assumes real
coefficients) silently miscomputes ‚Äî typically collapsing to 0 because
the imaginary contributions cancel between the two sub-factor calls.

Caught by the IntegrateRationalTests corpus (case 54, a Sqrt[3]-
coefficient quartic with negative inner discriminant) which the
ungated branch was returning 0 for instead of the correct
~LeafCount-1700 expression.

The gate uses the existing `intrat_sign_pos_assumption` helper.  The
practical effect: parametric biquadratic families like
`1/(a x^4 + b x^2 + c)` (whose inner disc `16 (b^2 - 4 a c)^3` has
sign-indeterminate `(b^2 - 4 a c)^3` under the positive-symbol model)
revert to NaiveLogPart's held RootSum form (correct, just unsimplified)
rather than risk a silent wrong answer.  The `1/(b + a x^4)` and
`1/(x^4 + a^4)` families remain closed via the older Sophie-Germain
branch (a different code path with a unconditional positive
discriminant).

This is the conservative-correctness trade.  A future "decide whether
b^2 > 4 a c is generically expected" hint would let the branch fire
for those parametric biquadratics too.

## poly: PolynomialGCD content-extraction handles Plus bases (2026-05-11)

Fixed a structural-correctness bug in `builtin_polynomialgcd`'s
integer-content (`numG`) accumulation:

  Cancel[4/(4 + 4 x)]      -> 4/(4 + 4 x)            (now: 1/(1 + x))
  Cancel[4/(4 + 8 x)]      -> 4/(4 + 8 x)            (now: 1/(1 + 2 x))
  Cancel[4 x/(4 + 8 x)]    -> (4 x)/(4 + 8 x)        (now: x/(1 + 2 x))

Root cause: the j-loop that peels integer content into `num_i` only
fired on `is_number(base)` factors of the bp-decomposition.  For an
input like `4 + 4 x` the bp had a single Plus factor (`Plus[4, 4 x]`,
exp=1); `is_number(Plus[...])` is false, so `num_i` stayed at 1.  The
asymmetry meant `PolynomialGCD[4, 4 + 4 x]` returned 1 (rather than 4),
and `Cancel` then refused to reduce.

Two cooperating edits in `src/poly.c`:

1. `get_int_content` extends to Plus expressions: returns the GCD of
   summand contents, with a short-circuit at 1.

2. The j-loop in `builtin_polynomialgcd` gains a Plus-base branch.
   When a bp factor is `Plus[...]` with integer content `c > 1`, it
   accumulates `c` into `num_i` AND replaces the base with its
   primitive part `Plus[a_i / c]` so the downstream `poly_gcd_internal`
   does not extract the same content again (which would multiply numG
   by the factor twice -- causing
   `PolynomialGCD[4 + 4 x, 8 + 8 x]` to incorrectly return
   `(4 + 4 x) * 4` instead of `4 + 4 x`).

Verified on Cancel cases above plus the existing
`Cancel[(4 + 4 x)/(8 + 8 x)] -> 1/2` regression (still passes).  All
`poly`, `rat`, `parfrac`, `facpoly`, `intrat`, `simp`, `simplify`,
`expand`, `deriv` test suites pass.

## intrat: empty LRT pair list returns NULL (silent wrong-zero fix) (2026-05-11)

The case above now correctly returns

  RootSum[Function[-1 - #1^8 + #1^16],
          Function[Log[x - #1] #1^3/(-8 #1^7 + 16 #1^15)]]

(differentiates back to the integrand under positive-symbol
assumption; verified num-diff = 0 at x = 7/4).

Pre-existing wrong-answer regression on `IntegrateRationalTests` case 33
is now correct.  Case 46 (degree-16 squared denominator) still
produces a wrong result -- separate bug, likely in the Hermite

## intrat: closed-form completions for five RootSum-leak inputs (2026-05-11)

Five `Integrate`BronsteinRational[...]` inputs reported as still
returning unevaluated `RootSum` (or otherwise unsimplified) now
close to real-elementary form:

  1.  `1/(b - a x^3)`                          (cubic / nth-root w. params)
  2.  `1/(x^5 + 1)`                            (palindromic-quartic Q)
  3.  `x^4/(a + b x^3)^2`                      (Hermite-reduced cubic)
  4.  `(-1 + x^2)/(1 - 2 x^2 + 2 x^4)`         (biquadratic Q, neg. inner disc)
  5.  `1/x/(1 - 2 x^2 + 2 x^4)`                (post-processing distribution)

Each issue corresponded to a distinct gap in the C port's
`logtoreal_dispatch` / `expand_simple_rootsum` / final-cleanup
chain.  The five fixes are independent:

**Phase A ‚Äî output post-processing.**  Two new passes in
`intrat_integrate_rational`:

  -  `intrat_distribute_plus(e)` calls `expr_expand` to flatten
     `Times[scalar, Plus[‚Ä¶]]` that the per-summand `c_k ¬∑ piece_int_k`
     accumulator leaves held (Times has no built-in Plus-distribution
     attribute).
  -  `intrat_strip_log_constants(e, x)` rewrites `Log[c ¬∑ p]` to
     `Log[p]` whenever `FreeQ[c, x]`.  The `Log[c]` summand is a
     constant of integration, harmless to drop, and the stripping
     lets the downstream `intrat_log_to_arctanh` pairing rule fire
     across previously-blocked term boundaries.

Both passes run before AND after `intrat_log_to_arctanh` so the
log-pairing rule sees fully-distributed sums and any newly-formed
`Times[scalar, Plus[‚Ä¶]]` from the pairing is re-distributed.

Fix for issue 5.

**Phase B ‚Äî Sophie-Germain-with-c2 for biquadratic with negative
inner discriminant.**  `logtoreal_dispatch` deg-4 `c1z && c3z`
branch previously bailed when `c2^2 ‚àí 4 c0 c4 < 0`; now it falls
into a new sub-branch that factors

    c4 t^4 + c2 t^2 + c0
       = c4 (t^2 + Œ± t + Œ≤)(t^2 ‚àí Œ± t + Œ≤)

with `Œ≤ = Sqrt[c0/c4]` and `Œ± = Sqrt[2 Œ≤ ‚àí c2/c4]`, both real under
the negative-inner-disc regime (algebraically: `c2^2 < 4 c0 c4`
forces `2 Sqrt[c0 c4] > |c2|`, hence `2 Sqrt[c0/c4] ‚àí c2/c4 > 0` when
`c4 > 0`).  Each real-coefficient quadratic factor has discriminant
`Œ±^2 ‚àí 4 Œ≤ < 0`, dispatching through `logtoreal_quadratic`'s ArcTan
branch.

Fix for issue 4.

**Phase C ‚Äî cubic dispatch in `logtoreal_dispatch`.**  New deg-3
branch for nth-root cubics `c3 t^3 + c0` (c1 = c2 = 0): real root
`r0 = ‚àí(c0/c3)^(1/3)` (principal real cube root, sign-selected by
`intrat_sign_pos_assumption` so positive-symbol parameters route
correctly).  Factor as

    c3 (t ‚àí r0)(t^2 + r0 t + r0^2)

and dispatch the linear factor through `build_log_term` and the
quadratic through `logtoreal_quadratic` (discriminant `‚àí3 r0^2 < 0`,
always ArcTan branch).

Fix for issues 1 and 3.  Non-depressed cubics (`c1` or `c2` non-zero)
remain unhandled ‚Äî Cardano's formula port is a future extension.

**Phase D2 ‚Äî palindromic-quartic real-form expansion in
NaiveLogPart.**  New helper `expand_palindromic_quartic_real`
detects palindromic `d_t = c0 t^4 + c1 t^3 + c2 t^2 + c1 t + c0`
(c0 == c4, c1 == c3), solves the `u = t + 1/t` substitution
`c0 u^2 + c1 u + (c2 ‚àí 2 c0) = 0` for u_¬±, and for each conjugate
root pair `Œ±_k = u_k/2 ¬± i Sqrt[(4 ‚àí u_k^2)/4]` builds the real

    C_re ¬∑ Log[(x ‚àí u_k/2)^2 + (4 ‚àí u_k^2)/4]

where `C = a(Œ±_k) / d'(Œ±_k) = C_re + i C_im` is split via the
existing `split_re_im` helper.  This bypasses the LogToReal /
LogToAtan chain entirely, which otherwise overwhelms on the
nested-radical S polynomials from the scaled-palindromic LRT path.

Fix for issue 2.

**Soundness gates.**  Every new sign decision is gated through
`intrat_sign_pos_assumption` first (symbolic positive-symbol walk),
then a fresh `intrat_numeric_sign` helper that runs `N[expr]` for
expressions like `Sqrt[5] ‚àí 5` that the symbolic walker leaves at
sign-unknown but a numerical sample resolves cleanly.  Failure to
decide a sign always falls through to the held-`RootSum` form
rather than risking a silent miscomputation.

## intrat: general nth-root cyclotomic factorisation in logtoreal_dispatch (2026-05-11)

Replaces the deg-3 nth-root and deg-4 Sophie-Germain shortcuts in
`logtoreal_dispatch` with a single sparse-polynomial handler
`logtoreal_nthroot_sparse(base, deg, s, x, t)` that closes
`c_n t^n + c_0` (all intermediate coefficients zero) for every
n ‚â• 3.

**Motivation.**  The 2026-05-11 Phase-C cubic fix only handled
`c_3 t^3 + c_0`.  Nearby inputs like `1/(b - a x^4)`,
`1/(b - a x^5)`, `1/(b - a x^6)`, `1/(b + a x^6)`, `1/(b + a x^8)`,
`1/(b + a x^9)` all still produced `RootSum[Function[‚Ä¶], ‚Ä¶]`
because their LRT-Q polynomials are sparse nth-root forms with
n ‚àâ {3} or the Sophie-Germain n = 4, q > 0 case.  Each of them is
factorable in real closed form via the standard cyclotomic
decomposition; the C port just wasn't doing it.

**Algorithm.**  For `c_n t^n + c_0` let `q = ‚àíc_0 / c_n`.

  -  If `q > 0`: roots are `r ¬∑ exp(2œÄ i k / n)` for k = 0..n‚àí1
     with `r = q^(1/n) > 0`.  Real roots are `t = r` (always) and
     `t = ‚àír` when n is even.  Complex pairs are indexed
     k = 1..‚åä(n‚àí1)/2‚åã (skipping k = n/2 when n even); the matching
     conjugate is at index `n ‚àí k`.

  -  If `q < 0`: roots are `r ¬∑ exp((2k+1) œÄ i / n)` for k = 0..n‚àí1
     with `r = |q|^(1/n) > 0`.  Real root `t = ‚àír` exists only for
     odd n (at k = (n‚àí1)/2, angle œÄ).  Complex pairs are k = 0..‚åä(n‚àí2)/2‚åã
     with the matching conjugate at `n ‚àí 1 ‚àí k` (skipping the real
     root index when n is odd).

Each complex pair contributes a real quadratic factor
`t^2 ‚àí 2 r cos(Œ∏_k) t + r^2` whose discriminant
`‚àí4 r^2 sin^2 Œ∏_k` is always strictly negative, so the ArcTan
branch of `logtoreal_quadratic` always fires.  Mathilda evaluates
`Cos[k œÄ / n]` to closed radical form for the cyclotomically nice
n (3, 4, 5, 6, 10, 12, ‚Ä¶); for other n (8, 9, 11, ‚Ä¶) the `Cos`
remains held but the result is still real-elementary (no `RootSum`
or `Function` leak).

**Sign selection.**  The branch fires when
`intrat_sign_pos_assumption(q)` is definite, falling back to
`intrat_numeric_sign(q)` (an `N[]`-based decision with a 1e-12
dead zone).  Both gates undecidable ‚áí bail to the held-`RootSum`
form ‚Äî sound under all sign instances.

**Subsumes.**  The previous deg-3 nth-root branch and the deg-4
Sophie-Germain (c_1 = c_2 = c_3 = 0) branch are now removed.
The deg-4 biquadratic (c_1 = c_3 = 0, c_2 ‚âÝ 0) and palindromic
sub-branches are kept for non-sparse quartics.  Behaviour for
inputs the old branches handled is unchanged.

**Tests.**  `tests/test_intrat.c` adds six new regressions
(`test_closed_nth_root_quartic_minus` / `_quintic_minus` /
`_sextic_minus` / `_sextic_plus` / `_octic_plus` /
`_nonic_plus`).  Each asserts no `RootSum` / `Function` head and
numerically verifies the derivative at two sample points.

**Regression risk.**  Full test suite shows 90 pass / 15 fail both
before and after this change ‚Äî identical failure set (canonical-
ordering pre-existing failures from commit `cc8b164`).  Zero new
regressions.
