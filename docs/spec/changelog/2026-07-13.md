# Changelog: week of 2026-07-13 (Mon) – 2026-07-19 (Sun)

Feature additions and fixes recorded during this week.

## Performance — Plus/Times like-term collection is O(n) (2026-07-16)

`builtin_plus` and `builtin_times` collected like terms (same base → summed
coefficients / summed exponents) with a linear scan of every existing group,
making a sum or product of *n* terms over *g* distinct bases O(n·g) — quadratic
when the terms don't collapse. Both now index groups in an open-addressing hash
table keyed on `expr_hash(base)`, so collection is O(1) expected per term. The
canonical output form is unchanged (group encounter order preserved; the
evaluator's Orderless pass still canonicalises the result), so results are
byte-for-byte identical to before — only faster.

Measured (Apple M-series):

- `Total[c·x^k]` over 10⁵ monomials (10³ distinct powers): **1.65 s → 0.50 s**.
- Product of 4·10⁴ distinct symbolic factors: **41 s → 0.23 s** (was O(n²)).

Both operations now scale linearly in the number of terms. Touches
`src/plus.c` and `src/times.c` only.

## Special functions — SinhIntegral and CoshIntegral (2026-07-16)

Added the hyperbolic sine and cosine integrals, the imaginary-axis siblings of
`SinIntegral` / `CosIntegral`:

- **`SinhIntegral[z]`** — Shi(z) = ∫₀ᶻ Sinh[t]/t dt. Entire and odd, no branch
  cut (Shi(z) = −i Si(i z)). New in `src/special_functions/sinhintegral.c`.
- **`CoshIntegral[z]`** — Chi(z) = EulerGamma + Log[z] + ∫₀ᶻ (Cosh[t] − 1)/t dt.
  Logarithmic singularity at 0, branch cut on `(−∞, 0]` (Chi(z) = Ci(i z) − i π/2).
  New in `src/special_functions/coshintegral.c`.

Each is `Listable, NumericFunction, Protected` with an Information string.
Highlights:

- **Numeric evaluation** for machine- and arbitrary-precision reals and
  complexes via MPFR / the `ncpx` toolkit. The convergent Maclaurin series are
  the trigonometric ones with the alternating sign removed (all terms positive),
  so the real paths carry no cancellation guard; large |z| uses the
  optimally-truncated asymptotic (`cosh F + sinh G` for Shi, `sinh F + cosh G`
  for Chi) with a Stokes constant off the real axis and the principal `Log`
  branch for Chi. Reals whose result overflows a C double (e.g.
  `SinhIntegral[10.^6] ≈ 1.5166×10^434288`) return an extended-exponent MPFR
  real rather than `Infinity`.
- **Exact special values**: `Shi[0]=0`, `Shi[±∞]=±∞`, `Shi[±I∞]=±Iπ/2`;
  `Chi[0]=−∞`, `Chi[±∞]=∞`, `Chi[±I∞]=±Iπ/2`; `ComplexInfinity → Indeterminate`.
  Negative real `Chi[-x]` returns the from-above branch value `Chi(|x|) + Iπ`.
- **Derivatives** (`src/calculus/deriv.c`): `D[SinhIntegral[z],z] = Sinh[z]/z`,
  `D[CoshIntegral[z],z] = Cosh[z]/z` (with chain rule).
- **Series** (`src/calculus/series.c`): Taylor at the origin (analytic for Shi;
  the `EulerGamma + Log[x] + …` logarithmic series for Chi) and the
  hyperbolic-prefactored asymptotic expansion at Infinity
  (`-Iπ/2 + …`).
- **Integrate** (`src/internal/CRCMathTablesIntegrals.m`):
  `Integrate[Sinh[a x]/x, x] = SinhIntegral[a x]`,
  `Integrate[Cosh[a x]/x, x] = CoshIntegral[a x]`. The trigonometric base cases
  `Integrate[Sin[a x]/x, x] = SinIntegral[a x]` and
  `Integrate[Cos[a x]/x, x] = CosIntegral[a x]` were added at the same time,
  completing the dangling Formula 387/388 negative-power reductions.

Extensive unit tests in `tests/test_sinhintegral.c` and
`tests/test_coshintegral.c`; no new memory leaks (valgrind-clean against the
macOS baseline).

## Parser — top-level newline terminates a statement (2026-07-16)

Fixes [GitHub issue #20](https://github.com/stblake/mathilda/issues/20): a file
loaded with `Get["file.m"]` whose expressions were on separate lines *without*
trailing `;` returned the wrong value. Two consecutive top-level expressions
such as

```
Series[Sin[x], {x, 0, 5}]
Integrate[Sin[x], x]
```

were being joined by **implicit multiplication** (`Series[…] * Integrate[…]`)
because the parser treated the line break as ordinary whitespace, so `Get`
returned the product instead of the last statement's value.

The parser now treats a line break at **bracket depth 0** as a soft statement
terminator (Mathematica semantics), while line breaks *inside* `()`, `[]`, `{}`,
`<|…|>`, or `[[…]]` remain insignificant. Concretely:

- `a\nb` → two statements `a`, `b` (was `a b`).
- `f[a,\n b]` and `{1,\n 2}` → still one expression each.
- A line ending in an operator (`1 +\n2`, `x =\n 5`) still continues onto the
  next line.

Implementation (`src/parse.c`): `ParserState` gains a `bracket_depth` counter
(maintained by the list/call/association/paren/part parsers) and a sticky
`saw_newline` flag (set by `skip_whitespace`, cleared when a primary token is
consumed). The implicit-multiplication branch of `parse_expression_prec` breaks
instead of inserting `Times` when `bracket_depth == 0 && saw_newline`. Regression
coverage: `test_parse_newline_separator` in `tests/test_parse.c`.

## CRCTable — circular ↔ hyperbolic symmetry fills (2026-07-16)

A systematic audit of the CRC integral table (`src/internal/CRCMathTablesIntegrals.m`)
surfaced a broad set of forms the circular trig rules covered but the hyperbolic
side (and, for reciprocal products, the circular side too) did not. Root causes
and fixes, all verified by differentiate-back to the integrand:

- **`Sinh^m Cosh^n` products** (`Integrate[Sinh[x]^4 Cosh[x]^3, x, Method -> "CRCTable"]`
  returned unevaluated). The Formula 547 reduction bottomed out on single-`Cosh`
  / single-`Sinh` factors that had no base rule. Added the `Sinh Cosh`,
  `Sinh Cosh^m`, `Sinh^m Cosh` bases and generalised Formula 547 to argument
  `a x`.

- **Canonical-head reciprocal products.** Mathilda canonicalises `1/Sin -> Csc`,
  `1/Cos -> Sec`, `1/Sinh -> Csch`, `1/Cosh -> Sech`, so the table's
  `1/(Sin^m Cos^n)` and `1/(Sinh^m Cosh^n)` rules never matched an evaluated
  integrand — `Csc[x]^4 Sec[x]^3` failed in **both** circular and hyperbolic.
  Added head-keyed reductions on the `Csc/Sec` and `Csch/Sech` heads. The
  recursive term is `+` for circular (`1 = Sin^2 + Cos^2`) and `−` for
  hyperbolic (`1 = Cosh^2 − Sinh^2`).

- **Argument generality (`a x`).** The single-function hyperbolic bases
  (`Sinh`, `Cosh`, `Tanh`, `Coth`, `Sech`, `Csch`), the squares, and
  `Sech Tanh` / `Csch Coth` bound a bare `x`, so every `a != 1` reduction that
  reached a base (e.g. `Sech[2 x]^3`) failed to close. Generalised to `a_. x_`.

- **`E^(a x) Sinh^n` / `E^(a x) Cosh^n`.** Ported the circular `E^(a x) Sin^n` /
  `Cos^n` reductions. The hyperbolic denominator is `a^2 − n^2 b^2` (vs the
  circular `a^2 + n^2 b^2`, never zero), which **vanishes at resonance**
  `a = n b` — e.g. `E^x Cosh[x]` (`a = b = 1`), where `e^x = cosh x + sinh x`
  forces a secular `x/2` term. The reductions carry an `a^2 =!= n^2 b^2` guard
  and resonant `a^2 === b^2` base cases (`e^(2 a x)/(4 a) ± x/2`). The `Cosh`
  recursion also carries a `−` on its integral term where `Sinh` carries `+`
  (the `cosh^2 = 1 + sinh^2` vs `sinh^2 = cosh^2 − 1` sign). The exotic
  intermediate resonance `a = n b, n > 1` (e.g. `E^(2x) Cosh[x]^2`) is left to
  the general cascade.

- **Mixed tangent/secant powers** `Tan^m Sec^n`, `Cot^m Csc^n` (circular) and
  `Tanh^m Sech^n`, `Coth^m Csch^n` (hyperbolic). These are the canonical forms
  Mathilda folds the Formula 307/308 quotients into (`Cos^2/Sin^3 -> Cot^2 Csc`,
  `Sin^2/Cos^3 -> Sec Tan^2`), so those quotient rules never matched either. The
  reductions strip the `Tan`/`Cot` power two at a time down to the single
  `Sec^n`/`Csc^n` rules, with the `Tan Sec^n` / `Cot Csc^n` bases (an optional
  exponent covers the single-`Sec`/`Csc` case, i.e. `Sec Tan` and `Csc Cot` —
  which the circular CRC tables omit though the hyperbolic `Sech Tanh` /
  `Csch Coth` carry them). Recursive-term sign is `−` for circular
  (`Sec^2 = 1 + Tan^2`) and `+` for hyperbolic (`Sech^2 = 1 − Tanh^2`).

- **Removed** the truncated, RHS-less Formula 528 statement
  (`E^(a x) Cos^m Sin^n`), which was dead syntax that never fired.

Not table forms (correctly still unresolved under `Method -> "CRCTable"`, but
handled by the default cascade via Risch): `E^(a x)` times reciprocal trig
powers such as `E^x Sech[x]^3`, `E^x Csch[x]^3` (and the circular `E^x Sec[x]^3`).

- **Polynomial × hyperbolic** (`x^n Sinh/Cosh` generalised from bare `x` to
  `a x`; `x Sinh^m` / `x Cosh^m` for Formulas 381-386; `x Csch^n` / `x Sech^n`
  = `x/Sinh^n`, `x/Cosh^n` for Formulas 410-413, keyed on the canonical
  `Csch`/`Sech` heads — even `n` closes on the `x Csch^2` / `x Sech^2` bases,
  odd `n` bottoms out on the non-elementary `x Csch` / `x Sech` and is left
  unresolved as the circular `x/Sin`, `x/Cos` are). The parallel circular
  `x Csc^n` / `x Sec^n` (dead for the same canonicalisation reason) were
  re-keyed too. `Sinh[a x]/x^m -> SinhIntegral` is intentionally **not** added:
  `SinhIntegral` / `CoshIntegral` carry no derivative rule yet, so the result
  would be unverifiable (a separate special-function task).

Tests: `test_crctable_hyperbolic_products`, `test_crctable_reciprocal_head_products`,
`test_crctable_exp_hyperbolic`, `test_crctable_hyperbolic_argument_generality`,
`test_crctable_mixed_tan_sec_families`, `test_crctable_polynomial_hyperbolic` in
`tests/test_integrate_dispatch.c`. The rational-in-hyperbolic (Weierstrass)
family (`1/(a + b Cosh)`, `1/(a Cosh^2 + b Sinh^2)`, …) remains a follow-up.

## CRCTable — Sec^n / Csc^n integer-power reduction (2026-07-16)

- `Integrate[Csc[x]^4, x, Method -> "CRCTable"]` (and the `Sec` sibling) formerly
  returned unevaluated: the CRC table only carried the reciprocal-power forms
  `1/Cos[a x]^m` and `1/Sin[a x]^m`, which never match the `Sec` / `Csc` heads,
  even though the hyperbolic `Sech[x]^n` / `Csch[x]^n` reductions were already
  present. Added the two missing rules in `src/internal/CRCMathTablesIntegrals.m`:

  ```
  ∫ Sec[a x]^n dx = Sec[a x]^(n-2) Tan[a x] / (a(n-1)) + (n-2)/(n-1) ∫ Sec[a x]^(n-2) dx
  ∫ Csc[a x]^n dx = -Csc[a x]^(n-2) Cot[a x] / (a(n-1)) + (n-2)/(n-1) ∫ Csc[a x]^(n-2) dx
  ```

  The `Csc` recursion carries a **+** on its integral term (`Csc^2 = 1 + Cot^2`),
  unlike the hyperbolic `Csch` analogue's **−** (`Csch^2 = Coth^2 − 1`) — the sign
  that a naive mirror of `Csch` gets wrong. Both reductions terminate at the
  degree-1 base cases (`Sec[a x]` / `Csc[a x]`, odd `n`) or a constant (even `n`).

- Audited the full integer-power reduction family: all six circular
  (`Sin/Cos/Tan/Cot/Sec/Csc`) and six hyperbolic (`Sinh/…/Csch`) functions now
  reduce; `Sec^n` and `Csc^n` were the only gaps.

- Tests: `test_crctable_secant_cosecant_powers` (diff-back of the new rules, even
  and odd `n`, with an `a x` argument) and `test_crctable_trig_power_family` (a
  12-head regression guard, `n = 2..6`) in `tests/test_integrate_dispatch.c`.

## Exact trig/exp zero-test — hyperbolic kernels + algebraic-extension points (2026-07-16)

Extended the exact grid zero-test (`simp_trigexp_zero.c`, shared by `Simplify`'s
`transform_trigexp_vanish`, the Risch verifier `rt_verify_antideriv`, and
`PossibleZeroQ`) from circular-only to hyperbolic and real-exponential kernels,
fixing the last flaky Risch trig-frontend decline, `1/(2 + Cosh[x])`.

- **Hyperbolic kernels.** `classify_kernel`/`kernel_value`/`poly_deg` now also
  recognise `Sinh`/`Cosh`/`Tanh`/`Coth`/`Sech`/`Csch` and the real exponential
  `E^(k x)`, with the I-free kernel `t = E^(x)` (the circular set keeps
  `t = E^(I x)`). `decide_with_var` picks the mode from which family is present;
  an expression mixing both (`Tanh[x] - Tan[x]`) needs two kernels and declines
  to the sampler. This closes `1/(3 + 2 Sinh[x])`, `Sinh`/`Cosh`/`Tanh`/… diff-backs,
  and hyperbolic identities under `PossibleZeroQ` (`Cosh[2x] - 2 Cosh[x]^2 + 1`,
  `Sinh[2x] - 2 Sinh[x] Cosh[x]`).
- **Algebraic-extension point values.** A radical-coefficient antiderivative
  (the `Q(Sqrt[3])` partial-fraction constants in the `1/(2 + Cosh[x])` diff-back)
  evaluates each grid point to an exact constant in `Q(alpha)` that the cheap
  classifier leaves undecided. The point evaluator now retries once through
  `Together` (exact rationalisation over the extension) when a point is
  undecided, so a genuine zero certifies and a genuine nonzero stays nonzero.
- **Soundness guard.** A commensurate *non-integer* kernel (a half-angle
  `Tanh[x/2]` alongside `Cosh[x]`, or `E^(x/2)`) is algebraically dependent on
  the integer-multiple kernels but not representable on the integer-power `t`
  axis; `collect` now declines on it instead of mis-treating it as an
  independent generator (which produced a spurious `False` for
  `1/(5 + 3 Cosh[x])`). Genuinely independent nested transcendentals
  (`Cosh[x^2]`, `Tanh[Log[x]]`) stay opaque as before. Verified: identities →
  `True`, non-identities → `False`, `zero_test`/`trigexp_zero` suites green, no
  added leaks.

## Risch trig frontend — Csc[x]^3 no longer flakily declines (exact zero-test fix) (2026-07-16)

`Integrate`RischTranscendental[Csc[x]^3, x]` returned unevaluated after ~21 s
(flaky — it occasionally succeeded on warm state) even though it produces a
correct antiderivative
(`-ArcTanh[E^(I x)] + E^(I x)/(-1 + E^((2 I) x)) + 2 E^(I x)/(1 + E^((4 I) x) - 2 E^((2 I) x))`).
The integrand's diff-back verification (`rt_verify_antideriv`) could not prove
`D[G] - Csc[x]^3 == 0`: the fast Laurent `Together` leaves `E^(±k I x)` poles
uncombined, the exact grid zero-test declined, and the residual `Simplify` ran
16 s and gave up.

- **Root cause** in the exact grid zero-test (`simp_trigexp_zero.c`, `poly_deg`):
  an I-laden diff-back carries explicit Gaussian-rational coefficients
  `Complex[a, b]`, which are `EXPR_FUNCTION` nodes with head `Complex`. `poly_deg`
  hit its unbounded-head fallback on them and returned "no degree bound," so the
  whole grid test declined — even though a `Complex[...]` literal is a constant
  (degree 0 in every axis), exactly as `collect()` already treated it. Fixed by
  giving `poly_deg` the same rule: any subterm free of the kernel variable is a
  constant coefficient regardless of head (`Complex[a, b]`, `Sqrt[2]`, `Pi`, …).
- **Effect:** the grid test now proves the Csc[x]^3 diff-back zero in ~9 ms
  (was: decline → 16 s Simplify → give up). `Integrate`RischTranscendental[Csc[x]^3, x]`
  succeeds in ~25 ms and is **deterministic** (5/5 → exact diff-back 0), closing
  the flaky `test_circular_trig_integration`. Rigorous both ways — a genuine
  non-identity with `Complex[...]` coefficients still certifies FALSE, no leaks.
  (Pre-existing and unrelated: even powers `Csc[x]^4`/`Sec[x]^4` remain slow via
  the realify path — confirmed slow with the fix reverted; not in the test suite.)

## Simplify — TrigToExp-collapse fallback for I-laden diff-backs (2026-07-16)

`Simplify` now recognises a further family of trig/exp expressions as
identically zero: those mixing a **real** exponential kernel `E^x` with the
**imaginary** kernel `E^(I x)` *and* carrying a bare polynomial coefficient in
the kernel variable — canonically the Risch antiderivative diff-back
`D[∫ x E^x Sin[x] dx] - x E^x Sin[x]`. Previously `Simplify` stalled on such
inputs at a non-zero intermediate like
`x (-E^x Sin[x] + (-I/2) E^((1+I) x) + (I/2) E^((1-I) x))` (non-idempotent —
a second `Simplify` pass reached 0), and the assert-active
`integrate_risch_transcendental_tests` aborted at `test_multikernel_case`.

- The rigorous exact grid zero-test (`trigexp_rational_is_zero`) declines by
  design on bare polynomial dependence on the kernel variable and on mixed
  real+imaginary exponential kernels it treats as independent opaque
  generators. `transform_trigexp_vanish` now adds a cheap, **sound** fallback
  invoked *only when that test declines* (never when it proves non-zero):
  `TrigToExp[e]` is an exact identity, and the evaluator's automatic
  `E^a·E^b → E^(a+b)` merge plus like-term collapse reduces a genuine identity
  to a literal `0` — a proof. It pays a single `TrigToExp` + evaluate (no
  `Together`, no common-denominator build, no grid), and is complexity-gated
  (≤ 512) so a large non-collapsing input bails fast.
- Fixes the whole `x^k E^(ax) Sin/Cos[bx]` diff-back family and, as a bonus,
  angle-addition identities in two variables (e.g.
  `Sin[x] Cos[y] + Cos[x] Sin[y] - Sin[x+y] → 0`). Soundness verified: genuine
  non-identities (`a Sin[x] + b Cos[x]`, `Sin[x] Sin[y] + Cos[x] Cos[y]`) are
  preserved, and `E^(I x) - Cos[x] - I Sin[x] → 0` stays correct. Zero added
  leaks (byte-identical valgrind totals vs. the non-fallback path). The
  previously pre-existing `test_multikernel_case` abort is now resolved.

## TrigToExp now recurses into nested trig arguments (2026-07-16)

`TrigToExp` previously used a single top-down `ReplaceAll`, which rewrites the
outermost trig heads but never re-descends into the RHS it substitutes. So a
trig-valued *argument* — e.g. the inner `Cos[x]-Sin[x]` in `Sin[Cos[x]-Sin[x]]`
— was left un-exponentialized, diverging from Wolfram Language (whose `TrigToExp`
is fully recursive). Switched to `ReplaceRepeated` (every rule's RHS is built
purely from `Exp`/`Log`/`Sqrt`, so the fixed-point iteration is guaranteed to
terminate). `TrigToExp[Sin[Cos[x]]]` now exponentializes the inner `Cos[x]` too.

- **Fixes the Risch transcendental trig frontend.** With the argument fully
  exponentialized, `Integrate[(Cos[x]+Sin[x]) Sin[Cos[x]-Sin[x]], x,
  Method -> "RischTranscendental"]` now closes to `Cos[Cos[x]-Sin[x]]` (was
  returned unevaluated — the frontend never saw a genuine exponential tower over
  `x`). Regression tests added in `test_integrate_risch_transcendental.c`
  (`test_trig_frontend`).
- No regressions across the trig, simplify, Risch, log/exp, and solve suites
  (failure sets identical before and after).

## FresnelC and FresnelS — the Fresnel integrals (2026-07-15)

New `FresnelC[z] = ∫₀ᶻ Cos[π t²/2] dt` and `FresnelS[z] = ∫₀ᶻ Sin[π t²/2] dt`
in `src/special_functions/fresnel.c` (one file serving both builtins, `bessel.c`
style). Both entire and odd, `Listable | NumericFunction | Protected`.

- **Numerics.** A shared kernel computes the pair (C, S) together: real inputs
  sum two convergent Maclaurin series (asymptotic DLMF 7.12 form for large `|x|`);
  complex inputs sum `A = C+iS` and its `i→−i` partner `B` via the shared `ncpx`
  MPFR-complex toolkit, then recover `C=(A+B)/2`, `S=(A−B)/(2i)`. Guard bits
  `≈(π/2)|z|²/ln2` absorb the partial-sum cancellation; odd symmetry folds
  negative / left-half-plane inputs. Machine and arbitrary (MPFR) precision, real
  and complex, tracking input precision.
- **Exact values.** `Fresnel*[0]=0`, `FresnelC[±∞]=±1/2`, `FresnelS[±∞]=±1/2`,
  `FresnelC[±I∞]=±I/2`, `FresnelS[±I∞]=∓I/2`, `ComplexInfinity/Indeterminate →
  Indeterminate`, and the symbolic odd rewrite `Fresnel*[-x]=-Fresnel*[x]`.
- **Calculus.** Derivatives `D[FresnelC[z],z]=Cos[π z²/2]`,
  `D[FresnelS[z],z]=Sin[π z²/2]` (`src/calculus/deriv.c`). Full Series parity in
  `src/calculus/series.c`: origin Taylor via the `kernel_coefs` table, the
  trig-prefactored asymptotic expansion at Infinity, and the symbolic-index
  `SeriesCoefficient` closed form as a `Piecewise` of Gamma quotients.
- **Wiring.** Symbols in `sym_names.{c,h}`, init in `core.c`, docstrings in
  `info.c`. Tests: `tests/test_fresnelc.c`, `tests/test_fresnels.c` (special
  values, machine/MPFR real & complex, odd symmetry, derivatives, all Series
  forms, Listable, attributes, arity) — references cross-checked against mpmath.

## Transcendental Risch — constant-base exponentials a^x (a ≠ e) (2026-07-15)

`Integrate[3^x/(E^(3^x)+1), x, Method -> "RischTranscendental"]` and
`Integrate[2^x/(4^x-2^x-1), x, …]` returned unintegrated even though both are
elementary. Root cause: `a^u` (base `a ≠ e`) is stored as `Power[a, u]` — the
evaluator collapses `E^(c Log a)` back to `a^c` (`power.c` `make_power`), so none
of the exponential recognizers (all keyed to `Exp` / `Power[E, ·]`) fired.

Fix (`integrate_risch_transcendental.c`): a **debasing** pre/post pass around the
recursive integrator. Every x-dependent constant-base power `a^u` is rewritten to
`E^(u · Σⱼ eⱼ Lⱼ)` where `a = ∏ⱼ pⱼ^eⱼ` (prime factorisation) and each `Lⱼ` is a
fresh OPAQUE symbol standing for `Log[pⱼ]`; because the exponent carries `Lⱼ`
(not the literal `Log[pⱼ]`), `make_power` no longer collapses it and the base-e
machinery sees a genuine `E^(…)` tower with a constant log-derivative. Sharing
`L` by prime makes commensurate bases commensurate (`4^x = E^(2 x L₂)`,
`2^x = E^(x L₂)` → one kernel). After integration `Lⱼ → Log[pⱼ]` is substituted
back and a `Log` recombination (`n Log a + m Log b → Log[a^n b^m]`) restores the
original base (`6^x/(6^x+1) → Log[1+6^x]/Log[6]`). The rewrite is an exact
identity, so correctness is preserved.

- `3^x/(E^(3^x)+1) → (3^x − Log[1+E^(3^x)])/Log[3]`; `2^x → 2^x/Log[2]`;
  `1/(2^x+1) → −2 ArcTanh[1+2·2^x]/Log[2]`; `6^x/(6^x+1) → Log[1+6^x]/Log[6]`.
- **Inexact bases stay conservative.** The general `Integrate` rationalises a
  float base upstream (`2.71828 → 67957/25000`); debasing that would fabricate a
  spuriously exact `Log`-based form that numericalises to a degraded exponent
  (`2.71828^(0.999999 x)`). A new `rt_transcendental_set_debase` toggle, disabled
  by the Integrate inexact path, keeps `Integrate[x·2.71828^(-x), x]` unevaluated
  (inexact-in / inexact-out).

## Risch — RT_TAN nonlinear-lower-monomial residue (2026-07-14)

Closes `∫(1+Tan²)/(Tan·Log[Tan]) → Log[Log[Tan[x]]]`, the repeated-pole
`Log[Tan]²` form (`→ −1/Log[Tan[x]]`), and the hyperbolic σ=−1

## IntegrateHypertangent — disc>0 real-algebraic residue hang fixed (2026-07-14)

## Hermite reduction over C(x)[t] — infinite-loop fix on irreducible-quadratic poles (2026-07-14)

- New `rf_normfrac` = `Cancel[Together[·]]`; the division / reduction loops now

## Risch `CoupledDECancelTan` — the tangent cancellation case (Bronstein §8.4) (2026-07-13)

Added the direct real recursion for the **tangent cancellation case** of the
coupled differential system — the algorithm whose whole point (Bronstein §8.4,
p.265) is to keep the hypertangent monomial `t` *real*, going complex only one
level down inside the base field `k`, rather than adjoining `√−1` to `k(t)` (the
"trivial route", which would rewrite any further tangent in `k` as a complex
hyperexponential and leave the answer in a field merely isomorphic to `k`).

New internal builtin in `src/calculus/risch_coupled.c`:

- `Risch\`CoupledDECancelTan[b0, b2, c1, c2, t, deriv, n]` — for a hypertangent
  monomial `t` (`Dt/(t²+1) = η ∈ k`), `b0, b2 ∈ k` and `c1, c2 ∈ k[t]`, solves
  the tangent-cancellation coupled system (symmetric diagonal `b0 − nηt`, forced
  by the complex-scalar structure of eq. 8.14)

  ```
  D[q1] + (b0 − nηt) q1 − b2 q2 == c1
  D[q2] + b2 q1 + (b0 − nηt) q2 == c2
  ```

  for `q1, q2 ∈ k[t]` of degree ≤ `n`. Follows the p.265 box verbatim: base case
  `n = 0` → `CoupledDESystem(b0, b2, c1, c2)`; otherwise evaluate `c1, c2` at
  `t = √−1` and Gaussian-split `z1 + z2√−1`, solve the lower coupled system
  `(s1, s2) ← CoupledDESystem(b0, b2 − nη, z1, z2)`, form and exactly divide
  `c = ((c1 − z1 + nη(s1 t + s2)) + (c2 − z2 + nη(s2 t − s1))√−1)/(t − √−1)`,
  split `c = d1 + d2√−1`, recurse `CoupledDECancelTan(b0, b2 + η, d1, d2, n−1)`,
  and reconstruct `(q1, q2) = (h1 t + h2 + s1, h2 t − h1 + s2)`. Returns
  `{q1, q2}` or stays unevaluated ("no solution"). Attributes
  `PROTECTED | READPROTECTED`, docstring. Reproduces **Bronstein Example 8.4.1**
  exactly — `CoupledDECancelTan[0, 4x, −t²+2t−8x²+1, 2(1−2x), tan(x), 2]` returns
  `{t − 1, 2x}`, with every intermediate `(z, s, c, d, h)` matching the book.

  Scope: base field `k = C(x)` (a single tangent tower over the rationals),
  where the base `CoupledDESystem` route over `C(i)(x)` is exact; deeper `k`
  carrying further tangent monomials needs the full mutually-recursive
  `CoupledDESystem` dispatcher (follow-up).

Shared refactor: extracted `rc_split_gaussian` (algebraic real/imaginary split by
conjugation, used for both `k`-constants and `k[t]`-polynomials) and `rc_base_var`
(the derivation's `Dx = 1` variable); `risch_coupled_desystem` and the
hypertangent-reduced code now reuse them.

Tests added to `tests/test_risch_coupled.c` (`test_cancel_tan_book`,
`test_cancel_tan_constructed`, `test_cancel_tan_stress`,
`test_cancel_tan_no_solution`, `test_cancel_tan_robustness`): Example 8.4.1 and its
recursion by exact value; constructed-solution diff-back against the symmetric

## Risch `IntegrateHypertangent` driver + `ResidueReduce` (Bronstein §5.10/§5.6) (2026-07-13)

Phase 1 of retiring the `TrigToExp` route for real tangent: the complete
**`IntegrateHypertangent`** driver (Bronstein §5.10, p.172 box) that integrates
`f ∈ k(t)` over a hypertangent monomial `t = tan(∫η)` *directly and real*, plus
the general residue criterion **`ResidueReduce`** (§5.6) it needs. Two new
internal builtins in `src/calculus/risch_hypertangent.c`:

- `Risch\`ResidueReduce[h, t, deriv]` — the Rothstein–Trager / LazardRiobooTrager
  residue criterion (§5.6) over `k(t)`: returns `{g2, β}` with `g2 = Σ cⱼ Log(gⱼ)`

  `Integrate\`TranscendentalLogPart` engine the tower integrator uses, with the
  derivation's lower variables `{x, t₀, …, t_{L−1}}` as the constant-residue gate.
- `Risch\`IntegrateHypertangent[f, t, deriv]` — the full §5.10 driver, returning

  box does, `HermiteReduce` (§5.3) → `ResidueReduce` (§5.6) →
  `IntegrateHypertangentReduced` (§5.10, the `t²+1` pole peel via `CoupledDESystem`)
  → `IntegrateHypertangentPolynomial` (§5.10, `PolynomialReduce` + the `c Log(t²+1)`
  term), reconciling the parts by `p = h − D[g2] + r` and `p − D[q1]`. Both
  `PROTECTED | READPROTECTED`, docstrings.

Reproduces every worked example in §5.10: **5.10.1** `∫(tan²+x tan+1)` → `{tan(x),
False}` (the `x tan` remainder is non-elementary); **5.10.2** `∫ sin(x)/x` →

residue logs (e.g. `∫ D[Log(tan(x)−2)]` → `Log(tan(x)−2)`) and detects the

## `Sum[]` cascade — all pre-existing memory leaks fixed (2026-07-13)

Every infinite `Sum` that flowed through the sub-algorithm cascade leaked a
handful of small `Expr` nodes (14–54 blocks per input, per `SUM_LEAKS.md`),
even when it produced the correct closed form. Root-caused and eliminated; all
five recorded reproducers and every stage now report **0 leaks / 0 bytes**
under macOS `leaks --atExit`.

**Two distinct ownership bugs, both in the stages that run and bail before the
closing stage:**

1. **`evaluate(expr_copy(f))` in 7 stages** — `evaluate()` *borrows* its
   argument (it internally does `current = expr_copy(e)` and never frees `e`),
   so the anonymous `expr_copy(f)` reference was never released, leaking the
   canonicalised summand tree. Fixed to the correct borrow-then-free idiom
   (`fc = expr_copy(f); fn = evaluate(fc); expr_free(fc);`) in
   `sum_alternating.c`, `sum_trigonometric.c`, `sum_dirichlet.c`, `sum_euler.c`,
   `sum_euler_nonlinear.c`, `sum_geometric.c`, and `sum_zetaseries.c`. `Sum\`Gosper`
   was already clean because it consistently frees the evaluate input.

2. **Redundant `expr_copy(base)` in `Sum\`Rational`** — `sr_term_contribution`
   uses its `base` argument only through copies (`sr_pdeg`/`sum_subst`), so it
   never needed to own it, yet the sole call site passed `expr_copy(base)` and
   the callee never freed it. Dropped the copy so `base` is borrowed from the
   `Apart` result — matching the already-clean sibling `alt_term_contribution`.

No behavioural change: `sum_tests` (64) and `nsum_tests` still pass, and the
closed forms are unchanged (`Sum[1/k^2]→Pi^2/6`, `Sum[1/(k(k+1))]→1`,
`Sum[(-1)^(k+1)/k]→Log[2]`, `Sum[1/(k^2+1)]→(Pi Coth[Pi]-1)/2`, …).

## Evaluator/symtab performance — Phase 2: unified interner + symbol table

The string interner (8191-bucket table of `InternEntry`) and the symbol table
(65535-bucket table of `SymEntry`→`SymbolDef`) are merged into a single 65535-bucket table
of self-chaining `SymbolDef` nodes. A symbol's name is interned and its definition looked up

The graphics engine links raylib, which on Linux is commonly a static archive

## SinIntegral and Sinc — sine integral and cardinal sine (2026-07-14)

New entire special functions:

- **`SinIntegral[z]`** = Si(z) = ∫₀ᶻ Sin[t]/t dt (`src/special_functions/sinintegral.c`).
  Odd, entire, no branch cuts. Exact values `0 → 0`, `±Infinity → ±Pi/2`,
  `±I Infinity → ±I Infinity`, `ComplexInfinity → Indeterminate`; symbolic odd
  fold `SinIntegral[-x] → -SinIntegral[x]`. Numeric via a convergent Maclaurin
  series (guard bits absorb the ~e^|z| cancellation) with an asymptotic
  `Pi/2 - cos f - sin g` fallback for large |z|; machine + arbitrary-precision
  reals and complexes (complex via the shared `ncpx` toolkit). Matches WL to 50
  digits: `N[SinIntegral[2], 50] = 1.605412976802694848576720148198588940848583422328…`.
- **`Sinc[z]`** = Sin[z]/z with `Sinc[0] = 1` (`src/special_functions/sinc.c`).
  Even, entire. Exact `0 → 1`, `±Infinity → 0`, `ComplexInfinity → Indeterminate`;
  numeric reals/complexes at machine or MPFR precision. Introduced so
  `D[SinIntegral[x], x]` returns `Sinc[x]` as in WL.
- **Derivatives** (`src/calculus/deriv.c`): `D[SinIntegral[z], z] = Sinc[z]`,
  `D[Sinc[z], z] = Cos[z]/z - Sin[z]/z^2`.
- **Series** (`src/calculus/series.c`): Taylor at 0 for `SinIntegral` via the
  odd-kernel-at-zero path (also composes, e.g. `SinIntegral[x^2]`); `Sinc`
  series at 0 via a `Sinc → Sin[x]/x` rewrite; and a trig-prefactored asymptotic
  expansion at Infinity for `SinIntegral`
  (`Pi/2 - Sin[x]/x^2 + Cos[x] (-1/x + 2/x^3)`). The symbolic
  `DirectedInfinity[z]`-direction constant form is deferred.
- Attributes for both: `Listable`, `NumericFunction`, `Protected`; docstrings in
  `info.c`; symbols in `sym_names.c`. Unit tests `tests/test_sinintegral.c`,
  `tests/test_sinc.c` (valgrind-clean).

## Together — combine algebraic-constant fractions with non-unit coefficients (2026-07-15)

Fixed a `Together`/`Cancel` deficiency where a sum of fractions over a
constant-radical extension (e.g. `Sqrt[5]`) was left completely uncombined
whenever a polynomial variable carried a coefficient other than 1:

```
Together[1/(2 x - Sqrt[5]) - 1/(2 x + Sqrt[5])]   (* was: unchanged *)
                                              -> (2 Sqrt[5])/(4 x^2 - 5)
```

Root cause was in `exact_poly_div` (`src/poly/poly.c`): the coefficient-level
base case (`var_count == 0`) only returned an exact quotient when **both**
operands were rational/Gaussian-rational, so the leading-coefficient recursion
of the LCM-denominator division `(4x^2 - 5) / (2x ± Sqrt[5])` refused to divide
the algebraic coefficient `-2 Sqrt[5]` by the rational `2` and returned NULL —
`together_recursive`'s strict-quotient combine then read that as "not exactly
divisible" and left the whole sum uncombined. The coefficient-1 case slipped
through only because the divisor was monic.

The fix keys the decision on the **divisor**: division by a nonzero rational (or
Gaussian-rational) scalar is exact in any Q(i)-algebra — such a `B` is a field
unit, so `A/B = A·B⁻¹` lands back in the working ring for every `A`, including
algebraic atoms like `Sqrt[2]` or a whole `Q[Sqrt2, Sqrt3, …]` coefficient. A
**non-field** divisor (`Sqrt[2]`, a symbolic radical) still returns NULL, so the
case-13 multivariate-GCD termination guard is unaffected. This also lets the
originally-reported mixed exponential/radical input combine (`Sqrt[5]` cancels,
leaving a clean fraction over `Q` in the `2^x` generator):

```
Together[4 Sqrt[5] 2^x (1/(-10 - 10 Sqrt[5] + 20 2^x)
                        - 1/(-10 + 10 Sqrt[5] + 20 2^x))]
    -> 2^x/(2^(2x) - 2^x - 1)          (* equals WL's factored form *)
```

Regression tests in `tests/test_rat.c` (`test_together_algebraic_generator`).

## RischTranscendental: trig/inverse-trig of a logarithm (transcendental power kernels)

`Integrate[f, x, Method -> "RischTranscendental"]` now integrates elementary
integrands whose exponential kernels are hidden as *general powers* `b^e`
(base `b ≠ e`, transcendental exponent). These arise whenever a circular or
hyperbolic function of a logarithm is exponentialized: `TrigToExp` turns
`Cos[x Log x]` into `(x^(I x) + x^(-I x))/2`, and the evaluator canonicalizes
`E^(c Log b) → b^c` (power.c `simplify_exp_log`), so the kernels
`E^(±I x Log x)` are *stored* as `x^(±I x)` — invisible to every exponential
recognizer, all keyed to `Exp` / `Power[E, ·]`. The integrand stranded
unintegrated.

```
Integrate[3 x^2 Cos[x Log[x]] - x^3 Sin[x Log[x]] - x^3 Log[x] Sin[x Log[x]], x,
          Method -> "RischTranscendental"]
    -> x^3 Cos[x Log[x]]                  (* was: unevaluated *)

Integrate[Cos[Log[x]], x, Method -> "RischTranscendental"]
    -> (x/2)(Cos[Log[x]] + Sin[Log[x]])   (* was: unevaluated *)

Integrate[Sin[Log[x]], x, Method -> "RischTranscendental"]  -> (x/2)(Sin[Log x] - Cos[Log x])
Integrate[x^I, x, Method -> "RischTranscendental"]           -> (1/2 - I/2) x^(1+I)
Integrate[x^x (1 + Log[x]), x, Method -> "RischTranscendental"] -> x^x
```

A new structural (non-evaluating) pre-pass `rt_powers_to_exp`
(`src/calculus/integrate_risch_transcendental.c`) re-spells each transcendental
`b^e` as a raw `Power[E, e Log b]` before the tower machinery runs. Because the
rewrite never passes through the evaluator, the `E^(c Log b) → b^c` collapse
never fires; the existing collectors/substitution (`rt_collect_exp_exponents`,
`rt_subst_kernels`, which already match `Power[E, ·]` structurally) then see a
genuine hyperexponential monomial, with `Log b` a real logarithmic sub-kernel
(`D[Log x] = 1/x`) — unlike the constant-base debasing, whose opaque `L_p`
stands for a *constant* `Log p`. Back-substituting the tower kernels lets the
evaluator re-collapse `E^(e Log b) → b^e`, rendering the antiderivative in the
original power form (then `ExpToTrig`/realify recover the real trig closed form).

Critically, the exponent is **split** on rewrite: the evaluator merges a
polynomial coefficient into the exponent (`x^2 x^x → x^(2+x)`,
`x^2 x^(I x) → x^(2+I x)`), which would read as a kernel independent of
`x^(3+x)` / `x^(3+I x)` (exponent ratio non-constant) and defeat the
commensurability reduction. `rt_powers_to_exp` peels the rational-constant
additive terms back into an algebraic power `b^c` (base field), keeping only the
transcendental part as the shared primitive kernel `x^x` / `x^(I x)`.

Genuinely non-elementary Log-in-argument integrands still decline via the field
decision (`Cos[x Log x]`, `Sin[Log x] Log[Log x]`, `x^x`). Every branch remains
behind the exact tower diff-back gate, so a mis-reduction can only decline,
never emit a wrong closed form. Regression coverage in
`tests/test_integrate_risch_transcendental.c` (`test_circular_trig_integration`).

## ElementaryIntegralQ — decides the trig class + algebraic-scope soundness guard (2026-07-16)

`Risch\`ElementaryIntegralQ[f, x]` is now a decision procedure for the
**trigonometric / inverse-trigonometric** class, and a latent unsoundness on
algebraic-of-x integrands is closed.

- **Trig `False` side.** The field decision (`rt_decide_field`) previously built
  its tower from the raw `f` — it never ran the integrator's trig front-end — so
  a bare `Sin`/`Cos`/`Tan` kernel produced no Risch tower and the verdict fell to
  `undec`. It now mirrors `rt_trig_frontend`: `TrigToExp` exponentializes the trig
  kernels (and `rt_powers_to_exp` re-exposes any collapsed `E^(c Log b)` general
  powers), so the decision routes through the *same* Gaussian exponential tower
  the True side uses and reads the residue / Risch-DE certificate off it. The
  classic trig special-function integrals now decide **`False`**:

  ```
  Risch`ElementaryIntegralQ[Sin[x]/x, x]      -> False   (* SinIntegral;  was undec *)
  Risch`ElementaryIntegralQ[Sin[x]/x^2, x]    -> False   (* was undec *)
  Risch`ElementaryIntegralQ[Sin[x^2], x]      -> False   (* FresnelS;     was undec *)
  Risch`ElementaryIntegralQ[Cos[E^x], x]      -> False   (* CosIntegral;  was undec *)
  Risch`ElementaryIntegralQ[Cos[x Log[x]], x] -> False   (* was undec *)
  ```

  Sound because `rt_decide` checks the True side first, so an exhibited elementary
  form always dominates; the field path runs only when the integrator found
  nothing, and its `False` fires solely behind an exact certificate.

- **Algebraic-of-x scope guard (soundness).** Exponentializing a trig/exp of a
  *radical* (`Cos[√x] → E^(±I√x)`) would let the tower builder spuriously accept
  the algebraic `√x` in an exponent and emit a false `False` — even though
  `∫Cos(√x)` is elementary via the algebraic substitution `u=√x` the
  transcendental algorithm does not perform. `rt_has_algebraic_of_x` now scans the
  original integrand for a non-integer-rational power of an x-dependent base (or a
  `Surd`/`Root` of x) and declines `rt_decide_field` to `UNKNOWN`:

  ```
  Risch`ElementaryIntegralQ[Cos[Sqrt[x]], x]   -> undec  (* elementary via u=Sqrt[x] *)
  Risch`ElementaryIntegralQ[Sin[Sqrt[x]], x]   -> undec
  Risch`ElementaryIntegralQ[Sin[x^(1/3)], x]   -> undec
  Risch`ElementaryIntegralQ[E^Sqrt[x], x]      -> undec  (* fixes a PRE-EXISTING false False *)
  ```

  Mixed algebraic-transcendental Risch (Bronstein's algebraic case) is not
  implemented, so this class is correctly out of scope rather than misdecided.
  `x^x = E^(x Log x)` and `x^Sqrt[2] = E^(Sqrt[2] Log x)` are **not** flagged
  (their exponents are transcendental, not `Rational`-headed).

- Regression coverage in `tests/test_risch_elementaryq.c` (`test_false_trig_frontend`,
  plus trig-True and algebraic-`undec` cases). No regressions across the Risch,
  transcendental-Risch, RischMacsyma, dispatch, DerivativeDivides, Goursat,
  Jeffrey, and unknown integrate suites. Valgrind totals identical to the
  startup baseline (no new leaks).

## Fix: `Integrate` `Method -> "RischTranscendental"` hang on trig-polynomial integrands

`Integrate[x^7 Cos[x] Sin[x]^5, x, Method -> "RischTranscendental"]` (and the
smaller `x Cos[x] Sin[x]^5`) never returned.

- **Root cause.** The trig front-end integrates the exponentialized form, verifies
  it, and reconstructs a real closed form in `rt_realify` (`risch_trig_frontend.c`).
  For a trig-POLYNOMIAL integrand `∫ x^n · trig-poly` the reconstructed real part is
  already the clean multiple-angle antiderivative
  (e.g. `(1/1152)(-90 x Cos[2x] + 36 x Cos[4x] - 6 x Cos[6x] + 45 Sin[2x] - 9 Sin[4x] + Sin[6x])`),
  but `rt_realify` handed it to full `Simplify` purely to beautify it. `Simplify`'s
  transform search `Factor`s the `TrigExpand` image, and the multivariate factorizer
  is pathologically slow (~6 s per term) over the OVERLAPPING generators
  `{x, Cos[x], Sin[x]}` — `Cos[x]`/`Sin[x]` themselves contain `x`. With seven
  exponential kernels the total blew up unboundedly.

- **Fix.** `rt_realify` now takes a fast path for a trig-polynomial real form
  (free of `Log`/`Arc*` heads and of denominators/radicals): normalize with the
  cheap, linear `TrigReduce` (product-to-sum) instead of `Simplify`, and return it
  once the existing exact diff-back gate (`Together[TrigToExp[D − f]]`, which reduces
  to 0 immediately for a denominator-free trig polynomial) certifies it. Rational-trig
  answers (`Sec[x]`, `1/(2+Cos[x])`, …) carry `Log`/`ArcTan`, are excluded from the
  fast path, and go through `Simplify` unchanged. Correctness is unaffected — the same
  diff-back gate certifies the result; only the beautification path changed.

- `x^7 Cos[x] Sin[x]^5` now returns a correct multiple-angle antiderivative
  (diff-back `= 0`); `x Cos[x] Sin[x]^5` and `x^2 Cos[x] Sin[x]^3` return in ~0.15 s.
  Regression cases added to `tests/test_integrate_risch_transcendental.c`
  (`test_reported_bug_fixes`). No regressions across the transcendental-Risch,
  risch_field, inverse-trig-Simplify, and trigrat suites.
