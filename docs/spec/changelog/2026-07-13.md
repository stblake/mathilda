## IntegrateHypertangent вҖ” disc>0 real-algebraic residue hang fixed (2026-07-14)

`Risch`IntegrateHypertangent` (and the shared hyperbolic driver `rh_integrate_core`
in `risch_hypertangent.c`) hung on a normal pole whose irreducible quadratic
factor has **positive discriminant** вҖ” real *irrational* residues, e.g.
`1/(tВІ+5t+3)` (disc 13). The disc<0 case (`tВІ+t+1`, complex residues вҶ’ `ArcTan`)
and the split-over-Q case always worked; only real-algebraic residues hung.

Every individual stage (`HermiteReduce`, `ResidueReduce`,
`IntegrateHypertangentReduced`) was fast. The hang was in the final
`IntegrateHypertangentPolynomial` stage, fed a reconciliation remainder
`p = h вҲ’ D_tower[gвӮӮ] + r` that was **not actually a polynomial**: for disc>0,
`ResidueReduce` emits the residues as conjugate radical pairs `a Вұ bвҲҡ13`, whose
contributions to `p` cancel to a polynomial over Q вҖ” but `Together` treats each
`вҲҡ13` as opaque, so the spurious `t`-poles survived and the polynomial driver's
field long-division spun on a phantom pole (the same non-termination pattern as
the C(x)[t] Hermite fix, now over Q(вҲҡ13)(x)).

Fix (`rh_integrate_core`): new `rh_polynorm` = `Cancel[Together[RootReduce[В·]]]`,
applied at both reconciliation points (`p` and `p2`). `RootReduce` reduces the
algebraic coefficients to canonical form *first* so the conjugate residues
combine over Q; `Cancel[Together]` then clears the genuinely-cancelling
denominators, yielding the true polynomial. On the common rational-residue case
(split-linear denominators, e.g. Weierstrass `Sec^n`) there are no radicals and
`RootReduce` is a fast no-op.

`1/(tВІ+5t+3)` now returns `{g, True}` in 3.3 s (was a hard hang), with
`f вҲ’ D_tower[g]` a `t`-free constant (`2/29`) вҖ” the expected ОІ=True base-field
remainder. The repeated-pole variant `1/(tВІ+5t+3)ВІ` closes in 0.03 s. All risch
suites (`risch_hermite`, `risch_hypertangent`, `risch_coupled`,
RischTranscendental 11.4 s) and `integrals_tests` pass; leak-clean (macOS
baseline only).

## Hermite reduction over C(x)[t] вҖ” infinite-loop fix on irreducible-quadratic poles (2026-07-14)

`Risch`HermiteReduce` (and the shared field arithmetic in `risch_field.c`) hung
indefinitely on normal poles whose squarefree factor is an **irreducible
quadratic with coefficients in `C(x)`** вҖ” e.g. `1/(tВІвҲ’x)Ві`, `1/(tВІвҲ’x)ВІ` under a
monomial derivation like `D[t] = ВҪ(tВІ+1)`. Split / rational-root poles
(`(1вҲ’tВІ)Ві`) and irreducible quadratics with *numeric* coefficients
(`(tВІ+5t+3)Ві`) were always fine; only the genuine `C(x)`-coefficient case hung.

Root cause was **not** coefficient swell but a true **non-termination** in
`risch_field_divmod_t` (and the identical loop in
`risch_field_polynomial_reduce`). Each long-division step subtracted the leading
term with a plain `Expand`. Over `k = C(x)` the coefficients are rational
functions of `x`, and `Expand` leaves a mathematically-zero leading coefficient
in a syntactically-nonzero shape (mismatched denominators), so `degree_t`
(via `CoefficientList`) never saw the degree drop and the loop spun forever. In
isolation every field op (`Diophantine`, `PolyDivide`, `ExtendedEuclidean`) was
fast; only the inter-step remainder `a = вҲ’jc вҲ’ uВ·D[b]`, with its messy `C(x)`
coefficients, tripped the loop.

Fix (both in `src/calculus/risch_field.c`):

- New `rf_normfrac` = `Cancel[Together[В·]]`; the division / reduction loops now
  reduce the running remainder to **canonical form each step** instead of just
  expanding, so a true leading zero over `C(x)` collapses and the `t`-degree
  strictly decreases. Termination is now provable вҖ” no arbitrary iteration cap
  (consistent with the "no caps in decision procedures" rule).
- `risch_field_xgcd_t` now delegates to the `PolynomialExtendedGCD[a,b,t]`
  builtin, whose 3-arg form has a FLINT-backed swell-free path (`gr_poly` over
  the rational-function ring `fmpz_mpoly_q`) for `k = C(x)`; the hand-rolled
  Euclidean loop is kept only as a no-FLINT fallback (`rf_xgcd_classical`).

`1/(tВІвҲ’x)Ві` now reduces in 0.15 s (was a hard hang) with
`f вҲ’ (D[g]+h+r) = 0` exact in the tower. `risch_hermite`, `risch_hypertangent`,
`risch_coupled`, the RischTranscendental suite (11.5 s) and `integrals_tests`
all pass; leak-clean (macOS baseline only).

## Literal Hermite reduction over the tower (Bronstein В§5.3) (2026-07-13)

Implemented Bronstein's **quadratic `HermiteReduce`** (Theorem 5.3.1, p.139)
verbatim over the monomial-tower derivation, in `src/calculus/risch_hermite.{c,h}`,
exposed as `Risch\`HermiteReduce[f, t, deriv] -> {g, h, r}` with the exact invariant

    f == D[g] + h + r,   h simple (squarefree normal denominator),  r reduced.

The algorithm: `CanonicalRepresentation` splits `f = f_p + f_s + f_n`; the normal
part `f_n = a/d` is reduced by, for each squarefree factor `v = d_i` of multiplicity
`i вүҘ 2` (Yun factorization in `t`, reusing the now-exposed `risch_squarefree_t`) and
each `j` from `iвҲ’1` down to `1`, solving the Diophantine equation
`b (u D[v]) + c v = вҲ’a/j` (`u = d/v^i`) and updating `g += b/v^j`, `a вҶҗ вҲ’j c вҲ’ u D[b]`
вҖ” where `D[v]`, `D[b]` are the **monomial derivation** (not `d/dt`). The remaining
`a/(uv)` is split by `PolyDivide` into the simple part `h` and a polynomial folded
into the reduced `r = q + f_p + f_s`.

**ansatz** (audit finding A2-1), whose numerator coefficients were restricted to
polynomials of a heuristic `x`-degree; the literal algorithm handles arbitrary
rational `k = C(x, вҖҰ)` coefficients. Reproduces Bronstein **Example 5.3.1** exactly

## Evaluator/symtab performance вҖ” Phase 0 baseline harness

`EVAL_SYMTAB_IMPROVEMENTS.md` (new, repo root) is a full implementation spec for the
next-generation evaluator + symbol-table rework: a **symbol-object model** that unifies the
string interner and symbol table, precomputes base attributes into the `SymbolDef`, and
caches a `SymbolDef*` on `EXPR_SYMBOL` so per-node dispatch becomes a pointer dereference
(no re-hashing, no ~140-entry `strcmp` scan in `get_attributes()`).

Phase 0 (guardrails) landed:

- **`tests/bench_eval.c`** (+ ctest target `bench_eval`) вҖ” evaluator baseline benchmark
  modeled on `bench_assoc`: median-of-5 wall time, machine-normalized against
  `Total[Range[40000]]`, gates at 2.5Г— a committed baseline. Six workloads targeting the
  dispatch path (pure distinct-node dispatch, DownValue rewrite, Listable threading,
  Orderless sort, ReplaceRepeated fold, nested Plus collapse) rather than heavy builtin
  bodies. Baselines pinned (Apple M-series, USE_ECM=OFF); the pure-dispatch row
  (`Table[hh[k,k],{k,8000}]`, ~2.7 Вөs/node) is the primary signal for the upcoming Phase 1

  normalized costs down.

## Evaluator/symtab performance вҖ” Phase 1: base-attribute fold

`get_attributes()` was on the evaluator hot path (called per function node per pass) yet did
a linear `strcmp` scan of the ~140-entry `builtin_attrs[]` table every call, then merged the
dynamic `def->attributes`. Phase 1 folds each symbol's immutable static "base" attributes
into the `SymbolDef` once (new `base_attributes` + `base_seeded` fields; seeded lazily on
first `get_attributes`, eagerly in `attr_init` for table symbols), so every call is now an
O(1) field read + OR: `base_attributes | def->attributes`.

- Bit-for-bit identical to the old `scan | dynamic`, including the un-clearable base floor
  (builtins remain effectively Protected against `Unprotect`, the documented behavior).
  `def->attributes` keeps its exact meaning, so match.c's direct attribute reads and the 13+
  Block-style save/restore sites (findmin, nsum/nlimit/nresidue/nseries/nderiv/nint, modular,
  core Protect/Unprotect/ClearAll) need no change.
- Measured on `tests/bench_eval` (Apple M-series): pure-dispatch workload
  (`Length[Table[hh[k,k],{k,8000}]]`) **вҲ’33%** (norm 0.60 вҶ’ 0.40); DownValue rewrite вҲ’16%,
  Listable threading вҲ’13%; matcher/compare-bound rows unchanged, as predicted. Suite green
  (clearall_remove_protect, core, eval, match, symtab, eval_timestamps, regression, list).
- Files: `src/symtab.h`, `src/symtab.c`, `src/attr.c`. Struct layout changed вҶ’ full rebuild
  (`make clean`) required.

## Evaluator/symtab performance вҖ” Phase 2: unified interner + symbol table

The string interner (8191-bucket table of `InternEntry`) and the symbol table
(65535-bucket table of `SymEntry`вҶ’`SymbolDef`) are merged into a single 65535-bucket table
of self-chaining `SymbolDef` nodes. A symbol's name is interned and its definition looked up
through **one** hash of **one** table instead of hashing the name twice across two.

- `SymbolDef` now owns its canonical name string and carries `is_system` (migrated from
  `InternEntry`), a `has_def` materialization flag, and a `next` bucket-chain pointer;
  `SymEntry` is deleted. The `sym_intern.h` API (`intern_symbol`, `intern_is_system`,
  `intern_mark_all_system`, `intern_clear`) is now implemented in `symtab.c` over the unified
  table; `sym_intern.c` is an empty stub.
- `has_def` preserves observable semantics bit-for-bit: `intern_symbol` creates an
  interned-only node (`has_def=0`), `symtab_get_def` materializes (`has_def=1`); `Names[]` /
  `symtab_for_each` enumerate and `symtab_lookup` reports only `has_def=1` nodes вҖ” exactly the
  pre-merge "has a SymbolDef" set. `symtab_clear` / `symtab_remove_symbol` reset a node to
  interned-only (payload freed, `has_def=0`) but keep the node + name string so `SYM_*` and
  live `Expr` pointers stay valid; `symtab_init` resets payloads instead of `memset`;
  `intern_clear` is the sole full teardown (frees nodes + names).
- Verified: clean build; green on names, symtab (3Г— re-init), context, core, eval, match,
  clearall_remove_protect, eval_timestamps, regression, rule_dispatch (exercises
  `intern_clear`). Valgrind leak total byte-identical to a bare `Sin[1.0]` session (no
  Mathilda-source frames in any leak stack). `bench_eval` gate green.
- Files: `src/symtab.h`, `src/symtab.c`, `src/sym_intern.c`. Struct layout changed вҶ’ full
  rebuild (`make clean`) required.

## Evaluator/symtab performance вҖ” Phase 3a: resolve the head's def once

`evaluate_step` now resolves the head symbol's `SymbolDef*` a single time per pass and
threads it through attribute lookup, DownValue dispatch, and builtin dispatch, rather than
re-hashing the head up to three times. New def-threaded fast paths `get_attributes_def()`
(attr.c) and `apply_down_values_def()` (symtab.c) do the work; the name-taking
`get_attributes` / `apply_down_values` are now thin resolve-then-delegate wrappers. The
Phase 2 def node is stable, so the cached pointer stays valid across the step and each use
reads its fields fresh. No struct change; green on the eval/match/core/symtab/attribute/
names/regression/rule_dispatch suites; no new allocations.

Phase 3b (caching the `SymbolDef*` on the `EXPR_SYMBOL` node) was **measured out**: after
Phase 1 removed the attribute strcmp scan, the pure-dispatch benchmark is flat across Phases
1/2/3a (the residual per-node symtab lookups are <2% of per-node cost), so the node cache вҖ”
which would require renaming ~2446 `data.symbol` sites across 277 files вҖ” buys ~nothing and
is left as documented future work. Files: `src/eval.c`, `src/attr.c/h`, `src/symtab.c/h`.

## Evaluator/symtab performance вҖ” Phase 3b: cache SymbolDef* on EXPR_SYMBOL

The `EXPR_SYMBOL` union member is now `struct { char* name; struct SymbolDef* def; }`. The
evaluator resolves the head's definition once and caches it on the symbol node, so repeated
passes over the same node reach attributes/DownValues/builtin by a pointer load instead of a
symbol-table lookup. `name` stays at union offset 0, so the historical EXPR_STRING type-pun
holds and `sizeof(Expr)` is unchanged.

- Mechanical rename of 2864 `data.symbol` вҶ’ `data.symbol.name` sites across 356 files; the 3
  in-place name writers now maintain the cache (`expr_new_symbol` NULLs it for lazy resolve,
  `expr_unshare` copies it, `alpha_rename` NULLs it since the name changed).
- Safe because Phase 2 made SymbolDef nodes stable (reset in place, never reallocated): a
  cached def always points to the current node for that name, and its fields are read fresh.
  Valgrind: 0 invalid reads/writes, leak total byte-identical to baseline.
- Performance: neutral. Pure-dispatch is flat (~14,200 Вөs) across Phases 1/2/3a/3b вҖ” after
  Phase 1 removed the attribute strcmp scan, the eliminated per-node lookup is <2% of node
  cost. Implemented for architectural completeness; the measurable win came from Phase 1.
- Files: `src/expr.h`, `src/expr.c`, `src/symtab.c`, `src/eval.c`, plus the mechanical rename
  across `src/` and `tests/`. Struct definition changed вҶ’ `make clean` required.

## Build: raylib static X11/GL transitive deps auto-linked (issue #18)

The graphics engine links raylib, which on Linux is commonly a static archive
(`libraylib.a`) whose transitive windowing deps вҖ” `-lX11 -lGL -lpthread -ldl
-lrt ...` вҖ” live in the `.pc` file's `Libs.private:` line. Both build systems
were pulling only the plain `Libs:` set, so test executables (and the makefile
build without `EXTRA_LIBS`) failed to link with undefined `XInternAtom`,
`XSync`, `XFree`, etc. Fixed to link the pkg-config *static* (private-dep-
inclusive) set:

- `tests/CMakeLists.txt`: link `${RAYLIB_STATIC_LIBRARIES}` / `_STATIC_LIBRARY_DIRS`
  (falling back to the plain set if empty) instead of `${RAYLIB_LIBRARIES}`.
- `makefile`: `pkg-config --static --libs raylib` instead of `--libs`.
- No-op on macOS, where `Libs.private` is empty (frameworks are recorded in the
  dylib) so the static set equals the plain set; verified the link line and a
  test build are byte-identical there. Removes the need for the manual
  `make EXTRA_LIBS="-lX11"` workaround on Linux.

## SinIntegral and Sinc вҖ” sine integral and cardinal sine (2026-07-14)

New entire special functions:

- **`SinIntegral[z]`** = Si(z) = вҲ«вӮҖб¶» Sin[t]/t dt (`src/special_functions/sinintegral.c`).
  Odd, entire, no branch cuts. Exact values `0 вҶ’ 0`, `ВұInfinity вҶ’ ВұPi/2`,
  `ВұI Infinity вҶ’ ВұI Infinity`, `ComplexInfinity вҶ’ Indeterminate`; symbolic odd
  fold `SinIntegral[-x] вҶ’ -SinIntegral[x]`. Numeric via a convergent Maclaurin
  series (guard bits absorb the ~e^|z| cancellation) with an asymptotic
  `Pi/2 - cos f - sin g` fallback for large |z|; machine + arbitrary-precision
  reals and complexes (complex via the shared `ncpx` toolkit). Matches WL to 50
  digits: `N[SinIntegral[2], 50] = 1.605412976802694848576720148198588940848583422328вҖҰ`.
- **`Sinc[z]`** = Sin[z]/z with `Sinc[0] = 1` (`src/special_functions/sinc.c`).
  Even, entire. Exact `0 вҶ’ 1`, `ВұInfinity вҶ’ 0`, `ComplexInfinity вҶ’ Indeterminate`;
  numeric reals/complexes at machine or MPFR precision. Introduced so
  `D[SinIntegral[x], x]` returns `Sinc[x]` as in WL.
- **Derivatives** (`src/calculus/deriv.c`): `D[SinIntegral[z], z] = Sinc[z]`,
  `D[Sinc[z], z] = Cos[z]/z - Sin[z]/z^2`.
- **Series** (`src/calculus/series.c`): Taylor at 0 for `SinIntegral` via the
  odd-kernel-at-zero path (also composes, e.g. `SinIntegral[x^2]`); `Sinc`
  series at 0 via a `Sinc вҶ’ Sin[x]/x` rewrite; and a trig-prefactored asymptotic
  expansion at Infinity for `SinIntegral`
  (`Pi/2 - Sin[x]/x^2 + Cos[x] (-1/x + 2/x^3)`). The symbolic
  `DirectedInfinity[z]`-direction constant form is deferred.
- Attributes for both: `Listable`, `NumericFunction`, `Protected`; docstrings in
  `info.c`; symbols in `sym_names.c`. Unit tests `tests/test_sinintegral.c`,
  `tests/test_sinc.c` (valgrind-clean).
