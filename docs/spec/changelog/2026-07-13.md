## Exact trig/exp zero-test ŌĆö hyperbolic kernels + algebraic-extension points (2026-07-16)

Extended the exact grid zero-test (`simp_trigexp_zero.c`, shared by `Simplify`'s
`transform_trigexp_vanish`, the Risch verifier `rt_verify_antideriv`, and
`PossibleZeroQ`) from circular-only to hyperbolic and real-exponential kernels,
fixing the last flaky Risch trig-frontend decline, `1/(2 + Cosh[x])`.

- **Hyperbolic kernels.** `classify_kernel`/`kernel_value`/`poly_deg` now also
  recognise `Sinh`/`Cosh`/`Tanh`/`Coth`/`Sech`/`Csch` and the real exponential
  `E^(k x)`, with the I-free kernel `t = E^(x)` (the circular set keeps
  `t = E^(I x)`). `decide_with_var` picks the mode from which family is present;
  an expression mixing both (`Tanh[x] - Tan[x]`) needs two kernels and declines
  to the sampler. This closes `1/(3 + 2 Sinh[x])`, `Sinh`/`Cosh`/`Tanh`/ŌĆ” diff-backs,
  and hyperbolic identities under `PossibleZeroQ` (`Cosh[2x] - 2 Cosh[x]^2 + 1`,
  `Sinh[2x] - 2 Sinh[x] Cosh[x]`).
- **Algebraic-extension point values.** A radical-coefficient antiderivative
  (the `Q(Sqrt[3])` partial-fraction constants in the `1/(2 + Cosh[x])` diff-back)
  evaluates each grid point to an exact constant in `Q(alpha)` that the cheap
  classifier leaves undecided. The point evaluator now retries once through
  `Together` (exact rationalisation over the extension) when a point is
  undecided, so a genuine zero certifies and a genuine nonzero stays nonzero.
- **Soundness guard.** A commensurate *non-integer* kernel (a half-angle
  `Tanh[x/2]` alongside `Cosh[x]`, or `E^(x/2)`) is algebraically dependent on
  the integer-multiple kernels but not representable on the integer-power `t`
  axis; `collect` now declines on it instead of mis-treating it as an
  independent generator (which produced a spurious `False` for
  `1/(5 + 3 Cosh[x])`). Genuinely independent nested transcendentals
  (`Cosh[x^2]`, `Tanh[Log[x]]`) stay opaque as before. Verified: identities ŌåÆ
  `True`, non-identities ŌåÆ `False`, `zero_test`/`trigexp_zero` suites green, no
  added leaks.

## Risch trig frontend ŌĆö Csc[x]^3 no longer flakily declines (exact zero-test fix) (2026-07-16)

`Integrate`RischTranscendental[Csc[x]^3, x]` returned unevaluated after ~21 s
(flaky ŌĆö it occasionally succeeded on warm state) even though it produces a
correct antiderivative
(`-ArcTanh[E^(I x)] + E^(I x)/(-1 + E^((2 I) x)) + 2 E^(I x)/(1 + E^((4 I) x) - 2 E^((2 I) x))`).
The integrand's diff-back verification (`rt_verify_antideriv`) could not prove
`D[G] - Csc[x]^3 == 0`: the fast Laurent `Together` leaves `E^(┬▒k I x)` poles
uncombined, the exact grid zero-test declined, and the residual `Simplify` ran
16 s and gave up.

- **Root cause** in the exact grid zero-test (`simp_trigexp_zero.c`, `poly_deg`):
  an I-laden diff-back carries explicit Gaussian-rational coefficients
  `Complex[a, b]`, which are `EXPR_FUNCTION` nodes with head `Complex`. `poly_deg`
  hit its unbounded-head fallback on them and returned "no degree bound," so the
  whole grid test declined ŌĆö even though a `Complex[...]` literal is a constant
  (degree 0 in every axis), exactly as `collect()` already treated it. Fixed by
  giving `poly_deg` the same rule: any subterm free of the kernel variable is a
  constant coefficient regardless of head (`Complex[a, b]`, `Sqrt[2]`, `Pi`, ŌĆ”).
- **Effect:** the grid test now proves the Csc[x]^3 diff-back zero in ~9 ms
  (was: decline ŌåÆ 16 s Simplify ŌåÆ give up). `Integrate`RischTranscendental[Csc[x]^3, x]`
  succeeds in ~25 ms and is **deterministic** (5/5 ŌåÆ exact diff-back 0), closing
  the flaky `test_circular_trig_integration`. Rigorous both ways ŌĆö a genuine
  non-identity with `Complex[...]` coefficients still certifies FALSE, no leaks.
  (Pre-existing and unrelated: even powers `Csc[x]^4`/`Sec[x]^4` remain slow via
  the realify path ŌĆö confirmed slow with the fix reverted; not in the test suite.)

## Simplify ŌĆö TrigToExp-collapse fallback for I-laden diff-backs (2026-07-16)

`Simplify` now recognises a further family of trig/exp expressions as
identically zero: those mixing a **real** exponential kernel `E^x` with the
**imaginary** kernel `E^(I x)` *and* carrying a bare polynomial coefficient in
the kernel variable ŌĆö canonically the Risch antiderivative diff-back
`D[Ōł½ x E^x Sin[x] dx] - x E^x Sin[x]`. Previously `Simplify` stalled on such
inputs at a non-zero intermediate like
`x (-E^x Sin[x] + (-I/2) E^((1+I) x) + (I/2) E^((1-I) x))` (non-idempotent ŌĆö
a second `Simplify` pass reached 0), and the assert-active
`integrate_risch_transcendental_tests` aborted at `test_multikernel_case`.

- The rigorous exact grid zero-test (`trigexp_rational_is_zero`) declines by
  design on bare polynomial dependence on the kernel variable and on mixed
  real+imaginary exponential kernels it treats as independent opaque
  generators. `transform_trigexp_vanish` now adds a cheap, **sound** fallback
  invoked *only when that test declines* (never when it proves non-zero):
  `TrigToExp[e]` is an exact identity, and the evaluator's automatic
  `E^a┬ĘE^b ŌåÆ E^(a+b)` merge plus like-term collapse reduces a genuine identity
  to a literal `0` ŌĆö a proof. It pays a single `TrigToExp` + evaluate (no
  `Together`, no common-denominator build, no grid), and is complexity-gated
  (Ōēż 512) so a large non-collapsing input bails fast.
- Fixes the whole `x^k E^(ax) Sin/Cos[bx]` diff-back family and, as a bonus,
  angle-addition identities in two variables (e.g.
  `Sin[x] Cos[y] + Cos[x] Sin[y] - Sin[x+y] ŌåÆ 0`). Soundness verified: genuine
  non-identities (`a Sin[x] + b Cos[x]`, `Sin[x] Sin[y] + Cos[x] Cos[y]`) are
  preserved, and `E^(I x) - Cos[x] - I Sin[x] ŌåÆ 0` stays correct. Zero added
  leaks (byte-identical valgrind totals vs. the non-fallback path). The
  previously pre-existing `test_multikernel_case` abort is now resolved.

## TrigToExp now recurses into nested trig arguments (2026-07-16)

`TrigToExp` previously used a single top-down `ReplaceAll`, which rewrites the
outermost trig heads but never re-descends into the RHS it substitutes. So a
trig-valued *argument* ŌĆö e.g. the inner `Cos[x]-Sin[x]` in `Sin[Cos[x]-Sin[x]]`
ŌĆö was left un-exponentialized, diverging from Wolfram Language (whose `TrigToExp`
is fully recursive). Switched to `ReplaceRepeated` (every rule's RHS is built
purely from `Exp`/`Log`/`Sqrt`, so the fixed-point iteration is guaranteed to
terminate). `TrigToExp[Sin[Cos[x]]]` now exponentializes the inner `Cos[x]` too.

- **Fixes the Risch transcendental trig frontend.** With the argument fully
  exponentialized, `Integrate[(Cos[x]+Sin[x]) Sin[Cos[x]-Sin[x]], x,
  Method -> "RischTranscendental"]` now closes to `Cos[Cos[x]-Sin[x]]` (was
  returned unevaluated ŌĆö the frontend never saw a genuine exponential tower over
  `x`). Regression tests added in `test_integrate_risch_transcendental.c`
  (`test_trig_frontend`).
- No regressions across the trig, simplify, Risch, log/exp, and solve suites
  (failure sets identical before and after).

## Transcendental Risch ŌĆö constant-base exponentials a^x (a ŌēĀ e) (2026-07-15)

`Integrate[3^x/(E^(3^x)+1), x, Method -> "RischTranscendental"]` and
`Integrate[2^x/(4^x-2^x-1), x, ŌĆ”]` returned unintegrated even though both are
elementary. Root cause: `a^u` (base `a ŌēĀ e`) is stored as `Power[a, u]` ŌĆö the
evaluator collapses `E^(c Log a)` back to `a^c` (`power.c` `make_power`), so none
of the exponential recognizers (all keyed to `Exp` / `Power[E, ┬Ę]`) fired.

Fix (`integrate_risch_transcendental.c`): a **debasing** pre/post pass around the
recursive integrator. Every x-dependent constant-base power `a^u` is rewritten to
`E^(u ┬Ę ╬ŻŌ▒╝ eŌ▒╝ LŌ▒╝)` where `a = ŌłÅŌ▒╝ pŌ▒╝^eŌ▒╝` (prime factorisation) and each `LŌ▒╝` is a
fresh OPAQUE symbol standing for `Log[pŌ▒╝]`; because the exponent carries `LŌ▒╝`
(not the literal `Log[pŌ▒╝]`), `make_power` no longer collapses it and the base-e
machinery sees a genuine `E^(ŌĆ”)` tower with a constant log-derivative. Sharing
`L` by prime makes commensurate bases commensurate (`4^x = E^(2 x LŌéé)`,
`2^x = E^(x LŌéé)` ŌåÆ one kernel). After integration `LŌ▒╝ ŌåÆ Log[pŌ▒╝]` is substituted
back and a `Log` recombination (`n Log a + m Log b ŌåÆ Log[a^n b^m]`) restores the
original base (`6^x/(6^x+1) ŌåÆ Log[1+6^x]/Log[6]`). The rewrite is an exact
identity, so correctness is preserved.

- `3^x/(E^(3^x)+1) ŌåÆ (3^x ŌłÆ Log[1+E^(3^x)])/Log[3]`; `2^x ŌåÆ 2^x/Log[2]`;
  `1/(2^x+1) ŌåÆ ŌłÆ2 ArcTanh[1+2┬Ę2^x]/Log[2]`; `6^x/(6^x+1) ŌåÆ Log[1+6^x]/Log[6]`.
- **Inexact bases stay conservative.** The general `Integrate` rationalises a
  float base upstream (`2.71828 ŌåÆ 67957/25000`); debasing that would fabricate a
  spuriously exact `Log`-based form that numericalises to a degraded exponent
  (`2.71828^(0.999999 x)`). A new `rt_transcendental_set_debase` toggle, disabled
  by the Integrate inexact path, keeps `Integrate[x┬Ę2.71828^(-x), x]` unevaluated
  (inexact-in / inexact-out).

Regressions: `test_constant_base_exponentials` (numeric diff-back for the reported
cases, bare/rational/polynomial-times `a^x`, composite bases, and the two inexact
`Head == Integrate` guards); `intrischnorman`, `risch_hypertangent`,
`risch_elementaryq`, `integrate_dispatch` pass; no new failures in the
transcendental suite. Leak-clean (macOS valgrind baseline only).

## `Sum[]` cascade ŌĆö all pre-existing memory leaks fixed (2026-07-13)

Every infinite `Sum` that flowed through the sub-algorithm cascade leaked a
handful of small `Expr` nodes (14ŌĆō54 blocks per input, per `SUM_LEAKS.md`),
even when it produced the correct closed form. Root-caused and eliminated; all
five recorded reproducers and every stage now report **0 leaks / 0 bytes**
under macOS `leaks --atExit`.

**Two distinct ownership bugs, both in the stages that run and bail before the
closing stage:**

1. **`evaluate(expr_copy(f))` in 7 stages** ŌĆö `evaluate()` *borrows* its
   argument (it internally does `current = expr_copy(e)` and never frees `e`),
   so the anonymous `expr_copy(f)` reference was never released, leaking the
   canonicalised summand tree. Fixed to the correct borrow-then-free idiom
   (`fc = expr_copy(f); fn = evaluate(fc); expr_free(fc);`) in
   `sum_alternating.c`, `sum_trigonometric.c`, `sum_dirichlet.c`, `sum_euler.c`,
   `sum_euler_nonlinear.c`, `sum_geometric.c`, and `sum_zetaseries.c`. `Sum\`Gosper`
   was already clean because it consistently frees the evaluate input.

2. **Redundant `expr_copy(base)` in `Sum\`Rational`** ŌĆö `sr_term_contribution`
   uses its `base` argument only through copies (`sr_pdeg`/`sum_subst`), so it
   never needed to own it, yet the sole call site passed `expr_copy(base)` and
   the callee never freed it. Dropped the copy so `base` is borrowed from the
   `Apart` result ŌĆö matching the already-clean sibling `alt_term_contribution`.

No behavioural change: `sum_tests` (64) and `nsum_tests` still pass, and the

## Evaluator/symtab performance ŌĆö Phase 2: unified interner + symbol table

The string interner (8191-bucket table of `InternEntry`) and the symbol table

## Together ŌĆö combine algebraic-constant fractions with non-unit coefficients (2026-07-15)

Fixed a `Together`/`Cancel` deficiency where a sum of fractions over a
constant-radical extension (e.g. `Sqrt[5]`) was left completely uncombined
whenever a polynomial variable carried a coefficient other than 1:

```
Together[1/(2 x - Sqrt[5]) - 1/(2 x + Sqrt[5])]   (* was: unchanged *)
                                              -> (2 Sqrt[5])/(4 x^2 - 5)
```

Root cause was in `exact_poly_div` (`src/poly/poly.c`): the coefficient-level
base case (`var_count == 0`) only returned an exact quotient when **both**
operands were rational/Gaussian-rational, so the leading-coefficient recursion
of the LCM-denominator division `(4x^2 - 5) / (2x ┬▒ Sqrt[5])` refused to divide
the algebraic coefficient `-2 Sqrt[5]` by the rational `2` and returned NULL ŌĆö
`together_recursive`'s strict-quotient combine then read that as "not exactly
divisible" and left the whole sum uncombined. The coefficient-1 case slipped
through only because the divisor was monic.

The fix keys the decision on the **divisor**: division by a nonzero rational (or
Gaussian-rational) scalar is exact in any Q(i)-algebra ŌĆö such a `B` is a field
unit, so `A/B = A┬ĘBŌü╗┬╣` lands back in the working ring for every `A`, including
algebraic atoms like `Sqrt[2]` or a whole `Q[Sqrt2, Sqrt3, ŌĆ”]` coefficient. A
**non-field** divisor (`Sqrt[2]`, a symbolic radical) still returns NULL, so the
case-13 multivariate-GCD termination guard is unaffected. This also lets the
originally-reported mixed exponential/radical input combine (`Sqrt[5]` cancels,
leaving a clean fraction over `Q` in the `2^x` generator):

```
Together[4 Sqrt[5] 2^x (1/(-10 - 10 Sqrt[5] + 20 2^x)
                        - 1/(-10 + 10 Sqrt[5] + 20 2^x))]
    -> 2^x/(2^(2x) - 2^x - 1)          (* equals WL's factored form *)
```

Regression tests in `tests/test_rat.c` (`test_together_algebraic_generator`).

## RischTranscendental: trig/inverse-trig of a logarithm (transcendental power kernels)

`Integrate[f, x, Method -> "RischTranscendental"]` now integrates elementary
integrands whose exponential kernels are hidden as *general powers* `b^e`
(base `b ŌēĀ e`, transcendental exponent). These arise whenever a circular or
hyperbolic function of a logarithm is exponentialized: `TrigToExp` turns
`Cos[x Log x]` into `(x^(I x) + x^(-I x))/2`, and the evaluator canonicalizes
`E^(c Log b) ŌåÆ b^c` (power.c `simplify_exp_log`), so the kernels
`E^(┬▒I x Log x)` are *stored* as `x^(┬▒I x)` ŌĆö invisible to every exponential
recognizer, all keyed to `Exp` / `Power[E, ┬Ę]`. The integrand stranded
unintegrated.

```
Integrate[3 x^2 Cos[x Log[x]] - x^3 Sin[x Log[x]] - x^3 Log[x] Sin[x Log[x]], x,
          Method -> "RischTranscendental"]
    -> x^3 Cos[x Log[x]]                  (* was: unevaluated *)

Integrate[Cos[Log[x]], x, Method -> "RischTranscendental"]
    -> (x/2)(Cos[Log[x]] + Sin[Log[x]])   (* was: unevaluated *)

Integrate[Sin[Log[x]], x, Method -> "RischTranscendental"]  -> (x/2)(Sin[Log x] - Cos[Log x])
Integrate[x^I, x, Method -> "RischTranscendental"]           -> (1/2 - I/2) x^(1+I)
Integrate[x^x (1 + Log[x]), x, Method -> "RischTranscendental"] -> x^x
```

A new structural (non-evaluating) pre-pass `rt_powers_to_exp`
(`src/calculus/integrate_risch_transcendental.c`) re-spells each transcendental
`b^e` as a raw `Power[E, e Log b]` before the tower machinery runs. Because the
rewrite never passes through the evaluator, the `E^(c Log b) ŌåÆ b^c` collapse
never fires; the existing collectors/substitution (`rt_collect_exp_exponents`,
`rt_subst_kernels`, which already match `Power[E, ┬Ę]` structurally) then see a
genuine hyperexponential monomial, with `Log b` a real logarithmic sub-kernel
(`D[Log x] = 1/x`) ŌĆö unlike the constant-base debasing, whose opaque `L_p`
stands for a *constant* `Log p`. Back-substituting the tower kernels lets the
evaluator re-collapse `E^(e Log b) ŌåÆ b^e`, rendering the antiderivative in the
original power form (then `ExpToTrig`/realify recover the real trig closed form).

Critically, the exponent is **split** on rewrite: the evaluator merges a
polynomial coefficient into the exponent (`x^2 x^x ŌåÆ x^(2+x)`,
`x^2 x^(I x) ŌåÆ x^(2+I x)`), which would read as a kernel independent of
`x^(3+x)` / `x^(3+I x)` (exponent ratio non-constant) and defeat the
commensurability reduction. `rt_powers_to_exp` peels the rational-constant
additive terms back into an algebraic power `b^c` (base field), keeping only the
transcendental part as the shared primitive kernel `x^x` / `x^(I x)`.

Genuinely non-elementary Log-in-argument integrands still decline via the field
decision (`Cos[x Log x]`, `Sin[Log x] Log[Log x]`, `x^x`). Every branch remains
behind the exact tower diff-back gate, so a mis-reduction can only decline,
never emit a wrong closed form. Regression coverage in
`tests/test_integrate_risch_transcendental.c` (`test_circular_trig_integration`).

## ElementaryIntegralQ ŌĆö decides the trig class + algebraic-scope soundness guard (2026-07-16)

`Risch\`ElementaryIntegralQ[f, x]` is now a decision procedure for the
**trigonometric / inverse-trigonometric** class, and a latent unsoundness on
algebraic-of-x integrands is closed.

- **Trig `False` side.** The field decision (`rt_decide_field`) previously built
  its tower from the raw `f` ŌĆö it never ran the integrator's trig front-end ŌĆö so
  a bare `Sin`/`Cos`/`Tan` kernel produced no Risch tower and the verdict fell to
  `undec`. It now mirrors `rt_trig_frontend`: `TrigToExp` exponentializes the trig
  kernels (and `rt_powers_to_exp` re-exposes any collapsed `E^(c Log b)` general
  powers), so the decision routes through the *same* Gaussian exponential tower
  the True side uses and reads the residue / Risch-DE certificate off it. The
  classic trig special-function integrals now decide **`False`**:

  ```
  Risch`ElementaryIntegralQ[Sin[x]/x, x]      -> False   (* SinIntegral;  was undec *)
  Risch`ElementaryIntegralQ[Sin[x]/x^2, x]    -> False   (* was undec *)
  Risch`ElementaryIntegralQ[Sin[x^2], x]      -> False   (* FresnelS;     was undec *)
  Risch`ElementaryIntegralQ[Cos[E^x], x]      -> False   (* CosIntegral;  was undec *)
  Risch`ElementaryIntegralQ[Cos[x Log[x]], x] -> False   (* was undec *)
  ```

  Sound because `rt_decide` checks the True side first, so an exhibited elementary
  form always dominates; the field path runs only when the integrator found
  nothing, and its `False` fires solely behind an exact certificate.

- **Algebraic-of-x scope guard (soundness).** Exponentializing a trig/exp of a
  *radical* (`Cos[ŌłÜx] ŌåÆ E^(┬▒IŌłÜx)`) would let the tower builder spuriously accept
  the algebraic `ŌłÜx` in an exponent and emit a false `False` ŌĆö even though
  `Ōł½Cos(ŌłÜx)` is elementary via the algebraic substitution `u=ŌłÜx` the
  transcendental algorithm does not perform. `rt_has_algebraic_of_x` now scans the
  original integrand for a non-integer-rational power of an x-dependent base (or a
  `Surd`/`Root` of x) and declines `rt_decide_field` to `UNKNOWN`:

  ```
  Risch`ElementaryIntegralQ[Cos[Sqrt[x]], x]   -> undec  (* elementary via u=Sqrt[x] *)
  Risch`ElementaryIntegralQ[Sin[Sqrt[x]], x]   -> undec
  Risch`ElementaryIntegralQ[Sin[x^(1/3)], x]   -> undec
  Risch`ElementaryIntegralQ[E^Sqrt[x], x]      -> undec  (* fixes a PRE-EXISTING false False *)
  ```

  Mixed algebraic-transcendental Risch (Bronstein's algebraic case) is not
  implemented, so this class is correctly out of scope rather than misdecided.
  `x^x = E^(x Log x)` and `x^Sqrt[2] = E^(Sqrt[2] Log x)` are **not** flagged
  (their exponents are transcendental, not `Rational`-headed).

- Regression coverage in `tests/test_risch_elementaryq.c` (`test_false_trig_frontend`,
  plus trig-True and algebraic-`undec` cases). No regressions across the Risch,
  transcendental-Risch, RischMacsyma, dispatch, DerivativeDivides, Goursat,
  Jeffrey, and unknown integrate suites. Valgrind totals identical to the
  startup baseline (no new leaks).
