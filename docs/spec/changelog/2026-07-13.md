## IntegrateHypertangent — disc>0 real-algebraic residue hang fixed (2026-07-14)

`Risch`IntegrateHypertangent` (and the shared hyperbolic driver `rh_integrate_core`
in `risch_hypertangent.c`) hung on a normal pole whose irreducible quadratic
factor has **positive discriminant** — real *irrational* residues, e.g.
`1/(t²+5t+3)` (disc 13). The disc<0 case (`t²+t+1`, complex residues → `ArcTan`)
and the split-over-Q case always worked; only real-algebraic residues hung.

Every individual stage (`HermiteReduce`, `ResidueReduce`,
`IntegrateHypertangentReduced`) was fast. The hang was in the final
`IntegrateHypertangentPolynomial` stage, fed a reconciliation remainder
`p = h − D_tower[g₂] + r` that was **not actually a polynomial**: for disc>0,
`ResidueReduce` emits the residues as conjugate radical pairs `a ± b√13`, whose
contributions to `p` cancel to a polynomial over Q — but `Together` treats each
`√13` as opaque, so the spurious `t`-poles survived and the polynomial driver's
field long-division spun on a phantom pole (the same non-termination pattern as
the C(x)[t] Hermite fix, now over Q(√13)(x)).

Fix (`rh_integrate_core`): new `rh_polynorm` = `Cancel[Together[RootReduce[·]]]`,
applied at both reconciliation points (`p` and `p2`). `RootReduce` reduces the
algebraic coefficients to canonical form *first* so the conjugate residues
combine over Q; `Cancel[Together]` then clears the genuinely-cancelling
denominators, yielding the true polynomial. On the common rational-residue case
(split-linear denominators, e.g. Weierstrass `Sec^n`) there are no radicals and
`RootReduce` is a fast no-op.

`1/(t²+5t+3)` now returns `{g, True}` in 3.3 s (was a hard hang), with
`f − D_tower[g]` a `t`-free constant (`2/29`) — the expected β=True base-field
remainder. The repeated-pole variant `1/(t²+5t+3)²` closes in 0.03 s. All risch
suites (`risch_hermite`, `risch_hypertangent`, `risch_coupled`,
RischTranscendental 11.4 s) and `integrals_tests` pass; leak-clean (macOS
baseline only).

## Hermite reduction over C(x)[t] — infinite-loop fix on irreducible-quadratic poles (2026-07-14)

`Risch`HermiteReduce` (and the shared field arithmetic in `risch_field.c`) hung
indefinitely on normal poles whose squarefree factor is an **irreducible
quadratic with coefficients in `C(x)`** — e.g. `1/(t²−x)³`, `1/(t²−x)²` under a
monomial derivation like `D[t] = ½(t²+1)`. Split / rational-root poles
(`(1−t²)³`) and irreducible quadratics with *numeric* coefficients
(`(t²+5t+3)³`) were always fine; only the genuine `C(x)`-coefficient case hung.

Root cause was **not** coefficient swell but a true **non-termination** in
`risch_field_divmod_t` (and the identical loop in
`risch_field_polynomial_reduce`). Each long-division step subtracted the leading
term with a plain `Expand`. Over `k = C(x)` the coefficients are rational
functions of `x`, and `Expand` leaves a mathematically-zero leading coefficient
in a syntactically-nonzero shape (mismatched denominators), so `degree_t`
(via `CoefficientList`) never saw the degree drop and the loop spun forever. In
isolation every field op (`Diophantine`, `PolyDivide`, `ExtendedEuclidean`) was
fast; only the inter-step remainder `a = −jc − u·D[b]`, with its messy `C(x)`
coefficients, tripped the loop.

Fix (both in `src/calculus/risch_field.c`):

- New `rf_normfrac` = `Cancel[Together[·]]`; the division / reduction loops now
  reduce the running remainder to **canonical form each step** instead of just
  expanding, so a true leading zero over `C(x)` collapses and the `t`-degree
  strictly decreases. Termination is now provable — no arbitrary iteration cap
  (consistent with the "no caps in decision procedures" rule).
- `risch_field_xgcd_t` now delegates to the `PolynomialExtendedGCD[a,b,t]`
  builtin, whose 3-arg form has a FLINT-backed swell-free path (`gr_poly` over
  the rational-function ring `fmpz_mpoly_q`) for `k = C(x)`; the hand-rolled
  Euclidean loop is kept only as a no-FLINT fallback (`rf_xgcd_classical`).

`1/(t²−x)³` now reduces in 0.15 s (was a hard hang) with
`f − (D[g]+h+r) = 0` exact in the tower. `risch_hermite`, `risch_hypertangent`,
`risch_coupled`, the RischTranscendental suite (11.5 s) and `integrals_tests`
all pass; leak-clean (macOS baseline only).

## Real hypertangent family — correct-by-construction, diff-back gates removed (2026-07-13)

Removed every diff-back / Simplify verification gate from the live real
hypertangent / hyperbolic-tangent integrator `rt_hypertan_family`
(`src/calculus/integrate_risch_transcendental.c`, driving `Tan`, `Cot`, `Tanh`,
`Coth`). The Bronstein §5.10 pipeline is **correct by construction**: the
structural gate already guarantees `F` is a genuine rational function of the
monomial `t` over `C(x)` with no trig/exp/log of `x` surviving, so
`F|_{t→kernel} = f` holds identically (for a direct kernel `t` *is* the kernel —
back-substitution is the identity). The §5.10 driver returns `g` with `β = True`
iff the transcendental part is elementary, leaving the `t`-free base-field
remainder `base = F − D_tower[g]` for a recursive integral in `x`. Since the
tower derivation `D[t]` is exactly the derivative of the kernel,
`d/dx[(g + ∫base)|_{t→kernel}] = f` is Bronstein Thm 5.10.1 — a theorem, not
something to re-verify.

The only runtime facts now consulted are **structural**: `β = True`
(elementary), `base` free of `t`, and `∫base dx` itself elementary (Integrate
does not leave it unevaluated). This eliminates a ~25 s `Simplify` diff-back that
did not even converge on back-substituted half-angle forms, restoring the
transcendental suite to its 14.3 s baseline while keeping `Tan`/`Cot`/`Tanh`/
`Coth` (and powers) exact — all external diff-backs `0`. The cosmetic
`Log[1 ± K²] → −2 Log[Cos/Sin/Cosh/Sinh]` rewrite is likewise an exact identity
(for `Coth`, `1 − Coth² < 0` contributes only a constant `iπ`), applied by
construction.

Diagnosis notes (informing the still-pending Weierstrass front-end for
`Sec^n`/`Csc^n`): Hermite reduction itself is *fast* on the split (rational-root)
poles that trig integration produces — the earlier "driver perf on split poles"
suspicion was wrong. The genuine slow case is an **irreducible-quadratic normal
pole** (`1/(t²+5t+3)³`, `1/(t²−x)³`), which hangs the `k(x)[t]` Diophantine
arithmetic; those carry algebraic residues and are correctly excluded by the
split-only normal-pole gate.

## Literal Hermite reduction over the tower (Bronstein §5.3) (2026-07-13)

Implemented Bronstein's **quadratic `HermiteReduce`** (Theorem 5.3.1, p.139)
verbatim over the monomial-tower derivation, in `src/calculus/risch_hermite.{c,h}`,
exposed as `Risch\`HermiteReduce[f, t, deriv] -> {g, h, r}` with the exact invariant

    f == D[g] + h + r,   h simple (squarefree normal denominator),  r reduced.

The algorithm: `CanonicalRepresentation` splits `f = f_p + f_s + f_n`; the normal
part `f_n = a/d` is reduced by, for each squarefree factor `v = d_i` of multiplicity
`i ≥ 2` (Yun factorization in `t`, reusing the now-exposed `risch_squarefree_t`) and
each `j` from `i−1` down to `1`, solving the Diophantine equation
`b (u D[v]) + c v = −a/j` (`u = d/v^i`) and updating `g += b/v^j`, `a ← −j c − u D[b]`
— where `D[v]`, `D[b]` are the **monomial derivation** (not `d/dt`). The remaining
`a/(uv)` is split by `PolyDivide` into the simple part `h` and a polynomial folded
into the reduced `r = q + f_p + f_s`.

This is the faithful replacement for the earlier undetermined-coefficient Hermite
**ansatz** (audit finding A2-1), whose numerator coefficients were restricted to
polynomials of a heuristic `x`-degree; the literal algorithm handles arbitrary
rational `k = C(x, …)` coefficients. Reproduces Bronstein **Example 5.3.1** exactly
(`∫ (x − tan x)/tan²x → −x/tan x − ∫ x`, i.e. `{−x/t, 0, −x}`).

Extensive suite `tests/test_risch_hermite.c`: the defining invariant `f = D[g]+h+r`
across log / exponential / hypertangent monomials (for `tan`, the special `t²+1`
poles correctly stay in `r`), rational-`C(x)`-coefficient cases (the ansatz gap),
higher pole orders and multiple normal factors, `h`-simple/`r`-reduced structural
checks, and robustness. All green; leak-clean (macOS baseline noise only).

**Wired into the live integrator — ansatz removed.** `rt_field_ratint` (the
proper-part integrator of the recursive transcendental Risch tower, log/primitive
case) now IS the literal pipeline: Hermite reduction (§5.3) peels the repeated
normal poles into `D_tower[g]`, and the simple remainder `h` is integrated by the
residue criterion / Lazard–Rioboo–Trager log part (§5.6). The former ~150-line
bounded undetermined-coefficient `SolveAlways` ansatz (`H(t)/Hden + Σ cⱼ Log gⱼ`,
polynomial `x`-coefficients) is **deleted** — this is the exact decision procedure,
not a search. A tower diff-back self-verifies the result. The full transcendental
integrator suite (diff-back verified) still passes with the ansatz gone, and cases
the ansatz could not reach now close, e.g.
`∫ D[x/((x+1) Log x)] dx = x/((x+1) Log x)` (a Hermite part with the rational
coefficient `x/(x+1)`) and `∫ (Log x − 1)/Log²x dx = x/Log x`.

**Exponential top wired to literal Hermite too — its ansatz removed.**
`rt_field_hyperexp_coupled` (the proper-part integrator for an exponential top
`t = E^w`) now delegates to a new `rt_field_hyperexp_hermite`, the literal Bronstein
hyperexponential pipeline (§5.3 + §5.6 + §5.9):

1. `HermiteReduce(f) → (g, h, r)` over `D_tower` — `g` is the exact rational part
   (repeated normal poles peeled, arbitrary rational lower-field coefficients),
   `h` the simple normal part, `r` the reduced Laurent polynomial in `t`
   (`CanonicalRepresentation` routes the `t^{-a}` special part into `r`).
2. residue criterion on `h` → `L = Σⱼ cⱼ Log(gⱼ)` (constant residues).
3. **coupling reconciliation** `P = h + r − D_tower[L]`. For an exponential monomial
   `D[Log g] = D[g]/g` is *not* proper (`deg_t D[g] = deg_t g`), so the logs of
   step 2 spill a `t`-polynomial into the Laurent part; subtracting `D_tower[L]`
   leaves `P` a genuine Laurent polynomial.
4. `IntegrateHyperexponentialPolynomial(P)` (`rt_int_hyperexp_poly`) → each `t^i`
   coefficient (`i≠0`) by a Risch DE `q_i' + i w' q_i = p_i`; the `t^0` coefficient
   recurses to the lower field.

Result `g + L + Q`, tower diff-back self-verified. The former unified `SolveAlways`
ansatz (bounded-degree polynomial Hermite/Laurent/log coefficients) is **deleted**.
Strictly more complete than the ansatz — it now closes repeated exponential poles
carrying rational lower-field coefficients, e.g. `∫ D[1/((1+x)(1+E^x)²)] dx`,
`∫ D[x²/(1+E^x)³] dx`, and `∫ x E^x/(1+E^x)² dx` all round-trip. Non-elementary
inputs still decline (or resolve to `Erf`/`ExpIntegralEi` via other paths). Full
transcendental integrator suite green with the ansatz gone; leak-clean.

## Risch `CoupledDECancelTan` — the tangent cancellation case (Bronstein §8.4) (2026-07-13)

Added the direct real recursion for the **tangent cancellation case** of the
coupled differential system — the algorithm whose whole point (Bronstein §8.4,
p.265) is to keep the hypertangent monomial `t` *real*, going complex only one
level down inside the base field `k`, rather than adjoining `√−1` to `k(t)` (the
"trivial route", which would rewrite any further tangent in `k` as a complex
hyperexponential and leave the answer in a field merely isomorphic to `k`).

New internal builtin in `src/calculus/risch_coupled.c`:

- `Risch\`CoupledDECancelTan[b0, b2, c1, c2, t, deriv, n]` — for a hypertangent
  monomial `t` (`Dt/(t²+1) = η ∈ k`), `b0, b2 ∈ k` and `c1, c2 ∈ k[t]`, solves
  the tangent-cancellation coupled system (symmetric diagonal `b0 − nηt`, forced
  by the complex-scalar structure of eq. 8.14)

  ```
  D[q1] + (b0 − nηt) q1 − b2 q2 == c1
  D[q2] + b2 q1 + (b0 − nηt) q2 == c2
  ```

  for `q1, q2 ∈ k[t]` of degree ≤ `n`. Follows the p.265 box verbatim: base case
  `n = 0` → `CoupledDESystem(b0, b2, c1, c2)`; otherwise evaluate `c1, c2` at
  `t = √−1` and Gaussian-split `z1 + z2√−1`, solve the lower coupled system
  `(s1, s2) ← CoupledDESystem(b0, b2 − nη, z1, z2)`, form and exactly divide
  `c = ((c1 − z1 + nη(s1 t + s2)) + (c2 − z2 + nη(s2 t − s1))√−1)/(t − √−1)`,
  split `c = d1 + d2√−1`, recurse `CoupledDECancelTan(b0, b2 + η, d1, d2, n−1)`,
  and reconstruct `(q1, q2) = (h1 t + h2 + s1, h2 t − h1 + s2)`. Returns
  `{q1, q2}` or stays unevaluated ("no solution"). Attributes
  `PROTECTED | READPROTECTED`, docstring. Reproduces **Bronstein Example 8.4.1**
  exactly — `CoupledDECancelTan[0, 4x, −t²+2t−8x²+1, 2(1−2x), tan(x), 2]` returns
  `{t − 1, 2x}`, with every intermediate `(z, s, c, d, h)` matching the book.

  Scope: base field `k = C(x)` (a single tangent tower over the rationals),
  where the base `CoupledDESystem` route over `C(i)(x)` is exact; deeper `k`
  carrying further tangent monomials needs the full mutually-recursive
  `CoupledDESystem` dispatcher (follow-up).

Shared refactor: extracted `rc_split_gaussian` (algebraic real/imaginary split by
conjugation, used for both `k`-constants and `k[t]`-polynomials) and `rc_base_var`
(the derivation's `Dx = 1` variable); `risch_coupled_desystem` and the
hypertangent-reduced code now reuse them.

Tests added to `tests/test_risch_coupled.c` (`test_cancel_tan_book`,
`test_cancel_tan_constructed`, `test_cancel_tan_stress`,
`test_cancel_tan_no_solution`, `test_cancel_tan_robustness`): Example 8.4.1 and its

dispatched in `rt_transcendental_case` immediately **before** `rt_trig_frontend`:
finds a single `Tan[u]` kernel with `u` rational in `x` (so `u' ∈ k = C(x)`),
substitutes `t = Tan[u]` to get `F` rational in `t` over `C(x)`, builds the tower
derivation `Dt = u'(t²+1)`, runs `Risch\`IntegrateHypertangent`, integrates the
leftover base-field element in `C(x)`, back-substitutes `t → Tan[u]`, collapses
`Log[1+Tan[u]²] → −2 Log[Cos[u]]`, and **diff-back verifies** (`Simplify[D[g]−f]≡0`)
before returning. Correct real closed forms now come out of the Risch stage:

| integrand | before (`TrigToExp`) | now (real) |
|---|---|---|
| `Tan[x]` | `I x − Log[1+Cos[2x]+I Sin[2x]]` | `−Log[Cos[x]]` |
| `Tan[x]²` | `(−2I−x−x Cos[2x]+…)/(1+Cos[2x]−I Sin[2x])` | `Tan[x] − x` |
| `Tan[x]⁴` | I-laden | `x + Tan[x]³/3 − Tan[x]` |
| `Tan[2x]` | I-laden | `−Log[Cos[2x]]/2` |
| `x Tan[x]²` | I-laden | `½(−x² + 2 Log[Cos[x]] + 2x Tan[x])` |
| `2x Tan[x²]` | I-laden | `−Log[Cos[x²]]` |

Covers polynomials in `Tan`, scaled/half-angle/nonlinear arguments (`Tan[2x]`,
`Tan[x/2]`, `Tan[x²]`), special-pole `(t²+1)^k` rational parts (via the coupled
reduced case), and simple linear normal poles (rational residues). Genuinely
non-elementary tangent integrands (`∫ x Tan[x]`, `∫ x² Tan[x]²`) are **proven
non-elementary** and left unevaluated. `Tanh` (special factor `t²−1`, not `t²+1`)
is correctly left on the `TrigToExp` path.

Scope: a normal-pole gate defers a denominator whose normal part (after stripping
`t²+1`) has degree ≥ 2 to the fallback path — such poles can carry irrational-
algebraic residues whose tower-derivation spillover swells the `p = h − D[g2] + r`
reconciliation over the residue field; the driver's efficient handling of those is
a follow-up. The gate keeps the live integrator responsive; the full driver stays
available as `Risch\`IntegrateHypertangent`.

Tests: four new groups in `tests/test_integrate_risch_transcendental.c`

risch + integrator suites green; leak-clean (valgrind == `Sin[1.0]` baseline,
zero hypertangent frames).

## Real `Cot` and `Tanh` integration — the hypertangent family (Bronstein §5.10 + hyperbolic) (2026-07-13)

Extended the real-tangent framework to **cotangent** and **hyperbolic tangent**,
so all three now integrate directly and real instead of via `TrigToExp`.

**Cot** — `D[Cot[u]] = −u'(1+t²)`, so `t=Cot[u]` is a hypertangent monomial
(special `t²+1`, `η=−u'`): it reuses the entire `IntegrateHypertangent` driver
unchanged; only the live wiring is new (cosmetic `Log[1+Cot[u]²] → −2 Log[Sin[u]]`).

**Tanh** — `D[Tanh[u]] = u'(1−t²)`, special `t²−1 = (t−1)(t+1)`, which **splits
over k**. New driver `risch_integrate_hypertanh` (`src/calculus/risch_hypertangent.c`)
mirrors the §5.10 composition but its reduced case exploits the split: peeling a
`(ct+d)/(t²−1)^m` pole gives a coupled system `[[0,−2mη],[−2mη,0]]` that
**decouples over the reals** — with `P=c+d`, `Q=c−d` it is two independent base
Risch DEs `DP−2mη P = a+b` and `DQ+2mη Q = a−b` (no `ℂ(i)`, unlike the circular
case). The `IntegrateHypertangent` driver was refactored into a shared
`rh_integrate_core` parametrized by the reduced/polynomial sub-builtins and the
special polynomial; three new internal builtins expose the hyperbolic pieces:
`Risch\`IntegrateHypertanh`, `Risch\`IntegrateHypertanhReduced`,
`Risch\`IntegrateHypertanhPolynomial` (all `PROTECTED | READPROTECTED`).

The live wiring (`rt_hypertangent_case`) is generalized into a family dispatcher
+ core `rt_hypertan_family` handling `Tan`/`Cot`/`Tanh` — each with its tower
derivative, special polynomial (for the normal-pole gate), driver, and real
cosmetic collapse. Results:

| integrand | now (real) |
|---|---|
| `Cot[x]` | `Log[Sin[x]]` |
| `Cot[x]²` | `−Cot[x] − x` |
| `Cot[2x]` | `½ Log[Sin[2x]]` |
| `Tanh[x]` | `Log[Cosh[x]]` |
| `Tanh[x]²` | `x − Tanh[x]` |
| `Tanh[x]⁴` | `x − Tanh[x] − Tanh[x]³/3` |
| `Tanh[2x]` | `½ Log[Cosh[2x]]` |
| `x Tanh[x]²` | `½(x² + 2 Log[Cosh[x]] − 2x Tanh[x])` |
| `1/(1−Tanh[x]²)` | `x/2 + Sinh[x]Cosh[x]/2` (reduced pole, real) |

Non-elementary cases (`∫ x Tanh[x]`, `∫ x Tanh[x]³`) are proven and left
unevaluated.

**Coth** was added too: `D[Coth[u]] = u'(1−t²)` is the *same* monomial as Tanh
(special `t²−1`), so it reuses the hyperbolic driver directly; the only
difference is the real cosmetic `Log[1−Coth[u]²] → −2 Log[Sinh[u]]` (since
`1−Coth² = −Csch²`, the sign folds into the derivative-preserving rewrite, which
the diff-back gate confirms). `∫Coth → Log[Sinh[x]]`, `∫Coth² → x − Coth[x]`,
`∫Coth[2x] → ½Log[Sinh[2x]]`. The live dispatcher (`rt_hypertan_family` refactored
to take a substitution rule + back-substitution value) now covers
Tan/Cot/Tanh/Coth uniformly. `Sec`/`Csc`/`Sech`/`Csch` remain separate kernel
heads on the fallback path.

*Explored but reverted:* routing rational functions of circular trig (`Sec^n`,
`Csc^n`) through the tangent path via the Weierstrass substitution `t = Tan[x/2]`.
The driver integrates the substituted forms correctly and fast in isolation
(e.g. `∫Csc → Log[Tan[x/2]]`), but a genuine driver perf pathology on
high-multiplicity *split* poles — `(1−t²)³` from `Sec[x]³` blows up the Hermite
reconciliation while `t³` from `Csc[x]³` is fine — made the transcendental suite
10× slower with failures, so the Weierstrass front-end and the gate relaxation it
needed were backed out. Prerequisite for retrying: fixing the driver's handling
of high-multiplicity split normal poles.

Tests: two new groups in `tests/test_risch_hypertangent.c` (`test_hypertanh_values`,
`test_hypertanh_reduced_and_constructed` — book-value forms, reduced-pole and

derivation rejection, non-elementary decline) and two in
`tests/test_integrate_risch_transcendental.c` (`test_real_cotangent`,
`test_real_hypertanh` — clean forms by value, real-ness + diff-back across powers,
scalings, half-angle, mixed and reduced-pole cases, non-elementary unevaluated).
All suites green; leak-clean (valgrind == `Sin[1.0]` baseline, zero new-code

## `Sum[]` cascade — all pre-existing memory leaks fixed (2026-07-13)

Every infinite `Sum` that flowed through the sub-algorithm cascade leaked a
handful of small `Expr` nodes (14–54 blocks per input, per `SUM_LEAKS.md`),
even when it produced the correct closed form. Root-caused and eliminated; all
five recorded reproducers and every stage now report **0 leaks / 0 bytes**
under macOS `leaks --atExit`.

**Two distinct ownership bugs, both in the stages that run and bail before the
closing stage:**

1. **`evaluate(expr_copy(f))` in 7 stages** — `evaluate()` *borrows* its
   argument (it internally does `current = expr_copy(e)` and never frees `e`),
   so the anonymous `expr_copy(f)` reference was never released, leaking the
   canonicalised summand tree. Fixed to the correct borrow-then-free idiom
   (`fc = expr_copy(f); fn = evaluate(fc); expr_free(fc);`) in
   `sum_alternating.c`, `sum_trigonometric.c`, `sum_dirichlet.c`, `sum_euler.c`,
   `sum_euler_nonlinear.c`, `sum_geometric.c`, and `sum_zetaseries.c`. `Sum\`Gosper`
   was already clean because it consistently frees the evaluate input.

2. **Redundant `expr_copy(base)` in `Sum\`Rational`** — `sr_term_contribution`
   uses its `base` argument only through copies (`sr_pdeg`/`sum_subst`), so it
   never needed to own it, yet the sole call site passed `expr_copy(base)` and
   the callee never freed it. Dropped the copy so `base` is borrowed from the
   `Apart` result — matching the already-clean sibling `alt_term_contribution`.

No behavioural change: `sum_tests` (64) and `nsum_tests` still pass, and the
closed forms are unchanged (`Sum[1/k^2]→Pi^2/6`, `Sum[1/(k(k+1))]→1`,
`Sum[(-1)^(k+1)/k]→Log[2]`, `Sum[1/(k^2+1)]→(Pi Coth[Pi]-1)/2`, …).

## Evaluator/symtab performance — Phase 0 baseline harness

string interner and symbol table, precomputes base attributes into the `SymbolDef`, and
caches a `SymbolDef*` on `EXPR_SYMBOL` so per-node dispatch becomes a pointer dereference
(no re-hashing, no ~140-entry `strcmp` scan in `get_attributes()`).

Phase 0 (guardrails) landed:

  modeled on `bench_assoc`: median-of-5 wall time, machine-normalized against

## Evaluator/symtab performance — Phase 1: base-attribute fold

`get_attributes()` was on the evaluator hot path (called per function node per pass) yet did
a linear `strcmp` scan of the ~140-entry `builtin_attrs[]` table every call, then merged the
dynamic `def->attributes`. Phase 1 folds each symbol's immutable static "base" attributes
into the `SymbolDef` once (new `base_attributes` + `base_seeded` fields; seeded lazily on
first `get_attributes`, eagerly in `attr_init` for table symbols), so every call is now an
O(1) field read + OR: `base_attributes | def->attributes`.

- Bit-for-bit identical to the old `scan | dynamic`, including the un-clearable base floor
  (builtins remain effectively Protected against `Unprotect`, the documented behavior).
  `def->attributes` keeps its exact meaning, so match.c's direct attribute reads and the 13+
  Block-style save/restore sites (findmin, nsum/nlimit/nresidue/nseries/nderiv/nint, modular,
  core Protect/Unprotect/ClearAll) need no change.
- Measured on `tests/bench_eval` (Apple M-series): pure-dispatch workload
  (`Length[Table[hh[k,k],{k,8000}]]`) **−33%** (norm 0.60 → 0.40); DownValue rewrite −16%,
  Listable threading −13%; matcher/compare-bound rows unchanged, as predicted. Suite green
  (clearall_remove_protect, core, eval, match, symtab, eval_timestamps, regression, list).
- Files: `src/symtab.h`, `src/symtab.c`, `src/attr.c`. Struct layout changed → full rebuild
  (`make clean`) required.

## Evaluator/symtab performance — Phase 2: unified interner + symbol table

The string interner (8191-bucket table of `InternEntry`) and the symbol table
(65535-bucket table of `SymEntry`→`SymbolDef`) are merged into a single 65535-bucket table
of self-chaining `SymbolDef` nodes. A symbol's name is interned and its definition looked up
through **one** hash of **one** table instead of hashing the name twice across two.

- `SymbolDef` now owns its canonical name string and carries `is_system` (migrated from
  `InternEntry`), a `has_def` materialization flag, and a `next` bucket-chain pointer;
  `SymEntry` is deleted. The `sym_intern.h` API (`intern_symbol`, `intern_is_system`,
  `intern_mark_all_system`, `intern_clear`) is now implemented in `symtab.c` over the unified
  table; `sym_intern.c` is an empty stub.
- `has_def` preserves observable semantics bit-for-bit: `intern_symbol` creates an
  interned-only node (`has_def=0`), `symtab_get_def` materializes (`has_def=1`); `Names[]` /
  `symtab_for_each` enumerate and `symtab_lookup` reports only `has_def=1` nodes — exactly the
  pre-merge "has a SymbolDef" set. `symtab_clear` / `symtab_remove_symbol` reset a node to
  interned-only (payload freed, `has_def=0`) but keep the node + name string so `SYM_*` and
  live `Expr` pointers stay valid; `symtab_init` resets payloads instead of `memset`;
  `intern_clear` is the sole full teardown (frees nodes + names).
- Verified: clean build; green on names, symtab (3× re-init), context, core, eval, match,
  clearall_remove_protect, eval_timestamps, regression, rule_dispatch (exercises
  `intern_clear`). Valgrind leak total byte-identical to a bare `Sin[1.0]` session (no
  Mathilda-source frames in any leak stack). `bench_eval` gate green.
- Files: `src/symtab.h`, `src/symtab.c`, `src/sym_intern.c`. Struct layout changed → full
  rebuild (`make clean`) required.

## Evaluator/symtab performance — Phase 3a: resolve the head's def once

`evaluate_step` now resolves the head symbol's `SymbolDef*` a single time per pass and
threads it through attribute lookup, DownValue dispatch, and builtin dispatch, rather than
re-hashing the head up to three times. New def-threaded fast paths `get_attributes_def()`
(attr.c) and `apply_down_values_def()` (symtab.c) do the work; the name-taking
`get_attributes` / `apply_down_values` are now thin resolve-then-delegate wrappers. The
Phase 2 def node is stable, so the cached pointer stays valid across the step and each use
reads its fields fresh. No struct change; green on the eval/match/core/symtab/attribute/
names/regression/rule_dispatch suites; no new allocations.

Phase 3b (caching the `SymbolDef*` on the `EXPR_SYMBOL` node) was **measured out**: after
Phase 1 removed the attribute strcmp scan, the pure-dispatch benchmark is flat across Phases
1/2/3a (the residual per-node symtab lookups are <2% of per-node cost), so the node cache —
which would require renaming ~2446 `data.symbol` sites across 277 files — buys ~nothing and
is left as documented future work. Files: `src/eval.c`, `src/attr.c/h`, `src/symtab.c/h`.

## Evaluator/symtab performance — Phase 3b: cache SymbolDef* on EXPR_SYMBOL

The `EXPR_SYMBOL` union member is now `struct { char* name; struct SymbolDef* def; }`. The
evaluator resolves the head's definition once and caches it on the symbol node, so repeated
passes over the same node reach attributes/DownValues/builtin by a pointer load instead of a
symbol-table lookup. `name` stays at union offset 0, so the historical EXPR_STRING type-pun
holds and `sizeof(Expr)` is unchanged.

- Mechanical rename of 2864 `data.symbol` → `data.symbol.name` sites across 356 files; the 3
  in-place name writers now maintain the cache (`expr_new_symbol` NULLs it for lazy resolve,
  `expr_unshare` copies it, `alpha_rename` NULLs it since the name changed).
- Safe because Phase 2 made SymbolDef nodes stable (reset in place, never reallocated): a
  cached def always points to the current node for that name, and its fields are read fresh.
  Valgrind: 0 invalid reads/writes, leak total byte-identical to baseline.
- Performance: neutral. Pure-dispatch is flat (~14,200 µs) across Phases 1/2/3a/3b — after
  Phase 1 removed the attribute strcmp scan, the eliminated per-node lookup is <2% of node
  cost. Implemented for architectural completeness; the measurable win came from Phase 1.
- Files: `src/expr.h`, `src/expr.c`, `src/symtab.c`, `src/eval.c`, plus the mechanical rename
  across `src/` and `tests/`. Struct definition changed → `make clean` required.

## Build: raylib static X11/GL transitive deps auto-linked (issue #18)

The graphics engine links raylib, which on Linux is commonly a static archive
(`libraylib.a`) whose transitive windowing deps — `-lX11 -lGL -lpthread -ldl
-lrt ...` — live in the `.pc` file's `Libs.private:` line. Both build systems
were pulling only the plain `Libs:` set, so test executables (and the makefile
build without `EXTRA_LIBS`) failed to link with undefined `XInternAtom`,
`XSync`, `XFree`, etc. Fixed to link the pkg-config *static* (private-dep-
inclusive) set:

- `tests/CMakeLists.txt`: link `${RAYLIB_STATIC_LIBRARIES}` / `_STATIC_LIBRARY_DIRS`
  (falling back to the plain set if empty) instead of `${RAYLIB_LIBRARIES}`.
- `makefile`: `pkg-config --static --libs raylib` instead of `--libs`.
- No-op on macOS, where `Libs.private` is empty (frameworks are recorded in the
  dylib) so the static set equals the plain set; verified the link line and a
  test build are byte-identical there. Removes the need for the manual
  `make EXTRA_LIBS="-lX11"` workaround on Linux.

## SinIntegral and Sinc — sine integral and cardinal sine (2026-07-14)

New entire special functions:

- **`SinIntegral[z]`** = Si(z) = ∫₀ᶻ Sin[t]/t dt (`src/special_functions/sinintegral.c`).
  Odd, entire, no branch cuts. Exact values `0 → 0`, `±Infinity → ±Pi/2`,
  `±I Infinity → ±I Infinity`, `ComplexInfinity → Indeterminate`; symbolic odd
  fold `SinIntegral[-x] → -SinIntegral[x]`. Numeric via a convergent Maclaurin
  series (guard bits absorb the ~e^|z| cancellation) with an asymptotic
  `Pi/2 - cos f - sin g` fallback for large |z|; machine + arbitrary-precision
  reals and complexes (complex via the shared `ncpx` toolkit). Matches WL to 50
  digits: `N[SinIntegral[2], 50] = 1.605412976802694848576720148198588940848583422328…`.
- **`Sinc[z]`** = Sin[z]/z with `Sinc[0] = 1` (`src/special_functions/sinc.c`).
  Even, entire. Exact `0 → 1`, `±Infinity → 0`, `ComplexInfinity → Indeterminate`;
  numeric reals/complexes at machine or MPFR precision. Introduced so
  `D[SinIntegral[x], x]` returns `Sinc[x]` as in WL.
- **Derivatives** (`src/calculus/deriv.c`): `D[SinIntegral[z], z] = Sinc[z]`,
  `D[Sinc[z], z] = Cos[z]/z - Sin[z]/z^2`.
- **Series** (`src/calculus/series.c`): Taylor at 0 for `SinIntegral` via the
  odd-kernel-at-zero path (also composes, e.g. `SinIntegral[x^2]`); `Sinc`
  series at 0 via a `Sinc → Sin[x]/x` rewrite; and a trig-prefactored asymptotic
  expansion at Infinity for `SinIntegral`
  (`Pi/2 - Sin[x]/x^2 + Cos[x] (-1/x + 2/x^3)`). The symbolic
  `DirectedInfinity[z]`-direction constant form is deferred.
- Attributes for both: `Listable`, `NumericFunction`, `Protected`; docstrings in
  `info.c`; symbols in `sym_names.c`. Unit tests `tests/test_sinintegral.c`,
  `tests/test_sinc.c` (valgrind-clean).
