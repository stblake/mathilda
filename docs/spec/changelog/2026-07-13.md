# Changelog: week of 2026-07-13 (Mon) – 2026-07-19 (Sun)

Feature additions and fixes recorded during this week.

## `Sum[]` cascade — all pre-existing memory leaks fixed (2026-07-13)

Every infinite `Sum` that flowed through the sub-algorithm cascade leaked a
handful of small `Expr` nodes (14–54 blocks per input, per `SUM_LEAKS.md`),
even when it produced the correct closed form. Root-caused and eliminated; all
five recorded reproducers and every stage now report **0 leaks / 0 bytes**
under macOS `leaks --atExit`.

**Two distinct ownership bugs, both in the stages that run and bail before the
closing stage:**

1. **`evaluate(expr_copy(f))` in 7 stages** — `evaluate()` *borrows* its
   argument (it internally does `current = expr_copy(e)` and never frees `e`),
   so the anonymous `expr_copy(f)` reference was never released, leaking the
   canonicalised summand tree. Fixed to the correct borrow-then-free idiom
   (`fc = expr_copy(f); fn = evaluate(fc); expr_free(fc);`) in
   `sum_alternating.c`, `sum_trigonometric.c`, `sum_dirichlet.c`, `sum_euler.c`,
   `sum_euler_nonlinear.c`, `sum_geometric.c`, and `sum_zetaseries.c`. `Sum\`Gosper`
   was already clean because it consistently frees the evaluate input.

2. **Redundant `expr_copy(base)` in `Sum\`Rational`** — `sr_term_contribution`
   uses its `base` argument only through copies (`sr_pdeg`/`sum_subst`), so it
   never needed to own it, yet the sole call site passed `expr_copy(base)` and
   the callee never freed it. Dropped the copy so `base` is borrowed from the
   `Apart` result — matching the already-clean sibling `alt_term_contribution`.

No behavioural change: `sum_tests` (64) and `nsum_tests` still pass, and the
closed forms are unchanged (`Sum[1/k^2]→Pi^2/6`, `Sum[1/(k(k+1))]→1`,
`Sum[(-1)^(k+1)/k]→Log[2]`, `Sum[1/(k^2+1)]→(Pi Coth[Pi]-1)/2`, …).

## Risch differential-field foundation — five new `Risch\`` field primitives + extensive unit tests (2026-07-13)

The Bronstein transcendental-Risch foundation modules (`risch_field`,
`risch_canonical`, `risch_structure`, `risch_hypertangent`, landed 2026-07-12)
were under-tested — 88 assertions across 14 functions, mostly one book example
per algorithm. Five core field-arithmetic primitives had **no builtin wrapper**
and so were exercised only indirectly, with zero tests in isolation.

**New internal builtins** (`READPROTECTED`, in `src/calculus/risch_canonical.c`),
thin wrappers over the existing `risch_field.c` primitives so the field
arithmetic of `k[t]` over `k = C(x, lower monomials)` is REPL-inspectable and
directly testable:

- `Risch\`FieldGCD[a, b, t]` — monic-in-`t` gcd over the field `k` (pure-`k`
  factors are units, so the result is `1` when `a`, `b` share only a `t`-free
  factor).
- `Risch\`DivExact[a, b, t]` — exact quotient `a/b` when it is a polynomial in
  `t`; stays unevaluated otherwise.
- `Risch\`NumDen[f, t]` — `{a, d}` with `a/d = f`, `d` monic in `t`.
- `Risch\`ExtendedEuclidean[a, b, t]` — `{g, u, v}` with `u a + v b = g`.
- `Risch\`Diophantine[dn, ds, r, t]` — `{b, c}` with `b dn + c ds = r`,
  `deg_t(b) < deg_t(ds)`, for coprime `dn`, `ds`.

**Test suite deepened from 88 → 278 assertions (14 → 34 functions):**

- New `tests/test_risch_field.c` (10 functions, 107 assertions) — the §3.4 field
  layer in isolation: the monomial derivation across exponential/log/hypertangent/
  nonlinear kinds and 2-monomial towers; `FieldGCD`/`DivExact`/`NumDen` including
  field-unit behaviour of pure-`k` factors; property round-trips for `PolyDivide`
  (`a = q b + r`, `deg r < deg b`), `ExtendedEuclidean` (Bézout `u a + v b = g`),
  and `Diophantine` (`b dn + c ds = r`, degree bound); exhaustive normal/special
  classification; `PolynomialReduce` reconstruction; and robustness (malformed
  derivation, wrong arity, division by zero all stay unevaluated).
- `tests/test_risch_canonical.c` (+4 functions) — `SplitFactor` edge cases
  (pure-normal, pure-special, constant, deep multiplicity) and its defining
  property `p_n p_s = p` with `p_s` special / `p_n` non-special; three-multiplicity
  `SplitSquarefreeFactor` with reconstruction; `CanonicalRepresentation` across
  all monomial kinds with poly + special + normal parts, and special-only /
  normal-only degenerate cases.
- `tests/test_risch_structure.c` (+3 functions) — signed/rational/three-generator
  `RationalSpan`; 3-deep towers, nested `log(log x)` / `exp(exp x)` (both correctly
  a genuinely new monomial), and product/power-law reductions.
- `tests/test_risch_hypertangent.c` (+3 functions) — the §5.10 certificate
  `p − D[q] − c·D(t²+1)/(t²+1) ∈ k` verified as **free of `t`** (not zero — a
  nonzero constant is the leftover `∫(element of k)` base-field part) for degrees
  3–6; the `Dc = 0` (elementary) vs `Dc ≠ 0` (non-elementary, Liouville) boundary;
  `tan(a x)` scaling `c = 1/(2a)`; and non-hypertangent monomials rejected.

Clean under `-Wall -Wextra`; leak-clean (valgrind identical to the macOS
dyld/Accelerate baseline noise, no `risch_*` frames in any leak stack).

## Evaluator/symtab performance — Phase 0 baseline harness

`EVAL_SYMTAB_IMPROVEMENTS.md` (new, repo root) is a full implementation spec for the
next-generation evaluator + symbol-table rework: a **symbol-object model** that unifies the
string interner and symbol table, precomputes base attributes into the `SymbolDef`, and
caches a `SymbolDef*` on `EXPR_SYMBOL` so per-node dispatch becomes a pointer dereference
(no re-hashing, no ~140-entry `strcmp` scan in `get_attributes()`).

Phase 0 (guardrails) landed:

- **`tests/bench_eval.c`** (+ ctest target `bench_eval`) — evaluator baseline benchmark
  modeled on `bench_assoc`: median-of-5 wall time, machine-normalized against
  `Total[Range[40000]]`, gates at 2.5× a committed baseline. Six workloads targeting the
  dispatch path (pure distinct-node dispatch, DownValue rewrite, Listable threading,
  Orderless sort, ReplaceRepeated fold, nested Plus collapse) rather than heavy builtin
  bodies. Baselines pinned (Apple M-series, USE_ECM=OFF); the pure-dispatch row
  (`Table[hh[k,k],{k,8000}]`, ~2.7 µs/node) is the primary signal for the upcoming Phase 1
  (base-attribute precompute) and Phase 3 (cached def) speedups, which should push the
  normalized costs down.

## Evaluator/symtab performance — Phase 1: base-attribute fold

`get_attributes()` was on the evaluator hot path (called per function node per pass) yet did
a linear `strcmp` scan of the ~140-entry `builtin_attrs[]` table every call, then merged the
dynamic `def->attributes`. Phase 1 folds each symbol's immutable static "base" attributes
into the `SymbolDef` once (new `base_attributes` + `base_seeded` fields; seeded lazily on
first `get_attributes`, eagerly in `attr_init` for table symbols), so every call is now an
O(1) field read + OR: `base_attributes | def->attributes`.

- Bit-for-bit identical to the old `scan | dynamic`, including the un-clearable base floor
  (builtins remain effectively Protected against `Unprotect`, the documented behavior).
  `def->attributes` keeps its exact meaning, so match.c's direct attribute reads and the 13+
  Block-style save/restore sites (findmin, nsum/nlimit/nresidue/nseries/nderiv/nint, modular,
  core Protect/Unprotect/ClearAll) need no change.
- Measured on `tests/bench_eval` (Apple M-series): pure-dispatch workload
  (`Length[Table[hh[k,k],{k,8000}]]`) **−33%** (norm 0.60 → 0.40); DownValue rewrite −16%,
  Listable threading −13%; matcher/compare-bound rows unchanged, as predicted. Suite green
  (clearall_remove_protect, core, eval, match, symtab, eval_timestamps, regression, list).
- Files: `src/symtab.h`, `src/symtab.c`, `src/attr.c`. Struct layout changed → full rebuild
  (`make clean`) required.

## Evaluator/symtab performance — Phase 2: unified interner + symbol table

The string interner (8191-bucket table of `InternEntry`) and the symbol table
(65535-bucket table of `SymEntry`→`SymbolDef`) are merged into a single 65535-bucket table
of self-chaining `SymbolDef` nodes. A symbol's name is interned and its definition looked up
through **one** hash of **one** table instead of hashing the name twice across two.

- `SymbolDef` now owns its canonical name string and carries `is_system` (migrated from
  `InternEntry`), a `has_def` materialization flag, and a `next` bucket-chain pointer;
  `SymEntry` is deleted. The `sym_intern.h` API (`intern_symbol`, `intern_is_system`,
  `intern_mark_all_system`, `intern_clear`) is now implemented in `symtab.c` over the unified
  table; `sym_intern.c` is an empty stub.
- `has_def` preserves observable semantics bit-for-bit: `intern_symbol` creates an
  interned-only node (`has_def=0`), `symtab_get_def` materializes (`has_def=1`); `Names[]` /
  `symtab_for_each` enumerate and `symtab_lookup` reports only `has_def=1` nodes — exactly the
  pre-merge "has a SymbolDef" set. `symtab_clear` / `symtab_remove_symbol` reset a node to
  interned-only (payload freed, `has_def=0`) but keep the node + name string so `SYM_*` and
  live `Expr` pointers stay valid; `symtab_init` resets payloads instead of `memset`;
  `intern_clear` is the sole full teardown (frees nodes + names).
- Verified: clean build; green on names, symtab (3× re-init), context, core, eval, match,
  clearall_remove_protect, eval_timestamps, regression, rule_dispatch (exercises
  `intern_clear`). Valgrind leak total byte-identical to a bare `Sin[1.0]` session (no
  Mathilda-source frames in any leak stack). `bench_eval` gate green.
- Files: `src/symtab.h`, `src/symtab.c`, `src/sym_intern.c`. Struct layout changed → full
  rebuild (`make clean`) required.

## Evaluator/symtab performance — Phase 3a: resolve the head's def once

`evaluate_step` now resolves the head symbol's `SymbolDef*` a single time per pass and
threads it through attribute lookup, DownValue dispatch, and builtin dispatch, rather than
re-hashing the head up to three times. New def-threaded fast paths `get_attributes_def()`
(attr.c) and `apply_down_values_def()` (symtab.c) do the work; the name-taking
`get_attributes` / `apply_down_values` are now thin resolve-then-delegate wrappers. The
Phase 2 def node is stable, so the cached pointer stays valid across the step and each use
reads its fields fresh. No struct change; green on the eval/match/core/symtab/attribute/
names/regression/rule_dispatch suites; no new allocations.

Phase 3b (caching the `SymbolDef*` on the `EXPR_SYMBOL` node) was **measured out**: after
Phase 1 removed the attribute strcmp scan, the pure-dispatch benchmark is flat across Phases
1/2/3a (the residual per-node symtab lookups are <2% of per-node cost), so the node cache —
which would require renaming ~2446 `data.symbol` sites across 277 files — buys ~nothing and
is left as documented future work. Files: `src/eval.c`, `src/attr.c/h`, `src/symtab.c/h`.

## Evaluator/symtab performance — Phase 3b: cache SymbolDef* on EXPR_SYMBOL

The `EXPR_SYMBOL` union member is now `struct { char* name; struct SymbolDef* def; }`. The
evaluator resolves the head's definition once and caches it on the symbol node, so repeated
passes over the same node reach attributes/DownValues/builtin by a pointer load instead of a
symbol-table lookup. `name` stays at union offset 0, so the historical EXPR_STRING type-pun
holds and `sizeof(Expr)` is unchanged.

- Mechanical rename of 2864 `data.symbol` → `data.symbol.name` sites across 356 files; the 3
  in-place name writers now maintain the cache (`expr_new_symbol` NULLs it for lazy resolve,
  `expr_unshare` copies it, `alpha_rename` NULLs it since the name changed).
- Safe because Phase 2 made SymbolDef nodes stable (reset in place, never reallocated): a
  cached def always points to the current node for that name, and its fields are read fresh.
  Valgrind: 0 invalid reads/writes, leak total byte-identical to baseline.
- Performance: neutral. Pure-dispatch is flat (~14,200 µs) across Phases 1/2/3a/3b — after
  Phase 1 removed the attribute strcmp scan, the eliminated per-node lookup is <2% of node
  cost. Implemented for architectural completeness; the measurable win came from Phase 1.
- Files: `src/expr.h`, `src/expr.c`, `src/symtab.c`, `src/eval.c`, plus the mechanical rename
  across `src/` and `tests/`. Struct definition changed → `make clean` required.
