# Changelog: week of 2026-06-22 (Mon) – 2026-06-28 (Sun)

Feature additions and fixes recorded during this week.

## Feature: `Frame -> True` for `Graphics`/`Plot` with two-tier ticks (2026-06-23)

`Graphics`, `Show` and `Plot` now honour `Frame`, drawing a tick-and-label
rectangle along the edges of the plot in place of the through-the-origin `Axes`
cross. New options: `Frame`, `FrameTicks`, `FrameStyle` (with new internal
symbols `SYM_Frame`/`SYM_FrameStyle`/`SYM_FrameTicks`).

- **Settings** — `Frame -> True` boxes all four edges; `Frame -> False`/`None`
  draws none; `Frame -> {{left, right}, {bottom, top}}` toggles each edge with
  `True`/`False`. In `Plot`, supplying `Frame -> True` withholds the default
  `Axes -> True` (pass `Axes -> True` explicitly to keep both), mirroring
  Wolfram; `Frame -> False` leaves the axes default intact.
- **Reserved margin + clipping** — the frame reserves a margin (~5% of each
  window dimension; bottom/left larger to hold labels and the help line) and the
  curve is fitted to and clipped against the interior, so it never spills past
  the box. Ticks point inward; numeric labels sit *outside* the frame in the
  margin. The bottom help line therefore also falls outside the frame.
- **Two-tier ticks** — major ticks land on the same "nice" values as the axes;
  minor sub-ticks subdivide each major interval, the subdivision count chosen
  from the major step's leading digit so minors stay on round values (step `1`
  → 5, `2` → 4, `5` → 5, across all magnitudes). Minor ticks are half-length and
  unlabeled; major ticks on the bottom/left edges (or top/right when those are
  the only ones drawn) carry numeric labels in the margin.
- **Weight** — the frame box, its ticks and its labels render at 1.5 px (new
  `hershey_draw_text_ex()` adds a thickness parameter with rounded joins for the
  labels). The `Axes` lines, ticks and labels were lifted to the same 1.5 px so
  the plot looks identical in weight whether it is framed (`Frame -> True`) or
  axed (`Frame -> False`); the axis strokes use `1.5/zoom` world width so they
  stay 1.5 px on screen at every zoom level.
- **`FrameTicks`** — `Automatic` (default) shows ticks on every drawn edge;
  `None` keeps the box but drops all ticks and labels; the per-edge
  `{{left, right}, {bottom, top}}` form selects `Automatic`/`None` per edge.
  Ticks appear only on edges that the `Frame` setting actually draws.
- **`FrameStyle`** — an `RGBColor`/`GrayLevel` colour for the box, ticks and
  labels (defaults to the gray axis tone).

The frame is a fixed screen-space rectangle while the data pans/zooms within it
(clipped); the tick *values*, read from the data coordinates at the current
frame edges, update live like a ruler. The tick-subdivision policy is factored
into `frame_minor_divs()` (declared in `render.h`) and unit-tested headless
alongside the option-passthrough and axes-suppression behaviour in
`tests/test_graphics.c`.

## Feature: `AspectRatio` reshapes the plot window (2026-06-23)

`AspectRatio` now sizes the **window itself** rather than only stretching the
data inside a fixed box, matching Wolfram semantics. The supported settings:

- `Automatic` — the height-to-width ratio is taken from the actual coordinate
  extent (true geometry), so `Graphics[Circle[]]` opens a square window.
- `Full` — keeps the `ImageSize` box and stretches the graphics to fill it.
- an explicit ratio `a` — the window height becomes `round(width * a)`, clamped
  to a screen-friendly `[100, 2000]` band. `a` may be a plain number or a
  symbolic-numeric value such as `1/GoldenRatio` or `GoldenRatio`; `Plot`
  defaults to `1/GoldenRatio` (a default `800`-wide plot opens at `800 × 494`).

`ImageSize -> w` sets the width with the height following from `AspectRatio`;
`ImageSize -> {w, h}` pins both dimensions, after which `AspectRatio` shapes the
data inside the fixed frame.

Implementation: `Plot` (`src/graphics/plot.c`) numericalizes any non-symbol
`AspectRatio` value through `N[]` before storing it on the returned `Graphics`,
so the evaluator-less renderer always receives a machine number; `Automatic`/
`Full` pass through verbatim. The window-shaping policy is factored into the
pure, unit-tested `gfx_window_height()` (`src/graphics/render.{c,h}`). New
symbol `Full` (`src/sym_names.c`); docstrings for `AspectRatio` and `ImageSize`
(`src/info.c`). Tests in `tests/test_graphics.c` cover every option setting
(option resolution at the `Graphics` level plus the window-height policy).

## Feature: `Mesh -> All` overlays evaluation points on `Plot` (2026-06-23)

`Plot[f, {x, xmin, xmax}, Mesh -> All]` now draws a dot at every point the
adaptive sampler evaluated, in addition to the curve, in the curve's colour.
`Mesh -> None` (the default) draws the line only; `Mesh -> True`/`False` are
accepted as synonyms for `All`/`None`. The dots are sized as a small fraction
of the x-span so they stay roughly constant on screen across plot ranges, and
the setting is carried through the zoom/pan re-sampler so the mesh tracks the
curve when magnified. Implemented in `src/graphics/plot.c`; the hidden
`$PlotResample` metadata node gained a sixth `mesh` field. Besides being a
requested feature, `Mesh -> All` is the quickest way to see where the sampler
is and isn't placing points when diagnosing a rough-looking curve.

## Fix: multi-probe adaptive flatness test — robust to oscillation (2026-06-23)

`Plot` rendered visibly piecewise-linear on functions with fine structure —
worst of all a sum of several sines
(`Plot[Sin[x] + Sin[2x] + Sin[7x] + Sin[11x] + Sin[15x] + Sin[22x], {x,-2,2}]`).
Root cause: the adaptive sampler's flatness test inspected only the **midpoint**
of each candidate segment. For an oscillatory function on a coarse grid the
two endpoints and the midpoint resonate — three nearly-collinear samples while
the curve wiggles between them — so the segment was accepted as flat and
refinement never started. Raising `MaxRecursion` did nothing (measured: error
pinned at 0.17 for depth 6, 8, and 10 alike) because the recursion is never
entered.

`segment_is_flat`/`subdivide` (`src/graphics/sampling.c`) now probe **three**
interior points (the 1/4, 1/2 and 3/4 positions) and measure the maximum
**vertical** gap between the curve and the chord the renderer would draw,
normalized by the spike-clamped displayed y-extent (`plot_robust_yrange` over a
first uniform pass — a lone asymptote spike cannot inflate the scale and
re-flatten everything). A false "flat" now requires four aligned samples, which
an off-period grid does not provide, so the wiggle is caught. Vertical gap is
exactly the on-screen error for a `y = f(x)` curve.

Measured on `PlotPoints=50`/`MaxRecursion=6`: the sine-sum's max polyline error
drops from **0.17 → 0.03** while using **fewer** stored points (698 → 207);
smooth curves (`Sin`, `x^3`) stay at the base 50 points with no over-refinement;
`1/x` localizes its pole in ~140 points. The default `PlotPoints` stays 50 and
`MaxRecursion` 6 — the fix is the test, not the budget.

## Tweak: denser default `PlotPoints` for smoother curves (2026-06-23)

`Plot`'s default initial sample count (`PlotPoints`) was doubled from 25 to
50. The adaptive bisection (`MaxRecursion`, default 6) still refines on top of
this denser grid, so curves such as `Plot[1/x, {x, -2, 2}]` render visibly
smoother out of the box. The same default applies to the hidden
`$PlotResample` zoom/pan path. Override with `PlotPoints -> n` as before.

## Feature: spike-resistant automatic vertical `PlotRange` (2026-06-23)

`Plot[Tan[x], {x, -2 Pi, 2 Pi}]` used to render with a vertical extent of
roughly ±10^16: the adaptive sampler climbs to enormous finite values near each
asymptote (`tan` never lands exactly on a pole, so there is no break to drop),
and the renderer's raw min/max bounding box let those spikes flatten the visible
curve to a line. `PlotRange -> Automatic` (the default) now frames the curve's
body instead.

Changes (`src/graphics/sampling.{c,h}`, `src/graphics/render.c`):

- **Robust band** (`plot_robust_yrange`, pure/unit-tested in `sampling.c`):
  computes a median ± `K·1.4826·MAD` fence (`K = 4`) and intersects it with the
  true data extent. The intersection is itself the spike guard — a genuine
  extreme inside the fence is kept; one that shoots far past is clamped — and it
  falls back to exact min/max when there are `< 8` points or the bulk is flat.
- **Runaway gate** (`prims_have_runaway` in `render.c`): the clip is applied
  *only* to a curve that contains a near-vertical asymptote, detected by a
  segment whose `|slope|` exceeds ~1000× the curve's median segment slope. This
  cleanly separates true poles (Tan ~10^17, `1/x` ~10^4, `Gamma` ~10^5) from
  merely steep smooth curves (`x^5` ~200, `x^3` ~16, `Exp` ~4) — and from curves
  with a vertical tangent (`Sqrt`) or a removable singularity (`Sinc`), all of
  which keep their full, legitimate extent.
- **Option parsing** (`gfx_options_parse`): `PlotRange -> Automatic` is now
  accepted explicitly (the default behaviour), and `PlotRange -> All` disables
  the clipping to show every sampled point. An explicit `{ymin, ymax}` still
  overrides outright.

Result: `Tan` frames to ≈ ±10, `1/x` to ≈ ±7, `Gamma` to ≈ [-8, 8.5], while
`Sin`, `x^2`, `x^3`, `x^5`, `Exp`, `Sqrt`, `Sinc` are unchanged. New unit tests
in `tests/test_graphics_sampling.c` cover the band heuristic directly (spike
clipping, legitimate-extrema preservation, the `< 8`-point fallback, and
symmetric two-sided spikes).

## Feature: `PlotRange -> {ymin, ymax}` short form + non-uniform y framing (2026-06-23)

Two changes in `src/graphics/render.c` so `PlotRange` actually controls the
vertical view:

- **Parsing** (`gfx_options_parse`): `PlotRange` now accepts a two-number list
  `{ymin, ymax}` in addition to the full `{{xmin,xmax},{ymin,ymax}}` form. The
  short form pins the y-axis while the x-extent stays automatic, matching
  Wolfram. The single `range_auto` flag was split into independent
  `x_auto`/`y_auto`, so the auto-fit pass overrides only the fixed axis.
- **Rendering**: the renderer used a single *uniform* camera zoom derived from
  the x-width and `AspectRatio`, so the y data-range never entered the zoom and
  `PlotRange` y was visually inert (it only re-centred). Introduced a
  non-uniform vertical scale `ysc = AspectRatio * data_w / data_h` applied in
  "render space": the x-range fills the width and the y-range fills its
  aspect-determined share of the height independently. With `AspectRatio`
  `Automatic` (bare `Graphics`) `ysc` collapses to `1` (true geometry, circles
  stay round); a forced `AspectRatio` (e.g. `Plot`'s `1/GoldenRatio`) now frames
  the requested y-band. The camera, mouse, pan, box-zoom and re-sampler all stay
  in render space; only primitive y-coordinates and y-tick labels convert.

Example: `Plot[Sin[1/x^2], {x, -2 Pi, 2 Pi}, PlotRange -> {-1.1, 1.1}]`.

## Feature: Plotly-style toolbar in the plot window (2026-06-22)

The interactive graphics window (`src/graphics/render.c`) now shows a
Plotly-style modebar of icon buttons in the top-right corner, all drawn as
uniform-gray hand-drawn vector glyphs (no image assets, matching the Hershey
font ethos). Buttons, each mirroring its Plotly counterpart:

- **Download PNG** (camera) — `TakeScreenshot` to `mathilda_plot.png` in the
  CWD. The capture frame suppresses all UI chrome (toolbar, help line,
  toast) so the saved image holds only the plot; the grab happens right
  after the buffer swap, matching raylib's own F12-capture point.
- **Zoom** (magnifier) — selects the box-zoom tool: left-drag a rectangle,
  release to fit the camera to it.
- **Pan** (four arrows) — selects the pan tool (default): left-drag to move.
- **Zoom in / Zoom out** (magnifier `+`/`−`) — scale ×1.3 about the center.
- **Autoscale** (outward arrows) and **Reset axes** (house) — restore the
  initial fit-to-data view.

The active tool (Pan/Zoom) is shown by a highlighted button background, not
by recolouring the icon, so the glyph set stays one consistent gray. Hover
shows a tooltip. Right/middle-drag still pans regardless of tool, and the
wheel/`Q`/`E`/`R`/`Esc` controls are unchanged. Plotly's Box-Select and
Lasso-Select are intentionally omitted — Mathilda renders continuous
primitives, not a discrete dataset, so there is nothing to select.

## Feature: close button + adaptive re-sampling on zoom (2026-06-22)

Two interactive-window improvements (`src/graphics/render.c`,
`src/graphics/plot.c`):

- **Close button** — a new "✕" button at the right end of the modebar
  closes the plot window (equivalent to `Esc` / the OS close box). It gets
  a red hover tint to flag its destructive action, distinct from the
  neutral highlight every other button shares.
- **Re-sampling on zoom/pan** — `Plot` now embeds its function, variable,
  and sampler options inside the returned `Graphics[...]` (as an internal,
  `HoldAll`, render-only `$PlotResample[...]` node that prints as nothing —
  `Graphics` still shows as `-Graphics-`). When the view changes enough —
  zoomed past ~30 % or panned to the edge of the sampled span — the
  renderer re-runs the adaptive sampler (`plot_resample`) over the new
  x-window. Curves like `Plot[Sin[1/x^2], {x, 1, 3}]` stay smooth when
  magnified instead of revealing the home view's coarse grid. Re-sampling
  is held off mid-drag (it lands on release) so gestures stay fluid;
  discrete scroll/zoom-button steps re-sample on the next frame. A
  hand-built (non-`Plot`) `Graphics` has no metadata, so the renderer
  detects that once and keeps the static primitives.

The re-sampling is internal: `Plot` output is unchanged structurally
(`g[[1]]` is still the primitive list) apart from the trailing metadata
node, and `Show[]` carries the metadata through so a re-`Show`n plot also
re-samples. No new leaks (valgrind: identical to baseline).

## Fix: mouse-wheel zooms about the cursor (2026-06-22)

Scroll-zoom (`src/graphics/render.c`) previously scaled `camera.zoom` while
leaving `camera.target` pinned to the plot centre, so every wheel notch dove
toward the *centre of the view* regardless of where the mouse pointed. On a
plot whose centre is its worst point — `Plot[Sin[1/x^2], {x, -2 Pi, 2 Pi}]`,
where the centre is the `x=0` essential singularity — this made zoom land on
the one spot no sampling can resolve, so the live re-sampler appeared not to
help even though it was running correctly. The wheel now zooms *about the
cursor*: the world point under the mouse is read before and after the zoom
step and `camera.target` is shifted to keep it fixed, so you magnify exactly
what you point at and can drive the view into a resolvable off-centre region
where the re-sampler's refinement is visible. Box-zoom (centres on the drawn
box) and the `+`/`−` toolbar buttons (centre zoom) are unchanged. Pure
camera-math change — no ownership, symbol, or sampling changes; all graphics
tests pass.

## Fix: top-level `Graphics` auto-renders; `Circle[]`/`Disk[]` defaults (2026-06-22)

A bare `Graphics[...]` result no longer just prints `-Graphics-` without a
window. Rendering is now owned by the REPL front end: `process_input`
(`src/repl.c`) renders any top-level result whose head is `Graphics`,
mirroring Mathematica's notebook auto-display. So
`{RGBColor[0,1,0], Circle[]} // Graphics` opens a window, with no explicit
`Show[]` needed.

To keep a single rendering path, `builtin_show` (`src/graphics/show.c`) and
`builtin_plot` (`src/graphics/plot.c`) no longer call `graphics_show`
themselves — they just build and return the `Graphics[...]` object, and the
REPL displays it. Benefits: `Show[g]`, `Plot[...]` and `g // Graphics` all
render identically, and a trailing `;` (yielding `Null`) now correctly
suppresses the window. The no-Raylib stub message was generalized from
"Show: …" to "Graphics: not rendered …".

`Circle`/`Disk` now apply Mathematica's argument defaults in the renderer
(`circle_params` in `src/graphics/render.c`, used by both the bounding-box
and draw passes): `Circle[{x,y}]` has radius 1 and `Circle[]` is the unit
circle at the origin (likewise `Disk`). The heads stay inert — the defaults
are interpreted at render time, not by evaluation.

## Feature: multi-function `Plot` with auto palette (2026-06-22)

`Plot` now accepts a list of functions — `Plot[{Sin[1/x], Cos[1/x]}, {x,
-Pi, Pi}]` — drawing each curve on the same axes in a distinct colour.
Colours come from Mathematica's default plot palette (`ColorData[97]`),
hand-tuned so any prefix of the list reads as distinct yet harmonious;
the palette cycles when there are more curves than entries.

In `src/graphics/plot.c`: per-curve sampling and run-splitting are
factored into `sample_lines`; `builtin_plot` detects a `List`-headed first
argument, samples each body under a single iterator shadow, and assembles
the primitive list with a `palette_color(i)` `RGBColor` directive before
each curve's runs. A single curve (the non-list form) is unchanged —
no colour is injected and the `PlotStyle` default still applies.

## Tweak: thicker default plot line (2026-06-22)

The default curve stroke in `Plot`/`Show` is now 1.5x its former width:
`init_state.thickness` in `src/graphics/render.c` changed from a 1-px
hairline to `1.5f / base_zoom` world units, which renders 1.5 px wide at
the home zoom and scales with zoom like `point_size` (instead of staying
pinned to a single pixel). An explicit `Thickness[t]` directive still
overrides it.

## Fix: graphics axis tick labels clear of tick marks (2026-06-22)

`Plot`/`Show` axis tick numbers (rendered with the Hershey single-stroke
vector font) were anchored 6 px from the *axis line* and, because the
Hershey baseline grows upward, actually overprinted the tick marks. In
`draw_axes_labels` (`src/graphics/render.c`) each label is now anchored to
the *outer end of its tick mark* — computed with the same world-space tick
half-length as `draw_axes_lines` — plus a small screen-pixel gap, and the
offset accounts for the glyph cap-height (x-labels drop a full cap below
the tick; y-labels recentre by half a cap). Numbers now sit clearly below
the horizontal axis and left of the vertical axis instead of on the ticks.

## Feature: `NIntegrate` Levin collocation — `Method -> "LevinRule"` (2026-06-22)

`Method -> "LevinRule"` now implements Levin's actual collocation method for
highly oscillatory integrals, replacing the previous behaviour where the name
aliased the between-the-zeros + Wynn engine.

New module `src/numerical_calculus/levincoll.{c,h}`.

### 1-D Levin collocation

For an integrand `f(x)·{Cos[g]|Sin[g]|Exp[I g]}` (slowly-varying amplitude `f`,
rapidly oscillating kernel) the method solves the Levin ODE
`p'(x) + i g'(x) p(x) = f(x)` in a Chebyshev basis at Chebyshev–Gauss–Lobatto
nodes (a small dense complex system, LAPACK `zgetrf` + an in-house complex
triangular solve, with a reciprocal-condition guard), so the integral is the
boundary term `p(b)e^{i g(b)} − p(a)e^{i g(a)}` (`Cos`/`Sin` = real/imaginary
part).  Accuracy **improves** with the oscillation rate:
`NIntegrate[Cos[100000 x], {x,0,1}]` is resolved exactly where Gauss–Kronrod
cannot.  The phase derivative `g'` is taken symbolically (`D`, with the variable
detached so a prior sample binding cannot collapse it); the order is doubled
until two estimates agree.

- **Arbitrary `WorkingPrecision`** via an in-house complex-MPFR (`ncpx`)
  collocation solve (Gaussian elimination with partial pivoting); the nodes,
  Chebyshev basis, matrix and boundary term are all carried at the target
  precision.
- **Automatic** selects Levin for a detected oscillatory kernel that the smooth
  (Gauss–Kronrod / tanh-sinh) rules fail to resolve.
- **Fallbacks**: a non-Levin integrand, a phase singular at an endpoint (e.g.
  `Sin[1/x]` at 0), or weak oscillation (ill-conditioned matrix) falls through
  to the existing oscillatory cascade — no regression.

### Multivariate Levin

Over a constant rectangular box, an oscillatory axis is reduced by collocation
so the inner integral becomes a single linear solve instead of a nested adaptive
quadrature; the remaining axis is integrated by the ordinary 1-D cascade, each
sample costing one inner solve.  When the reduction axis's phase derivative is
independent of the outer variable the collocation matrix is factored once and
re-used across all outer samples (`levin_prepare_machine` / `levin_prepared_solve`),
turning a per-sample `O(n^3)` factorisation into an `O(n^2)` back-substitution —
the difference between a tractable and an intractable reduction.  This makes the
separable oscillatory product

```
NIntegrate[Sin[1/x] Cos[1000 y], {x, 0, 1}, {y, 0, 1}, Method -> "LevinRule"]
```

evaluate in well under a second (previously non-terminating).  Two-dimensional;
higher dimensions fall through to cubature / iterated quadrature.

Tests: `tests/test_levincoll.c` (engine unit tests) and a `test_levin_rule`
group in `tests/test_nint.c` (1-D forms, ultra-high frequency, Automatic,
singular-phase fallback, arbitrary precision, multivariate).  Leak-checked under
valgrind (no Levin frames in lost blocks).

## Feature: `ProductLog` (Lambert W function) (2026-06-22)

New builtin `ProductLog` in `src/special_functions/productlog.c`, the Lambert W
function: `ProductLog[z]` is the principal branch W₀(z) (the solution `w` of
`z = w e^w`), and `ProductLog[k, z]` the k-th branch (`k` any integer, `k = 0`
principal). Attributes `Listable | NumericFunction | Protected | ReadProtected`.

### Numerics

A single complex-MPFR Halley core (built on the shared `ncpx` toolkit in
`numeric_complex.h`) evaluates machine- and arbitrary-precision real and complex
inputs on any branch. The initial approximation is chosen by region — a
branch-point series in `p = Sqrt[2(e z + 1)]` near `z = -1/e` (branches 0 and
-1), the Maclaurin seed `z(1 - z + 3/2 z²)` for the principal branch near 0, and
otherwise the asymptotic `L1 - L2 + L2/L1` (`L1 = Log z + 2πi k`, `L2 = Log L1`)
— then refined by Halley's cubically-convergent iteration to a guard-padded
working precision. A real seed keeps the iteration exactly real, so real-valued
branches return a real leaf with no imaginary noise. Reproduces the documented
reference values (`ProductLog[-1.5] = -0.0327837 + 1.54964 I`,
`Table[ProductLog[k, 2.3], {k, -2, 2}]`, `N[ProductLog[1/3], 100]`, …) and the
inverse identity `W(z) e^W(z) == z` to full precision.

### Exact values, derivative, series

- Exact: `ProductLog[0] = 0`, `ProductLog[E] = 1`, `ProductLog[-1/E] = -1`,
  `ProductLog[-Pi/2] = I Pi/2`, `ProductLog[Infinity] =
  ProductLog[ComplexInfinity] = Infinity`, `ProductLog[k, 0] = -Infinity`
  (`k ≠ 0`). Non-special exact arguments stay symbolic.
- Derivative (`src/calculus/deriv.c`, one- and two-argument arms):
  `D[ProductLog[z], z] = ProductLog[z]/(z (1 + ProductLog[z]))`.
- Series (`src/calculus/series.c`): Taylor at 0
  (`x - x² + 3/2 x³ - 8/3 x⁴ + …`, coefficient `(-k)^(k-1)/k!`); a Puiseux series
  in `Sqrt[x + 1/E]` at the branch point `-1/E`; and the nested-logarithm
  asymptotic expansion at Infinity. `SeriesCoefficient` reduces numeric indices
  and emits the symbolic general term
  `Piecewise[{{(-n)^(n-1)/n!, n ≥ 1}}, 0]`.

Docstring in `src/info.c`; symbol interned in `src/sym_names.c`. Tests in
`tests/test_productlog.c` (exact values, machine/arbitrary-precision real and
complex numerics via the inverse identity, branches, list threading, derivative,
all three series, SeriesCoefficient, diagnostics, attributes). Leak-checked
under valgrind.

## Feature: 2D graphics engine — `Graphics`, `Show`, `Plot` (2026-06-22)

A new `src/graphics/` subsystem brings Mathematica-style 2D graphics:
symbolic primitives composed into `Graphics[...]` objects, rendered
interactively on demand via a Raylib backend.

- **Primitives & style directives** (`Point`, `Line`, `Rectangle`, `Circle`,
  `Disk`, `Polygon`, `Text`, `RGBColor`, `GrayLevel`, `Opacity`,
  `Thickness`, `PointSize`): registered as inert, protected structural
  heads with docstrings — like `List`/`Rule`, they carry no builtin C
  function and simply stay unevaluated until rendered.
- **`Graphics[primitives, opts...]`**: the symbolic container; prints as
  `-Graphics-` (new branch in `print.c`'s `print_standard`, alongside the
  existing `Rational`/`Complex`/`SeriesData` special forms).
- **`Plot[f, {x, xmin, xmax}, opts...]`** (`HoldAll`): adaptively samples
  `f`, auto-displays the result, and returns it as a `Graphics[...]`
  object. The adaptive sampler (`src/graphics/sampling.c`) is a pure
  C-doubles-in/doubles-out algorithm with zero `Expr`/Raylib dependency —
  unit-testable headlessly and intended to be the one sampling
  implementation future plotting functions (`ListPlot`, `ParametricPlot`)
  share, so `PlotPoints`/`MaxRecursion`/`MaxPlotPoints` behave
  consistently everywhere. Implements Mathematica's three-way contract:
  `PlotPoints` (default 25) sets the initial uniform grid, `MaxRecursion`
  (default 6) caps curvature-driven bisection depth per interval, and
  `MaxPlotPoints` (default unbounded) caps the total point budget.
  Singularities/non-real results break the curve into separate
  `Line[...]` segments instead of bridging the gap.
- **`Show[graphics, opts...]`**: merges trailing options into `graphics`
  (later wins) and renders it; the same render entry point (`graphics_show`)
  is what `Plot` calls for its auto-display.
- **Rendering options**: `PlotRange`, `AspectRatio`, `PlotStyle`, `Axes`,
  `AxesLabel`, `PlotLabel`, `Background`, `ImageSize` — read once, in
  `render.c`, by whichever path (`Plot` or `Show`) reaches the renderer.
- **Interactive Raylib window** (`src/graphics/render.c`, built only when
  `USE_GRAPHICS` is on): pan (drag), zoom (scroll), rotate (`Q`/`E`), reset
  (`R`); a `Camera2D` maps math-space (y-up) to Raylib's draw-space
  (y-down) via a single, documented sign flip. Axes are drawn with
  "nice-number" tick spacing and screen-space (zoom-independent) tick
  labels.
- **Vector font** (`src/graphics/hershey_font.c`): a small hand-authored
  single-stroke alphabet (digits, uppercase letters, basic punctuation;
  lowercase folds to uppercase) for axis labels, `AxesLabel`/`PlotLabel`,
  and `Text[...]` — explicitly a placeholder pending proper font
  rendering, not a transcription of the historical Hershey datasets
  despite the similarity in technique.
- **Build**: new `USE_GRAPHICS` flag (default on) in `makefile` and
  `tests/CMakeLists.txt`, autodetecting Raylib via `pkg-config` with a
  graceful degrade to a text placeholder when it's absent — Raylib is a
  system library (like GMP/Readline), never vendored. A
  `MATHILDA_NO_GRAPHICS_WINDOW` environment-variable escape hatch keeps
  the test suite headless even on a machine where Raylib *is* installed.

New tests: `tests/test_graphics_sampling.c` (pure-numeric sampler
behavior: exact-grid with `MaxRecursion -> 0`, curvature-driven
refinement, singularity detection, `MaxPlotPoints` budget, degenerate
ranges) and `tests/test_graphics.c` (symbolic construction: `Graphics`/
`Plot` head shape, option validation, `Show` option merging) — both run
headless regardless of Raylib availability. Full docs in
[`docs/spec/builtins/graphics.md`](../builtins/graphics.md).

## Feature: ten more `Plot`/`Show` options (2026-06-23)

`render.c`'s `GfxOptions`/`gfx_options_parse` now give real rendering effect
to ten more of Mathematica's actual `Options[Plot]` names (previously they
passed through onto the `Graphics[...]` unevaluated like any other option,
but `render.c` ignored them):

- **`AxesOrigin -> {x, y}`** — pins exactly where the axes cross, overriding
  the auto rule (0 if in range, else clamp to the near edge). Factored the
  origin resolution into one `axes_origin()` helper shared by
  `draw_axes_lines`/`draw_axes_labels` so the override and the auto-default
  can't drift apart between the two.
- **`AxesStyle`/`TicksStyle`** — color of the axis lines+ticks vs. the tick
  *label* text, independently (previously both hardcoded `DARKGRAY`).
- **`FrameLabel -> {xlabel, ylabel}`** — the `Frame`-mode equivalent of
  `AxesLabel` (new `draw_frame_label`); `RotateLabel -> False` draws the
  y-label horizontal instead of the default rotated-90° orientation. The
  frame's reserved margin grows automatically when `FrameLabel` is present
  so the extra text line doesn't collide with the tick labels.
- **`PlotRangePadding`** — the auto-fit padding fraction (previously a
  hardcoded 8%) is now a number (both axes), `{xfrac, yfrac}`, or `None`
  for an exact fit with no padding.
- **`GridLines -> Automatic | None | {xspec, yspec}`** and
  **`GridLinesStyle`** — light lines at the same major tick positions as
  `Axes`/`Frame` (`Automatic`), or explicit per-axis numeric positions; new
  `draw_gridlines`, called inside `BeginMode2D` before the axes and curve so
  the curve draws on top.
- **`Prolog`/`Epilog`** — extra primitive(s) drawn via the existing
  `draw_primitive()` recursion, in data coordinates (so they pan/zoom with
  the plot): `Prolog` right after `BeginMode2D` (under the grid/axes/curve),
  `Epilog` right before `EndMode2D` (on top of the curve). Each gets its own
  fresh `DrawState` so style directives inside them don't leak into/out of
  the main curve's state.

**Deferred** (each is a separate rendering subsystem, not a one-option add):
`ColorFunction`, `ColorFunctionScaling`, `MeshFunctions`, `MeshShading`,
`Filling`, `FillingStyle`, `PlotLegends`, `PlotTheme`, `RegionFunction`,
`Exclusions`, `ExclusionsStyle`, `ClippingStyle`, `Method`, `LabelStyle`,
`BaseStyle`. **Inherently inert outside a notebook** (accepted, no-op):
`BaselinePosition`, `AlignmentPoint`, `CoordinatesToolOptions`,
`ImageSizeRaw`, `PlotRegion`, `ImageMargins`, `ImagePadding`,
`WorkingPrecision` (the sampler is machine-precision `double` throughout;
claiming to honor higher precision would be dishonest). All of these already
passed through onto the `Graphics[...]` without erroring before this change
and still do — only the ten options above gained actual rendering behavior.

New tests in `tests/test_graphics.c` (`test_axes_origin_passthrough`,
`test_axes_style_and_ticks_style_passthrough`,
`test_frame_label_and_rotate_label_passthrough`,
`test_plot_range_padding_passthrough`, `test_grid_lines_passthrough`,
`test_prolog_epilog_passthrough`), following the existing
`Cases[...]`-extraction pattern: each confirms `Plot[...]` carries the
option (default-absent and explicit forms) through onto the returned
`Graphics[...]` correctly. Full docs in
[`docs/spec/builtins/graphics.md`](../builtins/graphics.md).

## Feature: named color constants (Red, Blue, Green, ...) (2026-06-23)

`Red`, `Green`, `Blue`, `Black`, `White`, `Gray`, `LightGray`, `Cyan`,
`Magenta`, `Yellow`, `Orange`, `Pink`, `Purple`, `Brown` are now ordinary
protected `OwnValue`s evaluating to `RGBColor[...]`/`GrayLevel[...]`
(`src/graphics/graphics_init.c`'s new `register_color`/`rgb_color`/
`gray_color` helpers), usable anywhere a color literal is — `PlotStyle`,
`Background`, `FrameStyle`, `AxesStyle`, `TicksStyle`, `GridLinesStyle`, or
directly as a primitive-list style directive.

A bare (non-`Held`) `Graphics[...]`'s arguments already evaluate normally,
so a named color resolves there for free. The interesting fix was inside
`Plot` (`HoldAll`): its `split_options` previously copied every
unrecognized option's RHS through onto the `Graphics[...]` result
completely unevaluated (`expr_copy`), so `Plot[f, {x,...}, Epilog -> {Red,
...}]` stored the bare symbol `Red` — it never got the chance to evaluate.
`split_options` now evaluates each such RHS once before storing it
(mirroring how `PlotPoints`/`MaxRecursion`/`Mesh` were already evaluated
individually), exactly matching what a non-`Held` `Graphics[]` would do for
the same value.

**Side fix**: this surfaced a real, separate bug while testing — `?Red`
showed `Information[RGBColor[1, 0, 0]]` instead of Red's docstring, because
`Information` had no `Hold` attribute and was evaluating its argument
(Red's *value*) before inspecting it. Gave `Information` `HoldFirst`
(`src/core.c`; matches real Mathematica's actual attributes), so `?x`/
`Information[x]` always inspects the symbol itself — this was already
silently wrong for any user symbol with an eager `OwnValue` (e.g. `x = 5;
?x` previously tried to describe `5`, not `x`), not just the new color
constants. Verified no regression across every test file that exercises
`Information[...]` (`test_core`, `test_eigen`, `test_iter`, `test_matrank`,
`test_nullspace`, `test_random`, `test_graphics`).

New test `test_named_color_constants_resolve` in `tests/test_graphics.c`.
**Note for future test-writing**: `assert_eval_eq` (in `tests/test_utils.h`)
uses the libc `assert()` internally, which `-DNDEBUG` (this project's
Release CMake build) silently turns into a no-op — a mismatch prints a
`FAIL:` line to stderr but does **not** fail the test or its exit code. The
existing `tests/test_prolog_epilog_passthrough` had a now-stale assertion
(written before this change) that was silently passing despite printing
`FAIL:`; fixed to expect the now-evaluated `RGBColor[1, 0, 0]`. Always grep
a test binary's output for `FAIL` rather than trusting its own "all passed"
banner or exit code. (Three pre-existing, unrelated `FAIL:` lines were also
found this way in `core_tests`' `GCD[Rational[...], ...]` cases — not
caused by this change, not fixed here, flagged for separate follow-up.)

## Fix: symbolic-but-numeric coordinates (`Pi/2`, `Sqrt[2]`, ...) in graphics primitives (2026-06-23)

Found while testing `Epilog`: `Plot[..., Epilog -> {Red, Point[{Pi/2,
0.8}]}]` silently drew nothing. Root cause was general, not `Epilog`-
specific — `render.c`'s coordinate extraction (`expr_to_d`, used by every
primitive's draw call and by `compute_bbox`/`gather_ys`) only recognized
already-literal numeric atoms (`EXPR_INTEGER`/`EXPR_REAL`/`EXPR_BIGINT`/
`EXPR_MPFR`/exact-integer `Rational[n,d]`). Plain `evaluate()` never
reduces `Pi` to a number (only `N[]` does), so `Point[{Pi/2, 0.8}]`'s `x`
stayed the symbolic expression `1/2 Pi` all the way to the renderer, which
then couldn't parse it as a point at all — this would have broken *any*
hand-written `Graphics[{Point[{Pi/2, 1}]}]` too, not just inside `Plot`.

`expr_to_d` now falls back to numericizing via `N[]` (same approach as
`plot.c`'s `numericize_bound`, for the same reason) when a coordinate isn't
already a plain literal. The sampler's own curve points are real literals
already (never hit this fallback — no hot-path cost for the large-data
path); only hand-authored primitive coordinates (`Epilog`/`Prolog`/literal
`Graphics[]`) pay the extra `evaluate()` call, and those lists are
typically small. Verified directly (temporary `fprintf` at the draw site,
since this is a render-time/pixel-placement bug that a headless test can't
meaningfully exercise) that `Point[{Pi/2, 0.8}]` now resolves to `x=1.5708
y=0.8` every frame; removed before committing.

## Feature: `ColorFunction`, `Filling`, `PlotLegends`, `RegionFunction`, `Exclusions`, `LabelStyle`, `Hue` (2026-06-23)

The remaining *visual* `Plot` options, on top of the structural/layout
batch from the previous entry:

- **`Hue[h]`/`Hue[h,s,b]`/`Hue[h,s,b,a]`** (`render.c`): standard HSB→RGB,
  prerequisite for `ColorFunction` (which commonly returns `Hue[...]` in
  real Mathematica usage). Recognized everywhere `RGBColor`/`GrayLevel`
  already were — refactored all seven call sites (the six `gfx_options_parse`
  color options plus `draw_primitive`'s directive dispatch) through one new
  `resolve_color()` instead of each repeating the same two-line
  `RGBColor`/`GrayLevel` check, so a future color form only needs touching
  in one place.
- **`ColorFunction -> f`** (+ **`ColorFunctionScaling`**, default `True`):
  implemented at *construction* time in `plot.c` — instead of one
  `Line[allPoints]` per sampled run, `sample_lines` emits a color directive
  (`f` evaluated at the segment's midpoint, 2-arg `f[xscaled,yscaled]`
  tried first then 1-arg `f[xscaled]`, mirroring `RegionFunction`'s same
  fallback) + a 2-point `Line[]` per segment. `"Rainbow"` is a built-in
  `Hue` sweep over scaled x. Zero `render.c` changes needed — it already
  just walks a primitive list of interleaved directives and primitives.
- **`Filling -> Axis | Bottom | Top | n`** (+ **`FillingStyle`**): one
  `Polygon[]` per run, tracing the curve forward then the baseline
  backward, drawn before the run's own `Line[]`/colored segments. Default
  `FillingStyle` brackets the fill in `Opacity[0.3]` then restores full
  opacity and the curve's actual color (`single_color`, threaded through
  `split_options` → `build_plot_primitives` → `sample_lines`) so the
  outline itself isn't left translucent. Multi-curve curve-to-curve
  filling (`Filling -> {1 -> {2}}`) is **not** included.
- **`PlotLegends -> Automatic | "Expressions" | {labels...}`**: a new
  internal `$PlotLegendData[{color1,label1}, ...]` metadata primitive
  (same pattern as `$PlotResample`), built by `plot.c`'s
  `build_legend_meta` and read by a new `draw_legend()` in `render.c` —
  screen-space, top-right but *below* the toolbar (which already owns that
  corner) so they don't collide. `"Expressions"` labels come from
  `expr_to_string` on each curve's actual body.
- **`RegionFunction -> f`**: `f[x,y]` (2-arg tried first, then 1-arg `f[x]`)
  evaluated per sample in `plot_eval_fn`; a non-`True` region is rejected
  exactly like a singularity, reusing the sampler's existing gap/
  `break_before` machinery with zero `sampling.c` changes.
- **`Exclusions -> {x1, ...} | {x == a, ...} | None`**: `parse_exclusions`
  + `split_at_exclusions` split `[xmin,xmax]` into sub-ranges at each
  given x (with a tiny epsilon gap so the curve doesn't visually rejoin),
  each sampled independently via the same `plot_sample_adaptive` call
  multi-curve `Plot` already makes repeatedly — concatenated with a forced
  `break_before` at each sub-range boundary.
- **`LabelStyle -> color`**: `render.c`'s `gfx_options_parse` now runs a
  quick pre-scan for `LabelStyle` that seeds `axes_color`/`ticks_color`/
  `frame_color` *before* the main option pass, so any of `AxesStyle`/
  `TicksStyle`/`FrameStyle` the caller also gives still overrides it
  regardless of each option's position in the argument list.

`$PlotResample`'s metadata shape changed from 6 flat positional args to
`[var, {bodies}, {plotPoints, maxRecursion, maxPlotPoints, mesh,
regionFunction, exclusions, colorFunction, colorFunctionScaling, filling,
fillingStyle}]` — a single options `List[...]` as the 3rd arg — so
`ColorFunction`/`Filling`/`RegionFunction`/`Exclusions` survive re-sampling
on zoom instead of silently vanishing once the view changes.

**Still deferred** (re-confirmed, now alongside the previous batch's list):
`MeshFunctions`/`MeshShading` (finer control over the already-working
`Mesh`), curve-to-curve `Filling -> {1 -> {2}}`, `Method`, `ClippingStyle`,
`PlotTheme`, `BaseStyle`.

New tests in `tests/test_graphics.c`: `test_hue_color_directive`,
`test_color_function_builds_per_segment_colors`,
`test_filling_builds_polygon`, `test_plot_legends_metadata`,
`test_region_function_and_exclusions_split_domain`,
`test_label_style_passthrough` — content-shape assertions (e.g. "the
primitive count grows past 1 with `ColorFunction` set", "the first
primitive is `Opacity` then `Polygon` with `Filling` set") rather than
pure passthrough, since these options change *what* gets built. Every
assertion was hand-verified against the live binary's actual output before
being written into the test (not just reasoned about), after the
`assert_eval_eq`/`NDEBUG` silent-failure trap from the previous entry.
