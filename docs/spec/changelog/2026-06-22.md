# Changelog: week of 2026-06-22 (Mon) – 2026-06-28 (Sun)

Feature additions and fixes recorded during this week.

## Feature: `LegendreP` — Legendre polynomials and associated Legendre functions (2026-06-22)

New builtin `LegendreP` (`src/special_functions/legendre.{c,h}`):

- `LegendreP[n, x]` — Legendre polynomial / function `P_n(x)`.
- `LegendreP[n, m, x]` — associated Legendre function `P_n^m(x)` (type 1).
- `LegendreP[n, m, a, x]` — Legendre function of type `a` (`a ∈ {1, 2, 3}`,
  default `1`).

Evaluation layers:

- **Exact integer order** builds the explicit polynomial in `x` with exact
  rational coefficients from the three-term recurrence; negative orders fold via
  `P_{-1-n} = P_n` (e.g. `LegendreP[10, x]`). `LegendreP[n, 1] = 1` for any
  order.
- **Numeric** (non-integer order with an inexact argument) sums the Gauss
  series `P_n(x) = 2F1(-n, n+1; 1; (1-x)/2)` on the shared `ncpx` complex-MPFR
  toolkit, real or complex, at machine or arbitrary precision with precision
  tracking (`LegendreP[2.5, 2]`, `LegendreP[3/2 + I, 1.5 - I]`,
  `N[LegendreP[3/2, 2], 50]`, `LegendreP[3/2, 2\`300]`). An exact non-integer
  order with an exact argument stays symbolic and numericalizes only under `N`.
- **Associated, type 1** (integer `n`, `m ≥ 0`) uses the Rodrigues derivative
  form `(-1)^m (1-x²)^(m/2) d^m/dx^m P_n(x)`; `0` when `m > |n'|`
  (`LegendreP[2, 1, x] = -3 x Sqrt[1 - x²]`, `LegendreP[2, 2, 2] = -9`).
- **Types 2 and 3** multiply the terminating regularized Gauss polynomial
  `2F1Reg(-n, n+1, 1-m, (1-x)/2)` by `(1+x)^(m/2) (1-x)^(-m/2)` (type 2) or
  `(1+x)^(m/2) (-1+x)^(-m/2)` (type 3).

Registered in `src/sym_names.{c,h}` (`SYM_LegendreP`), `src/core.c`
(`legendre_init`), and `src/info.c` (docstring). Attributes: `Listable`,
`NumericFunction`, `Protected`. Extensive tests in `tests/test_legendre.c`;
no new memory leaks (valgrind clean against baseline).

Left symbolic (deferred): non-integer associated/type forms, negative `m`,
symbolic `Series` / `SeriesCoefficient`, `D[]` rules, and analytic continuation
of the numeric series for `|(1-x)/2| ≥ 1`.

## Feature: imaginary unit `I` documented and protected (2026-06-22)

`I` (the imaginary unit, `Complex[0, 1]`) now carries:
- an Information docstring (`?I`), registered in `src/info.c`;
- the `Protected` attribute (`src/attr.c`), so `Attributes[I] = {Protected}`
  and `I` can no longer be reassigned.

The `SYM_I` internal symbol was already interned in `src/sym_names.c`.

Also gave `Information` the `HoldAll` attribute (matching the Wolfram Language)
so `?sym` works for symbols that carry an OwnValue. Previously `?I` evaluated
its argument to `Complex[0, 1]` before the builtin could read the symbol name,
leaving the docstring (and that of any other OwnValue-bearing symbol)
unreachable.

## Feature: `NIntegrate` Levin collocation — `Method -> "LevinRule"` (2026-06-22)

`Method -> "LevinRule"` now implements Levin's actual collocation method for
highly oscillatory integrals, replacing the previous behaviour where the name
aliased the between-the-zeros + Wynn engine.

New module `src/numerical_calculus/levincoll.{c,h}`.

### 1-D Levin collocation

For an integrand `f(x)·{Cos[g]|Sin[g]|Exp[I g]}` (slowly-varying amplitude `f`,
rapidly oscillating kernel) the method solves the Levin ODE
`p'(x) + i g'(x) p(x) = f(x)` in a Chebyshev basis at Chebyshev–Gauss–Lobatto
nodes (a small dense complex system, LAPACK `zgetrf` + an in-house complex
triangular solve, with a reciprocal-condition guard), so the integral is the
boundary term `p(b)e^{i g(b)} − p(a)e^{i g(a)}` (`Cos`/`Sin` = real/imaginary
part).  Accuracy **improves** with the oscillation rate:
`NIntegrate[Cos[100000 x], {x,0,1}]` is resolved exactly where Gauss–Kronrod
cannot.  The phase derivative `g'` is taken symbolically (`D`, with the variable
detached so a prior sample binding cannot collapse it); the order is doubled
until two estimates agree.

- **Arbitrary `WorkingPrecision`** via an in-house complex-MPFR (`ncpx`)
  collocation solve (Gaussian elimination with partial pivoting); the nodes,
  Chebyshev basis, matrix and boundary term are all carried at the target
  precision.
- **Automatic** selects Levin for a detected oscillatory kernel that the smooth
  (Gauss–Kronrod / tanh-sinh) rules fail to resolve.
- **Fallbacks**: a non-Levin integrand, a phase singular at an endpoint (e.g.
  `Sin[1/x]` at 0), or weak oscillation (ill-conditioned matrix) falls through
  to the existing oscillatory cascade — no regression.

### Multivariate Levin

Over a constant rectangular box, an oscillatory axis is reduced by collocation
so the inner integral becomes a single linear solve instead of a nested adaptive
quadrature; the remaining axis is integrated by the ordinary 1-D cascade, each
sample costing one inner solve.  When the reduction axis's phase derivative is
independent of the outer variable the collocation matrix is factored once and
re-used across all outer samples (`levin_prepare_machine` / `levin_prepared_solve`),
turning a per-sample `O(n^3)` factorisation into an `O(n^2)` back-substitution —
the difference between a tractable and an intractable reduction.  This makes the
separable oscillatory product

```
NIntegrate[Sin[1/x] Cos[1000 y], {x, 0, 1}, {y, 0, 1}, Method -> "LevinRule"]
```

evaluate in well under a second (previously non-terminating).  Two-dimensional;
higher dimensions fall through to cubature / iterated quadrature.

Tests: `tests/test_levincoll.c` (engine unit tests) and a `test_levin_rule`
group in `tests/test_nint.c` (1-D forms, ultra-high frequency, Automatic,
singular-phase fallback, arbitrary precision, multivariate).  Leak-checked under
valgrind (no Levin frames in lost blocks).

## Feature: `ProductLog` (Lambert W function) (2026-06-22)

New builtin `ProductLog` in `src/special_functions/productlog.c`, the Lambert W
function: `ProductLog[z]` is the principal branch W₀(z) (the solution `w` of
`z = w e^w`), and `ProductLog[k, z]` the k-th branch (`k` any integer, `k = 0`
principal). Attributes `Listable | NumericFunction | Protected | ReadProtected`.

### Numerics

A single complex-MPFR Halley core (built on the shared `ncpx` toolkit in
`numeric_complex.h`) evaluates machine- and arbitrary-precision real and complex
inputs on any branch. The initial approximation is chosen by region — a
branch-point series in `p = Sqrt[2(e z + 1)]` near `z = -1/e` (branches 0 and
-1), the Maclaurin seed `z(1 - z + 3/2 z²)` for the principal branch near 0, and
otherwise the asymptotic `L1 - L2 + L2/L1` (`L1 = Log z + 2πi k`, `L2 = Log L1`)
— then refined by Halley's cubically-convergent iteration to a guard-padded
working precision. A real seed keeps the iteration exactly real, so real-valued
branches return a real leaf with no imaginary noise. Reproduces the documented
reference values (`ProductLog[-1.5] = -0.0327837 + 1.54964 I`,
`Table[ProductLog[k, 2.3], {k, -2, 2}]`, `N[ProductLog[1/3], 100]`, …) and the
inverse identity `W(z) e^W(z) == z` to full precision.

### Exact values, derivative, series

- Exact: `ProductLog[0] = 0`, `ProductLog[E] = 1`, `ProductLog[-1/E] = -1`,
  `ProductLog[-Pi/2] = I Pi/2`, `ProductLog[Infinity] =
  ProductLog[ComplexInfinity] = Infinity`, `ProductLog[k, 0] = -Infinity`
  (`k ≠ 0`). Non-special exact arguments stay symbolic.
- Derivative (`src/calculus/deriv.c`, one- and two-argument arms):
  `D[ProductLog[z], z] = ProductLog[z]/(z (1 + ProductLog[z]))`.
- Series (`src/calculus/series.c`): Taylor at 0
  (`x - x² + 3/2 x³ - 8/3 x⁴ + …`, coefficient `(-k)^(k-1)/k!`); a Puiseux series
  in `Sqrt[x + 1/E]` at the branch point `-1/E`; and the nested-logarithm
  asymptotic expansion at Infinity. `SeriesCoefficient` reduces numeric indices
  and emits the symbolic general term
  `Piecewise[{{(-n)^(n-1)/n!, n ≥ 1}}, 0]`.

Docstring in `src/info.c`; symbol interned in `src/sym_names.c`. Tests in
`tests/test_productlog.c` (exact values, machine/arbitrary-precision real and
complex numerics via the inverse identity, branches, list threading, derivative,
all three series, SeriesCoefficient, diagnostics, attributes). Leak-checked
under valgrind.

## Feature: `DigitSum` — sum of digits of an integer (2026-06-22)

New builtin `DigitSum` (`src/int.{c,h}`):

- `DigitSum[n]` — sum of the base-10 digits of `|n|`.
- `DigitSum[n, b]` — sum of the base-`b` digits of `|n|` (`b >= 2`).

Sign of `n` is discarded; `DigitSum[0]` returns `0`. Both machine integers and
arbitrary-precision bignums are supported via GMP (fast `mpz_tdiv_q_ui` path
for `ulong` bases; `mpz_tdiv_qr` slow path for bignum bases).

Attributes: `Listable | NumericFunction | Protected`. Docstring in `src/info.c`;
symbol interned in `src/sym_names.c`. Tests in `tests/test_digit_sum.c` (basic
decimal, zero, negatives, two-arg base form, cross-check invariant vs
`Total[IntegerDigits[n]]`, bignum inputs, arity/type diagnostics, symbolic
silence, `Listable` threading, attributes, docstring, stress loop).
