# Changelog: week of 2026-06-15 (Mon) – 2026-06-21 (Sun)

Feature additions and fixes recorded during this week.

## Fix: two-argument `Zeta` numeric value for negative `a` (2026-06-19)

`Zeta[3, -1.5]` returned `0.118102` (the principal-branch `HurwitzZeta` value)
instead of `16.7107`. Two-argument `Zeta` uses the *symmetric* power
`((a+k)^2)^(-s/2)` (= `|a+k|^-s` on the real axis), so for `Re(a) < 0` the head
terms whose base lies in the left half-plane must **add** rather than cancel —
but the numeric kernel `zeta_hurwitz_cx` (`src/special_functions/zeta.c`) summed
the principal branch `(a+k)^-s`, where those terms cancel pairwise.

Fix: the Euler–Maclaurin head sum now uses the symmetric power
`((a+k)^2)^(-s/2)`, and `N` is forced large enough that the tail base `a+N` lands
comfortably in the right half-plane (where symmetric and principal coincide).
Positive-`a` values and one-argument `Zeta[s]` are unchanged; `HurwitzZeta`
(which genuinely uses the principal branch) is a separate kernel and keeps its
cancelling value (`HurwitzZeta[3, -1.5] = 0.118102`). Now
`Zeta[3, -1.5] = 16.7107`, `Zeta[3, -3.5] = 16.798`. Test in `tests/test_zeta.c`.

## Fix: derivatives of `HarmonicNumber` and symbolic-order `HurwitzZeta` (2026-06-19)

Added missing derivative rules (`src/calculus/deriv.c`):

- **`HarmonicNumber`** now differentiates w.r.t. its (continuous) index:
  `D[HarmonicNumber[n], n] = Pi^2/6 - HarmonicNumber[n, 2]` and
  `D[HarmonicNumber[n, r], n] = r (Zeta[r+1] - HarmonicNumber[n, r+1])`
  (the `r`-derivative stays the generic `Derivative[0,1][HarmonicNumber][n,r]`).
  Previously these returned an inert `Derivative[…][HarmonicNumber][…]`.
- **`HurwitzZeta` symbolic-order derivative**: `compute_deriv_symbolic_order`
  gained a closed form for the second argument,
  `D[HurwitzZeta[s, a], {a, n}] = (-1)^n Pochhammer[s, n] HurwitzZeta[n+s, a]`,
  matching the per-step integer-order results (which already worked). Previously
  the symbolic order left `D[HurwitzZeta[s,a],{a,n}]` unevaluated.

Tests in `tests/test_harmonicnumber.c` and `tests/test_hurwitzzeta.c`.

## Fix: `LerchPhi` exact reductions at integer `a` (2026-06-19)

Several `LerchPhi[z, s, a]` special values were missing or wrong
(`src/special_functions/lerchphi.c`):

- **Non-positive integer `a`** now reduces onto `PolyLog` (Wolfram's convention
  drops the singular `k = -a` term): new `lp_nonpos_int_a` gives
  `LerchPhi[z, s, -m] = z^m PolyLog[s, z] + Σ_{j=1}^{m} z^(m-j) (-j)^(-s)`. In
  particular `LerchPhi[z, s, 0] = PolyLog[s, z]` (so
  `PolyLog[s, z] == LerchPhi[z, s, 0]` is now `True`), `LerchPhi[0, s, 0] = 0`,
  `LerchPhi[0, 0, 0] = 0`, and `LerchPhi[1/2, 1, 0] = Log[2]`. Previously these
  stayed unevaluated or hit a `0^0` / `0^(-s)` indeterminate.
- **`z = -1` with positive-integer `a`** now routes through the `PolyLog`
  reduction instead of the two-`Zeta` `z = -1` form (which is indeterminate at
  integer `s`): `LerchPhi[-1, 1, 1] = Log[2]`, and the divergent table case
  `Table[LerchPhi[z, 1, 1], {z, -1, 2}]` returns
  `{Log[2], 1, ComplexInfinity, -I Pi/2}`.

The dispatch in `lerchphi_single` was reordered so that the non-positive-integer
`a` reduction precedes the `z` special cases (so `a = 0` wins over `z = 0`), and
the positive-integer `a` reduction precedes the `z = -1` case, while `z = 1`
keeps the clean `Zeta[s, a]` form. Docs: [`special-functions.md`](../builtins/special-functions.md).

## Feature: `Divisible` — integer/numeric divisibility test (2026-06-19)

New builtin `Divisible` in `src/numbertheory.c` (registered via
`numbertheory_init()`, symbol in `sym_names.c`, docstring in `info.c`).
Attributes: `Listable`, `Protected`.

- `Divisible[n, m]` → `True` if `n` is an integer multiple of `m`, else
  `False`; effectively `Mod[n, m] == 0`, returning `False` unless `n` and `m`
  are manifestly divisible.

Two evaluation paths:
- **Integer fast path** — when both arguments are integer-like (machine or
  bigint), divisibility is `mpz_divisible_p`, exact at arbitrary precision
  (`Divisible[10^3000 + 1, 16001]` → `True`). GMP convention for `0`:
  divisible by `0` iff `n == 0`; sign is ignored.
- **General path** — forms and evaluates the quotient `n m^-1`, reporting
  `True` iff it collapses to an integer or Gaussian integer. Covers Gaussian
  integers (`Divisible[3 + I, 1 - I]` → `True`), rationals
  (`Divisible[3/2, 1/2]` → `True`), and exact numeric quantities
  (`Divisible[2 Pi, Pi/2]` → `True`, `Divisible[Sqrt[6], Sqrt[2]]` → `False`).
  A non-integer quotient is `False` only when both arguments are concrete
  numeric quantities; symbolic, non-numeric arguments (`Divisible[x, 2]`) are
  left unevaluated.

`Listable`, so list arguments thread element-wise
(`Divisible[{1,2,3,4,5,6}, 2]` → `{False, True, False, True, False, True}`).
Wrong arg counts emit `Divisible::argm` (too few) or `Divisible::argt` (too
many) and leave the call unevaluated.

Tests: `tests/test_divisible.c` (13 groups: machine ints, negatives, zero
edges, bignums, Gaussian ints, rationals, symbolic-numeric, numeric
quantities, listability incl. nested, symbolic-unevaluated, arg errors,
consistency with `IntegerQ`/`GCD`/`Mod`, attributes). Valgrind-clean (no
Mathilda-source frames in leak stacks).

## Feature: `CoprimeQ` — pairwise relative-primality test (2026-06-19)

New builtin `CoprimeQ` in `src/numbertheory.c` (registered via
`numbertheory_init()`, symbol in `sym_names.c`, docstring in `info.c`).
Attributes: `Listable`, `Orderless`, `Protected`.

- `CoprimeQ[n1, n2, ...]` → `True` if the arguments are *pairwise* relatively
  prime, else `False`. A pair is coprime when `GCD == 1`.

Two evaluation paths:
- **Integer path** — every pair is tested with `mpz_gcd`, exact for machine
  ints and bigints (`CoprimeQ[2^100 - 1, 3^100 - 1]` → `False`,
  `CoprimeQ[2^127 - 1, 2^61 - 1]` → `True`). Sign is ignored; since
  `GCD(0, n) = |n|`, `CoprimeQ[0, 1]` → `True` but `CoprimeQ[0, 5]` → `False`.
  Pairwise: `CoprimeQ[6, 35, 143]` → `True`, `CoprimeQ[2, 3, 4]` → `False`.
- **Gaussian path** — selected by `GaussianIntegers -> True` or by any argument
  being an exact Gaussian integer. Coprimality in `Z[i]` is decided by a
  Gaussian Euclidean algorithm with round-to-nearest complex division; a pair
  is coprime iff the Gaussian GCD has norm 1. `CoprimeQ[5 + I, 1 - I]` →
  `False` (common factor `1 + I`); `CoprimeQ[2, 5, GaussianIntegers -> True]` →
  `True`, `CoprimeQ[2, 10, GaussianIntegers -> True]` → `False`.

`Orderless`, so argument order is irrelevant and the `GaussianIntegers` option
may sit at any position. `Listable`, so list arguments thread element-wise
(`CoprimeQ[{1,2,3,4,5}, 6]` → `{True, False, False, False, True}`). As a `*Q`
predicate it always returns a Boolean: `CoprimeQ[]` → `False`, `CoprimeQ[n]` →
`True`, and any non-coprime-integer/Gaussian argument (rationals, reals,
symbols, malformed options) → `False` (`CoprimeQ[a, b]` → `False`).

Tests: `tests/test_coprimeq.c` (12 groups: pairs, negatives, zero edges,
multi-arg pairwise, bignums, auto Gaussian, `GaussianIntegers` option,
listability incl. nested, arity, symbolic→False, consistency with `GCD`,
attributes). Valgrind-clean (no Mathilda-source frames in leak stacks).

## Feature: `Rescale` — linear interval rescaling (2026-06-19)

New builtin `Rescale` in `src/list.c` (registered via `list_init()`, symbol in
`sym_names.c`, docstring in `info.c`). Attributes: `NumericFunction`,
`Protected`.

- `Rescale[x, {min, max}]` → `(x - min)/(max - min)` (runs `x` from 0 to 1).
- `Rescale[x, {min, max}, {ymin, ymax}]` → `ymin + (ymax - ymin)(x - min)/(max - min)`.
- `Rescale[list]` → `Rescale[list, {Min[list], Max[list]}]`.

Implementation defers all arithmetic to the core evaluator, so it works on
exact integers/rationals, reals (output precision tracks the input), bignums,
complex numbers, and symbolic quantities, and yields exact results for exact
input (`Rescale[3, {-9, 7}, {11, 28}]` → `95/4`). Threads element-wise over a
`List` first argument, recursively (maps over matrices). Endpoints rescale to
exactly 0/1 (and `ymin`/`ymax`); a degenerate target `{c, c}` collapses to `c`.
Wrong arg counts emit `Rescale::argb` and stay unevaluated; malformed ranges
are left unevaluated. Symbolic results are kept factored (e.g.
`Rescale[x, {a, b}, {c, d}]` → `c + ((-c + d)(-a + x))/(-a + b)`), verified
equal to the closed forms via `Simplify`.

Tests: `tests/test_rescale.c` (14 groups: numeric two/three-arg, endpoints,
real-precision, complex, high-precision `N`, degenerate target, symbolic forms
+ equivalence, one-arg list, threading incl. nested, arg/range errors,
attributes). Valgrind-clean (no Mathilda-source frames in leak stacks).

## Feature: `LerchPhi` — the Lerch transcendent (2026-06-19)

New builtin `LerchPhi[z, s, a]` = Σ_{k≥0} zᵏ/(k+a)ˢ in
`src/special_functions/lerchphi.c`, registered via `lerchphi_init()` in
`core_init()`. Attributes: `Listable`, `NumericFunction`, `Protected`. Symbols
`SYM_LerchPhi`, `SYM_DoublyInfinite`, `SYM_IncludeSingularTerm` added in
`sym_names.{c,h}`; docstring in `info.c`; derivative rules in
`src/calculus/deriv.c`.

`LerchPhi` is the common generalization of `Zeta`, `HurwitzZeta` and `PolyLog`
(`LerchPhi[1, s, a] = Zeta[s, a]`, `z LerchPhi[z, s, 1] = PolyLog[s, z]`).

- **Exact reductions** (symbolic, so they also numericalize via the reused
  kernels): `LerchPhi[0, s, a] = a^(-s)`; `LerchPhi[z, 0, a] = 1/(1-z)`;
  `LerchPhi[1, s, a] = Zeta[s, a]`; `LerchPhi[z, s, 1] = PolyLog[s, z]/z`;
  `LerchPhi[-1, s, 1/2] = 2^(-s)(Zeta[s, 1/4] - Zeta[s, 3/4])`; a positive
  integer `a` shifts to the `PolyLog` series (`LerchPhi[z, s, 2] =
  (PolyLog[s, z] - z)/z²`); a negative integer `s = -n` gives a rational
  function of `z` via the Euler operator `(z d/dz + a)^n[1/(1-z)]`
  (`LerchPhi[2, -1, a] = 2 - a`).
- **Options**: `DoublyInfinite -> True` sums k from −∞ to ∞ as
  `Φ(z,s,a) + z⁻¹ Φ(1/z, s, 1-a)` (`LerchPhi[1, 2, 0.5, DoublyInfinite -> True]
  = Pi² = 9.8696`); `IncludeSingularTerm -> True` keeps the `k + a = 0` term,
  giving `ComplexInfinity` at a non-positive integer `a`. Argument / option
  diagnostics `LerchPhi::argrx` and `LerchPhi::nonopt`.
- **Numerics**: a complex-MPFR power series (file-local `lcx` toolkit) for
  `|z| < 1`, machine and arbitrary precision, real and complex
  (`LerchPhi[0.5, 3, 2.5] = 0.0794983`,
  `LerchPhi[0.3 + 0.2 I, 2, 1.5] = 0.495505 + 0.0444653 I`,
  `N[LerchPhi[1/2, 2, 5/2], 30] = 0.219693113434910235649039949138…`). `|z| > 1`
  has no analytic continuation here and stays symbolic.
- **Derivatives**: `D[LerchPhi[z,s,a], z] = (LerchPhi[z,-1+s,a]
  - a LerchPhi[z,s,a])/z`, `D[LerchPhi[z,s,a], a] = -s LerchPhi[z,1+s,a]`,
  `s`-derivative is the generic `Derivative[0,1,0][LerchPhi][z,s,a]`.

Tests in `tests/test_lerchphi.c` (reductions, machine/MPFR real & complex
numerics cross-checked against `PolyLog`/`Zeta`, options, derivatives, Listable,
attributes, diagnostics). No new memory leaks (valgrind clean against the macOS
baseline).

Also added the half-integer closed form `Zeta[s, 1/2] = (2^s - 1) Zeta[s]` to
two-argument `Zeta` (`src/special_functions/zeta.c`, exact `s` only), matching
`HurwitzZeta[s, 1/2]`. So `Zeta[2, 1/2] = Pi^2/2`, `Zeta[3, 1/2] = 7 Zeta[3]`,
and `LerchPhi[1, 2, 1/2]` now reduces to `Pi^2/2`. Test in `tests/test_zeta.c`.

## Feature: `HurwitzZeta` — the Hurwitz zeta function (2026-06-19)

New builtin `HurwitzZeta[s, a]` = Σ_{k≥0} (k+a)⁻ˢ in
`src/special_functions/hurwitzzeta.c`, registered via `hurwitzzeta_init()` in
`core_init()`. Attributes: `Listable`, `NumericFunction`, `Protected`. Symbol
`SYM_HurwitzZeta` added in `sym_names.{c,h}`; docstring in `info.c`.

Unlike `Zeta[s, a]`, `HurwitzZeta` sums the principal-branch power `(k+a)⁻ˢ`, so
the two disagree for non-positive real `a` (`HurwitzZeta[3, -3.5] = 0.0307784`
vs `Zeta[3, -3.5] = 16.798`) and `HurwitzZeta` has poles at `a = 0, -1, -2, …`.

- **Exact reductions**: `HurwitzZeta[s, 1] = Zeta[s]`,
  `HurwitzZeta[s, 1/2] = (2^s − 1) Zeta[s]`, and a positive integer `a = m`
  reduces to `Zeta[s] − Σ_{k=1}^{m-1} k⁻ˢ` (`HurwitzZeta[3, 2] = -1 + Zeta[3]`,
  `HurwitzZeta[2, 5] = -205/144 + Pi^2/6`).
- **Poles / special points**: `HurwitzZeta[1, a] = ComplexInfinity`;
  at a non-positive integer `a`, positive-integer `s` gives `ComplexInfinity`
  (`HurwitzZeta[2, -2]`), non-positive-integer `s` gives the Bernoulli-polynomial
  value (`HurwitzZeta[0, 0] = 1/2`, `HurwitzZeta[-1, 0] = -1/12`).
- **Numerics**: real/complex, machine/MPFR via an Euler–Maclaurin complex kernel
  (DLMF 25.11.5) with optimal-truncation correction series; precision tracks the
  input (`N[HurwitzZeta[1/3, 8/7], 50]`, `HurwitzZeta[2.3, 8 + I]`).
- **Derivative** (`src/calculus/deriv.c`): `D[HurwitzZeta[s, a], a] =
  -s HurwitzZeta[1+s, a]`; the `s`-derivative is the generic
  `Derivative[1,0][HurwitzZeta][s, a]`.

New test suite `tests/test_hurwitzzeta.c` (14 groups: exact reductions, poles,
machine/complex/MPFR numerics, principal-branch behaviour, derivatives, Listable,
attributes, argument counting). Valgrind-clean (no leaks beyond the macOS dyld
baseline).

## Feature: `Sum`Rational` — infinite rational-function summation (2026-06-19)

New `Sum` stage `Sum`Rational` (`Method -> "Rational"`) closes
`Sum[p(i)/q(i), {i, imin, Infinity}]` for a rational summand with concrete
integer `imin` and `deg q >= deg p + 2`. New module `src/sum/sum_rational.c`,
`sum_rational_init()` wired into `sum_init()` and dispatched in the `Automatic`
cascade ahead of `Sum`Hypergeometric`. Attributes: `Protected`,
`ReadProtected`.

Decompose into linear partial fractions and sum each pole term with the master
identity: `k >= 2` → `Zeta[k, imin - rho]`; `k == 1` →
`-c (PolyGamma[0, imin - rho] + EulerGamma)` (residues sum to zero, so the
divergent simple-pole pieces combine to a convergent value).

- **Rational poles** (`Apart[f, i]` over Q): `Sum[1/i^2] = Pi^2/6`,
  `Sum[1/i^3] = Zeta[3]`, `Sum[1/i^4] = Pi^4/90`, `Sum[1/(i(i+2))] = 3/4`.
- **Complex-conjugate quadratic** (`disc < 0`, handled over Q): the constant-
  numerator part collapses to `Coth`, the linear-numerator part stays a
  conjugate digamma sum. `Sum[1/(i^2(i^2+1))] = 1/6 (3 + Pi^2 - 3 Pi Coth[Pi])`,
  `Sum[1/(i^2+1)] = 1/2 (-1 + Pi Coth[Pi])`. Generalises to `beta != 1`
  (`Coth[2 Pi]`) and shifted lower bounds (`t = imin - alpha`, with the finite
  correction `Sum_{j=1}^{t-1} 1/(j^2+beta^2)`).
- **Real radical roots** (`disc >= 0`): split over an auto-detected field
  extension (`I` and surd generators discovered from `Solve[q==0]`), giving
  `PolyGamma` at radical/complex arguments —
  `Sum[1/(i(i^2+1))] = 1/2 (2 EulerGamma + PolyGamma[0,1-I] + PolyGamma[0,1+I])`
  and the four-`PolyGamma` form for `1/((i^2+3i+1)(i^2+1))`.

Symbolic-coefficient denominators (direct-residue route) and finite/indefinite
rational sums remain out of scope. Tests: `tests/test_sum_rational.c`
(17 checks). Docs: [`calculus.md`](../builtins/calculus.md).

## Feature: `HarmonicNumber` — generalized harmonic numbers (2026-06-19)

`HarmonicNumber[n]` gives `H_n = Sum_{i=1}^n 1/i` and `HarmonicNumber[n, r]`
gives the order-`r` harmonic number `H_n^(r) = Sum_{i=1}^n 1/i^r`. New module
`src/special_functions/harmonicnumber.c`; symbol interned in `sym_names.c`,
docstring in `info.c`. Attributes: `Listable`, `NumericFunction`, `Protected`.

Rather than carry its own numeric kernels, `HarmonicNumber` reduces to existing
primitives and lets the evaluator finish:

- **Non-negative integer `n`** → explicit finite sum `Sum_{i=1}^n i^-r`: exact
  rational for integer `r` (`HarmonicNumber[10] = 7381/2520`), explicit sum for
  symbolic order (`HarmonicNumber[4, r] = 1 + 2^-r + 3^-r + 4^-r`).
  `HarmonicNumber[0, r] = 0`.
- **`HarmonicNumber[Infinity, r]` → `Zeta[r]`** (`Pi^2/6`, `Pi^4/90`, `Zeta[3]`,
  …); `HarmonicNumber[Infinity]` → `ComplexInfinity`.
- **Non-positive integer order** → the Faulhaber polynomial built from
  `BernoulliB` (`HarmonicNumber[n, -1] = n/2 + n^2/2`,
  `HarmonicNumber[z, -4] = -z/30 + z^3/3 + z^4/2 + z^5/5`,
  `HarmonicNumber[n, 0] = n`).
- **Inexact, numericizable `n`** → the analytic identity
  `H_n^(r) = Zeta[r] - Zeta[r, n+1]` (digamma form `EulerGamma +
  PolyGamma[0, n+1]` for `r = 1`), at machine or arbitrary (`MPFR`) precision
  with precision tracking and complex order
  (`N[HarmonicNumber[27, 5 - I]] = 1.02598 + 0.0251513 I`).
- Generic symbols, exact non-integer `n`, and negative-integer `n` stay symbolic;
  0 or ≥3 arguments emit `HarmonicNumber::argt`.

Tested in `tests/test_harmonicnumber.c` (11 groups: exact integers, symbolic
order, infinity, Faulhaber, machine/MPFR/complex numerics, symbolic
passthrough, Listable, `argt`, attributes). No new leaks under valgrind.

## Feature: `NSolve` — numerical solutions of equations and systems (2026-06-18)

`NSolve[expr, vars]` returns numerical approximations to the solutions of an
equation or system as a list of replacement-rule lists; `NSolve[expr, vars,
Reals]` restricts to real solutions (default domain: complexes). `vars` may be a
single symbol or a list, and `NSolve[{e1, …}, vars]` is the conjunction; a
working precision can be given via `WorkingPrecision` or as a trailing
positional argument. Results follow the Wolfram convention: `{}` (no solutions),
`{{x->s,…},…}` (solutions, repeated by multiplicity for one variable), `{{}}`
(universal). Integer, real, and complex coefficients are handled at machine and
arbitrary precision, with solutions over the complexes. Attribute: `Protected`.

- **Dispatch by structure.**
  - Univariate polynomial → `NRoots` (multiplicity repeated; `Reals` filter).
  - Square / zero-dimensional polynomial system → the **eigenvalue /
    multiplication-matrix (Möller–Stetter)** method: a Gröbner basis over `Q`
    gives the quotient ring `Q[x]/I`; eigenvalues + eigenvectors of multiplication
    by a generic linear form yield every isolated solution, coordinates recovered
    as `(M_{x_i} v)[j]/v[j]`. `Method -> "Symbolic"` (and the eigenvalue path's
    fallback) instead solves a lexicographic Gröbner basis triangularly by NRoots
    back-substitution. The four `Method` settings agree on the solution set.
  - Linear / underdetermined / radical / inverse-function equations → `Solve`,
    numericalized (parametric families for underdetermined, `{}` for
    inconsistent).
  - Univariate non-polynomial equations Solve cannot reduce → **FindRoot
    grid-seeding** (best-effort, finite sample).
- **New supporting primitive.** `eigen_all_eigenvectors_real_mpfr` (eigen.h) —
  a black-box twin of `eigen_all_eigenvalues_real_mpfr` returning eigenvalues
  *and* right eigenvectors of a real matrix at MPFR precision (reusing the
  existing Hessenberg + Francis QR + `schur_compute_eigvecs_M` pipeline that
  already backs MPFR `Eigenvectors[]`). Used for the system solver's coordinate
  recovery.
- **Options.** `MaxRoots`, `Method`
  (`Automatic | "EndomorphismMatrix" | "Homotopy" | "Symbolic"`),
  `WorkingPrecision`, `VerifySolutions` (residual check), `RandomSeeding`
  (generic linear form). Anti-hang gates bound the quotient-ring dimension and
  per-generator total degree; out-of-envelope inputs stay unevaluated rather
  than approximate silently.
- **Non-goals (documented).** Real-algebraic solving of `>`, `>=` (no CAD);
  `MaxRoots -> Infinity` symbolic families; quantified systems; bounded-region
  holomorphic solving; full transcendental completeness. `!=` is a post-filter
  only; `"Homotopy"` routes to the eigenvalue engine.
- **Files.** `src/numerical_roots/nsolve.c` (rewritten dispatcher),
  `src/numerical_roots/nsolve_system.{c,h}` (new), `src/linalg/eigen.{h}` +
  `eigen_direct.c` (eigenvector wrapper), `info.c` (docstring), `tests/
  test_nsolve.c` (new). Valgrind-clean (zero bytes beyond the dyld/Accelerate
  baseline) across the univariate, system, elimination, and seeding paths.

## Feature: `NRoots` — numerical roots of a polynomial equation (2026-06-17)

`NRoots[lhs == rhs, var]` yields a disjunction of equations
`var==r1 || var==r2 || …` approximating every root of the polynomial equation,
with a root of multiplicity `k` shown as `k` identical equations (a single root
gives a bare equation). Real and complex coefficients are supported at machine
and arbitrary precision; a vacuous equation collapses to `True`/`False`.

- **Engines (all native complex, all agree to tolerance).**
  - `"Aberth"` (default / `Automatic`): Aberth–Ehrlich simultaneous iteration with
    Bini Newton-polygon circle initialization — cubic, complex/precision-agnostic.
  - `"CompanionMatrix"`: Frobenius companion eigenvalues; real coefficients reuse
    the existing real MPFR QR (`eigen_all_eigenvalues_real_mpfr`), complex
    coefficients use the `C^{n×n}→R^{2n×2n}` real embedding with residual-based
    root selection and synthetic-division multiplicities.
  - `"JenkinsTraub"`: the three-stage shifted-deflation algorithm (CPOLY), used
    for real and complex coefficients (RPOLY's real-arithmetic specialization
    yields identical roots and is folded in).
- **Pipeline.** Symbolic coefficient extraction → numericalize to a complex MPFR
  array → deflate a trailing `x^m` factor (exact zero roots) → engine → noise
  chop, conjugate symmetrization, multiplicity clustering, canonical ordering,
  round to the target precision.
- **Exact squarefree path.** Integer-coefficient polynomials are squarefree-
  decomposed (Yun) and each squarefree factor solved separately with its known
  multiplicity, so high powers like `(x^2-2)^30` stay well-conditioned and exact
  (the expanded degree-60 form is otherwise hopelessly ill-conditioned).
- **Stress tested.** `tests/test_nroots.c` runs a 15-polynomial × 4-method ×
  2-precision battery (real/complex coefficients, real/complex roots,
  multiplicity) verified against known root sets, plus high-multiplicity cases.
- **Options.** `Method`, `PrecisionGoal` (Automatic = machine; digits ⇒ MPFR),
  `MaxIterations`, `StepMonitor` (accepted). Attribute: `Protected`.
- **Wiring / tests / docs.** New `src/numerical_calculus/nroots.{c,h}`,

## Fix: exact Bessel values at the origin for non-integer order (2026-06-16)

## Fix: spurious imaginary part in machine-precision `Power` of a negative real base (2026-06-16)

`Power[b, e]` with a machine-precision (`EXPR_REAL`) **negative** base and an
**integer** exponent leaked a non-trivial imaginary part. For example
`1/(-2.22045*10^(-16))` returned `-4.50359e+15 - 0.551531*I` instead of the
real `-4.50359e+15`.

Root cause: the double-precision fallback in `src/power.c` evaluated such cases
with `cpow(z, w)`. Since `clog(-x) = log|x| + I*pi`, the `sin(pi) ≈ 1.22e-16`
rounding error entered the imaginary component, then got scaled up by the large
result magnitude (`1e16 * sin(pi) ≈ 1.22`) so it survived the `< 1e-14` cleanup
threshold.

Fix: before falling through to `cpow`, route purely real operands whose result
is provably real — non-negative base, or any base with an integer exponent —
through `pow()` directly, which returns an exact real (including the correct
sign for negative-base integer powers, e.g. `(-2.0)^3 = -8.0`). Negative-base
**fractional** exponents (e.g. `(-2.0)^0.5`, `(-8.0)^(1/3)`) are unaffected and
still return their principal complex value. The MPFR path already handled this
correctly; only the machine-`double` path was affected.

## Fix: `D` of `Piecewise` / higher derivatives of `UnitStep` (2026-06-16)

Higher-order derivatives of `UnitStep` returned garbage. `UnitStep'[x]` is
`Piecewise[{{Indeterminate, x == 0}}, 0]`, but there was no rule to
differentiate a `Piecewise`, so the second and third derivatives fell through
to the generic chain rule:

```
In[1]:= D[UnitStep[x], {x, 2}]
(* before: {{0, False Derivative[1, 0][Piecewise][{{Indeterminate, x == 0}}, 0]}} *)
(* after:  Piecewise[{{Indeterminate, x == 0}}, 0] *)

In[2]:= D[UnitStep[x], {x, 3}]
(* before: {{0, {{0, False^2 Derivative[2, 0][Piecewise][...]}}}}  — garbage *)
(* after:  Piecewise[{{Indeterminate, x == 0}}, 0] *)
```

All derivative orders now agree, matching Mathematica.

**Fix** (`src/calculus/deriv.c`):
- Added a `Piecewise` derivative rule:
  `D[Piecewise[{{vi, ci}}, d], x] = Piecewise[{{D[vi, x], ci}}, D[d, x]]` —
  the value expressions and the default are differentiated; the conditions ride
  through unchanged. Since `Piecewise` is `HoldAll`, each value derivative is
  reduced (`eval_and_free`) before being placed back, so e.g. `2 x^(2-1)`
  collapses to `2 x`.
- Added `D[Indeterminate, x] = Indeterminate` (an indeterminate quantity has no
  well-defined derivative). Previously the generic free-of-`x` short-circuit
  collapsed it to `0`, which would have turned the second derivative of
  `UnitStep` into `Piecewise[{{0, x == 0}}, 0]`.

## Fix: `Series` honors the `Assumptions` option for `ExpIntegralEi`/`LogIntegral` branch (2026-06-16)

`Series[ExpIntegralEi[x], {x, 0, n}, Assumptions -> x < 0]` and the
corresponding `LogIntegral` form silently ignored the `Assumptions` option and
always emitted the principal (`x > 0`) branch, so the logarithmic terms printed
as `Log[x]` even when `x` was assumed negative:

```
In[1]:= Series[ExpIntegralEi[x], {x, 0, 5}, Assumptions -> x < 0]
(* before: EulerGamma + Log[x] + x + ...   — wrong, should be Log[-x] *)
(* after:  EulerGamma + Log[-x] + x + ... *)

In[2]:= Series[LogIntegral[x], {x, 0, 2}, Assumptions -> x < 0]
(* before: (120 + ... + Log[x]^5) x / Log[x]^6 + O[x]^3                  — wrong *)
(* after:  I Pi + (120 + ... + Log[-x]^5) x / Log[-x]^6 + O[x]^3 *)
```

**Root cause.** `builtin_series` (`src/calculus/series.c`) never separated the
`Assumptions -> …` option from the expansion specs. Because `Assumptions -> x < 0`
is a `Rule`, it was consumed as a (no-op) second expansion spec, so the option
was dropped without effect.

**Fix.** `builtin_series` now pulls out the `Assumptions` option (keyed on the
LHS symbol `Assumptions`, so it is not confused with a leading-term spec
`x -> x0`), interprets it via a new `assumption_sign_of` helper (recognizes
`Less`/`LessEqual`/`Greater`/`GreaterEqual` of the expansion variable against a
real numeric bound, in either argument order, and `And[…]` conjunctions), and
threads the resulting sign into `try_series_ei_at_zero` /
`try_series_logintegral_at_zero`. For a negative variable they emit `Log[-x]`,
and `LogIntegral` additionally leads with the `I Pi` from
`Log[x] = Log[-x] + I Pi`. The default (no assumptions, or an assumption that
does not fix the sign) is unchanged. The option is forwarded into each inner
variable for multivariate expansions. Matches Mathematica.

## Fix: `Normal` drops the O-term from nested `SeriesData` (2026-06-16)

`Normal` left the `O[…]` term in place whenever the `SeriesData` was not the
top-level head of its argument — exactly the case for expansions around
`+-Infinity`, which wrap their `SeriesData` inside `Plus`/`Times`:

```
In[1]:= Series[BesselJ[0, x], {x, Infinity, 2}] // Normal
(* before: ... (Sqrt[2/Pi] Sqrt[1/x] + O[1/x]^(5/2)) + ... — O-term retained *)
```

`builtin_normal` (`src/calculus/series.c`) only recognised a `SeriesData` head
at the very top of its argument; anything else was passed through verbatim
(`expr_copy`), so the inner `SeriesData` — and its `O[…]` — survived. The same
bug affected every trig-/exponential-prefactored asymptotic form
(`BesselJ`, `BesselK`, `AiryAi`, `AiryBi`, `AiryAiPrime`, `AiryBiPrime`, …).

Fix: the O-term-dropping logic was factored into `seriesdata_to_normal` (the
old body, unchanged) and `builtin_normal` now calls a new recursive walker
`normal_recurse` that descends through the whole tree (head and args),
replacing **every** `SeriesData` at any depth with its normalised sum. The
surrounding factors are preserved and recombined by the ordinary evaluator
(no `Simplify`/`Expand`), so the pure pass-through case (`Normal[a + b]`) and
the top-level case are byte-for-byte unchanged. Malformed `SeriesData` nodes
fall through to a structural copy.

```

Out[1]= Sqrt[2/Pi] Sqrt[1/x] Cos[1/4 Pi - x] - 1/8 Sqrt[2/Pi] (1/x)^(3/2) Sin[1/4 Pi - x]
```

`Normal` now also threads into lists/other containers of series. All
`series_tests` pass; spec entry in
`docs/spec/builtins/power-series.md` updated.

## Feature: multi-generator radical-rational normal form in `Simplify` (2026-06-15)

`Simplify` could not combine a rational function of two or more distinct radical
generators when collapsing it required the relations between them. The reported
case was the round-trip check on the radical integrator:

```
In[1]:= r = Integrate[1/(x^3 (a + b x)^(1/3)), x];
In[2]:= D[r, x] // Simplify
```

`D[r, x]` is a four-term sum of fractions over `a^(k/3)` and `(a+b x)^(k/3)`;
`Together`/`Cancel` handle a single radical generator but left these separate
because the denominators are different fractional powers of *two* bases. The
collapse needs reduction modulo the generator relations `s^3 = a`,
`t^3 = a+b x`.

New pass `simp_radical_rational` (`src/simp/radrat.c`, wired into `simp_dispatch`
in `src/simp/simp_search.c`, mirroring `simp_trig_rational`) works in the
quotient ring `K[g_1, …, g_n] / <g_k^{q_k} − base_k>`:

- **Detect** the distinct radical bases `base_k` (each `base_k^(p/q_k)`,
  `q_k > 1`); require at least two and every base **provably positive** (so
  `base_k^(p/q) → g_k^p` is branch-safe). Single-generator and
  negative/complex-numeric-base inputs are left to the existing path.
- **Substitute** outer-base-first to fresh polynomial generators
  (`poly_subst_radical_to_gen`), build the relation `g_k^{q_k} − V_k` for each
  compound base (where `V_k` expresses `base_k` in the other generators), keeping
  only relations that stay inside the substituted ring.
- **Together**, reduce numerator and denominator modulo the relation ideal
  (`PolynomialRemainder`), rationalise the denominator (`PolynomialExtendedGCD`),
  substitute the radicals back, and finish with `Cancel[N / Factor[D]]`.
- **Strict score gate** — the result is returned only when its `SimplifyCount`
  beats the input, so the pass can never regress a case; it returns NULL
  otherwise and `simp_dispatch` continues unchanged.

Result: `D[Integrate[1/(x^3 (a+b x)^(1/3)), x], x] // Simplify` now returns
`1/(x^3 (a+b x)^(1/3))`. Scope is `Simplify` only; `Together`/`Cancel` keep their
single-generator behaviour. `src/simp/radrat.c` is added to the makefile glob and
to `tests/CMakeLists.txt` `COMMON_SRC`.

## Fix: post-integration result normalisation (`Integrate`) (2026-06-15)

`Integrate[1/(x^3 (a + b x)^(1/3)), x]` (and the linear/quadratic/ratio
radical family generally) returned a correct but needlessly verbose
antiderivative — unsimplified rational pieces over denominators in the
compound radicand and stray `(1/a)^(k/3)` constants. A post-method
normalisation pass, `intsimp_finalize` (`src/calculus/intsimp.c`, wired at the
one chokepoint in `builtin_integrate`, `src/calculus/integrate.c`), now cleans
**radical-bearing** antiderivatives (those carrying a `Power[f(x), p/q]` of the
integration variable). Non-radical / partially-closed (`nested Integrate`)
results pass straight through untouched — the deficiency only manifests in the
substitution integrators, and reshaping rational/Risch output perturbed forms
the corpus verifier pins. The pass:

- **x-free reciprocal/product powers.** `(1/a)^(2/3) → a^(-2/3)` etc. via
  `PowerExpand` applied only to x-free `Power[c, p/q]` subexpressions whose base
  is positive under the symbol-positive convention (`it_normalize_xfree_powers`
  + `it_pos_base`) — the reason `a^(7/3)` denominators and clean `Log`/`ArcTan`
  arguments now appear. The positive-base guard is essential: PowerExpand on a
  negative/complex numeric constant (`(-3)^(1/2)`) would cross a branch cut and
  silently change a coefficient's value.
- **Algebraic recombination.** The back-substituted Hermite sum is combined and,
  crucially, its `Numerator` / `Denominator` are `Expand`-ed so integer-power
  compounds collapse in x (`6 a^4 − 12 a^3 (a+b x) + 6 a^2 (a+b x)^2 →
  6 a^2 b^2 x^2`) before a final `Cancel` (`it_recombine_algebraic`). Gated to
  replace the original only when it strictly shrinks (leaf count), size-capped
  to avoid multivariate-GCD blow-up.
- **Inverse-trig arguments** are distributed and sign-normalised
  (`ArcTan[-(p+q)] → -ArcTan[p+q]`), distributing the negation so ArcTan's own
  oddness does not re-pull it.

The reported case now returns
`(4 (a+b x)^(5/3) − 7 a (a+b x)^(2/3))/(6 a^2 x^2) + (2 b^2 ArcTan[1/Sqrt[3] +
2 (a+b x)^(1/3)/(Sqrt[3] a^(1/3))])/(3 Sqrt[3] a^(7/3)) − …`, matching
Mathematica up to an immaterial `Log` sign (constant of integration).
Correctness re-verified with `PossibleZeroQ[D[result, x] − integrand]`.
`intsimp_finalize` takes ownership of the result and is a single tree-walk for
the non-radical fast path; no new builtins or attributes; prototype added to
`intsimp.h`. Regression-checked: all integration unit suites pass, the
`IntegrateRationalTests.m` corpus is unchanged (≤ baseline diffs, no new
crashes), and valgrind shows no new leaks vs the `Sin[1.0]` baseline.

## Feature: `AiryAi` — Airy function Ai(z) (2026-06-15)

Implemented `AiryAi` in `src/special_functions/airyai.c`, with its derivative
head `AiryAiPrime`. `AiryAi[z]` is the Airy function Ai(z), the solution of
y″ = z y that decays as z → +∞ — an entire function of z.

- **Exact values.** `AiryAi[0] = 1/(3^(2/3) Gamma[2/3])` and
  `AiryAi[±Infinity] = 0`. Other exact arguments (e.g. `AiryAi[2]`) stay
  unevaluated and numericalize only under `N`.
- **Unified numeric core.** A file-local complex-MPFR toolkit (`acx`, pairs of
  `mpfr_t`, the `ecx`/`pcx` pattern) computes Ai(z) and Ai′(z) together for real
  and complex inputs at machine and arbitrary (MPFR) precision. It routes on
  `r = |z|` and target precision `P`:
  - **Maclaurin series** for small/moderate `r`: recurrence
    `a_n = a_{n-3}/(n(n-1))` from Ai″ = z Ai, with `(2/3) r^{3/2}/ln2` guard
    bits absorbing the partial-sum cancellation. The closed-form constants
    `a_0 = Ai(0)`, `a_1 = Ai′(0)` are formed with the `2/3`, `1/3` fractions
    built exactly in MPFR (not via doubles).
  - **Asymptotic series** (DLMF 9.7.5) for large `r`: ζ = (2/3) z^{3/2},
    `u_k = (6k-5)(6k-3)(6k-1)/((2k-1)·216·k) u_{k-1}`, summed to optimal
    truncation. Near the negative real axis (2π/3 < |arg z| ≤ π) the DLMF 9.2.12
    connection relation `Ai(z) = -[ω Ai(ωz) + ω̄ Ai(ω̄z)]`, ω = e^{2πi/3}, maps
    the argument into two well-conditioned in-sector points so the oscillation
    emerges with no catastrophic cancellation.
  - Real arguments route through the same core (imaginary part 0), validated
    against MPFR's `mpfr_ai` over a half-integer grid in the unit tests.
  - Machine-precision results whose magnitude overflows the double range (e.g.
    `AiryAi[150. I] ≈ 8.1951×10^374 + …`) are promoted to MPFR rather than
    becoming `Inf`.

  Examples: `AiryAi[1.8] = 0.0470362`, `AiryAi[2.5 + I] = -0.00191209 -
  0.0180329 I`,
  `N[AiryAi[2], 50] = 0.034924130423274379135322080791807609761060213897583`.
- **Derivatives** (`src/calculus/deriv.c`). `D[AiryAi[z], z] = AiryAiPrime[z]`
  and `D[AiryAiPrime[z], z] = z AiryAi[z]`, so higher derivatives reduce
  (`D[AiryAi[x], {x, 2}] = x AiryAi[x]`).
- **Taylor series at 0** comes from the generic Taylor-via-D path once the exact
  values at 0 exist: `Series[AiryAi[x], {x, 0, 7}]` reproduces the closed-form
  coefficients.
- **Asymptotic series at ∞** (`src/calculus/series.c`,
  `try_series_airyai_at_infinity`). `Series[AiryAi[x], {x, Infinity, n}]` returns
  `Times[Exp[-(2/3) x^(3/2)], SeriesData[1/x, …]]` — a Puiseux series in
  `(1/x)^(1/4)` with the essential-singularity prefactor kept symbolic, matching
  Mathematica (e.g. `E^(-2/3 x^(3/2)) ((1/x)^(1/4)/(2 Sqrt[Pi]) -
  5 (1/x)^(7/4)/(96 Sqrt[Pi]) + O[1/x]^(13/4))`).
- **`AiryAiPrime`** is a minimal head: it has the derivative rule and the exact
  value `AiryAiPrime[0] = -1/(3^(1/3) Gamma[1/3])` (needed by the series) but no
  general numeric evaluator (`N[AiryAiPrime[2.0]]` stays unevaluated).

## Feature: `AiryBi` — Airy function Bi(z) (2026-06-15)

Implemented `AiryBi` in `src/special_functions/airybi.c`, with its derivative
head `AiryBiPrime`, as the companion of `AiryAi`. `AiryBi[z]` is the second
solution of y″ = z y — entire in z, growing exponentially as z → +∞ and
oscillating with decaying amplitude as z → −∞.

- **Exact values.** `AiryBi[0] = 1/(3^(1/6) Gamma[2/3])`,
  `AiryBi[Infinity] = Infinity`, `AiryBi[-Infinity] = 0`. Other exact arguments
  (e.g. `AiryBi[2]`) stay unevaluated and numericalize only under `N`.
- **Unified numeric core.** A file-local complex-MPFR toolkit (`acx`) computes
  Bi(z) and Bi′(z) together for real and complex inputs at machine and arbitrary
  (MPFR) precision, routing on `r = |z|`, `θ = arg z`, and target precision `P`:
  - **Maclaurin series** for small/moderate `r`: recurrence
    `b_n = b_{n-3}/(n(n-1))` from Bi″ = z Bi, seeded by the closed-form constants
    `b_0 = Bi(0) = 1/(3^(1/6) Gamma(2/3))`, `b_1 = Bi′(0) = 3^(1/6)/Gamma(1/3)`
    (fractions built exactly in MPFR), with `(2/3) r^{3/2}/ln2` guard bits.

    sector: ζ = (2/3) z^{3/2}, the same `u_k`/`v_k` as Ai but with no `(-1)^k`
    and prefactor `1/Sqrt[Pi]`, used only where the recessive companion
    `~exp(-2 Re ζ)` is below 2^−P (`Re ζ = (2/3) r^{3/2} cos(3θ/2) > (P ln2)/2`),
    keeping the exponentially large positive axis at full precision.
  - Near/left of the anti-Stokes line `|arg z| = π/3` (incl. the oscillatory
    negative real axis) the **DLMF 9.2.10 connection to Ai**
    `Bi(z) = e^{iπ/6} Ai(z e^{2πi/3}) + e^{-iπ/6} Ai(z e^{-2πi/3})` (and the
    matching `Bi′` relation), evaluated by a file-local Ai asymptotic kernel
    (direct series + the DLMF 9.2.12 wrapper), so the oscillation emerges with no
    catastrophic cancellation.
  - Machine-precision results whose magnitude overflows the double range (e.g.
    `AiryBi[1000.] ≈ 5.4077×10^9154`) are promoted to MPFR rather than `Inf`.

  Examples: `AiryBi[1.8] = 2.59587`, `AiryBi[2.5 + I] = 0.512544 + 5.335 I`,

  values at 0 exist: `Series[AiryBi[x], {x, 0, n}]` reproduces the closed-form
  coefficients.
- **Asymptotic series at ∞** (`src/calculus/series.c`,
  `try_series_airybi_at_infinity`). `Series[AiryBi[x], {x, Infinity, n}]` returns
  `Times[Exp[(2/3) x^(3/2)], SeriesData[1/x, …]]` with the essential-singularity

  5 (1/x)^(7/4)/(48 Sqrt[Pi]) + O[1/x]^(13/4))`).
- **`AiryBiPrime`** is a minimal head: derivative rule plus the exact value
  `AiryBiPrime[0] = 3^(1/6)/Gamma[1/3]`, no general numeric evaluator.
- **Attributes.** `AiryBi` and `AiryBiPrime`: `Listable`, `NumericFunction`,

  docstring in `info.c`; `argrx` arity diagnostic.

## Fibonacci: exact non-integer order for the polynomial form

`Fibonacci[n, x]` now evaluates when the order `n` is an exact **non-integer**
number (e.g. a `Rational`) and `x` is numeric — previously only an *inexact*
order triggered the generalized closed form, so exact rational orders fell
through to the unevaluated form. The gate (`src/special_functions/fibonacci.c`)
now keys on `expr_is_numeric_like(n) && expr_is_numeric_like(x)` and routes:

- any inexact input → `numericalize` the closed form
  `(β^n - Cos[π n] β^{-n}) / Sqrt[x²+4]`, e.g. `Fibonacci[1/2, 3.2] = 0.494833`;
- all-exact inputs → evaluate the closed form exactly and keep the result only
  when it collapses to a number, e.g. `Fibonacci[1/2, 0] = 1/2`,
  `Fibonacci[1/3, 0] = 1/4` (from `F_n(0) = (1 - Cos[π n])/2`). A non-collapsing
  exact `x`, or a symbolic `x`, stays unevaluated — mirroring the long-standing
  one-argument rule that exact non-integer orders stay symbolic until they
  reduce to a value.

Fixes `Fibonacci[1/2, 0]`, `Fibonacci[1/2, x]`, and `Fibonacci[1/2, 3.2]` to
match Mathematica (`1/2`, unevaluated, `0.494833`). The one-argument form and
all integer-order / numeric-order behaviour are unchanged. New regression tests
in `tests/test_fibonacci.c` (`test_rational_order` plus a numeric case).

## BesselJ / BesselI: integer-order argument parity

`BesselJ` and `BesselI` now apply the argument-parity reflection for **integer**
order — both are entire in z, so

- `BesselJ[n, -z] = (-1)ⁿ BesselJ[n, z]`
- `BesselI[n, -z] = (-1)ⁿ BesselI[n, z]`

e.g. `BesselJ[0, -z] → BesselJ[0, z]`, `BesselJ[1, -z] → -BesselJ[1, z]`,
`BesselJ[1, -2 x] → -BesselJ[1, 2 x]`, and likewise for `BesselI`. This matches
Mathematica; previously the negated argument was left unevaluated (only negative
**order** was reflected, in `src/internal/bessel.m`).

Implemented as a C-side fold (`bessel_z_parity_fold` in
`src/special_functions/bessel.c`) reusing `expr_is_superficially_negative`, the
same negative-argument detector `Sin`/`Cos`/etc. use. It fires only for
symbolic / exact arguments (a concrete inexact argument such as
`BesselJ[1, -2.5]` still evaluates numerically), only for integer order, and
composes with the existing negative-order reflection (e.g.
`BesselJ[-1, -z] → BesselJ[1, z]`).

`BesselK` deliberately gets **no** such fold: it has a branch cut along the
negative real axis, so `BesselK[n, -z]` carries no clean parity and stays
unevaluated (matching Mathematica). New regression tests:
`test_besselj_argument_parity`, `test_besseli_argument_parity`, and
`test_besselk_no_argument_parity`.

## Feature: `Product` — symbolic products + q/Barnes-G prerequisites (2026-06-18)

`Product[f, {i, imin, imax}]` (and the `{i,imax}`, `{i,imin,imax,di}`,
`{i,{i1,…}}`, multi-iterator, and indefinite `Product[f, i]` forms) is the
multiplicative analogue of `Sum`, implemented natively in C under
`src/product/`: a dispatcher plus one file per algorithm, each exposed as a
context-qualified builtin. Attributes: `HoldAll`, `Protected`. Empty/reversed
ranges fold to `1`. `N[Product[...]]` routes to `NProduct`.

- **Method polyalgorithm.** `Method -> Automatic` (default) runs a cascade
  Telescoping → Rational → Geometric → QProduct → Special → Infinite (cleanest
  closed form wins); each is reachable directly via
  `Method -> "Telescoping" | "Rational" | "Geometric" | "QProduct"`.
  - **`Product`Telescoping`** — Gamma-free rational anti-quotients via
    integer-spaced root chains (`Product[1-1/k^2,{k,2,n}]` → `(1+n)/(2n)`).
  - **`Product`Rational`** — any rational `f` with linear factors over `Q` →
    `Pochhammer`/`Factorial` (`Product[k,{k,1,n}]` → `n!`,
    `Product[x+i,{i,0,n-1}]` → `Pochhammer[x,n]`).
  - **`Product`Geometric`** — `base^(p(i))` factors routed through `Sum`
    (`Product[2^k,{k,1,n}]` → `2^(n(n+1)/2)`); never `Together`/`Factor`s a
    `base^var` (avoids the symbolic-exponent hang).
  - **`Product`QProduct`** — factors linear in `q^i` →
    `QPochhammer[a q^imin, q, imax-imin+1]`
    (`Product[1-a q^k,{k,0,n-1}]` → `QPochhammer[a,q,n]`).
  - **`Product`Special`** — `Product[i^i,{i,1,n}]` → `Hyperfactorial[n]`,
    `Product[Gamma[i],{i,1,n-1}]` → `BarnesG[n]`.
  - **`Product`Infinite`** — `imax == Infinity`: rational convergence gate
    (equal degree, leading and next-to-leading coefficients) + limit of the
    finite closed form, plus the Weierstrass family
    `Product[1+c/k^2,{k,1,Infinity}]` → `Sinh[Pi Sqrt[c]]/(Pi Sqrt[c])`. A
    divergent product prints `Product::div` and stays unevaluated. Option
    `VerifyConvergence` (default `True`).

- **New special functions** (`src/special_functions/`): `Hyperfactorial[n]`
  (`prod k^k`), `BarnesG[z]` (`G(n+1)=prod k!`, `0` at non-positive integers),
  and `QPochhammer[a,q,n]` / `QPochhammer[a,q]` (q-shifted factorial; finite
  exact/symbolic, infinite numeric for `|q|<1`). Each `Listable`,
  `NumericFunction`, `Protected`, with machine and MPFR `N`.

Tests: `tests/test_product.c` (201 unit checks across all six stages),
`tests/test_product_stress.c` (oracle-verified generated families), and
`tests/test_product_special.c` (the three special functions, incl. `N[x,35]`).
Valgrind-clean (no new leaks).
