# Changelog: week of 2026-06-08 (Mon) – 2026-06-14 (Sun)

Feature additions and fixes recorded during this week.

## Feature: `Gamma` — Euler & incomplete gamma functions (2026-06-08)

New builtin `Gamma` (`src/gamma.c`), covering the full Mathematica signature:

- `Gamma[z]` — Euler gamma function Γ(z).
- `Gamma[a, z]` — upper incomplete gamma Γ(a, z) = ∫_z^∞ tᵃ⁻¹ e⁻ᵗ dt.
- `Gamma[a, z0, z1]` — generalized incomplete = Γ(a, z0) − Γ(a, z1).

Attributes: `Listable`, `NumericFunction`, `Protected`. Docstring in
`src/info.c`; symbol already interned in `src/sym_names.c`.

Evaluation is layered for efficiency across all three argument regimes:

- **Exact symbolic.** Integer and half-integer `Gamma[z]` reduce to `(z-1)!`
  by reusing the existing `Factorial` machinery (exact int64, GMP BigInt, and
  rational·`Sqrt[Pi]` half-integers). Non-positive integers give
  `ComplexInfinity`. `Infinity`/`-Infinity`/`ComplexInfinity` map to
  `Infinity`/`Indeterminate`/`ComplexInfinity`. Incomplete-form rewrites:
  `Gamma[a,0] = Gamma[a]`, `Gamma[1,z] = E^-z`, `Gamma[a,Infinity] = 0`.

```
Table[Gamma[n], {n, 10}]   -> {1, 1, 2, 6, 24, 120, 720, 5040, 40320, 362880}
Gamma[30]                   -> 8841761993739701954543616000000
Gamma[5/2]                  -> 3/4 Sqrt[Pi]
Gamma[-1/2]                 -> -2 Sqrt[Pi]
Gamma[0]                    -> ComplexInfinity
```

- **Machine precision.** Real → libm `tgamma`; complex → Lanczos
  approximation (g = 7) with reflection for the left half-plane.

```
Gamma[2.2]      -> 1.1018
Gamma[2.3 + I]  -> 0.719141 + 0.540614 I
```

- **Arbitrary precision.** Real → MPFR `mpfr_gamma`; incomplete real (machine
  or MPFR) → `mpfr_gamma_inc`. Output precision tracks the input.

```
N[Gamma[22/10], 50] -> 1.1018024908797127327691419862229964808241863995904
Gamma[1.5, 7.5]     -> 0.00160996
Gamma[1, 1.1, 2.2]  -> 0.222068
```

Tests: `tests/test_gamma.c` (exact values, poles, half-integers, infinities,
symbolic fall-through, machine real/complex, MPFR precision tracking,
incomplete & generalized incomplete, Listable threading, attributes). Valgrind
clean (totals identical to the `Sin[1.0]` baseline; no Gamma frames in any lost
stack).

## Closing the Mathematica gaps for `Gamma` (2026-06-08)

Follow-up work to bring `Gamma` in line with Mathematica across more of its
domain (`src/gamma.c`, `src/calculus/deriv.c`, `tests/test_gamma.c`):

- **Integer incomplete gamma closes to a finite form.** `Gamma[n, z]` for a
  positive integer `n` now reduces to `(n-1)! e^-z Σ_{k=0}^{n-1} z^k/k!` for
  symbolic or exact `z` (integer coefficients via GMP, BigInt-safe). So
  `Gamma[2, 3] -> 4/E^3`, `Gamma[2, x] -> (1 + x) E^-x`,
  `Gamma[3, x] -> (2 + 2 x + x^2) E^-x`. (`Gamma[1, z] = E^-z` is now the
  `n = 1` case of this rule.)

- **Derivatives.** `D[Gamma[a, z], z] = -z^(a-1) E^-z`, with the chain rule on
  both arguments; the first-argument derivative is the generic
  `Derivative[1,0][Gamma][a,z]` (a closed form needs `PolyGamma`, not yet
  implemented, so one-argument `D[Gamma[z], z]` likewise stays generic).

- **Arbitrary-precision complex `Gamma[z]`.** Replaces the previous
  "left symbolic" limitation with Spouge's approximation, whose coefficients
  are generated at runtime to the requested precision (reflection for
  `Re(z) < 1/2`). E.g.
  `N[Gamma[I], 50] = -0.15494982830181068512… − 0.49801566811835604271… I`.
  Verified against the independent machine Lanczos path.

## Fix: `Gamma` argument-count diagnostic (2026-06-08)

`Gamma` now emits a Mathematica-compatible `Gamma::argt` warning when called
with the wrong number of arguments (0, or 4+), and leaves the call
unevaluated (`src/gamma.c`):

```
Gamma[]          -> Gamma::argt: Gamma called with 0 arguments; 1 or 3 arguments are expected.
Gamma[1, 2, 3, 4] -> Gamma::argt: Gamma called with 4 arguments; 1 or 3 arguments are expected.
```

The valid 1-, 2-, and 3-argument signatures are unaffected.

- **Complex incomplete gamma `Gamma[a, z]`.** Numeric complex `a`/`z` (machine
  or arbitrary precision) now evaluate via a lower-incomplete power series
  (`Re(z) < Re(a)+1`) or a Lentz continued fraction otherwise, finished with
  the `e^-z z^a` prefactor. E.g. `Gamma[2.0, 1 + I] = (2 + I) e^-(1+I)` and
  `N[Gamma[3/2, 2 + I], 30] = 0.160487401929263240… − 0.176588715957602346… I`
  (confirmed via the recurrence `Γ(a+1,z) = a Γ(a,z) + z^a e^-z` to 30 digits).

The complex arithmetic uses a small `mpfr_t`-pairs toolkit local to
`src/gamma.c` (add/sub/mul/div/exp/log/sin/pow); MPC is not a dependency.
Exact non-integer or exact-complex incomplete forms (`Gamma[3/2, z]`,
`Gamma[3/2, I]`) stay symbolic, matching Mathematica. Valgrind totals remain
identical to the `Sin[1.0]` baseline with no Gamma/deriv frames in any lost
stack.

## Feature: `Pochhammer` — the rising factorial (2026-06-08)

New builtin `Pochhammer` (`src/pochhammer.c`): the Pochhammer symbol
(a)ₙ = a (a+1) … (a+n-1) = Γ(a+n)/Γ(a). Attributes `Listable`,
`NumericFunction`, `Protected`; docstring in `src/info.c`; symbol interned in
`src/sym_names.c`; wired from `core_init()` next to `gamma_init()`.

Following the `Gamma` precedent, the implementation holds almost no numeric
code of its own — it delegates to two existing, fully-tested mechanisms:

- **Exact integer order → a `Times` product.** For an integer `n`
  (|n| ≤ 1000) it builds `Times[a, a+1, …, a+n-1]` (or the reciprocal for
  negative `n`) and evaluates it. That one path collapses to an exact
  Integer/Rational for numeric `a`, preserves MPFR precision and complex
  arithmetic, and stays a polynomial product for symbolic `a`.

- **Other numeric arguments → the Gamma ratio.** It evaluates
  `Gamma[a+n]/Gamma[a]`, reusing Gamma's exact half-integer reductions, its
  libm/MPFR real paths and its machine-complex Lanczos path. A residual-`Gamma`
  guard keeps genuinely-symbolic inputs unevaluated.

Short-circuits: `Pochhammer[a, 0] = 1`, `Pochhammer[0, n>0] = 0` (so even
`Pochhammer[0, 1285] = 0` without forming a huge factorial),
`Pochhammer[Infinity, n>0] = Infinity`. Wrong argument counts emit a
Mathematica-compatible `Pochhammer::argrx` diagnostic and stay unevaluated.

```
Pochhammer[10, 6]                 -> 3603600
Pochhammer[n, 5]                  -> n (1 + n) (2 + n) (3 + n) (4 + n)
Pochhammer[n, -5]                 -> 1/((-5 + n) (-4 + n) (-3 + n) (-2 + n) (-1 + n))
Pochhammer[3/2, 1/2]              -> 2/Sqrt[Pi]
Pochhammer[2.4, 8.5]              -> 2.31022e+06
N[Pochhammer[1/3, 7], 50]         -> 505.97165066300868770004572473708276177411979881116
Pochhammer[2. + 5 I, 8 I]         -> 2.13868e-06 - 1.42187e-05 I
Pochhammer[{2, 3, 5, 7, 11}, 3]   -> {24, 60, 210, 504, 1716}
```

Tests: `tests/test_pochhammer.c` (exact integer numeric/symbolic, negative
order, the `0`/`Infinity` short-circuits, half-integer reduction, machine
real/complex, MPFR precision tracking, symbolic fall-through, Listable,
attributes, arity diagnostics). Valgrind totals identical to the macOS
baseline with no Pochhammer frames in any lost stack. Derivatives and series
(which Mathematica expresses through `PolyGamma`) are deferred.
