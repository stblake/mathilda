# Changelog: week of 2026-06-08 (Mon) – 2026-06-14 (Sun)

Feature additions and fixes recorded during this week.

## Feature: `NSeries` — numerical Taylor/Laurent series (2026-06-13)

New builtin `NSeries[f, {x, x0, n}]` giving a numerical approximation to the
series expansion of `f` about `x = x0`, including the terms `(x-x0)^-n` through
`(x-x0)^n`, as a `SeriesData` object. Third member of the **numerical calculus**
group; implemented in `src/numerical_calculus/nseries.{c,h}`; attribute
`Protected` (not `Listable`); symbol `NSeries` interned in `sym_names.{c,h}`;
registered from `core_init` via `nseries_init`; docstring in `info.c`.

**Method (Lyness & Sande 1971; Bornemann, *FoCM* 2011).** `f` is sampled at `N`
equispaced points on a circle of radius `r` centred at `x0`,
`z_j = x0 + r e^{2πi j/N}`, and a discrete Fourier transform of the samples
recovers the Laurent coefficients via Cauchy's integral formula:
`c_k = (1/N) Σ_j f(z_j) e^{-2πi jk/N}`, then `a_e = c_{e mod N} · r^{-e}` for
`e = -n..n`. The upper-half DFT bins (`k = N-m`) supply the **negative**-power
coefficients, so one transform yields both the principal part and the analytic
part — exact when `f` is analytic on an annulus containing the circle.

- **Why this beats symbolic `Series`.** It needs only to sample `f` numerically,
  so it handles functions with no closed-form coefficients and, crucially,
  **Laurent expansions about essential singularities** where `Series` bails:
  `NSeries[Sin[x + 1/x], {x, 0, 10}] // Chop` recovers the Laurent series, while
  `Series[Sin[x + 1/x], {x, 0, 10}]` returns unevaluated.
- **`Radius` selects the annulus.** For a function with several singularities,
  `Radius` picks which annulus the Laurent series converges in:
  `NSeries[1/((1+x)(3+x)), {x,0,10}, Radius->5]` gives the far-annulus integer
  coefficients `9841/x^10 - 3280/x^9 + … + 1/x^2`.
- **Direct `O(N²)` DFT, no FFT dependency.** `N = 2^{⌈log2 n⌉+2}` is small
  (≤ a few hundred) and each sample is a full symbolic evaluation of `f`, which
  dominates the runtime; an FFT would optimise the part that is not slow and
  could not serve the MPFR path anyway. The same code serves machine
  (`double _Complex`) and arbitrary-precision (MPFR) computations. The `+2`
  oversampling (vs Mathematica's documented `+1`) widens the anti-alias margin.
- **Numerical hygiene.** Round-off breaks the conjugate symmetry of
  real-coefficient functions, leaving tiny spurious residuals — `Chop` them;
  `WorkingPrecision -> d` (MPFR) shrinks both the coefficient error and the
  spurious imaginary parts.

Options: `Radius` (default `1`), `WorkingPrecision` (default `MachinePrecision`).
Result `SeriesData[x, x0, {a_-n..a_n}, -n, n+1, 1]` (the original symbolic `x0`
is kept as the base, so `NSeries[Exp[x],{x,I,5}]` prints in `(x - I)`).
Diagnostics: `NSeries::ivar` (bad spec / non-symbol variable / negative order),
`NSeries::nnum` (`x0` or a sample not numeric), `NSeries::badopt`. Reference in
`docs/spec/builtins/calculus.md`; unit tests in `tests/test_nseries.c` (10 cases:
Taylor at real/complex centres, MPFR precision, essential-singularity Laurent,
`Radius`-controlled annulus, result shape, edge cases, attributes, memory loop),
all validated against the documented Mathematica outputs and valgrind-clean (only
the documented macOS dyld/Accelerate baseline noise).

## Feature: `ND` — numerical derivative (2026-06-13)

New builtin `ND[expr, x, x0]` / `ND[expr, {x, n}, x0, opts]` giving a numerical
approximation to the `n`-th derivative of `expr` at `x = x0`. Implemented in
`src/numerical_calculus/nderiv.{c,h}`; attribute `Protected`; interned symbols
`ND`, `Scale`, `Terms`, `EulerSum`, `NIntegrate` added in `sym_names.c`.

Two methods:

- **`Method -> EulerSum`** (default): Richardson (Romberg/Neville)
  extrapolation of the `n`-th **forward** finite difference along the complex
  direction `Scale`. The forward (one-sided) stencil reproduces Mathematica's
  directional behaviour — left/right derivatives of `Abs`, complex directions
  such as `Scale -> 1 + I` — and the all-powers error expansion gives the
  `2^j - 1` Richardson denominator. Works for non-analytic `expr`; needs
  integer order `n >= 0`. Machine `double _Complex` and MPFR complex paths.
- **`Method -> NIntegrate`**: Cauchy integral formula evaluated by reusing
  `NResidue` — `Gamma(n+1) * NResidue[expr/(x-x0)^(n+1), {x, x0}, Radius ->
  Scale]`. Passing `Radius -> Scale` (default `1`) avoids the heavy `1/r^n`
  cancellation of `NResidue`'s tiny `1/100` default radius, recovering full
  accuracy. `Gamma(n+1)` (vs `n!`) allows fractional/complex order
  (`ND[x, {x, -1/2}, 1]` = `4/(3 Sqrt[Pi])`). Requires `expr` analytic near
  `x0`.

Options: `Method`, `Scale` (default 1), `Terms` (default 7), `WorkingPrecision`,
`PrecisionGoal`, `MaxRecursion`. Threads element-wise over a list in the first
argument (manual threading; not `Listable`). Docstring in `info.c`; reference in
`docs/spec/builtins/calculus.md`; unit tests in `tests/test_nderiv.c` (validated
against the documented Mathematica outputs, valgrind-clean).

## Fix: `Limit` of special functions with a logarithmic leading term (2026-06-13)

`Limit[ExpIntegralEi[x], x -> 0]` returned unevaluated where Mathematica gives
`-Infinity`. `ExpIntegralEi[0]` itself evaluates to `-Infinity`, but the
direct-substitution fast paths deliberately refuse divergent substitutions
(they would mis-handle indeterminate forms), and the Series layer bailed: the
expansion `EulerGamma + Log[x] + x + ...` has its leading (`x^0`) coefficient
`EulerGamma + Log[x]`, which still mentions `x`, so the leading-term reader
treated it as the unresolvable logarithmic part of an asymptotic expansion.

Two coordinated changes in `src/calculus/limit.c`:

- `read_leading_term_limit`: when the leading exponent is exactly `0` and the
  coefficient still contains the limit variable, recurse on
  `Limit[coefficient]`. Every higher-order term carries a strictly positive
  power of `x` and vanishes at the point, so `lim f = lim(leading coefficient)`.
  Only an answer free of `x` is accepted; otherwise the old bail is preserved.
- `layer_plus_termwise`: its single-dominant-divergent case now also fires at
  finite limit points (previously ±Infinity only), so the recursion above
  resolves `EulerGamma + Log[x]` (finite `EulerGamma` + a term `-> -Infinity`).
  The growth-exponent shortcut and the all-finite termwise sum remain gated to
  infinity, keeping established finite-point behaviour unchanged.

`Limit[ExpIntegralEi[x], x -> 0]` now returns `-Infinity`;
`Limit[LogIntegral[x], x -> 0]` stays `0` (the finite value `li(0)`). Regression
test: `test_special_log_leading` in `tests/test_limit.c`.

## Fix: `Series[LogIntegral[x], {x, 0, n}]` generalized series (2026-06-13)

`Series[LogIntegral[x], {x, 0, 2}]` returned unevaluated (leaking
`Power::infy` / `Infinity::indet`). `li(x) = Ei(Log[x])` has no Taylor series
at `x = 0`: as `x -> 0+`, `Log[x] -> -Infinity` puts it in Ei's asymptotic
regime, so naive Taylor-via-`D` hit the `1/0` pole of the second derivative
`-1/(x Log[x]^2)` and bailed.

`do_series_single` (`src/calculus/series.c`) now recognises `LogIntegral[x]`
at `x = 0` via the new `try_series_logintegral_at_zero` helper (a sibling of
the existing `try_series_ei_at_infinity`) and emits the generalized series
directly:

```
Series[LogIntegral[x], {x, 0, 2}]
  = ((120 + 24 Log[x] + 6 Log[x]^2 + 2 Log[x]^3 + Log[x]^4 + Log[x]^5) x)
      / Log[x]^6 + O[x]^3
```

i.e. `x Σ_{k=0}^{2n+1} k!/Log[x]^(k+1) + O[x]^(n+1)` (DLMF 6.12.2 applied to
`Ei(Log[x])`). The only `x`-dependence is the prefactor `E^(Log[x]) = x`, so
every term carries exactly `x^1` and the `x^1` slot holds
`Together[Σ k!/Log[x]^(k+1)]`. This reproduces Mathematica's
`Assumptions -> x > 0` output; the no-assumptions `Floor[Arg[...]]`
branch-discriminator form is not emitted (Mathilda always returns the
principal `x > 0` form). Regression test: `test_series_logintegral_at_zero`.

## Fix: `Series[ExpIntegralEi[x], {x, 0, n}]` logarithmic series (2026-06-13)

`Series[ExpIntegralEi[x], {x, 0, 2}]` returned unevaluated: `Ei(0) = -Infinity`
blocks naive Taylor-via-`D`. `do_series_single` now emits the DLMF 6.6.2
logarithmic series directly via the new `try_series_ei_at_zero` helper:

```
Series[ExpIntegralEi[x], {x, 0, 2}] = EulerGamma + Log[x] + x + x^2/4 + O[x]^3
```

i.e. `EulerGamma + Log[x] + Σ_{k≥1} x^k/(k k!)`, emitted as
`SeriesData[x, 0, {EulerGamma + Log[x], 1, 1/4, ..., 1/(n n!)}, 0, n+1, 1]`
with the `EulerGamma + Log[x]` branch term in the `x^0` slot. Regression test:
`test_series_expintegralei_at_zero`.

## Fix: `Series` spurious `Power::infy` + `ExpIntegralEi` asymptotic at Infinity (2026-06-13)

`Series[ExpIntegralEi[x], {x, Infinity, 3}]` printed `Power::infy: Infinite
expression 1/0 encountered.` and returned unevaluated. Two fixes in
`src/calculus/series.c`:

- **Asymptotic expansion at Infinity.** `do_series_single` now recognises
  `ExpIntegralEi[x]` at `x = Infinity` and emits its asymptotic series directly
  (DLMF 6.12.2): `Ei(x) ~ E^x Σ_{k≥0} k!/x^(k+1)`. The `E^x` essential
  singularity stays as a symbolic prefactor, so the result is
  `Times[Power[E, x], SeriesData[Power[x, -1], 0, {0!,1!,…,(n-1)!}, 1, n+1, 1]]`,
  printing as `E^x (1/x + 1/x^2 + 2/x^3 + O[1/x]^4)` — matching Mathematica.
- **No more spurious `Power::infy`.** The root cause was general: the naive-Taylor
  singularity probe (`series_taylor_via_D`) substituted `x -> x0` with
  `replace_all_of` (non-evaluating), so a pole like `1/0 = Power[0,-1]` stayed
  unevaluated, slipped past the `has_infinity` guard, and only blew up — leaking
  the warning — when a coefficient was later simplified. The probe now
  **evaluates** the substituted value with `arith_warnings_mute_push/pop` around
  it, so the pole collapses to `ComplexInfinity` / `Indeterminate` and is caught
  silently. Cases like `Series[Gamma[x], {x, Infinity, 2}]` and
  `Series[ExpIntegralEi[x], {x, 0, 3}]` now return unevaluated with no stray
  warning.

Regression test `test_series_expintegralei_at_infinity` added; `series_tests`
passes, 0 leaks under macOS `leaks`.

## Fix: `ExpIntegralEi` large-|z| abort + asymptotic expansion (2026-06-13)

`N[ExpIntegralEi[-10^60 I] + I Pi, 20]` aborted with an `mpfr_init2`
assertion (`init2.c:53`). The complex and real-negative numeric kernels
sized their MPFR working precision as `out_prec + (long)(|z|/ln2)` guard
bits; for `|z| ≈ 1e60` the cast to `long` overflowed, producing a bogus
(out-of-range) precision passed to `mpfr_init2`. The convergent series the
guard bits served is in any case infeasible at that magnitude (it needs
~`2|z|` terms).

Fix adds an **asymptotic-expansion path** (DLMF 6.12.2) used whenever `|z|`
is large enough that its smallest term sits below the target precision —
exactly the regime where the convergent series breaks down:

- `Ei(z) ~ (e^z/z) Σ_{k≥0} k!/z^k + i π sign(Im z)`, summed to the smallest
  term (optimal truncation). The `i π sign(Im z)` constant is the principal
  branch jump across the cut, so `ExpIntegralEi[±I Infinity] = ±I Pi` is
  recovered as a limit.
- New file-local helpers: `ecx_exp`, `ecx_div` (complex `exp`/divide in the
  existing `ecx` MPFR pair toolkit), `ei_ecx_asymptotic` (complex) and
  `ei_real_asymptotic` (real).
- The routing threshold (`|z| > (out_prec + 24)·ln2 + 8`) also keeps the
  convergent branch's working precision bounded, so the overflow cannot
  recur. The convergent path is otherwise unchanged.

Cross-checked: asymptotic and convergent routes agree on the overlap
(`ExpIntegralEi[-50 I] = -0.00562839 − 3.12241 I` from either side), with
correct branch signs in all quadrants. New `test_ei_large_magnitude`
regression test; zero leaks (macOS `leaks`).

## Feature: `NResidue` — numerical residue by contour integration (2026-06-13)

Opens a new functionality group, **numerical calculus**, under
`src/numerical_calculus/`. The first member is `NResidue[expr, {z, z0}]`,
which numerically estimates the residue of `expr` at `z = z0` (the
coefficient of `(z - z0)^-1` in the Laurent expansion) by integrating
around a small circle in the complex plane. Unlike the symbolic `Residue`
(which needs a power series at `z0`), `NResidue` handles **essential
singularities** such as `Exp[1/x]` and `Sin[1/x]`.

Two modules:

- **`quadrature.{c,h}`** — a reusable, expression-agnostic
  periodic-trapezoidal contour-integration core (designed so a future
  `NIntegrate` can reuse it). For a function analytic on the circle the
  trapezoidal rule converges geometrically (Trefethen & Weideman, *SIAM
  Review* 2014). It doubles `N` (reusing samples) until `|S_{2N} - S_N|`
  meets the precision goal, applies Aitken/Shanks extrapolation to the
  doubling sequence, and runs at machine precision (`double _Complex`) or
  in MPFR complex arithmetic (a file-local `qcx` pair-of-`mpfr_t` toolkit,
  mirroring `erf.c`'s `ecx`).
- **`nresidue.{c,h}`** — `builtin_nresidue` + `nresidue_init`. Parses the
  `{z, z0}` spec like `D[f, {x, n}]`, peels trailing options, and samples
  the integrand via Block-style temporary `OwnValues` (as `FindRoot` does).

Registered from `core_init` via `nresidue_init`; `SYM_NResidue`,
`SYM_Radius`, and `SYM_MaxRecursion` added to `sym_names.{c,h}`; docstring
in `info.c`. Attribute: **`Protected`** only — list-threading over arg 1 is
done manually (so the `{z, z0}` spec is never split), which is exactly why
`NResidue` is *not* `ATTR_LISTABLE`.

Options: `Radius` (default `1/100`, or `Automatic`), `WorkingPrecision`,
`PrecisionGoal`, `MaxRecursion` (default `10`), `Method` (`Trapezoidal`).

State-of-the-art additions beyond Mathematica's `NResidue`:

- **`Radius -> Automatic`** — a Fornberg/Bornemann-style adaptive search
  that favours fast-converging radii, removing the manual radius tuning the
  reference implementation needs (its own `Exp[1/x]` example fails at the
  default radius until you guess `Radius -> 1`).
- **Branch-cut detection** — a relative-jump + decay-rate diagnostic flags
  a cut that *crosses* the contour (`NResidue::bcut`) instead of silently
  returning garbage. A cut lying entirely inside the disk (e.g.
  `Sqrt[x-1] Sqrt[x+1]` on `Radius -> 2`, analytic on the circle) is
  undetectable on the contour and returns Mathematica's value.
- **Aitken/Shanks extrapolation** of the doubling sequence with a reported
  error estimate; an `NResidue::ncvi` message on non-convergence (the best
  estimate is still returned).

Examples (`docs/spec/builtins/calculus.md`):
`NResidue[1/x, {x, 0}]` → `1.`;
`NResidue[Sin[1/(10x)], {x, 0}] // Chop` → `0.1`;
`NResidue[{Exp[1/x], Sin[1/x], Cos[1/x]}, {x, 0}, Radius -> 1] // Chop` →
`{1., 1., 0}`;
`10! NResidue[Zeta[x]/x^11, {x, 0}, Radius -> 1/2, WorkingPrecision -> 30]` →
`Derivative[10][Zeta][0]`. Tests: `tests/test_nresidue.c` (15 cases;
valgrind clean — only the documented macOS dyld/Accelerate baseline noise).

## Feature: `Beta` — Euler, incomplete, and generalized beta function (2026-06-13)

`Beta` was previously a bare interned symbol with no kernel implementation
(every form stayed unevaluated). It is now a first-class special function
across all three Wolfram argument forms, implemented in
`src/special_functions/beta.{c,h}` (registered from `core_init` via
`beta_init`, with `SYM_Beta` added to `sym_names.{c,h}`).

- **`Beta[a, b]`** — Euler beta `B(a,b) = Gamma[a]Gamma[b]/Gamma[a+b]`, reduced
  through the existing `Gamma` builtin so it inherits all of `Gamma`'s exact
  reductions. When one argument is a positive integer `n` the gamma ratio
  collapses to `(n-1)!/Pochhammer[b, n]`, giving exact rationals even for
  rational partners (`Beta[2,3]=1/12`, `Beta[5,4]=1/280`, `Beta[3,1/3]=27/14`,
  `Beta[7/2,2]=4/63`); half-integers fold to rational multiples of `Pi`
  (`Beta[5/2,7/2]=3/256 Pi`); other exacts keep the residual gamma form
  (`Beta[1/3,1/3]=Gamma[1/3]^2/Gamma[2/3]`). Machine / MPFR real and complex
  arguments evaluate numerically with precision tracking.
- **Pole structure.** A surviving gamma pole gives `ComplexInfinity`
  (`Beta[0,b]`, `Beta[a,0]`, `Beta[0,n]`, `Beta[-2,5]`, `Beta[Infinity,0]`);
  a cancelling pair of poles reduces by the finite limit of the gamma ratio
  (`Beta[-2,1]=-1/2`, `Beta[2,-5]=1/20`).
- **`Beta[z, a, b]`** — incomplete beta via
  `B_z(a,b) = z^a/a * Hypergeometric2F1[a, 1-b, a+1, z]`: special values
  `Beta[0,a,b]=0`, `Beta[1,a,b]=Beta[a,b]`, `Beta[z,a,1]=z^a/a`; a
  positive-integer `b` terminates the series to an exact closed form
  (`Beta[1/2,3,4]=7/640`), and fully numeric calls evaluate to a number.
- **`Beta[z0, z1, a, b]`** — generalized incomplete beta, evaluated as
  `Beta[z1,a,b] - Beta[z0,a,b]` when all arguments are numeric.
- **Derivatives** (in `src/calculus/deriv.c`):
  `D[Beta[a,b],a] = Beta[a,b](PolyGamma[0,a]-PolyGamma[0,a+b])` (symmetric in
  `b`, higher orders composing through `PolyGamma`);
  `D[Beta[z,a,b],z] = z^(a-1)(1-z)^(b-1)` with the inert
  `Derivative[0,1,0]/[0,0,1][Beta]` partials in `a`, `b`; and the endpoint
  integrands `±z^(a-1)(1-z)^(b-1)` for the four-argument form.
- **Attributes** `Listable, NumericFunction, Protected, ReadProtected`;
  docstring in `info.c`; arity diagnostic `Beta::argb` for fewer than 2 or more
  than 4 arguments. Extensive unit coverage in `tests/test_beta.c`; valgrind
  shows no new leaks (identical totals to the `Sin[1.0]` baseline).

## Feature: `FullSimplify` + runtime module loading (`LoadModule`) (2026-06-11)

First cut of `FullSimplify`, implemented in the **Mathilda language**
(`src/internal/simp/FullSimplify.m`) rather than C, plus the runtime
`.m`-module-loading framework it relies on.

- **Module loader (`src/loadmodule.{c,h}`).** New `mathilda_load_module(relpath)`
  resolves an internal module against `$MATHILDA_HOME` then a relative ladder
  (`src/internal/`, `../src/internal/`, `../../src/internal/`,
  `../../../src/internal/`), so the same load works from the REPL (repo root) and
  the CMake test binaries (`tests/build/`). Loads each module **once**. The
  file-read/parse/eval loop was lifted out of `Get` into the shared
  `mathilda_run_file`; `builtin_get` (`src/readwrite.c`) is now a thin wrapper.
  New builtin **`LoadModule["relpath"]`** (`Protected`, returns `True`/`False`)
  exposes it to Mathilda code. `core_init` eager-loads the FullSimplify driver
  last (after every builtin it calls is registered) and applies `Protected`
  afterwards so the `.m` can define its own DownValues. Generalises the bespoke
  three-path CRC-table fallback and fixes the "tests never load any `.m`" gap.
- **`FullSimplify[expr]` / `FullSimplify[expr, assum]`** — a wrapper around
  `Simplify`: a relevance engine scans `expr` for the function heads it contains
  (`Cases[expr, _Symbol, {0, Infinity}, Heads -> True]`), lazily loads the
  identity library registered for each head, and feeds the union of their
  transformation functions to `Simplify` via
  `TransformationFunctions -> {Automatic, ...}`. Because `Simplify`'s user-
  transform pass only ever keeps a strictly-lower-complexity candidate,
  **FullSimplify is automatically at least as simple as Simplify** (no second
  pass needed). `Protected`; docstring set C-side in `core.c`.
- **Relevance-driven, lazily-loaded identity libraries** under
  `src/internal/simp/transforms/`, loaded only when a matching head appears so
  the collection scales without taxing the common case: `gamma.m`
  (`Gamma[z+1] -> z Gamma[z]`, `LogGamma`/`PolyGamma` recurrences,
  `Pochhammer`/`Beta`/`Factorial -> Gamma`), `erf.m` (`Erf[z]+Erfc[z] -> 1`),
  `polylog.m` (dilogarithm duplication
  `PolyLog[2,z]+PolyLog[2,-z] -> PolyLog[2,z^2]/2`, reflection), `powerradical.m`
  (`Surd[x,n]^n -> x`). Closes gaps Simplify alone leaves open, e.g.
  `FullSimplify[Gamma[x+1]/Gamma[x]] -> x`,
  `LogGamma[x+1]-LogGamma[x] -> Log[x]`, `Erf[x]+Erfc[x] -> 1`,
  `Pochhammer[a,n]/Gamma[a+n] -> 1/Gamma[a]`.
- **Options:** `ComplexityFunction -> f` and `TransformationFunctions -> {…}`
  (user transforms merged with the relevance set; `Automatic` always retained)
  forwarded to `Simplify`; `TimeConstraint -> {tLoc, tTot}` (bare `t` = `{t,t}`,
  default `Infinity`) wraps each transform in a per-transform `TimeConstrained`
  budget (injected via `With[]`, since a nested `Function` does not close over
  the wrapped function) and the whole call in a total budget whose plain-Simplify
  fallback is the HoldAll-protected third argument — so it is computed only on
  timeout, never doubling work.
- **Tests.** `tests/test_fullsimplify.c` (framework: genuine gaps, ≥-Simplify
  sanity, assumptions, all three option families, protection) and
  `tests/test_fullsimplify_corpus.c` + `tests/fullsimplify_corpus.m` (per-rule
  corpus, 9 cases, modelled on `test_crc_corpus.c`). `src/loadmodule.c` added to
  the CMake `COMMON_SRC`; `readwrite_tests` and `simplify_tests` unaffected.
  Builds clean under `-std=c99 -Wall -Wextra`.
- **First-cut limitations** (documented in
  `docs/spec/builtins/simplification.md`): the gamma recurrence matches a literal
  `+1` shift only; an identity over an `Orderless` sum must match the whole sum
  (no partial-sum matching, so `a + Erf[x] + Erfc[x]` is not collapsed). The
  `Infinity::indet` notice on `FullSimplify[Gamma[x+1]-x Gamma[x]]` originates in
  Simplify's own built-in pipeline (plain `Simplify` emits it too) and is
  harmless — the result is `0`.

## Feature: `LogIntegral` (logarithmic integral) (2026-06-11)

New builtin implementing the logarithmic integral li(z), the principal value of
∫₀^z dt/ln t, with a branch cut on (−∞, +1). Implemented from the identity
**li(z) = Ei(Log z)** in `src/special_functions/logintegral.c` (with
`logintegral.h`), reusing the `ExpIntegralEi` numeric kernels in full (no
duplicated numeric code). Internal symbol in `src/sym_names.c`, registration in
`src/core.c`, docstring in `src/info.c`, derivative rule in
`src/calculus/deriv.c`. Tests in `tests/test_logintegral.c` (10 groups: exact,
symbolic, machine & MPFR real, machine & MPFR complex, derivatives, Listable,
attributes, arity). Documented in `docs/spec/builtins/special-functions.md`.
Valgrind-clean (byte-identical to the macOS `Sin[1.0]` baseline; no
`logintegral`/`expintegralei` frames in any leak stack). Reference values
cross-checked against the running binary and the task specification.

- **`LogIntegral[z]`** — single-argument only; attributes
  `{Listable, NumericFunction, Protected}`. Wrong arity emits `LogIntegral::argx`.
  - Exact: `LogIntegral[0] = 0`, `LogIntegral[1] = -Infinity`,
    `LogIntegral[Infinity] = Infinity`; `ComplexInfinity` / `Indeterminate` →
    `Indeterminate`. Exact non-special arguments (`LogIntegral[2]`,
    `LogIntegral[1/2]`) stay symbolic; numeric values come from a `Real`/MPFR
    argument or from `N`.

## Build: Perl-free ECM build path / graceful autoreconf fallback (`build_ecm.sh`) (2026-06-10)

Removed Perl from the *normal* ECM compilation path and made the one remaining
Perl-only step fail gracefully. Investigation showed that the only Perl consumer
in the ECM build is the `autoreconf -i` bootstrap (it drives `autoconf` /
`automake` / `libtoolize`, which are Perl programs). The generated `configure`
script, libtool, ECM's `Makefile.am`, and `make` itself are all pure POSIX
shell / m4 and need no Perl — verified by building `libecm.a` end-to-end on a
host with a broken Perl, with `configure` already present.

`build_ecm.sh` now:
- Only runs `autoreconf` when `configure` is **absent** (unchanged), and when it
  must, first **probes `autoreconf --version`** so a stale/broken
  `#!/usr/bin/perlX.Y` shebang (kernel "bad interpreter", exit 126) is caught.
- On a missing/non-functional `autoreconf`, prints an actionable error (copy a
  pre-generated build system from a working machine, or install Perl + autotools,
  or `make USE_ECM=0`) instead of the cryptic interpreter crash.

Net effect: any checkout that already carries a generated `configure` (the common
case, and what GMP-ECM release tarballs ship) builds ECM with **zero Perl**. A
fresh clone still needs Perl once to generate `configure`; that gap is now a
clear message rather than an opaque failure.

## Feature: `Unset` (`lhs =.`) (2026-06-10)

New builtin to remove a single definition by left-hand side, the inverse of
`Set`. Implemented in `src/core.c` (`builtin_unset`) with a new
`symtab_remove_matching_rule()` primitive in `src/symtab.c`; docstring in
`src/info.c`, internal symbol in `src/sym_names.c`, and the postfix `=.`
operator in `src/parse.c`. Tests in `tests/test_unset.c` (20 cases, parsing +
semantics + memory hygiene).

- **`Unset[lhs]` / `lhs =.`** — removes any rule whose LHS is `lhs`. A bare
  symbol clears its `OwnValue`; a function form clears the matching `DownValue`
  on the head symbol. Attributes `{HoldFirst, Protected}`. Always returns `Null`.
- **Rule matching is up to renaming of bound pattern variables**
  (alpha-equivalence): `f[x_] =.` removes a rule defined as `f[y_] := ...`, while
  a specific rule like `fact[1] =.` leaves the general `fact[n_]` rule intact.
  `symtab_remove_matching_rule()` canonicalizes the query LHS exactly as
  `add_rule` did, then alpha-normalizes both sides (Pattern variables rewritten
  to positional placeholders) before comparing with `expr_eq`.
- **Parser.** `=.` is a low-precedence postfix operator (precedence 40, like
  `Set`) so `a b =.` parses as `Unset[a b]`; a trailing-digit guard keeps
  `k =.5` parsing as `Set[k, 0.5]`.
- **Guards.** `Protected`/`Locked` symbols are never modified (so built-ins are

## Feature: continuous Weierstrass-substitution integrator (`Integrate`Weierstrass`, 2026-06-09)

New integration method implementing the Jeffrey–Rich continuous Weierstrass
substitution (*The Evaluation of Trigonometric Integrals Avoiding Spurious
Discontinuities*, ACM TOMS 20(1), 1994), generalised to hyperbolic integrands.
New files `src/calculus/integrate_jeffrey.{c,h}`; tests in
`tests/test_integrate_jeffrey.c`.

- **What it does.** Integrates a rational function of the trigonometric kernels
  `Sin/Cos/Tan/Cot/Sec/Csc[x]` (or the hyperbolic kernels
  `Sinh/Cosh/Tanh/Coth/Sech/Csch[x]`) by substituting `u = Tan[x/2]`
  (`Tanh[x/2]` for hyperbolic), integrating the resulting rational function of
  `u` via `Integrate`BronsteinRational`, and back-substituting.
- **Continuity correction (trig).** Adds the secular term `K Floor[(x - b)/p]`
  (`b = Pi`, `p = 2 Pi`) that removes the spurious jump discontinuities the
  classical substitution introduces at the poles of `Tan[x/2]`.  `K` is the
  jump, computed as the difference of the one-sided limits of the
  `u`-antiderivative at `±Infinity` (summed term-by-term, since the core `Limit`
  engine does not distribute over `Plus`/`Times` at infinity; `ArcTan`/`Log`
  limits are resolved through the limit of their inner argument).  A genuine
  singularity (divergent limit) drops the correction — paper §4.  Example:
  `Integrate[3/(5 - 4 Cos[x]), x]` → `2 ArcTan[3 Tan[x/2]] + 2 Pi Floor[(x-Pi)/(2 Pi)]`
  (paper eq. 10); `Integrate[1/(2 + Cosh[x]), x]` (previously unevaluated) →
  `2 ArcTanh[Tanh[x/2]/Sqrt[3]]/Sqrt[3]`.
- **Hyperbolic needs no `Floor`.** `Tanh[x/2]` is a smooth monotone bijection
  `R -> (-1,1)` with no poles, so the substitution introduces no spurious
  discontinuity and no correction is applied.
- **`TrigExpand` pre-pass.** When the verbatim integrand does not match (a kernel
  argument is a multiple/sum of `x`, e.g. `Cosh[x] Cosh[2 x]`), the method retries
  on `TrigExpand[f]`, which reduces those to kernels of the bare variable.
- **Cascade placement.** Runs in the Automatic cascade **ahead of**
  `DerivativeDivides` and `RischNorman` — it is domain-specific, deterministic,
  and correct by construction, so it yields a real, continuous antiderivative
  rather than pmint's complex-logarithm form.  Gated in Automatic mode to genuine
  rational integrands (a kernel in a denominator), so polynomial trig such as
  `Integrate[Sin[x], x]` keeps its cleaner table form.
- **Exposure.** `Method -> "Weierstrass"` and the package head
  `Integrate`Weierstrass[f, x]` (no denominator gate, `Protected`,
  `ReadProtected`).  Correct by construction — no differentiate-back gate (the
  `Floor` term defeats symbolic `D`); verified in tests via
  `PossibleZeroQ[D[Integrate[f,x] /. Floor[_] -> 0, x] - f]`.  No new memory
  leaks (valgrind: leak stacks free of `integrate_jeffrey` frames).

## Fix: `PossibleZeroQ` false positives on branch-dependent non-zeros (2026-06-09)

`PossibleZeroQ[D[2 Sqrt[1 - Cos[x]], x] - Sqrt[1 + Cos[x]]]` returned `True`
~6.5 % of the time and `False` otherwise (13/200 over a `Table`). The
flakiness leaked into `Integrate[…, Method -> "DerivativeDivides"]`, whose
verification gate `differentiates_back`
(`src/calculus/integrate_derivdivides.c`) accepts a candidate antiderivative on
`PossibleZeroQ[D[r,x] - f] === True` — so identical `Integrate` calls returned
unevaluated on one line and `2 Sqrt[1 - Cos[x]]` on the next.

- **Root cause (`src/zero_test.c`).** The expression is **not** an identical
  zero: it vanishes on `(0, π)`, `(2π, 3π)`, … but is genuinely non-zero on
  `(π, 2π)`, `(3π, 4π)`, … (the two `Sqrt` factors live on different analytic
  branches, split by branch points at the zeros of `1 ± Cos[x]`). `False` is the
  correct verdict. The Schwartz–Zippel stage drew only 4 samples and returned
  `True` iff all 4 read zero; each random real part lands in a zero interval ~½
  the time, so `P(all 4 zero) ≈ (1/2)^4 = 6.25 %` — exactly the observed rate. A
  small imaginary perturbation does not help: by the identity theorem the
  function is zero on a whole complex neighbourhood of each zero interval, so
  only the sample's **real-part branch interval** matters — more sample
  *points*, not complex ones, defeat it.

- **Fix.** `decide_schwartz_zippel` is now two-phase, exploiting the measured
  cost asymmetry (a non-zero branch sample is `O(1)` at machine precision and
  rejected at ladder rung 0; confirming a genuine identity needs the full MPFR
  ladder per sample). A **screen** phase (`ZT_SCREEN_SAMPLES = 24`) evaluates
  many points at machine precision only and settles `False` on the first
  decisively non-zero point; a **confirm** phase (`ZT_CONFIRM_SAMPLES = 4`) then
  climbs the full precision ladder on a few points to reject
  cancellation-hidden small non-zeros before declaring an identity. A shared
  `sz_trial` helper drives both. The half-and-half branch case now false-
  positives with probability ~`(1/2)^24`.

- **Result.** `Tally[Table[PossibleZeroQ[D[2 Sqrt[1 - Cos[x]], x]
  - Sqrt[1 + Cos[x]]], {200}]]` → `{{False, 200}}`;
  `Integrate[Sqrt[1 + Cos[x]], x, Method -> "DerivativeDivides"]` is now
  deterministically unevaluated (the candidate is branch-wrong — its derivative
  is `-Sqrt[1 + Cos[x]]` on half the real line — so the gate correctly rejects
  it). Genuine identities are unaffected (`Sin[x]^2 + Cos[x]^2 - 1`,
  `Sqrt[Tan[x]]` DerivativeDivides still close). Throughput: branch non-zeros
  ~3× faster (machine-precision short-circuit instead of climbing the ladder);
  genuine identities +~0.8 ms/call (24 extra cheap screens; the ladder still
  runs only 4×).

Tests: `test_zero_test.c` (`stage3_branch_dependent_nonzero`, sweeping 8 seeds)
plus the existing Stage-3 identities; `zero_test_tests` and
`integrate_derivdivides_tests` pass; valgrind clean (no `zero_test.c` frames in
any lost stack; totals match the documented macOS dyld/Accelerate baseline).

## LogGamma: full implementation (2026-06-09)

`LogGamma[z]` is no longer an inert placeholder — it is now a fully evaluating
builtin (`src/loggamma.c`, `Listable | NumericFunction | Protected`), modelled
on `Gamma`. `LogGamma` is the analytic log-gamma (single branch cut on the
negative reals), distinct from `Log[Gamma[z]]`, and stays finite where `Gamma`
overflows.

- **Exact closed forms.** Integers reduce as `LogGamma[n] = Log[(n-1)!]`
  (`LogGamma[5] → Log[24]`); positive half-integers give `Log` of the exact
  `Sqrt[Pi]` form; negative half-integers add the branch term
  `-Ceiling[-z] Pi I`, e.g. `LogGamma[-3/2] → -2 I Pi + Log[(4 Sqrt[Pi])/3]`.
  Non-positive integers diverge to `Infinity`.
- **Symbolic infinities.** `Infinity → Infinity`, `-Infinity → Indeterminate`,
  `I Infinity` / `ComplexInfinity → ComplexInfinity`.
- **Numerics.** Machine real via libm `lgamma`; arbitrary-precision real via
  MPFR `lgamma`; machine complex via a Lanczos log-series; arbitrary-precision
  complex via a **Stirling series with argument reduction** — the continuous
  branch, whose imaginary part grows past π where a single `Log[Gamma]` would
  wrap. Negative reals return `log|Γ(z)| - Pi Ceiling[-z] I`.
- **Calculus.** `D[LogGamma[z], z] = PolyGamma[0, z]` (new rule in
  `deriv.c`); higher derivatives raise the `PolyGamma` order.

Docstring updated in `info.c`; registration moved out of `polygamma_init` into
the new `loggamma_init`. Tests in `tests/test_loggamma.c`.

## PolyLog: zeta-expansion speedup for real order (2026-06-09)

`pl_zeta_cx` (the `zeta(s-k)` evaluator inside the Jonquière/zeta expansion) now
applies the functional equation itself for a **real** argument in the left
half-plane (`w < 1/2`, away from integers), reusing the fast positive-side

## PolyLog: the polylogarithm (2026-06-09)

New builtin `PolyLog` (`src/polylog.{c,h}`), modelled on the `Gamma` / `Zeta`
modules. Attributes `Listable`, `NumericFunction`, `Protected`. Registered in
`core.c`, `sym_names.{c,h}`, with a docstring in `info.c`; tests in
`tests/test_polylog.c`.

- **`PolyLog[n, z]` — the polylogarithm** Liₙ(z) = Σ_{k≥1} zᵏ/kⁿ.
  `PolyLog[n, p, z]` (Nielsen generalized polylogarithm) is accepted but left
  symbolic; other argument counts raise `PolyLog::argt`.
- **Exact closed forms.** `PolyLog[n, 0] = 0`; `PolyLog[1, z] = -Log[1-z]`;
  `PolyLog[0, z] = z/(1-z)`; negative integer orders give Eulerian-number
  rational functions (`PolyLog[-2, z] = (z+z²)/(1-z)^3`), built from the Eulerian
  triangle with GMP; for integer `n ≥ 2`, `PolyLog[n, 1] = Zeta[n]` and
  `PolyLog[n, -1] = (2^(1-n)-1) Zeta[n]`; and the famous
  `PolyLog[2, 1/2] = Pi²/12 - Log[2]²/2`,
  `PolyLog[3, 1/2] = Log[2]³/6 - Pi² Log[2]/12 + 7 Zeta[3]/8`.
- **Numerics (MPFR).** A direct power series for `|z| ≤ 1/2` (with a pure-real
  fast path for real order and real −1 < z < 1) and the Jonquière / zeta
  expansion for 1/2 < |z| with |ln z| < 2π cover machine- and
  arbitrary-precision real and complex inputs, e.g. `PolyLog[2, 0.9] = 1.29971`,
  a 100-digit `PolyLog[3.0`100, 0.5`100]` matching Mathematica to all digits, and
  `PolyLog[.2 + I, .5 - I] = -0.0898526 - 0.595865 I`. Complex-order ζ values use
  the ζ functional equation in the left half-plane to avoid the catastrophic
  cancellation of a raw Euler–Maclaurin sum at large negative real part. The
  branch cut runs from 1 to ∞ and is taken continuous from below, matching
  Mathematica (Im Li₂(x) = −π Log[x] for real x > 1; the branch point
  `PolyLog[2, 1.0]` is the finite real `Pi²/6`).
- **Derivatives.** `src/calculus/deriv.c` gains
  `D[PolyLog[n, z], z] = PolyLog[n-1, z]/z`, with the order derivative left as the
  generic `Derivative[1, 0][PolyLog][n, z]`.
- **Listable**, including the empty list: `PolyLog[{2, 4}, -1]`,
  `PolyLog[{}, x] = {}`.
- Valgrind-clean (no leaks beyond the macOS dyld/Accelerate baseline).

## EulerE: Euler numbers and polynomials (2026-06-09)

## Integrate: strip the constant of integration from Risch-Norman results (2026-06-09)

The Risch-Norman (Bronstein "poor man's integrator") path could leave a stray
additive constant in its answer — its parametric linear solve has an arbitrary
constant in the homogeneous direction. For example `Integrate[x Sin[x^2], x]`,
when it reached this fallback, returned `1/2 (-1 - Cos[x^2])` instead of
`-1/2 Cos[x^2]`.

- **Fix.** `rischnorman_integrate` (`src/calculus/intrischnorman.c`) now ends
  with `strip_constant_of_integration`: it `Expand`s a copy of the result so a
  scalar `c·(… + const)` distributes, then keeps only the summands that depend
  on the integration variable `x`. An antiderivative is defined only up to an
  additive constant, so any `FreeQ[·, x]` summand is removable.
- **Conservative.** The rewrite is applied only when a constant is actually
  dropped *and* an `x`-dependent part remains; otherwise the original (often
  more compact) form is preserved, so results with no spurious constant are
  untouched.

Tests: `intrischnorman_tests` gains `test_phase7_no_const_x_sin_x2` /
`_x_cos_x2`, which assert both derivative-correctness and that `Expand[result]`
has no `x`-free summand. Full `intrischnorman`, `integrals`, and
`integrate_dispatch` suites pass.

## Special functions: `PolyGamma` (digamma / polygamma) + inert `LogGamma` (2026-06-08)

Added `PolyGamma[z]` and `PolyGamma[n, z]` — the digamma ψ(z) and the n-th
polygamma ψ⁽ⁿ⁾(z) — modelled on the `Gamma` implementation
(`src/polygamma.c`, registered in `src/core.c`, symbols in `src/sym_names.c`,
docstrings in `src/info.c`). Attributes: `Listable`, `NumericFunction`,
`Protected`.

- **Canonicalisation.** `PolyGamma[z]` rewrites to `PolyGamma[0, z]`.
- **Special values.** Non-positive integer arguments → `ComplexInfinity`;
  `PolyGamma[0, Infinity]` → `Infinity`, `PolyGamma[n, Infinity]` → `0` (n ≥ 1);
  `ComplexInfinity`/`Indeterminate` arguments → `Indeterminate`.

  forms at non-integer rationals (`PolyGamma[0, 1/2]`), and dedicated `Series`
  expansions (only the generic `D`-based Taylor fallback applies). A future

## Feature: `GoldenAngle`, `Glaisher`, `Khinchin` constants (2026-06-08)

Three more first-class mathematical constants, all `{Constant, Protected}` with
docstrings (`src/info.c`) and arbitrary-precision `N`. They join the central
numeric constant table (`src/numeric.c`), so `numeric_init` stamps their
attributes automatically, and they were added to the constant lists consulted

(`src/zero_test.c`). `GoldenAngle` is newly interned in `src/sym_names.c`;
`Glaisher`/`Khinchin` were already interned.

- **`GoldenAngle`** = `(3 − Sqrt[5]) Pi` = `2 Pi / GoldenRatio^2` ≈ 2.39996 rad
  (137.5°). MPFR filler evaluates the closed form directly.

- **`Glaisher`** — Glaisher–Kinkelin constant *A* ≈ 1.2824271291, satisfying
  `Log[A] = 1/12 − ζ'(−1)`. MPFR has no zeta-derivative primitive, so
  arbitrary precision is computed from
  `ln A = (γ + ln 2π)/12 − ζ'(2)/(2π²)`, with `ζ'(2) = −Σ ln(n)/n²` evaluated by
  Euler–Maclaurin summation; the even Bernoulli numbers it needs are obtained
  from `mpfr_zeta_ui(2k)` (so `B_{2k}/(2k)!` reduces to a `ζ(2k)` expression),
  and the divergent asymptotic correction series is truncated at its smallest
  term.

- **`Khinchin`** — Khinchin's constant *K* ≈ 2.6854520011, the limiting
  geometric mean of continued-fraction partial quotients. Computed from the
  geometrically convergent Bailey–Borwein–Crandall zeta series
  `ln K · ln 2 = Σ_{n≥1} (ζ(2n) − 1)/n · Σ_{k=1}^{2n−1} (−1)^(k+1)/k`.

## Feature: `Gamma` — Euler & incomplete gamma functions (2026-06-08)

New builtin `Gamma` (`src/gamma.c`), covering the full Mathematica signature:

- `Gamma[z]` — Euler gamma function Γ(z).
- `Gamma[a, z]` — upper incomplete gamma Γ(a, z) = ∫_z^∞ tᵃ⁻¹ e⁻ᵗ dt.
- `Gamma[a, z0, z1]` — generalized incomplete = Γ(a, z0) − Γ(a, z1).

Attributes: `Listable`, `NumericFunction`, `Protected`. Docstring in
`src/info.c`; symbol already interned in `src/sym_names.c`.

Evaluation is layered for efficiency across all three argument regimes:

- **Exact symbolic.** Integer and half-integer `Gamma[z]` reduce to `(z-1)!`
  by reusing the existing `Factorial` machinery (exact int64, GMP BigInt, and
  rational·`Sqrt[Pi]` half-integers). Non-positive integers give
  `ComplexInfinity`. `Infinity`/`-Infinity`/`ComplexInfinity` map to

```
Gamma[5/2]                  -> 3/4 Sqrt[Pi]
Gamma[-1/2]                 -> -2 Sqrt[Pi]
Gamma[0]                    -> ComplexInfinity
```

  approximation (g = 7) with reflection for the left half-plane.

```
Gamma[2.2]      -> 1.1018
Gamma[2.3 + I]  -> 0.719141 + 0.540614 I
```

- **Arbitrary precision.** Real → MPFR `mpfr_gamma`; incomplete real (machine


## Feature: `Pochhammer` — the rising factorial (2026-06-08)

New builtin `Pochhammer` (`src/pochhammer.c`): the Pochhammer symbol
(a)ₙ = a (a+1) … (a+n-1) = Γ(a+n)/Γ(a). Attributes `Listable`,
`NumericFunction`, `Protected`; docstring in `src/info.c`; symbol interned in
`src/sym_names.c`; wired from `core_init()` next to `gamma_init()`.

Following the `Gamma` precedent, the implementation holds almost no numeric
code of its own — it delegates to two existing, fully-tested mechanisms:

- **Exact integer order → a `Times` product.** For an integer `n`
  (|n| ≤ 1000) it builds `Times[a, a+1, …, a+n-1]` (or the reciprocal for
  negative `n`) and evaluates it. That one path collapses to an exact
  Integer/Rational for numeric `a`, preserves MPFR precision and complex
  arithmetic, and stays a polynomial product for symbolic `a`.

  `Gamma[a+n]/Gamma[a]`, reusing Gamma's exact half-integer reductions, its
  libm/MPFR real paths and its machine-complex Lanczos path. A residual-`Gamma`
  guard keeps genuinely-symbolic inputs unevaluated.

`Pochhammer[0, 1285] = 0` without forming a huge factorial),
`Pochhammer[Infinity, n>0] = Infinity`. Wrong argument counts emit a
Mathematica-compatible `Pochhammer::argrx` diagnostic and stay unevaluated.

```
Pochhammer[n, -5]                 -> 1/((-5 + n) (-4 + n) (-3 + n) (-2 + n) (-1 + n))

Pochhammer[{2, 3, 5, 7, 11}, 3]   -> {24, 60, 210, 504, 1716}
```

## Fix: bare `N[expr]` no longer collapses high-precision numbers (2026-06-08)

`N[Pi, 100] // N` (i.e. `N[N[Pi, 100]]`) was returning a machine-precision
`3.14159` instead of the 100-digit value — `Precision` reported
`MachinePrecision`. The same precision loss hit every constant (`E`,
`GoldenRatio`, …) and any already-arbitrary-precision literal (`N[2.5`100]`).

Root cause: `numericalize`'s `EXPR_MPFR` branch (`src/numeric.c`) **always**
down-converted an MPFR value to a machine `double` under machine mode. That is
correct for inexact contagion (`1. + N[Pi,100]` must collapse to machine), but
wrong for bare `N`: Mathematica's `N[expr]` numericalizes only the **exact**
parts of `expr` and leaves the precision of already-approximate numbers
untouched.

Fix: a `preserve_inexact` flag on `NumericSpec` (`src/numeric.h`), set **only**
by the one-argument `N` builtin. When set, the machine-mode `EXPR_MPFR` branch
keeps the value at its existing precision (`expr_new_mpfr_copy`). Contagion
(`numeric_contagion_args`) and the explicit two-argument `N[expr, prec]` form
leave the flag clear, so their (correct) down-converting / precision-setting
behaviour is unchanged. The flag is read solely in the machine branch and is
initialised to `false` by `numeric_machine_spec()`, through which every
machine-mode spec in the codebase is built — so existing call sites are
unaffected.

```
N[Pi, 100] // N                  -> 3.1415926535897932384626…  (100 digits)
Precision[N[Pi, 100] // N]       -> 100.243
N[(1 + Sqrt[5])/2, 200] // N     -> 1.6180339887…              (200 digits)
N[2.5`100]                       -> Precision 100.243   (was: MachinePrecision)
N[Pi]                            -> 3.14159              (exact symbol → machine, unchanged)
1. + N[Pi, 100]                  -> 4.14159              (machine contagion, unchanged)
N[N[Pi, 100], 20]                -> 3.1415926535897932385 (explicit reduce to 20, unchanged)
```

`numeric_tests` pass.

## Integrate `Sqrt[Tan[x]]`: radical-trig Simplify, `PossibleZeroQ`, DerivativeDivides cascade (2026-06-09)

Closed the worked example `Integrate[Sqrt[Tan[x]], x]` end to end. The integral
is now found by the default cascade and by `Method -> "DerivativeDivides"`, and
its closed form `D[…, x] // Simplify` reduces back to `Sqrt[Tan[x]]` quickly
instead of hanging. The underlying enabler is a new family of algebraic
generators in the trig-rational normal form, plus two supporting fixes.

- **Trig-rational quadratic-radical generators (`src/simp/trigrat.c`).**
  `Simplify` can now normalise rational functions of a quadratic radical of a
  trig kernel — e.g. `Sqrt[Tan[x]]` (and `Tan[x]^(3/2)`, `1/Sqrt[Tan[x]]`, …),
  which are rational in `t = Sqrt[Tan[x]]` since `Tan = t^2`, `Sec^2 = 1 + t^4`.
  A kernel `Sqrt[g]` is treated as an algebraic element `l` with the relation
  `l^2 = g` (structurally identical to the trig "odd" generator `s` with
  `s^2 = 1 - c^2`): the existing ideal-reduction and conjugate-rationalisation
  machinery is reused, with radicals cleared from denominators before the trig
  generators. All half-integer powers of one `g` collapse to integer powers of
  one `l`. Works for non-trig radicands too (`Simplify[(x + Sqrt[x])/Sqrt[x]]`
  → `1 + Sqrt[x]`). Examples:
  ```
  Simplify[Tan[x]/Sqrt[Tan[x]]]                          -> Sqrt[Tan[x]]
  Simplify[(Sqrt[Tan[x]] + Tan[x]^(3/2))/(1 + Tan[x])]   -> Sqrt[Tan[x]]
  D[Integrate[Sqrt[Tan[x]], x], x] // Simplify           -> Sqrt[Tan[x]]   (was: hang)
  ```
  Previously `Sqrt[Tan[x]]` was an opaque atom unrelated to `Tan[x]`, so the
  normal form could not reduce it and `Simplify` fell through to `Factor`, which
  blew up factoring a multivariate rational over `Q(Sqrt[2])`. (Also fixed a
  pre-existing leak in the module's raw `Times`/`Power` constructors.)

- **`PossibleZeroQ` sampling (`src/zero_test.c`).** The Schwartz–Zippel sampler
  drew unbounded imaginary parts; for `Tan`/`Tanh` kernels, `|Im| ≳ 18` saturates
  the function to `±I`, where `1 + Tan^2` cancels below machine epsilon and a true
  zero reads as `O(1)`. Imaginary parts are now bounded in magnitude (`|Im| ≤ 8`,
  full denominator granularity preserved for the SZ guarantee; branch cuts depend
  on the sign, not the magnitude). Fixes false `False` verdicts on radical-trig
  identities such as `D[Integrate[Sqrt[Tan[x]], x], x] - Sqrt[Tan[x]] == 0`.

- **DerivativeDivides Eliminate/Solve in the default cascade
  (`src/calculus/integrate.c`, `integrate_derivdivides.{c,h}`).** The thorough
  Eliminate/Solve branch-search (which finds radical substitutions like
  `u = Sqrt[Tan[x]]`) now runs in the Automatic cascade as well as under the
  explicit method — the new `integrate_derivdivides_full` runs the fast direct
  quotient first and only then the search. So bare `Integrate[Sqrt[Tan[x]], x]`
  closes, not just `Method -> "DerivativeDivides"`.

- **Quiet internal `Eliminate` (`src/poly/eliminate.{c,h}`).** A new
  `eliminate_suppress_messages` flag mutes Eliminate's advisory `::ifun` / `::alg`
  / `::nlin` diagnostics while the integrator drives it as a private sub-step, so
  the user no longer sees branch caveats for an elimination they did not request.

Performance note: `Simplify` of the radical-trig derivative now returns in
~0.3 s (was a non-terminating hang). The residual cost is the general multivariate

## Fix: `Integrate[Sqrt[Cot[x]], x]` (radical-trig with an inverse odd generator) (2026-06-09)

`Integrate[Sqrt[Cot[x]], x]` raised `Power::infy` (`1/0`) three times and
returned unevaluated, while the symmetric `Integrate[Sqrt[Tan[x]], x]` closed.
It now closes to the expected mirror form

```
-ArcTan[-1 + Sqrt[2] Sqrt[Cot[x]]]/Sqrt[2] - ArcTan[1 + Sqrt[2] Sqrt[Cot[x]]]/Sqrt[2]
  - Log[1 + Cot[x] - Sqrt[2] Sqrt[Cot[x]]]/(2 Sqrt[2])
  + Log[1 + Cot[x] + Sqrt[2] Sqrt[Cot[x]]]/(2 Sqrt[2])
```

- **Root cause (`src/simp/trigrat.c`).** The integral's verification gate
  `Simplify[D[r,x] - Sqrt[Cot[x]]]` drives the trig-rational normal form. The
  radical kernel `Sqrt[Cot[x]]` carries `l^2 = Cot = Cos/Sin = c·s^(-1)` — a
  radicand that is *rational with the odd generator `s = Sin` in its
  denominator*. Reducing `l^2` then injects `Power[s, -1]` into the working
  numerator/denominator; the conjugate denominator-rationalisation clears `s`
  via the substitution `den|_{s -> 0}`, which evaluates `Power[0, -1]` — the
  literal `1/0`. `Simplify` returned `Indeterminate`, the differentiation gate
  failed, and `Integrate` gave up. (`Sqrt[Tan[x]]` escaped because its inverse
  generator is the *even* `c = Cos`, which is never substitute-zeroed.)

- **Fix.** Before the conjugate substitution, `tr_rationalise_denom` now clears
  any inverse power of an odd generator `s_i` (`tr_has_neg_sgen_power` /
  `tr_clear_neg_sgen`): it recombines `num/den` over a common denominator,
  re-splits, and re-reduces so both are polynomials in every `s_i`. This is
  reached only after the radical generators are cleared, so the re-reduction
  cannot reintroduce a fresh `.../s_i` term (no `l^2` remains to expand). The
  natural low-degree relation `l^2 = g` is kept (a `Sqrt[N D]/D` normalisation
  was rejected — it raises polynomial degree and made the case ~20× slower).
  The detector keys on odd-generator inverse powers only, so the `Sqrt[Tan[x]]`
  path (`c^(-1)` only, handled later by `Cancel`) is byte-for-byte unchanged.

The same clearing fixes `Csc` (`1/Sin`) radicands:
`Simplify[Cot[x]/Sqrt[Cot[x]]]` → `Sqrt[Cot[x]]`,
`Simplify[Csc[x]/Sqrt[Csc[x]]]` → `Sqrt[Csc[x]]`. `Integrate[Sqrt[Cot[x]], x]`
runs in ~1.6 s (comparable to the `Sqrt[Tan[x]]` case).

Tests: `test_trigrat.c` (`sqrt_cot_derivative`, `sqrt_cot_rational_combos`),
`test_integrate_derivdivides.c` (`radical_substitution_cot`). All
integrate/simp/trig/zero-test suites pass; valgrind clean (matches the
dyld/Accelerate baseline; no leak stack enters `trigrat.c`).

## Perf: DerivativeDivides verification gate drops the `Simplify` confirm (2026-06-09)

- **Symptom.** `Integrate[Sqrt[Tan[x]], x]` took ~1.5 s even though its
  decomposed steps are fast (`Eliminate` ~0.03 s, the rational integral
  ~0.03 s). Profiling the post-integration verification in
  `differentiates_back` (`src/calculus/integrate_derivdivides.c`) located the
  cost: `Simplify[D[r, x] - f] === 0` alone ran ~1.14 s, a heavy `trigrat`
  normalization of the back-substituted radical-trig difference.

- **Fix.** The verification gate now accepts a candidate antiderivative on
  `PossibleZeroQ[D[r, x] - f] === True` and no longer follows up with the
  rigorous `Simplify` (and `Cancel[Together[Expand[.]]]` fallback).
  `PossibleZeroQ`'s numeric sampling already settles these branches — including
  the entangled `Sqrt`/`Sec`/`Tan` forms — decisively, so the symbolic confirm
  was pure overhead. The gate also now hands its difference straight to
  `PossibleZeroQ` (no copy-then-free), trimming one allocation per branch.

- **Result.** `Integrate[Sqrt[Tan[x]], x]` drops from ~1.5 s to ~0.20 s
  (~7.6×) with a byte-identical answer; `Integrate[Sqrt[Cot[x]], x]` likewise
  drops to ~0.27 s. (Pre-existing and unrelated: `Integrate[x Sin[x^2], x]`
  hangs on both this and the prior revision — a separate DerivativeDivides bug,
  not touched here.)

Tests: `test_integrate_derivdivides.c` (all six cases pass); `leaks` reports
0 leaks on the `Sqrt[Tan[x]]` integral.

## Fix: DerivativeDivides no longer hangs on recursive substitutions (2026-06-09)

- **Symptom.** `Integrate[x Sin[x^2], x]` (and `Integrate[Sin[u]/2, u]`,
  `Integrate[x Cos[x^2], x]`, …) hung indefinitely. A stack sample showed an
  unbounded `dd_core → integrate_in → Integrate → dd_core` recursion in
  `src/calculus/integrate_derivdivides.c`. Two compounding causes: every
  reduced sub-integral `Integrate[g, u]` re-enters the *full* derivative-divides
  stage (the Automatic cascade calls `integrate_derivdivides_full`), and each
  level mints a fresh substitution variable so overlapping branches regenerate
  the *same* integrand without structural-equality ever noticing. With the
  `Eliminate`/`Solve` branch-search firing at every node this fans out into an
  exponential tree of `Eliminate + Solve + Factor + PowerExpand` calls — an
  effective hang even though the number of *distinct* integrands is tiny.

- **Fix.** Two guards in `dd_core`:
  1. **Integrand memo (loop guard).** Every integrand attempted during the
     current top-level descent is memoised, canonicalised by renaming the
     integration variable to a fixed sentinel so gensym'd duplicates compare
     equal. A canonical integrand seen before short-circuits to `NULL`
     (integrability is path-independent, so re-attempting it can only loop or
     repeat work). The memo is owned by the outermost frame and freed on its
     exit, so independent top-level integrals never share state. This makes the
     recursion provably terminate and collapses overlapping subproblems to one
     attempt apiece.
  2. **`Eliminate`/`Solve` search only at the outermost call.** The memo alone
     terminates but leaves a 7–18 s near-hang, because the heavyweight
     branch-search costs ~0.1–1 s per node. It only earns its keep on the
     *original* integrand (e.g. `u = Sqrt[Tan[x]]`, whose reduction is rational
     and is finished by the rational stage); reduced sub-integrals are better
     finished by the cheaper cascade stages. The cheap *direct-quotient*
     derivative-divides strategy still runs on every recursive sub-integral.

- **Result.** `Integrate[x Sin[x^2], x]` → `-Cos[x^2]/2` in ~0.004 s (was a
  hang); `Integrate[Sin[u]/2, u]` ~0.1 s; `Integrate[x Cos[x^2], x]` ~0.006 s.
  Radical substitutions are unchanged: `Integrate[Sqrt[Tan[x]], x]` ~0.16 s,
  `Integrate[Sqrt[Cot[x]], x]` ~0.25 s. `Integrate[Sin[x^2], x]` correctly
  stays unevaluated (non-elementary). This supersedes the "pre-existing and
  unrelated" caveat in the prior entry.

Tests: `integrate_derivdivides_tests` (6/6) plus `integrate_dispatch`,
`integrate_linrad`, `integrate_linratiorad`, `integrate_quadrad`,
`integrate_unknown`, and `integrals_tests` all pass; `leaks` reports 0 leaks
across the formerly-hanging integrals and `Sqrt[Tan[x]]`.

```
PolyLog[1/2, 1.3429`50]

```

## Fix: precision contagion in multi-argument special functions

Several special functions computed a numeric result's precision by taking the
**maximum** MPFR precision among their arguments and treating a machine-precision
`Real` as carrying *no* constraint. That violated Mathematica's contagion rule,
under which a machine-precision argument forces a machine-precision result:

```
PolyLog[N[1/2], 1.3429`50]
  was: -1.5227...0177 - 3.2642...2453*I   (Precision 50.272)
  now: -1.52272 - 3.26428*I              (MachinePrecision)
```

A new shared helper `numeric_min_inexact_bits` (in `src/numeric.c` /
`numeric.h`) returns the **minimum** precision among a value's inexact
(Real/MPFR) leaves — descending into `Complex[...]` — with a machine `Real`
counting as 53 bits and exact atoms (Integer/Rational/BigInt) imposing no
constraint. Each affected function now derives its working/output precision
from the min across its arguments, floored at machine (53 bits):

- **`PolyLog[n, z]`** (`polylog.c`)
- **`Zeta[s, a]`** Hurwitz zeta (`zeta.c`)
- **`Gamma[a, z]` / `Gamma[a, z0, z1]`** incomplete gamma, and the result-type
  decision now keys off the governing precision rather than "any operand is
  MPFR" (`gamma.c`)
- **`HypergeometricPFQ[{…}, {…}, z]`** — takes the MPFR path only when every
  inexact argument exceeds machine precision (`hypergeopfq.c`)
- **`Gamma[z]`, `LogGamma[z]`, `PolyGamma[n, z]`** complex paths — min across
  the real/imaginary parts (matters only for a mixed-precision `Complex`)

Exact-order calls are unchanged: `PolyLog[1/2, 1.3429`50]` and
`Zeta[2.0`50, 1.5`50]` still return 50-digit results. `BernoulliB[n, x]`,
`EulerE[n, x]`, and `Pochhammer[a, n]` were already correct — they build
symbolic expressions evaluated through standard arithmetic, which already
implements min-contagion. `gamma_tests`, `zeta_tests`, `polylog_tests`,
`hypergeopfq_tests`, `polygamma_tests`, `loggamma_tests`, and `numeric_tests`
pass.

## Perf: `Simplify` of high-generator trig rationals (~5× on the Weierstrass case, 2026-06-09)

`Simplify[D[ArcTanh[Tanh[x/2]] + 1/(1+Tanh[x/2]) - 1/(1+Tanh[x/2]^2+2 Tanh[x/2]), x]]`
(the derivative of a `Integrate`Weierstrass` result) took **~18.6 s** and was
routinely `Ctrl-C`'d; it now returns `1/2 + 1/2 E^(-2 x)` in **~3.7 s** with the
same answer. Root cause: the expression is a rational function in *several*
transcendental generators of related angles (`{Sinh[x/2], Cosh[x/2], Sinh[x],
Cosh[x], ...}`), and `Simplify`'s bottom-up search re-runs the multivariate
polynomial machinery (`PolynomialGCD`, `Factor`) at every node — both
exponential in the generator count. Four independent, correctness-preserving
fixes:

- **Multivariate-GCD explosion guard (`src/poly/poly.c`, `builtin_polynomialgcd`).**
  The recursive content/pseudo-remainder GCD is exponential in the number of
  generators (`poly_content` recurses one variable smaller for every
  coefficient). With `>= 3` generators and `> 200` total leaves, skip the
  polynomial GCD and return only the numeric/monomial content. Sound: the
  content is always a divisor of the true GCD, so `Cancel` under-reduces rather
  than reduces incorrectly — the same pessimism the existing pseudo-remainder
  and subresultant-PRS size budgets already rely on. Protects `Together` /
  `Cancel` / `Integrate` everywhere, not just `Simplify`.

- **Recursive `heuristic_factor` memoization (`src/poly/facpoly_heuristic.inc`).**
  `factor_via_z_independent_split` re-factored the *same* residual sub-polynomial
  along many variable-specialisation branches (144× on one identical 16-leaf
  trivariate poly here); `builtin_factor` cached only its top-level call, so the
  internal recursion bypassed the cache. `heuristic_factor` now consults the
  active `FactorMemo`. Keys are wrapped in a private `$HeuristicFactor` head so a
  (possibly less-complete) heuristic result never collides with `builtin_factor`'s
  own raw keys; stores are gated to shallow depth so a depth-cap-truncated result
  is never cached and served to a shallower caller.

- **`z`-split nesting cap (`src/poly/facpoly_z_split.inc`).** The speculative
  "factor `P` specialised at each variable" step can re-enter itself, giving an
  `O(v_count!)` blow-up (`v_count` reached 6, four levels deep). Capped to 2
  levels of nesting; beyond it the caller falls through to the other multivariate
  paths or returns `P` unfactored — less-factored, never incorrect.

- **Generator-count gate on the Factor family in `Simplify`
  (`src/simp/simp_search.c`, `transform_can_fire`).** `Factor` / `FactorSquareFree`
  / `FactorTerms` / `Apart` are skipped as candidate transforms when the seed has
  `>= 4` distinct polynomial generators, where multivariate factoring is both
  exponential and (for these trig forms) never the simplest result. Direct
  `Factor[...]` is **unaffected** — the gate lives only in `Simplify`'s transform
  selection. Univariate `Factor`, 2-generator Pythagorean identities, and
  bi/trivariate polynomials stay below the threshold; the simplest form (when one
  exists at this generator count) is still reached via
  `Together`/`Cancel`/`TrigReduce`/`TrigToExp`.

No regressions: `simplify`, `factor_baseline`, `factor_recombine`, `factor_terms`,
`mvfactor`, `mvfactor3`, `qafactor`, `rat`, `trigreduce`, `trigrat`, `trig`,
`integrals`, `expand`, `extended_gcd`, and the radical/trig `*_simplify` suites
pass (the 2 pre-existing `simplify_tests` radical FAILs and 1 `trigfactor_tests`
FAIL are identical on `main`). Valgrind: no `poly.c` GCD-guard, `heuristic_factor`,
`z_split`, or `simp_search` generator-counter frames in any lost/error stack.
Remaining cost is `TrigReduce`'s per-node internal `Together` on the multi-angle
form — bounded but not eliminated (closing the gap to Mathematica's instant
result needs a single-canonical-form rewrite, out of scope here).

## Feature: `FullSimplify` becomes a proper package; `MessageName` (`::`); package-context fixes (2026-06-13)

Reworked `FullSimplify` into a genuine Mathematica-style package and added the
language plumbing that a proper package needs (`::usage` messages, correct
System-symbol resolution inside a package context, statement-by-statement
loading of context-changing prologues). Also taught the gamma identity library
to consolidate products into a single `Gamma`.

- **`FullSimplify` is now a package** (`src/internal/simp/FullSimplify.m`).
  `BeginPackage["FullSimplify`"]` exports `FullSimplify` and
  `RegisterTransforms` via `::usage` messages; all helpers live in the
  `FullSimplify`Private`` context opened by `Begin["`Private`"]`; the file ends
  with `SetAttributes[FullSimplify, Protected]` and `EndPackage[]` (which leaves
  `FullSimplify`` on `$ContextPath`). The C side (`src/core.c`) no longer sets a
  docstring or `Protected` bit for `FullSimplify` — the package is
  self-documenting and self-protecting.
- **`MessageName` (`::`)** — new operator + builtin. `sym::tag` parses to
  `MessageName[sym, "tag"]` (tag read literally as a string; parser in
  `src/parse.c`, printer in `src/print.c`, `HoldFirst | Protected` in
  `src/attr.c`, interned in `src/sym_names.c`). `sym::tag = "text"` stores the
  message as a retrievable `DownValue`; `sym::usage = "text"` additionally calls
  `symtab_set_docstring` so `?sym` / `Information[sym]` surface it
  (`src/eval.c`: `assignment_target_symbol` recurses through `MessageName` so the
  `Protected` check targets `sym`, not `MessageName`). This is the idiomatic way
  to declare package exports.
- **Statement-by-statement module loading** (`src/parse.c`,
  `parse_next_expression`). The file loader now parses one top-level statement at
  a time (minimum precedence just above `CompoundExpression`'s `;`) and consumes
  the trailing `;`, instead of folding a whole file into one `CompoundExpression`
  up front. This lets context-changing prologues (`BeginPackage[...]`,
  `Begin["`Private`"]`) take effect on the symbols parsed in *subsequent*
  statements — previously every symbol in a `Get`/`LoadModule`-loaded package was
  resolved under the context in force before the prologue ran.
- **System-symbol resolution inside packages** (`src/context.c`,
  `src/sym_intern.{c,h}`, `src/sym_names.c`, `src/core.c`). The context resolver
  now resolves a bare name to `System`` when it has a `SymbolDef` *or* is flagged
  as a System symbol. `intern_mark_all_system()` flags every name interned at
  kernel-init time (run from `core_init` before the first package loads), so pure
  structural heads and constants created lazily on first use (`List`, `Rule`,
  `Integer`, `Automatic`, `Infinity`, `None`, `Heads`, ...) still resolve to
  `System`` from inside a `BeginPackage[...]` context rather than being qualified
  into the package's private context. `Surd` and `Beta` (System symbols with no
  kernel implementation, used as symbolic heads by the identity libraries) are
  interned here so they resolve consistently.
- **Gamma consolidation.** `simp/transforms/gamma.m` gains the raising identity
  `z Gamma[z] == Gamma[z+1]` (an Orderless `Times` match that absorbs a factor
  equal to the gamma argument). Because both forms have equal `SimplifyCount`,
  Simplify's strictly-decreasing search never takes it, so the `FullSimplify`
  driver runs a final **consolidation sweep**: it applies the relevance-selected
  identities once more to the Simplify result and keeps any rewrite that does not
  raise the complexity score. Net effect: `FullSimplify[x Gamma[x]]` &rarr;
  `Gamma[1 + x]`. The `>=`-Simplify guarantee holds (complexity never rises).

No regressions: `fullsimplify`, `fullsimplify_corpus` (9/9), `context`, `parse`,
`simplify` suites pass. `context_tests` updated to expect `FullSimplify`` on the
initial `$ContextPath` (standard package behaviour). The pre-existing
`Simplify[Gamma[x+1] - x Gamma[x]]` `0 ComplexInfinity` stderr note is unchanged
(emitted by plain `Simplify`, independent of these changes).
