# Changelog: week of 2026-06-08 (Mon) – 2026-06-14 (Sun)

Feature additions and fixes recorded during this week.

## Special functions: `PolyGamma` (digamma / polygamma) + inert `LogGamma` (2026-06-08)

Added `PolyGamma[z]` and `PolyGamma[n, z]` — the digamma ψ(z) and the n-th
polygamma ψ⁽ⁿ⁾(z) — modelled on the `Gamma` implementation
(`src/polygamma.c`, registered in `src/core.c`, symbols in `src/sym_names.c`,
docstrings in `src/info.c`). Attributes: `Listable`, `NumericFunction`,
`Protected`.

- **Canonicalisation.** `PolyGamma[z]` rewrites to `PolyGamma[0, z]`.
- **Special values.** Non-positive integer arguments → `ComplexInfinity`;
  `PolyGamma[0, Infinity]` → `Infinity`, `PolyGamma[n, Infinity]` → `0` (n ≥ 1);
  `ComplexInfinity`/`Indeterminate` arguments → `Indeterminate`.
- **Exact at positive integers.** `PolyGamma[0, m] = H_{m-1} - EulerGamma`
  (e.g. `PolyGamma[5] = 25/12 - EulerGamma`, `PolyGamma[100]` an exact rational
  minus `EulerGamma`). Odd order n ≥ 1 closes via ζ(n+1) into a rational plus a
  rational multiple of `Pi^(n+1)` (`PolyGamma[1,1] = Pi^2/6`,
  `PolyGamma[3,5] = -22369/3456 + Pi^4/15`), driven by a lazily-cached exact
  Bernoulli-number generator. Even order n ≥ 1 (ζ of an odd integer) stays
  symbolic (`PolyGamma[2,1]`, `PolyGamma[4,1]`).
- **Numeric.** Inexact real arguments evaluate at machine or arbitrary (MPFR)
  precision, tracking the input precision; `n = 0` uses `mpfr_digamma`, higher
  orders use a recurrence-shift + Bernoulli asymptotic series (MPFR has no
  polygamma above digamma and no Hurwitz zeta). Complex arguments evaluate by
  the same asymptotic in a small self-contained complex-MPFR helper
  (`PolyGamma[2.5 + 3 I] = 1.28127 + 0.979805 I`).
- **Negative order.** `PolyGamma[-1, z]` emits `LogGamma[z]`, registered as a
  new inert (`Listable`, `Protected`, no rules) log-gamma symbol. Orders ≤ −2
  stay symbolic.
- **Derivatives** (`src/calculus/deriv.c`). `D[PolyGamma[n, z], z] =
  PolyGamma[n+1, z]`. `Gamma` now differentiates through `PolyGamma`:
  `D[Gamma[z], z] = Gamma[z] PolyGamma[0, z]`, so
  `D[Gamma[z], {z, 2}] = Gamma[z] PolyGamma[1, z] + Gamma[z] PolyGamma[0, z]^2`
  (this replaces the previous inert `Derivative[1][Gamma][z]`; the
  `test_gamma.c` expectation was updated accordingly).
- **Tests.** New `tests/test_polygamma.c` (canonicalisation, argt errors,
  poles/infinities, exact n=0 and odd/even n≥1, inert `LogGamma`, machine and
  MPFR real with digit-exact precision tracking, complex, derivatives,
  listability, attributes). Valgrind clean (no leaks from Mathilda frames).
- **Out of scope** (deliberately symbolic): complex / fractional **order** `n`
  (`PolyGamma[6 + I, z]`), `PolyGamma[n, z]` for `n ≤ −2`, Gauss-digamma exact
  forms at non-integer rationals (`PolyGamma[0, 1/2]`), and dedicated `Series`
  expansions (only the generic `D`-based Taylor fallback applies). A future
  cleanup could share the complex-MPFR toolkit currently duplicated between
  `gamma.c` (`gcx`) and `polygamma.c` (`pcx`).

## Docs: documentation-center function regrouping (2026-06-08)

Overhauled how the documentation center (`site/`) groups built-in functions.
Previously a function received a category only if its name appeared literally in
an `## H2` heading of a `docs/spec/builtins/*.md` file, so functions documented
under *thematic* headings (`## Trig Functions`) or not yet documented fell
through to a 104-strong "Other & Advanced" bucket, and ~25 polynomial/algebra
functions were mis-filed under Structural Manipulation.

- **New categories.** Split the combined `arithmetic-and-algebra.md` into
  `arithmetic.md` (**Arithmetic**) and `algebra.md` (**Algebra**); split the
  mathematical constants out of `special-functions.md` into
  `mathematical-constants.md` (**Mathematical Constants** — `Pi`, `E`, `Degree`,
  `EulerGamma`, `Catalan`, `GoldenRatio`, `GoldenAngle`, `Glaisher`,
  `Khinchin`); added `comparisons.md` (**Comparisons** — `Equal`, `Unequal`,
  `Less`, `LessEqual`, `Greater`, `GreaterEqual`, `SameQ`, `UnsameQ`,
  `Inequality`).
- **Curated grouping map.** `site/generate.py` gained an authoritative
  `CATEGORY_OVERRIDES` table that pins thematically-documented and
  previously-uncategorized functions to the correct category — trig/log/exp →
  Elementary Functions, the `Blank*`/`Optional`/`Repeated`/`MatchQ`/`Longest`/
  `Shortest`/`Default`/`HoldPattern` pattern constructs → Pattern Matching,
  `Rule`/`Clear`/`AddTo`/`DownValues`/… → Assignment and Rules, the polynomial
  functions (`Factor`, `Expand`, `GroebnerBasis`, `PolynomialGCD`,
  `Coefficient`, `Resultant`, …) → Algebra, and so on. "Other & Advanced" is now
  empty.
- **Internal helpers excluded.** Context-qualified internal symbols
  (`Solve`SolveLinearSystem``, `Sum`Gosper``, `Integrate`CRCTable``, …) are no
  longer emitted as public documentation pages.
- The per-function "Specification" reference now points at the file that
  actually documents the function, independent of its display category.
- `Mathilda_spec.md` category index updated to match.

## Feature: `GoldenAngle`, `Glaisher`, `Khinchin` constants (2026-06-08)

Three more first-class mathematical constants, all `{Constant, Protected}` with
docstrings (`src/info.c`) and arbitrary-precision `N`. They join the central
numeric constant table (`src/numeric.c`), so `numeric_init` stamps their
attributes automatically, and they were added to the constant lists consulted
by `D` (`src/calculus/deriv.c`), `NumericQ` (`src/core.c`), `Rationalize`
(`src/rationalize.c`), and the zero-test free-symbol detector
(`src/zero_test.c`). `GoldenAngle` is newly interned in `src/sym_names.c`;
`Glaisher`/`Khinchin` were already interned.

- **`GoldenAngle`** = `(3 − Sqrt[5]) Pi` = `2 Pi / GoldenRatio^2` ≈ 2.39996 rad
  (137.5°). MPFR filler evaluates the closed form directly.

- **`Glaisher`** — Glaisher–Kinkelin constant *A* ≈ 1.2824271291, satisfying
  `Log[A] = 1/12 − ζ'(−1)`. MPFR has no zeta-derivative primitive, so
  arbitrary precision is computed from
  `ln A = (γ + ln 2π)/12 − ζ'(2)/(2π²)`, with `ζ'(2) = −Σ ln(n)/n²` evaluated by
  Euler–Maclaurin summation; the even Bernoulli numbers it needs are obtained
  from `mpfr_zeta_ui(2k)` (so `B_{2k}/(2k)!` reduces to a `ζ(2k)` expression),
  and the divergent asymptotic correction series is truncated at its smallest
  term.

- **`Khinchin`** — Khinchin's constant *K* ≈ 2.6854520011, the limiting
  geometric mean of continued-fraction partial quotients. Computed from the
  geometrically convergent Bailey–Borwein–Crandall zeta series
  `ln K · ln 2 = Σ_{n≥1} (ζ(2n) − 1)/n · Σ_{k=1}^{2n−1} (−1)^(k+1)/k`.

Verified end-to-end against Mathematica/high-precision references:

```
Attributes[Khinchin]   -> {Constant, Protected}
D[Glaisher x, x]       -> Glaisher
NumericQ[Khinchin]     -> True
N[GoldenAngle, 50]     -> 2.3999632297286533222315555066336138531249990110581
N[Glaisher, 250]       -> 1.28242712910062263687534256886979172776768892732500…  (matches to 250 digits)
N[Khinchin, 250]       -> 2.68545200106530644530971483548179569382038229399446…  (matches to 250 digits)
```

`N[Glaisher, 1000]` and `N[Khinchin, 500]` complete in well under a second;
`numeric_tests` and `core_tests` pass.

## Feature: `Gamma` — Euler & incomplete gamma functions (2026-06-08)

New builtin `Gamma` (`src/gamma.c`), covering the full Mathematica signature:

- `Gamma[z]` — Euler gamma function Γ(z).
- `Gamma[a, z]` — upper incomplete gamma Γ(a, z) = ∫_z^∞ tᵃ⁻¹ e⁻ᵗ dt.
- `Gamma[a, z0, z1]` — generalized incomplete = Γ(a, z0) − Γ(a, z1).

Attributes: `Listable`, `NumericFunction`, `Protected`. Docstring in
`src/info.c`; symbol already interned in `src/sym_names.c`.

Evaluation is layered for efficiency across all three argument regimes:

- **Exact symbolic.** Integer and half-integer `Gamma[z]` reduce to `(z-1)!`
  by reusing the existing `Factorial` machinery (exact int64, GMP BigInt, and
  rational·`Sqrt[Pi]` half-integers). Non-positive integers give
  `ComplexInfinity`. `Infinity`/`-Infinity`/`ComplexInfinity` map to
  `Infinity`/`Indeterminate`/`ComplexInfinity`. Incomplete-form rewrites:
  `Gamma[a,0] = Gamma[a]`, `Gamma[1,z] = E^-z`, `Gamma[a,Infinity] = 0`.

```
Table[Gamma[n], {n, 10}]   -> {1, 1, 2, 6, 24, 120, 720, 5040, 40320, 362880}
Gamma[30]                   -> 8841761993739701954543616000000
Gamma[5/2]                  -> 3/4 Sqrt[Pi]
Gamma[-1/2]                 -> -2 Sqrt[Pi]
Gamma[0]                    -> ComplexInfinity
```

- **Machine precision.** Real → libm `tgamma`; complex → Lanczos
  approximation (g = 7) with reflection for the left half-plane.

```
Gamma[2.2]      -> 1.1018
Gamma[2.3 + I]  -> 0.719141 + 0.540614 I
```

- **Arbitrary precision.** Real → MPFR `mpfr_gamma`; incomplete real (machine
  or MPFR) → `mpfr_gamma_inc`. Output precision tracks the input.

```
N[Gamma[22/10], 50] -> 1.1018024908797127327691419862229964808241863995904
Gamma[1.5, 7.5]     -> 0.00160996
Gamma[1, 1.1, 2.2]  -> 0.222068
```

Tests: `tests/test_gamma.c` (exact values, poles, half-integers, infinities,
symbolic fall-through, machine real/complex, MPFR precision tracking,
incomplete & generalized incomplete, Listable threading, attributes). Valgrind
clean (totals identical to the `Sin[1.0]` baseline; no Gamma frames in any lost
stack).

## Closing the Mathematica gaps for `Gamma` (2026-06-08)

Follow-up work to bring `Gamma` in line with Mathematica across more of its
domain (`src/gamma.c`, `src/calculus/deriv.c`, `tests/test_gamma.c`):

- **Integer incomplete gamma closes to a finite form.** `Gamma[n, z]` for a
  positive integer `n` now reduces to `(n-1)! e^-z Σ_{k=0}^{n-1} z^k/k!` for
  symbolic or exact `z` (integer coefficients via GMP, BigInt-safe). So
  `Gamma[2, 3] -> 4/E^3`, `Gamma[2, x] -> (1 + x) E^-x`,
  `Gamma[3, x] -> (2 + 2 x + x^2) E^-x`. (`Gamma[1, z] = E^-z` is now the
  `n = 1` case of this rule.)

- **Derivatives.** `D[Gamma[a, z], z] = -z^(a-1) E^-z`, with the chain rule on
  both arguments; the first-argument derivative is the generic
  `Derivative[1,0][Gamma][a,z]` (a closed form needs `PolyGamma`, not yet
  implemented, so one-argument `D[Gamma[z], z]` likewise stays generic).

- **Arbitrary-precision complex `Gamma[z]`.** Replaces the previous
  "left symbolic" limitation with Spouge's approximation, whose coefficients
  are generated at runtime to the requested precision (reflection for
  `Re(z) < 1/2`). E.g.
  `N[Gamma[I], 50] = -0.15494982830181068512… − 0.49801566811835604271… I`.
  Verified against the independent machine Lanczos path.

## Fix: `Gamma` argument-count diagnostic (2026-06-08)

`Gamma` now emits a Mathematica-compatible `Gamma::argt` warning when called
with the wrong number of arguments (0, or 4+), and leaves the call
unevaluated (`src/gamma.c`):

```
Gamma[]          -> Gamma::argt: Gamma called with 0 arguments; 1 or 3 arguments are expected.
Gamma[1, 2, 3, 4] -> Gamma::argt: Gamma called with 4 arguments; 1 or 3 arguments are expected.
```

The valid 1-, 2-, and 3-argument signatures are unaffected.

- **Complex incomplete gamma `Gamma[a, z]`.** Numeric complex `a`/`z` (machine
  or arbitrary precision) now evaluate via a lower-incomplete power series
  (`Re(z) < Re(a)+1`) or a Lentz continued fraction otherwise, finished with
  the `e^-z z^a` prefactor. E.g. `Gamma[2.0, 1 + I] = (2 + I) e^-(1+I)` and
  `N[Gamma[3/2, 2 + I], 30] = 0.160487401929263240… − 0.176588715957602346… I`
  (confirmed via the recurrence `Γ(a+1,z) = a Γ(a,z) + z^a e^-z` to 30 digits).

The complex arithmetic uses a small `mpfr_t`-pairs toolkit local to
`src/gamma.c` (add/sub/mul/div/exp/log/sin/pow); MPC is not a dependency.
Exact non-integer or exact-complex incomplete forms (`Gamma[3/2, z]`,
`Gamma[3/2, I]`) stay symbolic, matching Mathematica. Valgrind totals remain
identical to the `Sin[1.0]` baseline with no Gamma/deriv frames in any lost
stack.

## Feature: `Pochhammer` — the rising factorial (2026-06-08)

New builtin `Pochhammer` (`src/pochhammer.c`): the Pochhammer symbol
(a)ₙ = a (a+1) … (a+n-1) = Γ(a+n)/Γ(a). Attributes `Listable`,
`NumericFunction`, `Protected`; docstring in `src/info.c`; symbol interned in
`src/sym_names.c`; wired from `core_init()` next to `gamma_init()`.

Following the `Gamma` precedent, the implementation holds almost no numeric
code of its own — it delegates to two existing, fully-tested mechanisms:

- **Exact integer order → a `Times` product.** For an integer `n`
  (|n| ≤ 1000) it builds `Times[a, a+1, …, a+n-1]` (or the reciprocal for
  negative `n`) and evaluates it. That one path collapses to an exact
  Integer/Rational for numeric `a`, preserves MPFR precision and complex
  arithmetic, and stays a polynomial product for symbolic `a`.

- **Other numeric arguments → the Gamma ratio.** It evaluates
  `Gamma[a+n]/Gamma[a]`, reusing Gamma's exact half-integer reductions, its
  libm/MPFR real paths and its machine-complex Lanczos path. A residual-`Gamma`
  guard keeps genuinely-symbolic inputs unevaluated.

Short-circuits: `Pochhammer[a, 0] = 1`, `Pochhammer[0, n>0] = 0` (so even
`Pochhammer[0, 1285] = 0` without forming a huge factorial),
`Pochhammer[Infinity, n>0] = Infinity`. Wrong argument counts emit a
Mathematica-compatible `Pochhammer::argrx` diagnostic and stay unevaluated.

```
Pochhammer[10, 6]                 -> 3603600
Pochhammer[n, 5]                  -> n (1 + n) (2 + n) (3 + n) (4 + n)
Pochhammer[n, -5]                 -> 1/((-5 + n) (-4 + n) (-3 + n) (-2 + n) (-1 + n))
Pochhammer[3/2, 1/2]              -> 2/Sqrt[Pi]
Pochhammer[2.4, 8.5]              -> 2.31022e+06
N[Pochhammer[1/3, 7], 50]         -> 505.97165066300868770004572473708276177411979881116
Pochhammer[2. + 5 I, 8 I]         -> 2.13868e-06 - 1.42187e-05 I
Pochhammer[{2, 3, 5, 7, 11}, 3]   -> {24, 60, 210, 504, 1716}
```

Tests: `tests/test_pochhammer.c` (exact integer numeric/symbolic, negative
order, the `0`/`Infinity` short-circuits, half-integer reduction, machine
real/complex, MPFR precision tracking, symbolic fall-through, Listable,
attributes, arity diagnostics). Valgrind totals identical to the macOS
baseline with no Pochhammer frames in any lost stack. Derivatives and series
(which Mathematica expresses through `PolyGamma`) are deferred.

## Fix: bare `N[expr]` no longer collapses high-precision numbers (2026-06-08)

`N[Pi, 100] // N` (i.e. `N[N[Pi, 100]]`) was returning a machine-precision
`3.14159` instead of the 100-digit value — `Precision` reported
`MachinePrecision`. The same precision loss hit every constant (`E`,
`GoldenRatio`, …) and any already-arbitrary-precision literal (`N[2.5`100]`).

Root cause: `numericalize`'s `EXPR_MPFR` branch (`src/numeric.c`) **always**
down-converted an MPFR value to a machine `double` under machine mode. That is
correct for inexact contagion (`1. + N[Pi,100]` must collapse to machine), but
wrong for bare `N`: Mathematica's `N[expr]` numericalizes only the **exact**
parts of `expr` and leaves the precision of already-approximate numbers
untouched.

Fix: a `preserve_inexact` flag on `NumericSpec` (`src/numeric.h`), set **only**
by the one-argument `N` builtin. When set, the machine-mode `EXPR_MPFR` branch
keeps the value at its existing precision (`expr_new_mpfr_copy`). Contagion
(`numeric_contagion_args`) and the explicit two-argument `N[expr, prec]` form
leave the flag clear, so their (correct) down-converting / precision-setting
behaviour is unchanged. The flag is read solely in the machine branch and is
initialised to `false` by `numeric_machine_spec()`, through which every
machine-mode spec in the codebase is built — so existing call sites are
unaffected.

```
N[Pi, 100] // N                  -> 3.1415926535897932384626…  (100 digits)
Precision[N[Pi, 100] // N]       -> 100.243
N[(1 + Sqrt[5])/2, 200] // N     -> 1.6180339887…              (200 digits)
N[2.5`100]                       -> Precision 100.243   (was: MachinePrecision)
N[Pi]                            -> 3.14159              (exact symbol → machine, unchanged)
1. + N[Pi, 100]                  -> 4.14159              (machine contagion, unchanged)
N[N[Pi, 100], 20]                -> 3.1415926535897932385 (explicit reduce to 20, unchanged)
```

`numeric_tests` pass.
