# Changelog: week of 2026-05-18 (Mon) – 2026-05-24 (Sun)

Feature additions and fixes recorded during this week.

## PolynomialGCD / PolynomialLCM: multi-α tower path (2026-05-24)

`PolynomialGCD[..., Extension -> Automatic]` and
`PolynomialLCM[..., Extension -> Automatic]` now handle inputs whose
auto-detected algebraic extension contains two or more independent
generators (e.g. both `Sqrt[2]` and `Sqrt[3]`).  Previously the
multi-generator tower was constructed and then dropped on the floor;
the request fell through to the no-extension BPList path which only
recognises common literal factors, leaving e.g.

```mathematica
PolynomialLCM[x - Sqrt[2], x - Sqrt[3], Extension -> Automatic]
   -> (-Sqrt[2] + x) (-Sqrt[3] + x)         (* unevaluated symbolic product *)
```

The new path lifts each input to `QAUPoly[x]` over the compositum
`Q(γ)`, runs `qaupoly_gcd` directly (LCM = a·b / gcd folded
left-to-right), and renders the result with the tower's surface form
for γ.  Same input now collapses to:

```mathematica
PolynomialLCM[x - Sqrt[2], x - Sqrt[3], Extension -> Automatic]
   -> Sqrt[6] - Sqrt[2] x - Sqrt[3] x + x^2
PolynomialLCM[x - Sqrt[2] - Sqrt[3], x - Sqrt[2] + Sqrt[3], Extension -> Automatic]
   -> -1 - 2 Sqrt[2] x + x^2
PolynomialGCD[x^2 - 2 Sqrt[3] x + 1, x - Sqrt[3] - Sqrt[2], Extension -> Automatic]
   -> -Sqrt[2] - Sqrt[3] + x
```

Scope (tier-1 limitation): the substituted inputs must share exactly
one polynomial variable besides the internal γ.  Multivariate inputs
fall through to the existing no-extension path unchanged.  Without
`Extension -> Automatic` the behaviour is unchanged — the standard
BPList path still produces the symbolic structural form.

Implementation: `qa_polynomialgcd_with_tower` and
`qa_polynomiallcm_with_tower` in `src/poly/qafactor.c`, sharing the
`tower_substitute_alphas` / `tower_lift_inputs` scaffold with
`qa_cancel_with_tower`.  Tests in `tests/test_extension_auto_builtins.c`
(seven `test_pgcd_multigen_*` / `test_plcm_multigen_*` cases).

## Power: split `Integer^(p/q)` into distinct-prime radicals (2026-05-24)

`Power[Integer, Rational]` now factorises the integer base into its
prime factorisation and emits a canonical product of distinct-prime
powers grouped by their reduced effective exponent, matching
Mathematica's surface form:

```mathematica
18^(1/3)  -> 2^(1/3) 3^(2/3)
12^(1/3)  -> 2^(2/3) 3^(1/3)
60^(1/3)  -> 2^(2/3) 15^(1/3)        (primes 3, 5 share eff 1/3)
50^(1/3)  -> 2^(1/3) 5^(2/3)
18^(2/3)  -> 3 2^(2/3) 3^(1/3)       (4/3 contributes int_part 1)
18^(5/3)  -> 54 2^(2/3) 3^(1/3)
98^(1/3)  -> 2^(1/3) 7^(2/3)
```

Existing extractions are unaffected:

```mathematica
6^(1/3)   -> 6^(1/3)                  (uniform exps — split adds no info)
30^(1/3)  -> 30^(1/3)
36^(1/3)  -> 6^(2/3)                  (find_min_perfect_base)
72^(1/3)  -> 2 3^(2/3)
8^(1/3)   -> 2
50^(1/2)  -> 5 Sqrt[2]
```

Implementation: `power_split_residue` in `src/power.c` trial-divides
the residue `r` (the part of `abs(n)` left after
`factor_out_kth_power` extracts the largest perfect q-th power), then
for each prime `p_i` with exponent `a_i` computes
`(int_i, d_i) = divmod(a_i * b_rem, q)`. The integer parts feed a
rational coefficient `∏ p_i^int_i`; the residual primes are grouped by
their reduced `(d_i / q)` exponent into one `Power[∏p_group, eff]` per
group. Triggers only when splitting strictly adds information (some
`int_i != 0` or the reduced exponents differ across primes) so
uniform-exponent inputs like `6^(1/3)` and `30^(1/3)` keep their
compact form.

Restricted to positive base with positive `b_rem`. Negative bases stay
in the existing `Power[res_base, b_rem/q]` shape so the `(-1)^(b/q)`
sign aggregation logic doesn't need to be rewired. Cases like
`18^(5/3)` (block-extracted as `18 · 18^(2/3)`) reach the split
naturally on the recursive evaluator pass, since `18^(2/3)` re-enters
`builtin_power` and hits the same fall-through.

The complementary `times.c` "generalised radical fusion" rule —
`Power[a, e_i] * Power[b, e_j]` collapse for related exponents —
was the canonical-form sibling of the old combined `Power[N, p/q]`
shape. It now skips fusion when both bases are coprime positive
integers and `|k| > 1`, so the new split form is stable instead of
being re-fused back. Useful fusions still fire:

```mathematica
Sqrt[2] Sqrt[3]            -> Sqrt[6]      (k = 1)
2^(1/3) 3^(1/3)            -> 6^(1/3)      (k = 1)
2^(1/3) 8^(2/3)            -> 4 2^(1/3)    (gcd 2, |k|=2 — cancellation)
12^(1/3) 2^(-2/3)          -> 3^(1/3)      (gcd 2, |k|=2 — cancellation)
2^(1/3) 3^(2/3)            -> 2^(1/3) 3^(2/3)  (coprime, |k|=2 — stays split)
```

## Added: `PossibleZeroQ` — hybrid symbolic-numeric zero recognition (2026-05-24)

`PossibleZeroQ[expr]` heuristically decides whether `expr` is identically
zero. Returns `True` / `False`; collapses uncertain cases to `True` per
Mathematica's `ztest1` policy. Attributes: `Listable | Protected`.

The implementation (`src/zero_test.c`, header `src/zero_test.h`) is a
four-stage pipeline, each stage exiting as soon as it produces a
definite verdict:

- **Stage 0** — constant-time structural shortcuts: literal `0`,
  `Complex[0, 0]`, named non-zero constants like `Pi`.
- **Stage 1** — rational normalisation via `Together`, `Cancel`,
  `Expand`, then `is_zero_poly` (existing in `src/poly/poly.c`). Decides
  every identity in `Q(x_1, …, x_n)` exactly.
- **Stage 2** — numeric precision ladder (53 → 200 → 500 → 1000 bits).
  Compares `|numericalize(e)|` against a lenient cancellation threshold
  `scale * 2^{-p + 30}`. Detects "MPFR didn't help" by checking whether
  the residual magnitude actually shrinks rung-over-rung — if it doesn't
  (e.g. `Sin` of `Complex[Real, Real]` which Mathilda evaluates with
  doubles regardless of precision), the prior verdict is taken as
  final. Uses a Plus-aware operand-magnitude scale that captures
  intermediate cancellation depth.
- **Stage 3** — Schwartz–Zippel probabilistic identity test for inputs
  with free symbols. Substitutes each free variable with an
  independently-drawn random sample from `R ∪ Complex[R, R]` and recurses
  into Stage 2. Requires four consecutive `True` trials.

Reference examples all match Mathematica behaviour:

```mathematica
PossibleZeroQ[E^(I Pi/4) - (-1)^(1/4)]              (* True  *)
PossibleZeroQ[(x + 1)(x - 1) - x^2 + 1]             (* True  *)
PossibleZeroQ[(E + Pi)^2 - E^2 - Pi^2 - 2 E Pi]     (* True  *)
PossibleZeroQ[E^Pi - Pi^E]                          (* False *)
PossibleZeroQ[2^(2 I) - 2^(-2 I) - 2 I Sin[Log[4]]] (* True  *)
PossibleZeroQ[1/x + 1/y - (x + y)/(x y)]            (* True  *)
PossibleZeroQ[Sqrt[x^2] - x]                        (* False *)
```

The full design rationale and literature survey (Richardson 1968,
Schwartz 1980, Zippel 1979, Bronstein 2005, Higham 2002, …) is in
`ZERO_RECOGNISE_PLAN.md` at the repository root.

Symbol registered in `src/sym_names.{c,h}`, attributes table updated in
`src/attr.c`, docstring added in `src/info.c`, dispatcher init wired
into `core_init` in `src/core.c`. Spec entry under
`docs/spec/builtins/expression-information.md`.

Tests: `tests/test_zero_test.c` — 72 cases across all four stages,
including a memory-leak smoke load and catastrophic-cancellation
regressions (`10^20 + 1 - 10^20 != 0`).

## Fix: `N[]` no longer returns NaN for complex MPFR powers (2026-05-24)

At MPFR precision, expressions like `N[(-1)^(1/4), 32]` and
`N[E^(I Pi/4), 64]` used to return `nan.0 + nan.0*I`. The `cpow` fallback
in `power.c` extracted operand values via a chain of
`type == EXPR_REAL ? ... : EXPR_INTEGER ? ... : EXPR_BIGINT ? ... : 0`,
which silently coerced any `EXPR_MPFR` operand to `0`. `cpow(0+0i, 0+0i)`
then produced NaN.

The build doesn't link MPC, so true complex MPFR arithmetic isn't
available. Instead `numeric.c` now provides `numeric_mpfr_complex_pow`,
a polar-form implementation that handles complex-valued powers at
arbitrary precision:

    base = r * exp(i*theta),  exp = c + i*d
    base^exp = r^c * exp(-d*theta)
               * (cos(c*theta + d*ln(r)) + I*sin(c*theta + d*ln(r)))

All sub-operations (`mpfr_hypot`, `mpfr_atan2`, `mpfr_pow`, `mpfr_log`,
`mpfr_exp`, `mpfr_cos`, `mpfr_sin`) operate at the combined target
precision, so the result carries the requested number of digits rather
than being demoted to machine precision.

`src/power.c` calls the new helper immediately after the existing real
MPFR `mpfr_pow` fast path and before the `cpow` fallback, gated on
`numeric_any_mpfr(base, exp)`. The helper returns `NULL` for inputs it
can't approximate, leaving downstream paths intact.

`tests/test_numeric.c::test_n_prec_complex_pow` exercises the two cases
from the bug report plus `I^I`.

## Fix: Element[{x,y,...}, dom] now behaves as a conjunction (2026-05-23)

`Element[{x1, ..., xN}, dom]` is shorthand for
`Element[x1, dom] && ... && Element[xN, dom]`. Previously,
`builtin_element` always threaded the call into a `List` of per-element
`Element[xi, dom]` calls. Because `Simplify` carries `ATTR_LISTABLE` on
its assumption argument, this caused inputs like
`Simplify[Log[E^(x+y)], Element[{x, y}, Reals]]` to thread `Simplify`
itself over the list of partial facts, so each branch saw only one of
the variables. The result was a bogus list
`{Log[E^(x+y)], Log[E^(x+y)]}` instead of the expected `x + y`.

Two coordinated changes:

- `src/simp/simp_builtins.c::builtin_element`: when the first arg is a
  list, evaluate `Element[xi, dom]` on each component and collapse to a
  single `True` / `False` only when every sub-call decides. When any
  component is undecided, return `NULL` so the original
  `Element[{x1,...,xN}, dom]` expression stays in place.
- `src/simp/simp_assume.c::ctx_walk`: recognise
  `Element[{x1,...,xN}, dom]` as a fact and split it into per-variable
  `Element[xi, dom]` facts when building the `AssumeCtx`, so
  `prov_re` / `fact_in_domain` continue to see the joint assumption
  per-variable.

Also introduces `SYM_Element` in `sym_names.{c,h}` for the
pointer-compare fast path in `ctx_walk`.

`tests/test_element.c::test_element_threads_over_list` updated to
reflect the new semantics (joint `True` / `False` rather than a
threaded list) and now also covers the undecided-and-Simplify case.

## simp_log -- dedicated logarithm-simplification sub-module (2026-05-23)

A new `src/simp/simp_log.{c,h}` sub-module fires inside both
`simp_pipeline_logexp` and the simp_search seed phase to close a class
of Log identities that the existing positivity-aware LogExp cascade
missed. It implements two orthogonal primitives:

- **Prime decomposition of Log[positive rational]**: every Log[r]
  subexpression whose argument is a positive integer or rational p/q is
  replaced by the integer linear combination
  `Sum e_i Log[p_i] - Sum f_j Log[q_j]` obtained by prime-factoring p
  and q through `FactorInteger`. Also recognises `Log[Power[r, k]]`,
  rewriting as `k * decomposition(Log[r])`. Sound unconditionally.
  Closes the previously unsimplified `-4 Log[4] + 8 Log[2] -> 0`,
  `Log[72] - 3 Log[2] - 2 Log[3] -> 0`, `Log[2/3] + Log[3/4] +
  Log[4/5] - Log[2/5] -> 0`, `Log[Sqrt[12]] - Log[2] - 1/2 Log[3]
  -> 0`, and `Log[2,5] Log[5,8] - 3 -> 0`.

- **Linear-combination-of-logs fuser**: every Plus subexpression
  whose addends decompose as `Sum c_i Log[a_i] + R` is rewritten by
  fusing the log block into a single `Log[Product a_i^c_i]`. The fused
  argument is canonicalised via `Together`, `Expand`, and a single
  depth-bounded `Simplify` so that algebraic identities like
  `(x^2 - y^2) / ((x-y)(x+y)) = 1` and the Pythagorean
  `(Sec[x] - Tan[x])(Sec[x] + Tan[x]) = 1` collapse the fused Log to
  Log[1] = 0. Gated on provable positivity of every a_i when the
  fused form is not a near-atomic constant.

The two passes iterate with the existing `LogExpRules` cascade inside
`simp_pipeline_logexp` to a small fixed point (cap 3 rounds), so a
fused `Log[E^a]` is then collapsed to `a` via the realness-aware
`Log[E^k] -> k` rewrite for real-symbolic a.

Recursion into `simp_log_apply` from the recursive `Simplify` call is
bounded by `simp_log_depth`, capped at 2 to keep the cost bounded.

A new 20-case unit suite in `tests/test_simp_log.c` exercises every
test in the original user logTestSuite, plus three regression guards
that confirm single-Log inputs, `Log[1]`, and a single Plus log term
are not over-rewritten.

## System floating-point constants $MachinePrecision, $MachineEpsilon, $MinMachineNumber, $MaxMachineNumber, $MaxNumber, $MinNumber (2026-05-23)

Registered six read-only system variables describing the local
floating-point range and granularity. Each is bound as an `OwnValue` in
a new `system_constants_init()` helper in `src/core.c` (called from
`core_init()` immediately after `eval_init()`) and marked `Protected`.

Values are derived from the platform's `<float.h>` macros — not
hard-coded — so they track the float ABI Mathilda was compiled
against:

| Symbol | Source |
|--------|--------|
| `$MachinePrecision` | `DBL_MANT_DIG * Log10[2]` (≈ 15.9546 on IEEE 754) |
| `$MachineEpsilon`   | `DBL_EPSILON` |
| `$MinMachineNumber` | `DBL_MIN` |
| `$MaxMachineNumber` | `DBL_MAX` |
| `$MaxNumber`        | `USE_MPFR`: `mpfr_nextbelow(+inf)` at `DBL_MANT_DIG` bits; else `DBL_MAX` |
| `$MinNumber`        | `USE_MPFR`: `mpfr_nextabove(0)` at `DBL_MANT_DIG` bits; else `DBL_MIN` |

The previous hard-coded `15.954589770191003` magic number that the
parser used as the *machine-precision threshold* (above which a real
literal is promoted to MPFR) is now driven by the same derivation:
`NUMERIC_MACHINE_PRECISION_DIGITS` in `src/numeric.h` is defined as
`((double)DBL_MANT_DIG * NUMERIC_LOG10_2)`. `src/parse.c` and
`src/precision.c` use that macro instead of the previous literal.

Docstrings are registered in `src/info.c`; new `SYM_DollarMachineEpsilon`,
`SYM_DollarMachinePrecision`, `SYM_DollarMaxMachineNumber`,
`SYM_DollarMaxNumber`, `SYM_DollarMinMachineNumber`, `SYM_DollarMinNumber`
entries are added to `src/sym_names.{c,h}` (the first `$`-prefixed
interned symbols in the file).

```mathematica
In[1]:= $MachinePrecision
Out[1]= 15.9546

In[2]:= $MachinePrecision == 53 Log[10, 2]
Out[2]= True

In[3]:= {$MinMachineNumber, $MaxMachineNumber}
Out[3]= {2.22507*10^-308, 1.79769*10^308}

In[4]:= {MachineNumberQ[$MaxMachineNumber], MachineNumberQ[$MaxNumber]}
Out[4]= {True, False}

In[5]:= $MachinePrecision = 42
Set::wrsym: Symbol $MachinePrecision is Protected.
```

## MachineNumberQ predicate (2026-05-23)

Added `MachineNumberQ[expr]` in `src/numeric.c`. Returns `True` iff
`expr` is a machine-precision IEEE-double real (finite — `+inf`, `-inf`,
and `NaN` are rejected via `isfinite()`) or a `Complex` of two such
reals. Exact numbers (Integer, BigInt, Rational), arbitrary-precision
reals (`EXPR_MPFR`, e.g. from `N[Pi, 30]` or any literal with > ~16
mantissa digits), and non-numeric atoms all return `False`. A `Complex`
needs *both* parts to be finite machine reals — `Complex[1, 2]` (an
exact Gaussian integer) and `Complex[N[1, 30], N[1, 30]]` both yield
`False`, matching Wolfram Language semantics.

Attributes: `Protected` (intentionally *not* `Listable`; `MachineNumberQ`
on a list inspects the `List` atom as a whole).

```mathematica
In[1]:= MachineNumberQ[Sin[1000.]]
Out[1]= True

In[2]:= MachineNumberQ[Exp[1000.]]            (* overflows to +inf *)
Out[2]= False

In[3]:= MachineNumberQ[-29037945.290347]
Out[3]= True

In[4]:= MachineNumberQ[-29037852093587905730945.29034875093457832094573984537498]
Out[4]= False                                 (* MPFR via parser auto-promote *)
```

Test coverage: `tests/test_machine_number_q.c` (27 tests) — documented
examples, every numeric leaf type, IEEE special values, MPFR via
`N[..., n]` and high-precision literal, Complex variants, non-numeric
inputs, arity mismatches, attributes/docstring/symbol introspection,
and a repeated-evaluation stress loop for valgrind coverage.

## Precision-aware real-literal parsing (2026-05-23)

The parser now picks `EXPR_REAL` vs `EXPR_MPFR` automatically based on
the number of significant digits typed in the *mantissa* of a real
literal. Previously every literal with a `.` or `e/E` became a machine
double, silently discarding the extra precision a user typed. Now:

- A literal whose implied precision is **≤ MachinePrecision** (≈ 15.95
  decimal digits) parses as `EXPR_REAL` — unchanged behaviour for
  ordinary inputs such as `3.14`, `0.5`, `1.0e22`.
- A literal whose implied precision **exceeds MachinePrecision** parses
  as `EXPR_MPFR` at `ceil(implied × log2(10))` bits, fed directly from
  the literal text via `mpfr_set_str` (no lossy double round-trip).

Implied precision uses the mantissa only — the exponent does not count:

    implied = digits_after_decimal + log10(|mantissa_value|)

This matches Mathematica, where `1.0e22` keeps `MachinePrecision`
(mantissa `1.0` has implied precision ≈ 1) but
`1.123456789012345678e22` is high precision (mantissa precision ≈ 18).

**Examples** (matching the Mathematica reference output):

    In[1] := -29037945.290347
    Out[1] = -2.90379e+07         (* EXPR_REAL *)
    In[2] := Precision[%]
    Out[2] = MachinePrecision     (* implied ≈ 13.46, below threshold *)

    In[3] := -29037852093587905730945.29034875093457832094573984537498
    Out[3] = -29037852093587905730945.29034875093457832094573984537498
    In[4] := Precision[%]
    Out[4] = 54.4864              (* implied ≈ 54.463, MPFR @ 181 bits *)
    In[5] := Accuracy[%%]
    Out[5] = 32.0235

    In[6] := -29037852093587905730945.2903487509345783209457398453749803489530945034950
    In[7] := Precision[%]
    Out[7] = 71.6451              (* MPFR @ 238 bits *)

The reported `Precision[]` differs from Mathematica's value by less than
0.2 digits because `Precision[]` returns `bits / log2(10)` with bits
quantised to integers; Mathematica appears to store the precision as a
continuous real. `Accuracy[]` follows the same offset because
`Accuracy = Precision − log10(|x|)`. End-to-end ordering (which side of
MachinePrecision a literal falls on, and which MPFR bit count it gets)
is identical.

**Always machine precision (never auto-promoted)** — preserves
existing semantics for code that relies on machine doubles:
- pure integer literals (`123`, `99999999999999999999`) — still
  `EXPR_INTEGER` / `EXPR_BIGINT`;
- real literals without a `.` (`1e10`, `5e-3`);
- zero literals regardless of trailing zeros (`0.0`, `0.000`,
  `0.0000000000000000000`);
- low-mantissa-precision scientific / scaled-scientific (`1.0e22`,
  `1.0*^22`).

**Backwards compatibility** — explicit Mathematica-style precision
suffixes still override the implicit decision. `3.14`50`,
`3.14``49`, and `3``50` all behave exactly as before.

**Implementation** — `src/parse.c` `parse_number()` real branch:
the literal is scanned once into a mantissa span and a full-literal
span, the implied precision is computed from the mantissa portion
only, and the result is built either via `expr_new_real(strtod(...))`
or `expr_new_mpfr_from_str(..., bits)`. The change is entirely local
to the literal-construction step; the existing precision-suffix
handler and `*^`-scaling code at the end of `parse_number()` are
unchanged.

**Tests** — new `tests/test_parser_precision.c` (23 test functions
across 13 categories) covers: basic machine/MPFR paths, negative
literals, threshold boundary, the three user-provided examples,
scientific and scaled notation, zero handling, integer pass-through,
backtick-suffix backwards compatibility, `Precision[]`/`Accuracy[]`
integration, MPFR-value round-trip (parser feeds the literal text to
`mpfr_set_str` directly, not via a double), 100-digit literals, and
fractions without integer parts.

## RealExponent (2026-05-23)

New built-in `RealExponent[x]` / `RealExponent[x, b]` -- the base-10 (or
base-`b`) real exponent of a real number, `Log[b, |x|]`.  The 1-argument
form defaults to base 10.

- Implementation: added in `src/real.c` (+ `src/real.h`), reusing the
  numeric classifier `rd_classify` and the rational-bigint detector
  `rd_is_rational_bigint` from the existing `RealDigits` /
  `MantissaExponent` machinery.  Registered in `real_init()` with
  `Protected | Listable` attributes.  Symbol `SYM_RealExponent`
  interned in `src/sym_names.{h,c}`; docstring added in `src/info.c`.
- Accepted numeric kinds: `Integer`, `BigInt`, `Rational`, `Real`, and
  (under `USE_MPFR`) `MPFR`.  Symbolic numeric leaves (`Pi`, `E`,
  `EulerGamma`, `Catalan`, `GoldenRatio`, `Degree`) and numeric-valued
  composites such as `Pi^Pi` or `1/Pi` route through `numericalize` at
  the combined working precision, then take the same path as a numeric
  literal.  Plain symbols with no numeric value are left unevaluated
  with no diagnostic.
- Output kind contagion -- only the **original** inputs (pre-
  numericalize) drive the choice:
  - Either `x` or `b` is MPFR  ->  result is MPFR at the higher of the
    input precisions.
  - Neither is MPFR  ->  result is a machine `Real`.
  This avoids spuriously promoting `RealExponent[Pi^Pi]` to MPFR just
  because the internal numericalize step uses MPFR at 64 working bits.
- Sign of `x` is discarded (`RealExponent[-x] == RealExponent[x]`).
- Zero handling follows Mathilda's `Accuracy` convention (every zero
  has `Accuracy = Infinity`, so `-Accuracy[0] = -Infinity`):
  - `RealExponent[0]`     -> `-Infinity`
  - `RealExponent[0.]`    -> `-Infinity`
  - `RealExponent[0``p]`  -> `-Infinity`
  This is a documented divergence from Mathematica's
  `RealExponent[0.] = -307.65...`; matching that requires changing
  `Accuracy[0.]` in `src/precision.c` first and is deferred.
- Base validation: the base must be a real number `> 1`.  Integer 1, 0,
  negative integers, `Rational` `<= 1`, `Real` `<= 1`, MPFR `<= 1`, and
  complex bases all emit `RealExponent::ibase` and leave the call
  unevaluated.
- Diagnostics (stderr; call is then left unevaluated):
  - `RealExponent::argt: RealExponent called with N argument(s); 1 or 2
    arguments are expected.` -- argc outside `[1, 2]`.
  - `RealExponent::realx: The value <x> is not a real number.` --
    `Complex[re, im]` with non-zero imaginary part at position 1.
  - `RealExponent::ibase: Base <b> is not a real number greater than 1.`
    -- base `<= 1`, non-positive, or complex.
- Computation path (USE_MPFR build):
  1. Numericalize symbolic inputs at working precision = `max(64,
     prec(x) + 32, prec(b) + 32)`.
  2. Convert `|x|` and `b` to MPFR via `mpfr_set_si` / `mpfr_set_z` /
     `mpfr_set_d` / `mpfr_set_q` / `mpfr_set` as appropriate.
  3. Compute `log(|x|) / log(b)` at working precision + 32 guard bits.
  4. Round to output precision (machine `Real` via `mpfr_get_d`, or
     MPFR via `expr_new_mpfr_move`).
- Tests: `tests/test_real_exponent.c` -- integer / bigint / rational /
  machine-real / MPFR coverage, base 2 / 3 / 5 / 7 / 10 / 16 / `E`,
  identities `RealExponent[xy] == RealExponent[x] + RealExponent[y]`

## RealDigits (2026-05-23)

New built-in `RealDigits[x]` / `RealDigits[x, b]` / `RealDigits[x, b, len]`
/ `RealDigits[x, b, len, n]` -- positional-notation digit expansion of an
approximate or exact real number. Result is `{digits, exp}` with the first
digit being the coefficient of `b^(exp - 1)`. Sign of `x` is discarded.

- Implementation: new files `src/real.c` (+ `src/real.h`); registered in
  `core_init()` between `int_init()` and `attr_init()`. Attributes
  `Protected | Listable` applied in `real_init()`. Symbol `SYM_RealDigits`
  interned in `src/sym_names.{h,c}`. Docstring registered in `src/info.c`.
- Numeric kinds handled uniformly: `Integer`, `BigInt`, `Rational`,
  machine `Real`, and `EXPR_MPFR` (under `USE_MPFR`). Symbolic numeric
  constants such as `Pi`, `E`, and `GoldenRatio` are numericalized via
  `numericalize` to enough MPFR bits to satisfy the requested base/length
  when an explicit `len` is given.
- Algorithm:
  - Exact (Integer / Rational) with no explicit `len`: remainder-tracked
    long division detects terminating vs recurring expansions. Recurring
    rationals return a nested-list block, e.g.
    `RealDigits[19/7] == {{2, {7, 1, 4, 2, 8, 5}}, 1}`.
  - General path (explicit `len` and/or `n`, or inexact `x`): scale
    `|x|` by `base^(-low)` where `low = start_pos - len + 1`, floor to an
    mpz, and read off base-`b` digits LSD-first. Pure mpz / mpq arithmetic
    for exact inputs guarantees bit-exact answers regardless of `low`.
  - Inexact path: `mpfr_get_str(.., MPFR_RNDN, avail)` produces the
    canonical round-to-nearest digits at the value's actual precision
    (`53` bits for machine doubles, `mpfr_get_prec` for MPFR). This
    yields the "shortest round-trip" decimal form Mathematica uses --
    `RealDigits[123.55555]` returns the literal `1,2,3,5,5,5,5,5,0,...`
    instead of the IEEE binary tail.
  - Available precision in base `b` is `Round[Precision[x]/Log10[b]]`.
    Requested positions below that are filled with `Indeterminate`;
    positions above the natural exponent are filled with `0`.
  - Zero special case: `RealDigits[0] -> {{0}, 0}`,
    `RealDigits[0.] -> {{0}, -Floor[Accuracy[0.]]}`. For machine doubles
    `Accuracy[0.] = 15.95` so the result is `{{0}, -15}`.
- Restrictions: bases must be integers `>= 2` and fit in `unsigned long`.
  Non-integer bases (e.g. `GoldenRatio`) trip `RealDigits::ibase` and
  leave the call unevaluated; supporting them requires MPFR floor
  iteration and has been deferred.
- Diagnostics (stderr; call is then left unevaluated):
  - `RealDigits::argb` -- argc outside `[1, 4]`.
  - `RealDigits::realx` -- concrete non-real numeric (Complex with
    non-zero imaginary part).
  - `RealDigits::ibase` -- base not an integer `>= 2` (covers `0`, `1`,
    negative, non-integer, and bignum bases > ULONG_MAX).
  - `RealDigits::intnn` -- non-negative integer expected at position 3
    (`len`).
  - `RealDigits::int` -- integer expected at position 4 (`n`).
  - `RealDigits::ovfl` -- requested `len` exceeds `REAL_MAX_LEN`
    (1_000_000 digits).
- Test suite: `tests/test_real_digits.c` (49 cases, registered in
  `tests/CMakeLists.txt` as `real_digits_tests`). Covers documented
  Mathematica examples verbatim (`58127`, `123.55555`, `Pi @ 25 digits`,
  `19/7` with and without explicit `len`, `Pi @ 20 digits starting at
  10^-5` / `10^5`, `1.234` in base 2 to 15 digits, `5.635` with
  Indeterminate fill and offset start), exact rationals (terminating
  and recurring: `1/3, 1/6, 1/7, 1/8`), MPFR inputs via `N[Pi, 50]` and
  `N[Pi, 100]`, sign discard, bignum integers (`2^100` in base 2 = 101
  digits), listable threading, zero special cases, every error path,
  attribute / docstring / interned-symbol introspection, and a 30-iteration
  repeated-evaluation stress loop for valgrind. All tests pass cleanly
  under `valgrind --leak-check=full` (system-library leaks from libobjc
  / libdyld are unrelated to Mathilda).

New built-in `FromDigits[list]` / `FromDigits[list, b]` /
`FromDigits["string"]` / `FromDigits["string", b]` -- the inverse of
`IntegerDigits` / `IntegerString`. Reconstructs a number from its
most-significant-first digit list (or character string) in a chosen
base, matching Mathematica's surface semantics.

- Implementation: extends `src/int.c` (+ `src/int.h`). Registered in
  `int_init()` alongside `IntegerDigits` / `IntegerLength` /
  `DigitCount`. Attribute `Protected` (intentionally **not**
  `Listable` -- the first argument *is* a list) declared in
  `src/attr.c` and re-applied in `int_init()`. Symbol
  `SYM_FromDigits` interned in `src/sym_names.{h,c}`.
- The all-integer fast path (every list digit AND the base are
  integer-like) is computed via Horner in GMP and demoted to
  `EXPR_INTEGER` when the result fits in `int64`, else returned as
  an `EXPR_BIGINT`. Empty list -> 0. Digits in the list need not be
  smaller than the base; they are "carried" through the Horner
  accumulation (matching Mathematica:
  `FromDigits[{7, 11, 0, 0, 0, 122}] == 810122`). Negative digits
  are also handled correctly.
- The symbolic / non-integer path (any Real, Rational, Complex, or
  symbolic digit OR base) builds the explicit Horner sum
  `Plus[Times[d[0], Power[b, n-1]], ..., d[n-1]]` and hands it back
  to the evaluator. Single code path covers symbolic base
  (`FromDigits[{1, 2, 3}, b] == 3 + 2 b + b^2`), mixed digits
  (`FromDigits[{1, a, 3}, 10] == 103 + 10 a`), full symbolic
  (`FromDigits[{a, b, c, d, e}, x] == e + d x + c x^2 + b x^3 + a x^4`),
  Real / Rational base (`FromDigits[{2, 3, 4}, 2.5] == 24.0`,
  `FromDigits[{1, 2, 3}, 1/2] == 17/4`).
- The string form decodes each character against the `[0-9a-zA-Z]`
  alphabet (`0`-`9` -> `0`-`9`, `a`-`z` / `A`-`Z` -> `10`-`35`)
  case-insensitively. Characters greater than the base are still
  decoded and carried (matching `FromDigits["1A3C"] == 2042`). Empty
  string -> 0. Base over a string is required to be an
  integer >= 2; a symbolic base over a string leaves the call
  unevaluated silently.
- Round-trip property `FromDigits[IntegerDigits[n]] == Abs[n]`
  (sign-discarding form) is asserted across machine-int, bignum, and

  == 100!`.
- Diagnostics (stderr; call is then left unevaluated):
  - `FromDigits::argb: FromDigits called with N argument(s); 1 or 2
    arguments are expected.` -- fires for `argc == 0` or `argc >= 3`.
  - `FromDigits::nlst: Argument <arg> at position 1 in <call> should
    be a list or a string.` -- concrete-but-not-list-or-string first
    arg (Integer, Real, Rational, Complex). Pure symbolic first args
    flow through silently.
  - `FromDigits::ibase: Base argument must be an integer >= 2.` --
    fires for an integer base `<= 1` (or any integer base over the
    string form that is `<= 1`).
  - `FromDigits::char: Invalid digit character <c> in string at
    position <pos> in <call>.` -- character outside `[0-9a-zA-Z]` in
    a string input.
- Test suite: `tests/test_from_digits.c` (46 cases, registered in
  `tests/CMakeLists.txt` as `from_digits_tests`). Covers all
  documented examples verbatim, the inverse-of-`IntegerDigits`
  round trip, bignum results (`2^100`, `10^30 + 12345`, `100!`),
  carrying with digits `>=` base, negative digits, symbolic / mixed
  / Real / Rational base, hex strings with mixed case, base 36
  ("Z" -> 35), 50-digit bignum strings, edge cases (empty list,
  empty string, leading zeros, single-digit lists), and every error
  path including silent unevaluated symbolic forms. A 50-iteration
  repeated-evaluation stress loop catches double-frees and mpz
  leaks under valgrind. Verified leak-free on the per-digit

  (arbitrary-precision bases) fall back to a repeated-division loop.
  Net effect: O(log n) for typical small bases, regardless of how many
  digits `n` has.
- `IntegerLength[0]` is `0` in any base (zero has no significant
  digits). Sign of `n` is discarded.
- Listable: `IntegerLength[{1, 10, 100}]` threads element-wise; same for
  the base argument or both arguments together.

  - `IntegerLength::argt: IntegerLength called with N argument(s);

    `IntegerLength[]` returns `IntegerLength[]` and
    `IntegerLength[1, 2, 3, 4]` returns `IntegerLength[1, 2, 3, 4]`,

    `test_zero_args_argt`, `test_three_args_argt`,

    integer base of 1, 0, or negative.

## UpperTriangularMatrixQ (2026-05-22)

New predicate `UpperTriangularMatrixQ` with two surface arities:

- `UpperTriangularMatrixQ[m]` returns `True` iff every entry strictly
  below the main diagonal of `m` is zero, `False` otherwise.
- `UpperTriangularMatrixQ[m, k]` returns `True` iff every entry
  `m[[i, j]]` with `j - i < k` is zero.  Positive `k` selects a
  superdiagonal above the main diagonal (the test gets stricter --
  even the main diagonal must vanish); negative `k` selects a
  subdiagonal below it (the first `|k|` subdiagonals are allowed to be
  nonzero).  Works on rectangular as well as square matrices.

An option `Tolerance -> t` relaxes the zero test to `Abs[e] <= t`,
mirroring `DiagonalMatrixQ` / `HermitianMatrixQ` / `SymmetricMatrixQ`.
`Tolerance -> Automatic` falls through to the structural (exact-zero)
test.  Without a tolerance the predicate is conservative: only literal
numeric zeros (`Integer 0`, `Real 0.0`, `BigInt 0`) count as zero, so a
symbolic sub-diagonal entry causes the matrix to be rejected.

Shape handling:
- Non-list, scalar, vector, ragged, and rank-3+ inputs return `False`.
- The empty list `{}` is not a matrix and returns `False`.
- An `n`-by-`0` matrix (e.g. `{{}, {}}`) has no entries and is vacuously
  upper triangular -- returns `True`.

## Conjugate: auto-simplify on real-valued and involutive subexpressions (2026-05-22)

`Conjugate` now reduces a small family of obvious cases that previously
returned the input unchanged:

```
Conjugate[Conjugate[z]]  ->  z          (involution)
Conjugate[Re[z]]         ->  Re[z]
Conjugate[Im[z]]         ->  Im[z]
Conjugate[Abs[z]]        ->  Abs[z]
Conjugate[Arg[z]]        ->  Arg[z]
```

nested `Conjugate`s reduces to a single `Conjugate`, an even number to

top of the function, before the existing numeric / complex-literal

pattern to hand the inner expression to the evaluator without a copy.

## HermitianMatrixQ (2026-05-22)

New predicate `HermitianMatrixQ[m]` that returns `True` iff `m` is
explicitly Hermitian (`m == ConjugateTranspose[m]`), and `False`
otherwise.  Diagonal entries must be self-conjugate.

The default test is structural and handles symbolic `Conjugate[...]`
pairs directly so it does not have to rely on `Conjugate[Conjugate[x]]`
folding (although as of 2026-05-22 that fold is now an automatic
simplification -- see the Conjugate note above).  Specifically, the
pair test accepts any of:

  1. `a == Conjugate[c]` structurally and `c == b` (covers (Conjugate[a], a));
  2. `b == Conjugate[c]` structurally and `c == a` (covers (a, Conjugate[a]));
  3. `Conjugate[b]` evaluates to a structurally-equal expression of `a`.

Two options are supported, matching Mathematica's surface API:

- `SameTest -> f`: pairs are accepted iff `f[m[i,j], Conjugate[m[j,i]]]`
  evaluates to `True`.  `SameTest -> Automatic` falls through to the
  structural test.
- `Tolerance -> t`: pairs are accepted iff
  `Abs[m[i,j] - Conjugate[m[j,i]]] <= t`.  `Tolerance -> Automatic`
  falls through to the structural test.

Non-matrix, non-square, ragged, empty, and rank-3+ tensor inputs
return `False`.  Unknown options or non-`Rule` trailing arguments
leave the call unevaluated.

## LUDecomposition: Hager-Higham MPFR condition estimator (2026-05-22)

The MPFR kernel now uses the Hager-Higham one-norm estimator (the
algorithm LAPACK's `*lacn2` uses inside `*gecon`) for the condition
number of real matrices.  The estimator runs the iteration

```
x = (1/n) e
loop:
    y = A^{-T} x
    gamma = ||y||_1
    xi = sign(y)
    z = A^{-1} xi
    if convergence test passes, return gamma
    x = e_{argmax |z|}
```

with `A^{-T}` and `A^{-1}` applied via the existing LU factorisation
plus two new helpers in `src/linalg/ludecomp_mpfr.c`:

- `lum_solve_LU_inplace` — in-place LU solve with arbitrary RHS
  (the prior `lum_solve_LU_canonical` is now a thin wrapper that
  initialises the RHS to a basis vector).
- `lum_solve_LU_T_inplace` — in-place `(LU)^T = U^T L^T` solve.
- `lum_apply_perm` / `lum_apply_perm_T` — forward and inverse row
  permutation, used to apply `P` and `P^T` between LU solves.

Cost per condition-number call drops from `O(n^3)` (one LU-solve per
column of the identity) to `O(n^2)` (2-5 LU-solve pairs total).  The
estimate is a lower bound on the true `||A^{-1}||_inf` but for
well-conditioned matrices typically agrees with the explicit-inverse
value to all available bits -- e.g. on the symmetric `{{4,1,2},
{1,3,0},{2,0,5}}` test matrix at 80 bits the two approaches agree to
ULP.

Complex matrices keep the explicit-inverse approach (the HH
sign-vector iteration in the complex case requires a separate
implementation; left for a follow-up).

Tests: `test_lu_mpfr_hh_estimate_accurate` in
`tests/test_ludecomposition_mpfr.c` pins the new estimator's value
for the symmetric test matrix; the existing MPFR cond-positive tests
all pass against the new estimator.

## LUDecomposition: smallest-|value| pivot for exact numerics (2026-05-22)

The symbolic Doolittle kernel now picks the pivot with the **smallest
absolute value** when every entry in the active column is an exact
numeric (`Integer`, `BigInt`, `Rational`, or `Complex` of those).
This matches Mathematica's behaviour on exact numeric input — probed
empirically against `wolframscript`:

```mathematica
LUDecomposition[{{4, 5}, {2, 3}}]
(* {{{2, 3}, {2, -1}}, {2, 1}, 0}  -- pivot 2 < 4 *)

LUDecomposition[{{1/2, 1/3}, {1/5, 1/7}}]
(* {{{1/5, 1/7}, {5/2, -1/42}}, {2, 1}, 0}  -- pivot 1/5 < 1/2 *)

LUDecomposition[{{2 + I, 5}, {1, 3}}]
(* {{{1, 3}, {2 + I, -1 - 3 I}}, {2, 1}, 0}  -- |1| < |2 + I| *)
```

Smallest `|pivot|` tends to keep intermediate `L` entries integer for
integer input and avoids unnecessary fraction expansion for rational
input.  For any column containing a free symbol, `Sqrt`, or other
non-exact-numeric leaf, the pivot rule falls back to "first non-zero"
— preserving the documented behaviour for `LUDecomposition[{{a, b},
{c, d}}]` returning `p = {1, 2}`.

Implementation: `numeric_abs_sq_as_mpq` and `lu_column_all_numeric`
helpers in `src/linalg/ludecomp.c` (extracts `|x|^2` exactly as
`mpq_t` for the recognised numeric forms).  The symbolic core's pivot
loop branches on `lu_column_all_numeric`: smallest-`|x|^2` for the
numeric branch, first-non-zero for the symbolic branch.

## LUDecomposition: rectangular input (2026-05-22)

`LUDecomposition[m]` now accepts any non-empty `rows x cols` matrix,
not just square `n x n`.  This matches Mathematica's contract for
rectangular input: the returned `{lu, p, c}` triple has

- `lu` of shape `rows x cols` (input shape preserved),
- `p` of length `rows` (the full row permutation; rows past
  `Min[rows, cols]` are never touched by elimination and keep their
  identity values),
- `c` the exact `Integer 0` for non-square input (the L∞ condition
  number is mathematically undefined for non-square `A`).

The Doolittle elimination terminates at step `Min[rows, cols] - 1`,
after which trailing rows of `U` are zero by construction and trailing
columns of `U` have already been filled by the Schur update.  All
three kernels (`src/linalg/ludecomp.c`, `ludecomp_machine.c`,
`ludecomp_mpfr.c`) take separate `rows, cols` parameters; LAPACK's
`dgetrf` / `zgetrf` accept rectangular input directly.  The condition
number is only computed for square input (`dgecon` / `zgecon` and the
MPFR explicit-inverse estimator both require square `A`).

Side-by-side examples vs Mathematica that now match exactly:

```mathematica
LUDecomposition[{{1, 2}, {3, 4}, {5, 6}}]
(* {{{1, 2}, {3, -2}, {5, 2}}, {1, 2, 3}, 0} *)

LUDecomposition[{{1, 2, 3}, {4, 5, 6}}]
(* {{{1, 2, 3}, {4, -3, -6}}, {1, 2}, 0} *)
```

Tests: rectangular cases added to
`tests/test_ludecomposition.c`,
`tests/test_ludecomposition_machine.c`, and
`tests/test_ludecomposition_mpfr.c` (tall + wide variants,
shape assertions, reconstruction-identity assertions).

## LUDecomposition (2026-05-22)

`LUDecomposition[m]` now factors a square matrix `m` into the standard
Mathematica `{lu, p, c}` triple:

- `lu` packs Doolittle's `L` (strictly-lower triangle, unit diagonal
  implicit) and `U` (upper triangle) into one `n x n` matrix.
- `p` is the 1-indexed row-permutation vector with
  `m[[p]] == l . u` where
  `l = LowerTriangularize[lu, -1] + IdentityMatrix[n]` and
  `u = UpperTriangularize[lu]`.
- `c` is the L∞ condition-number estimate for approximate numerical
  matrices, or the exact `Integer 0` for exact / symbolic inputs.

The implementation follows the same kernel-router pattern as
`QRDecomposition`:

- A LAPACK fast path (`src/linalg/ludecomp_machine.c`) runs
  `dgetrf` / `zgetrf` for the factorisation and
  `dgecon` / `zgecon` (with `dlange` / `zlange` for `‖A‖∞`) for the
  condition estimate; matrices dominated by IEEE doubles take this
  branch.
- An MPFR Doolittle kernel (`src/linalg/ludecomp_mpfr.c`) runs over
  row-major MPFR arrays at the input's working precision, with
  largest-magnitude pivot selection.  The condition number is
  computed from the explicit inverse via two back-substitutions per
  column, costing the same O(n³) as the factorisation itself.
- A symbolic Doolittle core (`src/linalg/ludecomp.c`) drives every
  arithmetic primitive through the Mathilda evaluator, so exact
  integer / rational / Complex / Sqrt-bearing / free-symbolic
  matrices share one code path.  Pivoting is "first non-zero" — the
  natural pivot is only displaced when it is provably zero — which
  matches the Mathematica example `LUDecomposition[{{a, b}, {c, d}}]`
  returning `{1, 2}` for `p`.

A singular `m` emits `LUDecomposition::sing` and the factorisation
completes with a zero on `U`'s diagonal at the singular step,
matching Mathematica.  Non-square or empty matrices emit
`LUDecomposition::matsq` and leave the call unevaluated.  When
BLAS/LAPACK or MPFR is unavailable at build time the corresponding
fast kernel is short-circuited and `lu_dispatch` falls back to the
symbolic core (which itself rationalises inexact input, runs the
exact pipeline, then numericalises).

New LAPACK Fortran-ABI wrappers added in `src/linalg/lapack.h` /
`lapack.c`: `mat_lapack_dgetrf`, `mat_lapack_zgetrf`,
`mat_lapack_dlange`, `mat_lapack_zlange`, `mat_lapack_dgecon`,
`mat_lapack_zgecon`.  All ship with USE_LAPACK=0 stubs returning a
negative info code so call sites link cleanly without LAPACK.

Tests live in `tests/test_ludecomposition.c` (symbolic + exact),
`tests/test_ludecomposition_machine.c` (machine precision +
condition number sanity), and `tests/test_ludecomposition_mpfr.c`
(high-precision MPFR coverage).

## D: distribute over `Equal` + `NonConstants` option (2026-05-22)

`D[a == b, x]` now distributes the derivative across both sides of an
`Equal` expression rather than treating the relation as a constant
sub-tree and collapsing to `0`. The classic implicit-circle calls
behave as expected:

```mathematica
In[1]:= D[x^2 + y^2 == 1, x]
Out[1]= 2 x == 0
```

The constant short-circuit in `compute_deriv` was suppressing the
Equal-distribution path; an explicit `f_is_equal` guard lets the
distribution fire even when both arms are syntactically free of `x`,
so the outer evaluator gets a chance to fold `Equal[0, 0]` to `True`.

`D` also now accepts a `NonConstants` option:

```mathematica
In[2]:= D[x^2 + y^2 == 1, x, NonConstants -> y]
Out[2]= 2 x + 2 y D[y, x, NonConstants -> {y}] == 0
```

Each symbol listed in `NonConstants` is treated as an implicit
function of the differentiation variable. The implementation threads
a `nonconsts` parameter through the entire `compute_deriv` recursion
(`Plus`, `Times`, `Power`, `Log`, `Equal`, `List`, `RootSum`, the
elementary chain rule, the unknown-head chain rule, and
`Derivative[...]` advancement). When the recursion reaches a symbol
listed in `NonConstants`, it returns the literal unevaluated form
`D[sym, x, NonConstants -> {...}]`; `builtin_d` recognises this
fixed-point shape on re-entry and returns the canonical
List-wrapped form so the evaluator does not loop and so the
shorthand `NonConstants -> y` is normalised to `NonConstants -> {y}`.

`expr_contains_nonconst` complements the existing `expr_free_of`
short-circuit so that sub-trees mentioning an implicitly-dependent
symbol (e.g. `2 y` with `NonConstants -> {y}`) no longer get
mis-folded to `0` before the per-head dispatch runs.

`SYM_NonConstants` is registered in `sym_names.{c,h}`. The `D`
docstring and `docs/spec/builtins/calculus.md` reflect the new
calling conventions. `tests/test_deriv.c` gains
`test_equal_distribution` (4 cases) and `test_nonconstants` (6
cases) covering the canonical examples and the option-parsing
canonicalisation. The symbolic-order path (`D[f, {x, k}]` for
symbolic `k`) intentionally does not thread `NonConstants` through;
mixed use falls back to the unevaluated `D` form.

## times, complex: two small canonicalization fixes (2026-05-21)

Two narrow simplifications now line up with Mathematica's output:

```
-1/5/Sqrt[2/5]         ->  -1/Sqrt[10]      (was unevaluated)
Conjugate[3/Sqrt[11]]  ->  3/Sqrt[11]       (was unevaluated)
```

`builtin_times` (`src/times.c`): the Sqrt-coefficient absorption guard
previously required a perfect-square extraction (`p_sq > 1 || q_sq > 1`)
before allowing `c * Power[r, -1/2]` to fold into a single radical.
That guard rejected the no-extraction case where the residual radical
collapses to an integer base (`p_rest == 1` or `q_rest == 1`), which is
exactly the shape needed for inputs like `-1/5/Sqrt[2/5]` and
`1/3/Sqrt[2/3]`. The `useful_extraction` predicate now also accepts an
`integer_radical` flag set when `p_rest == 1 || q_rest == 1`, leaving
the `2/Sqrt[30] -> Sqrt[2/15]` "Sqrt[Rational] shuffle" still blocked
(both residual parts > 1).

`builtin_conjugate` (`src/complex.c`): Conjugate already returned the
input unchanged for `EXPR_INTEGER`, `EXPR_REAL`, and `Rational`, but
left symbolic real numerics (e.g. `Sqrt[11]`, `3/Sqrt[11]`, `Pi`,
`Sqrt[2] + 3/Sqrt[5]`) unevaluated. It now also calls the existing
`is_numeric_real` predicate (same reality test used by
`complex_decompose`) before falling through to NULL, so any expression
that numericalizes to a machine real is Conjugate-fixed. Tests for the
existing complex paths (`Conjugate[Complex[2,3]]`, `Conjugate[1 + I x]`,
etc.) are unaffected because they take earlier branches.

## D: array (Outer-style) derivatives -- gradient, Hessian, Jacobian (2026-05-21)

`D[f, {{x,y}, 2}]` (and the related vector-derivative spec
`D[f, {{x1,...,xN}}]`) previously short-circuited to `0`: the parser
treated the inner `{x, y}` list as the differentiation "variable", and
the `expr_free_of` check then reported that `f` did not contain
`List[x, y]` structurally. The result was that gradients, Hessians,
Jacobians and higher-array derivatives were all silently wrong.

`parse_var_spec` in `src/calculus/deriv.c` now recognises the two
array-spec shapes:

```text
{{x1, ..., xN}}        -- gradient, order 1
{{x1, ..., xN}, n}     -- n-th array derivative; n >= 0 integer
```

and dispatches them through a new helper that implements
`First[Outer[D, {f}, array]]` directly: it recurses into `f` at every
`List` level and at each non-`List` leaf lays down the array's nested
skeleton populated with `D[leaf, v_i]` entries. The result therefore
has shape `shape(f) ++ shape(array)`, matching Mathematica's
documented semantics. For higher orders the step is applied `n` times
with intermediate evaluation (mirroring `higher_order_partial`).

Because the array-spec path is independent of the existing scalar
path, the form `D[f, {array1}, {array2}, ...]` -- equivalent to
`First[Outer[D, {f}, array1, array2, ...]]` -- falls out for free
through sequential application of single-step specs.

```mathematica
In[1]:= D[x^2 + 5 y^3, {{x, y}}]              (* gradient *)
Out[1]= {2 x, 15 y^2}

In[2]:= D[x^2 + 5 y^3, {{x, y}, 2}]           (* Hessian -- bug case *)
Out[2]= {{2, 0}, {0, 30 y}}

In[3]:= D[{x^2 + y, x y, x y z}, {{x, y, z}}] (* Jacobian *)
Out[3]= {{2 x, 1, 0}, {y, x, 0}, {y z, x z, x y}}

In[4]:= D[x^2 + 5 y^3, {{x, y}}, {{x, y}}]    (* chained = Hessian *)
Out[4]= {{2, 0}, {0, 30 y}}
```

Behavioural notes:

- Order 0 gives a deep copy of `f` (matches MMA).
- Empty array `{}` gives an empty list of derivatives.
- Variables not appearing in `f` produce 0 entries in the appropriate
  slot, preserving the result shape.
- Symbolic / non-integer orders with an array spec are not handled;
  the call stays unevaluated.

New tests: `tests/test_deriv_array.c` (10 test groups covering
gradient, Hessian, Jacobian, chained arrays, edge cases, and
regression checks for the original scalar forms).

Implementation: new helpers `outer_d_step`, `d_at_array_leaves`,
`array_higher_order` in `src/calculus/deriv.c`; extended
`parse_var_spec` with an `is_array` out-parameter; updated
`builtin_d` and `builtin_dt` dispatch accordingly.

## Derivative: operator-form reduction for user-defined functions (2026-05-21)

`f'[x]` (parsed as `Derivative[1][f][x]`) previously returned
unevaluated whenever `f` was a user-defined symbol with DownValues. The
evaluator now recognises `Derivative[n1,...,nm][f]` as a reducible head
when `f` is a symbol that has at least one DownValue: it synthesises
`Function[{t1,...,tm}, f[t1,...,tm]]` (with the DownValue rewrite
applied to the body), then runs the existing
`derivative_of_pure_function` pipeline. The resulting `Function[...]`
becomes the new head, which is then applied to the outer arguments
through the regular pure-function path.

```mathematica
In[1]:= f[x_] := x^5 + 6 x^3
In[2]:= f'[x]
Out[2]= 18 x^2 + 5 x^4

In[3]:= f'[5]
Out[3]= 3575

In[4]:= g[x_, y_] := x^2 y^3
In[5]:= Derivative[1, 1][g][a, b]
Out[5]= 6 a b^2
```

Fresh temporary symbol names are minted via a static counter
(`Derivative$<id>$<i>`) without registering them in the symbol table,
so they cannot collide with anything in the rewritten body. When `f`
has no DownValues, or the constructed call does not rewrite (arity or
pattern mismatch), the original `Derivative[...][f][...]` form is
returned unchanged.

Implementation: `derivative_of_symbol` in `src/calculus/deriv.c`,
dispatched from the §7b' branch of `evaluate_step` in `src/eval.c`
(immediately after the existing `Derivative[...][Function[...]]`
reducer).

## D: symbolic-order n-th derivatives of Sin/Cos/Sinh/Cosh (2026-05-21)

`D[F[u], {x, n}]` for `F` in `{Sin, Cos, Sinh, Cosh}` now produces
a closed form when `u` is linear in `x`. Linearity is detected by
computing `du/dx` once and checking it is free of `x`; the residual
is then exactly the linear coefficient `a`. With `u = a x + b`:

- `D[Sin[u],  {x, n}] -> a^n Sin[n Pi/2 + u]`
- `D[Cos[u],  {x, n}] -> a^n Cos[n Pi/2 + u]`
- `D[Cosh[u], {x, n}] -> a^n (-I)^n Cos[n Pi/2 - I u]`
- `D[Sinh[u], {x, n}] -> I a^n (-I)^n Sin[n Pi/2 - I u]`

`compute_deriv_symbolic_order` (`src/calculus/deriv.c`) alongside
the existing `Power[x, n]` / `Power[b, x]` symbolic-order cases.

## simp: numerical-sign fallback + general Sqrt[e^2] rewrite (2026-05-21)

Two related Simplify improvements driven by the failing case
`Simplify[Sqrt[(1 + Cos[2] + Cos[3] + Sqrt[1 + Sqrt[3]])^2] - (1 + Cos[2]
+ Cos[3] + Sqrt[1 + Sqrt[3]])]`, which previously returned the residual
`-1 - Cos[2] - Cos[3] - ... + Sqrt[(...)^2]` instead of `0`.

1. **`numeric_sign_fallback` in `prov_pos` / `prov_neg`** (`src/simp/simp.c`).
   When the structural Plus/Times sign analysis can't decide, the
   expression is numericalized (MPFR @ 100 bits when available, else
   machine double) and a real result whose magnitude exceeds a
   ~1e-20 / ~1e-10 safety margin yields a definite sign. The margin
   leaves five decades of headroom over MPFR roundoff so expressions
   that are algebraically zero do not falsely classify as nonzero. The
   `Plus` and `Times` branches now fall through to this fallback
   instead of short-circuiting `return false` when a single sub-term
   isn't provably nonneg — necessary for sums like `1 + Cos[2] + ...`
   where Cos[2] is locally negative but the total is unambiguously
   positive.

2. **`SqrtSquareRules` transform** (`src/simp/simp.c`). New walker
   `apply_sqrt_of_square_rules` rewrites
   `Power[Power[base, 2], Rational[1, 2]]` (= `Sqrt[base^2]`) based on
   the sign of `base`:

   - `base >= 0` → `base`
   - `base <= 0` → `-base`
   - `base` real, sign undetermined → `Abs[base]`

   The existing per-symbol rules only covered bare symbols
   (`Power[Power[x_symbol, 2], 1/2] :> x`); this walker generalises to
   any subexpression. Wired into `simp_search` as a Phase-0 pre-apply
   and re-applied per candidate, mirroring the `AbsRules` walker.

Composed: `Sqrt[(1+Cos[2])^2]` now simplifies to `1+Cos[2]` (the walker
turns it into `Abs[1+Cos[2]]`, then the numeric-sign fallback proves
`1+Cos[2] > 0`, and existing `Abs[nonneg] -> nonneg` rules finish).

## times: Sqrt-coefficient absorption for c / Sqrt[r] (2026-05-21)

`builtin_times` (`src/times.c`) now folds a non-trivial rational/integer
coefficient `c` into a single `Power[r, -1/2]` group by rewriting
`c * Sqrt[r]^-1` as `sign(c) * Sqrt[c^2 / r]` and re-running Power's
rational-base perfect-square extraction. This produces the canonical
Mathematica forms

```
14 / Sqrt[10]        ->  7 Sqrt[2/5]
78 / Sqrt[66]        ->  13 Sqrt[6/11]
(3/5) / Sqrt[2/5]    ->  3 / Sqrt[10]
```

Gates: applied only when `complex_val == NULL`, `group_count == 1`,
`num_prod` is rational/integer with `|num_prod| > 1`, the group
exponent is exactly `-1/2`, and the merged radicand exposes a
perfect-square factor that was not already extractable from the input.
The `useful_extraction` predicate skips purely-structural reshuffles
like `2/Sqrt[30] -> Sqrt[2/15]` that would otherwise interfere with
Plus's like-term collection during Gram-Schmidt-style pipelines.

The eps = +1 branch (`c * Sqrt[r]` with positive exponent) is left to
Power's existing integer-base extraction; folding positive-exponent
Sqrt[Rational] forms in `builtin_times` was tried but diverged from
the integer-base `c / Sqrt[r]` shapes that QR and Together /
PolynomialGCD pipelines rely on, causing Together recursion blowups
on symbolic QR inputs.

Tests added in `tests/test_radical_canonical.c`:
`test_sqrt_absorb_integer_base`, `test_sqrt_absorb_rational_base`,
`test_sqrt_absorb_collapses_to_integer`,
`test_sqrt_absorb_unchanged_when_canonical`.

Known regression: `test_qr_rank_deficient_4x3` in
`tests/test_qrdecomposition.c` now fails because `Simplify` does not
recognize `21/5 Sqrt[2/213]` and `7/5 Sqrt[6/71]` as the same value --

## linalg: add `QRDecomposition` (2026-05-21)

New builtin `QRDecomposition[m]` returning `{q, r}` such that
`m == ConjugateTranspose[q] . r`. `q` is row-orthonormal
(row-unitary for complex `m`), `r` is upper triangular, and both
have `MatrixRank[m]` rows ("thin" QR).

- Algorithm: Modified Gram-Schmidt on the columns of `m`, driven
  through the evaluator. One code path serves every input family:
  exact integer / rational, complex, machine-precision Real,
  arbitrary-precision MPFR, and free-symbolic. Inexact inputs follow
  the shared `common_rationalize_input` → exact pipeline →
  `common_numericalize_result` round-trip (same as PseudoInverse /
  Eigenvalues / Solve), so the answer comes back at the input
  precision.
- `QRDecomposition[m, Pivoting -> True]` returns `{q, r, p}` where
  `p` is a `p x p` permutation matrix with
  `m . p == ConjugateTranspose[q] . r`. Pivot selection picks the
  column with the largest residual squared-norm at each step,
  putting the diagonal of `r` in order of decreasing magnitude.
- Rank-deficient inputs (columns in the span of earlier columns)
  produce a shorter `q` / `r` automatically; the all-zero matrix
  returns `{{}, {}}`.
- For real-valued inputs (no `Complex` head, no `I` leaf) the `q`
  rows are built without `Conjugate` so the printed form stays
  free of `Conjugate[Sqrt[...]]` residues that Mathilda's
  simplifier does not currently reduce. Complex inputs go through
  the proper Hermitian inner product.
- New interned symbols: `SYM_QRDecomposition`, `SYM_Pivoting`,
  `SYM_TargetStructure` in `sym_names.{c,h}`. `TargetStructure ->
  "Structured"` is reserved for a future release (would need new
  `OrthogonalMatrix` / `UnitaryMatrix` / `UpperTriangularMatrix` /
  `PermutationMatrix` heads) and leaves the call unevaluated for
  now.
- Implementation in `src/linalg/qrdecomp.c` (`builtin_qrdecomposition`,
  registered by `qrdecomp_init`).  Attributes: `Protected`.  Docstring
  registered in `src/info.c` and reachable via `?QRDecomposition`.
- Tests in `tests/test_qrdecomposition.c` cover: 2x2 / 3x2 / 2x3 /
  3x3 / 3x4 exact-integer matrices, identity matrices, rank-deficient
  inputs (the famous `{{1,2,3},{4,5,6},{7,8,9}}`), all-zero matrix,
  exact rationals, complex 2x2 and 3x3 (verified numerically through
  `N[]` + `Chop` to sidestep Mathilda's `Conjugate[Conjugate[..]]`
  residues), machine-precision Real 2x2 / 3x3 / 3x4, MPFR at 30
  digits, `Pivoting -> True` (item count, permutation shape,
  `m . p == Transpose[q] . r` identity in both numeric and exact
  paths), `Pivoting -> False`, unknown-option recovery, vector-input
  recovery, exact-value regression for the 2x2 spec example, and a
  `q . Transpose[q] == IdentityMatrix[3]` orthogonality check.
  Standalone valgrind run on the QR-only entry points shows zero
  Mathilda-code definite leaks.

## funcprog: add `Thread` (2026-05-20)

New builtin `Thread` for threading a function over arguments that are lists
(or have any specified head):

- `Thread[f[args]]` -- threads `f` over any lists in `args`.
- `Thread[f[args], h]` -- threads over arguments with head `h`.
- `Thread[f[args], h, n]` -- threads only over the positions selected by
  `n`, which uses the standard sequence specification: `All`, `None`, a
  positive integer (first `n`), a negative integer (last `-n`), `{k}`
  (only position `k`, supports negatives), `{m, n}` (range), or
  `{m, n, s}` (range with step `s`).

- Implementation in `src/funcprog.c` (`builtin_thread`, helper
  `thread_parse_spec`): builds `h[f[...], f[...], ...]` with `L` copies,
  where `L` is the common length of all threadable args whose head is `h`;
  arguments outside the position set (or whose head differs from `h`) are
  broadcast unchanged into every replica. The result is re-evaluated so
  inner `Listable` / `Orderless` heads collapse normally.
- All `args` whose heads are `h` must be of the same length, otherwise
  the expression is returned unchanged (matches Mathematica's
  `Thread::tdlen` recovery shape; the message itself is omitted).
- Atoms pass through unchanged (`Thread[5]` -> `5`).
- New interned symbol `SYM_Thread` in `sym_names.{c,h}`.
- Attributes: `Protected`.
- Docstring registered in `src/info.c` and reachable via `?Thread`.

## linalg: add `MatrixRank` (2026-05-20)

Added the `MatrixRank[m]` (and `MatrixRank[m, Method -> "<name>",
Tolerance -> t]`) builtin in a new translation unit
`src/linalg/matrank.c`.

- Returns the number of linearly independent rows of `m`
  (equivalently, of linearly independent columns). Always evaluates
  to a non-negative `Integer`; otherwise the call is left
  unevaluated.
- Supports numerical (Integer / Rational / Bignum / Real / MPFR /
  Complex) and symbolic matrices, square or rectangular. Empty or
  non-rank-2 inputs emit `MatrixRank::matrix` and remain
  unevaluated.
- **Two execution paths.** When every entry is exact and the user
  did not supply a `Tolerance`, MatrixRank routes through
  `RowReduce[m, Method->...]` and returns the count of non-zero
  rows in the RREF (using `is_zero_poly` for structural zero,
  matching the existing pivot-finder convention in
  `src/linalg/inv.c`). When the matrix contains any Real / MPFR
  leaf, or the user supplies `Tolerance`, MatrixRank runs a
  partial-pivot Gaussian forward-elimination over a portable
  `double`-complex `cplx_t` struct (avoiding `_Complex` for C99
  portability) with tolerance-aware pivot selection: a column is
  skipped when its largest sub-pivot `|entry|` is `<= t`.
- **Tolerance grammar.** Accepts `Tolerance -> Automatic` (the
  default; uses `max(rows, cols) * MachineEpsilon * max(|entries|)`
  for inexact matrices, 0 otherwise), `Tolerance -> 0` (no
  tolerance, exact agreement with the symbolic path on integer /
  rational input), `Tolerance -> <non-negative number>` (Integer,
  Real, Rational, or `Power[10, k]`-style expressions resolved via
  `N[...]`). Negative or symbolic tolerances emit `MatrixRank::opt`
  and leave the call unevaluated.
- **Method grammar.** Shares `matsol_parse_method_option` with
  RowReduce / LinearSolve / Inverse / NullSpace
  (`src/linalg/linsolve.h`). Method only affects the exact path;
  the numerical path always uses the partial-pivot Gauss kernel.
- **Spec parity.** Reproduces the spec's tolerance ladder
  exactly: on `m = {{1,1,1},{0,10^-10,0},{0,0,10^-20}}`,
  `MatrixRank[m]` = 3 (exact),
  `MatrixRank[N[m]]` = 2 (default tolerance drops 10^-20),
  `MatrixRank[N[m], Tolerance -> 0]` = 3 (no tolerance),
  `MatrixRank[N[m], Tolerance -> 10^-8]` = 1 (only the first row
  survives).
- Attributes set to `{Protected}` in `matrank_init`. Symbol
  interned as `SYM_MatrixRank` in `src/sym_names.{c,h}`. Docstring
  added in `src/info.c`.
- Wired into `core_init` via `matrank_init` immediately after
  `matnull_init` (preserving the matsol → matinv → mateigen →
  matnull → matrank ordering).
- Coverage: `tests/test_matrank.c` registers `matrank_tests` in
  `tests/CMakeLists.txt` with 36 named tests grouped into
  exact-form (3×3, 4×4, rectangular, identity, diagonal, rational,
  bigint, symbolic, zero, 1×3, 3×1), Method (DivisionFree, OneStep,
  Cofactor, Automatic symbol + string, unknown), Tolerance / numerical
  (the four-step spec ladder above, complex dependency,
  exact-with-tolerance, rational tolerance, invalid tolerance,

## cond: add `Piecewise` (2026-05-20)

Added the `Piecewise[{{val_1, cond_1}, {val_2, cond_2}, ...}, default]`
builtin to `src/cond.c`. Semantics match Mathematica's `Piecewise`:

- `HoldAll`, `Protected` attributes — values are not evaluated unless
  their clause survives simplification. Only the surviving `val_i` is
  returned (held) for the outer evaluator to reduce.
- The first argument must be a `List` of two-element `{val_i, cond_i}`
  pairs. Malformed shapes (non-`List` first arg, pairs of the wrong
  arity, more than two top-level arguments, zero arguments) leave the
  call unevaluated so the user can see what they wrote.
- The optional second argument is the default value used when no
  condition fires; it defaults to `0`. `Piecewise[conds]` rewrites to
  `Piecewise[conds, 0]`.
- Conditions are evaluated left-to-right via the standard evaluator:
  - `{val_i, False}` clauses are dropped.
  - At the first `{val_i, True}` all later clauses (and the default)
    are dropped; the `True` clause becomes the final unconditional
    case in the symbolic form.
  - If every preceding condition is literally `False`, the value at
    the first `True` is returned directly (and evaluated by the
    surrounding loop).
- Consecutive surviving clauses with structurally equal values are
  merged: their conditions are combined via `Or` and the combined
  condition is re-evaluated so that boolean simplifications (`d ||
  True -> True`, `d || False -> d`) kick in.
- When the simplification leaves the input structurally unchanged, the
  builtin returns `NULL` so the outer fixed-point loop terminates
  efficiently without an `expr_eq` compare.

(used by the `Limit`/`D` machinery). The new builtin registers it via
`symtab_add_builtin`; docstring added to `src/info.c`; spec entry added

## cond: add `Switch` (2026-05-20)

Added the `Switch[expr, form_1, value_1, form_2, value_2, ...]`
control-flow builtin to `src/cond.c`. Semantics match Mathematica's
`Switch`:

- `HoldRest`, `Protected` attributes — `expr` is evaluated normally by
  the outer evaluator, but the form/value pairs are held and processed
  lazily one pair at a time by the builtin itself.
- Each `form_i` is evaluated immediately before its match is tried.
  The first `form_i` that pattern-matches `expr` (via the standard
  `match()` engine, so typed blanks like `_Integer`, `_Plus`,
  predicates `_?test`, and conditioned patterns `p /; cond` all work)
  causes `Switch` to return the corresponding `value_i` unevaluated;
  the outer fixed-point loop then evaluates only that value, leaving
  every other branch untouched.
- A trailing form of `_` (Blank) acts as a catch-all default clause,
  fall-through behavior that emerges naturally from "blank matches
  anything" rather than being special-cased.
- If no form matches `expr`, the call is returned unevaluated so a
  subsequent `/.` substitution can resolve it later.
- Wrong arity (`Switch[]`, `Switch[x]`, `Switch[x, a]`, or any odd
  number of arguments after `expr`) is a usage error; the call is
  returned unevaluated.
- Pattern bindings produced by the form (e.g. `{x_, y_}`) are *not*
  substituted into `value_i`; the form acts purely as a discriminator,

## cond: add `Which` (2026-05-20)

Added the `Which[test1, value1, test2, value2, ...]` control-flow
builtin to `src/cond.c`. Semantics match Mathematica's `Which`:

- `HoldAll`, `Protected` attributes — tests and values are evaluated
  lazily by the builtin itself, one pair at a time.
- The first `test_i` that evaluates to `True` causes `Which` to return
  the corresponding `value_i` (which is then evaluated normally by the
  outer fixed-point loop).
- Tests that evaluate to `False` are skipped.
- A test that evaluates to anything other than `True` or `False` causes
  `Which` to return symbolically as `Which[test_i_eval, value_i, ...]`
  with the remaining (still-held) pairs preserved verbatim. This makes
  `Which` work cleanly with later `/.` substitution.
- All tests `False` (or no arguments at all) returns `Null`.
- A trailing test of `True` acts as a default clause, mirroring the
  idiom `sign[x_] := Which[x < 0, -1, x > 0, 1, True, Indeterminate]`.
- An odd number of arguments is a usage error; the call is returned
  unevaluated so the user can see what they wrote.

## mateigen.c — "FEAST" Eigenvalues / Eigenvectors method, machine + MPFR (Phases 1–5, 2026-05-20)

`Eigenvalues` and `Eigenvectors` now accept `Method -> "FEAST"` for
Hermitian (real symmetric or complex Hermitian) inexact numeric input.
Unlike Direct / Arnoldi / Banded, FEAST returns only the eigenpairs
whose eigenvalues lie in a user-supplied real interval `[a, b]` — a
spectral-slice query rather than a full decomposition.  Automatic
never routes to FEAST (it requires the user to commit to an interval);
explicit `Method -> "FEAST"` is the only way in.  The sub-option
grammar mirrors Arnoldi / Banded:

```mathematica
Method -> "FEAST"
Method -> {"FEAST", "Interval" -> {a, b},
                    "ContourPoints"  -> 8,
                    "SubspaceSize"   -> 16,
                    "MaxIterations"  -> 20,
                    "Tolerance"      -> 10^-10}
```

`"Interval"` is the only required key; the parser auto-swaps `{b, a}`
to `{a, b}` and a degenerate interval `{c, c}` (or one whose bounds
NaN-fail to coerce) trips the kernel's `high <= low` guard and falls
through to Direct.  Unknown keys are silently ignored, matching the
Arnoldi / Banded conventions.

### Algorithm

FEAST (Polizzi 2009) is a contour-integral spectral-projector method:

1. Pick `m_0` (default `max(20, n/4)`, capped at `n`) and initialise a
   random `n x m_0` subspace `Y`.
2. Approximate `P_[a,b](A) Y = (1/(2 pi i)) ∫_C (z I - A)^-1 Y dz` by
   `Ne`-point Gauss-Legendre quadrature on the upper half of the
   elliptic contour through `(a, 0)` and `(b, 0)`; Schwarz symmetry
   halves the number of complex linear solves to `Ne / 2`.  Supported
   `Ne` values are 2, 4, 8 (default), and 16.
3. Form `A_q = Q^* A Q`, `B_q = Q^* Q` and solve the small
   generalised Hermitian-definite problem `A_q y = lambda B_q y` by
   Cholesky reduction `B_q = L L^*` + the existing Direct kernel on
   `L^-1 A_q L^-*`.
4. Filter Ritz pairs against `[a, b]`, lift back to `A`-eigenvectors,
   and iterate until the residual converges or `MaxIterations` is
   reached.  Output is sorted by `|lambda|` descending so `k_spec`
   (`Eigenvalues[m, k]`, `Eigenvalues[m, -k]`, `Eigenvalues[m, UpTo[k]]`)
   composes naturally with the in-interval result.

The complex linear solve at each Gauss-Legendre node is a partial-
pivoted LU on paired re/im storage (`feast_complex_lu_factor` /
`_solve` at machine precision, `_M` variants at MPFR precision), with
no dependency on `libmpc`.

### Fail-soft cascade

The dispatcher returns `NULL` (and the wrapper falls through to
Direct) on any of the following:

- non-Hermitian input,
- `"Interval"` missing or non-coercible to two finite doubles,
- `interval_high <= interval_low` (catches the degenerate
  `{c, c}` and `{NaN, ...}` cases),
- generalised eigenproblem (`Eigenvalues[{m, a}]`),
- Cholesky failure on `B_q` (subspace too small for the spectral
  count in the interval),
- LU singular at any quadrature node,
- non-convergence within `MaxIterations`.

The first such fall-back per process emits a single
`Eigenvalues::feast: FEAST machine kernel could not converge (...);
falling back to the Direct method.` stderr line tagged with the
reason, so a user who passes `Method -> "FEAST"` always gets *some*
sensible answer back — at worst, the full Direct spectrum.

### Reuse map

- `direct_sort_perm_desc_abs` / `_M`, `direct_apply_k_spec_list`,
  `direct_build_real_eigenvalue_list` / `_M`,
  `direct_build_real_eigenvector_list` / `_M`,
  `direct_build_complex_hermitian_eigvec_list` / `_M` — shared
  emit-side helpers, unchanged.

  `matM_is_hermitian` — Hermitian detection, unchanged.

## mateigen.c — "Banded" Eigenvalues / Eigenvectors method, machine + MPFR (Phase 4, 2026-05-19)

`Eigenvalues` and `Eigenvectors` now accept `Method -> "Banded"` for
Hermitian (real symmetric or complex Hermitian) inexact numeric input.
The Automatic dispatcher routes here when the matrix is Hermitian,
`n > 8`, and the half-bandwidth `b ≤ max(8, n/4)`; outside that
window Automatic stays on `"Direct"` (or `"Arnoldi"` for the small-k
path).  Banded explicitly refuses (returns `NULL`, the wrapper falls
back to Direct) on non-Hermitian inputs and on fully dense matrices
(`b == n - 1`), so explicit `Method -> "Banded"` is always safe to
pass — it simply elides the band optimisation when none is available.

### Algorithm

Two-sided Givens rotations zero one off-band entry at a time and
chase the bulge they introduce until it falls past the matrix edge
(Schwarz 1968 / Rutishauser).  For each anchor column `k = 0..n-3`
and each subdiagonal `d = b, b-1, ..., 2`, a Givens in plane
`(k+d-1, k+d)` zeroes `A[k+d, k]` using the in-band entry
`A[k+d-1, k]`.  The two-sided application produces a fresh bulge at
`(p, q+b) ≡ (q+b, p)`; a chase Givens in plane `(q+b-1, q+b)`
eliminates it and propagates a new bulge another `b` columns
along, until `cq + b ≥ n`.  After all outer steps the matrix is
symmetric tridiagonal and is handed off to the existing Phase 2
symmetric tridiag QR (`direct_symtridiag_qr` / `_M`) — no new QR
code is added.

Complex Hermitian banded reuses the Phase 2 phase-correction step
(`direct_phase_correct_tridiag`): the band-Givens produces a
Hermitian tridiagonal with real diagonal and complex sub-diagonal,
the phase correction makes the sub-diagonal real-positive, and the
final QR runs at real symmetric tridiagonal precision.  The Givens
itself uses Wilkinson's real-c / complex-s parameterisation
(`c = |a|/r`, `s = b · conj(a) / (|a| · r)`), which keeps the
mathematics in paired re/im doubles (no `libmpc`).

- `direct_phase_correct_tridiag` and `_M` (Phase 2) — unchanged.
- `direct_compose_complex_Q_real_Z` and `_M` (Phase 2) — unchanged.
- `direct_sort_perm_desc_abs` / `_M`, `direct_build_real_eigenvalue_list`

- `matD_is_real_symmetric`, `matD_is_hermitian`, `matM_is_real_symmetric`,
  `matM_is_hermitian` — shared with Direct's dispatcher.
- New: `matD_bandwidth_real` / `_complex`, `matM_bandwidth_real` /

- New: `banded_givens_two_sided_real` / `_complex` (machine) and

## mateigen.c — "Arnoldi" Eigenvalues / Eigenvectors method, machine precision (Phase 3 steps 3a–3e, 2026-05-19)

`Eigenvalues` and `Eigenvectors` now accept `Method -> "Arnoldi"` (and
the sub-option form `Method -> {"Arnoldi", "BasisSize" -> m,
"MaxIterations" -> N, "Tolerance" -> t}`) for inexact numeric input.
The Automatic dispatcher routes here when `n > 32` and a small `k` is
requested (`k <= max(20, n/10)`); small matrices always go through
`"Direct"` because for those Direct is both faster and exact.

### Algorithm

An m-step Krylov subspace `K_m(A, v_0) = span{v_0, A v_0, ..., A^{m-1} v_0}`
is built via classical Gram–Schmidt with one re-orthogonalisation pass
("twice is enough" — Daniel et al. 1976), producing an orthonormal
basis V_m and an m × m upper Hessenberg `H_m` such that
`A V_m = V_m H_m + h_{m+1,m} v_{m+1} e_m^T`.  H_m is then diagonalised
by the **same Phase 2 Francis QR pipeline** used by Direct
(`direct_qr_real_general` + `schur_compute_eigvecs`), and Ritz vectors
`V_m y_i` are lifted back to A-eigenvectors.  Default basis size is
`max(2k, 20)` capped at n; on lucky breakdown (||w|| below tolerance
at step j) Arnoldi terminates early with j+1 exact eigenpairs.

Complex inputs use paired re/im storage for V_m and H_m (no `libmpc`
dependency), and route H_m's diagonalisation through a 2mu × 2mu real
block embedding `M = [[Re H_m, -Im H_m], [Im H_m, Re H_m]]` — the
same trick that `direct_complex_general_machine` uses on A directly.
Per-distinct-eigenvalue grouped Gram–Schmidt deduplicates M's doubled
spectrum back to mu A-eigenpairs.

### Sub-option grammar

```
Method -> "Arnoldi"
Method -> {"Arnoldi", "BasisSize" -> 20,
                     "MaxIterations" -> 500,
                     "Tolerance"     -> 1.0*^-10}
```

`BasisSize` overrides the default m; `MaxIterations` is parsed and
reserved for the Phase 3f explicit-restart path; `Tolerance` sets the
||w|| breakdown threshold (default `1e-12 * n * ||A||_inf` capped at
`1e-16 * ||A||_inf`).

### Code

- `ArnoldiOpts` + `arnoldi_set_defaults` + `arnoldi_parse_subopts`:
  parse and default the three sub-options from a
  `Method -> {"Arnoldi", ...}` RHS.
- `arnoldi_target_k`, `arnoldi_default_basis_size`,
  `arnoldi_automatic_prefers`: the routing heuristic.
- `mateigen_build_complex_eigvec_list_rect`: rectangular variant of
  the existing eigvec emitter for the mu < n case.
- `arnoldi_real_general_machine`: real Arnoldi factorisation + lift.
- `arnoldi_complex_general_machine`: complex (paired re/im) Arnoldi
  factorisation + 2mu × 2mu block-embed + lift.
- `arnoldi_dispatch_machine` and the top-level `arnoldi_dispatch`:
  shape selection + sub-option plumbing.

Hermitian / symmetric inputs flow through these same general kernels
— H_m comes out (nearly) symmetric / Hermitian for free.  A dedicated
Lanczos fast path is intentionally deferred (the test-corpus matrices
are ≤ 50, so per-step savings are minor).

### Builtin wiring

`builtin_eigenvalues` and `builtin_eigenvectors` now try the
`arnoldi_dispatch` path first when `opts.method == ARNOLDI` (always)
or when `Automatic` is paired with a small `k`.  On Arnoldi failure
they fall through to `direct_dispatch`, then to the symbolic
characteristic-polynomial path — so an explicit Arnoldi request never
returns an error, only the best available result.

### Tests

15 new tests in `tests/test_mateigen_arnoldi.c` covering the real
symmetric / general / complex Hermitian / complex general kernels,
sub-option parsing, the Automatic-with-small-k routing, the UpTo[k]
form, residual checks for Ritz vectors, and the 1×1 edge case.  Each
non-trivial fixture compares Arnoldi's output against the matching
Direct call, so any divergence above `1e-9` flags a regression.

### MPFR (step 3f, 2026-05-19)

MPFR Arnoldi mirrors the machine kernels with arbitrary-precision
arithmetic, scratch-pool discipline (no `mpfr_init` / `mpfr_clear` in
inner loops), and reuse of the 2d-A..D infrastructure:

- `arnoldi_real_general_mpfr`: paired mpfr_t V_m / H_m, then route
  H_m through `direct_qr_real_general_M` + `schur_compute_eigvecs_M`
  (the same Phase 2d-B + Schur back-sub used by `direct_real_general_mpfr`).
- `arnoldi_complex_general_mpfr`: paired re/im mpfr_t V_m / H_m,
  2mu × 2mu real-block embedding, then grouped complex Gram–Schmidt
  at MPFR precision (twice for stability) to disambiguate the
  doubled spectrum.  Group tolerance scales like `N · ||spec|| · 2^{-bits/2+6}`
  and the extraction threshold like `sqrt(mu) · 2^{-bits/2+6}`.
- `arnoldi_dispatch_mpfr`: top-level shape selector for the MPFR path;
  the public `arnoldi_dispatch` now checks
  `common_scan_inexact(m).min_bits > 53` and prefers MPFR when it
  fires, otherwise falls through to the machine kernel.

Hermitian / symmetric inputs again flow through the same general MPFR
kernels — the special-case Lanczos fast path is intentionally deferred.

### Tests

15 machine-precision tests + 6 MPFR tests in
`tests/test_mateigen_arnoldi.c` covering each kernel.  The MPFR tests
compare Arnoldi's |λ| against Direct's |λ| at 60 and 80 bits with a
1e-15 tolerance; the eigenvector tests verify return shape and that
input precision carries through to result components.

### Code (2d-C: complex Hermitian)

- `matM_norm_inf_complex` and `matM_is_hermitian` — paired with the
  existing real helpers from 2d-A so the dispatcher detects shape on
  both real and complex MPFR inputs uniformly.
- `direct_tridiag_complex_hermitian_M` — paired-`mpfr_t` (real / imag)
  Hermitian Householder tridiagonalisation; mirrors the machine

  onto the tridiagonal so the subdiagonal becomes real-positive,
  updating Q by right-multiplication.  After this step the tridiagonal

### Code (2d-D: complex general)

- `direct_complex_general_mpfr` reuses the 2d-B real-general MPFR
  pipeline via the same 2n × 2n real block embedding
  M = [[R, −S], [S, R]] used by the machine kernel

- After running M through `direct_hessenberg_real_M` +
  `direct_qr_real_general_M` + `schur_compute_eigvecs_M`, the

  (twice for stability) extracts the n distinct A-eigenvectors

- Tolerances:
  - `group_tol = N · spec_norm · 2^{−bits/2 + 6}` — matches the
    LAPACK-style "n · sqrt(eps)" rule, generous enough to group

  - `extract_threshold = sqrt(n) · 2^{−bits/2 + 6}` — between the
    +J subspace norm (~ sqrt(2)) and the −J subspace norm (~ 2^−bits).

## mateigen.c — real general (non-symmetric) MPFR "Direct" kernel (Phase 2 step 2d-B, 2026-05-19)

`Eigenvalues` and `Eigenvectors` now route real **non-symmetric** MPFR
inputs through an arbitrary-precision Hessenberg + Francis QR + Schur
back-substitution kernel, rather than downgrading to the machine path.
With 2d-A and 2d-B combined, all real-matrix shapes now compute
spectrum and eigenvectors at the matrix's input precision.

### Code

- `direct_hessenberg_real_M`, `direct_francis_step_M`,
  `direct_split_2x2_real_M`, and `direct_qr_real_general_M` —
  MPFR transcriptions of their `double` counterparts. Each `double`
  arithmetic op becomes an `mpfr_*` call rounded `MPFR_RNDN`; the
  Francis step takes a 14-cell caller-supplied scratch pool so
  `mpfr_init`/`mpfr_clear` stay out of the bulge-chase inner loop.
- New exceptional-shift mechanism inside `direct_francis_step_M`
  (LAPACK `dlahqr`-style): the QR driver passes `exceptional = true`
  every 10th iteration, replacing the trailing-2×2 shift with
  `s = 1.5 · (|H[p-1, p-2]| + |H[p-2, p-3]|)`. Without this the
  high-precision sweep can stall in a 2-cycle for matrices whose
  trailing 2×2 has the same complex shift after each pass
  (observed on `{{1, -1, 0}, {1, 1, -1}, {0, 1, 1}}` at 50+ digits).
  Machine precision masks this naturally via roundoff; MPFR does not.
- `schur_eigvec_real_M` and `schur_eigvec_complex_M` — back-
  substitution on `(H − λI)` for real and complex eigenvalues in the
  Schur basis, mirroring the machine versions. `schur_compute_eigvecs_M`
  composes `V = Q · v_schur` and normalises to unit 2-norm.
- `direct_sort_perm_desc_abs_complex_M`,
  `direct_build_complex_eigenvalue_list_M`, and
  `direct_build_complex_eigenvector_list_M` — emit `List[mpfr, ...]`
  or `List[Complex[mpfr_re, mpfr_im], ...]` depending on whether the
  imaginary part is zero.
- `direct_real_general_mpfr` is the top-level entry that owns the
  workspace lifecycle (`H`, optional `Q`, Householder vector,
  14-cell scratch, eigenvalue arrays).
- `direct_dispatch_mpfr` now selects between `direct_real_sym_mpfr`
  and `direct_real_general_mpfr` based on `matM_is_real_symmetric`,
  matching the machine dispatcher's structure.
- Deflation tolerance: `rel_tol = 2^{-bits+10}` (1024 ULPs), about
  one decimal digit looser than the symmetric tridiag tolerance.
  Non-symmetric implicit-shift QR accumulates more per-sweep roundoff
  than the symmetric path (factor ~n^2 from bulge chase plus back-
  transformation), so a slightly more permissive threshold is needed
  for clean convergence on adversarial high-precision inputs.
- `max_iter` raised to `60·n + 30` (was `30·n` in the machine kernel).
  Even with exceptional shifts, MPFR convergence at very high bit
  counts can require more sweeps before deflation fires.

## mateigen.c — complex non-Hermitian "Direct" kernel (Phase 2 step 2c-B, 2026-05-19)

`Eigenvalues` and `Eigenvectors` now also handle **complex
non-Hermitian** matrices at machine precision; the symbolic
characteristic-polynomial fallback is reserved for arbitrary-precision
MPFR and the symbolic path proper.  With this commit the Direct method
covers every machine-precision matrix shape.

### Algorithm — real block embedding

A complex n × n matrix `A = R + i S` is embedded into the real
2n × 2n block matrix

    M = [[ R, -S ],
         [ S,  R ]]

whose spectrum is `spec(A) ∪ conj(spec(A))` as a multiset.  The
existing real machinery handles M:

1. **Hessenberg reduction** via `direct_hessenberg_real` with `Q`
   accumulation.
2. **Francis double-shift QR** via `direct_qr_real_general`.
3. **Schur back-substitution** via `schur_compute_eigvecs` to get M's
   eigenvectors (always — `Q` is needed even for `WANT_VALUES` to
   disambiguate the conjugate-pair doubling).

### Eigenvector recovery + the +J / −J split

For each M-eigenvector `w = w_re + i w_im` with `M w = mu w`, the
candidate

    x = (a - d) + i (b + c)

where `w_re = [a; b]` and `w_im = [c; d]` are the n-vector top / bottom
splits, satisfies `A x = mu x`.  When `mu` is in `spec(A)` the candidate
is non-zero; when `mu` is only in `conj(spec(A))` (i.e. `conj(mu) ∈
spec(A)` but `mu` itself is not) the candidate is zero.  This is the
+J / −J splitting of M's eigenspace into the A-action and conj(A)-
action subspaces.

### Grouped Gram-Schmidt for multiplicity

Algebraic multiplicity is handled by grouping M-eigenvalues that fall
within `1e-8 ‖M‖ n` of each other (one Schur position-order pass
identifies adjacent equivalents), then running complex modified
Gram-Schmidt twice over each group's candidate `x`'s.  The "twice is
enough" rule keeps the orthogonalisation numerically stable; vectors
that fall below `sqrt(n) · 1e-9` after orthogonalisation are dropped as
−J contributions.  The extracted basis size per group equals `m_A(mu)`
and the totals sum to exactly `n`.

This correctly handles the "mixing" case where both `λ` and
`conj(λ)` are in `spec(A)` for some non-real λ (e.g.
`[[0,1,0],[-1,0,0],[0,0,2+i]]` has spec `{i, -i, 2+i}` and the simple
pair-up algorithm would emit `{i, i, 2+i}` here -- the grouped GS
correctly emits `{2+i, i, -i}`).

### Dispatch

`direct_dispatch_machine` now routes complex non-Hermitian inputs to
`direct_complex_general_machine`; the symbolic char-poly fallback is
preserved as the final escape hatch.

### LAPACK-HOOK annotations

`zgehrd + zhseqr + ztrevc3` (or `zgeev` as a one-call wrapper) is the
LAPACK drop-in.  The block embedding nominally costs 8x more flops at
`-O3` than a native complex Hessenberg + complex QR; vendor LAPACK on
the complex path will recover that 8x.

## mateigen.c — complex Hermitian "Direct" kernel (Phase 2 step 2c-A, 2026-05-19)

`Eigenvalues` and `Eigenvectors` now route complex **Hermitian**
matrices at machine precision through a hand-rolled native-complex
kernel instead of falling back to the symbolic
characteristic-polynomial pipeline.

### Algorithm

1. **Complex Householder tridiagonalisation** of the Hermitian input.
   Reflectors `H_k = I - 2 u u^H / ||u||^2` are applied symmetrically
   via the rank-2 update `A <- A - u q^H - q u^H` with `q = 2 A u -
   2 alpha_v u` (`alpha_v = u^H A u`, real for Hermitian).  The result
   is a Hermitian tridiagonal `T` with real diagonal but generally
   complex sub-diagonal.  Complex unitary `Q` accumulates the
   reflectors when eigenvectors are requested.
2. **Diagonal phase correction**: a unitary diagonal `D = diag(d_k)`
   with `|d_k| = 1` and `d_{k+1} = d_k * sub[k] / |sub[k]|` is applied
   so `D^H T D` has real-positive sub-diagonal, i.e. becomes a real
   symmetric tridiagonal.  `Q` is updated to `Q D`.
3. **Real symmetric tridiag QR** (existing `direct_symtridiag_qr`)
   finds eigenvalues `Lambda` and accumulates the real orthogonal
   `Z` with `Z^T T_real Z = Lambda`.
4. **Final eigenvectors** `V = Q Z` (complex `Q` times real `Z`).

This avoids needing a separate complex tridiagonal QR; the only added
cost over the real symmetric path is an `O(n^2)` diagonal-phase pass.
See Wilkinson 1965, "The Algebraic Eigenvalue Problem", §5.45.

### Dispatch

`direct_dispatch_machine` now splits real vs complex.  Complex inputs
that pass `matD_is_hermitian` (within `1e-12 n ||A||_inf` tolerance on
both the imag-diagonal and the conjugate-symmetry checks) route to
`direct_complex_hermitian_machine`; complex non-Hermitian inputs still
fall through to the symbolic path (step 2c-B).

### LAPACK-HOOK annotations

The new section is annotated for future drop-in replacement: `zhetrd`
(step 1, with `tau` covering the implicit phase reduction) +
`dsteqr` / `dstedc` (step 3), or `zheevd` as a one-call wrapper.

## matsol.c — LinearSolve accepts array-shaped m and b (2026-05-19)

`LinearSolve[m, b]` previously rejected anything other than a rank-2
`m` with a rank-1 or rank-2 `b`. Mathematica accepts higher-rank
tensors: a rank-N `m` with dimensions `{d_1, ..., d_{N-1}, n}` is
interpreted as a `(d_1 * ... * d_{N-1}) x n` linear system, and `b`
with dimensions `{d_1, ..., d_{N-1}, e_1, ..., e_p}` produces a
result of shape `{n, e_1, ..., e_p}`. The reported failure case was

```
a = RandomInteger[{-3, 3}, {2, 3, 6}];
b = RandomInteger[{-3, 3}, {2, 3, 4, 5}];
Dimensions[LinearSolve[a, b]]   (* expected {6, 4, 5} *)
```

`builtin_linearsolve` in `src/matsol.c` now:
- accepts `m_rank >= 2`, computing `r` as the product of `m`'s leading
  dims and `c` as its last dim;
- validates that `b`'s leading `m_rank - 1` dimensions match `m`'s
  leading dims (otherwise `LinearSolve::lvec1`);
- computes `trail_dims` / `trail_rank` from `b`'s remaining dims and
  passes them to each method-specific solver.

The three solvers (`linearsolve_divfree`, `linearsolve_onestep`,
`linearsolve_cofactor`) now build their flat `c * k` answer array and
hand it to a shared `build_solution_tensor` helper which assembles
the nested `{c, trail_dims...}` `List`. This collapses the previous
`b_rank == 1 ? flat-list : list-of-lists` branch into a single code
path that generalises to any trailing rank.

`docs/spec/builtins/linear-algebra.md` and `tests/test_linearsolve.c`
gained a `test_linearsolve_array_inputs` covering the bug-report
case, rank-3 m with vector b (`trail_rank == 0`), rank-3 m with
rank-3 b (`trail_rank == 1`), and a leading-dim mismatch error.

## core.c, part.c — `Dimensions[expr, n]` and `Head[expr, h]` (2026-05-18)

Added the second-argument forms of `Dimensions` and `Head` to match

`?Head`) with full descriptions and examples.

- `Dimensions[expr, n]` truncates the result to the first `n` levels.
  The internal `get_dimensions` walker now carries a `max_depth` cap and
  short-circuits once that depth is reached, so deep structures aren't
  fully walked when only the top levels are requested.  Raggedness is
  still detected within the requested level range.  `n = 0`, atomic

  in WL.  Non-integer or negative `n` leaves the call unevaluated.
- `Head[expr, h]` returns `h[Head[expr]]`; the wrapper `h` is left to

- Test coverage in `tests/test_eval.c::test_dimensions` and
  `tests/test_eval.c::test_builtin_head` covers both forms, ragged
  structures, atomic inputs, the empty list, depth-cap truncation, and
  wrapper evaluation.



All four `LeastSquares` Method options now have dedicated algorithmic
implementations instead of dispatching through `"Direct"`.

- **`"IterativeRefinement"`** is now a real loop: `x <- x +
  PseudoInverse[m] . (b - m . x)` repeated until
  `Total[Flatten[dx]^2] <= Tolerance^2` or a 50-iteration cap.  For exact
  inputs the first correction is exactly zero (Moore-Penrose identity
  `pinv m pinv = pinv` gives `dx = pinv (b - m pinv b) = 0`) so the loop
  converges in one pass; for inexact inputs the loop drives round-off
  down to Tolerance.  The squared-norm test uses
  `Total[Flatten[dx]^2]` which is correct for both vector and matrix
  RHS.
- **`"Krylov"`** is Conjugate-Gradient-on-Least-Squares: starting from
  `x_0 = 0` (so the iterate stays in `range(A^T)` and converges to the
  minimum-norm LS solution for rank-deficient `m`), the loop maintains
  `r = b - A x`, `s = A^T r`, `p`, `gamma = |s|^2`.  Each step computes
  `q = A p`, `alpha = gamma / |q|^2`, then updates `x`, `r`, `s`,
  `beta = |s_new|^2 / |s|^2`, and `p <- s_new + beta p`.  Exits on
  `gamma <= Tolerance^2` (numeric) or `gamma == 0` (exact), on a
  null-direction `|q|^2 == 0`, or at the `2 cols(A) + 10` cap.  Matrix
  RHS are solved column-by-column and recombined via `Transpose`.
  Symbolic inputs (the convergence test is undecidable for them) fall
  back to `"Direct"`.
- **`"LSQR"`** is the canonical Paige-Saunders 1982 algorithm: Lanczos
  bidiagonalisation of `A` combined with a Givens-rotation update of
  the upper bidiagonal least-squares factor.  Convergence test uses
  `|phi_bar * alpha_{k+1}|`, the Paige-Saunders estimate of
  `||A^T r||`, scaled against the initial gradient `||A^T b||`.
  Dispatches by input grammar: free-symbol inputs go to Direct; exact
  (Integer / Rational) and Complex inputs go to CGLS (LSQR is
  mathematically equivalent to CGLS but produces unsimplified
  square-root expressions over exact rationals); pure-real inputs with
  at least one Real entry run the canonical double-precision algorithm.
  Detects rank deficiency by monitoring `alpha_new` and `beta_new`
  against `LSTSQ_LSQR_BREAKDOWN_EPS * max(|A_ij|)`, avoiding the
  catastrophic blowup of dividing by a near-zero `alpha`; near-zero
  `beta_new` triggers a final no-rotation Givens update that
  incorporates the last Krylov direction (so full-rank, well-conditioned
  systems do not miss the last iteration).

`"Automatic"` and `"Direct"` remain `PseudoInverse[m, Tolerance -> t]
. b`, the fundamental LeastSquares specification.

Tests in `tests/test_matlstsq.c` expanded with twenty new cases:
five for IterativeRefinement (Real 3x3 vs Direct, exact zero
Tolerance, ultra-tight Tolerance still terminates, singular 3x3,
matrix RHS), eight for Krylov (4x3 overdetermined, singular 3x3,

## matlstsq.c — `LeastSquares` (2026-05-18)

New module `src/matlstsq.[ch]` implementing
`LeastSquares[m, b]`, `LeastSquares[m, b, Method -> "<name>"]`, and
`LeastSquares[m, b, Tolerance -> t]`.  `LeastSquares` returns an `x`
minimising `Norm[m . x - b]`; for full-column-rank `m` this is the
unique least-squares minimiser, for rank-deficient `m` it is the
minimum-norm minimiser (the Moore-Penrose pseudoinverse solution
`PseudoInverse[m] . b`).

- New symbol: `SYM_LeastSquares`, cached in `sym_names.[ch]` and
  interned in `sym_names_init()` alongside the other matrix-routine
  symbols.
- Attributes: `Protected`.
- Docstring lives in `src/info.c` next to `LinearSolve` /
  `PseudoInverse` and is exposed via `Information[LeastSquares]` and
  the `?LeastSquares` REPL shortcut.
- Registered from `core_init()` immediately after `matinv_init()` so
  the underlying `PseudoInverse` builtin is available.

Behaviour:

- The matrix `m` may be square or rectangular and of any rank.  The
  right-hand side `b` may be a vector (rank-1) or a matrix (rank-2,
  one column per RHS).  For matrix `b` the result has one column per
  RHS, identical to a column-wise vector solve.
- Works on every input family supported by `PseudoInverse`:
  integer / rational, symbolic, machine-precision `Real` / MPFR, and
  exact / inexact `Complex`.
- Method and Tolerance options may appear in any order; each at most
  once.  Duplicates or unknown options leave the call unevaluated, so
  user-supplied DownValues / pattern rules can intercept it.
- Accepted Method names:
  - `Automatic` (symbol) / `"Automatic"` / `"Direct"` — Moore-Penrose
    solve `PseudoInverse[m] . b` (the workhorse).
  - `"IterativeRefinement"` — Direct then one refinement pass:
    `x <- PseudoInverse[m] . b`, `dx <- PseudoInverse[m] . (b - m . x)`,
    return `x + dx`.  For exact inputs the correction is exactly zero
    and the answer equals Direct; for inexact inputs the second pass
    reduces residual round-off.
  - `"LSQR"` — Paige-Saunders LSQR.  Method name is accepted by the
    parser; current dispatch goes through `"Direct"`.  A dedicated
    iterative implementation is a future extension.
  - `"Krylov"` — iterative Krylov / CGNR method.  Currently dispatched
    to `"Direct"` (likewise reserved).
- `Tolerance -> Automatic` (default), or a non-negative integer / real
  / `Rational`.  Forwarded verbatim to the underlying `PseudoInverse`
  call so any future singular-value-truncation pass in `PseudoInverse`
  is picked up automatically.

Diagnostics: `LeastSquares::matrix` (non-rank-2 or empty `m`),
`LeastSquares::lvec` (`b` not rank-1 or rank-2),
`LeastSquares::lvec1` (`b`'s leading dimension does not match `m`'s
row count) — each leaves the call unevaluated.

Tests: `tests/test_matlstsq.c` covers all spec examples (over-
determined integer 3x2 / 4x3, singular 3x3, symbolic 2x2,
matrix-RHS, `IdentityMatrix . b == b`, identity vs `LinearSolve` for
consistent systems, identity vs `PseudoInverse . b`, approximate
3x3, 1x1 / row-vector / column-vector degenerates), every Method
name (including the `Automatic` symbol and string variants), every
Tolerance type (Automatic, Integer, Real, Rational, negative),
mixed-option orderings, duplicate detection, and every shape-error
path.  The stress loop runs 5 representative shapes a total of 200
iterations through the PseudoInverse + Dot pipeline; valgrind shows
no Mathilda-originated leaks (only the unavoidable macOS dyld /

## core.c — `Clip` (2026-05-18)

`Clip[x]`, `Clip[x, {min, max}]`, and
`Clip[x, {min, max}, {vmin, vmax}]` are now built-ins in `src/core.c`
(`builtin_clip` plus the file-local helpers `clip_to_double_value`,
`clip_has_imaginary_part`, `clip_classify_infinity`).  The symbol is
cached as `SYM_Clip` in `sym_names.[ch]` and is registered with
`NumericFunction | Protected` (Listable is intentionally **not** set:
see below).

Behaviour:

- Default interval is `[-1, 1]`.  `{min, max}` and `{vmin, vmax}` are
  ordinary `List[...]` arguments; the bounds are coerced to a `double`
  for the comparison via `clip_to_double_value`, which accepts
  `Integer`, BigInt, `Rational[n, d]`, `Real`, MPFR (when
  `USE_MPFR=1`), and any symbolic expression that `numericalize`
  reduces to one of those (e.g. `Pi`, `E`, `Sqrt[2]`).
- The original symbolic `x` is returned when it lies inside the
  interval -- numericalization is used only for the decision, never
  for the result, so `Clip[Pi, {-9, 7}, {11, 28}]` is `Pi`, not
  `3.14159...`.
- `Infinity` and `-Infinity` are classified before any numericalize
  call (`Infinity` is not a machine real and `numericalize` cannot
  reduce it) and yield `vmax` / `vmin` respectively.
- A `Complex[re, im]` with a structurally non-zero imaginary part
  trips a rate-limited `Clip::ncompl` warning (via the shared
  `matsol_warn_once` idiom from `src/matsol.c`) and the call stays

- Any other case where `clip_to_double_value` cannot reduce `x`,
  `min`, or `max` to a number also leaves the call unevaluated so
  user-supplied DownValues / pattern rules can intercept it.

`ATTR_LISTABLE` threads over *every* list argument, but Clip must
keep `{min, max}` and `{vmin, vmax}` as configuration lists.  Setting
`Listable` would incorrectly thread `Clip[1, {-2, 2}]` to
`{Clip[1, -2], Clip[1, 2]}`.  The fix is to drop `Listable` and
implement first-argument threading manually in `builtin_clip`: if
`args[0]` is a `List[x1, x2, ...]`, the call rewrites to
`List[Clip[x1, ...], Clip[x2, ...], ...]` and each inner call is
re-evaluated through `eval_and_free`, leaving the bounds /
replacement lists untouched.

## matinv.c — Method options for `Inverse` (2026-05-18)

`Inverse[m, Method -> "<name>"]` now mirrors the dispatcher that
`RowReduce` and `LinearSolve` gained earlier in the month, so every
matrix-solver builtin offers the same `Method` grammar.

- Three explicit algorithms are exposed, all in `src/matinv.c`:
  - `"DivisionFreeRowReduction"` — the historical Bareiss-like
    fraction-free Gauss-Jordan on the augmented matrix `[A | I]`.  The
    body is the same one that previously lived inside `builtin_inverse`,
    now lifted into `inverse_divfree(arg, n)`.
  - `"OneStepRowReduction"` — classical Gauss-Jordan on `[A | I]` with
    one division per pivot per row entry.  Each entry is canonicalised
    through `matsol_canon_entry` (`Together`) so symbolic cancellations
    are still detected and a singular symbolic matrix produces a
    definite `Inverse::sing` rather than a silently wrong answer.
  - `"CofactorExpansion"` — adjugate / determinant formula
    `A^-1[i, j] = (-1)^(i + j) Det[M[j, i]] / Det[A]`, with `Det`
    computed via the same `laplace_det` primitive used by `Det` itself
    and by the `LinearSolve` / `RowReduce` cofactor workers.  Time
    complexity is `O(n! * n^2)`; intended for small `n` or for
    obtaining a closed-form symbolic inverse of small matrices.  A 1x1
    fast path bypasses `laplace_det`'s `n >= 1` precondition.
- `Method -> Automatic` (the symbol) and `Method -> "Automatic"` both
  alias to `"DivisionFreeRowReduction"`, matching Mathematica.
- An unknown method name emits a rate-limited `Inverse::method`
  warning and leaves the call unevaluated.  Non-square / empty input
  still emits `Inverse::matsq`.  Every method emits `Inverse::sing` on
  a singular matrix and returns unevaluated.
- The previously file-private `parse_method_option`, `warn_once`,
  `canon_entry`, and `div_entry` helpers in `src/matsol.c` were
  promoted to `matsol_parse_method_option`, `matsol_warn_once`,
  `matsol_canon_entry`, and `matsol_div_entry`, and exposed in
  `src/matsol.h` alongside the `MatsolMethod` enum so `matinv.c` can
  reuse the same Method grammar and the same `Together` /
  exact-division primitives without duplication.
- `PseudoInverse` continues to call `Inverse` internally (through the
  evaluator) with no `Method` argument, so its behaviour is unchanged
  and it still goes through `inverse_divfree`.

Tests live in `tests/test_matinv_methods.c` (a new test executable
wired into `tests/CMakeLists.txt`).  Coverage mirrors
`test_matsol_methods.c`: each method is exercised against tiny
invertible integer matrices, bignum-valued matrices, fully symbolic
invertible matrices (back-substitution check, since methods differ in
canonical form), exact complex matrices, machine-precision Real
matrices, MPFR matrices when compiled in, and the singular /
non-square / invalid-method error paths.

## matinv.c — `PseudoInverse` builtin, and `Inverse` moved out of `linalg.c` (2026-05-18)

A new module `src/matinv.c` (with `src/matinv.h`) consolidates the two
matrix-inversion builtins.

- `Inverse[m]` is moved from `src/linalg.c` to `src/matinv.c` verbatim.
  The Bareiss-like fraction-free Gauss-Jordan algorithm on `[A | I]` is
  unchanged.  Registration happens in the new `matinv_init`, called
  from `core_init` immediately after `matsol_init` (so `Inverse` keeps
  resolving in the same evaluator load order).  `linalg_init` no
  longer registers `Inverse`.
- `PseudoInverse[m]` and `PseudoInverse[m, Tolerance -> t]` are new.
  The implementation computes a full-rank decomposition `m = B . C` by
  row-reducing `m` to identify rank `r` and the pivot columns, builds
  `B` (the `m x r` matrix of pivot columns) and `C` (the leading `r`
  rows of the RREF), and returns
  `PseudoInverse[m] = ConjugateTranspose[C] . Inverse[C . ConjugateTranspose[C]] . Inverse[ConjugateTranspose[B] . B] . ConjugateTranspose[B]`.
  The two `r x r` mid-products are invertible by construction.  For the
  `m x n` zero matrix the result is the `n x m` zero matrix.  For
  non-singular square `m` the result coincides with `Inverse[m]`.
- Inexact (`Real` / MPFR) inputs are routed through the
  `common_scan_inexact` → `common_rationalize_input` → exact pipeline →
  `common_numericalize_result` flow, so `PseudoInverse` honours
  inexact-in / inexact-out semantics at the input precision while
  still computing rank in exact rational arithmetic.
- `PseudoInverse` is `Protected`.  Information string registered in
  `info.c`.  Symbol interned as `SYM_PseudoInverse` (and the option
  symbol `SYM_Tolerance`) in `sym_names.c` / `sym_names.h`.
- The `Tolerance` option recognises `Automatic` (default), non-negative
  `Integer`, non-negative `Real`, and non-negative `Rational[n, d]`.
  Negative values, unknown option names, and non-numeric values leave
  the call unevaluated.
- New test binary `tests/test_matinv.c` (added to
  `tests/CMakeLists.txt` as `matinv_tests`) covers: invertible square
  (collapses to `Inverse`), the rank-1 singular 3x3 spec example,
  over- and under-determined rectangular inputs with the Moore-Penrose
  identity check `m . p . m == m && p . m . p == p`, zero matrices,
  `1x1` / `1xN` / `Nx1` shapes, exact rationals, exact and inexact
  complex (including the Wolfram reference 3x3 complex example),
  machine-precision real (including a singular row), the `Tolerance`
  option in all three forms (`Automatic`, numeric, invalid), shape
  errors, and a 120-call stress loop for memory hygiene.

Fixed a pre-existing leak of the temporary argument array in
`builtin_conjugate` (`src/complex.c`) that was newly exposed by
`PseudoInverse`'s use of `Conjugate[Transpose[...]]`.

## common.c — shared rationalize / numericalize preprocessor; Solve & Integrate routed through it (2026-05-18)

Both `Solve` and `Integrate` need the same "if the input has any inexact
(Real / MPFR) leaves, rationalise them before the exact-arithmetic core runs,
then numericalise the answer on the way out" preprocessing pass.  That dance
now lives in a single new module — `src/common.c` / `src/common.h` — exposing
three building blocks:

- `common_scan_inexact(e)` — single-pass walk returning a
  `CommonInexactInfo {has_inexact, min_bits}` struct.  `min_bits` is the
  *minimum* bit-precision seen across every inexact leaf: 53 for any Real
  (IEEE 754 double), `mpfr_get_prec(...)` for each MPFR.  The minimum
  policy means a mixed input propagates the *lower* precision through to
  the answer — the answer is no more precise than the least-precise
  input, which is the standard inexact-arithmetic contract.
- `common_rationalize_input(e, bits)` — force-rationalise every inexact
  leaf at the given precision.  Always tries the relaxed convergent-bound
  algorithm first (so `1.5` recovers as `3/2`, `1/9.` as `1/9`); when no
  convergent meets the bound the fallback is `RATIONALIZE_ZERO` (½ ulp at
  the input's double precision) for `bits ≥ 52` and `RATIONALIZE_TOLERANCE`
  with `dx = 2^(-bits)` for `bits < 52` (only seen when the caller has an
  MPFR leaf with explicit low precision).  Symbolic constants (`Pi`, `E`,
  ...) and Hold-forms pass through unchanged.
- `common_numericalize_result(e, bits)` — round-trip an exact symbolic
  answer back to floating-point at the requested precision.  `bits ≤ 53`
  → IEEE 754 double; `bits > 53` → MPFR at exactly `bits` bits (when
  compiled with `USE_MPFR`).  The traversal recurses into `List` and
  `Rule` so `{{x -> 1/2}}` cleanly becomes `{{x -> 0.5}}` while leaving
  the unknown LHS symbol alone.

The implementation delegates to `internal_force_rationalize_bits`
(rationalize.c, newly exposed) and `numericalize` (numeric.c) — the new
module is purely a stable API surface so future callers (`NSolve`,
`DSolve`, ...) plug in the same way.

### Precision propagation in action

```
Precision[(Solve[N[1, 30] x == 7, x][[1]][[1]])[[2]]]              -> 30.103
Precision[(Solve[(0.5 + N[Pi, 30]) x == 1, x][[1]][[1]])[[2]]]     -> MachinePrecision

Solve[N[Pi, 50] x == 1, x]
  -> {{x -> 0.31830988618379067945899344644312539938328968316824}}

Integrate[N[Pi, 50] x, x]
  -> 1.5707963267948965801416385921020337420225281697321 x^2
```

The 30-digit MPFR input flows through to a 30-digit result; the moment a
machine-precision Real joins it, the answer drops to machine precision.

Caveat: when the minimum precision exceeds 53 bits (only achievable with
explicit-MPFR inputs), the rationalisation step is still performed via the
double-based `internal_rationalize_double` core, so the intermediate
rational is bit-exact at double precision rather than at the requested
MPFR precision.  The numericalisation step does run at the full MPFR
precision, so the *output* carries the requested precision — it just isn't
fully *informed* by sub-double-precision bits of the input.  A
precision-true MPFR rationalisation path would close this gap and is left
as future work.

### Solve — approximate-number input support

`Solve[1.5 x + 3 == 0, x]` previously fell through the polynomial specialist
(which expects exact coefficients) and returned unevaluated.  After this
change `Solve` detects inexact leaves up front, rationalises the equation
(`{3/2, 3}`), dispatches to the appropriate specialist exactly as before,
and numericalises the bindings before return:

```
Solve[1.5 x + 3 == 0, x]                    -> {{x -> -2.0}}
Solve[x^2 - 2.25 == 0, x]                   -> {{x -> -1.5}, {x -> 1.5}}
Solve[0.5 x^2 - x - 1.5 == 0, x]            -> {{x -> -1.0}, {x -> 3.0}}
Solve[{1.5 x + y == 4.5, x - y == 0.5},
      {x, y}]                               -> {{x -> 2.0, y -> 1.5}}
Solve[N[Pi] x == 1, x]                      -> {{x -> 0.31831}}
```

The exact-integer path is untouched — `Solve[x^2 - 4 == 0, x]` still returns
`{{x -> -2}, {x -> 2}}` (no float anywhere in the input means no
post-numericalisation).

The `vars` argument is never rationalised — it stays a symbol or a list of
symbols, which is what the specialists require.

### Integrate — bit-exact rationalisation fallback removes `Integrate::inexact`

`Integrate` previously called `Rationalize` in the relaxed default mode and,
on a transcendental float it could not convert to a small-denominator
rational, emitted `Integrate::inexact` and returned unevaluated.  Routing
through `common_rationalize_input` (bit-exact ½-ulp fallback) means the
integrator always sees an exact integrand and the warning is gone:

```
Integrate[1.5 x, x]      -> 0.75 x^2
Integrate[N[Pi] x, x]    -> 1.5708 x^2     (* used to emit Integrate::inexact *)
Integrate[x^2, x]        -> 1/3 x^3        (* exact path: untouched *)
```

Same `inexact-in / inexact-out` contract as Solve and the existing
exact-symbolic builtins (`Apart`, `Cancel`, `Together`, `Factor`, ...).

## linalg: add LinearSolve (2026-05-18)

`LinearSolve[m, b]` finds `x` satisfying `m . x == b`.  The matrix `m`
may be square or rectangular; `b` may be a vector (LinearSolve returns
a vector) or a matrix with one solution column per RHS column.
Implemented in `src/linalg.c` as fraction-free Gauss-Jordan elimination
on the augmented matrix `[m | b]` — the same Bareiss-like routine that
backs `RowReduce` and `Inverse`, so exact integer / rational / symbolic
input flows through with no spurious denominator blow-up.

### Behaviour

- Square invertible system → unique solution.
- Rectangular over- or under-determined system → particular solution
  with every free (non-pivot) variable taken to be 0, matching
  Mathematica's convention.
- Inconsistent system → emits `LinearSolve::nosol` to stderr and
  returns the call unevaluated.
- Shape error (`b`'s leading dimension does not match `m`'s row count,
  or either argument is not a matrix / vector) → emits
  `LinearSolve::lvec1` / `::lvec` / `::matrix` and returns unevaluated.

### Symbols / docstring

- `SYM_LinearSolve` added to `sym_names.{h,c}`.
- `Attributes[LinearSolve] = {Protected}` (assigned in `linalg_init`).
- Docstring registered in `info.c`, accessible via `?LinearSolve`.

### Verification

New test binary `linearsolve_tests` exercises 31 cases:
identity / diagonal / 1×1 systems, exact integer & rational 2×2 and
3×3 round-trips, matrix-RHS shapes, 3-column rectangular and 4×2
rectangular shapes, symbolic 2×2 / 3×3 verified by
`m . LinearSolve[m, b] == b`, Gaussian-integer system from the spec,
floating-point round-trip, singular / inconsistent / shape-error
fail paths (verifying unevaluated return), and equivalence with
`Inverse[m]` via `LinearSolve[m, IdentityMatrix[n]]`.

Valgrind (`--leak-check=full`) reports zero leaks attributable to our
code; the surviving "definitely lost" blocks all trace into
`libobjc.A.dylib` / `dyld4` initialisation, the standard macOS
baseline.


## matsol.c — Method options for RowReduce and LinearSolve (2026-05-18)

`RowReduce[m]` and `LinearSolve[m, b]` now accept an optional
`Method -> "<name>"` argument that selects the elimination algorithm.
The bodies of both builtins were lifted out of `src/linalg.c` into a
new `src/matsol.c` (the new file is wired in via `core_init` →
`matsol_init`).  `src/linalg.h` now exports the four primitives the
two routines share with the rest of `linalg.c`:
`get_tensor_dims`, `flatten_tensor`, `laplace_det` and
`exact_div_wrapper`.

### Accepted methods

| Name (string)                | Description |
|------------------------------|-------------|
| `Automatic` (default)        | Alias for `DivisionFreeRowReduction`. |
| `DivisionFreeRowReduction`   | Bareiss-like fraction-free Gauss-Jordan on `m` (or `[m | b]`). Preserves the prior default exactly. |
| `OneStepRowReduction`        | Classical Gauss-Jordan with one division per pivot per element. Each per-cell update goes through `Together` so symbolic rationals reduce. |
| `CofactorExpansion`          | For `LinearSolve`: Cramer's rule via Laplace expansion of `Det[m]` (square non-singular m only). For `RowReduce`: identity matrix when `Det[m] != 0`, else falls back to `DivisionFreeRowReduction`. |

`Method -> Automatic` (the symbol, not the string) is also accepted.

### Error messages

| Message                       | Cause |
|-------------------------------|-------|
| `RowReduce::cofnsq`           | `CofactorExpansion` requested but `m` is singular or non-square; falls back. |
| `RowReduce::method`           | Unrecognised method name. |
| `LinearSolve::cofnsq`         | `CofactorExpansion` requested for non-square `m`; call stays unevaluated. |
| `LinearSolve::cofsng`         | `CofactorExpansion` requested but `Det[m] == 0`; call stays unevaluated. |
| `LinearSolve::method`         | Unrecognised method name. |

All four messages use the `last_warned_hash` rate-limit idiom so
nested test loops don't spam stderr.

### Power[Rational, integer] fix (pre-requisite)

Adding `OneStepRowReduction` over bignum matrices surfaced a
pre-existing bug in `Power[Rational[a, b], n]` where the `is_rational`
fast path bailed out as soon as `a` or `b` overflowed `int64`. That
caused `Power[Rational[10^50-1, 10^25], -1]` (a typical
post-elimination shape on bignum matrices) to stay unsimplified,
which in turn broke `Together` and `is_zero_poly`.

The fix in `src/power.c` adds a bigint-aware Rational-base path that
walks the `Rational[X, Y] ^ k` triple through `mpz_pow_ui` and emits a
canonical reduced `Rational` (or demoted integer) result. Soundness is
preserved: only integer-like components are accepted, the sign is
normalised onto the numerator, and the GCD is cancelled after raising
both halves to the power.

`tests/test_matsol_methods.c` covers every `(builtin, method)` pair
across: tiny exact integer, bignum, machine-precision `Real`,
arbitrary-precision MPFR (gated on `USE_MPFR`), symbolic invertible,
symbolic singular, rectangular, and complex matrices.  Inconsistent
systems are verified to print back unevaluated for all three
methods.  Existing `linalg_tests` and `linearsolve_tests` continue to

## complex, comparisons, info: Abs/Re/Im of symbolic Complex sums, NumericQ comparisons, fuller iter docstrings

### Symbolic decomposition in Abs, Re, Im, ReIm, Conjugate

`Abs`, `Re`, `Im`, `ReIm`, and `Conjugate` now decompose expressions
that are syntactically a `Plus` or `Times` carrying nested `Complex[a, b]`
literals.  Previously these built-ins only matched the canonical
`Complex[re, im]` head, so expressions like `1 + I Sqrt[3]` (which
Mathilda stores as `Plus[1, Times[Complex[0, 1], Power[3, 1/2]]]` —
the `Sqrt[3]` coefficient blocks the usual Plus-time folding into a
single `Complex[]`) stayed under an unevaluated `Abs[...]`.

A new file-static helper `complex_decompose` in `src/complex.c` walks
the expression structurally:

- `Complex[a, b]` literals contribute `(a, b)` directly.
- `Plus[t1, t2, ...]` recurses on each term and sums the real and
  imaginary parts (terms with no detectable imaginary component fall
  into the real bucket).
- `Times[f1, f2, ...]` recurses on each factor, threading a running
  complex product via `(a + b i)(c + d i) = (ac - bd) + (ad + bc) i`.

Subexpressions that aren't a `Complex` literal (symbols, `Sqrt[n]`,
unknown functions, ...) are treated as real-valued, matching
Mathematica's default `Re`/`Im` behaviour.  `Conjugate` builds
`re + (-I)*im` from the decomposition, so e.g. `Conjugate[1 + I Sqrt[3]]`
now reduces to `1 - I Sqrt[3]`.

### Less/Greater/Equal/Unequal handle NumericQ-style expressions

`compare_numeric` in `src/comparisons.c` previously short-circuited
to `can_compare = false` for anything that wasn't an `Integer`,
`Real`, or `Rational` leaf.  It now falls back to `numericalize` at
machine precision, so symbolic-but-exact numeric expressions
(`Sqrt[3]`, `Pi`, `Pi + Sqrt[2]/2`, ...) reduce to a machine double
and participate in comparisons.  This means:

- `Sqrt[3] < 20` -> `True`
- `Pi < 4` -> `True`, `Pi < 3` -> `False`
- `Sqrt[2] <= Sqrt[2]` -> `True`
- `Sqrt[3] == 2` -> `False`, `Sqrt[3] != 2` -> `True`
- `x < 5`, `Sqrt[x] < 5` -> unchanged (numericalize leaves these
  symbolic, so `compare_numeric` still reports `can_compare = false`).

`Equal` was extended to surface a definite `False` whenever
`compare_numeric` returns a non-zero result (mirroring the symmetric
path already present in `Unequal`), so `Sqrt[3] == 2` no longer leaks
through as unevaluated.  The result of `compare_numeric` is marked
inexact in the new path so the existing `2^-46` floating-point
tolerance branch is used.

### End-to-end: Select[Abs[#] < c &]

The original report was `Select[Flatten[Table[a + b*Exp[2 Pi I/3], {a,-2,2}, {b,-2,2}], 1], Abs[#] < 20 &]` returning only the
five integer entries — every other entry stayed `Abs[...] < 20`
because `Abs` couldn't reduce the Plus-form complex and `Less`
couldn't compare the resulting `Sqrt[..]`.  After this change the
predicate returns `True` for all 25 lattice points, matching
Mathematica.

### Fuller docstrings for Do and For

`docs/spec/builtins/control-flow.md`-tracked builtins `Do` and `For`
had terse docstrings that omitted the multi-iterator and empty-body
forms as well as the `Break[]` / `Continue[]` / `Return[v]` flow
controls.  The strings registered in `src/info.c` now cover all the
iterator-spec variants (`{n}`, `{i, imax}`, `{i, imin, imax, di}`,
`{i, {i1, i2, ...}}`, multi-iterator threading), the `HoldAll`
attribute, and the loop-control built-ins.  `While`'s docstring was
already complete and is unchanged.

## Eigenvalues / Eigenvectors: numerical "Direct" kernel for real symmetric machine-precision (Phase 2 step 1)

The first real numerical eigensolver kernel lands.  For an
approximate-numeric real symmetric matrix, `Method -> Automatic` and
`Method -> "Direct"` now route through a hand-rolled hot-path kernel
in `src/mateigen.c` instead of the symbolic
Faddeev-Leverrier-plus-`Solve` pipeline.

Algorithm:

1. Convert the Expr List-of-Lists to a flat row-major `double` workspace
   (`MatD`) at the dispatcher boundary -- the inner loops never touch
   `Expr` nodes.
2. Hermitian / real-symmetric structure is auto-detected; non-symmetric
   numeric matrices and complex / MPFR inputs return `NULL` from this
   step's dispatcher so the caller falls through to the symbolic path.
3. **Householder tridiagonalisation** of the symmetric matrix via the
   rank-2 update `A <- A - u q^T - q u^T` with `q = p - 2 K u`,
   `p = 2 A u`, `K = u^T A u` (Golub & Van Loan Algorithm 8.3.1).
   Scratch buffers are pre-allocated so no `malloc` happens inside the
   reduction loop.  When eigenvectors are requested, an orthogonal
   accumulator `Q` collects the Householder reflectors.
4. **Implicit-shift symmetric QR** with Wilkinson shift on the
   tridiagonal `(diag, sub)`.  Each sweep applies Givens rotations
   along the active block via the closed-form update for the symmetric
   2x2 similarity `Q^T A Q`, chasing the bulge through the band.
   Deflation triggers when an off-diagonal entry falls below
   `1e-14 * (|d_k| + |d_{k+1}|)`.  Max-sweeps guard at `30 * n` to
   prevent theoretical hangs.
5. Sort by descending `|lambda|` (stable on ties) and emit a List of
   `EXPR_REAL` eigenvalues or a List-of-Lists of orthonormal
   eigenvectors.  `k`-spec (`+k` / `-k` / `UpTo[k]`) is applied after
   sort.

The kernel reuses Phase 1's `MateigenWant` flag so a single dispatcher
serves both `Eigenvalues` and `Eigenvectors`; the eigenvector
accumulation work is skipped when the caller only asked for values.
Every numerical site carries a `LAPACK-HOOK` comment naming the
LAPACK routine that would replace it once `USE_LAPACK` is wired
(`dsytrd` for tridiagonalisation, `dsteqr` / `dstedc` for the
symmetric tridiagonal QR step).

A new `tests/test_mateigen_direct.c` (19 assertions) and a shared
`tests/eigen_corpus.{h,c}` test harness verify the kernel against a
broad real-symmetric corpus:

- Diagonal, identity, zero, and 2x2 hand-computed cases
  (`[[4,1],[1,4]]`, `[[1,2],[2,-1]]`).
- Toeplitz tridiagonal matrices at sizes 3, 5, 6 with the closed-form
  spectrum `lambda_k = 2 (1 - cos(k pi / (n+1)))`.
- The Hilbert-4 matrix (ill-conditioned but well-defined symmetric
  spectrum).
- Deterministically-seeded random symmetric matrices at sizes 5 and 10

## Eigenvalues: numerical "Direct" kernel for real non-symmetric machine-precision matrices (Phase 2 step 2a)

The `"Direct"` kernel now covers real **non-symmetric** matrices at
machine precision too -- the path between dispatcher and the existing
real-symmetric kernel from step 2.1 picks up everything that fails the
Hermitian-detection test.  For an approximate-numeric real general
matrix, `Method -> Automatic` and `Method -> "Direct"` now route
through a hand-rolled Householder reduction to upper Hessenberg form
followed by implicit double-shift Francis QR sweeps, instead of the
symbolic Faddeev-Leverrier + `Solve` pipeline.

Algorithm:

1. **Householder reduction** to upper Hessenberg.  Two-sided update
   `A <- H A H` (no symmetry shortcut; left- and right-multiplication
   are done explicitly).  O(10 n^3 / 3) flops.  No scratch allocations
   inside the reduction loop.
2. **Francis double-shift QR.**  Each sweep:
   - Looks for the leftmost negligible sub-diagonal entry to find the
     active sub-block `[q..p-1]`.
   - Deflates 1x1 trailing blocks as real eigenvalues and 2x2
     trailing blocks as either a real pair (positive discriminant) or
     a complex conjugate pair.
   - Otherwise applies one Francis step on the block: derives the
     implicit double-shift from the trailing 2x2 (`s = trace`,
     `t = det`), seeds the bulge from the first three entries of the
     polynomial `M = (A - mu_1 I)(A - mu_2 I)` (real arithmetic
     throughout since the shift pair is a conjugate pair when
     complex), then chases the bulge through the active block via
     size-3 Householder reflectors with a size-2 closer at the end.
   - Deflation tolerance:
     `|H[i, i-1]| <= 1e-14 * (|H[i-1,i-1]| + |H[i,i]|)`.
   - Max iterations: `30 * n` total sweeps.
3. Eigenvalues are written into parallel `re[]` / `im[]` arrays as
   they deflate, sorted by descending `|lambda|` (stable on ties),
   and emitted as `EXPR_REAL` (for real eigenvalues) or
   `Complex[re, im]` (for complex pairs).  `k`-spec applies after
   sort.

LAPACK-HOOK comments mark every replaceable site: `dgehrd` for the
Hessenberg reduction, `dhseqr` for the Francis QR loop, and
`dtrevc3` + `dorghr` for the eigenvector back-transform that arrives
in step 2b.

This commit covers **eigenvalues only**.  Real non-symmetric
`Eigenvectors` calls continue to fall through to the symbolic
null-space pipeline until step 2b lands the Hessenberg
back-substitution and the complex-eigenvalue handling.

A new 12-test suite in `tests/test_mateigen_direct.c` covers:

- 2x2 real eigenvalues (`[[1,2],[3,4]]`).
- 2x2 pure rotation `[[0,1],[-1,0]]` -> `+/- i`.
- 2x2 general complex pair `[[1,-1],[1,1]]` -> `1 +/- i`.
- 3x3 block-diagonal mixing a complex pair with a real eigenvalue.
- Deterministically-seeded random non-symmetric matrices at sizes
  3, 5, 10 (Lehmer LCG) verified via the trace + `trace(A^2) =
  sum(lambda^2)` similarity invariants (the latter holds for any
  matrix, unlike the normal-matrix-only Frobenius identity).
- Companion matrix of `(x-1)(x-2)(x-3)` -> `{3, 2, 1}`.
- Jordan block 2x2 -> `{2, 2}` (defective).
- Sort order verification for matrices with both real and complex
  eigenvalues.
- `Method -> "Direct"` produces the same numerical result as
  `Automatic` for a non-symmetric input.
- `Eigenvectors` on a non-symmetric numeric input still falls back to
  the symbolic path (intentional for step 2a; step 2b will replace
  this behaviour).

continue to pass unchanged.

## Eigenvectors: numerical "Direct" kernel for real non-symmetric machine-precision matrices (Phase 2 step 2b)

Real non-symmetric eigenvectors now route through the same Hessenberg
+ Francis QR pipeline that step 2a established for eigenvalues,
followed by Schur-form back-substitution and a Q back-transform.  This
brings `Eigenvectors` to feature parity with `Eigenvalues` for all
real matrix shapes (symmetric, general, real-spectrum, complex-
spectrum) at machine precision.

Pipeline:

1. **Q accumulation**: Hessenberg reduction and every Francis QR
   reflector (including the new 2x2-real-eigenvalue Givens splitter)
   now right-multiply an n x n orthogonal accumulator `Q`, so the
   final Schur form satisfies `H = Q^T A Q`.  When eigenvectors are
   not requested, `Q` is passed as `NULL` and the accumulation work is
   skipped -- step 2a stays as fast as before.
2. **2x2 real-eigenvalue split**: a new `direct_split_2x2_real`
   helper handles the case where the trailing 2x2 block of the active
   Schur sub-block has real (not complex) eigenvalues.  Instead of
   reading the eigenvalues from the diagonal after Francis sweeps
   (which loses precision for ill-conditioned cases), we compute them
   analytically from the 2x2 block, then apply a Givens similarity to
   triangularise the block and right-multiply `Q`.  The diagonals are
   overwritten with the analytic eigenvalues so subsequent back-
   substitution sees self-consistent (`eval_re[k] == H[k, k]`) data.
3. **Back-substitution on Schur form**:
   - `schur_eigvec_real(H, n, k, lambda, v)` solves `(H - lambda I)
     v = 0` from row k-1 down to 0, with 2x2 sub-blocks handled by
     a 2x2 real solve.
   - `schur_eigvec_complex(H, n, k, a, b, v_re, v_im)` solves the
     same equation for `lambda = a + b i` (b > 0).  Each step is a
     2x2 complex solve (size-2 within a 1x1 or 2x2 Schur block,
     coded via complex 2x2 inverse), expanded into paired real
     `v_re`, `v_im` updates.  Initial values come from the 2x2 block
     at Schur positions (k, k+1) using `x_1 = 1`, `x_0 = (lambda -
     delta)/gamma`.
4. **Back-transform**: `w = Q . v_schur` for real eigenvalues, or
   `(w_re, w_im) = (Q v_re_schur, Q v_im_schur)` for complex.  All
   eigenvectors are normalised to unit 2-norm (for real) or
   `sum(w_re^2 + w_im^2) = 1` (for complex).
5. **Output**: rows of the returned `List[List[...]]` are sorted by
   descending `|lambda|` to match the eigenvalue order; complex
   entries become `Complex[re, im]`, real entries `Real`.

A new 8-test suite in `tests/test_mateigen_direct.c` exercises the
non-symmetric eigenvector path:

- 2x2 real `[[1,2],[3,4]]` residual.
- 2x2 pure rotation `[[0,1],[-1,0]]` (complex `+/- i`) residual.
- 3x3 block-diagonal `1 +/- i` plus real 2.
- 3x3 upper triangular `{6, 4, 1}`.
- Generic 3x3 `[[1,2,3],[4,5,6],[7,8,0]]` (well-conditioned).
- Deterministic random 5x5 and 10x10 non-symmetric (mixed real /
  complex spectrum).
- 3x3 mixing `{5, 2+i, 2-i}` (real + complex pair).

Each test verifies `||A v - lambda v||_inf` against `256 n eps
||A||_inf` for real eigenpairs and the equivalent complex-arithmetic
norm `||A v - lambda v||` for complex pairs.  The companion matrix of
`(x-1)(x-2)(x-3)` (textbook ill-conditioned case for eigenvector
computation due to extreme Vandermonde eigenvector sensitivity) is
deliberately excluded; its eigenvalue-only test from step 2a passes
at 1e-10 tolerance and remains the right characterisation.

All 76 existing `tests/test_eigen.c` tests continue to pass unchanged
-- including the eigenvector tests for numeric symmetric and complex-
rotation matrices, which previously routed through the symbolic
fallback and now flow through the numerical kernel.
