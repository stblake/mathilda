# Changelog: week of 2026-07-20 (Mon) ‚Äì 2026-07-26 (Sun)


Feature additions and fixes recorded during this week.

## `StringPartition` ‚Äî partition a string into fixed-length substrings (2026-07-20)

Added `StringPartition` in a new translation unit `src/strings/stringpartition.c`,
registered in `strings_init()` (no regex dependency). It carves a string into
fixed-length substrings, covering the Wolfram surface:

- `StringPartition["123456789123456789", 9]` ‚Üí `{"123456789", "123456789"}` ‚Äî
  non-overlapping length-`n` blocks (offset `d = n`); a trailing run too short to
  fill a block is dropped, e.g. `StringPartition["ababababab", 3]` ‚Üí
  `{"aba", "bab", "aba"}`.
- `StringPartition["string", n, d]` starts a length-`n` block every `d`
  characters. `StringPartition["123456789", 2, 1]` ‚Üí
  `{"12","23",...,"89"}` (overlap, `d < n`); `d > n` skips characters in the
  middle. All blocks stay length `n`.
- `StringPartition["string", UpTo[n]]` allows a shorter final block so every
  character appears: `StringPartition["123456789", UpTo[2]]` ‚Üí
  `{"12","34","56","78","9"}`. Combinable with an explicit offset (`UpTo[n], d`).
- A list of strings threads (`StringPartition[{"abcd","ef"}, 2]` ‚Üí
  `{{"ab","cd"},{"ef"}}`). Non-string subjects, non-integer/non-positive `n`/`d`
  leave the call unevaluated; other arities emit `StringPartition::argt`
  (2 or 3 arguments expected).

Strings are treated as raw byte arrays (matching `StringTake`/`StringDrop`); no
UTF-8 codepoint decoding is performed. Attributes: `Protected`. Docstring in
`info.c`; symbol `SYM_StringPartition` interned in `sym_names.c`. Covered by the
new `tests/test_stringpartition.c` (unconditional, not `USE_REGEX`-gated).

## `StringExtract` ‚Äî split into blocks and select by position (2026-07-20)

Added `StringExtract` in a new translation unit `src/strings/stringextract.c`,
registered in `regex_init()` alongside its `StringSplit` dependency. It splits a
string into blocks and selects blocks by position, covering the Wolfram surface:

- `StringExtract["a bbb  cccc aa   d", 2]` ‚Üí `"bbb"` ‚Äî whitespace-delimited nth
  block; `-n` counts from the end (`... , -1]` ‚Üí `"d"`).
- Position specs: `n`, `-n`, `{n1, n2, ...}` (collection), `n1;;n2` (span), and
  `All`. `StringExtract["a bbb  cccc aa   d", 2;;4]` ‚Üí `{"bbb","cccc","aa"}`;
  `{1,4,5}` ‚Üí `{"a","aa","d"}` (equivalently `Part[StringSplit[...], {1,4,5}]`).
- `sep -> pos` delimits blocks with a string pattern: `"a--bbb--ccc--dddd" ,
  "--" -> 3]` ‚Üí `"ccc"`, and `sep -> All` equals `StringSplit["string", sep]`.
- Multi-level extraction `StringExtract["string", pos1, pos2, ...]` delimits with
  whitespace at the lowest level, `"\n"` at the next, `"\n\n"` above that, and so
  on (or explicit `sepi -> posi` per level). E.g. on a 3-paragraph text
  `StringExtract[text, 2, 2, 3]` ‚Üí the 3rd word of the 2nd line of the 2nd
  paragraph.
- Absent blocks yield `Missing["PartAbsent", pos]`:
  `StringExtract["a 1\nb 2\nc 3 x", All, 3]` ‚Üí
  `{Missing["PartAbsent",3], Missing["PartAbsent",3], "x"}`.
- Threads over a list of subject strings. A non-string, non-list subject leaves
  the call unevaluated; fewer than 2 arguments emits `StringExtract::argm`
  ("2 or more arguments are expected").

The implementation is a thin, DRY layer: each level builds `StringSplit[str,
sep]` and evaluates it to produce the blocks, so the entire string-pattern engine
(`wl_pattern_to_regex`), whitespace-run collapsing, and empty-end trimming are
reused verbatim ‚Äî making the `sep -> All == StringSplit` and `{pos...} ==
Part[StringSplit[...], {pos...}]` equivalences exact. Position resolution reuses
`expr_part` for spans; integer/out-of-range handling produces `Missing` directly.
Attributes: `Protected`. Docstring in `info.c`; symbols `SYM_StringExtract` /
`SYM_StringSplit` added to `sym_names.{c,h}`. Eight test groups in
`tests/test_stringfns.c` cover every worked example; valgrind-clean.

## `StringTrim` ‚Äî trim matching substrings from both ends (2026-07-20)

Added `StringTrim` in a new translation unit `src/strings/stringtrim.c`, covering
the Wolfram surface:

- `StringTrim["  aaa bbb  "]` ‚Üí `"aaa bbb"` ‚Äî the 1-argument form trims runs of
  whitespace (spaces, tabs, newlines) from both ends.
- `StringTrim["string", patt]` trims substrings matching `patt` from the start
  and the end, e.g. `StringTrim["++++x----", ("+"|"-")...]` ‚Üí `"x"` and
  `StringTrim["007bond007", DigitCharacter..]` ‚Üí `"bond"`. Each end is trimmed to
  a fixed point, so a literal single-character pattern removes a whole run
  (`StringTrim["xxabcxx", "x"]` ‚Üí `"abc"`).
- A pattern that anchors itself to the start only trims the front:
  `StringTrim["   aaa   ", RegularExpression["^ *"]]` ‚Üí `"aaa   "`.
- Threads over a list of subject strings, passing non-string elements through:
  `StringTrim[{"  a  ", 5, x}]` ‚Üí `{"a", 5, x}`.
- A non-string subject leaves the call unevaluated; `StringTrim[]` emits
  `StringTrim::argt` ("1 or 2 arguments are expected") and is left unevaluated.

The trim pattern is translated to PCRE by the existing shared string-pattern
engine (`wl_pattern_to_regex`, `string_pattern.c`), so it accepts the same
vocabulary as `StringSplit` / `StringReplace` / `StringCases` / `StringMatchQ`
(literal strings, `RegularExpression`, the character-class heads, `~~`, `|`,
`Repeated`/`RepeatedNull`, `Except`, ‚Ä¶). Rather than the whole-string `\A‚Ä¶\z`
anchor used by `StringMatchQ`, `StringTrim` compiles two one-end matchers ‚Äî a
front matcher anchored at the current position (via a start-offset check, since
PCRE `\A`/`^` bind to absolute offset 0) and a back matcher `(?:patt)\z` matched
against a truncated subject length so `\z` tracks the shrinking end. Registered
as `Protected` in `regex_init()`; docstring in `info.c`; symbol `SYM_StringTrim`
in `sym_names.c`. Tests in `tests/test_stringfns.c`; no memory leaks (valgrind
totals identical to baseline).
