# Changelog: week of 2026-07-20 (Mon) – 2026-07-26 (Sun)


Feature additions and fixes recorded during this week.

## Numeric loop fast-path + iteration allocation cleanups (2026-07-20)

Closed the large gap to Mathematica on tight numeric loops. Reference (10^6
iterations, Apple M-series):

| Benchmark | before | after | Mathematica |
|-----------|-------:|------:|------------:|
| `Do[x = 3.5 x (1 - x), {10^6}]` | 1.16 s | **0.013 s** | 0.78 s |
| `Nest[3.5 # (1 - #)&, 1./Pi, 10^6]` | 1.48 s | **0.014 s** | 0.029 s |
| `For[i=0, i<10^6, i++, x = 3.5 x (1-x)]` | 2.34 s | **0.014 s** | 1.30 s |
| `Nest[Cos, 1.0, 10^6]` | ~1.4 s | **0.011 s** | — |

Two stages, no semantic change.

**Stage 1 — iteration allocation cleanups.**
- Scalar `Nest`/`Fold`/`FixedPoint` no longer retain the full O(n) iteration
  history: the shared `iter_run` (`src/funcprog.c`) takes a bounded-history
  window (1 for these; `*List` variants keep everything), so a 10^6-step run
  holds O(1) live nodes instead of 10^6 — which stops it defeating the bounded
  Expr node pool. (`NestWhile` still retains history: its step gates on the
  running count.)
- Residual per-iteration heap churn removed: a **bucketed free-list for
  function-node args arrays** (`src/expr.c`, lengths 1..8, capped per bucket
  like the node pool), stack buffers for the `Plus`/`Times` result vector
  (`final_args`), and a caller-provided buffer for `numeric_contagion_args`
  (`src/numeric.c`, small cases on the stack). These alone brought `Do` to
  0.78 s and `Nest`/`For` down proportionally.

**Stage 2 — automatic numeric fast-path (`src/numloop.{c,h}`, new module).**
A transparent `Compile`-like path: when a loop body is numeric-closed arithmetic
over machine-real variables, it is compiled once to a tiny `double` stack-machine
bytecode and the loop runs with zero per-iteration `Expr` allocation. Wired into
`Nest`, `Do` (both `Do[…, {n}]` and the integer range `Do[…, {i, imin, imax,
di}]`), `For` (canonical counter loop), `While`, `Fold` (seeded scalar),
`FixedPoint`, and `NestWhile` (default form). Bare numeric heads
(`Nest[Cos, 1.0, n]`, `FixedPoint[Cos, 1.0]`) are compiled too, not only pure
`Function[…]` bodies.

The imperative loops (`Do`/`For`/`While`) compile a *block* of assignments, not
just one: a `CompoundExpression` body `x = 3.5 x (1-x); y = 4 x; x = y/4` with
multiple state variables and temporaries compiles all statements over one shared
register file, running them in order each iteration (`Do[…; …; …, {10^6}]` went
1.85 s → 0.027 s). A variable read before it is assigned, a non-assignment
statement (e.g. `Print`), or a symbolic operand makes the block decline and run
interpreted.

**Dense `NDArray[…]` bodies** are handled by *loop fusion*: for `Nest[f, arr, n]`
and `Do[a = …, {n}]` where the state variables are small float64 NDArrays of one
shape, the compiled scalar bytecode is run *per element* over the flat buffers,
fusing the whole element-wise map with no intermediate array temporaries and no
per-iteration allocation. This is gated to `≤ 256` elements: above that the
interpreter's vectorized NDArray kernels (tight C / BLAS loops) already win, so
larger arrays decline the fast-path. On small arrays in tight loops the win is
large — `Nest[3.5 # (1-#)&, arr8, 200000]` (8-element array) went 0.199 s →
0.025 s (~8×). Complex/float32 arrays, read-only array operands, and `For`/
`While`/`Fold` over arrays fall through to the interpreter.

- Supported ops: `Plus`, `Times`, `Power` (via libm `pow`, matching the
  interpreter), and the total-on-ℝ unary functions (`Sin`/`Cos`/`Tan`/`Sinh`/
  `Cosh`/`Tanh`/`Exp`/`Log`/`Sqrt`/`Abs`/`ArcTan`); any variable-free
  subexpression (`Pi`, `Sqrt[2]`, rationals) folds to a constant.
- **Correctness gate:** the fast path fires only when every variable
  numericalizes to a machine real *and* the result is guaranteed inexact — the
  seed is already a `Real`/MPFR, or the body/list carries a `Real` literal.
  Exact or symbolic loops (`Nest[#^2&, 2, 4]` → `65536`, `Nest[Sin, 2, 3]` →
  `Sin[Sin[Sin[2]]]`, `Do[x = x+1, {5}]` with integer `x`) decline it and run
  the interpreter unchanged. A non-finite intermediate (overflow, `Log` of a
  negative — where the interpreter would go complex) aborts the attempt and
  falls back. Results agree with the interpreter to floating-point rounding;
  they are *not* bit-identical, because the interpreter's Orderless `Plus`/
  `Times` sort operands by runtime value before folding (a data-dependent order
  a static compile can't reproduce, and floating `*`/`+` aren't associative).
- Control flow (`Break`/`Return`/`Continue`), iterator-form `Do[…, {i, a, b}]`,
  list/symbolic bodies, and custom `SameTest` all fall through to the
  interpreter. New `tests/test_numloop.c` (`numloop_tests`) checks each shape
  differentially against the interpreter (fast path toggled via
  `numloop_set_enabled` / `MATHILDA_NO_NUMLOOP`) plus the fallbacks;
  `bench_assoc` gate and valgrind remain clean.

## Evaluator: allocation + arithmetic hot-path optimizations (2026-07-20)

Performance work on the core evaluator. Profiling the logistic-map benchmark
`Module[{x = 1/Pi}, Do[x = 3.5 x (1 - x), {10^6}]; x]` showed ~75% of wall time
in the system allocator (`malloc`/`free`/`calloc`) churning short-lived,
fixed-size `Expr` nodes — the evaluator was allocation-bound, not compute-bound.
The reference benchmark went from **`Do` 3.10 s → 1.10 s (~2.8×)** and **`Nest`
3.46 s → 1.50 s (~2.3×)**, with identical results, via:

1. **Bounded Expr node pool** (`src/expr.c`). Every `Expr` is exactly
   `sizeof(Expr)` bytes, so `expr_free` recycles nodes onto a singly-linked
   free-list (link stored in the dead node's payload) and `expr_new_*` /
   `expr_unshare` pop from it. The list is **capped** (`EXPR_POOL_CAP` = 8192):
   the evaluator's tens-of-nodes working set always hits the hot cap, while a
   large batch (e.g. `KeySort` building a 40k-key association) drains the cap
   and falls to fresh `malloc`, whose burst locality keeps sort-heavy scaling
   at baseline. An *unbounded* pool measurably regressed `KeySort`'s O(n log n)
   doubling ratio (~2.2 → ~3.8) by scattering the recycled set across memory;
   the cap fixes that. `expr_pool_free_all()` (new, in `expr.h`) drains the list
   and is registered with `atexit()` so every binary leaves a clean heap.
2. **Stack buffer for `evaluate_step`'s argument scratch** (`src/eval.c`). Calls
   with ≤ `EVAL_SMALL_ARGS` (8) arguments build their evaluated-argument array
   on the stack, removing a `malloc`/`free` pair per function node per pass;
   the held-`Unevaluated` tracking array is likewise allocated lazily.
3. **Symbol OwnValue fast path** (`src/symtab.c`). Reading a plain `x = value`
   binding no longer allocates a `MatchEnv` or runs the pattern matcher — the
   bare-symbol pattern matches unconditionally, so it clones the value directly.
   The symbol node also caches its resolved `SymbolDef`, skipping a table hash
   per read.
4. **`Plus`/`Times` small-collector on the stack** (`src/plus.c`, `src/times.c`).
   The like-term/like-base collector and its open-addressing hash table are
   stack-allocated for ≤ 8 terms (three `malloc`/`free` pairs removed per small
   `Plus`/`Times`); larger sums/products still use the heap.
5. **Cheaper numeric fast paths**: `numeric_contagion_args` returns early when
   all operands are already machine reals (no array or per-arg reals allocated);
   scalar `Plus`/`Times` skip the NDArray machinery (three arg scans) unless an
   NDArray operand is present; the Flat-flatten hot path no longer re-interns an
   already-interned head symbol.

No semantic change — 13 evaluator/arithmetic/pattern/functional/list/poly/numeric
suites pass, the `bench_assoc` performance gate passes (`KeySort` back to ~2.2),
and valgrind reports no Mathilda-attributed errors or leaks. New coverage in
`tests/test_expr_pool.c` (`expr_pool_tests`).

## `StringPosition` — positions of substrings matching a pattern (2026-07-20)

Added `StringPosition` in a new translation unit
`src/strings/regex/stringposition.c`, registered in `regex_init()` (backed by
the shared PCRE2 pattern engine, so not `Listable`; docstring in `info.c`). It
returns `{start, end}` position pairs in the form consumed by `StringTake` /
`StringReplacePart`.

- `StringPosition["abXYZaaabXYZaaaaXYZXYZ", "XYZ"]` →
  `{{3, 5}, {10, 12}, {17, 19}, {20, 22}}`.
- `StringPosition["s", patt, n]` keeps the first `n` matches; a list of subjects
  threads; a list of patterns merges each pattern's matches ordered by start.
- Option `Overlaps -> True` (default, one match per start with overlaps),
  `False` (no overlaps, greedy/global), or `All` (every matching substring at
  every start): `StringPosition["AAAA", x__, Overlaps -> All]` →
  `{{1,4},{1,3},{1,2},{1,1},{2,4},{2,3},{2,2},{3,4},{3,3},{4,4}}`. Option
  `IgnoreCase -> True` folds case. New interned symbol `SYM_Overlaps`; `Overlaps`
  registered as an inert protected head. `Options[StringPosition]` is registered
  as `{IgnoreCase -> False, Overlaps -> True}`; the builtin seeds its option
  state from these defaults so `SetOptions[StringPosition, ...]` takes effect,
  with explicit trailing options overriding.

Shared string-pattern translator (`string_pattern.c`) extended to support the
patterns these examples need (a strict improvement also benefiting `StringCases`
/ `StringReplace` / `StringMatchQ`):

- **Named backreferences**: a repeated `Pattern` name now compiles to `\g{n}`
  rather than a second independent capture, so `x_ ~~ x_` matches two *equal*
  characters (previously matched any two characters).
- **`BlankSequence` / `BlankNullSequence`**: `__` → `.+`, `___` → `.*`.
- **Nested list as alternatives**: a `List` inside a pattern (e.g. inside
  `StringExpression`) compiles to an alternation, so
  `StringExpression[{x__, "A"}]` behaves like `x__ | "A"`.

## `StringRiffle` — assemble a string from a list with separators (2026-07-20)

Added `StringRiffle` in a new translation unit `src/strings/stringriffle.c`,
registered in `strings_init()` (no regex dependency; symbol interned as
`SYM_StringRiffle`, docstring in `info.c`). The inverse of `StringSplit`;
byte-length based like the rest of the string subsystem. Non-string leaves are
converted with `ToString`.

- `StringRiffle[{"a", "b", "c", "d", "e"}]` → `"a b c d e"`. Nested lists use the
  default scheme — spaces at the innermost level, one extra newline per level up:
  a 2-D array joins rows with `"\n"` and cells with `" "`; 3-D joins blocks with
  `"\n\n"`.
- `StringRiffle[list, sep]` inserts the string `sep` between the top-level
  elements (deeper levels default): `StringRiffle[{"a", …}, ", "]` →
  `"a, b, c, d, e"`.
- `StringRiffle[list, {"left", "sep", "right"}]` is a delimiter triple —
  `StringRiffle[{"a", …}, {"(", " ", ")"}]` → `"(a b c d e)"`.
- `StringRiffle[list, sep1, sep2, …]` sets `sep_i` (a string or a triple) at
  level `i`; e.g. `StringRiffle[{{"a", 27}, {"b", 28}, {"c", 29}}, {"{", ", ",
  "}"}, ": "]` → `"{a: 27, b: 28, c: 29}"`.
- An empty list gives `""` (delimiters only for a triple). A first argument that
  is neither a list nor a string, or a malformed separator (not a string, not a
  3-string list), leaves the call unevaluated; no arguments emits
  `StringRiffle::argm` (1 or more arguments expected).
- **Attributes**: `Protected`. Tests in `tests/test_stringriffle.c`; leak-clean
  (`leaks`: 0 leaks).

## `StringPadLeft` / `StringPadRight` — pad or truncate strings to a length (2026-07-20)

Added `StringPadLeft` and `StringPadRight` in a new translation unit
`src/strings/stringpad.c` (both `Protected`; symbols `SYM_StringPadLeft` /
`SYM_StringPadRight`; docstrings in `info.c`). Byte-length based, like the rest
of the string subsystem.

- `StringPadLeft["abcde", 10]` → `"     abcde"`; `StringPadRight` pads on the
  right. `StringPad*["string", n, "padding"]` pads with cyclic copies of
  `padding` (`padding[i mod Length]`, truncated at width `n`), e.g.
  `StringPadLeft["abcde", 10, "."]` → `".....abcde"`.
- When `n` is shorter than the string it truncates: `StringPadLeft` keeps the
  last `n` characters, `StringPadRight` the first `n`.
- Threads over a list of strings: the one-argument list form
  `StringPadLeft[{"a", "ab", …, "abcde"}]` pads each to the longest length;
  `StringPad*[{s1, …}, n, …]` pads/truncates each to `n`.
- A non-string subject (not a list of strings), a non-integer or negative `n`, a
  non-string / list-valued `padding`, an empty `padding` when padding is needed,
  or a non-string list element leaves the call unevaluated. Other arities emit
  `StringPadLeft::argb` / `StringPadRight::argb` (1 to 3 arguments expected).
- Tests in `tests/test_stringpad.c`; no new memory leaks (valgrind parity with
  the sibling `stringrepeat_tests` baseline).

## `StringRepeat` — build a string by repeating a base string (2026-07-20)

Added `StringRepeat` in a new translation unit `src/strings/stringrepeat.c`,
registered in `strings_init()` (no regex dependency; symbol interned as
`SYM_StringRepeat`, docstring in `info.c`).

- `StringRepeat["a", 50]` → 50 `a`s; `StringRepeat["abc", 10]` →
  `"abcabcabcabcabcabcabcabcabcabc"`.
- `StringRepeat["string", n, max]` truncates to a total length of at most `max`,
  allowing a partial final copy: `StringRepeat["ab", 10, 19]` →
  `"abababababababababa"` (19 chars). The fill is cyclic (`buf[i] = str[i % len]`),
  so a mid-block cutoff is handled naturally.
- `n == 0`, `max == 0`, or an empty base string gives `""`. A non-string subject,
  a non-integer or negative `n`/`max` leaves the call unevaluated; other arities
  emit `StringRepeat::argt`. The `n * len` product is guarded against `size_t`
  overflow (unbounded overflow → unevaluated; a `max` cap keeps the result finite).
- **Attributes**: `Protected`. Tests in `tests/test_stringrepeat.c`.

## `StringPartition` — partition a string into fixed-length substrings (2026-07-20)

Added `StringPartition` in a new translation unit `src/strings/stringpartition.c`,
registered in `strings_init()` (no regex dependency). It carves a string into
fixed-length substrings, covering the Wolfram surface:

- `StringPartition["123456789123456789", 9]` → `{"123456789", "123456789"}` —
  non-overlapping length-`n` blocks (offset `d = n`); a trailing run too short to
  fill a block is dropped, e.g. `StringPartition["ababababab", 3]` →
  `{"aba", "bab", "aba"}`.
- `StringPartition["string", n, d]` starts a length-`n` block every `d`
  characters. `StringPartition["123456789", 2, 1]` →
  `{"12","23",...,"89"}` (overlap, `d < n`); `d > n` skips characters in the
  middle. All blocks stay length `n`.
- `StringPartition["string", UpTo[n]]` allows a shorter final block so every
  character appears: `StringPartition["123456789", UpTo[2]]` →
  `{"12","34","56","78","9"}`. Combinable with an explicit offset (`UpTo[n], d`).
- A list of strings threads (`StringPartition[{"abcd","ef"}, 2]` →
  `{{"ab","cd"},{"ef"}}`). Non-string subjects, non-integer/non-positive `n`/`d`
  leave the call unevaluated; other arities emit `StringPartition::argt`
  (2 or 3 arguments expected).

Strings are treated as raw byte arrays (matching `StringTake`/`StringDrop`); no
UTF-8 codepoint decoding is performed. Attributes: `Protected`. Docstring in
`info.c`; symbol `SYM_StringPartition` interned in `sym_names.c`. Covered by the
new `tests/test_stringpartition.c` (unconditional, not `USE_REGEX`-gated).

## `StringExtract` — split into blocks and select by position (2026-07-20)

Added `StringExtract` in a new translation unit `src/strings/stringextract.c`,
registered in `regex_init()` alongside its `StringSplit` dependency. It splits a
string into blocks and selects blocks by position, covering the Wolfram surface:

- `StringExtract["a bbb  cccc aa   d", 2]` → `"bbb"` — whitespace-delimited nth
  block; `-n` counts from the end (`... , -1]` → `"d"`).
- Position specs: `n`, `-n`, `{n1, n2, ...}` (collection), `n1;;n2` (span), and
  `All`. `StringExtract["a bbb  cccc aa   d", 2;;4]` → `{"bbb","cccc","aa"}`;
  `{1,4,5}` → `{"a","aa","d"}` (equivalently `Part[StringSplit[...], {1,4,5}]`).
- `sep -> pos` delimits blocks with a string pattern: `"a--bbb--ccc--dddd" ,
  "--" -> 3]` → `"ccc"`, and `sep -> All` equals `StringSplit["string", sep]`.
- Multi-level extraction `StringExtract["string", pos1, pos2, ...]` delimits with
  whitespace at the lowest level, `"\n"` at the next, `"\n\n"` above that, and so
  on (or explicit `sepi -> posi` per level). E.g. on a 3-paragraph text
  `StringExtract[text, 2, 2, 3]` → the 3rd word of the 2nd line of the 2nd
  paragraph.
- Absent blocks yield `Missing["PartAbsent", pos]`:
  `StringExtract["a 1\nb 2\nc 3 x", All, 3]` →
  `{Missing["PartAbsent",3], Missing["PartAbsent",3], "x"}`.
- Threads over a list of subject strings. A non-string, non-list subject leaves
  the call unevaluated; fewer than 2 arguments emits `StringExtract::argm`
  ("2 or more arguments are expected").

The implementation is a thin, DRY layer: each level builds `StringSplit[str,
sep]` and evaluates it to produce the blocks, so the entire string-pattern engine
(`wl_pattern_to_regex`), whitespace-run collapsing, and empty-end trimming are
reused verbatim — making the `sep -> All == StringSplit` and `{pos...} ==
Part[StringSplit[...], {pos...}]` equivalences exact. Position resolution reuses
`expr_part` for spans; integer/out-of-range handling produces `Missing` directly.
Attributes: `Protected`. Docstring in `info.c`; symbols `SYM_StringExtract` /
`SYM_StringSplit` added to `sym_names.{c,h}`. Eight test groups in
`tests/test_stringfns.c` cover every worked example; valgrind-clean.

## `StringTrim` — trim matching substrings from both ends (2026-07-20)

Added `StringTrim` in a new translation unit `src/strings/stringtrim.c`, covering
the Wolfram surface:

- `StringTrim["  aaa bbb  "]` → `"aaa bbb"` — the 1-argument form trims runs of
  whitespace (spaces, tabs, newlines) from both ends.
- `StringTrim["string", patt]` trims substrings matching `patt` from the start
  and the end, e.g. `StringTrim["++++x----", ("+"|"-")...]` → `"x"` and
  `StringTrim["007bond007", DigitCharacter..]` → `"bond"`. Each end is trimmed to
  a fixed point, so a literal single-character pattern removes a whole run
  (`StringTrim["xxabcxx", "x"]` → `"abc"`).
- A pattern that anchors itself to the start only trims the front:
  `StringTrim["   aaa   ", RegularExpression["^ *"]]` → `"aaa   "`.
- Threads over a list of subject strings, passing non-string elements through:
  `StringTrim[{"  a  ", 5, x}]` → `{"a", 5, x}`.
- A non-string subject leaves the call unevaluated; `StringTrim[]` emits
  `StringTrim::argt` ("1 or 2 arguments are expected") and is left unevaluated.

The trim pattern is translated to PCRE by the existing shared string-pattern
engine (`wl_pattern_to_regex`, `string_pattern.c`), so it accepts the same
vocabulary as `StringSplit` / `StringReplace` / `StringCases` / `StringMatchQ`
(literal strings, `RegularExpression`, the character-class heads, `~~`, `|`,
`Repeated`/`RepeatedNull`, `Except`, …). Rather than the whole-string `\A…\z`
anchor used by `StringMatchQ`, `StringTrim` compiles two one-end matchers — a
front matcher anchored at the current position (via a start-offset check, since
PCRE `\A`/`^` bind to absolute offset 0) and a back matcher `(?:patt)\z` matched
against a truncated subject length so `\z` tracks the shrinking end. Registered
as `Protected` in `regex_init()`; docstring in `info.c`; symbol `SYM_StringTrim`
in `sym_names.c`. Tests in `tests/test_stringfns.c`; no memory leaks (valgrind
totals identical to baseline).
