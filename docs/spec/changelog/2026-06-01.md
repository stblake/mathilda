## Fix: `Power[-1, p/q]` floors the exponent into `[0,1)` (2026-06-07)

For a negative base, `builtin_power` (`src/power.c`) now uses **floor**
division of the rational exponent `p/q` so the residual `(-1)^(b/q)` lands in
`[0, 1)`, pulling out a `(-1)^a = Вұ1` sign that merges into the coefficient вҖ”
Mathematica's canonical form. Previously truncation toward zero left negative
residual exponents (`|p| < q` skipped extraction entirely), so `(-1)^(-1/5)`
stayed `Power[-1, Rational[-1, 5]]`. Now:

```
(-1)^(-1/5)  -> Times[-1, Power[-1, Rational[4, 5]]]
(-1)^(-2/3)  -> -(-1)^(1/3)
(-1)^(-7/5)  -> (-1)^(3/5)
(-8)^(-1/3)  -> Times[Rational[-1, 2], Power[-1, Rational[2, 3]]]
```

Positive bases keep truncation toward zero (unchanged): `2^(-5/3)`, `2^(7/3)`
are byte-identical to before. The `q == 2` arm (`(-1)^(3/2) -> -I`) and the
even-`q вүҘ 4` non-`-1` bail are unaffected. One downstream test expectation was
updated: `(-I Pi)^(1/2)` now prints `-(-1)^(3/4) Sqrt[Pi]` (вүЎ the old
`Sqrt[Pi]/(-1)^(1/4)`), matching Mathematica. Verified against
`power_corpus_tests`, `radical_canonical_tests`, `radicals_tests`,
`solve_tests`, `piecewise_tests`, `series_tests`, `simplify_tests`,
`numeric_complex_tests`, `bigint_tests`.

## Fix: `Exp[I q Pi]` canonicalizes to `(-1)^q` (2026-06-07)

`builtin_exp` (`src/logexp.c`) previously expanded `Exp[I*q*Pi]` via Euler's
formula into `Cos[q Pi] + I Sin[q Pi]`, which then collapsed eagerly into trig
radicals вҖ” so even bare `Exp[I Pi/5]` evaluated to
`1/4 (1 + Sqrt[5]) + (I/4) Sqrt[10 - 2 Sqrt[5]]`, and `Simplify`/`Together`
inherited that bloated form. It now produces `Power[-1, q]`, matching
Mathematica: `Exp[I Pi/5] -> (-1)^(1/5)`, `Simplify[Exp[I Pi/5]] -> (-1)^(1/5)`,
`Together[Exp[I Pi/5]] -> (-1)^(1/5)`. `Power[-1, q]` still reduces the
special cases on its own (`Exp[I Pi] -> -1`, `Exp[I Pi/2] -> I`,
`Exp[2 I Pi] -> 1`, `Exp[3 I Pi/2] -> -I`), so `Table[Exp[I n Pi/2], {n,0,4}]`
remains `{1, I, -1, -I, 1}`. Verified against `logexp_tests`, `trig_tests`,
`squarefreeq_tests`, `solve_tests`, `core_tests`, `simplify_tests`.

## HypergeometricPFQ вҖ” generalized hypergeometric function (2026-06-07)

New builtin `HypergeometricPFQ` (`src/hypergeopfq.c` / `.h`, registered in the new
`hypergeopfq_init`, called from `core_init`). Attributes `NumericFunction |
Protected`; symbols `HypergeometricPFQ`, `Hypergeometric0F1`, `Hypergeometric1F1`,
`Hypergeometric2F1`, `HypergeometricPFQRegularized` in `sym_names.{c,h}`;
docstrings in `info.c`; reference page
`docs/spec/builtins/special-functions.md` (new category row in `Mathilda_spec.md`).

`HypergeometricPFQ[{a1,вҖҰ,ap}, {b1,вҖҰ,bq}, z]` is the series
`Sum_{k>=0} (prod_i Pochhammer[a_i,k] / prod_j Pochhammer[b_j,k]) z^k / k!`.
Evaluation, in order:

- **Structural.** `pFq[a,b,0] -> 1`; threading over a `List` third argument
  (`HypergeometricPFQ[{1,2,3,4},{5,6,7},{0.1,0.3,0.5}]` вҶ’
  `{1.01164, 1.03627, 1.06296}`); generic cancellation of parameters common to the
  upper and lower lists (`pFq[{a,b,c},{a,d,e},z] -> pFq[{b,c},{d,e},z]`), which is
  suppressed for a non-positive integer common element so the polynomial
  interpretation wins (`pFq[{-2,a,b,c},{-2,d,e},z]`).
- **Termination.** A non-positive integer upper parameter truncates the series to
  an explicit polynomial, valid for symbolic `z`
  (`pFq[{-3,a},{b,c},x]` вҶ’ the degree-3 polynomial).
- **Reductions** to elementary closed forms (analytic continuations, needing no new
  function heads): `0F0 -> E^z`, `1F0(a) -> (1-z)^(-a)` (so the geometric case
  `1F0(1) -> 1/(1-z)`), `0F1(1/2) -> Cosh[2 Sqrt[z]]`,
  `0F1(3/2) -> Sinh[2 Sqrt[z]]/(2 Sqrt[z])`, `1F1(1;2) -> (E^z-1)/z`,
  `2F1(1,1;2) -> -Log[1-z]/z`.
- **Numeric** by direct series summation at machine precision (`double complex`),
  arbitrary precision (MPFR, via `(re,im)` pairs вҖ” there is no MPC), and for complex
  arguments/parameters, with output precision tracking the input
  (`N[HypergeometricPFQ[{1,1,1},{3/2,3/2,3/2},10],50]` вҶ’
  `530.19188827362590438855961685444087792733053398358`;
  `HypergeometricPFQ[{I,I,I},{2,2,2},-1.0 I]` вҶ’ `0.870032 - 0.00484538 I`).
  The summation is gated to the convergent regime вҖ” `p<=q` (entire, all finite `z`)
  and `p==q+1` with `|z|<1`; outside it (and `p>q+1`, asymptotic only) the call
  stays unevaluated, since analytic continuation outside the unit disk is a
  deliberate follow-on.

`Hypergeometric0F1[b,z]`, `Hypergeometric1F1[a,b,z]`, `Hypergeometric2F1[a,b,c,z]`
are thin convenience heads that rewrite to `HypergeometricPFQ`.

Derivative rule added to `src/calculus/deriv.c`:
`D[pFq[{a},{b},x],x] = (prod a_i / prod b_j) pFq[{a_i+1},{b_j+1},x]` (times the
chain factor); deferred to the generic chain rule when the parameter lists depend
on the differentiation variable.

Extensive tests in `tests/test_hypergeopfq.c` (`hypergeopfq_tests`): origin,
exact-stays-symbolic, cancellation (incl. negative-integer guard), termination,
threading, all reductions, convenience heads, machine/complex/MPFR numerics, the
convergence gate, and the derivative. Leak-clean under macOS `leaks --atExit`
(0 bytes), strict-C99 `-Wall -Wextra` clean.

**Deferred** (research-grade, aligned with the `Sum` development plan's special
functions): analytic continuation for `p==q+1` outside `|z|<1` and asymptotics for
`p>q+1`; `Series` at `z->Infinity` and the branch point `z==1`; `FunctionExpand` to
Bessel and `PolyLog`-based reductions; the full `HypergeometricPFQRegularized` and
`Integrate` rules producing `pFq`.

## Sum: infinite hypergeometric series via HypergeometricPFQ (2026-06-07)

New summation stage `Sum`Hypergeometric` (`src/sum/sum_hypergeometric.c`),
appended last in the `Automatic` cascade in `src/sum/sum.c` (after
`Sum`Gosper`). It closes infinite sums `Sum[t(k), {k, imin, Infinity}]` whose
summand is a **hypergeometric term** вҖ” `t(k+1)/t(k)` a rational function of `k`
вҖ” for a concrete integer lower bound.

The (reindexed) term ratio is factored into monic linear factors over Q
(`Solve` for each root, `PolynomialQuotient` to deflate): numerator roots give
the upper parameters, the denominator roots minus the canonical `(m+1)`
factorial factor give the lower parameters, and the ratio of leading
coefficients is the argument `z`. The result `t(imin) HypergeometricPFQ[{a},{b},z]`
is then reduced by `HypergeometricPFQ` to a closed form when possible. This
makes, among others:

```
Sum[z^k/k!, {k, 0, Infinity}]        -> E^z
Sum[z^k/k!, {k, 1, Infinity}]        -> -1 + E^z
Sum[1/k!, {k, 0, Infinity}]          -> E
Sum[x^k, {k, 0, Infinity}]           -> 1/(1 - x)          (* subsumes geometric *)
Sum[z^k/(2 k)!, {k, 0, Infinity}]    -> Cosh[Sqrt[z]]
Sum[1/(2 k + 1)!, {k, 0, Infinity}]  -> Sinh[1]
```

**Convergence gate.** `p<=q` is entire; `p==q+1` is summed only for `|z|<1`
when `z` is numeric. Outside that (and `p>q+1`) the sum stays unevaluated rather
than returning the analytic continuation of a divergent series вҖ” so
`Sum[2^k, {k,0,Infinity}]` and `Sum[k, {k,1,Infinity}]` are left held (not
`1/(1-2) = -1`, etc.). When `z` is symbolic the closed form is returned as the
conventional CAS answer (e.g. `Sum[x^k,вҖҰ] -> 1/(1-x)`).

Tests in `tests/test_sum.c` (the closed forms above plus the two divergent
hold-cases); `sum_tests` and `hypergeopfq_tests` leak-clean under `leaks`.

## Fix: `N` on a huge exact integer overflowed to `inf` (2026-06-07)

`N[1001!]` (and `2^2000 // N`, etc.) printed `inf.0`. In machine mode,
`leaf_from_bigint` (`src/numeric.c`) converted the bigint with `mpz_get_d`,
which overflows to `Вұinf` once the magnitude exceeds `DBL_MAX` (~1.8e308).
Mathematica's machine-precision numbers carry an arbitrary exponent, so the
machine path now detects the `isinf` overflow and falls back to a
machine-precision (`DBL_MANT_DIG`-bit) MPFR real, which is finite for any
magnitude вҖ” `N[1001!]` вҶ’ `4.027896473371709e+2570`. In-range integers still
produce ordinary IEEE doubles, and MPFR mode (`N[1001!, 30]`) is unchanged.

The same overflow lurked in the second leaf path: `N`'s `EXPR_MPFR` вҶ’
machine down-conversion used a bare `mpfr_get_d`. An already-out-of-range MPFR
argument вҖ” `1.5 + 1001!` is promoted to an MPFR ~4e2570 by `Plus`'s inexact
contagion before `N` runs вҖ” re-overflowed to `inf` on the way down. That path
now keeps any finite-but-out-of-range value as a `DBL_MANT_DIG`-bit MPFR
(guarded by `mpfr_number_p`, so a genuinely infinite MPFR still yields `inf`).
`N[1.5 + 1001!]` and `N[1.5 * 1001!]` are now finite.
Regression test: `test_n_bigint_machine_overflow` in `tests/test_numeric.c`.

## Simplify: `TransformationFunctions` option + docstring restructure (2026-06-07)

`Simplify` now honours the **`TransformationFunctions`** option, matching
Mathematica's semantics for choosing which functions are applied to try to
transform parts of an expression:

- `TransformationFunctions -> Automatic` (default) вҖ” the built-in collection
  (Together, Cancel, Expand, Factor, FactorSquareFree, Apart, TrigExpand,
  TrigFactor, the TrigToExp/ExpToTrig roundtrip, вҖҰ), unchanged behaviour.
- `TransformationFunctions -> {f1, f2, вҖҰ}` вҖ” applies **only** the `fi`; the
  built-in pipeline is suppressed.
- `TransformationFunctions -> {Automatic, f1, вҖҰ}` вҖ” the built-in functions
  **together with** the `fi`.

The `fi` may be any function вҖ” a builtin head (`Together`, `Cancel`, вҖҰ) or a
pure function such as `(# /. a -> 0 &)`. New module
`src/simp/simp_transform.c` provides `simp_apply_transformations`, a bounded
global-best fixed point: each round applies every user function to the whole
expression and to every subexpression, re-evaluates, and keeps the candidate of
lowest complexity (scored with the same `ComplexityFunction` measure used
throughout the search). `builtin_simplify` (`src/simp/simp_builtins.c`) parses
the option, gates the built-in pipeline on whether `Automatic` is present, and
then applies the user functions. The option also propagates through Simplify's
list/relation threading and the inexact-input rationalise/numericalise path.

The `Simplify` docstring was restructured into a forms/options/behaviour layout
and now documents all three options; `TransformationFunctions` gains its own
docstring (discoverable via `?TransformationFunctions`). Seven regression tests
added in `tests/test_simplify.c` (default, suppression, user-only, user+builtin,
custom pure function). Leak-clean (macOS `leaks`), strict-C99 clean.

Also added a dedicated **Simplification** reference page,
`docs/spec/builtins/simplification.md`, covering `Simplify` (transforms,
specialised simplifications, assumptions, options), `SimplifyCount`,
`TransformationFunctions`, `Assuming`, `$Assumptions`, `Element`, and
`$SimplifyDebug`, with a new category row in the `Mathilda_spec.md` index. All
examples on the page are verified against the current build.

## Integrate: linear-ratio-radicals (MГ¶bius) rationalising substitution (2026-06-06)

Added `Integrate`LinearRatioRadicals[f, x]` вҖ” integration of a **rational
function of `x` and radicals `((a x + b)/(c x + d))^(m/n)` of one shared
linear-fractional (MГ¶bius) argument** by the substitution
`u = ((a x + b)/(c x + d))^(1/n)`, `n = LCM[n1, n2, ...]`.  New module
`src/calculus/integrate_linratiorad.c` (+ `.h`); wired into
`src/calculus/integrate.c` as a new cascade stage and a
`Method -> "LinearRatioRadicals"` value.  Sibling of the recent
`Integrate`LinearRadicals` / `Integrate`QuadraticRadicals` stages.

Inverting the MГ¶bius map gives `x = (d u^n - b)/(a - c u^n)`,
`dx = n (a d - b c) u^(n-1)/(a - c u^n)^2 du`, so the integrand becomes a
rational function of `u` that `Integrate`BronsteinRational` integrates exactly:

```
Integrate[R[x, ((a x+b)/(c x+d))^(m1/n1), ...], x]
  = Integrate[ R[(d u^n - b)/(a - c u^n), u^M1, ...]
               В· n (a d - b c) u^(n-1)/(a - c u^n)^2, u ],  Mk = mk n/nk.
```

Identification scans `f` for `Power[base, p/q]` with `|q| > 1`; all bases must be
**structurally identical** (`n = LCM` of the `|q|`).  Crucially the base is
**not** required to be a polynomial вҖ” it is the ratio
`Times[a x + b, Power[c x + d, -1]]`, which is what cleanly partitions this
method from `LinearRadicals` / `QuadraticRadicals` (whose scans demand a
polynomial base, so a ratio radical falls through to here).  The shared base is
canonicalised with `Cancel[Together[.]]`, its MГ¶bius coefficients read off the
numerator (`a, b`) and denominator (`c, d`); the denominator must be genuinely
linear (degree 1 вҖ” a constant denominator is `LinearRadicals`' job) and the map
non-degenerate (`a d - b c != 0`).  The forward rewrite reuses
`poly_subst_radical_to_gen`; the substituted integrand is confirmed rational in
`{x, u}` before recursing into the full `Integrate`, then back-substituted
`u -> ((a x + b)/(c x + d))^(1/n)`.

**No differentiate-back gate** (matching the other radical stages): the MГ¶bius
substitution is an exact bijection on the relevant domain, so the result is
correct by construction once the rational sub-integral closes.  A depth guard
(8) and per-call fresh symbols keep the recursion finite.

**Cascade placement:** after `QuadraticRadicals`, before `DerivativeDivides`.
Headline wins the other stages miss cleanly:
`Integrate[1/Sqrt[(x + 1)/(x - 1)], x]` and
`Integrate[((x - 1)/(x + 1))^(1/3)/x, x]` (the latter rationalises to
`6 u^3/((u^3 + 1)(1 - u^3))`, a degree-3 rational integral).

Regression coverage in `tests/test_integrate_linratiorad.c`
(`integrate_linratiorad_tests`): automatic cascade, method plumbing, a cube-root
and an LCM=6 case verified numerically (their residuals carry radicals Simplify
cannot collapse), the inexact-coefficient path, strict no-match cases, and a
cascade-partition check (linear/quadratic cases still route to their own
stages).  Docs: `docs/spec/builtins/calculus.md`
(`### Integrate`LinearRatioRadicals`).

The new module is **leak-clean** under macOS `leaks` (0 bytes for the matching
and no-match paths).  Implementing it surfaced one **pre-existing** leak in the
rational integrator's log-combiner `intsimp_log_to_arctanh`
(`src/calculus/intsimp.c:456` вҖ” an `expr_expand` input subtree never freed),
reachable with no radicals at all via `Integrate[6 u^3/((u^3+1)(1-u^3)), u]`;
logged in `MEMORY_LEAKS.md`, not introduced here.

## Factor: fix dropped integer content in the bivariate Hensel path (2026-06-06)

`Factor` silently dropped a constant factor for squarefree two-variable inputs
with distinct factors and a non-unit leading coefficient вҖ” e.g.
`Factor[4 x^2 - 4 y^2]` returned `(x + y) (x - y)` (= P/4) instead of
`4 (x + y) (x - y)`. The Stage-2 Wang x-scale recovery in
`try_bivariate_via_hensel_with_main` (`src/poly/facpoly_bv_hensel.inc`)
reconstructs the product of **primitive** factors, so the integer content of
the input was never re-multiplied. Perfect-square / repeated-factor inputs were
unaffected (they fall back through the content-correct
`FactorSquareFree`/`heuristic_factor` pipeline), which is why the bug evaded
the existing suite.

The fix computes the input's integer content (`bpoly_int_content`) before the
lift and re-multiplies it into the assembled factorisation (a no-op when the
content is 1, i.e. for the MONIC/NEGATE leading-coefficient cases).

**Downstream impact on Simplify of single-radical fractions.** `Simplify`'s
algebraic-extension reduction (`simp_algebraic`) substitutes `Sqrt[u] -> g` and
calls `Factor` on the resulting polynomial in `(x, g)`; the dropped content
corrupted the value by a constant factor. With the fix,
`Simplify[D[Integrate[Sqrt[x^2 - x - 1]/x, x], x]]` now recovers the integrand
`Sqrt[-1 - x + x^2]/x` exactly (previously it returned a value 4Г— too large).

Regression coverage: `test_factor_bivariate_content_regression` in
`tests/test_facpoly.c` and `test_radical_integrand_recovery_sqrt_quadratic` in
`tests/test_radical_simplify.c`.

## Integrate: derivative-divides substitution method (2026-06-06)

Added `Integrate`DerivativeDivides[f, x]` вҖ” integration **by substitution**
(the classical "derivative-divides" method of Moses' SIN / Maxima's
`diffdiv`).  New module `src/calculus/integrate_derivdivides.c` (+ `.h`); wired
into `src/calculus/integrate.c` as a new cascade stage and a
`Method -> "DerivativeDivides"` value.

It recognises `f(x) = c В· h(u(x)) В· u'(x)`, reduces to `Integrate[h[u], u]`,
and back-substitutes `u -> u(x)`.  Candidate kernels `u(x)` are the distinct
`x`-dependent subexpressions of `f` (the C analogue of
`Cases[Union[Level[f, {0, Infinity}]], e_ /; !FreeQ[e, x]]`).  Two
complementary strategies, each gated by the unconditional verification
`Simplify[D[result, x] - f] === 0`:

- **Direct quotient** `q = Cancel[Together[f / D[u(x), x]]]; q /. u(x) -> u`
  вҖ” cheap, no diagnostics, handles transcendental compositions like
  `x Exp[x^2]`, and picks the correct radical branch inherently.  This is the
  only strategy run inside the Automatic cascade.
- **Eliminate / Solve** (explicit method only) вҖ” the differential relation
  `Eliminate[{Dt[y] == f Dt[x], u == u(x), Dt[u == u(x)]}, {x, Dt[x]}]`,
  solved for `Dt[y]` and reduced with `Factor //@ `, `PowerExpand`,
  `Cancel[. / Dt[u]]`.  Closes cases the direct strategy cannot (e.g. where
  `Cancel` rewrites `1/Cos[x]` to `Sec[x]`).  Because the algebraisation
  squares radicals / inverts functions, `Solve` returns multiple branches; the
  verification gate selects the one that differentiates back to `f` вҖ” directly
  addressing the inverse-function branch-selection problem.

**Cascade placement:** after `BronsteinRational`, before `RischNorman`.  The
Automatic path is direct-only so it stays quiet (the Eliminate path can emit
`Eliminate::ifun` / `::alg` branch-caveat diagnostics) and fast.  Headline win
the other stages miss: `Integrate[Sin[x] Sqrt[1 - Cos[x]], x]` вҶ’
`2/3 (1 - Cos[x])^(3/2)`.  A depth guard (8) and per-call fresh substitution
symbols keep the recursion finite and collision-free.

Regression coverage added in `tests/test_integrate_derivdivides.c`
(`integrate_derivdivides_tests`).  Docs: `docs/spec/builtins/calculus.md`
(`### Integrate`DerivativeDivides`).

## Integrate: linear-radicals rationalising substitution (2026-06-06)

Added `Integrate`LinearRadicals[f, x]` вҖ” integration of a **rational function
of `x` and radicals `(a x + b)^(m/n)` of one shared linear argument** by the
substitution `u = (a x + b)^(1/n)`, `n = LCM[n1, n2, ...]`.  New module
`src/calculus/integrate_linrad.c` (+ `.h`); wired into
`src/calculus/integrate.c` as a new cascade stage and a
`Method -> "LinearRadicals"` value.

With `x = (u^n - b)/a`, `dx = (n/a) u^(n-1) du` the integrand becomes a rational
function of `u`, which `Integrate`BronsteinRational` integrates exactly:

```
Integrate[R[x, (a x+b)^(m1/n1), ...], x]
  = (n/a) Integrate[ R[(u^n - b)/a, u^M1, u^M2, ...] u^(n-1), u ],  Mk = mk n/nk.
```

Identification scans `f` for `Power[base, p/q]` with `q > 1`; each `base` must be
degree-1 in `x` and all must be the same `a x + b` (distinct or non-linear bases
are rejected).  The forward rewrite reuses `poly_subst_radical_to_gen` (shared
with the algebraic-factoring path); the substituted integrand is confirmed
rational in `{x, u}` before recursing into the full `Integrate`, then
back-substituted `u -> (a x + b)^(1/n)`.  A depth guard (8) and per-call fresh
symbols keep the recursion finite.

**No differentiate-back gate (update 2026-06-06).** Unlike the heuristic
methods, the result is accepted as soon as the rational sub-integral closes вҖ”
there is *no* `Simplify[D[result, x] - f] === 0` verification.  The substitution
is an exact bijection introducing no branch issues, so the antiderivative is
correct by construction.  The gate was also a correctness-irrelevant performance
trap: on integrands with symbolic parameters (e.g.
`Integrate[(1 - b x)^(1/3)/(1 - x^2), x]`) the `D[result, x] - f` residue is a
nest of symbolic radicals that stalls `PossibleZeroQ`/`Simplify` indefinitely вҖ”
removing the gate turns that hang into a ~0.1 s answer.  (This also removes the
only path by which this method touched the pre-existing `PossibleZeroQ`
SchwartzвҖ“Zippel leak logged in `MEMORY_LEAKS.md`.)

**Cascade placement:** after `BronsteinRational`, before `DerivativeDivides`.
Headline wins the other stages miss cleanly:
`Integrate[1/Sqrt[x + 1], x]` вҶ’ `2 Sqrt[1 + x]`;
`Integrate[1/(1 + x^(1/3)), x]` вҶ’
`-3 x^(1/3) + 3/2 x^(2/3) + 3 Log[1 + x^(1/3)]`.

Docs: `docs/spec/builtins/calculus.md` (`### Integrate`LinearRadicals`).  Note: a
**pre-existing** nondeterministic leak in `PossibleZeroQ`'s SchwartzвҖ“Zippel stage
(`src/zero_test.c`) was originally surfaced by this method's differentiate-back
gate over radical residues вҖ” logged in `MEMORY_LEAKS.md`, not introduced here.
With the gate now removed (see the update above) this method no longer reaches
that path; the underlying leak remains and is still reproducible via bare
`PossibleZeroQ`.

## Integrate: quadratic-radicals Euler substitution (2026-06-06)

Added `Integrate`QuadraticRadicals[f, x]` вҖ” integration of a **rational function
of `x` and square roots `(a x^2 + b x + c)^(m/2)` of one shared quadratic
argument** by a classical **Euler substitution**.  New module
`src/calculus/integrate_quadrad.c` (+ `.h`); wired into
`src/calculus/integrate.c` as a new cascade stage and a
`Method -> "QuadraticRadicals"` value.

Identification scans `f` for `Power[base, p/q]` whose `x`-dependent `base` is a
degree-2 polynomial; each must be a square root (`q == 2`) and all must be the
same radicand `rad = a x^2 + b x + c` (a cube root of a quadratic, a square root
of a cubic, or distinct radicands are rejected; a linear radical belongs to
`Integrate`LinearRadicals`).  The forward rewrite reuses
`poly_subst_radical_to_gen` (`rad^(p/2) -> y^p`); the substituted integrand is
confirmed rational in `{x, y}` before substituting.

To return **real-valued** antiderivatives, exactly **one** Euler substitution is
chosen (not all three, per design refinement) by the sign of the leading
coefficient `a` вҖ” and, when `a < 0`, of the discriminant `b^2 - 4 a c`:

* `a > 0` вҶ’ Euler I, `Sqrt[rad] = Sqrt[a] x + u`;
* `a < 0` and `b^2 - 4 a c > 0` вҶ’ Euler III, `Sqrt[rad] = (x - alpha) u`;
* `a` symbolic вҶ’ Euler I (best-effort `a > 0` branch).

For numeric `a < 0` a real radicand forces real roots, so Euler III subsumes the
`c > 0` (Euler II) case.  The chosen branch gives `x = X(u)`, `dx = X'(u) du` and
the radical image; the rationalised integrand `f dx /. {y -> ..., x -> X}`
re-enters the full `Integrate` and is back-substituted `u -> U(x)`.  As with
`Integrate`LinearRadicals`, the Euler substitution is an exact bijection, so the
result is accepted without a `Simplify[D[result, x] - f] === 0` gate.  A depth
guard (8) and per-call fresh symbols keep the recursion finite.

**Cascade placement:** after `LinearRadicals`, before `DerivativeDivides`.
Headline wins:
`Integrate[1/Sqrt[x^2 + 1], x]` вҶ’ `-Log[-x + Sqrt[1 + x^2]]`;
`Integrate[1/Sqrt[1 - x^2], x]` вҶ’ `-2 ArcTan[Sqrt[1 - x^2]/(-1 + x)]`;
`Integrate[1/Sqrt[a x^2 + 1], x]` вҶ’
`-Log[-Sqrt[a] x + Sqrt[1 + a x^2]]/Sqrt[a]` (symbolic leading coefficient).

Regression coverage in `tests/test_integrate_quadrad.c`
(`integrate_quadrad_tests`).  Docs: `docs/spec/builtins/calculus.md`
(`### Integrate`QuadraticRadicals`).

## Power: stop merging `(B^p)^q` for symbolic inner exponents (2026-06-06)

The power-of-power composition in `builtin_power` (`src/power.c`) folded
`(B^p)^q -> B^(p q)` whenever the inner base `B` was provably positive,
regardless of the inner exponent `p`. That is only sound when `p` is real
(so `B^p` is a positive real); for symbolic `p` the base may be complex and the
identity crosses a branch cut. As a result `(E^x)^y` collapsed to `E^(x y)` and
`(2^x)^y` to `2^(x y)`, diverging from Mathematica, which leaves both
unevaluated.

The composition now requires either an integer outer exponent (sound for any
base) or a positive base **and** a real (integer/bigint/rational/real/MPFR)
inner exponent вҖ” gated by a new `is_real_number` predicate. Numeric cases such
as `Sqrt[(1/54)^(2/3)]` and `(E^2)^3` are unaffected; `(E^x)^y`, `(2^x)^y` and
`Sqrt[2^a]` (symbolic `a`) now stay nested. `PowerExpand` remains the route to
the merged form (and as a side effect now applies to `(E^x)^y`, which it could
not see before). Regression coverage added to
`tests/test_power_corpus.c::test_power_of_power_positive_base` and
`tests/test_powerexpand.c`.

## PowerExpand (2026-06-06)

New builtin `PowerExpand` (`src/expand_power.c` / `.h`, registered in the new
`expand_power_init`, called from `core.c` right after `expand_init`). Attribute
`Protected`; symbol `SYM_PowerExpand` in `sym_names.{c,h}`; docstring in
`info.c`; reference page in `docs/spec/builtins/structural-manipulation.md`
(next to `Expand`).

- Distributes powers over products and collapses nested powers / logs / args:
  `(a b)^c -> a^c b^c`, `(a^b)^c -> a^(b c)`, `Log[a b] -> Log[a] + Log[b]`,
  `Log[a^b] -> b Log[a]`, `Arg[a b] -> Arg[a] + Arg[b]`. Since `Sqrt[x]` is
  `Power[x, 1/2]` and `Log[1/z]` is `Log[z^-1]`, these subsume
  `Sqrt[x y] -> Sqrt[x] Sqrt[y]`, `Sqrt[z^2] -> z`, `Log[1/z] -> -Log[z]`.
- Rules apply **top-down to a fixed point** (mirroring `ReplaceRepeated`), so
  `Log[(a b)^c]` collapses to `c (Log[a] + Log[b])` rather than expanding the
  inner power first. The branch-correction terms are kept inert (not
  reprocessed), which both is correct and avoids an infinite recursion on the
  `1/(2 Pi) = Power[Times[2,Pi],-1]` factor inside them.
- Inverse-function collapse `f^-1[f[x]] -> x` for the inverse-trig /
  inverse-hyperbolic pairs.
- **`Assumptions` option** (`Automatic` | `True` | `assum`):
  - `Automatic` (default): the textbook transforms, correct for positive-real
    bases / integer exponents; branch cuts ignored.
  - `True`: emits the universally-correct formula with a branch-correction term
    `Floor[1/2 - Sum Arg[zi]/(2 Pi)]` (products) or
    `Floor[1/2 - Im[x Log a]/(2 Pi)]` (nested powers), assembled from
    `Floor`/`Arg`/`Im`/`E`/`I`/`Pi`, plus the universal `ArcTan[Tan[x]]`
    template (with `Re`/`UnitStep`).
  - `assum`: builds the `True` formula then refines it under the assumptions вҖ”
    `Arg`/`Im` of known-sign reals resolve to `0`/`Pi`, `Floor` is evaluated
    over assumption-bounded intervals, and `E^(n I Pi)` collapses to `(-1)^n`.
    Reuses the `src/simp/` assumption engine (`assume_ctx_from_expr`,
    `assume_known_real`). Reproduces the documented examples
    (`Sqrt[z^2]` under `z<0` -> `-z`; `(z^p)^(1/p)` under `0<p<1` -> `z`;
    `Log[(a b)^c]` under `3<a<5 && -2<b<-1 && 7<c<9` ->
    `-8 I Pi + c (Log[a] + Log[b])`).
- `PowerExpand[expr, {x1, ...}]` expands only subexpressions mentioning a listed
  variable; threads over `List`, equations, inequalities, and logic functions;
  `PowerExpand[]` emits `PowerExpand::argt` and stays unevaluated.
- Known limitations (best-effort beyond the documented cases): in `True` mode
  the nested-power correction keeps `Im[...]` rather than rewriting it to
  `Arg[...]`; and the interval `Floor` reducer can over-reduce at half-open
  `Arg` branch boundaries under weak assumptions.
- Extensive tests in `tests/test_powerexpand.c` (`powerexpand_tests`):
  Automatic transforms, variable restriction, threading, `True`-mode formulas,
  and the assumption-refinement cases. Verified leak-free against a `Sin[1.0]`
  baseline вҖ” no `expand_power` frames in any leak stack (the only delta is a
  pre-existing `builtin_inequality`/`builtin_and` leak, recorded in
  `MEMORY_LEAKS.md`, triggered merely by evaluating the inequality assumptions).

### PowerExpand: keep roots real by folding negative coefficients (2026-06-06)

When expanding a product raised to a **non-integer (root) exponent**,
`split_power_product` (`src/expand_power.c`) now folds a negative numeric
coefficient into a `Plus` factor instead of emitting a spurious `I`:
`Sqrt[-4 Dt[u]^2 (-1 + u)]` -> `2 Dt[u] Sqrt[1 - u]` rather than
`2 I Dt[u] Sqrt[-1 + u]`. The rewrite negates the coefficient (`-k -> k`) and
one `Plus` factor (`p -> -p`), distributing the `-1` through the sum so it
displays as `1 - u`; it is value-preserving since `(-1)^c p^c = (-p)^c`. It is
gated on (a) a numeric non-integer exponent (`pe_exp_nonint`), and (b) the
presence of *both* a negative numeric factor and a `Plus` factor, so integer
exponents (`(-2 (1 + x))^2 -> 4 (1 + x)^2`) and genuinely-imaginary results
with no `Plus` to absorb the sign (`Sqrt[-4 Dt[u]^2] -> 2 I Dt[u]`) are left
untouched. New regression cases in `tests/test_powerexpand.c`.

## Eliminate: general transcendental (trig / exp / log) algebraisation (2026-06-06)

`Eliminate` (`src/poly/eliminate.c`) gained a general transcendental
algebraisation pre-pass, the sibling of the existing radical pass, so an elim
variable can now be eliminated even when it sits inside circular/hyperbolic
trig, exponential (`b^x`), or logarithmic functions in **multiple** places вҖ”
the case the single-layer inverse-function pre-pass could not handle (it would
bail with `Eliminate::nlin`).

Algorithm: each equation is expanded so every kernel lands on an *atomic*
argument вҖ” `TrigExpand` for trig (multiple/sum angles), `b^(p+q) -> b^p b^q` /
`b^(k m) -> (b^m)^k` for exponentials, `Log[a b] -> Log[a]+Log[b]` /
`Log[a^n] -> n Log[a]` for logarithms вҖ” so e.g. `Sin[x]` and `Sin[3x]`, or
`Exp[x]` and `Exp[2x]`, share one auxiliary. Each `Sin[Оё]`/`Cos[Оё]` (resp.
`Sinh`/`Cosh`) is replaced by a fresh aux pair `$tsN$`,`$tcN$` carrying the
Pythagorean constraint `s^2+c^2==1` (circular) / `c^2-s^2==1` (hyperbolic);
each `b^Оё`/`Log[Оё]` becomes a single algebraically-free aux `$teN$`/`$tlN$`.
The auxes join the elimination block, the GrГ¶bner engine runs unchanged, and
`Eliminate::ifun` is emitted (branch/sign info may be lost). Two conservative
soundness gates bail to `Eliminate::nlin` rather than emit a wrong relation:
(A) an elim variable shared across two distinct kernels (`Sin[x]` & `Sin[x*y]`,
or `Sin[x]` & `Exp[x]`), and (B) an elim variable still present as a bare
polynomial atom after substitution (`x` alongside `Sin[x]`).

This makes the integration-by-substitution discovery pattern work, e.g.
`Eliminate[{Dt[y] == Cos[x] Sqrt[1-Sin[x]] Dt[x], u == Sin[x], Dt[u] == Cos[x] Dt[x]}, {x, Dt[x]}]`
returns `Dt[y]^2 == (1-u) Dt[u]^2`, and
`Eliminate[{Dt[y] == Exp[x]/(1+Exp[x]) Dt[x], u == Exp[x], Dt[u] == Exp[x] Dt[x]}, {x, Dt[x]}]`
returns `Dt[u] == (1+u) Dt[y]`. New tests in `tests/test_eliminate.c` cover
trig (Pythagorean, double-angle, tangent, hyperbolic), exp (double-angle,
general base, integration substitution), log (power, squared), and the
soundness bails; all 41 cases pass leak-free under `leaks`.

Also fixed a pre-existing leak in `TrigExpand` (`src/simp/trigsimp.c`): the
Equal/List/logic-threading branch returned without freeing its temporary
`new_args` buffer (16 bytes per threaded call), surfaced by the new pass's
heavy use of `TrigExpand` over equations.

## Integration of undefined (unknown) functions вҖ” Roach В§1.7 (2026-06-06)

New module `src/calculus/integrate_unknown.c` (+ `.h`) and builtin
`Integrate`Undefined[f, x]`, wired as the first stage of the `Integrate`
dispatch cascade (gated to run only when the integrand contains an
undefined-function derivative such as `f'[x]`) and selectable strictly via
`Method -> "Undefined"`. Implements the algorithm of Kelly Roach, *Indefinite
and Definite Integration* (1992), В§1.7 ("Undefined Functions").

Each undefined function value `u[g]` and its derivative tower
`u'[g], u''[g], вҖҰ` is treated as a differential-field generator. The integrand
is reduced by an integration-by-parts step on the highest-order generator `A`
present: writing `f = CВ·A + D` (linear in `A`) and `B` for `A`'s antiderivative
atom, `вҲ«f dx = W вҲ’ вҲ«(dW/dx вҲ’ CВ·A вҲ’ D) dx` where `W = вҲ«(C/g') dB` and
`g' = D[arg, x]` is the chain factor. The inner `вҲ«вҖҰ dB` step is delegated to the
existing rational integrator over a substituted generator symbol, which means a
single mechanism subsumes Roach's polynomial, fraction, **and** log parts:

```
Integrate[f'[x], x]                                  -> f[x]
Integrate[x f'[x] + f[x], x]                         -> x f[x]
Integrate[f'[x] g[x] + f[x] g'[x], x]                -> f[x] g[x]
Integrate[f''[x] g[x] + 2 f'[x] g'[x] + f[x] g''[x], x] -> f[x] g'[x] + f'[x] g[x]
Integrate[f'[x]/f[x], x]                             -> Log[f[x]]
Integrate[f'[x]/f[x]^2, x]                           -> -1/f[x]
Integrate[(f'[x] g'[x] - f[x] g''[x])/(f[x]^2 + g'[x]^2), x] -> -ArcTan[g'[x]/f[x]]
Integrate[2 x f'[x^2], x]                            -> f[x^2]   (* chain rule *)
```

Robustness: the by-parts residual is handed back to the integrator as a *whole*
(sums are not split term-by-term, since e.g. `f'[x]g'[x] + f[x]g''[x]` is an
exact derivative whose pieces individually have no closed form), and a
canonical-form cycle guard makes genuinely non-elementary inputs like
`Integrate[f'[x] g'[x], x]` terminate unevaluated rather than looping.
Canonicalisation throughout uses `Cancel[Together[Expand[вҖҰ]]]` so that
mathematically-zero residuals collapse to literal `0`. Memory-clean under
valgrind (no leak stacks in the module).

A logarithm of an unknown-function expression, `Log[eta]`, is additionally
recognised as a transcendental generator (Roach В§1.6/В§1.7) and reduced by parts,
`Integrate[C L + D, x] = G L вҲ’ Integrate[G L' вҲ’ D, x]` with `G = Integrate[C, x]`
and `L' = eta'/eta`. A self-referential resolution closes perfect-power cases
where the residual is a constant multiple of the original integrand
(`Integrate[Iorig] = bdy вҲ’ k Integrate[Iorig] вҮ’ bdy/(1+k)`):

```
Integrate[(f[x] - x f[x] + f[x] Log[x f[x]] + x f'[x])/f[x], x]
                                            -> -x^2/2 + x Log[x f[x]]
Integrate[Log[f[x]] f'[x]/f[x], x]          -> Log[f[x]]^2/2
```

New unit + stress suite `tests/test_integrate_unknown.c` (`integrate_unknown_tests`):
exact-form assertions, a randomized `D`-round-trip harness, negative cases, a
cycle-termination check, and `Log[eta]`-generator cases. Known limitations:
transcendental generators other than `Log` (`ArcTan[eta]`, `Exp[eta]`, вҖҰ) and
nested unknown arguments `f[g[x]]` are deferred.

## Integrate of InterpolatingFunction objects (2026-06-06)

`Integrate[InterpolatingFunction[...][x], x]` now returns the antiderivative as
a fresh applied `InterpolatingFunction[...][x]`, closing the asymmetry where
`D` differentiated such objects but `Integrate` left them unevaluated (matching

produces (integrating the sampled derivative recovers the lower order). Only the

## Parser: leading minus binds looser than `^` on numeric literals (2026-06-06)

Fixed a precedence bug where a leading minus on a *number* was folded into the
literal before `^` was applied: `-3^(2/3)` parsed as `Power[-3, Rational[2,3]]`

## Floor/Ceiling/Round/IntegerPart/FractionalPart on exact real numerics (2026-06-06)

`Round`, `Floor`, `Ceiling`, `IntegerPart`, and `FractionalPart` now resolve
**exact real numeric arguments** that previously fell through unevaluated вҖ”
named constants and their products/powers such as `10000000 3^(2/3)` or
`25000000000000000000 Pi`:

```mathematica
In[1]:= Round[10000000*3^(2/3)]
Out[1]= 20800838          (* was: Round[10000000 3^(2/3)] *)

In[2]:= Round[25000000000000000000 Pi]
Out[2]= 78539816339744830962
```

**Implementation** (`src/piecewise.c`, MPFR-only). When a leaf branch
(`Integer`/`BigInt`/`Real`/`MPFR`/`Rational`/`Complex`) and the symbolic
simplifications (sign extraction, idempotency) all decline, `do_piecewise_1`
calls the new `do_piecewise_numeric_exact`. It `numericalize()`s the argument
to MPFR at 256 bits, then doubles (to a 65536-bit / ~19,700-digit cap),
applying the rounding op at each precision and accepting the result only once
**two successive precisions agree on the integer**. This interval-style
certification means the answer is never wrong: a value too close to a rounding
boundary to certify, or one that does not collapse to a pure real number
(still symbolic, or `Complex[...]`), is left unevaluated rather than guessed.
`FractionalPart` is kept exact as `x - IntegerPart[x]` (e.g.
`FractionalPart[10000000 3^(2/3)]` вҶ’ `-20800838 + 10000000 3^(2/3)`), matching
Mathematica. Verified leak-free (`leaks --atExit`). New cases added to
`tests/test_piecewise.c`.

No new builtins or attributes; behaviour of the existing five heads extended.

## Fit and DesignMatrix: linear least-squares regression (2026-06-06)

New module `src/fit.c` adds the `Fit` and `DesignMatrix` builtins (registered in

## CRC table: branch-correct 1/Sqrt[1 Вұ Cos/Sin[x]] (Formulas 398вҖ“401, 2026-06-05)

Replaced the four reciprocal-square-root antiderivatives in
`src/internal/CRCMathTablesIntegrals.m` with branch-correct forms. The classical
`Sqrt[2] Log[Tan[x/4 (+ shift)]]` primitives are real on only every *other*
inter-pole interval of the integrand (elsewhere the `Tan` argument turns
negative and the `Log` goes complex), and even where real the sign is wrong вҖ”
the integrand carries an absolute value, e.g.
`1/Sqrt[1 - Cos[x]] = 1/(Sqrt[2] |Sin[x/2]|)`. These integrands have poles, so
no continuous antiderivative spans one; correctness is per pole-bounded
interval.

- **Surfaced by the corpus** as the lone DIFF-NONZERO regression once the
  Optional-dispatch fix made `1/Sqrt[1 - Sin[x]]` reach Formula 400: it returned
  `Sqrt[2] Log[Tan[x/4 - Pi/8]]`, correct at the old sample point xвүҲ2.6 but
  complex/wrong at xвүҲ0.7. (The three siblings shared the latent bug but happened
  to sample inside their good intervals.)
- **Fix.** Keep the integrand's radical literally; the factor
  `Sqrt[1 - Cos[x]]/(Sqrt[2] Sin[x/2]) = sign(Sin[x/2])` restores the sign the
  `ArcTanh` primitive of the *signed* reciprocal drops:
  - `1/Sqrt[1 - Cos[x]]` вҶ’ `-ArcTanh[Cos[x/2]] Csc[x/2] Sqrt[1 - Cos[x]]`
  - `1/Sqrt[1 + Cos[x]]` вҶ’ `ArcTanh[Sin[x/2]] Sec[x/2] Sqrt[1 + Cos[x]]`
  - `1/Sqrt[1 - Sin[x]]` вҶ’ `-ArcTanh[Cos[x/2 - Pi/4]] Csc[x/2 - Pi/4] Sqrt[1 - Sin[x]]`
  - `1/Sqrt[1 + Sin[x]]` вҶ’ `ArcTanh[Sin[x/2 - Pi/4]] Sec[x/2 - Pi/4] Sqrt[1 + Sin[x]]`
  This mirrors the Phase C/D radical-literal treatment already applied to the
  non-reciprocal Formulas 394вҖ“397.
- Pinned by `test_crctable_reciprocal_sqrt_branch` in
  `tests/test_integrate_dispatch.c` (numeric `D[result, x] вҲ’ integrand` at points
  on the previously-broken interval for all four).

  report TIMEOUT rather than DIFF-ZERO. Integration and the derivative are
  instant; the cost is entirely in the corpus's symbolic verifier

## Pattern dispatch: Optional-headed first-arg slots (2026-06-05)

Fixed a В§3.5 DownValue dispatch-filter bug that silently skipped any rule whose
first argument is a function-call pattern with an `Optional` argument
(`src/symtab.c`, `pattern_arg_head_canon`). Such a slot can collapse to its base
via the Optional default вҖ” e.g. `Power[b_, n_.]` matches a bare `b` as `b^1`,
`Times[a_., x_]` matches `x` вҖ” so a matching input need **not** be headed by the
pattern's head. The filter was caching the head anyway (e.g. `"Power"`) and
rejecting any input whose first-arg head differed (e.g. a bare symbol, `"Symbol"`)
**before** the matcher ran, even though `MatchQ`/`ReplaceAll` (which bypass the
filter) matched fine.

- **User-visible symptom (CRC integral table).** `Integrate[a x, x,
  Method -> "CRCTable"]` returned unevaluated because the monomial rule
  `IntegrateTable[x_^n_., x_] /; FreeQ[n, x] && n =!= -1 := x^(n+1)/(n+1)`
  never reached the matcher for the `IntegrateTable[x, x]` reduction
  (`x` вүЎ `x^1`). `a x^2` and `a x^(-1)` worked because their exponents are
  explicit Powers. Now `Integrate[a x, x, Method -> "CRCTable"]` вҶ’ `1/2 a x^2`.
- **Fix.** `pattern_arg_head_canon` returns `NULL` (wildcard, scan-always) for a
  plain function-call pattern when any of its arguments is an `Optional` (detected
  via a new `pattern_arg_is_optional` helper that also sees through a
  `Pattern[name, вҖҰ]` wrapper). The dispatch filter is a pure optimization, so
  widening to wildcard only ever costs an extra `match()` call вҖ” never
  correctness. Other slot shapes are unaffected, so head specialization is
  retained everywhere it is sound.
- Pinned by `tests/test_condition_downvalue.c::test_crc_table_formula_fires`
  (which already asserted the `a x` case) plus the `crc_corpus`, `match`,
  `match_extensive`, `patterns`, `symtab`, and `integrate_dispatch` suites.

## Interpolation: O(1) re-evaluation + cached grid (2026-06-05)

Two-part performance fix for `InterpolatingFunction` application
(`src/interp.c`). Per evaluated point the system did two O(n)-in-the-node-count
passes over the n-point table that have nothing to do with the interpolation
maths: (1) the evaluator re-evaluated the object's entire `table` argument on
every application, and (2) `interp_apply` rebuilt the parsed tensor grid from
scratch. Together these dominated per-point cost and made it scale with `n`.
At n=1600 (1000 off-node points) the fix takes per-point eval from вүҲ1.93 ms to
вүҲ0.38 ms (**вүҲ5Г—**), with the speedup growing with `n`.

- **`HoldAll` on `InterpolatingFunction`** (the larger win, вүҲ3Г— alone). The
  object's `domain`/`table` arguments are constant, already-evaluated numeric
  data, but the symbol had no Hold attribute, so each `ifun[x]` made the
  evaluator re-evaluate the whole table вҖ” an O(n) deep re-traversal every call.
  Holding them keeps the object a true normal form and makes evaluating it O(1).
  The application's own argument `x` is still evaluated normally (governed by the
  object head, not by `InterpolatingFunction`).
- **Single-entry grid cache** of the most-recently-built `IFun`, keyed on the
  **structural identity** (`expr_eq`) of `(domain, table, periodic)`.
  `interp_eval_double` borrows the grid from `grid_cache_get` instead of
  building/freeing one per call (which included an O(nВІ) `grid_insert` sort); the
  cache owns deep copies of the keys (so `entryAt` stays valid) and frees the
  previous entry on every rebuild. Correct by construction: redefining the data
  yields a structurally different `table`, which misses and rebuilds; the single
  entry serves the build-once / evaluate-many loop (`Table[f[x], {x, вҖҰ}]`), and
  interleaving distinct objects rebuilds each time (no stale values).
- Note: the bracketing-interval search (`bracket_interval`) was **already** a
  binary search; the earlier review's "O(n) window scan" diagnosis was
  inaccurate вҖ” the real costs were the table re-evaluation and the grid rebuild.
- Machine-precision grid cache only; the MPFR kernels (`interp_mpfr.c`) are
  unchanged. No behavioural change: every value, the `dmval` extrapolation
  warning, and the full `tests/test_interp.c` suite are byte-identical
  before/after (also verified against the `match`/`replace`/`patterns`/`eval`
  suites for the attribute change). Leak-clean under `leaks --atExit` (0 leaks).
  Strict C99 `-Wall -Wextra` clean.

## InterpolatingFunction (2026-06-04)

New builtin object `InterpolatingFunction` (`src/interp.c` / `src/interp.h`,
registered in `interp_init`, called from `core.c`). Attribute `Protected`;
symbol `SYM_InterpolatingFunction` in `sym_names.{c,h}`; docstring in
`info.c`; reference page in `docs/spec/builtins/functional-programming.md`.

- `InterpolatingFunction[domain, table]` represents an approximate function over
  tabulated data; the object is a normal form that persists unevaluated, like
  `Function[...]`. `domain = {{x1min,x1max}, ...}` carries one interval per
  dimension; `table = {{coord, val}, ...}` (scalar `coord` for 1-D, an
  `{x1,...,xm}` list for m-D).
- `InterpolatingFunction[...][x1,...,xm]` evaluates the approximate function by
  **tensor-product** piecewise-polynomial interpolation of default order 3
  (cubic) in each dimension, with a sliding window centred on the bracketing
  interval and a Newton divided-difference polynomial. Order drops to
  `min(3, n_k - 1)` per dimension for small grids. The windowing reproduces
  Mathematica's default piecewise interpolant (verified against `wolframscript`
  for non-polynomial 1-D and 2-D data, values agreeing to full double
  precision; e.g. hill table `ifun[2.5] вҶ’ 3.6875`).
- **Multi-dimensional**: data must fill the full Cartesian product of the
  per-dimension grids; points may be supplied in any order (sorted internally).
- **Derivatives**: `Derivative[d1,...,dm][InterpolatingFunction[...]]` (so
  `ifun'`, `D[ifun[x],x]`, `D[f[x,y],{x,2}]`) returns another
  `InterpolatingFunction` carrying the accumulated orders вҖ” like Mathematica вҖ”
  which evaluates the mixed partial of the windowed interpolant; orders are
  additive and an order beyond the local degree gives `0`. Implemented via
  `interp_make_derivative` (dispatched from `eval.c`'s Derivative branch) plus a
  generic-chain-rule fall-through for applied `InterpolatingFunction` objects in
  `src/calculus/deriv.c`. Verified vs `wolframscript`: `g'[2.5]=1.04166вҖҰ`,
  `g''=-1.5`, `g'''=-1`, `g''''=0`, 2-D `D[1,0]=3`, `D[0,1]=18.75`, `D[1,1]=0`.
- Application is dispatched in `eval.c` alongside `Function`/`Composition`
  (function-headed expression branch), via `interp_apply`.
- Exact node-tuple hits return the stored value exactly (`ifun[2] вҶ’ 3`,
  2-D `f[2,3] вҶ’ 31`); other queries return a machine real. Out-of-domain
  arguments are extrapolated with an `InterpolatingFunction::dmval` warning.
  Symbolic, wrong-arity (вү  `m`), or malformed objects (fewer than two grid
  points, incomplete tensor grid, non-numeric data) are left unevaluated.
- Standard output shows only the domain, abbreviating the rest as `<>`
  (`InterpolatingFunction[{{0, 5}}, <>]`); `FullForm` shows the full structure.
- 48 unit tests in `tests/test_interp.c` (linear/quadratic/cubic, exact cubic
  reproduction, reference windows, exact nodes, extrapolation, 1-D derivatives,
  derivative-as-object, `D` chain rule, 2-D interpolation + mixed partials,
  unsorted input, unevaluated and malformed cases, printing). Leak-clean (the
  only `leaks` hit is the pre-existing `system_constants_init` global). Strict
  C99 `-Wall -Wextra` clean.

## Interpolation: Spline/Hermite methods, derivative data, arbitrary precision (2026-06-04)

Major extension of `Interpolation` toward Mathematica fidelity (`src/interp.c`
rewritten; new `src/interp_mpfr.c` for the MPFR kernels). Standard,
mathematically-correct interpolants вҖ” pass through every data point and honour
every supplied derivative; interior values need not match Wolfram's proprietary
B-spline scheme.

- **`Method -> "Spline"`**: natural cubic spline (CВІ, second derivative 0 at the
  ends; linear for two points), tensor-product over the full grid via a
  tridiagonal Thomas solve.
- **`Method -> "Hermite"`**: piecewise cubic Hermite with node slopes estimated by
  3-point finite differences (reproduces quadratics exactly).
- **Derivative-supplied data** `{{{x,...}, f, df, ddf, ...}, ...}` (1-D and n-D):
  interpolated by tensor-product Hermite of per-dimension order `k = max(K,1)`
  (`K` = highest supplied derivative order). A full mixed-derivative tensor is
  built from the supplied gradient/Hessian/вҖҰ tensors; mixed partials absent from
  the data (e.g. `f_xy` from gradient-only data) are estimated by central finite
  differences across the grid. Evaluated by a direct tensor-Hermite-basis sum
  over the bracketing cell. Reproduces a cubic from `{f,f'}` and a quintic from
  `{f,f',f''}` exactly; separable polynomials in m-D reproduced exactly.
  Derivatives of the interpolant (`D`, `Derivative`, `f'`) compose for every form.
- **Arbitrary precision**: when the data, domain or evaluation point carry MPFR
  precision the MPFR kernels run at the data's working precision and an
  `EXPR_MPFR` is returned (otherwise machine `double`/`EXPR_REAL`). Precision is
  detected with `common_scan_inexact` / `numeric_expr_is_mpfr`; values coerced via
  `get_approx_mpfr`. Verified to agree with the machine path (e.g. spline interior
  value to 30+ digits, `Precision` = 30.103).
- The method/order annotations are carried on the `InterpolatingFunction` object
  (`[domain, table, ders, orders, method]`) and preserved through differentiation.
- The object representation is unchanged for value-only data; derivative-supplied
  data is stored as longer table entries `{coord, val, df, ...}`.
- 35 new unit tests in `tests/test_interp.c` (111 total): Spline (node
  reproduction, n=2 linear, linear-data, derivative), Hermite method (quadratic
  reproduction, derivative), 1-D supplied `{f,f'}`/`{f,f',f''}` (cubic/quintic
  exact, derivative), n-D gradient (separable cubic + partials) and Hessian
  (biquintic) data, and arbitrary-precision (MPFR result type + value) across all
  methods/forms. Leak-clean under `leaks` (only the pre-existing
  `system_constants_init` global). Both `interp.c` and `interp_mpfr.c` compile
  under strict C99 with and without `USE_MPFR`.

## MinimalPolynomial (2026-06-04)

New builtin `MinimalPolynomial` (`src/poly/minpoly.c`, registered in the new

symbol in `sym_names.c`; docstring in `info.c`; reference page in
`docs/spec/builtins/arithmetic-and-algebra.md`.

- `MinimalPolynomial[s, x]` gives the lowest-degree polynomial in `x` with
  integer coefficients, positive leading coefficient and content `1`, having the
  algebraic number `s` as a root (e.g. `MinimalPolynomial[Sqrt[2] + Sqrt[3], x]`
  вҶ’ `1 - 10 x^2 + x^4`).
- `MinimalPolynomial[s]` returns the minimal polynomial as a pure function
  (`1 - 10 #1^2 + #1^4 &`).
- `MinimalPolynomial[s, x, Extension -> a]` gives the characteristic polynomial
  of `s` in `Q(a)` over `Q(a)` вҖ” `m_s(x)^([Q(a):Q]/[Q(s):Q])` when `s вҲҲ Q(a)`,
  else left unevaluated (`MinimalPolynomial::ext`).
- `s` may be built from integers/rationals, radicals (`Sqrt`, `Power[_, p/q]`),
  the imaginary unit, roots of unity (`Power[E, I Pi r]`), `Root[f &, k]`
  objects, and `Plus`/`Times`/`Power`. The first argument threads over lists.
- **Algorithm (exact resultant elimination):** each algebraic atom is replaced
  by a fresh auxiliary variable with a polynomial defining relation вҖ” negative
  powers introduce a reciprocal variable (`D w вҲ’ 1`) so every relation stays
  polynomial and no `Together`/`Numerator` of the system is needed. The
  auxiliaries are eliminated from `x вҲ’ s` by repeated `internal_resultant`
  (highest index first; introduction order guarantees each is present when
  eliminated). The univariate result is denominator-cleared
  (`Together`вҶ’`Numerator`), made primitive over `Z`, and factored
  (`internal_factor`); the unique irreducible factor vanishing at `s` is chosen
  by a high-precision (`80`-digit MPFR) numeric test, then returned primitive
  with positive leading coefficient (`zupoly_primitive_part`).
- The `Extension` form reduces to the tower law on degrees from the same core;
  membership `s вҲҲ Q(a)` is verified via the primitive-element degree
  `[Q(a, s):Q] == [Q(a):Q]` (max minimal-polynomial degree of `a + r s` over
  `r вҲҲ {1, 2}`).
- Non-algebraic input (`Pi`, `Log[2]`, вҖҰ) is left unevaluated.
- Tests: `tests/test_minimalpolynomial.c` (radicals, nested radicals, complex /
  Gaussian numbers and roots of unity, `Root[]` objects, rationals/integers,
  listable threading, pure-function form, the `Extension` form including the
  non-membership case, and unevaluated non-algebraic input). Valgrind-clean вҖ”
  byte-for-byte identical leak totals to a baseline test that never calls it, so
  no leaks attributable to the new code.

## FindIntegerNullVector (2026-06-04)

  `ZeroTest вҶ’ Automatic | f`.

- Diagnostics: `::norel`, `::lgrelb`, `::rnfb`, `::rnfu`, `::ztest1`.
- Relations are canonical only up to sign and (complex) a Gaussian-unit

## LatticeReduce (2026-06-04)

## Ratios (2026-06-01)

New builtin `Ratios` (`src/list.c`), the multiplicative analog of
`Differences` and inverse of `FoldList[Times, ...]`. Attribute `Protected`.
Symbol registered in `sym_names.c`; docstring in `info.c`.

- `Ratios[list]` gives the successive ratios `{l[[2]]/l[[1]], l[[3]]/l[[2]],
  ...}`, of length `s - 1`. Ratios divides successive elements by preceding
  ones, so `Ratios[{x1, x2}]` gives `{x2/x1}`.
- `Ratios[list, n]` applies the ratio operator `n` times, giving length
  `s - n`. `n` must be a non-negative integer (`n = 0` returns `list`
  unchanged); e.g. `Ratios[{a, b, c, d, e}, 2]` is
  `{(a c)/b^2, (b d)/c^2, (c e)/d^2}`.
- `Ratios[list, {n1, n2, ...}]` gives the successive `nk`-th ratios at level
  `k` of a nested list, and is equivalent to
  `Ratios[Ratios[list, n1], {0, n2, ...}]`. Each `nk` must be a non-negative
  integer.
- Division threads element-wise over sublists via the `Listable` `Power`/`Times`
  that `Divide` rewrites to, so for a matrix `m`, `Ratios[m]`
  (= `Ratios[m, 1]` = `Ratios[m, {1, 0}]`) takes ratios of successive rows
  within each column, while `Ratios[m, {0, 1}]` takes ratios of columns within
  each row.
- The head of the input is preserved; a list of length `<= n` reductions yields
  the empty list. Works on machine integers, GMP arbitrary-precision integers
  (exact `Rational` results), machine-precision doubles, and symbolic
  expressions. First ratios are constant for a geometric sequence, and
  `FoldList[Times, x, Ratios[list]]` reconstructs the original list.

Implemented by mirroring the `Differences` helpers: `ratio_divide`,
`ratio_once`, `ratio_n`, and `ratio_levels` build `Divide[hi, lo]` and let the
evaluator reduce it. 28 unit tests in `tests/test_ratios.c`; valgrind/`leaks`
clean.

## GroebnerBasis: weight-matrix MonomialOrder + GroebnerWalk method (2026-06-04)

`GroebnerBasis` now accepts a custom term order given by an integer **weight
matrix** and a new `Method -> "GroebnerWalk"` engine. Both extend the existing
Buchberger subsystem (`src/poly/groebner.{c,h}`, `groebnerbasis.c`) and share
the same weight-matrix machinery.

**Weight-matrix `MonomialOrder`.** `MonomialOrder -> {{...}, ...}` (a
rectangular integer matrix with one column per main variable) ranks monomials
by the lexicographic comparison of the weight vectors `MВ·a` vs `MВ·b`. The
standard orders are special cases (the identity matrix is `Lexicographic`,
`{{1,1,1},{0,0,-1},{0,-1,0}}` is `DegreeReverseLexicographic`).

- New `GB_ORDER_MATRIX` order plus a borrowed, caller-owned `GBWeightMatrix`
  carried on each `GBPoly`. Comparison (`cmp_matrix_desc`) accumulates row
  weights in `int64` with `__builtin_*_overflow` guards, falling back to exact
  `mpz` only on overflow, so the hot compare path stays fast.
- `gb_wmat_validate` is a hard gate: it requires full column rank (via exact
  rational Gaussian elimination вҖ” otherwise distinct monomials could tie and
  `gb_poly_normalize` would silently merge them) **and** a positive first
  non-zero entry in every column (well-foundedness, so Buchberger terminates).
  Invalid matrices, or matrices combined with elimination/parameter variables,
  emit `GroebnerBasis::nimpl` and fall back to `Lexicographic`.

**`Method -> "GroebnerWalk"`** (new file `src/poly/groebnerwalk.c`,
`gb_groebner_walk`). Implements the CollartвҖ“KalkbrenerвҖ“Mall walk: compute a
cheap `DegreeReverseLexicographic` basis, then convert it to the target order
(Lexicographic or a weight matrix) by walking the weight path
`w(t) = (1-t)В·(1,вҖҰ,1) + tВ·Пғ` through the GrГ¶bner fan. At each cone wall it
takes the initial-form ideal's basis under the target order and lifts it back
via a new quotient-tracking division `gb_divmod`; `gb_initial_form` extracts
the `w`-initial forms. The path parameter is exact (`mpq`), so simultaneous
wall crossings are handled without perturbation. By reduced-GB uniqueness the
output is identical to a direct Buchberger run, so any safety-guard trip
(overflow, a degenerate lift, or a step-count blow-up) falls back to
`gb_buchberger` with no loss of correctness. The walk is honoured only without
elimination/parameter variables; otherwise Buchberger runs directly. The
shared finalize tail (interreduce вҶ’ fixed point вҶ’ primitive-over-Z вҶ’ sort) was
factored out of `gb_buchberger` into `gb_finalize_basis`.

Docstring (`info.c`) and the `GroebnerBasis` reference
(`docs/spec/builtins/structural-manipulation.md`) updated; `Method ->
"GroebnerWalk"` and explicit weight matrices removed from the "deferred" list.
New differential tests in `tests/test_groebner.c` (identity matrix вүЎ
Lexicographic, grevlex matrix вүЎ DegreeReverseLexicographic, walk вүЎ Buchberger
for lex and matrix targets on the unit-sphere/cyclic system, single-variable
and `вҹЁ1вҹ©` short-circuits, and the rank-deficient-matrix fallback). `leaks`-clean.

## Simplify: combine single-radical fraction sums over a common denominator

Extended `Simplify`'s algebraic-top `Together` fast path
(`builtin_simplify`, `src/simp/simp_builtins.c`) so it also fires for a sum of
fractions over a *single* polynomial-radicand radical, not just multi-generator
algebraic-number towers. Previously the path was gated on
`extension_autodetect` returning a tower with `n >= 2`, which is NULL for a
radical whose base is a polynomial (e.g. `(1 - b x)^(1/3)`, since the QAExt
machinery only represents integer-base minimal polynomials). Such sums fell
through to `simp_bottomup`, which simplifies each `Plus` child separately and
never combines them over a common denominator вҖ” so e.g.
`D[Integrate[(1 - b x)^(1/3)/x, x], x] // Simplify` returned a four-term mess
in ~1.5 s instead of recovering the integrand `(1 - b x)^(1/3)/x`.

Now, when the autodetect tower is single-generator (or NULL), the fast path
runs **plain** `Together` (which collapses the single poly-radical via its
Phase E poly-radical reduction in `builtin_together`); the multi-generator
case still uses `Together[expr, Extension -> Automatic]`. The combined fraction
is kept only when its `SimplifyCount` is strictly below the input's, so there
is no regression when `Together` is a no-op or worse. On the example above
`Simplify` now returns `(1 - b x)^(1/3)/x` in ~0.02 s (~68Г— faster) and the
differentiate-back round-trip recovers the integrand exactly.

The outer gate switched from `has_non_integer_power` to the new
`simp_has_rational_root` predicate (`src/simp/simp_util.c`, exposed in
`simp_internal.h`; it is the former static `subtree_has_root`, generalised and
renamed) вҖ” a *genuine* `Power[_, Rational[_, q>=2]]` radical вҖ” so a
symbolic-exponent power such as `a^x` is never fed to `Together` (which can
hang on symbolic exponentials). All existing integer-base multi-generator cases
still satisfy this gate.

The user-facing test `test_simplify_user_real_contagion_factored`
(`tests/test_simplify.c`), which previously pinned the expanded denominator as a
placeholder "until factoring in a rational-power generator lands", was updated
to the now-reachable partially factored form `1/((1.0 + x) (Sqrt[x] + x))`
(SimplifyCount 17 vs the expanded 18; LeafCount 15 vs 17) вҖ” a step toward
Mathematica's fully factored output.
