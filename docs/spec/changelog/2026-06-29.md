# Changelog: week of 2026-06-29 (Mon) – 2026-07-05 (Sun)

Feature additions and fixes recorded during this week.

## Fix: solid fill color for ParametricPlot/ParametricPlot3D surfaces (2026-06-30)

The two-iterator forms of `ParametricPlot` and `ParametricPlot3D` previously
hardcoded `Opacity[0.5]` before every polygon strip, making all surface fills
50% transparent regardless of `PlotStyle`. The hardcoded opacity is removed;
surfaces are now **solid** by default. Transparency is now opt-in via
`PlotStyle -> {color, Opacity[a]}`, which works because both the 2D and 3D
renderers process a `List` of style directives recursively.

## Feature: Lambertian shading for Plot3D and ParametricPlot3D (2026-06-30)

Per-face flat Lambertian shading is now applied to all `Polygon[]` primitives
rendered by `graphics3d_show` (i.e. `Plot3D`, `ParametricPlot3D`, and any
hand-built `Graphics3D[...]` object). Shading is on by default and gives
surfaces an immediately 3D appearance without any geometry changes.

**Model**: `I = 0.3 + 0.7 × |n·L|` where `n` is the polygon's unit face normal
(cross product of first two edges, normalised) and `L` is a directional light
fixed in **camera space** (`0.3 right + 1.0 up + 0.5 forward`, normalised to
world space each frame). Being camera-relative means the shading updates as the
user orbits — the lit face shifts with the viewpoint, revealing surface shape
from any angle. The absolute value gives two-sided lighting — both faces of a
polygon are shaded as if lit from their respective front side, which is correct
since backface culling is disabled for mathematical surfaces. The ambient floor
of 0.3 prevents any face from going completely dark.

**Option**: `Lighting -> Automatic` (default) enables shading;
`Lighting -> None` (or `False`) falls back to the previous flat-color behavior —
useful when `ColorFunction` encodes all the information (a Rainbow-colored sphere
looks better unshaded) or when debugging geometry.

**Implementation**: `render3d.c` only — `shade_color()` with a static normalized
`LIGHT_DIR` constant, applied in `draw_primitive3`'s Polygon branch.
No changes to `plot3d.c` or `parametricplot3d.c` since shading is purely a
renderer concern.

## Feature: ParametricPlot3D (2026-06-30)

Added `ParametricPlot3D`, a 3D parametric plotting function complementing
`ParametricPlot` with full Graphics3D output and orbit-camera rendering.

**One-iterator form** — adaptive 3D space curves:
- `ParametricPlot3D[{fx,fy,fz}, {t,tmin,tmax}]` samples a parametric space
  curve. The adaptive sampler uses a 3D Euclidean chord-deviation test
  (normalized by the curve's bounding-box diagonal) with the same three-probe
  anti-aliasing as `ParametricPlot`, extended to 3 coordinates.
- Multi-curve: `ParametricPlot3D[{body1, body2, ...}, {t,...}]` draws each in
  a distinct palette colour.

**Two-iterator form** — parametric surface patches:
- `ParametricPlot3D[{fx,fy,fz}, {t,tmin,tmax}, {u,umin,umax}]` samples a
  uniform `PlotPoints × PlotPoints` grid in `(t,u)` space, maps each point
  to `(x,y,z)`, and emits `Polygon[]` quads, producing a parametric surface.

**Options** — full parity with `ParametricPlot`:
`PlotPoints` (default 25), `MaxRecursion` (default 6), `MaxPlotPoints`,
`Mesh` (default `None`; `All`/`True` overlays dots or grid), `PlotLegends`,
`ColorFunction` (`"Rainbow"` or `f[x,y,z]`), `ColorFunctionScaling`,
`RegionFunction` (`f[x,y,z]` then `f[x,y]` fallback), `PlotStyle`, `Axes`,
plus pass-through display options. `AspectRatio` is silently dropped (no
meaning for the orbit camera).

**Symbol**: `HoldAll`, `Protected`; auto-displays via the `Graphics3D` path.

## Feature: user-pinned substitution for Integrate's DerivativeDivides (2026-06-29)

`Integrate`'s `Method` option now accepts a list form for `"DerivativeDivides"`
that pins the substitution kernel:

```mathematica
Integrate[Sqrt[x]/(1 + Sqrt[x]), x,
          Method -> {"DerivativeDivides", "Substitution" -> Sqrt[x]}]
(* -2 Sqrt[x] + x + 2 Log[1 + Sqrt[x]] *)
```

Without the sub-option, `"DerivativeDivides"` collects **every** `x`-dependent
subexpression as a candidate kernel and trials each. The
`"Substitution" -> u` sub-option restricts the search to the single kernel
`u(x)`, running both the direct-quotient and the Eliminate/Solve strategies on
it. Behaviour is strict, as for all named methods: if `u` does not close the
integral the call returns unevaluated, with no fallback to the automatic kernel
search (e.g. `Substitution -> x^2` on the non-elementary `Sin[x^2]`).

Implementation: a new `forced_kernel` parameter on `dd_core`
(`src/calculus/integrate_derivdivides.c`) plus the public entry
`integrate_derivdivides_with_sub`; `parse_method_option`
(`src/calculus/integrate.c`) gained a `{"<method>", subopt -> val, ...}` list
branch mirroring the existing `FactorInteger` list-form precedent
(`src/facint.c`), with the `method_from_string` mapping factored out for reuse.
Only `"Substitution"`, and only for `"DerivativeDivides"`, is recognised; any
other sub-option key is `Integrate::method`-invalid. Covered by
`test_substitution_option` in `tests/test_integrate_derivdivides.c`.

## Feature: Chebychev binomial integrator `Integrate`ChebychevAlgebraic` (2026-06-29)

A new, fast integration method for Chebychev binomial differentials
`x^p (a x^r + b)^q` (`p, q, r` rational, `a, b` free of `x`).  By Chebychev's
theorem (1853) such an integral is elementary iff one of `q`, `(p+1)/r`,
`q+(p+1)/r` is an integer, each giving a substitution that turns the integrand
into a *rational function*:

- **Type I** (`q ∈ ℤ`): `x = u^N`, `N = LCM(den p, den r)`.
- **Type II** (`(p+1)/r ∈ ℤ`): `u^s = a x^r + b`, `s = den q`.
- **Type III** (`q+(p+1)/r ∈ ℤ`): `u = x^r`, then `t^s = (a u + b)/u`.

```mathematica
Integrate[x^(1/3) (a Sqrt[x] + b)^3, x]                     (* Type I  *)
Integrate[x^(3/2) (a x^(5/6) + b)^(1/3), x]                 (* Type II *)
Integrate[x^(-5/6) Sqrt[x^(1/3) - 1], x]
(* 3 x^(1/6) Sqrt[-1 + x^(1/3)] - 3 ArcTanh[Sqrt[-1 + x^(1/3)]/x^(1/6)]  Type III *)
```

Recognition is a single structural scan, so the method is cheap enough to run
ahead of `DerivativeDivides`'s Eliminate/Solve search (it sits between
`LinearRatioRadicals` and `Weierstrass` in the cascade — stage 6 of 10).  The
reduced rational integral is handled by the `BronsteinRational` stage and the
result is back-substituted.  Types I and II introduce a single principal radical
and are correct by construction (no differentiate-back verification, which is
branch-fragile with symbolic `a, b`); Type III is branch-sensitive and uses a
**split** back-substitution `t -> (a x^r + b)^(1/s) / x^(r/s)` so every radical
in the antiderivative is the principal root of a sub-expression of the original
integrand.

Non-elementary binomials (none of the three quantities integral, e.g.
`(a x^2 + b)^(1/3)`, or `1/Sqrt[x^4 + 1]`) return `NULL`, so under the default
`Method -> Automatic` the cascade falls through to the remaining methods; only
the forced `Method -> "ChebychevAlgebraic"` form surfaces as unevaluated.

Also available as `Integrate[f, x, Method -> "ChebychevAlgebraic"]` and the
package head `Integrate`ChebychevAlgebraic[f, x]`.

Implementation: `src/calculus/integrate_chebychev.{c,h}` (modelled on
`integrate_linrad.c` / `integrate_linratiorad.c`), wired into the cascade,
`method_from_string`, the `Method` switch and warning, and `integrate_init`
(`src/calculus/integrate.c`).  Covered by `tests/test_integrate_chebychev.c`
(all of `chebychev_problems.tex` problems 1 and 3, the worked Type III example,
problem-4 non-elementary cases, and method plumbing).

## Fix: Plot adaptive sampler density was inverted (2026-06-29)

`Plot[Sin[x], {x, 0, 2 Pi}, Mesh -> All]` placed **more** sample points on the
straight zero-crossings than on the curvy peaks — the opposite of curvature-
proportional sampling. Root cause: the absolute screen-space chord-**length**
cap (`MAX_CHORD_FRAC`), which keys on *slope* not curvature, was tight enough
(≈1/44 of the frame) to out-compete the curvature (vertical-gap) test on
ordinary moderately-sloped curves, pouring samples into the steep-but-straight
stretches while the loose curvature tolerance (`FLAT_TOL`) let the actually-
curvy peaks pass as "flat" at the 50-point grid scale.

Fix (`src/graphics/sampling.c`): tightened `FLAT_TOL` `0.0025 → 0.0006` so the
curvature test is the primary driver (density now tracks `|f''|`), and relaxed
`MAX_CHORD_FRAC` `0.0225 → 0.08` to a loose backstop that only binds on genuine
near-vertical gaps (asymptotes). Verified across `Sin` (peaks now ~2× denser
than zero-crossings), `x^3` (densest at the curved ends), `x^2` (uniform —
constant curvature), a straight line and `Exp` (no wasted refinement), the
`Runge` spike (peaked at centre), and `Sin[22 x]` (oscillation still fully
resolved). Total point counts dropped (less wasteful) while the shape is now
correct. Clean build; the chord cap's asymptote/`Log` protection is preserved.

## Real Hershey vector font for graphics text (2026-06-29)

Graphics text (`Text[]`, axis tick labels, `AxesLabel`/`PlotLabel`) now uses a
faithful transcription of the classic Hershey **Roman Simplex** single-stroke
font, replacing the previous small hand-authored stick alphabet.

- Glyph data is generated from the historical Hershey dataset
  (`tools/hershey.dat`) by `tools/gen_hershey.py` into
  `src/graphics/hershey_glyphs.inc`. The build and runtime read only the
  generated `.inc`; re-run the generator (and rebuild) to change the font set.
- Full printable ASCII (32–126): digits, **real upper- and lowercase** letters
  (no more folding lowercase to uppercase), punctuation and symbols, with
  **proportional advance widths** (`hershey_text_width` sums per-glyph bearings).
- Raw Hershey coordinates (design cap height 21, baseline-down) are normalised
  at draw time into the renderer's existing 7-unit cap-height frame
  (`HERSHEY_CAP_HEIGHT`), so all `scale`/positioning math and every caller in
  `render.c` are unchanged and text sizes match the prior build. Glyph shapes,
  lowercase, and spacing are substantially improved.
- Strokes are drawn at 2× weight via a single `HERSHEY_STROKE_SCALE` knob in
  `hershey_font.c`, with rounded joins, so labels read more boldly.
- The makefile now lists `hershey_glyphs.inc` as a prerequisite of
  `hershey_font.o`, so regenerating the font forces a recompile.

## Arithmetic of `SeriesData` (2026-06-29)

`Plus`, `Times`, and `Power` now combine `SeriesData[x, x0, {coefs}, nmin,
nmax, den]` objects, matching Mathematica. `Divide` and `Subtract` are covered
automatically because they reduce to `Times[a, Power[b,-1]]` and
`Plus[a, Times[-1, b]]`. Previously a `SeriesData` was opaque to the arithmetic
builtins, so `series + 1`, `x series`, `series^3`, `s1 s2`, `s1/s2` all stayed
unevaluated.

- **Plus / Times** fold every operand into a single `SeriesData` about the
  common `(x, x0)`, truncated to the **minimum** O-term order of the operands.
  A non-series operand free of `x` folds into the `a0` coefficient (fast path
  for `series + scalar`); the bare variable, polynomials, and transcendental
  functions are series-expanded with the full engine and combined. Operands
  about a **different** variable or expansion point are left as a symbolic
  `Plus`/`Times` (unevaluated), as in Mathematica.
- **Power** handles integer exponents (negative via series inversion) and
  scalar exponents free of the series variable directly. An exponent that
  depends on the series variable — or an ordinary base raised to a series
  exponent, e.g. `2^Series[x,{x,0,3}]` — is evaluated as `Exp[exp*Log[base]]`
  and re-expanded.
- **Mixed exact/approximate precision** works per-coefficient: coefficient
  arithmetic flows through the evaluator (`Plus`/`Times`), so integer, bigint,
  rational, machine-real, and MPFR coefficients mix with the usual contagion.
  Adding a real to a series folds it into `a0` as a real while the remaining
  coefficients stay exact.

Verified end-to-end: `Series[Exp[x],{x,0,2}]+1` → `2+x+x^2/2+O[x]^3`;
`x Series[Sin[x],{x,0,5}]` → `x^2-x^4/6+x^6/120+O[x]^7`;
`Series[Exp[x],{x,0,2}] Series[Exp[x],{x,0,3}]` → `1+2x+2x^2+O[x]^3` (order is
the min); `Series[Exp[x],{x,0,2}]^3` → `1+3x+9x^2/2+O[x]^3`;
`2^Series[x,{x,0,3}]` → `1+Log[2] x+Log[2]^2/2 x^2+Log[2]^3/6 x^3+O[x]^4`;
`Series[Exp[x],{x,0,3}]^Series[x,{x,0,3}]` → `1+x^2+O[x]^4` (= `Exp[x^2]`).

**Implementation.** Public entry points `series_combine_plus`,
`series_combine_times`, `series_power`, and the `is_series_data` predicate in
`src/calculus/series.c` (declared in `series.h`), driving the existing
`SeriesObj` algebra (`so_add`, `so_mul`, `so_pow_int`, `so_pow_expr`, `so_inv`,
`so_align_den`). A new `SeriesData` leaf case in `series_expand` lets a series
embedded in an expression (the `Exp[exp*Log[base]]` rewrite) flow through the
expander. Cheap `is_series_data` dispatch added before the grouping loops in
`builtin_plus` / `builtin_times` (ahead of inexact contagion so it cannot
re-`N` the structural integers) and after the identity reductions in
`builtin_power`. Tests in `tests/test_series.c`; valgrind shows no new leaks vs.
the macOS dyld baseline. Documented in
`docs/spec/builtins/power-series.md`.

Two guards keep folding faithful and terminating. A non-series operand whose
dependence on the series variable is *non-analytic* (a piecewise-constant head
such as `Floor`, `Arg`, `Sign`, ...) cannot be power-series expanded, so it
makes the combine bail to a symbolic `Plus`/`Times` instead of looping in
`series_expand` — this preserves the branch-point wrapper
`Times[(-1)^Floor[...], SeriesData[...]]` produced by `Series[ArcCosh[x+1], ...]`
and friends. And for a series at `Infinity` (where the formal variable is
`1/x`), a factor that is structurally free of `1/x` but still depends on the
underlying `x` (e.g. `E^x` in `Times[E^x, SeriesData[1/x, ...]]` from
`Series[ExpIntegralEi[x], {x, Infinity, n}]`) is *not* treated as a constant and
stays outside the series, matching Mathematica.

## Differentiation and integration of `SeriesData` (2026-06-29)

`D` and `Integrate` now act on `SeriesData[x, x0, {coefs}, nmin, nmax, den]`
objects term-by-term, returning a new `SeriesData`, matching Mathematica.
Previously `D[SeriesData[...], x]` fell through to the generic chain-rule
fallback (producing a meaningless `Derivative[1,0,...][SeriesData][...]`) and
`Integrate[SeriesData[...], x]` matched no method and stayed unevaluated.

- **Differentiation w.r.t. the series variable** applies the power rule: the
  coefficient of `(x - x0)^((nmin+i)/den)` is scaled by `(nmin+i)/den` and the
  exponent drops by one. `nmin` and `nmax` each decrease by `den`; the
  differentiated constant becomes a leading zero and is trimmed.
- **Integration w.r.t. the series variable** scales each coefficient by
  `den/(nmin+i+den)` and raises the exponent by one; `nmin` and `nmax` each
  increase by `den`. The integration constant is `0` (as in Mathematica). A
  nonzero `(x - x0)^-1` residue term would integrate to a `Log` that
  `SeriesData` cannot represent, so `Integrate` is left unevaluated in that
  case.
- **W.r.t. a different variable** (when `x0` is free of it) both operations
  thread into the coefficients, leaving the powers of `(x - x0)` unchanged.

Verified end-to-end: `Integrate[Series[Exp[x],{x,0,8}], x]` →
`x + x^2/2 + ... + x^9/362880 + O[x]^10`; `D[Series[Exp[x],{x,0,8}], x]` →
`1 + x + ... + x^7/5040 + O[x]^8`; the two round-trip; `1/x^2 + 1` integrates to
`-1/x + x + O[x]^5` while a true `1/x` residue stays unevaluated.

**Implementation.** Two helpers `series_differentiate` / `series_integrate`
(plus a private `so_from_seriesdata` reader) in `src/calculus/series.c`, reusing
the existing `SeriesObj` machinery (`so_alloc`, `so_set_coef`,
`so_trim_leading`, `so_to_expr`); one-line dispatch on `SYM_SeriesData` added to
`compute_deriv` in `src/calculus/deriv.c` and to `builtin_integrate` in
`src/calculus/integrate.c`. Tests in `tests/test_series.c`; valgrind-clean
(no new leaks vs. the macOS dyld baseline). Documented in
`docs/spec/builtins/power-series.md`.
