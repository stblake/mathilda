# Changelog: week of 2026-06-29 (Mon) – 2026-07-05 (Sun)

Feature additions and fixes recorded during this week.

## Limit: Method option (2026-07-04)

`Limit` now accepts a `Method` option that selects which internal strategy runs.
The evaluator has always been a cascade of strategy layers (`src/calculus/limit.c`,
`compute_limit`); those layers are now grouped under six user-facing names.

- **`Options[Limit]`** gains `Method -> Automatic` (alongside `Direction` and
  `Assumptions`).
- **`Method -> Automatic`** (default) runs the full cascade in order, exactly as
  before — behaviour is unchanged for existing inputs.
- **Named methods** — `"Substitution"`, `"RationalFunction"`, `"Series"`,
  `"LHospital"`, `"Asymptotic"` (the `ArcTan`/`Log`/dominant-term/`f^g` layers),
  and `"Bounded"` (envelope + oscillation) — restrict the **outermost** call to
  that one group. If the group resolves nothing, the whole `Limit` is left
  **unevaluated**; an unrecognised value emits `Limit::method` and is likewise
  left unevaluated. The restriction is confined to depth 1 (a new `method` field
  on `LimitCtx`, gated by `depth == 1`), so recursive sub-limits — one-sided
  probes, L'Hospital iterations, `Abs` splitting — keep the full cascade.
- Docstring, `docs/spec/builtins/calculus.md` (new `## Limit` section), the
  calculus tutorial, and the site overlay/impl pages updated. New tests in
  `tests/test_limit.c` (`test_method`) cover per-method success, the
  fail-⇒-unevaluated contract, unknown-method handling, and the depth-1
  recursion carve-out. Valgrind-clean (no leaks beyond the macOS init baseline).

## RootReduce: WL-faithful algebraic-number canonicalisation (G1–G5) (2026-07-04)

Brought `RootReduce` up to the core Wolfram-Language contract. Previously it only
rationalised radical denominators over a parametric function tower; it now also
canonicalises **constant algebraic numbers** and threads over relations. New
engine `src/poly/flint_qqbar.c` backed by FLINT `qqbar` (exact real/complex
algebraic numbers); `src/rootreduce.c` becomes a dispatcher.

- **G1 — canonical form for constant algebraic numbers.** An expression of
  integers, rationals, radicals, roots of unity, the imaginary unit and `Root[]`
  objects (no free symbol) folds bottom-up into one `qqbar`, then emits a
  **rational**, a **quadratic radical**, or `Root[Function[minpoly&], k]`
  (degree ≥ 3). `RootReduce[Sqrt[2]+Sqrt[3]]` → `Root[1 - 10 #1^2 + #1^4 &, 4]`;
  provably-equal nested radicals canonicalise identically. The `Root` index `k`
  uses Wolfram ordering (reals ascending, then non-reals) — indices match WL on
  the reference examples (including the degree-15 Root-product example).
- **G2 — nested radicals reduce.** `RootReduce[(Sqrt[18]+Sqrt[27])/Sqrt[5+2Sqrt[6]]]`
  → `3`; `RootReduce[(Sqrt[2]+Sqrt[3]+Sqrt[6]+3)/Sqrt[5+2Sqrt[6]]]` → `1+Sqrt[3]`.
- **G3 — diagnostics.** Wrong argument count emits `RootReduce::argx`; an unknown
  `Method` value emits `RootReduce::mtd`; the call is left unevaluated.
- **G4 — threads over relations/logic.** `Equal`, `Unequal`, `Less`,
  `LessEqual`, `Greater`, `GreaterEqual`, `And`, `Or`, `Not`, `Xor`, `Implies`.
  Binary (in)equalities of constant algebraic numbers are decided exactly via
  `qqbar` (`qqbar_equal` / `qqbar_cmp_re`): the Wolfram equality example
  `Sqrt[2]+Sqrt[3]+Sqrt[5] == Sqrt[10+2Sqrt[15]+4Sqrt[4+Sqrt[15]]]` → `True`.
- **G5 — `Method` option.** `Options[RootReduce] = {Method -> Automatic}`.
  `"Recursive"`/`"Automatic"` fold `qqbar` arithmetic directly; `"NumberField"`
  builds a primitive element of the common field and re-expresses the value via
  `qqbar_express_in_field`. All three agree (verified on the degree-21 example).
- **Bug fixed while wiring the `Root[]`-object input path:** the univariate
  poly-Expr → `fmpz_poly` builder reused a temp polynomial across `Plus`/`Times`
  terms without zeroing it in the variable case, so `t^3 + t + c` came out as
  `2 t^3 + …` (wrong minimal polynomial, and a resulting arithmetic blow-up).
- Dispatch is additive: parametric towers (free-variable radicands, e.g. the
  Goursat `k^(1/3)` families) still route to `flint_algebraic_field_canonical`;
  all prior `rootreduce_tests` remain green. `test_rootreduce.c` extended with
  G1–G5 cases. Degree cap (120) guards against pathological `qqbar` blow-up.

## RootReduce + WL-faithful Together/Cancel over cube-and-higher-root towers (2026-07-04)

Milestone B of the genuine-algebraic FLINT programme: full **field arithmetic**
(not just the zero test) over the tower `K = Q(params)(α_1..α_r)`, delivering two
user-visible capabilities.

- **New builtin `RootReduce`** (`src/rootreduce.c`, engine
  `flint_algebraic_field_canonical` in `src/poly/flint_bridge.c`). Rewrites an
  algebraic expression in canonical form with the **denominator rationalised** —
  cleared of every radical / root-of-unity generator. `K` is a finite-dimensional
  `Q(params)`-vector space with the monomial basis
  `{ gen_0^{e_0} … gen_{r-1}^{e_{r-1}} : 0 ≤ e_i < d_i }` (`d_i` = degree of
  `gen_i`'s minimal polynomial: `Q_i` for a radical, `deg Φ_{2Q_i}` for a root of
  unity). Multiplication-by-the-denominator is a `Q(params)`-linear map — each
  product reduced to normal form via `fmpz_mpoly_divrem_ideal` — and inverting it
  with a linear solve over the field `Q(params)` (`gr_mat_nonsingular_solve` over
  `fmpz_mpoly_q`) gives `N/D` with denominator `Norm_{K/Q(params)}(D)`, free of
  every generator, over which the coordinate solutions are recombined (lcm of
  their denominators) into a single fraction. Rigorous and complete when the
  minimal polynomials are irreducible; **no numeric zero oracle**.
  Examples:
  - `RootReduce[1/(1 + k^(1/3))]` → `(1 - k^(1/3) + k^(2/3))/(1 + k)`
  - `RootReduce[1/(a + b k^(1/3))]` → `(a² - a b k^(1/3) + b² k^(2/3))/(a³ + b³ k)`
  - `RootReduce[1/(1 + Sqrt[2])]` → `-1 + Sqrt[2]`
  - `RootReduce[1/(a + b Sqrt[2] + c Sqrt[3])]` →
    `(…)/(a⁴ - 4 a² b² + 4 b⁴ - 6 a² c² - 12 b² c² + 9 c⁴)`

  Listable, Protected. Leaves an expression with no algebraic generator unchanged
  (matching WL). Idempotent.

- **`Together`/`Cancel` now combine a sum of cube-and-higher-root fractions**
  (`flint_algebraic_field_together`, wired into `flint_cancel_fraction`). The
  radicals are treated as **free kernels** — `Power[base, p/q] → gen^p`, combined
  over a common denominator in lowest terms via `fmpz_mpoly_q` — so this keeps
  radicals in the denominator and does **not** rationalise (WL-faithful,
  generalising the sqrt-only `flint_parametric_field_normalize` to any index).
  It fires only on `Plus` inputs with genuine (symbol/polynomial-radicand)
  generators, so it never pre-empts the relation-aware single-fraction GCD path.
  - `Together[1/(x - k^(1/3)) + 1/(x + k^(1/3))]` → `2 x/(x² - k^(2/3))`
    (previously left uncombined)
  - `Cancel[(x^3 - k)/(x - k^(1/3))]` → `k^(2/3) + k^(1/3) x + x²` (unchanged —
    single-fraction path still handles relation-dependent cancellation)
  - constant-radical (`Sqrt[2]`) sums still combine via the number-field path

Tests: `tests/test_rootreduce.c` (`rootreduce_tests`) — exact canonical forms,
value preservation cross-checked by the **independent** ideal-reduction zero test
(`flint_algebraic_field_normalize` agrees with the matrix inversion), rationalised
denominators, idempotence, WL-faithful Together/Cancel, and a stress batch of
towers. No regressions across the FLINT / rat / simp / Goursat / extension /
radical / trigrat / intrat suites.

## Rigorous FLINT reduction over genuine-algebraic parametric towers (2026-07-04)

`Simplify`/`Together`/`Cancel` could not verify the cube-root Goursat identity
`D[Integrate[f,x],x] - f == 0` (with `f = (2-(k+1)x)/((1-(k+1)x)(x(1-x)(1-kx))^(1/3))`):
the derivative-minus-integrand is a rational function over the tower
`Q(x,k)(α,β,ω)` with `α = (x(1-x)(1-kx))^(1/3)` (a **genuine** degree-3 algebraic
generator — polynomial radicand), `β = k^(1/3)`, and `ω = (-1)^(1/3)`. This is the
parametric-algebraic regime the sqrt-only path (`flint_parametric_*`, which only
accepts `Sqrt[symbol]`) and the constant-radicand number-field path both reject,
so it fell to the classical paths, which under-reduce or exhaust memory.

- **New engine** `flint_algebraic_field_normalize` (`src/poly/flint_bridge.c`).
  Models `K = Q(params)(α_1..α_r)` as the quotient ring `Q[params, gens]/I` with
  `I = (gen^q − radicand, Φ_{2Q}(ω), …)`. Detects genuine-algebraic generators
  (radicals `Power[base, p/q]` of **any** index `q ≥ 2` whose radicand carries a
  free symbol — cube roots and polynomial/symbol radicands — plus roots of unity
  `(-1)^(p/q)`), canonicalising radicands by `Expand` so two printed forms of one
  radicand (`x(1-x)(1-k x)` vs `x - x² - k x² + k x³`) collapse to a single
  generator. With the generators ordered as the leading LEX variables the monic
  minimal polynomials form a Gröbner basis (pairwise-coprime leading monomials),
  so `fmpz_mpoly_divrem_ideal` yields the canonical normal form. An expression
  written `N/D` over `Q[params,gens]` is zero in `K` iff `N` reduces to 0 modulo
  `I` — a **rigorous** zero test (sound always; complete for irreducible minimal
  polynomials), with **no numeric zero oracle** (`Simplify` never calls
  `PossibleZeroQ`) and no primes/CRT.

- **Wired into `Cancel`/`Together`** (`flint_cancel_fraction`, `src/rat.c`) as the
  first whole-expression try. It returns the integer `0` exactly when the input is
  identically zero in the field, else `NULL` — so it can only *add* the ability to
  reach `0` rigorously and never worsens an existing reduction. `Simplify` picks
  this up through its `Together`/`Cancel` transforms. `Simplify[D[F,x]-f]`,
  `Together[…]`, `Cancel[…]` now all give `0`, while perturbed / genuinely-nonzero
  variants (e.g. `D[F,x]-2f`, `(x(1-x)(1-kx))^(1/3)+k^(1/3)`) are left untouched.

- Tests: `test_algebraic_field_normalize` + `test_goursat_identity_reduces` in
  `tests/test_flint_bridge.c` (direct zero/NULL contract and the end-to-end
  Goursat identity, including a non-zero guard). Regression suites green.

## Cube-root Goursat: third-kind logarithmic part + Cancel/Together fix (2026-07-03)

Two connected changes let `Integrate[]` close the pseudo-elliptic cube-root
family `(2 - (k+1) x)/((1 - (k+1) x) (x (1-x)(1-k x))^(1/3))` — including with a
**symbolic parameter `k`** — both through the default cascade and through
`Method -> "GoursatAlgebraic"`.

- **General `Cancel`/`Together` fix over algebraic number fields** (`src/rat.c`).
  `Cancel[Together[e, Extension -> Automatic], Extension -> Automatic]` failed to
  reduce a rational function whose coefficients carry algebraic denominators
  (e.g. `1/Sqrt[3]`) over a multi-generator field such as `Q(I, Sqrt[3]) =
  Q(zeta_12)`: FLINT's number-field GCD (`flint_cancel_fraction`) and the QA
  tower path both returned the fraction un-cancelled. Fixes: (1)
  `cancel_with_extension` now `Expand`s numerator/denominator before the
  `PolynomialGCD`, so an unexpanded product of algebraic-coefficient factors
  reduces; (2) a new `cancel_auto_gcd_quotient` reduces a single fraction
  directly through `PolynomialGCD` + `PolynomialQuotient` with
  `Extension -> Automatic` (each re-detects the whole splitting field), run as a
  strict-shrink first pass ahead of the FLINT/tower routing for algebraic-atom
  inputs. Plain-rational cancellation is untouched. The concrete failure — the
  cube-root descent's `R(t(z))(1-z)^3` coming back as degree-6-over-`(z-1)^3`
  with a zero `z^3` coefficient — now reduces to the correct cubic.

- **Third-kind cube-root logarithmic-part reduction** (`goursat_cubic_thirdkind`
  in `src/calculus/integrate_goursat.c`). When the order-3 eigendescent
  obstruction `H1 != 0` but the cofactor `F` has a pole at a NON-branch point,
  `F/R^(1/3)` is still elementary and its antiderivative is the logarithmic part
  `C * Sum_{j=0..2} omega^(2 j) Log[R^(1/3) - omega^j kappa t]`
  (`kappa = (lead R)^(1/3)`, `omega = e^(2 pi i/3)`). The coefficient direction
  `(1, omega^2, omega)` is forced by killing the trace and `omega^1` components;
  the reduction differentiates the ansatz modulo `Y^3 = R`, checks the `Y^0`/`Y^1`
  slots vanish, and reads the scale `C = F N / (R p2)` (`N = R - (lead R) t^3`) by
  sampling. Tried before the eigendescent (cheaper, and avoids its budget-
  exhausting parametric Möbius canonicalisation); every result is verified by the
  differentiate-back guard, so the polynomial-cofactor / genuinely-elliptic cases
  (`t/(t^3-1)^(1/3)`) still decline. Tests: `test_integrate_goursat.c::test_thirdkind`.

  Known limitation: over the two-generator field `Q(kappa, omega)` the descent's
  canonicalisation exercises a pre-existing, bounded (size-scaled, per-call) leak
  in the extension tower LCM/`Together` path; the new integrator code itself is
  valgrind-clean. Only the `m = 0` linear form (F's pole a root of
  `N = R - (lead R) t^3`) is handled; a general affine form `kappa t + m` is
  deferred.

## New infinite Sum / Product closed-form families (2026-07-03)

Six new algorithmic families close a batch of previously-unevaluated infinite
`Sum[]`/`Product[]` inputs, plus a warning fix. Each is a self-contained stage
in the existing cascades (`src/sum/sum.c`, `src/product/product.c`).

- **`Sum`Dirichlet`** (`src/sum/sum_dirichlet.c`) — Dirichlet series of recognised
  multiplicative arithmetic functions map to ratios of Riemann zeta values:
  `Sum[MoebiusMu[k]/k^s] = 1/Zeta[s]`, `Sum[LiouvilleLambda[k]/k^s] = Zeta[2s]/Zeta[s]`,
  `Sum[EulerPhi[k]/k^s] = Zeta[s-1]/Zeta[s]`, `Sum[DivisorSigma[0,k]/k^s] = Zeta[s]^2`,
  `Sum[DivisorSigma[0,k]^2/k^s] = Zeta[s]^4/Zeta[2s]`, `Sum[DivisorSigma[a,k]/k^s] =
  Zeta[s] Zeta[s-a]`, `Sum[MoebiusMu[k]^2/k^s] = Zeta[s]/Zeta[2s]`. Table-driven;
  even zetas fold (e.g. `Sum[LiouvilleLambda[k]/k^2] = Pi^2/15`). Runs before
  `Sum`Rational`.
- **`Sum`ZetaSeries`** (`src/sum/sum_zetaseries.c`) — sums of functions of
  `Zeta[a k + b]` by interchanging summation order: `Sum[Zeta[k]-1, {k,2,Inf}] = 1`,
  `Sum[(-1)^k (Zeta[k]-1), {k,2,Inf}] = 1/2`, `Sum[Zeta[2k]-1, {k,1,Inf}] = 3/4`;
  unblocks `Product[Exp[Zeta[k]-1], {k,2,Inf}] = E` via `Product`LogSum`.
- **`Sum`LogZeta` generalised** (`src/sum/sum_logzeta.c`) — now emits the inert
  `-c Derivative[1][Zeta][s]` for any integer `s >= 2` (previously only the `s==2`
  Glaisher form), so `Sum[Log[k]/k^3] = -Zeta'[3]` and, via `Product`LogSum`,
  `Product[k^(1/k^3)] = Exp[-Zeta'[3]]`.
- **`Sum`EulerNonlinear`** (`src/sum/sum_euler_nonlinear.c`, `sum_euler_internal.h`)
  — nonlinear Euler sums (products of `HarmonicNumber[k,p]`, and `PolyGamma[m,k]`
  factors) reduced to polynomials in Riemann zeta values via multiple zeta value
  reduction, gated to total weight `<= 7` (first irreducible MZV is `zeta(6,2)` at
  weight 8; heavier sums fall through, never a wrong value). Pipeline: quasi-shuffle
  (stuffle) expansion → `T`/`That` lift → depth-1/2 reduction (Euler, reflection,
  odd-weight double-zeta — shared with `Sum`Euler`) + a curated weight-<=7 depth->=3
  table. Examples: `Sum[HarmonicNumber[k]^3/k^4]`, `Sum[HarmonicNumber[k]
  HarmonicNumber[k,2]/k^3]`, `Sum[PolyGamma[1,k]^2] = 3 Zeta[3]`. Runs after the
  narrow `Sum`Euler` (existing fast paths untouched).
- **`Product`QProduct` infinite branch** (`src/product/product_qproduct.c`) —
  `Product[1 - a q^(b k + c), {k,imin,Inf}] = QPochhammer[a q^(b imin+c), q^b]`, e.g.
  `Product[1 - q^k] = QPochhammer[q,q]`, `Product[1/(1-x^k)] = 1/QPochhammer[x,x]`,
  `Product[1 - q^(2k)] = QPochhammer[q^2,q^2]`.
- **`Product`BesselZero`** (`src/product/product_besselzero.c`) + new symbol
  **`BesselJZero`** (`src/special_functions/besseljzero.{c,h}`) — the Bessel
  Hadamard product `Product[1 - x^2/BesselJZero[n,k]^2, {k,1,Inf}] =
  Gamma[n+1] (2/x)^n BesselJ[n,x]`, so `Product[1 - x^2/BesselJZero[0,k]^2] =
  BesselJ[0,x]`. `BesselJZero` stays symbolic for now.
- **`Product`Infinite` warning fix** (`src/product/product_infinite.c`) — the
  speculative Weierstrass-sinh recogniser (`try_sinh_family`) now skips
  symbolic-power bodies and mutes its speculative `Simplify`, eliminating the ~24
  spurious `Power::infy: 1/0` warnings on
  `Product[(1+x/k)Exp[-x/k], {k,1,Inf}]` (answer unchanged).

Tests in `tests/test_sum_product_families.c` (29 checks); `tests/test_sum_log.c`
updated for the new `Sum`LogZeta` output. Not addressed (deferred): the Chudnovsky
1/π series, Rogers–Ramanujan q-series, sums over `ZetaZero`, and the Glaisher-type
log product `Product[(1+1/k)^k Exp[-1+1/(2k)]]` (needs LogGamma/Barnes-G asymptotic
expansion in the Series/Limit engine).

## PrimeNu: number of distinct prime factors (2026-07-03)

New number-theory builtin `PrimeNu[n]` (`src/numbertheory/primenu.c`) giving
`nu(n)`, the number of **distinct** prime factors of `n`. It is the companion to
`PrimeOmega[n]` (`Omega(n)`, prime factors with multiplicity) and shares the
same factoring machinery — the implementation is `primeomega.c` with the final
reducer returning the *count* of factors rather than the exponent sum. `nu` and
`Omega` coincide exactly when `n` is square-free.

- Computed from the prime factorisation via the shared `df_factor_mpz` helper;
  machine integers and GMP bigints (e.g. `PrimeNu[50!] -> 15`) are handled
  uniformly. `PrimeNu[1]` and `PrimeNu[-1]` are `0`; the sign of `n` is ignored
  (`nu(-n) = nu(n)`). A prime power gives `1` (`PrimeNu[32] -> 1`).
- Gaussian integers: `PrimeNu[n, GaussianIntegers -> True]`, or a non-real
  `Complex[a, b]` argument, factors over `Z[i]` via `df_gaussian_prime_factor`
  and counts distinct Gaussian prime factors (a rational prime `p ≡ 1 (mod 4)`
  splits into two conjugate primes, so `PrimeNu[105, GaussianIntegers -> True]`
  is `4` versus `PrimeNu[105] -> 3`; `PrimeNu[3 + I] -> 2`).
- Additive on coprime arguments (`nu(m n) = nu(m) + nu(n)` when `GCD[m,n] == 1`);
  for a square-free `n`, `MoebiusMu[n] == (-1)^PrimeNu[n]` and
  `LiouvilleLambda[n] == (-1)^PrimeNu[n]`.
- Attributes `Listable`, `Protected`. Non-integer or zero `n` is left
  unevaluated; a wrong argument count emits `PrimeNu::argt`.
- Docstring in `info.c`; symbol `SYM_PrimeNu` registered in `sym_names.{c,h}`;
  unit tests in `tests/test_primenu.c` (valgrind-clean — no leak frames in the
  new code, macOS dyld/libobjc baseline noise and vendored-ECM `possibly lost`
  blocks only).

## Fix: FactorInteger mislabelled prime powers of a large prime as prime (2026-07-03)

`FactorInteger[(2^61-1)^2]` returned `{{5316911983139663487003542222693990401,
1}}` — declaring the composite square of a Mersenne prime to be prime. The root
cause was in `factorize_mpz` (`src/facint.c`): a prime power `p^k` of a large
prime defeats the general search methods (Pollard rho collides on the repeated
factor, and ECM has no small cofactor to grab and cannot crack the large `p`
directly), so the "gave up" fallback added `n` with exponent `1`, silently
lying about primality. `PrimeQ` was unaffected (it uses `mpz_probab_prime_p`
directly); the same latent bug also corrupted `EulerPhi`, `DivisorSigma`, and
`Divisors` on such inputs, since they share `factorize_mpz`.

- Added a **perfect-power reduction** at the top of `factorize_mpz` (after the
  primality test, before method dispatch): if `mpz_perfect_power_p(n)`, extract
  exact integer roots to a fixpoint to reduce `n = base^mult` to a primitive
  base, factor the base once into a private list, then merge the base's factors
  with each exponent scaled by `mult`. This sidesteps the search entirely for
  the whole prime-power / perfect-power class.
- Handles multi-prime bases and stacked exponents: `(2^61-1)^3`, `(2^61-1)^6`,
  `6^10 -> {{2,10},{3,10}}`, `(2^31-1)^2 (2^61-1)^3`, and negatives all factor
  correctly. Genuine Mersenne primes (`2^127-1`) still return `{{p, 1}}`.
- The `FactorInteger[n, k]` factor-count cap and explicit `Method ->` selections
  are respected inside the new branch (the merge honours `k_limit`).
- The residual limitation (a hard composite the bounded search can't split —
  e.g. a semiprime of two large distinct primes) no longer fails silently: the
  `Automatic` give-up path now emits a `FactorInteger::nofac` warning to stderr
  before returning the composite with exponent 1. The warning honours the shared
  `arith_warnings_muted()` gate, so it stays quiet inside numeric integration,
  `Limit`, `Series`, and other contexts that mute arithmetic messages.

## PrimeOmega: number of prime factors with multiplicity (2026-07-03)

New number-theory builtin `PrimeOmega[n]` (`src/numbertheory/primeomega.c`)
giving `Omega(n)`, the number of prime factors of `n` counted with multiplicity
(the exponent sum in the factorisation). It is the additive companion to the
newly added `LiouvilleLambda[n] = (-1)^Omega(n)`, and reuses the same factoring
machinery — the implementation is `liouvillelambda.c` with the final reducer
returning `Omega` itself instead of `(-1)^Omega`.

- Computed from the prime factorisation via the shared `df_factor_mpz` helper;
  machine integers and GMP bigints (e.g. `PrimeOmega[30!] -> 59`) are handled
  uniformly. `PrimeOmega[1]` and `PrimeOmega[-1]` are `0`; the sign of `n` is
  ignored (`Omega(-n) = Omega(n)`).
- Gaussian integers: `PrimeOmega[n, GaussianIntegers -> True]`, or a non-real
  `Complex[a, b]` argument, factors over `Z[i]` via `df_gaussian_prime_factor`
  and counts Gaussian prime factors with multiplicity (so
  `PrimeOmega[12, GaussianIntegers -> True]` is `5` from `(1+i)^4 3`, and
  `PrimeOmega[5 + 9 I]` is `2`).
- Attributes `Listable`, `Protected`. Non-integer or zero `n` is left
  unevaluated; a wrong argument count emits `PrimeOmega::argt`.
- Docstring in `info.c`; symbol `SYM_PrimeOmega` registered in
  `sym_names.{c,h}`; unit tests in `tests/test_primeomega.c` (valgrind-clean —

## LiouvilleLambda: Liouville function (2026-07-03)

New number-theory builtin `LiouvilleLambda[n]` (`src/numbertheory/liouvillelambda.c`)
giving the Liouville function `lambda(n) = (-1)^Omega(n)`, where `Omega(n)` counts
the prime factors of `n` with multiplicity. It is completely multiplicative and
returns `1` (even number of prime factors) or `-1` (odd).

- Computed from the prime factorisation via the shared `df_factor_mpz` helper;
  machine integers and GMP bigints (e.g. `LiouvilleLambda[10^30 + 1]`) are
  handled uniformly. The sign of `n` is ignored.
- Gaussian integers: `LiouvilleLambda[n, GaussianIntegers -> True]`, or a
  non-real `Complex[a, b]` argument, factors over `Z[i]` via
  `df_gaussian_prime_factor` and counts Gaussian prime factors with
  multiplicity (so `LiouvilleLambda[2, GaussianIntegers -> True]` is `1`).
- Attributes `Listable`, `Protected`. Non-integer/zero `n` is left unevaluated;

## TimeConstrained: async-signal-safe abort (fixes intermittent SIGILL crash) (2026-07-03)

`TimeConstrained` aborts an over-budget computation with a `siglongjmp` taken
from its asynchronous `SIGPROF` handler. `SIGPROF` can land on any instruction —
including deep inside `malloc`/`realloc`/`free` while libmalloc holds an internal
zone lock. Unwinding past that point never releases the lock, so the *next*
allocation deadlocks; on macOS libmalloc detects the still-owned lock and traps
with `_os_unfair_lock_recursive_abort`, i.e. `Illegal instruction: 4`.

This was reproducible (~1 in 8 runs) on integrands whose algebraic-extension
`Together`/`Cancel` churns giant GMP integers, e.g.

```
Integrate[(2 - (k + 1) x)/((1 - (k + 1) x) (x (1 - x) (1 - k x))^(1/3)), x,
          Method -> "GoursatAlgebraic"]
```

The 6 s Goursat budget expires while a multi-megabyte `__gmpz_mul` is inside
`large_malloc`, leaking the large-zone lock.

**Fix** (`src/core.c`, `src/core.h`): the large allocations that leak the lock
originate exclusively from GMP (and MPFR, which shares GMP's allocators), so GMP
is routed through guarded memory functions (`mp_set_memory_functions`, installed
first thing in `core_init`). While a real `malloc`/`realloc`/`free` is in flight
a `tc_gmp_alloc_busy` flag is set; the `SIGPROF` handler then *defers* the jump
(records the timeout and returns) instead of unwinding mid-lock, and the
allocator takes the deferred jump from `tc_alloc_safepoint()` the instant the
real call returns and releases libmalloc's lock. Code outside a GMP allocation
keeps the immediate async jump, so `TimeConstrained` still interrupts long pure-C
GMP loops (e.g. `FactorInteger` on a large composite) — now at the next
allocation boundary rather than at an arbitrary lock-holding instruction.

Verified: 60/60 crash-free on the reproducer (was ~12.5 %); `TimeConstrained`
still returns `$Aborted` on a genuine CPU-time overrun; `core_tests` and
`integrate_goursat_tests` pass; heavy bignum/MPFR paths (`FactorInteger`,
`Factorial[2000]`, `N[Sqrt[2], 1000]`, `Together`, `GCD`) unaffected. The
underlying algebraic-extension `Together` slowness on this integrand is a
separate, tracked performance limitation.

## Sum: infinite polynomial × geometric sums (2026-07-02)

`Sum[p(k) r^k, {k, k0, Infinity}]` with `|r| < 1` now evaluates to its exact
rational closed form instead of falling through to a `HypergeometricPFQ`. For
example `Sum[k^2/2^k, {k, 0, Infinity}]` → `6`, `Sum[k^3/2^k, {k, 0, Infinity}]`
→ `26`, `Sum[1/2^k, {k, 0, Infinity}]` → `2`.

- **Mechanism.** `Sum`Geometric` already builds the antidifference `F(k) = q(k) r^k`
  for finite ranges. For an infinite range with `|r| < 1` the boundary term
  `q(n+1) r^(n+1) -> 0`, so the sum is exactly `-F(imin)`. The stage now takes
  this branch after a convergence gate (`|r| < 1` decided via `Abs[r] < 1`);
  symbolic `r` or `|r| >= 1` (including the divergent `r = -1`/`r = 1` edges) is
  left unevaluated rather than given a false analytic continuation.
- **Geometric-factor detection generalised.** `find_geometric` now recognises a
  geometric factor by the representation-independent ratio test `g(i+1)/g(i)`
  free of `i`, so `2^(-k)` (parsed as `(2^k)^(-1)`), `(1/2)^k`, `r^(a i + b)`,
  etc. are all handled. The polynomial part is accumulated by multiplying in
  each geometric factor's var-free constant `C = g|_{i->0}` (rather than
  dividing the summand by a reconstructed `r^i`, which failed to cancel when the
  ratio's base differed structurally from the factor's, e.g. `1/2` vs `2`).
- **Files.** `src/sum/sum_geometric.c`; regression tests in `tests/test_sum.c`
  (60 pass). No change to the finite/indefinite results (`Sum[a^i, i]`,
  `Sum[q1^i q2^i, i]`, …).

## Simplify: numeric contagion must not cross List boundaries (2026-07-02)

`{1., 1} // Simplify` returned `{1.0, 1.0}` — the exact `1` was numericalised
merely because it shared a list with the inexact `1.`. More visibly,
`Simplify[{0.982981, r}]` (e.g. the first element being a `Timing[]` result)
numericalised the *whole* symbolic `r`, turning an exact closed form into
`2.0 k - 4.0 k x - …`.

- **Cause.** `Simplify` is not `Listable`, so a list argument was handed whole to
  the `internal_rationalize_then_numericalize` round-trip (rationalise every
  inexact leaf → run the exact pipeline → **numericalise the entire result
  tree**). The final blanket `numericalize` swept exact numbers that never
  touched an inexact input. (`Together`/`Cancel` escape this because they *are*
  `Listable` and thread before the round-trip.)
- **Fix.** When the argument is a `List` and contains an inexact leaf, thread
  `Simplify` over the elements (`simplify_thread_list` in `simp_builtins.c`),
  deciding inexactness independently per element and preserving trailing
  options/assumptions. Nested lists (matrices) are handled by recursive
  re-entry. Genuine within-expression contagion is unchanged
  (`Simplify[(x-1.5)/(x^2-2.25)] == 1/(1.5+x)`).
- **Result.** `{1., 1} // Simplify == {1., 1}`; `{1.5, x, {2, 3.}} // Simplify ==
  {1.5, x, {2, 3.}}`; `Simplify[{0.982981, r}]` keeps `r` exact.

## Cancel / Together / Simplify: FLINT fast path for the internal multivariate GCD (2026-07-02)

`Simplify` of the (exact) Goursat antiderivative

## Integrate`GoursatAlgebraic: Theorem-1 fast path — no needless character split (2026-07-02)

`Integrate[(1 + k x)/((-1 + k x) Sqrt[(1-x) x (1-k² x)]), x, Method -> "GoursatAlgebraic"]`
produced a *correct* but enormous antiderivative (9495 leaves, two `ArcTanh` + two
`Log` with 40 nested radicals). Root cause: `goursat_v4` always ran the **Theorem-2**
Klein-four character decomposition `F = F₁ + F₂ + F₃`, even when the cofactor `F` is
*already* anti-invariant under one of the three root-pairing involutions
(`F + F(S_k) == 0`, Goursat's **Theorem 1**). Splitting an already-anti-invariant `F`
turns a *constant* genus-0 integrand into two *non-constant* ones, so the two piece
reductions bloat and — lacking a full `Simplify` — never recombine.

- **Fix.** Before the character split, test each involution for `F + F(S_k) == 0` (a
  pure rational zero test, so `PossibleZeroQ` is reliable) and, on a hit, run a single
  direct `reduce_v4_piece(F, S_k)`. Falls back to the Theorem-2 split when no single
  involution works or the direct reduction does not close. Verified by the existing
  differentiate-back guard.
- **Result.** The named parametric integral: **9495 → 444 leaves**, a single correct
  term, `PossibleZeroQ[D[result,x] - integrand] == True`. No regressions across the
  goursat / quadrad / jeffrey / chebychev / derivdivides / linrad / linratiorad /
  dispatch integrate suites.
- **Former remaining gap — now closed** (see the next section). The 444-leaf term still
  carried the single radical `Sqrt[R]` *split* as `Sqrt[x] Sqrt[(1-x)(1-k² x)]` across a
  numerator/denominator, which is a spurious branch point and several independent radical
  generators that made a downstream `Simplify` fail to converge. That split is now
  recombined inside the integrator, so `Simplify` completes.

## Integrate`GoursatAlgebraic: recombine split radicals into the single generator Sqrt[R] (2026-07-02)

Follow-up to the Theorem-1 fast path above. The 444-leaf antiderivative of
`Integrate[(1 + k x)/((-1 + k x) Sqrt[(1-x) x (1-k² x)]), x, Method -> "GoursatAlgebraic"]`
was correct but presented the one radical `Sqrt[R]` (R = x(1-x)(1-k² x)) *split* as a
product of factor roots — e.g. a term `(1/k+x)² Sqrt[R] / (Sqrt[x] (-1/k+x)² Sqrt[(1-x)(1-k² x)])`
that is, on the integrand's domain, just the rational `(1/k+x)²/(-1/k+x)²`. Two symptoms,
one cause:

- an apparent `Sqrt[x]` branch point the true antiderivative does not have, and
- three independent radical generators, so a following `Simplify` blew up on the
  multivariate GCD/`Together` cost and never converged (effectively hung).

**Where the split comes from.** The descent's own output carries the radical as a *single*
`Sqrt[c·poly]`. `reexpress_over_radical` re-expresses each nested radical as
`rational·Sqrt[R]` one `Sqrt` node at a time; once the evaluator has distributed a factored
radicand across a term, that per-node rewrite leaves `Sqrt[R]` split across the term's
numerator and denominator. Individually neither `x/R` nor `(1-x)(1-k² x)/R` is a perfect
square, so the per-node pass cannot re-merge them.

**Fix (general, correct-by-construction).** After `reexpress_over_radical`, evaluate to the
flat form where the split is visible and run a new `combine_radicals` pass: at every `Times`
node it merges the *x-dependent* half-power factors `Power[base, m/2]` into one rational
radicand ρ (bases of `+m/2` into the numerator, `−m/2` into the denominator), reduces
`Sqrt[ρ]` over R (→ a rational when ρ is a perfect square, → `rational·Sqrt[R]` when ρ/R is),
reattaches the integer-power leftovers and all non-radical (and *constant*-radical) factors,
and fixes the one remaining global sign numerically. Constant radicals such as
`Sqrt[-1-1/k²+2/k]` are left untouched (they keep the `Log` real). The rewrite is adopted
only if the differentiate-back guard still passes, so it can never introduce a wrong answer.
New helpers: `is_half_power`, `numeric_sign_ratio`, `reduce_sqrt_over_R`,
`combine_times_radicals`, `combine_radicals` in `src/calculus/integrate_goursat.c`.

- **Result.** The named integral now returns with a single x-radical `Sqrt[R]` and no
  spurious `Sqrt[x]` (`FreeQ[result, Power[x, -1/2]] == True`); the differentiate-back

## NIntegrate: fast, correct high-precision semi-infinite oscillatory integrals (2026-07-02)

Fixed a wrong-answer bug and a severe performance gap in the arbitrary-precision
(`WorkingPrecision`) branch of `NIntegrate` for semi-infinite oscillatory
(Fourier-type) integrals such as `∫_0^∞ x Sin[x]/(x²+4) dx = π/(2 E²)`.

- **The bug.** At machine precision the semi-infinite path already fell back from
  exp-sinh to an integrate-between-the-zeros + Wynn-epsilon engine when the
  oscillatory tail defeated exp-sinh. The MPFR path had **no such fallback**, so
  `WorkingPrecision -> 200` ran exp-sinh to non-convergence and returned a wrong,
  un-converged value (`0.434…` for the case above) with an `ncvb` warning.
- **Correctness fix.** Added `osc_integrate_mpfr` (`src/numerical_calculus/oscint.c`):
  the MPFR counterpart of the machine oscillatory integrator — cheap machine-precision
  zero-finding for the lobe geometry, each half-period lobe integrated to full precision
  with tanh-sinh, partial sums accelerated with the MPFR Wynn epsilon (degree scaled to
  the requested digits, not the machine-path cap of 6). Wired in as the MPFR AUTO/Levin
  fallback. This alone makes the answer correct, but per-lobe high-precision quadrature is
  slow (hundreds of thousands of samples for 200 digits).
- **Speed fix (the elegant path).** New `src/numerical_calculus/oscde.{c,h}` implements
  the **Ooura–Mori double-exponential rule for Fourier integrals** (Ooura & Mori, *JCAM*
  112, 1999): `x = (M/ω) φ(t)`, `φ(t) = t/(1 − exp(−2t − α(1−e^{−t}) − β(e^t−1)))`,
  `M = π/h`, whose abscissae land on the oscillation's zeros `kπ/ω` (sine) or `(k+½)π/ω`
  (cosine) double exponentially. `NIntegrate`'s Automatic/Levin dispatch now recognises an
  aligned Fourier integrand (`amp·{Sin|Cos}(ω x)` from lower limit 0, via a flattened
  reuse of the Levin kernel detector) and routes to it. **~200 digits in ~0.6 s** vs the
  minutes-and-wrong prior behaviour; agrees with `N[π/(2 E²), 200]` to ~194 digits.
  Gotcha fixed during development: the abscissa scale `M = π/h` must be formed in MPFR —
  a double-rounded `M` makes `x_k = kπ(1+ε)` and floors `sin(x_k)` at ~1e-16 in the tail,
  capping accuracy at 15 digits *and* preventing the node march from terminating.
- Verified: `x Sin[x]/(x²+4)` (→`π/2E²`), `Cos[x]/(x²+1)` (→`π/2E`), `Sin[x]/x` (→`π/2`),
  `x Sin[2x]/(x²+1)` (→`π/2 e^{-2}`) at 30/50/80/100/200 digits; `BesselJ[0,x] Exp[-x/10]`
  (→`1/√1.01`) still correct via the general fallback. New assertions in
  `tests/test_nint.c::test_high_precision`; all `nint_tests` pass; valgrind-clean
  (macOS dyld/libobjc baseline noise only).

## PossibleZeroQ: rigorous rational `False`, ladder early-exit, and a resolved "failure" (2026-07-02)

Another hardening round on `src/zero_test.c` (`PossibleZeroQ`), driven by the two
`Integrate\`GoursatAlgebraic`-verification cases in `POSSIBLE_ZEROQ_FAILURES.md`.

- **Case B1 was not a bug.** `PossibleZeroQ[D[Integrate[f, t, "GoursatAlgebraic"], t] - f]`
  returning `False` on the period-3 cyclotomic case is *correct*: the antiderivative is
  only valid on the positive real branch, and the residual is a genuine non-zero off it
  (`t = -3.4 → 0.20`). B1 shares the exact numeric signature of `Sqrt[x^2] - x` and
  `Log[x^2] - 2 Log[x]` — zero on the positive axis, non-zero on the negative — both of
  which must return `False`. No real/complex/exact-algebraic sampling policy can return
  `True` for B1 without breaking those; only genuine algebraic-function-field zero testing
  could, which is out of scope. The Goursat descent's positive-real-only numeric gate
  (`diff_back_ok`) is the correct domain-aware verifier and is retained. Reclassified in
  `POSSIBLE_ZEROQ_FAILURES.md`.

- **Rigorous `False` for pure rational functions (Zippel).** New
  `is_pure_rational_function` gate: when the whole input is a rational function of its free
  symbols over `Q` (exact rational coefficients, free symbols, `Plus`/`Times`/integer
  `Power` only), Stage-1 normalisation is complete, so a non-zero normalised numerator is a
  **rigorous** `False` — exact DeMillo–Lipton–Schwartz–Zippel, no sampling. `x + y`,
  `(x + 1)/(x - 1)`, `(x + y)^2 - x^2 - y^2` now decide `False` deterministically and
  instantly. Named algebraic constants (`GoldenRatio`) and any transcendental/radical atom
  are deliberately excluded, so identities like `Sin[2 x] - 2 Sin[x] Cos[x]` and
  `GoldenRatio^2 - GoldenRatio - 1` still route to the paths that return `True`.

- **Precision-ladder early-exit (case B2 performance).** Once a numeric residual has *both*
  shrunk geometrically and dropped far below any plausible cancellation floor
  (`ZT_DEEP_ZERO_BITS`), the identity is settled `True` without climbing the costly
  500/1000-bit rungs. Combined with reusing the (precision-independent) operand scale
  across rungs instead of recomputing it, `PossibleZeroQ[D[r, x] - f]` on the 9495-leaf
  parametric Goursat output drops from **4.8 s → 1.9 s** (well inside the descent's 6 s
  budget) with every verdict unchanged.

- **Tests.** `tests/test_zero_test.c` gains groups 12–14: Zippel-rigor cases (with
  determinism-across-hostile-seeds and not-misapplied-to-transcendentals guards),
  precision-ladder early-exit cases (deep radical denestings stay `True`, resolvable tiny
  non-zeros stay `False`), and a large-tree performance guard. Suite valgrind-clean

## Integrate: closed-form quadratic-radical integrator + parametric Goursat fixes (2026-07-01)

**`Integrate\`QuadraticRadicals` now emits ArcTan / ArcTanh / ArcSinh / ArcSin /
Log directly** instead of the Log-of-radical forms the Euler substitution
produced. `src/calculus/integrate_quadrad.c` gained a closed-form path (`qr_direct`,
tried before Euler, falling back to it on anything it cannot fully decompose): it
splits the integrand as `E(x) + H(x)/Sqrt[rad]`, integrates the rational part `E`,
partial-fractions `H`, and assembles the standard building blocks
`I_n = ∫x^n/Sqrt[rad]` (order reduction to `I0`) and `J(p) = ∫1/((x−p)Sqrt[rad])`
(→ `ArcTanh` if `rad(p)>0`, `ArcTan` if `rad(p)<0`). Each block is an exact
identity, so the assembled answer needs no verification. `I0` picks `ArcSinh`/`ArcSin`
only when `Sqrt[|disc|]` is an exact rational (so the derivative stays
Simplify-reducible), else a clean-cancelling `Log` form; a symbolic-leading-coefficient
pure-polynomial integrand is declined to the Euler path. E.g.
`∫1/Sqrt[x^2+1] = ArcSinh[x]`, `∫1/(x Sqrt[x^2+x+1]) = -ArcTanh[(2+x)/(2 Sqrt[1+x+x^2])]`.

**Parametric square-root Goursat (`R = F(x) (a x^2+b x+c param)^(...)`) no longer
hangs.** In `src/calculus/integrate_goursat.c`: (1) `monomialize_root_of_unity`
now requires `NumberQ[N[base]]` before denesting, so a symbolic parameter (the `k`
in `R = x(1-x)(1-k^2 x)`) no longer triggers the numeric root-of-unity heuristic
and injects spurious `Arg[k^2]`/`Abs[k^2]` terms; (2) the V4 fixed points are
`PowerExpand`-ed so `Sqrt[k^2]/k^2 → 1/k`, keeping the descent over `Q(k)` with a
single radical instead of carrying `Sqrt[k^2]` as a phantom second generator that
caused the genus-0 integrand to recurse. With these and the cleaner quadratic-radical
output, `Integrate[(1+k x)/((-1+k x) Sqrt[(1-x) x (1-k^2 x)]), x, Method ->
"GoursatAlgebraic"]` returns a verified antiderivative (was: non-terminating).

Verification note: the descent's differentiate-back gate should be
`PossibleZeroQ[D[result,x]-f]` (never `Simplify`), but `PossibleZeroQ` currently
misfires on the period-3 cases' cyclotomic-tower nested radicals and is too slow on
the large parametric outputs; the gate therefore uses a numeric differentiate-back
as an interim measure. Both `PossibleZeroQ` failure modes are written up in
`POSSIBLE_ZEROQ_FAILURES.md` (cases B1/B2). Tests: `integrate_quadrad_tests`,
`integrate_goursat_tests` (all green); no regressions across the integrate/rat/simp/

## Perf: FLINT-backed squarefree test in algebraic Factor (Trager) (2026-07-01)

`Factor[p, Extension -> α]` (Trager's algorithm, `src/poly/qafactor.c`) must find
the smallest linear shift `s` for which the norm `Norm(f(x − sα)) ∈ Q[x]` is
squarefree, testing squarefreeness by computing `gcd(R, R′)`. That GCD ran
through the classical content/pseudo-remainder `poly_gcd_internal`, which
sampling showed to be **~88 %** of total runtime on higher norm degrees (norm
degree = `deg f × deg minpoly`). `expr_qx_is_squarefree` now computes that GCD
via FLINT (`flint_multivariate_gcd` → `fmpq_mpoly_gcd`; the univariate rational
norm is squarely in FLINT's wheelhouse), falling back to the classical path
unchanged when FLINT returns NULL (numeric/non-polynomial input, or `USE_FLINT`
off). Only the GCD degree is consulted, so FLINT's monic normalisation is
irrelevant and factor outputs are byte-identical. Measured speedups:
`Factor[x^16 - 2, Extension -> 2^(1/4)]` 1.31 s → 0.076 s (**17×**),
`x^12 - 15 x^8 + 15 x^4 - 1` over Q(√2) 0.32 s → 0.011 s (**30×**),
`(x^2-2)(x^2-3)(x^2-5)(x^2-7)` over Q(√2) 0.20 s → 0.006 s (**32×**). After the
fix the profile is dominated by the genuine algebraic work — norm factorisation
(already FLINT `fmpz_poly_factor`) and the Trager Q(α) GCD lifts. The norm
*factorisation* was never the bottleneck; the squarefree-shift search was.

## Feature: transparent multivariate Factor via FLINT (2026-07-01)

`Factor[p]` on a genuine multivariate polynomial over Q now routes through
FLINT's `fmpq_mpoly_factor` (public `flint_polynomial_factor`, guarded fast path
at the top of `builtin_factor`, right after the univariate path and gated to the
plain single-argument form). The classical heuristic multivariate factoriser is
exponential in the variable count and effectively hangs on inputs such as
`Factor[x^99 - y^99]` (> 20 s); the FLINT path returns it in a few
milliseconds. Inputs that are *not* a multivariate polynomial over Q — a
denominator (rational function), a fractional or symbolic exponent, or a

coefficient** (highest total degree, deglex tie-break — Mathematica's
convention), with the discarded sign folded into a separated rational content
(the factoring context now uses `ORD_DEGLEX` so FLINT's term 0 is that leading
term). This makes the result *more* canonical than the classical output, which
spread factor signs inconsistently: `Factor[y^2 - x^2]` → `-(x - y)(x + y)`

factored product is identical; the evaluator re-sorts the `Orderless` `Times`, so
factor ordering is independent of FLINT. Two pinned classical-convention tests
(`test_facpoly.c`) were rebaselined to the verified-equivalent normalised forms.

## Fix: PossibleZeroQ Schwartz–Zippel false-zeros on algebraic-coefficient rationals (2026-06-30)

`PossibleZeroQ` (`src/zero_test.c`) wrongly returned `True` for several genuine
non-zeros that mix a free variable with an algebraic constant, e.g.
`-Sqrt[2] + (1 + u^3) Sqrt[2]` (= `Sqrt[2] u^3`), `-(-1)^(1/3) + (1 + u^3) (-1)^(1/3)`,
and `(u (-1)^(1/3))/(-1 + u^3)`.  The latter's verdict was even *flaky* — it
depended on the variable name (hence on the structural-hash sample seed).

Two independent root causes in the Stage-3 sampler, both fixed:

- **Sample points clustered near 0.** `sample_random_value` drew `n/d` with
  `|n| ≤ 64` but `d` up to `2^16`, so most samples had `|value| ≪ 1`.  A
  high-degree monomial (`u^3`) then collapses far below the operand scale and is
  misread as an identity.  Samples now have magnitude bounded **below by 1** (an
  integer part in `[1, 64]` plus a full-granularity fractional part, random
  sign); by the identity theorem an analytic identity true on `[1, 64]` is true
  everywhere, so no decision power is lost.
- **`magnitude_scale` mis-scored reciprocals.** The cancellation-threshold
  scale used `pow(base, |exponent|)`, so `Power[denom, -1]` was scored as
  `denom` instead of `1/denom`, hugely inflating the operand scale of any
  expression with a denominator and pushing genuine non-zeros below the noise
  floor.  Now uses the **signed** exponent value (new `expr_signed_double`).

A third, related false-zero was found and fixed in the same pass: `decide_numeric`'s
"residual never shrank across the precision ladder → assume zero (lenient)"
fallback misfired on **algebraic** expressions whose operand scale is inflated by
internal cancellation, e.g. `Sqrt[10^12+1] - 10^6` (≈ `5e-7`, scored as noise).
For a purely-algebraic expression MPFR honours precision exactly, so a residual
that stays above the machine-noise floor (`scale·2^-48`) while *not* shrinking is
a genuine non-zero, not a cancellation-hidden zero. New `is_algebraic_expr`
predicate gates this: non-shrinking algebraic residuals above the floor settle
`False`; transcendental heads (whose MPFR path may stay pinned to machine
precision) keep the lenient zero verdict. A true algebraic zero shrinks under MPFR
and never reaches this branch.

No regressions: `zero_test_tests` and the PossibleZeroQ-heavy suites
(simplify/solve/series/limit/intrat/rat/radical/trigrat/sum_rational/trig) show

## Progress: V4 cyclotomic-tower descent — denest fixed points + select clean involutions (2026-06-30)

Groundwork for the Legendre–Clausen integral `t/((t³+8) Sqrt[t³-1])` (and the
broader cyclotomic-root V4 class). `Solve[S == t]` returns each V4 involution's
fixed points via the quadratic formula, so they carry an opaque nested radical
`Sqrt[disc]` whose discriminant is a cyclotomic constant (e.g. `Sqrt[12 (-1)^(2/3)]`).
Left opaque, that radical reads as an independent variable: `R(t(u))·(1-u)⁴`
never reduces to a polynomial, the descended `lc`/`Q`/`gx` keep spurious
`(1-u)` denominators and `Sqrt[cyclotomic]` coefficients, and `integrate_in`
re-recognises a cubic-radical Goursat problem and recurses until the time guard
fires (the old ~12 s hang).

Two changes in `src/calculus/integrate_goursat.c`:

- **Denest cyclotomic fixed points** (`denest_const_radicals`): the discriminant
  is, by construction, a rational multiple of a root of unity `c (-1)^(p/q)`,
  whose square root IS a tower element (`Sqrt[12 (-1)^(2/3)] = 2 Sqrt[3] (-1)^(1/3)`).
  Detect the monomial numerically (`c = Rationalize[Abs[N[disc]]]`,
  `q = Rationalize[Arg[N[disc]]/Pi]`), verify it equals the discriminant exactly,
  then `PowerExpand` the monomial. The fixed points become explicit
  cyclotomic-tower elements and the descent stays rational over that field.
- **Select the cleanest involution** (`involution_score`): each projection is
  anti-invariant under two involutions; pick the one whose denested fixed points
  are real (smallest tower) rather than the first with a single finite fixed
  point. For `t³-1` this picks the real `1 ± Sqrt[3]` pairing, and that piece's
  genus-0 reduction now closes.

Net: the descent no longer grows an unbounded radical tower, and the lead
projection closes. The named integral still declines (~8 s, time-budget), now
for a *different* and newly-isolated reason: its remaining genus-0 pieces are
`∫ (rational over Q(ζ₁₂)) / Sqrt[quadratic over Q(√3)]`, which Mathilda's
elementary integrator does not yet close over a number field — the next target,
separate from Goursat. All existing `integrate_goursat` / `integrate_jeffrey`
tests pass; behaviour on rational-coefficient V4 integrands is unchanged.

## Fix: `Integrate` `GoursatAlgebraic` returns the antiderivative over the integrand's own radical (2026-06-30)

`Integrate[(x-1)/((x+2) Sqrt[x^3-1]), x]` (and other square-root pseudo-elliptic
integrands routed through the V4 / period-3 descent) produced a **correct but
unverifiable** antiderivative: the closed form carried a *nested* radical
`Sqrt[g]` in the Möbius-substituted coordinate (e.g. `Sqrt[w² − w]`,
`w = ((x-1)/(x+2))³`) instead of the integrand's own radical `Sqrt[x³−1]`. Since
the answer and the integrand then contained *different* radicals,
`D[result, x] // Simplify` could not reduce to the integrand (it would have to
relate two distinct nested roots), so the result failed a routine
differentiate-back verification even though it was correct.

By the Goursat construction `g` is always `R` times a perfect rational square,
so `Sqrt[g] = (rational)·Sqrt[R]`. A new output-normalisation pass in
`src/calculus/integrate_goursat.c` re-expresses every such nested square root
over the integrand's own `Sqrt[R]`:

- `reexpress_over_radical` collects the radicands `g` of `Sqrt[g]` in the result,
  and for each computes `h = Factor[Cancel[Together[g/R]]]`. Factoring exposes the
  square part; `PowerExpand[Sqrt[h]]` yields a *radical-free* rational `s` exactly
  when `h` is a perfect rational square (the gate: `s` contains no `Power[_,1/2]`
  and `s² = h`). The sole branch freedom — a global `±` sign — is fixed
  numerically by sampling `s·Sqrt[R] / Sqrt[g]`.
- The rewrite is **correct-by-construction** (the integrator knows the
  perfect-square relationship) and is adopted only if the existing
  differentiate-back guard still passes on the rewritten form. It is confined to
  the integrator — `Simplify` is unchanged (pulling a polynomial square factor out
  of a radical is branch-cut-sensitive in general, so `Simplify` rightly refuses
  it without assumptions).

Result: `Integrate[(x-1)/((x+2) Sqrt[x^3-1]), x]` now returns a `Log` in the
single radical `Sqrt[x³−1]` and `D[%, x] // Simplify` reduces to the integrand
(`= 0`). The V4 sqrt cases whose radical was already `Sqrt[R]` are unchanged
(the pass skips `g == R`). Regression: `test_period3` in
`tests/test_integrate_goursat.c` now also asserts the symbolic `Simplify`
verification, not only a numeric differentiate-back.

## Feature: `Integrate`GoursatAlgebraic` period-3 reduction for √(cubic) (Goursat 1887 §4) (2026-06-29)

`Integrate[(x-1)/((x+2) Sqrt[x^3-1]), x]` (and other `F(t)/Sqrt[t^3-1]` with the
`t^3-1` higher symmetry) now integrates — to
`-(2/3) ArcTan[Sqrt[((x-1)^3(1-((x-1)/(x+2))^3))/(x+2)^3]/(((x-1)/(x+2))^3-1)]`,
equal to the closed form `-(2/3) ArcTan[3 Sqrt[x^3-1]/(x-1)^2]` up to a constant
— in ~0.1s, under both `Method -> "GoursatAlgebraic"` and the default cascade.

This is the case Goursat's **Section 4** covers and the V4 Theorems 1–2
(`goursat_v4`) do **not**: there is an order-3 Möbius `S` fixing one of the four
ramification points `{roots(R), Infinity}` and cycling the other three, and `F`
is a non-trivial period-3 character `F(S) = α F` (`α = Exp[2πI/3]`). The V4
trivial projection is non-zero here, but the period-3 trivial projection
`F + F(S) + F(S^2)` vanishes.

**Reduction** (`goursat_period3` / `period3_reduce` in
`src/calculus/integrate_goursat.c`): with `z = (t-a)/(t-a1)` for the two fixed
points `a, a1` of `S`, the map becomes `z -> α z`, the differential
`dt/Sqrt[R]` becomes `dz/Sqrt[z(A z^3 + B)]`, and the α-character `F(t(z))`
becomes `z·φ(z^3)`; setting `x = z^3` yields the elementary
`(1/3) φ(x)/Sqrt[x(A x + B)]`, which the cascade integrates and the result is
back-substituted `x -> ((t-a)/(t-a1))^3`. Reuses the existing `cyclic_mobius`,
`fixed_points`, and `to_function_of_power`; the two fixed-point orientations are
tried to cover the α vs α² character. Avoids the cyclotomic `Together` blowup by
substituting the *original* (simple) `F` into `t(z)` before any projection, and
gates each stage with the numeric `is_cube_function` / `sample_clearly_nonzero`
checks; `Solve`'s fixed points are `Simplify`-ed (they come back as
un-simplified nested cube-roots-of-unity, e.g. collapsing to `1` and `-2`).
Correctness is enforced by `gs_core`'s `diff_back_ok` guard.

Scope: this implements the single-fixed-point period-3 reduction. The
cross-character A4 cases (Goursat **Section 5**, e.g. the Legendre–Clausen
`Integrate[t/((t^3+8) Sqrt[t^3-1]), t]`, which is a combination of period-3
characters under *different* fixed points) are not yet handled and still
decline. Tests in `tests/test_integrate_goursat.c::test_period3`.

## Perf: `Integrate`GoursatAlgebraic` fast numeric elementarity gate (2026-06-29)

Integrands whose radicand `R` has cyclotomic roots but that **fail the V4 /
period-3 eigenspace criterion** took ~12s to decline — they exhausted the 6
CPU-second `TimeConstrained` budget inside the criterion's `canonic`/`Simplify`
(Together/Cancel over the splitting field of `R`, which blows up
super-polynomially). The criterion only needs a *zero test* of a projection, so
it is now decided numerically on the **uncombined** projection before any
symbolic combination — turning a 12s timeout-decline into a fast decline:

- **Square-root (V4) case.** The trivial-character projection
  `P0 = (f0+f1+f2+f3)/4` is sampled at a spread of fixed rationals via
  `N[Abs[…], 30]` (a `sample_clearly_nonzero` helper). A decisively non-zero,
  finite sample declines instantly; only a vanishing sample authorises building
  the three non-trivial projections and descending.
- **Cube-root / fourth-root cases.** The discarded criterion eigenprojection
  (`H1` for `p=1/3`; `H0` for `p=2/3`; `H1,H2` for `p=1/4`; `H0,H1` for `p=3/4`)
  is built in a *raw* `evaluate`-only form (`eigenproj3_raw`/`eigenproj4_raw`,
  no `canonic`/`Simplify`) from the pre-`canonic` integrand source and sampled
  the same way — sidestepping both the `H` canonicalisation and the eigenproj
  `Simplify`.

The gate is a fast-reject only: a passing (zero-ish) sample falls through to the
identical symbolic path, and every emitted antiderivative is still independently
verified by `gs_core`'s `diff_back_ok` differentiate-back check, so a numeric
false-positive-for-zero cannot yield a wrong answer. A sample is counted as
evidence of non-vanishing only when `|·|` is both above the `1e-6` noise floor
**and** below `1e9` (a point landing on a pole of some `f_j` numericalises to
`Infinity` though the symbolic projection is identically zero, and must be
skipped, not mistaken for a genuine non-zero).

Measured: `Integrate[(x-1)/((x+2) Sqrt[x^3-1]), x, Method -> "GoursatAlgebraic"]`
12.1s → 0.04s; `Integrate[(1-x^3)^(1/3)/(1+x), …]` 12.0s → 0.16s; the
`integrate_goursat` test suite 16.8s → 4.7s. All elementary cases still compute
their full antiderivatives and all tests pass. Implementation in
`src/calculus/integrate_goursat.c` (`sample_clearly_nonzero`, `eigenproj3_raw`,
`eigenproj4_raw`, and the three gate sites in `goursat_v4`/`goursat_cubic`/
`goursat_quartic`).

## Fix: `Integrate`GoursatAlgebraic` positive-power radicals + a hang guard (2026-06-29)

`Integrate[(1 - x^3)^(1/3)/x, x, Method -> "GoursatAlgebraic"]` returned
unevaluated even though it is a textbook cube-root Goursat integral. It now
closes (to `(1-x^3)^(1/3) - ArcTan[…]/Sqrt[3] - …`), both under
`Method -> "GoursatAlgebraic"` and in the default `Integrate` cascade.

**Recognition: any rational exponent, not just radicals in the denominator.**
The recogniser previously matched only the exact exponents
`-1/2, -1/3, -2/3, -1/4, -3/4` — i.e. a radical already in the denominator. A
*positive*-power radical such as `(1-x^3)^(1/3)` was rejected outright. The
recogniser now accepts any rational exponent `n/d` whose reduced denominator is
`2`, `3`, or `4`, splitting `R^(n/d) = R^k · R^(-p)` where the radical order
`p = pnum/d ∈ {1/2,1/3,2/3,1/4,3/4}` (with `pnum = (-n) mod d`) and the integer
remainder `k = (n+pnum)/d` folds into the rational cofactor `F` as `R^k`. So
`(1-x^3)^(1/3)/x = (1-x^3)/x · (1-x^3)^(-2/3)` is recognised as the `p = 2/3`
case with `F = (1-x^3)/x`. The old exact-negative exponents are the `k = 0`
special case, so previously-working inputs are unchanged.

**Termination guard.** The cube-/fourth-root descents canonicalise rational
functions over the splitting field of `R`. When `R` has cyclotomic roots
(e.g. `1 - x^3`, whose roots are the cube roots of unity), the core
`Together`/`Cancel`/`Simplify` over that algebraic-number field can blow up
super-polynomially for unlucky cofactors — `Sqrt[1-x^3]/x` and
`(1-x^3)^(1/3)/(1+x)` spin for minutes inside `Cancel`/`Simplify`, even though
the same machinery dispatches `(1-x^3)^(1/3)/x` in a tenth of a second. Because
`Integrate`GoursatAlgebraic` sits in the *default* `Integrate` cascade, an
unbounded attempt would hang every `Integrate[]` of such an integrand. The
outermost attempt is now wrapped in a CPU-time budget (`TimeConstrained`, whose
SIGPROF abort can interrupt the in-flight C canonicaliser); on exhaustion the
integrator declines and the integral is left for the rest of the cascade.
Recursive genus-0 descents run inside that single timer. (Genuinely solving the
cyclotomic-cofactor cases such as `(1-x^3)^(1/3)/(1+x)` awaits a faster
algebraic-number `Together`/`Cancel`; today they decline rather than hang.)

## Fix: `Integrate`GoursatAlgebraic` branch-cut error in the square-root reduction (2026-06-29)

`Integrate[(x^4 + 2 x^3 - 4)/(x^2 Sqrt[(x^2-1)(x^2-4)]), x, Method -> "GoursatAlgebraic"]`
— a textbook pseudo-elliptic integral — returned unevaluated with a
`Power::infy: 1/0` message instead of its (elementary) antiderivative.

**Root cause.** In Goursat's V4 (square-root) reduction, each anti-invariant
eigenpiece descends to `Integrate[gx / Sqrt[Q], x]` with `Q` the *monic*
quadratic obtained by dividing the descended radicand `H` by its leading
coefficient `lc`, with `Sqrt[lc]` pulled out into the prefactor `(α-β)/(2
Sqrt[lc])`. That factoring is invalid across the branch cut: `Sqrt[lc] Sqrt[Q]
!= Sqrt[lc Q]` whenever the monic `Q` evaluates negative (here `lc = -2` and the
relevant `Q` is negative on the whole integration interval), so the assembled
antiderivative was wrong and the differentiate-back guard correctly rejected it
— but only after a stray `1/0`.

**Fix** (`src/calculus/integrate_goursat.c`):
- New `rebase_radical`: after the (fast, exact) monic integration, rewrite every
  `Q^p` in the result as `(lc Q)^p · lc^(-p)` — an identity that re-expresses the
  radical as the *true* `H = lc Q` (which, after back-substitution, collapses to
  the clean real `Sqrt[R]·rational`) while `lc` survives only as a constant that
  cancels the prefactor's `1/Sqrt[lc]`. No branch cut is straddled on a variable
  radicand, and the recursive reduction stays monic (Euler-I, integer
  coefficients) so it remains fast.
- The differentiate-back guard now verifies numerically (`diff_back_ok`):
  `D[result,x] - f` is sampled at several fixed rational points with `Abs`
  (robust to the `(x-β)` poles and the complex `R<0` stretch that defeat
  `PossibleZeroQ`'s real sampling on the un-simplified nested radical). It still
  rejects the Ferrari-quartic misfire (a wrong piece differs from `f` by O(1)).

The integral now closes in ~0.9s — both under `Method -> "GoursatAlgebraic"` and
via the default `Integrate` cascade — and differentiates back to the integrand.
The cube-/fourth-root reductions are unaffected (their `cval^(1/3)`, `cval^(1/4)`
roots are odd and single-valued on the reals). Regression test in
`tests/test_integrate_goursat.c`.

```mathematica
Integrate[Sqrt[x]/(1 + Sqrt[x]), x,
          Method -> {"DerivativeDivides", "Substitution" -> Sqrt[x]}]
(* -2 Sqrt[x] + x + 2 Log[1 + Sqrt[x]] *)
```

Without the sub-option, `"DerivativeDivides"` collects **every** `x`-dependent
subexpression as a candidate kernel and trials each. The
`"Substitution" -> u` sub-option restricts the search to the single kernel
`u(x)`, running both the direct-quotient and the Eliminate/Solve strategies on
it. Behaviour is strict, as for all named methods: if `u` does not close the
integral the call returns unevaluated, with no fallback to the automatic kernel
search (e.g. `Substitution -> x^2` on the non-elementary `Sin[x^2]`).

Implementation: a new `forced_kernel` parameter on `dd_core`
(`src/calculus/integrate_derivdivides.c`) plus the public entry
`integrate_derivdivides_with_sub`; `parse_method_option`
(`src/calculus/integrate.c`) gained a `{"<method>", subopt -> val, ...}` list
branch mirroring the existing `FactorInteger` list-form precedent
(`src/facint.c`), with the `method_from_string` mapping factored out for reuse.
Only `"Substitution"`, and only for `"DerivativeDivides"`, is recognised; any
other sub-option key is `Integrate::method`-invalid. Covered by
`test_substitution_option` in `tests/test_integrate_derivdivides.c`.

## Feature: Chebychev binomial integrator `Integrate`ChebychevAlgebraic` (2026-06-29)

A new, fast integration method for Chebychev binomial differentials
`x^p (a x^r + b)^q` (`p, q, r` rational, `a, b` free of `x`).  By Chebychev's
theorem (1853) such an integral is elementary iff one of `q`, `(p+1)/r`,
`q+(p+1)/r` is an integer, each giving a substitution that turns the integrand
into a *rational function*:

- **Type I** (`q ∈ ℤ`): `x = u^N`, `N = LCM(den p, den r)`.
- **Type II** (`(p+1)/r ∈ ℤ`): `u^s = a x^r + b`, `s = den q`.
- **Type III** (`q+(p+1)/r ∈ ℤ`): `u = x^r`, then `t^s = (a u + b)/u`.

```mathematica
Integrate[x^(1/3) (a Sqrt[x] + b)^3, x]                     (* Type I  *)
Integrate[x^(3/2) (a x^(5/6) + b)^(1/3), x]                 (* Type II *)
Integrate[x^(-5/6) Sqrt[x^(1/3) - 1], x]
(* 3 x^(1/6) Sqrt[-1 + x^(1/3)] - 3 ArcTanh[Sqrt[-1 + x^(1/3)]/x^(1/6)]  Type III *)
```

Recognition is a single structural scan, so the method is cheap enough to run
ahead of `DerivativeDivides`'s Eliminate/Solve search (it sits between
`LinearRatioRadicals` and `Weierstrass` in the cascade — stage 6 of 10).  The
reduced rational integral is handled by the `BronsteinRational` stage and the
result is back-substituted.  Types I and II introduce a single principal radical
and are correct by construction (no differentiate-back verification, which is
branch-fragile with symbolic `a, b`); Type III is branch-sensitive and uses a
**split** back-substitution `t -> (a x^r + b)^(1/s) / x^(r/s)` so every radical
in the antiderivative is the principal root of a sub-expression of the original
integrand.

Non-elementary binomials (none of the three quantities integral, e.g.
`(a x^2 + b)^(1/3)`, or `1/Sqrt[x^4 + 1]`) return `NULL`, so under the default
`Method -> Automatic` the cascade falls through to the remaining methods; only
the forced `Method -> "ChebychevAlgebraic"` form surfaces as unevaluated.

Also available as `Integrate[f, x, Method -> "ChebychevAlgebraic"]` and the
package head `Integrate`ChebychevAlgebraic[f, x]`.

Implementation: `src/calculus/integrate_chebychev.{c,h}` (modelled on
`integrate_linrad.c` / `integrate_linratiorad.c`), wired into the cascade,
`method_from_string`, the `Method` switch and warning, and `integrate_init`
(`src/calculus/integrate.c`).  Covered by `tests/test_integrate_chebychev.c`
(all of `chebychev_problems.tex` problems 1 and 3, the worked Type III example,
problem-4 non-elementary cases, and method plumbing).

## Feature: Goursat pseudo-elliptic integrator `Integrate`GoursatAlgebraic` (2026-06-29)

A new integration method for **pseudo-elliptic integrals** `F(t)/R(t)^p`
(`F` rational, `R` a polynomial, `p ∈ {1/2, 1/3, 2/3, 1/4, 3/4}`) — integrands
that live on a positive-genus algebraic curve `y^n = R(t)` (`n = den p`) yet
integrate elementarily.  Goursat's 1887 algorithm and its cube-/fourth-root
generalisations (Blake 2026) detect this via a Möbius automorphism cycling the
roots of `R`, which splits the integrand into eigencomponents; when the
obstructive components vanish the rest descend to genus-0 curves with rational
antiderivatives.

```mathematica
Integrate[t/Sqrt[(t^2-1)(t^2-4)], t]      (* p=1/2, V4 — elementary *)
Integrate[t^2/(t^3-1)^(1/3), t]           (* (t^3-1)^(2/3)/2 *)
Integrate[t^2/(t^3-1)^(2/3), t]           (* (t^3-1)^(1/3)   *)
Integrate[t^3/(t^4-1)^(1/4), t]           (* (t^4-1)^(3/4)/3 *)
Integrate[t^3/(t^4-1)^(3/4), t]           (* (t^4-1)^(1/4)   *)
```

Criteria by exponent (else obstructed → `NULL`): `p=1/2` the Klein four-group
`V4` of root-pairing involutions, elementary iff the trivial-character
projection `F^(0)` vanishes (`R` cubic or quartic); `p=1/3, 2/3` an order-3
cyclic Möbius, elementary iff the `ω`-eigencomponent (`H1` at 1/3, `H0` at 2/3)
vanishes (`R` cubic, or quadratic); `p=1/4, 3/4` an order-4 cyclic Möbius on
roots in **harmonic position** (cross-ratio `-1`), elementary iff two
eigencomponents vanish (`V1,V2` at 1/4; `V0,V1` at 3/4).

The method orchestrates existing builtins (`Solve` with `Cubics/Quartics ->
True`, `Together`/`Cancel` over algebraic extensions, `PossibleZeroQ`,
`Coefficient`, `Numerator`/`Denominator`), recurses `Integrate` on the rational
reductions, and back-substitutes.  `Simplify` collapses roots-of-unity
identities (e.g. `1 + ω + ω^2 → 0`) that `Cancel`/`Together` leave intact.  It
runs ahead of `DerivativeDivides`/Risch in the cascade (stage 7 of 11), filling
a gap no other radical method covers (cubic/quartic radicands).

A differentiate-back guard `PossibleZeroQ[D[result, t] - f]` runs before
returning: the eigenspace zero-test can misfire on deeply nested radical roots
(an irreducible quartic solved via Ferrari) where catastrophic cancellation
numericalises a genuinely nonzero projection to ~0, so the guard rejects such
cases (declining rather than emitting a wrong answer).  Every canonical
factorable/biquadratic example passes it trivially.

Also available as `Integrate[f, x, Method -> "GoursatAlgebraic"]` and the
package head `Integrate`GoursatAlgebraic[f, x]`.

Implementation: `src/calculus/integrate_goursat.{c,h}`, wired into the cascade,
`method_from_string`, the `Method` switch and warning, and `integrate_init`
(`src/calculus/integrate.c`).  Reference: `goursat/GoursatAppendix.wl` and the
two preprints.  Covered by `tests/test_integrate_goursat.c` (square-, cube-, and
fourth-root worked examples, obstructed/non-harmonic declines, method plumbing).

## Fix: Plot adaptive sampler density was inverted (2026-06-29)

`Plot[Sin[x], {x, 0, 2 Pi}, Mesh -> All]` placed **more** sample points on the
straight zero-crossings than on the curvy peaks — the opposite of curvature-
proportional sampling. Root cause: the absolute screen-space chord-**length**
cap (`MAX_CHORD_FRAC`), which keys on *slope* not curvature, was tight enough
(≈1/44 of the frame) to out-compete the curvature (vertical-gap) test on
ordinary moderately-sloped curves, pouring samples into the steep-but-straight
stretches while the loose curvature tolerance (`FLAT_TOL`) let the actually-
curvy peaks pass as "flat" at the 50-point grid scale.

Fix (`src/graphics/sampling.c`): tightened `FLAT_TOL` `0.0025 → 0.0006` so the
curvature test is the primary driver (density now tracks `|f''|`), and relaxed
`MAX_CHORD_FRAC` `0.0225 → 0.08` to a loose backstop that only binds on genuine
near-vertical gaps (asymptotes). Verified across `Sin` (peaks now ~2× denser
than zero-crossings), `x^3` (densest at the curved ends), `x^2` (uniform —
constant curvature), a straight line and `Exp` (no wasted refinement), the
`Runge` spike (peaked at centre), and `Sin[22 x]` (oscillation still fully
resolved). Total point counts dropped (less wasteful) while the shape is now
correct. Clean build; the chord cap's asymptote/`Log` protection is preserved.

## Arithmetic of `SeriesData` (2026-06-29)

`Plus`, `Times`, and `Power` now combine `SeriesData[x, x0, {coefs}, nmin,
nmax, den]` objects, matching Mathematica. `Divide` and `Subtract` are covered
automatically because they reduce to `Times[a, Power[b,-1]]` and
`Plus[a, Times[-1, b]]`. Previously a `SeriesData` was opaque to the arithmetic
builtins, so `series + 1`, `x series`, `series^3`, `s1 s2`, `s1/s2` all stayed
unevaluated.

- **Plus / Times** fold every operand into a single `SeriesData` about the
  common `(x, x0)`, truncated to the **minimum** O-term order of the operands.
  A non-series operand free of `x` folds into the `a0` coefficient (fast path
  for `series + scalar`); the bare variable, polynomials, and transcendental
  functions are series-expanded with the full engine and combined. Operands
  about a **different** variable or expansion point are left as a symbolic
  `Plus`/`Times` (unevaluated), as in Mathematica.
- **Power** handles integer exponents (negative via series inversion) and
  scalar exponents free of the series variable directly. An exponent that
  depends on the series variable — or an ordinary base raised to a series
  exponent, e.g. `2^Series[x,{x,0,3}]` — is evaluated as `Exp[exp*Log[base]]`
  and re-expanded.
- **Mixed exact/approximate precision** works per-coefficient: coefficient
  arithmetic flows through the evaluator (`Plus`/`Times`), so integer, bigint,
  rational, machine-real, and MPFR coefficients mix with the usual contagion.
  Adding a real to a series folds it into `a0` as a real while the remaining
  coefficients stay exact.

Verified end-to-end: `Series[Exp[x],{x,0,2}]+1` → `2+x+x^2/2+O[x]^3`;
`x Series[Sin[x],{x,0,5}]` → `x^2-x^4/6+x^6/120+O[x]^7`;
`Series[Exp[x],{x,0,2}] Series[Exp[x],{x,0,3}]` → `1+2x+2x^2+O[x]^3` (order is
the min); `Series[Exp[x],{x,0,2}]^3` → `1+3x+9x^2/2+O[x]^3`;
`2^Series[x,{x,0,3}]` → `1+Log[2] x+Log[2]^2/2 x^2+Log[2]^3/6 x^3+O[x]^4`;
`Series[Exp[x],{x,0,3}]^Series[x,{x,0,3}]` → `1+x^2+O[x]^4` (= `Exp[x^2]`).

**Implementation.** Public entry points `series_combine_plus`,
`series_combine_times`, `series_power`, and the `is_series_data` predicate in
`src/calculus/series.c` (declared in `series.h`), driving the existing
`SeriesObj` algebra (`so_add`, `so_mul`, `so_pow_int`, `so_pow_expr`, `so_inv`,
`so_align_den`). A new `SeriesData` leaf case in `series_expand` lets a series
embedded in an expression (the `Exp[exp*Log[base]]` rewrite) flow through the
expander. Cheap `is_series_data` dispatch added before the grouping loops in
`builtin_plus` / `builtin_times` (ahead of inexact contagion so it cannot
re-`N` the structural integers) and after the identity reductions in
`builtin_power`. Tests in `tests/test_series.c`; valgrind shows no new leaks vs.
the macOS dyld baseline. Documented in
`docs/spec/builtins/power-series.md`.

Two guards keep folding faithful and terminating. A non-series operand whose
dependence on the series variable is *non-analytic* (a piecewise-constant head
such as `Floor`, `Arg`, `Sign`, ...) cannot be power-series expanded, so it
makes the combine bail to a symbolic `Plus`/`Times` instead of looping in
`series_expand` — this preserves the branch-point wrapper
`Times[(-1)^Floor[...], SeriesData[...]]` produced by `Series[ArcCosh[x+1], ...]`
and friends. And for a series at `Infinity` (where the formal variable is
`1/x`), a factor that is structurally free of `1/x` but still depends on the
underlying `x` (e.g. `E^x` in `Times[E^x, SeriesData[1/x, ...]]` from
`Series[ExpIntegralEi[x], {x, Infinity, n}]`) is *not* treated as a constant and
stays outside the series, matching Mathematica.

## Differentiation and integration of `SeriesData` (2026-06-29)

`D` and `Integrate` now act on `SeriesData[x, x0, {coefs}, nmin, nmax, den]`
objects term-by-term, returning a new `SeriesData`, matching Mathematica.
Previously `D[SeriesData[...], x]` fell through to the generic chain-rule
fallback (producing a meaningless `Derivative[1,0,...][SeriesData][...]`) and
`Integrate[SeriesData[...], x]` matched no method and stayed unevaluated.

- **Differentiation w.r.t. the series variable** applies the power rule: the
  coefficient of `(x - x0)^((nmin+i)/den)` is scaled by `(nmin+i)/den` and the
  exponent drops by one. `nmin` and `nmax` each decrease by `den`; the
  differentiated constant becomes a leading zero and is trimmed.
- **Integration w.r.t. the series variable** scales each coefficient by
  `den/(nmin+i+den)` and raises the exponent by one; `nmin` and `nmax` each
  increase by `den`. The integration constant is `0` (as in Mathematica). A
  nonzero `(x - x0)^-1` residue term would integrate to a `Log` that
  `SeriesData` cannot represent, so `Integrate` is left unevaluated in that
  case.
- **W.r.t. a different variable** (when `x0` is free of it) both operations
  thread into the coefficients, leaving the powers of `(x - x0)` unchanged.

Verified end-to-end: `Integrate[Series[Exp[x],{x,0,8}], x]` →
`x + x^2/2 + ... + x^9/362880 + O[x]^10`; `D[Series[Exp[x],{x,0,8}], x]` →
`1 + x + ... + x^7/5040 + O[x]^8`; the two round-trip; `1/x^2 + 1` integrates to
`-1/x + x + O[x]^5` while a true `1/x` residue stays unevaluated.

**Implementation.** Two helpers `series_differentiate` / `series_integrate`
(plus a private `so_from_seriesdata` reader) in `src/calculus/series.c`, reusing
the existing `SeriesObj` machinery (`so_alloc`, `so_set_coef`,
`so_trim_leading`, `so_to_expr`); one-line dispatch on `SYM_SeriesData` added to
`compute_deriv` in `src/calculus/deriv.c` and to `builtin_integrate` in
`src/calculus/integrate.c`. Tests in `tests/test_series.c`; valgrind-clean
(no new leaks vs. the macOS dyld baseline). Documented in
`docs/spec/builtins/power-series.md`.

## Infra: FLINT engine — M2, GCD over a cyclotomic field Q(ζ_n) + generalised single-generator core (2026-06-30)

Generalised the M2 number-field GCD from "Q(√d) only" to **any simple algebraic
extension Q(s)** given a minimal polynomial, and added the **cyclotomic** case
Q(ζ_n) on top of it.

**Generalised core** (`absfield_gcd_core` in `src/poly/flint_bridge.c`). A
`GenSpec` maps an input atom to a power `s^k` of the generator; the core converts
each operand to an `fmpq_mpoly` in `{x, s}`, reduces each x-coefficient modulo
the minimal polynomial with `fmpq_poly_rem` (replacing the bespoke `s²=d` fold),
runs `gr_poly_gcd` over `Q(s)`, and reads the monic result back in the power
basis `{1, s, …, s^{m-1}}` (`nf_elem_get_fmpq_poly`). The Q(√d) path is now a
thin wrapper (`sqrt_atom_power`, minpoly `y²−d`); its existing tests are
unchanged and still pass, confirming the refactor.

**Cyclotomic** (`flint_cyclotomic_gcd`). Recognises roots of unity
`(-1)^(p/q) = Power[-1, Rational[p,q]]`, takes the field `Q(ζ_N)` with
`N = 2·lcm(q)` and minimal polynomial `Φ_N` (`fmpz_poly_cyclotomic`), maps each

## Infra: FLINT engine — M2, radical-tower collapse Q(√d₁,…,√d_r) (2026-06-30)

Completed the algebraic-extension **tower** case (plan §3.2.1): GCD over
`Q(√d₁,…,√d_r)`, r ≥ 2, by collapsing the tower to a single primitive element θ.

**Method** (`tower_build` in `src/poly/flint_bridge.c`) — fully-rational linear
algebra, no resultants. Each `√d_j` multiplies by a known matrix in the product
(bitmask) basis `{∏ √d_i^{e_i}}`, so `T = Σ c_j·(mult-by-√d_j)` is multiplication
by `θ = Σ c_j √d_j`. The columns `{1, θ, …, θ^{D−1}}` (D = 2^r) form `P`; when `P`
is invertible θ is primitive, the absolute minimal polynomial is read from
`θ^D = P·(P⁻¹θ^D)`, and each `√d_i`'s image in `Q(θ)` is `P⁻¹` applied to its
product-basis unit vector. Independent retries over small coefficient vectors
`c`; bails if the radicals are dependent (no full-degree θ).

The GCD runs through the shared `absfield_gcd_core` (now generalised to an
atom→image-polynomial `GenSpec` and a `CoefBuilder` readback). The tower readback
maps the power-basis result back through `P` into the **product basis**, so
coefficients print as clean single radicals — e.g.
`gcd((x−√2)(x−√3)(x−1), (x−√2)(x−√3)(x−2)) = x² − (√2+√3)x + √6` (the `√2·√3`
cross term reconstructed as `√6`), and `gcd((x−√2−√3)(x−1), (x−√2−√3)(x−2)) =
x − √2 − √3`. `` Flint`GCD `` dispatches rational → Q(√d) → Q(ζ_n) → tower.

## FLINT engine — parametric radical GCD/division Q(t)(√k) (M3, 2026-07-01)

GCD and exact division over the **parametric radical field** `Q(t_1,…,t_p)(√k)`
where the radicand `k` is a free *symbol* (parameter), not an integer — the
Goursat blocker ring `Q(a,b,k)(√k)[x]`.

**Key simplification.** `√k` is transcendental over `Q(t)` with `k = (√k)²`, so
`Q(a,b,k)(√k)` **is** the rational function field `Q(a,b,√k)` — no genuine
algebraic relation to reduce. The regime collapses to ordinary multivariate GCD
over `Q` via the substitution `√k → S, k → S²`, `fmpq_mpoly_gcd` (already
rigorous and fast), then readback `S → √k` (folding `S² → k` under evaluation).
So *no `fq_nmod` residue field, CRT, or Encarnación outer loop* is needed for
this regime — those stay for genuine constant-coefficient minimal polynomials
(later milestone).

`flint_multivariate_divexact` (`fmpq_mpoly_divides`); `ps_detect` recognises the
single-symbolic-radicand regime (bails on a second radical, cube root, root of
unity, or two distinct radicands). Wired into `flint_extension_gcd` and
`flint_extension_divexact`, so `PolynomialGCD` / `Cancel` / `Together` reduce over
`Q(a,b,k)(√k)` **sub-second** (the multivariate-`PolynomialExtendedGCD` /
`exact_poly_div` chain that hung before). `Cancel[(x²−k)/(x−√k)] = √k + x`;
`Cancel[((x−√k)(a x²+b x+k))/((x−√k)(x³+a k))] = (a x²+b x+k)/(x³+a k)` (verified
via `PossibleZeroQ`).

denominator is a numeric constant (the unit `−1` from a GCD FLINT normalised
monic in its own variable order rather than in `x`), the result is now
distributed (`√k + x`, not `−(−√k − x)`); guarded against a use-after-free in the
numeric-check.

## FLINT parametric resultant; Rothstein-Trager diagnosis (2026-07-01)

**Why Rothstein–Trager chokes on the Goursat inner integral even though the
answer is a single ArcTan:** RT computes the log part from the roots of a
*resultant* `R(t) = Res_x(A − t·D', D)`. The answer being a simple ArcTan means
`R(t)` has a conjugate root pair — but RT still has to *compute* `R(t)` first, and
its cost is set by `deg D` and the *size of D's coefficients*, not by the answer.
Here `deg D = 4` (the √quadratic was rationalised by a `QuadraticRadicals`/Euler
substitution, doubling the degree) and D's coefficients are large polynomials in
`a, b, k` with `k^(1/2)…k^(9/2)`. The classical symbolic subresultant PRS on a
7×7 Sylvester system with those parametric-radical entries blows up — measured
**>90 s** just for the one resultant.

**Fix:** `flint_parametric_sqrt_resultant` (`src/poly/flint_bridge.c`) computes
`Res_var(a, b)` over `Q(t..)(√k)` via the same `√k → S, k → S²` collapse used by
the parametric GCD, then FLINT's `fmpq_mpoly_resultant`; readback `S → √k`. Hooked
into `builtin_resultant` for the parametric regime. The `k^(9/2)`-coefficient
resultant that timed out now returns in **~0.5 ms**, verified to agree exactly
with the classical resultant evaluated at a concrete `k`.
`flint_multivariate_resultant` (plain `Q`) added as the underlying primitive.

**Tests:** `test_parametric_resultant` in `tests/test_flint_bridge.c`
(`Res[x−√k, x+√k, x] = 2√k`; shared-root → 0; numeric agreement with the
classical resultant on the `k^(9/2)` case). Regressions green; valgrind-clean.

**M7 status (honest):** the two named integrals still do not close. The inner
`∫G/√quadratic` is a *chain* of bottlenecks over `Q(a,b,k)(√k)`: `canonic`
(Cancel/Together — fixed) → `Apart`/`Factor` of the degree-4 denominator (number-
field factoring over a parametric field — the current blocker, hangs before RT) →
RT resultant (now fixed) → resultant factoring / root-finding. The real
architectural fix is to integrate `∫rational/√quadratic` *directly* (a low-degree
reduction to ArcTan/ArcTanh) instead of rationalising it to a degree-4 rational
and invoking full Rothstein–Trager — which would sidestep the whole chain.

## Apart Bézout partial fraction over radical fields; rational-function-field path confirmed (2026-07-01)

fraction solve builds an S×(S+1) matrix and `RowReduce`s it; over `Sqrt[k]`-laden

`∫1/((x−√k)(x+√k))dx = −ArcTanh[x/√k]/√k` and similar close end-to-end;

## Cancel/Together buffer-leak fixes + cube-root Goursat leak analysis (2026-07-03)

Fixed three instances of a single memory-leak bug class in `src/rat.c`: a
locally `malloc`'d `Expr**` array is passed to `expr_new_function`, which
*copies* the array (memcpy) and adopts the element pointers but not the buffer —
so the caller must `free` the buffer. Three sibling branches that build a
`Times`/`Plus` from a multi-element buffer forgot the `free` (while their
`n == 1` counterparts did it):

- `rat_symbolic_content` (the `n_running >= 2` return) — the dominant source; it
  fires on every `Cancel`-over-extension whose symbolic content has ≥2 algebraic
  factors (e.g. powers of `κ = (lead R)^(1/3)` and `ω` in the cube-root Goursat
  descent's `canonic`).
- `rat_strip_symbolic_common` (the `num_nf >= 2` branch).
- `rat_div_distribute` (the `Plus` branch's `return`).

All three are exercised by `Integrate[(2-(k+1)x)/((1-(k+1)x)(x(1-x)(1-kx))^(1/3)),
x, Method -> "GoursatAlgebraic"]`; after the fix these frames no longer appear in
any valgrind leak stack for that integral. Affected suites (`rat`, `intrat`,
`integrate_goursat`, `extension_*`) all pass; the integral still solves.

Also corrected a stale diagnosis: the previously-suspected "cube-root Goursat
leak" reproduced via an NDJSON pipe input with *unescaped* quotes
(`Method -> "GoursatAlgebraic"` instead of `\"..\"`), which truncates the `expr`
string mid-expression — the leak measured there was the parser's handling of the
malformed/truncated `->` input, not the integrator. The properly-escaped
integral has no parse-symbol leak. A larger, ASLR-nondeterministic per-call leak
remains in the shared algebraic-extension arithmetic (poly GCD/LCM/quotient and
`Expand` over `Q(α)`, reached from `together_recursive_ext`/`cancel_with_extension`/
`get_coeff`); it does not reproduce when the same sub-expressions are round-tripped
through print+reparse in isolation (the top-level evaluator normalises them first),
so it needs a C-level harness that drives the internal `canonic` on the un-normalised
in-memory trees — deferred.

## Associations: the `<| ... |>` data structure (2026-07-05)

Mathilda gains a first-class **Association** data structure modelled on the
Wolfram Language, with `<|key -> value, ...|>` literal syntax. Associations are
ordinary expressions (`Association[Rule[k,v], ...]`) with unique,
insertion-ordered keys (first occurrence fixes position, last fixes value), so
the whole generic toolchain (`Length`, `Map`, `ReplaceAll`, `===`, `FullForm`)
works on them unchanged. Bulk operations are backed by a transient
open-addressing hash index (keyed by `expr_hash`/`expr_eq`), giving amortised
`O(n)` construction, de-duplication, grouping and lookup instead of the naive
`O(n^2)`.

New builtins (all `Protected`; `AssociateTo` is `HoldFirst`), documented in
[`builtins/data-structures.md`](../builtins/data-structures.md):

- **`Association`** / `<|...|>` — construct/normalise (splices lists of rules and
  nested associations; de-duplicates keys).
- **`AssociationQ`**, **`Keys`**, **`Values`**.
- **`Lookup`** — single key (with optional default → `Missing["KeyAbsent", key]`),
  or a list of keys resolved with one index build (`O(n+m)`).
- **`KeyExistsQ`**, **`KeyDrop`**, **`KeyTake`**, **`KeyValueMap`**.
- **`AssociationThread`**, **`Counts`**, **`GroupBy`**, **`Merge`**.
- **`AssociateTo`** — in-place add/update on a held symbol.

Also: `assoc[[key]]` / `assoc[[Key[k]]]` / `assoc[[i]]` indexing via `Part`
(`src/part.c`); `Normal[assoc]` → list of rules (`src/calculus/series.c`);
`<|...|>` StandardForm, TeXForm and KaTeX (notebook) printing
(`src/print.c`, `src/print_latex.c`). Implementation in `src/assoc.c`, parser
support in `src/parse.c`. Covered by 41 end-to-end tests in
`tests/test_association.c` (0 leaks under macOS `leaks`).

## Fix: KeyValuePattern (and Except) in DownValue LHSs (2026-07-05)

Eighteenth increment — a matcher/dispatch fix.

- `f[KeyValuePattern[{...}]] := ...` now matches: associations can be
  destructured directly in a function definition (e.g.
  `area[KeyValuePattern[{"w" -> w_, "h" -> h_}]] := w h`). The same fix also
  repairs `Except` in a DownValue LHS.
- Root cause: the §3.5 DownValue dispatch filter (`src/symtab.c`,
  `pattern_arg_head_canon`) treated a first-argument pattern head of
  `KeyValuePattern`/`Except` as a literal head, so it skipped the rule when the
  input's first-arg head differed (e.g. `Association`). These heads are now
  treated as wildcards (return NULL), so the filter never skips them — strictly
  more conservative, so it cannot cause missed matches elsewhere. (Matching
  itself always worked via MatchQ/Cases; only the DownValue fast-path filter was
  wrong.)
