# Changelog: week of 2026-07-06 (Mon) – 2026-07-12 (Sun)

Feature additions and fixes recorded during this week.

## Integrate: complex line / contour integration (2026-07-07)

New `src/calculus/integrate_line.c` adds contour integration in the complex
plane, wired into `Integrate`'s definite-integral dispatch (`integrate.c`) and
exposed explicitly as `Integrate\`LineIntegral` and `Integrate\`PathSingularPoints`.

- **New surface.** `Integrate[f, {x, a, b}]` with a non-real endpoint, and the
  polyline form `Integrate[f, {x, z0, z1, …, zn}]`, evaluate the contour integral
  of `f` along the straight segments `z0 → z1 → … → zn`.  Real-endpoint two-point
  specs are unchanged (still the real-axis Newton-Leibniz path).
  - `Integrate[1/x, {x, 1 - I, 2 + 3 I}]` → `Log[2 + 3 I] - Log[1 - I]`
  - `Integrate[z^2, {z, 0, 1 + I}]` → `-2/3 + 2/3 I`; `Integrate[Cos[z], {z, 0, I}]`
    → `I Sinh[1]`; `Integrate[1/(z^2 + 1), {z, 0, 1 + I}]` → `ArcTan[1 + I]`.
- **Real-parameter reduction.** Each segment `a → b` is parametrised by a real
  `t ∈ [0, 1]`, `γ(t) = a + t(b − a)`.  The antiderivative is computed in the
  friendly variable `x`; on-path singularities become real roots `t* ∈ (0, 1)` of
  `Denominator[Together[f(γ(t))]]`, and endpoint singularities are resolved by
  **real one-sided limits in `t`** (so a complex-ray approach is a limit the
  existing `Limit` engine can take, e.g. the integrable branch-point endpoint of
  `Integrate[1/Sqrt[z], {z, 0, 1 + I}]` → `2 Sqrt[1 + I]`).
- **On-path singularity detection / divergence.** `Integrate\`PathSingularPoints[
  1/z, {z, -1 - I, 1 + I}]` → `{0}`; a genuine singularity on the contour makes
  the integral divergent — `Integrate[1/z, {z, -1 - I, 1 + I}]` warns
  `Integrate::idiv` and is left unevaluated.
- **Branch-correct residues.** For rational integrands whose antiderivative is a
  sum of `Log`/`ArcTan`/`ArcTanh` of **affine** arguments, the continuous change
  along a segment is computed as a principal `Log` of a ratio `Log[(u(b))/(u(a))]`
  — exact because a straight segment subtends `< π` at any off-path point.  Closed
  contours therefore close: `Chop[N[Integrate[1/z, {z, 1, I, -1, -I, 1}]]]` →
  `0.0 + 6.28319 I` (`2 Pi I`), and closed contours of entire functions vanish.
- **Never-wrong.** Every segment value is numerically cross-checked (loose 1e-3
  tolerance) against a complex quadrature of `f(γ(t)) γ'(t)`; an uncorrectable
  branch crossing leaves the integral unevaluated rather than return a wrong
  branch.
- Attributes `Protected, ReadProtected` on both builtins; docstrings via
  `symtab_set_docstring`.  Tests in `tests/test_integrate_line.c` cover detection,
  segment values, complex-plane limits, divergence, polyline path independence,
  and the `2 Pi I` residue.  Memory: the only leaks under valgrind are the
  pre-existing `Solve` / Goursat (`gs_guarded`) leaks that any `Integrate`/`Solve`
  caller triggers; no allocation in `integrate_line.c` leaks.

## Integrate: continuous branch-form definite integrals + `Integrate::idiv` (2026-07-07)

Follow-up to the Newton-Leibniz definite integrator: a class of *continuous*
integrands whose antiderivative is a branch form now evaluates, and divergent
integrals warn like Mathematica.  All changes in
`src/calculus/integrate_newton_leibniz.c`.

- **Continuous Weierstrass branch forms now close.** `Integrate[1/(2+Cos[x]),
  {x,0,2Pi}]` → `(2 Pi)/Sqrt[3]`, `Integrate[1/(5+4Cos[x]),{x,0,2Pi}]` →
  `2/3 Pi`, `Integrate[1/(3+2Sin[x]),{x,0,2Pi}]` → `(2 Pi)/Sqrt[5]`. The
  Weierstrass integrator emits an *already-continuous*, `Floor`-corrected
  antiderivative, so `F(b)−F(a)` by substitution is exact.  Previously these
  bailed because pole detection ran on `F` too and tripped on the spurious
  `Tan[x/2]` denominator.  Pole detection now runs on the **integrand `f`
  only** (a continuous `f` has a pole-free antiderivative).
- **"Never-wrong" continuity guard.** A finite value from an antiderivative
  that could carry a genuine uncorrected branch jump — one containing a step
  head (`Floor`/`Ceiling`/`Round`/`Sign`/…) or an inverse-trig node over a
  pole-bearing trig head (`ArcTan[Tan[x/2]…]`) — is accepted only if a numeric
  `NIntegrate` cross-check agrees (loose 1e-3 tolerance); otherwise the integral
  is left unevaluated.  The exact symbolic value is returned, never the numeric.
- **Spurious-pole numeric resolution.** When `Solve` cannot place a trig
  denominator's roots (e.g. it returns complex `ArcCos[2]` conditional
  solutions for the nowhere-zero `2+Cos[x]`), a numeric sign-change scan of the
  denominator over `(a,b)` settles whether a real interior pole exists; if none,
  integration proceeds under the cross-check guard.
- **`Integrate::idiv` warning.** A genuinely divergent definite integral now
  emits `Integrate::idiv: Integral of <f> does not converge on {a, b}.` and is
  left **unevaluated** (Mathematica-faithful).  Previously `Integrate[1/x,
  {x,-1,1}]` returned `Indeterminate` and `Integrate[1/x^2,{x,-1,1}]` returned
  `Infinity`, both silently; these now warn and echo the unevaluated integral.
- A boundary substitution that merely *contains* a singular sentinel (e.g.
  `ArcTan[ComplexInfinity/Sqrt[3]]` from evaluating a Weierstrass form exactly
  on a `Tan[x/2]` pole) is no longer accepted as a closed form — such partial
  intervals fall back to a one-sided limit and, if unresolved, stay unevaluated
  rather than emitting `ComplexInfinity`-laced output.

Tests: `tests/test_integrate_newton_leibniz.c` gains `test_continuous_periodic`
(5 exact cases) and 3 NIntegrate stress rows; `test_divergent` updated for the
unevaluated-echo policy.

## Integrate: definite integration via Newton-Leibniz (2026-07-07)

`Integrate` now evaluates definite and iterated integrals by the fundamental
theorem of calculus.  New module `src/calculus/integrate_newton_leibniz.c`.

- **`Integrate[f, {x, xmin, xmax}]`** — antidifferentiates `f` with the existing
  indefinite cascade to get `F`, then forms `F(xmax) − F(xmin)` through the
  `Limit` engine.  Boundaries at infinite endpoints use a plain limit; interior
  poles use one-sided limits (`Direction -> "FromBelow"/"FromAbove"`) after
  splitting the interval; ordinary finite endpoints use direct substitution.
  Improper integrals converge (`Integrate[1/Sqrt[x], {x, 0, 1}] -> 2`,
  `Integrate[1/(1 + x^2), {x, 0, Infinity}] -> Pi/2`); divergent ones surface
  `Infinity` / `ComplexInfinity` / `Indeterminate`
  (`Integrate[1/x^2, {x, -1, 1}] -> Infinity`,
  `Integrate[1/x, {x, -1, 1}] -> Indeterminate`).  An unknown antiderivative or
  an undecidable pole position leaves the integral unevaluated — never a wrong
  value.
- **Iterated integrals** — `Integrate[f, {x,a,b}, {y,c,d}, …]` reduces
  innermost-first (last spec is the inner integral), so an inner bound may
  depend on an outer variable (`Integrate[1, {x,0,1}, {y,0,x}] -> 1/2`).
- **`Integrate`NewtonLeibniz[f, {x, a, b}]`** — explicit entry point;
  `Method -> "NewtonLeibniz"` selects the definite mechanism, and any other
  `Method` name on a definite integral is passed through to the inner
  indefinite integration.
- **`Integrate`SingularPoints[expr, {x, a, b}]`** — the interior-pole detector,
  exposed for inspection/reuse: sorted real roots of
  `Denominator[Together[expr]]` strictly inside `(a, b)`.
- **Attributes.** `Integrate`NewtonLeibniz` and `Integrate`SingularPoints` are
  `Protected | ReadProtected`.  `Integrate` stays non-`Listable`.
- **Scope.** The detector covers rational (denominator) poles.  Branch-cut
  discontinuities of `F` that are not denominator roots are out of scope; a
  transcendental denominator that `Solve` cannot place is treated as
  undecidable → unevaluated.
- **Tests.** `tests/test_integrate_newton_leibniz.c` — 8 groups: pole detector,
  proper integrals, improper/infinite-bound, divergent, iterated, explicit
  entry / method pass-through, unevaluated-when-unknown, and a 14-case numeric
  stress batch cross-checked against `NIntegrate`.
- **See** [`docs/spec/builtins/calculus.md`](../builtins/calculus.md) →
  *Definite integration (Newton-Leibniz)*.

## Build: GMP-ECM is now a system dependency (2026-07-07)

GMP-ECM (advanced integer factorization, used by `FactorInteger`) is no longer
vendored as the `src/external/ecm/` git submodule. It is now a plain system
library, aligning it with GMP, MPFR and FLINT.

- **Install:** `brew install gmp-ecm` (macOS) or `sudo apt install libecm-dev`
  (Debian/Ubuntu); `gmp-ecm-devel` on Fedora/RHEL.
- **Detection:** the makefile autodetects it with a compile-link probe (GMP-ECM
  ships no `pkg-config` `.pc` file) and links `-lecm`; the CMake test build uses
  `find_path`/`find_library`. When absent, both build paths degrade gracefully
  to `-DNO_ECM` (equivalent to `make USE_ECM=0`) with a one-line install hint,
  matching the `USE_FLINT`/`USE_MPFR` policy.
- **Removed:** the `src/external/ecm` submodule, the `.gitmodules` entry, the
  `build_ecm.sh` autotools/Perl bootstrap, and the `--recurse-submodules` clone
  step. No C sources changed — `facint.c` already used the public `ecm.h` API
  behind `#ifndef NO_ECM`.
- **Docs:** README, `SPEC.md`, `docs/building.md`, and the docs-site build page
  (`site/docs/index.md`) updated with the new install instructions.

## StringInsert (2026-07-06)

Added the `StringInsert` string builtin (`src/picostrings.c`).

- `StringInsert["string", "snew", n]` inserts `"snew"` so its first character
  becomes the nth character of the result; `StringInsert["string", "snew", -n]`
  makes the last character of `"snew"` the nth character from the end. Positive
  n maps to a 0-based insertion offset `n - 1`; negative n maps to `len + n + 1`.
- `StringInsert["string", "snew", {n1, n2, ...}]` inserts a copy of `"snew"` at
  each position. All positions refer to the ORIGINAL string, before any
  insertion is done, so the offsets are resolved up front against the untouched
  input (a `counts` array over `[0, len]` records how many copies land before
  each character, then a single pass builds the result). Duplicate positions
  yield multiple copies.
- `StringInsert[{s1, s2, ...}, "snew", spec]` maps over a list of strings.
- An out-of-range position (outside `1..len+1` / the negative equivalent),
  position 0, or a non-integer/symbolic position leaves the call unevaluated;
  a non-string first or second argument also stays unevaluated so symbolic
  arguments flow through. A call whose argument count is not three emits
  `StringInsert::argrx` (matching Mathematica's diagnostic).
- The result buffer is freed after `expr_new_string` copies it (leak-checked
  clean with `leaks` across the multi-position, list-mapping, error, and
  symbolic paths).
- **Attributes**: `Protected`. Symbol registered in `src/sym_names.c`;
  docstring in `src/info.c`.
- 31 unit tests in `tests/test_strings.c` (every spec form, front/end/negative
  positions, `Range`-based multi-insertion, positions-refer-to-original,
  duplicate/mixed-sign positions, list mapping, empty insertion/string,
  out-of-range and symbolic flow-through, `argrx`, and newline preservation).

See [`builtins/string-operations.md`](../builtins/string-operations.md#stringinsert).

## StringReverse (2026-07-06)

Added the `StringReverse` string builtin (`src/picostrings.c`).

- `StringReverse["string"]` reverses the order of the characters in `"string"`,
  returning a new string. Implemented by copying the characters into a fresh
  buffer in reverse order; the buffer is freed after `expr_new_string` copies
  it (no leak — confirmed valgrind-clean against the startup baseline).
- **Listable**: `StringReverse[{s1, s2, ...}]` threads element-wise over the
  list, e.g. `StringReverse[{"cat", "dog"}]` → `{"tac", "god"}`.
- A wrong argument count emits `StringReverse::argx` (matching Mathematica's
  diagnostic) and leaves the call unevaluated; a single non-string argument
  also stays unevaluated so symbolic arguments flow through.
- **Attributes**: `Listable`, `Protected`. Symbol registered in
  `src/sym_names.c`; docstring in `src/info.c`.
- Extensive unit tests in `tests/test_strings.c` (basic reversal, palindrome,
  spaces/digits, involution, Listable threading, empty strings, `StringJoin`
  composition, symbolic/integer flow-through, and the two `argx` cases).

See [`builtins/string-operations.md`](../builtins/string-operations.md#stringreverse).

## StringDrop (2026-07-06)

Added the `StringDrop` string builtin (`src/picostrings.c`), the complement of
`StringTake`.

- Supports the full standard Wolfram Language sequence specification: drop the
  first n (`StringDrop["s", n]`) or last n (`StringDrop["s", -n]`) characters,
  a single character (`{n}`), a range (`{m, n}`), a stepped range
  (`{m, n, s}`), and `UpTo[n]` (drop n, or as many as are available). Negative
  indices count from the end; a decreasing range `{m, n}` with m > n is empty
  and drops nothing.
- Implemented via a "keep" mask over the character positions: the spec marks
  the characters to drop, and the survivors are concatenated into a fresh
  buffer (freed after `expr_new_string` copies it — valgrind-clean against the
  startup baseline).
- `StringDrop[{s1, s2, ...}, spec]` maps over a list of strings. An
  out-of-range position or a symbolic argument leaves the call unevaluated.
- A call whose argument count is not two emits `StringDrop::argrx` (matching
  Mathematica's diagnostic) and leaves the call unevaluated.
- **Attributes**: `Protected`. Symbol registered in `src/sym_names.c`;
  docstring in `src/info.c`.
- Extensive unit tests in `tests/test_strings.c` (46 cases: every spec form,
  negative/mixed-sign indices, empty ranges, `UpTo` saturation, list mapping,
  out-of-bounds and symbolic flow-through, `argrx`, empty strings, and a
  cross-check against `StringJoin[Drop[Characters[...], spec]]`).

See [`builtins/string-operations.md`](../builtins/string-operations.md#stringdrop).

## StringReplacePart (2026-07-06)

Added the `StringReplacePart` string builtin (`src/picostrings.c`).

- `StringReplacePart["string", "snew", {m, n}]` replaces the characters at
  positions m through n by `"snew"`. `StringReplacePart["string", "snew",
  {{m1, n1}, ...}]` applies `"snew"` at several ranges, and
  `StringReplacePart["string", {"snew1", ...}, {{m1, n1}, ...}]` pairs each
  range with its own new string (the two lists must be the same length).
  Position specs use the `StringPosition` form (pairs `{m, n}`); negative
  positions count from the end, and all positions refer to the ORIGINAL string
  before any replacement is done.
- The flat single-range form `{m, n}` and the nested multi-range form
  `{{m, n}, ...}` are distinguished by whether the first element is itself a
  list. Ranges are resolved and validated up front; overlap is rejected in the
  order given via a `covered` bitmap — the first claimant of a character wins,
  and a later overlapping range emits `StringReplacePart::ovlp` and is dropped.
  Accepted ranges are insertion-sorted by start, then the result is assembled
  in a single left-to-right pass (length computed first, then filled).
- `StringReplacePart["string", "", ...]` deletes the selected characters. An
  empty position list makes no replacements.
- `StringReplacePart[{s1, s2, ...}, snew, part]` maps over a list of strings.
- Operator form: `StringReplacePart[new, part]` (a two-argument call) returns
  the pure function `Function[StringReplacePart[#1, new, part]]`, so
  `StringReplacePart[new, part][old]` reduces to
  `StringReplacePart[old, new, part]` (mirroring how `Cases[patt][expr]` is
  implemented).
- A malformed or out-of-range position, a reversed range (m > n after
  resolution), a non-string argument, or a new-string/position length mismatch
  leaves the call unevaluated so symbolic arguments flow through. A call whose
  argument count is not two or three emits `StringReplacePart::argt` (matching
  Mathematica's diagnostic).
- The result buffer is freed after `expr_new_string` copies it (leak-checked
  clean with `leaks`: `0 leaks for 0 total leaked bytes` across the single/
  multi-range, list-of-strings, operator, overlap, delete, list-mapping, and
  error paths).
- **Attributes**: `Protected`. Symbol registered in `src/sym_names.c`;
  docstring in `src/info.c`.
- 28 unit tests in `tests/test_strings.c` (every spec form and example,
  broadcast vs per-range new strings, both operator forms, overlap rejection
  including a mid-list skip, deletion, single/nested range, first/last/full
  replacement, negative and adjacent ranges, empty position list, list
  mapping, special characters, `argt` for 0/1/4 args, and symbolic/
  out-of-range/reversed/length-mismatch flow-through).

See [`builtins/string-operations.md`](../builtins/string-operations.md#stringreplacepart).

## RegularExpression + regex string functions (2026-07-06)

Added regular-expression matching for strings, built on **PCRE2** (`libpcre2-8`,
the same engine the Wolfram Language uses). New builtins live under
`src/strings/regex/`: `RegularExpression`, `StringMatchQ`, `StringCases`,
`StringReplace`, and `StringSplit`.

- **Build wiring** — PCRE2 is an optional pkg-config dependency
  (`-DUSE_REGEX -DPCRE2_CODE_UNIT_WIDTH=8`), autodetected and graceful-degrading
  exactly like Raylib/FLINT: when absent, `make` still succeeds and the regex
  builtins warn (`Name::regavail`) and stay unevaluated. A single wrapper
  (`regex_engine.{c,h}`) is the only translation unit that includes `<pcre2.h>`;
  its body is under `#ifdef USE_REGEX` with link-compatible stubs otherwise.
  Best-effort JIT (`pcre2_jit_compile`) is enabled.
- **`RegularExpression["re"]`** is an inert data head (evaluates to itself) that
  carries a PCRE pattern for the string functions. It best-effort-validates the
  pattern and emits `RegularExpression::regex` on a syntax error.
- **`StringMatchQ[s, patt]`** — whole-string match (the pattern is anchored
  `\A(?:...)\z`), returning `True`/`False`.
- **`StringCases[s, patt]`** — list of non-overlapping matches, left to right;
  with a rule `patt -> rhs` returns the rhs per match with `$0`/`$1…` expanded.
- **`StringReplace[s, rule | {rules…}]`** — replaces each non-overlapping match
  by the (`$n`-expanded) rhs; with several rules the leftmost match wins, ties
  broken by rule order; zero-width matches (e.g. `\b`) are handled without
  dropping characters.
- **`StringSplit[s, patt]`** — substrings between delimiter matches, dropping
  empty pieces (Wolfram default); zero-width delimiters like `(?m)^` / `(?m)$`
  split at positions.
- All four accept `RegularExpression["re"]`, a **literal string** (matched
  literally — PCRE metacharacters are escaped), or a **List** of
  alternatives/rules; a **list of subjects** threads element-wise (the builtins
  are deliberately not `Listable`, so the pattern/rule argument is never
  threaded — matching the `StringReplacePart`/`Replace` precedent). Non-string
  subjects flow through unevaluated.
- The shared layer `regex_common.{c,h}` compiles a pattern argument into a rule
  set, expands `$n`/`$0` (and `$$` → `$`) replacement templates, and provides a
  small growable buffer; `regex_rules_free` releases every compiled program.
- **Attributes**: `Protected`. `SYM_RegularExpression` registered in
  `src/sym_names.c`; docstrings for all five in `src/info.c`.
- Tests: `tests/test_regex.c` (engine wrapper — compile/match/unset-group/bad-
  pattern — and `$n` expansion, plus the inert head) and
  `tests/test_stringfns.c` (the worked spec examples across all four functions:
  character classes/ranges, `\d \D \s \S \w \W`, named classes, quantifiers and
  non-greedy `*?`, anchors and `\b`, inline `(?i)`, alternation, `$1`/multi-rule
  replacement, `(?m)^`/`(?m)$` splitting with real newlines, literal patterns,
  list threading, and unevaluated flow-through). Leak-checked clean (`leaks`:
  `0 leaks for 0 total leaked bytes`) across every path.

Note: Mathilda's string lexer does not interpret C-style escapes (`\n`, `\t`)
in string literals, so a newline in a *subject* must be a real newline byte;
this is a pre-existing parser property, independent of the regex layer.

See [`builtins/string-operations.md`](../builtins/string-operations.md#regularexpression).

## Integrate `"GoursatAlgebraic"` — denominator-radical normalisation fix (2026-07-07)

Fixed a branch-cut normalisation bug in the Goursat V4 (square-root) descent
(`src/calculus/integrate_goursat.c`). On e.g.
`Integrate[Sqrt[1 + x^4]/(1 - x^4), x, Method -> "GoursatAlgebraic"]` the result
carried a spurious nested radical
`Sqrt[2 (1 + (1+x)^4/(-1+x)^4 + 6 (1+x)^2/(-1+x)^2)]` in the ArcTan/ArcTanh
arguments, which is really `4 Sqrt[1 + x^4]/(-1 + x)^2` over the integrand's own
generator.

- **Root cause.** The eigendescent emits the radical in *denominator* positions
  as `Power[g, -1/2]` (`1/Sqrt[g]`), but the radical-over-R normaliser
  (`reexpress_over_radical`) only recognised and rewrote the `+1/2` form:
  `collect_sqrt_radicands` matched the exponent exactly `1/2`, and the rewrite
  rule was `Sqrt[g] -> cand` alone. So `+1/2` occurrences (in the `Log` term)
  healed to `Sqrt[1 + x^4]` while the `-1/2` occurrences stayed nested.
- **Fix.** `collect_sqrt_radicands` now gathers radicands from any half-integer
  power `Power[g, m/2]` (via `is_half_power`), so it also sees `1/Sqrt[g]`; and
  `reexpress_over_radical` now rewrites **both** signs — `Sqrt[g] -> cand` and
  `1/Sqrt[g] -> 1/cand` — in one `ReplaceAll`. `cand` is the sign-fixed
  `+Sqrt[g]`, so its reciprocal is `1/Sqrt[g]` with the matching sign; the
  differentiate-back guard still gates adoption.
- The antiderivative is now written entirely over `Sqrt[1 + x^4]` and
  differentiates back to the integrand (verified to 25 digits at several
  rational points).
- Regression tests added to `tests/test_integrate_goursat.c` (`test_sqrt`):
  numeric differentiate-back plus a structural assertion that no t-dependent
  half-power radical other than `Sqrt[1 + t^4]` survives.
