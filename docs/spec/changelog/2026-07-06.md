# Changelog: week of 2026-07-06 (Mon) – 2026-07-12 (Sun)

Feature additions and fixes recorded during this week.

## Integrate: recursive Risch integrator ported from Maxima (2026-07-10)

New indefinite-integration method `Integrate\`RischMacsyma[f, x]`
(`src/calculus/integrate_risch_macsyma.c`), also reachable as
`Integrate[f, x, Method -> "RischMacsyma"]` and inserted into the Automatic
cascade **after** `Integrate\`RischNorman`.

This is a faithful port of the algorithm structure of Maxima's `risch.lisp`:
the **recursive** Risch algorithm (a decision procedure), deliberately distinct
from the existing `Integrate\`RischNorman`, which is the **parallel** Risch
(Bronstein pmint / Norman) heuristic.  It never falls back on the parallel-Risch
engine.  Arithmetic is grounded in Mathilda's existing `Expr`/poly/rat
primitives rather than a reimplementation of Maxima's CRE representation; the
rational base case delegates to `Integrate\`BronsteinRational`.

Every branch is **correct by construction** — each case emits a closed form only
behind an exact structural certificate (a logarithmic derivative that Cancels to
a polynomial, an exactly linear denominator, a genuinely `x`-free cofactor, ...),
so results are **not** checked by differentiation.

Cases implemented:

- **Logarithmic polynomial** `P(x, Log[u])` with `u` rational in `x`: the
  recursive primitive-polynomial coefficient matching
  (`q_i' + (i+1) q_{i+1} eta = p_i`), with a limited-integration oracle that
  folds a would-be new logarithm back into the tower — e.g.
  `Integrate[Log[x], x] = x Log[x] - x`, `Integrate[Log[x]^2, x]`,
  `Integrate[Log[2 x + 3], x]`, `Integrate[Log[x^2 + 1], x]`,
  `Integrate[Log[x]/x, x] = Log[x]^2/2`.
- **Exponential (Laurent) polynomial** `sum_i p_i(x) E^(i u)` with `u` a
  polynomial in `x` (`theta' = u' theta`), `i` ranging over positive AND
  negative powers of a single primitive kernel `E^u`: the exponential powers
  decouple, the `i = 0` term integrates in `K`, and each `i != 0` term solves
  the Risch differential equation `q_i' + i u' q_i = p_i` by a bounded
  polynomial ansatz solved exactly with `SolveAlways` — e.g.
  `Integrate[x E^x, x] = (x - 1) E^x`, `Integrate[x E^(x^2), x] = E^(x^2)/2`,
  `Integrate[x E^(-x^2), x] = -E^(-x^2)/2`, and the Laurent
  `Integrate[(E^x + E^(-x))/2, x] = Sinh[x]`,
  `Integrate[E^x + E^(-x) + E^(2 x), x]`.  When the RDE has no polynomial
  solution the exp case declines (so `E^(-x^2)` itself falls through to Erf).
- **Hermite reduction (repeated transcendental poles)** for both the
  logarithmic (`theta = Log[u]`) and exponential (`theta = E^u`) kernels: a
  proper rational function of `theta` whose denominator has a repeated factor
  reduces to `Q = H(theta)/Hden(theta) + sum_j c_j Log(g_j)`, where
  `Hden = gcd(D, dD/dtheta)` is the repeated part, `g_j` the squarefree factors
  of the radical `D/Hden`, and `H` a polynomial in `theta` with bounded-degree
  polynomial-in-`x` coefficients; all constant unknowns are solved at once by
  `SolveAlways[Q' - f == 0, {t, x}]` (result then `Cancel`ed) — e.g.
  `Integrate[1/(x (1 + Log[x])^2), x] = -1/(1 + Log[x])`,
  `Integrate[1/(x Log[x]^2), x] = -1/Log[x]`,
  `Integrate[(1 + Log[x])/(x Log[x]^2), x] = Log[Log[x]] - 1/Log[x]` (mixed
  Hermite + log part), and `Integrate[E^x/(1 + E^x)^2, x] = -1/(1 + E^x)`.  The
  exponential kernel is handled when `D` is coprime to `theta` (no `theta = 0`
  pole; that Laurent-coupled case is left to the hyperexponential path).
- **Coupled hyperexponential case** (now including Laurent-coupled Hermite —
  Phase A): a general rational function of a single `theta = E^u` (`u`
  polynomial in `x`) whose integral mixes a Laurent-polynomial part with a log
  part — the case where `D(g)/g` is itself improper in `theta`, so the two
  parts do not separate.  A unified ansatz
  `Q = sum_i w_i(x) theta^i + H(theta)/Hden(theta) + sum_j c_j Log(g_j)` is
  solved for all its constant coefficients at once by
  `SolveAlways[Q' - f == 0, {t, x}]` — correct by construction (an exact
  solution certifies the identity; an imperfect degree bound only declines).
  The `theta`-coprime denominator `Dtil = den/theta^a` is split into its
  repeated part `Hden = gcd(Dtil, dDtil/dtheta)` (absorbed by the Hermite term
  `H/Hden`, `H` proper in `theta` with polynomial-in-`x` coefficients) and its
  squarefree radical `rad = Dtil/Hden` (whose distinct factors give the log
  terms); when `Dtil` is squarefree `Hden = 1` and the ansatz collapses to the
  previous Laurent + log form.  Closes
  `Integrate[1/(1 + E^x), x] = x - Log[1 + E^x]`,
  `Integrate[1/(2 + E^x), x]`, `Integrate[E^(2 x)/(1 + E^x), x]`, and now the
  **repeated / `theta = 0` exponential poles** the pure Hermite path (cannot
  supply the `x`/Laurent term) and the squarefree hyperexp path both declined:
  `Integrate[1/(1 + E^x)^2, x] = x + 1/(1 + E^x) - Log[1 + E^x]`,
  `Integrate[1/(E^x (1 + E^x)^2), x]` (a `theta = 0` pole coupled with a
  repeated pole at `theta = -1`), `Integrate[1/(2 + E^x)^2, x]`,
  `Integrate[1/(1 + E^x)^3, x]`.
- **Multi-kernel sum of exponentials** (Phase B, first increment): an integrand
  that exponentializes to a sum `sum_k p_k(x) E^(W_k)` of NON-commensurate
  exponentials (exponents `W_k` not integer multiples of one primitive — e.g. the
  `(1 ± I) x` pair from `E^x Sin[x]` after `TrigToExp`) is a set of independent
  transcendental extensions.  Since `d/dx` never mixes distinct exponents, the
  terms decouple and each integrates by its own Risch DE `q_k' + W_k' q_k = p_k`
  (`W_k` polynomial in `x`), plus any `W_k = 0` term in `K`.  The term splitter
  descends into nested (un-flattened) `Times`, so `p_k` is the full
  exponential-free cofactor.  Correct by construction (each `q_k` is
  `SolveAlways`-certified; `d/dx sum_k q_k E^(W_k) = f` by linearity); declines
  whole if any term is non-elementary (`E^(x^2)`) or has a non-polynomial
  exponent.  Closes `Integrate[E^x Sin[x], x]`, `Integrate[E^x Cos[x], x]`,
  `Integrate[x E^x Sin[x], x]`, `Integrate[E^(2 x) Sin[3 x], x]`,
  `Integrate[x^2 E^x Cos[x], x]`.  **Known simplification gap** (same family as
  `Tan`/`Tanh` below): through the complex exponentials `E^((a ± b I) x)`,
  `ExpToTrig` leaves the answer in a correct but I-laden `Cosh`/`Sinh`-of-complex
  form; the diff-back is exactly `0`.
- **Nested logarithmic tower** (Phase B, second increment): a rational function
  of a CHAIN of nested logarithms `t_1 = Log[u_1](x)`, `t_2 = Log[u_2](x, t_1)`,
  ..., `t_n = Log[u_n](x, t_1..t_{n-1})` (`n >= 2`) — which the single-extension
  cases cannot model.  The single-kernel derivation `D = d/dx + Dt d/dt`
  generalizes to the whole tower `D = d/dx + sum_i Dt_i d/dt_i` (each
  `Dt_i = Cancel[D[u_i,x]/u_i]` substituted to the `t`'s, triangular: `Dt_i` lies
  in `C(x, t_1..t_{i-1})`), and one unified ansatz
  `Q = sum_k P_k(x, t_1..t_{n-1}) t_n^k + sum_j c_j Log(g_j)` (`P_k` bounded-degree
  polynomials, `g_j` the squarefree `t_n`-denominator factors, `c_j` constants) is
  solved at once by `SolveAlways[D_tower[Q] - f == 0, {t_n,...,t_1,x}]`.  Correct
  by construction (the `t_i` are algebraically independent transcendentals, so a
  certified polynomial identity in them proves the functional identity).  Closes
  `Integrate[1/(x Log[x] Log[Log[x]]), x] = Log[Log[Log[x]]]`,
  `Integrate[Log[Log[x]]/(x Log[x]), x] = Log[Log[x]]^2/2`,
  `Integrate[Log[Log[x]]/x, x] = Log[x] Log[Log[x]] - Log[x]` (a lower-field
  coefficient), depth-3 `1/(x Log[x] Log[Log[x]] Log[Log[Log[x]]])`, and
  independent (non-nested) chains such as `Log[x] + Log[x + 1]`.  **Safety gate**
  (never wrong; decline instead): after substituting all kernels to `t`, the
  integrand must be a genuine rational function of the *whole* tower
  `{x, t_1..t_n}` (`PolynomialQ[num, {x,t_1..t_n}]` and the same for `den`) —
  without it a residual non-rational inner kernel (`Sin[Log[x]] -> Sin[t_1]`) would
  slip through and the ansatz could certify a wrong closed form.  Repeated
  top-kernel poles and rational lower-field coefficients are out of scope for this
  increment and decline cleanly.
- **Nested exponential tower** (Phase B, third increment): dual of the log tower
  for a chain of nested exponentials `t_i = E^(u_i)` (`t_1 = E^x`, `t_2 = E^(E^x)`,
  ...).  An exponential kernel's derivative is a multiple of itself, so
  `Dt_i = (D[u_i,x]|_{ker->t}) t_i`, and the ansatz is **Laurent** (inner exp
  kernels are invertible, so their coefficient exponents may be negative — e.g.
  `INT E^(x+E^x) dx = E^(E^x) = t_2/t_1` needs `t_1^{-1}`).  Closes
  `Integrate[E^x E^(E^x), x] = E^(E^x)`, `Integrate[E^(2 E^x) E^x, x]`.  **Both
  tower cases are now diff-back verified** (`Simplify[D[result,x]-f] == 0`, else
  decline): unlike the single-kernel decision-procedure cases they are
  bounded-ansatz searches whose multi-variable `SolveAlways` certificate can be
  spurious for a non-elementary integrand (e.g. `E^(E^x)/(1+E^(E^x))` is
  non-elementary and is rejected).
- **Fix (correctness): single-kernel cases certified a wrong nested-exponential
  answer.**  `rm_frac_case` (and the other single-kernel cases) substituted only
  ONE exponential kernel, leaving a nested inner kernel (`E^x` inside `E^(E^x)`)
  in the derivation `Dt = u' theta`; `SolveAlways` then treated it as a free
  parameter and certified `Integrate[E^(E^x)/(1+E^(E^x))] = Log[1+E^(E^x)]/E^x`,
  which is WRONG (the integral is non-elementary) and leaked into the default
  `Integrate` cascade.  Fixed with a nesting gate (`rm_kernel_simple`): every
  single-kernel case now requires its kernel's defining `u` to be rational in `x`
  alone; nested integrands fall through to the tower cases (which decline this one
  after the diff-back check).
- **Trigonometric / hyperbolic front-end** (Maxima's exponentialize path):
  `TrigToExp` rewrites the trig/hyperbolic kernels to complex exponentials,
  the exponential machinery integrates the resulting Laurent-rational function
  of `E^(I x)` / `E^x`, and `ExpToTrig` converts back — both exact rewrites, so
  the result stays correct by construction.  Closes `Integrate[Sin[x], x]`,
  `Cos[x]`, `Sinh[x]`, `Cosh[x]`, `Sin[x]^2`, `Sin[x] Cos[x]`, `Tan[x]`,
  `Tanh[x]`, ... .  **Known simplification gap:** through the complex
  substitution `u = I x`, `Tan`/`Tanh` close to a correct but I-laden form such
  as `I x - Log[1 + E^(2 I x)]` (`= -Log[Cos[x]]`) that no current simplifier
  (`Simplify`/`FullSimplify`/`TrigReduce`/`PowerExpand`) collapses to real
  closed form.  The correct answer is returned (completeness preferred over
  declining); teaching `Simplify` the log-of-product / half-angle collapse is a
  follow-up improvement.
- **Fractional (Rothstein–Trager) log-part**: a proper rational function of a
  single transcendental `theta` (`theta` in the denominator) with a squarefree
  denominator `d = prod g_i` integrates to `sum_i c_i Log(g_i)` with each `c_i`
  a constant.  The residues are found by solving the exact identity
  `num = sum_i c_i D(g_i) (d/g_i)` (with the transcendental derivation
  `D = d/dx + (D theta) d/dtheta`) for all `theta` and `x` via
  `SolveAlways[..., {t, x}]` — e.g. `Integrate[1/(x (1 + Log[x])), x] =
  Log[1 + Log[x]]`, `Integrate[E^x/(1 + E^x), x] = Log[1 + E^x]`,
  `Integrate[1/(x Log[x] (1 + Log[x])), x] = Log[Log[x]] - Log[1 + Log[x]]`,
  `Integrate[1/(x (Log[x]^2 - 1)), x]` (two residues).  Integrands with a
  polynomial/Hermite remainder or a repeated pole decline (no wrong output).
- **Special functions** (Maxima's `erfarg` / `dilog` and the Ei/li reductions),
  now produced by default (WL-faithful), closing integrals the whole elementary
  cascade otherwise leaves open:
  `Integrate[E^(-x^2), x] = (Sqrt[Pi]/2) Erf[x]` and the general Gaussian
  `E^(a x^2 + b x + c)`; `Integrate[E^x/x, x] = ExpIntegralEi[x]`;
  `Integrate[1/Log[x], x] = LogIntegral[x]`;
  `Integrate[Log[1 - x]/x, x] = -PolyLog[2, x]`.

Remaining phases (integrands needing them return unevaluated): *mixed* exp-log
towers, algebraically dependent kernels (the evaluator merges `E^x E^(E^x) ->
E^(x+E^x)`), repeated top-kernel poles (tower Hermite), and rational (not
polynomial) lower-field coefficients — which need a genuine
one-extension-at-a-time recursion with a structure-theorem kernel basis;
rational-argument exponentials / non-polynomial `u` (e.g. `E^(1/x)`); and
algebraic extensions.  Tests: `tests/test_integrate_risch_macsyma.c`.

## Names: enumerate symbol-table names by pattern (2026-07-10)

New builtin `Names` (`src/names.c`) lists the names of symbols in the symbol
table, matching Wolfram-Language semantics:

- `Names["string"]` / `Names[patt]` — names matching a string pattern.
- `Names[{p1, p2, ...}]` — names matching any of the patterns.
- `Names[]` — every name in the table.

String patterns are matched against the whole (anchored) name with two
metacharacters: `*` (zero or more characters) and `@` (one or more
**non-uppercase** characters); every other character, including the context
backtick, is literal. A pattern element may instead be
`RegularExpression["re"]`, routed through the existing PCRE2 engine
(`src/strings/regex`) and anchored `\A(?:re)\z`; when Mathilda is built without
PCRE2 such a pattern emits `Names::regavail` and stays unevaluated.

The result is a `List` of `String`s sorted with the system's canonical string
order (`expr_compare`), so `Names[patt]` is always identical to
`Sort[Names[patt]]`. All symbols are candidates — no internal-symbol filtering.

Enumeration is backed by a new public `symtab_for_each` iterator
(`src/symtab.{c,h}`) over the existing chained hash table. `Names` is
`Protected`, registered in `sym_names.c`, documented via `info.c`, and covered
by `tests/test_names.c` (12 tests). Valgrind-clean: a Names-heavy session shows
byte-for-byte the same leak profile as a bare `Sin[1.0]` baseline.

## ByteCount: count NDArray, BigInt, and MPFR payloads (2026-07-10)

`ByteCount` previously counted only the fixed `sizeof(Expr)` for any leaf whose
type was not a symbol/string/function — so `EXPR_NDARRAY`, `EXPR_BIGINT`, and
`EXPR_MPFR` were all badly undercounted. Worst was `NDArray`: `ByteCount`
returned a constant independent of the array's size, ignoring the entire data
buffer that is the object's whole purpose (a 1000×1000 float64 array and a 2×2
one reported the same count).

`byte_count_internal` (`src/core.c`) now adds the out-of-node storage each of
these atoms owns:

- **`NDArray`** — the owned `dims[]` array (`rank` × `int64_t`) plus the flat
  row-major data buffer (`ndarray_size(e)` × `ndt_elem_size(dtype)`). The count
  now scales with element count and honors dtype width (float32 < float64 <
  complex64).
- **`BigInt`** — GMP significand limbs (`mpz_size` × `sizeof(mp_limb_t)`).
- **`MPFR`** — significand storage via `mpfr_custom_get_size`, scaling with the
  value's precision.

Extensive assertions added to `test_bytecount` (`tests/test_core.c`), including
exact per-element and per-dtype scaling checks for `NDArray`.

## Linear algebra: NDArray fast paths for every routine (2026-07-10)

Every linear-algebra builtin now accepts `NDArray[...]` (`EXPR_NDARRAY`) input
and evaluates it on the flat `double` buffer, never falling back to the
symbolic path. A new module `src/linalg/ndlinalg.c` centralises this: each
builtin adds one guard,

```c
if (linalg_call_has_ndarray(res)) return ndla_<op>(res);
```

Two tiers:

- **Buffer fast paths** returning a closed-system `NDArray`/`Real`/`Complex`
  result: `Det`, `Inverse`, `LinearSolve`, `MatrixRank`, `Tr`, `Norm`,
  `Normalize`, `Cross`. These load via `numarray.c`'s `na_load_*` (which reads
  `ndt_get` straight into a col-/row-major buffer, widening `float32`,
  marshalling complex) and compute with an in-house partial-pivot `double` LU
  (`nd_lu_real`) for the real heavy four — so `Det[NDArray[...]]` is O(n³) in
  every build, never the O(n!) Laplace path. Complex heavy ops use LAPACK `z*`
  when available.
- **Delist-to-numeric** for the factorisations, predicates, constructors, and
  lattice routines (`Eigenvalues`/`Eigenvectors`, `QRDecomposition`,
  `LUDecomposition`, `SingularValueDecomposition`, `LeastSquares`,
  `PseudoInverse`, `NullSpace`, `RowReduce`, `MatrixPower`,
  `PositiveDefiniteMatrixQ`, `NegativeDefiniteMatrixQ`, `DiagonalMatrix`,
  `ToeplitzMatrix`, `HankelMatrix`, `VandermondeMatrix`, `LatticeReduce`,
  `FindIntegerNullVector`): `linalg_delist_and_reeval` rebuilds the call with
  each `NDArray` argument converted to a nested `List` and re-evaluates it, so
  the routine's existing numeric-List kernel runs. This also fixed a crash
  where `Eigenvalues[NDArray[...]]` reached List-indexing code past the shape
  gate.

Examples:

```mathematica
Det[NDArray[{{1., 2.}, {3., 4.}}]]                 (* -2.0 *)
Inverse[NDArray[{{4., 3.}, {6., 3.}}]]             (* NDArray[{{-0.5, 0.5}, {1.0, -0.666667}}] *)
LinearSolve[NDArray[{{2., 0.}, {0., 4.}}], NDArray[{2., 8.}]]   (* NDArray[{1.0, 2.0}] *)
Tr[NDArray[{{1., 2.}, {3., 4.}}]]                  (* 5.0 *)
Eigenvalues[NDArray[{{2., 0.}, {0., 3.}}]]         (* {3.0, 2.0} *)
```

Tested by `tests/test_ndarray_linalg.c` (float64/float32/complex64, fast-path
vs List agreement, edge shapes) and `tests/bench_ndarray_linalg.c` (O(n³)
doubling-ratio gate + large-size stress). Both `USE_LAPACK=1` and
`USE_LAPACK=0` builds pass; valgrind/`leaks`-clean.

## NDArray: warn on symbolic `Plus`/`Times`/`Power` (2026-07-10)

An `NDArray[...]` is purely numeric, so combining one with a symbolic operand
can never be carried out elementwise. `Plus`, `Times`, and `Power` now detect
this case — at least one `NDArray` operand plus at least one operand that is
neither an `NDArray` nor numeric (per `expr_is_numeric_like`) — and print a
one-line `NDArray::sym` warning before leaving the expression unevaluated:

```mathematica
In[1]:= NDArray[{1., 3.}] + a
NDArray::sym: NDArray objects are purely numeric and cannot be added with a
symbolic expression; the expression is left unevaluated.
Out[1]= a + NDArray[{1.0, 3.0}]

In[2]:= c NDArray[{1., 3.}]
NDArray::sym: ... cannot be multiplied ...
Out[2]= c NDArray[{1.0, 3.0}]

In[3]:= NDArray[{1., 3.}]^n
NDArray::sym: ... cannot be raised to a power ...
Out[3]= NDArray[{1.0, 3.0}]^n
```

A numeric scalar operand (Integer/Real/Rational/Complex) still broadcasts
silently — `1 + NDArray[...]`, `3 NDArray[...]`, `2 ^ NDArray[...]` are
unaffected. The warning shares the existing `ndarray_warn_once` dedup (one print
per evaluation) and honours the arithmetic-warning mute, so it stays quiet
inside internal computations (`Simplify`, etc.). New helper
`ndarray_warn_symbolic()` in `src/ndarray.c`; callers in `plus.c`, `times.c`,
`power.c`.

## NDArray `DataType` (dtype) support (2026-07-10)

`NDArray[...]` now carries an element data type (dtype), analogous to numpy's
`dtype`, defaulting to `float64` and also supporting `float32`, `complex64`, and
`complex32`. The four types map one-to-one onto BLAS's `s`/`d`/`c`/`z`
precisions (the numeric suffix is the per-component bit width); complex is stored
as interleaved `(re, im)` plain floats/doubles (no `<complex.h>`, BLAS-compatible).

- **Construction & introspection.** `NDArray[list, DataType -> "float32"]` packs
  at the chosen type (real dtypes accept `Integer`/`Real`; complex dtypes also
  accept `Complex[...]`). `Options[NDArray]` is `{DataType -> "float64"}`;
  `DataType[a]` returns an array's dtype string (numpy `.dtype`), unevaluated on
  a non-array. dtype is part of array identity (`SameQ`, ordering, hashing).

- **Fast paths.** Elementwise `Plus`/`Times`/`Dot` and a new elementwise `Power`
  operate directly on the typed buffers with a promotion lattice (complex ≻ real,
  64-bit ≻ 32-bit component); the float64 real case keeps its dedicated tight
  `double` loop. `Power` promotes to complex when a real base with a non-integer
  exponent leaves the real axis (`NDArray[{-1.0}]^0.5`); integer exponents on
  complex bases are exact (`I^2 == -1` via binary exponentiation).

- **numpy-style scalar broadcasting.** Numeric scalars combine elementwise with
  arrays: `1 + NDArray[...]`, `3 * NDArray[...]`, `2 ^ NDArray[...]`, and `-A` /
  `A - B`. Scalars are "weak" — an `Integer`/`Real` scalar keeps the array's
  float width (`float32 + 1 -> float32`), a `Complex` scalar moves it onto the
  complex axis at the same width (`float32 + I -> complex32`).

- **Closed system under linalg.** The BLAS/LAPACK bridges accept all four dtypes
  as input (widening `float32` → `double` for the compute kernel) and build
  results back into dtyped `NDArray`s — real → real array, complex → `complex64`
  array (previously `List[Complex[...]]`).

- **Implementation.** `NDArrayData` gains a `dtype` field and a `void*` buffer;
  `ndt_*` helpers (`src/ndarray.c`, declared in `ndarray.h`) are the single choke
  point that widen/narrow between the stored representation and a `(double re,
  double im)` pair, keeping every generic consumer (copy/eq/hash in `expr.c`,
  `sort`, `precision`, list predicates) dtype-agnostic. `SYM_DataType` in
  `sym_names.c`, docstring in `info.c`.

- **Tests.** `tests/test_ndarray.c` gains dtype construction/introspection,
  round-trip, narrowing, dtype-distinguishing equality, complex arithmetic,
  mixed-dtype promotion, scalar broadcasting, `Power` (scalar/elementwise/scalar-
  base, complex-exact, real→complex promotion), and complex/float32 `Dot` cases.

## Direct BLAS / LAPACK REPL access via `` BLAS` `` and `` LAPACK` `` (2026-07-10)

Mathilda already links BLAS/LAPACK (Apple Accelerate on macOS, a system
BLAS/LAPACK on Linux) for its machine-precision linear algebra, but those
kernels were reachable only through the high-level operators. Two new backtick
contexts now expose them directly, mirroring the existing `` FLINT` `` context:

- **`` BLAS` `` (~22 kernels)** — Level 1 (`ddot`, `dnrm2`, `dasum`, `idamax`,
  `daxpy`, `dscal` + complex `zdotu`, `zdotc`, `dznrm2`, `zaxpy`, `zscal`),
  Level 2 (`dgemv`, `dger`, `dtrmv`, `zgemv`), and Level 3 (`dgemm`, `dsymm`,
  `dtrsm`, `dsyrk` + complex `zgemm`, `zherk`). Run in CBLAS row-major mode.
- **`` LAPACK` `` (~24 drivers)** — solves (`dgesv`, `dtrtrs`, `dgels`),
  factorizations (`dgetrf`, `dgetri`, `dgeqrf` → `{Q,R}`, `dpotrf`), SVD
  (`dgesdd`/`dgesvd` → `{U,S,VT}`), eigenproblems (`dgeev`, `dsyev` →
  `{values,vectors}`), and `dlange`, each with its complex `z*` counterpart.

The bindings are ergonomic: dimensions and leading dimensions are inferred from
the array shapes, so only the meaningful arguments (`alpha`, `beta`, operand
arrays) are passed. Arrays may be `NDArray[...]` values or nested `List`s; real
results come back as `NDArray`, complex results as nested `List`s of
`Complex[...]`. A shape mismatch, singular/indefinite input, or non-numeric
entry leaves the call unevaluated.

## Limit: bounded base raised to a divergent positive power (2026-07-09)

`Limit[(Sin[1/x]/2)^(1/x^2), x -> 0]` now returns **0** (was left
unevaluated). `Sin[1/x]` has an essential singularity at `x = 0`, so the
series/L'Hospital layers can make no progress; the only sound route is a
magnitude squeeze. The existing bounded-envelope layer knew `|Sin[g]| <= 1`
and integer powers, but had no rule for a bounded base raised to a *growing*
exponent, and was gated to `+Infinity` only.

Added `envelope_bounded_power` in `src/calculus/limit.c`, run at the front of
`layer_bounded_envelope` (so it also fires at finite points). For
`f = base^exp` where `base` carries a bounded oscillatory head
(Sin/Cos/Tanh/ArcTan/ArcCot) and `exp -> +Infinity`, it uses
`|base^exp| = |base|^exp <= B^exp` with `B` the constant magnitude bound of the
base; when `B^exp -> 0` the squeeze forces `f -> 0` (the value is complex where
`base < 0`, but its magnitude still vanishes, so the two-sided complex limit is
0). Soundness gates: the exponent limit must be `+Infinity` (so `exp` is
eventually positive — the inequality direction and the decay both require it),
and `B` must be x-free. Consequently `(2 Sin[1/x])^(1/x^2)` (base bound `>= 1`,
genuinely unbounded) and `(Sin[1/x]/2)^(-1/x^2)` (negative exponent) correctly
stay unevaluated rather than yielding a false `0`. The `+Infinity` case
(`... , x -> Infinity` — exponent `-> 0`, limit `1`) is unaffected. New
coverage in `tests/test_limit.c::test_bounded_envelope`; leak-clean under
valgrind (no new frames vs. baseline).

## Limit: bounded-power squeeze — shrinking bound, divergent base, hang fix (2026-07-09)

Three follow-up gaps in the bounded-base-power machinery above, all in
`src/calculus/limit.c`:

- **Shrinking (x-dependent) magnitude bound.**
  `Limit[(x Sin[1/x]/2)^(1/x^2), x -> 0]` now returns **0**. The magnitude
  bound `B` is now built with a sound `Abs`-wrapped bare-variable case
  (`|x Sin[1/x]/2| <= Abs[x]/2`) and the squeeze fires whenever
  `Limit[B]` is a concrete real in `[0, 1)` — so a bound that *shrinks to 0*
  succeeds, not just a constant one. (Previously any x-dependent `B` was
  rejected.) For an x-free `B` this is identical to the old `Limit[B^exp] = 0`
  test, so no case regresses.

- **Base bounded below by a constant > 1 → `Infinity`.**
  `Limit[(2 + Sin[1/x]/2)^(1/x^2), x -> 0]` now returns **Infinity**. For a
  base `c + g(x)` with `c` an x-free positive real and `g` the bounded
  remainder, `|base| >= |c| - U` (with `U >= |g|`); if that lower bound has a
  limit `> 1` and `c > 0`, the base is eventually positive and `> 1`, so
  `base^exp -> +Infinity`.

- **Termination fix for the general squeeze (removes hangs).**
  `Limit[(Cos[x]/2)^x, x -> Infinity]`, `Limit[(Sin[x]/2)^x, x -> Infinity]`
  and `Limit[(1 + Sin[x]/x)^(x^2), x -> Infinity]` previously **hung**.
  `layer_bounded_envelope`'s general squeeze recursively re-limited its own
  magnitude bound; when the oscillator survived inside an `Abs[Log[...]]`
  default-wrap (the reduction couldn't replace it by a constant envelope),
  each recursion re-wrapped another `Abs` down to the depth cap, with an
  expensive `simp` per level. It now bails when `contains_bounded_head(bound)`
  is still true — the guard the surrounding comment already described but was
  never implemented. The two `(·/2)^x` cases now settle to `0` via the
  squeeze; the `1^Infinity` case terminates unevaluated (its limit genuinely
  does not exist); and `Limit[Cos[1/x]^(x^2), x -> Infinity]` — a hang that was
  actually a convergent `1^Infinity` — is now recovered by Series as
  `1/Sqrt[E]`.

New coverage in `tests/test_limit.c::test_bounded_envelope`; leak-clean under
valgrind (definitely-lost byte count identical to baseline).

## Rename Matrix → NDArray + N-dimensional introspection (2026-07-09)

Renamed the dense-array type from `Matrix[...]` to `NDArray[...]` (numpy
`ndarray` naming) so it no longer collides with the existing list-based
linear-algebra `Matrix*` predicates (`MatrixQ`, `MatrixPower`, `MatrixRank`).
The rename is full and internal: `EXPR_MATRIX` → `EXPR_NDARRAY`,
`MatrixData` → `NDArrayData`, `src/matrix.{c,h}` → `src/ndarray.{c,h}`,
`matrix_*` → `ndarray_*`, `SYM_Matrix` → `SYM_NDArray`.

The type was already rank-generic (flat row-major `double` buffer + rank/dims,
with N-dimensional access computed by offset arithmetic); this change makes the
N-dimensional nature first-class in the introspection surface:

- **`Depth[NDArray[...]]`** now returns `ndim + 1` (matching the equivalent
  nested `List`), instead of treating the array as an opaque depth-1 atom.
- **`Length[NDArray[...]]`** now returns the leading-axis length (numpy `len`
  = `shape[0]`), instead of `0`.
- **`Dimensions[NDArray[...]]`** remains the full shape (O(1) from `dims`).
- New **`NDArrayQ[expr]`** predicate: `True` iff `expr` is an `NDArray`
  (`Protected`).

Adding/multiplying `NDArray` values of **incompatible shape** now prints a
one-line `NDArray::shape` warning (naming the two shapes) and leaves the
expression unevaluated, mirroring `Dot::dotsh`, instead of silently degrading
to opaque symbolic terms. Constructing a **ragged** (non-rectangular)
`NDArray[list]` — unequal sublist shapes, or a mix of list and non-list
siblings — now prints an `NDArray::ragged` warning and stays unevaluated,
instead of silently leaving a valid-looking `NDArray[...]`. (An empty or
non-machine-precision list still stays unevaluated silently, since a symbolic
entry may become packable later.) Both warnings are de-duplicated within a
single evaluation (guarded on the eval clock) so each prints once per input.

`docs/spec/builtins/linear-algebra.md` `## Matrix` section became `## NDArray`.
Tests moved to `tests/test_ndarray.c` (adds rank-3/4, `Depth`, `Length`,
`NDArrayQ` cases). numpy broadcasting and higher-rank `Dot` batching remain
deferred.

## Matrix: a first-class, visibly-distinct dense ndarray (2026-07-07)

Added `Matrix[...]`, a genuine new `Expr` variant (`EXPR_MATRIX`) that packs a
rectangular, machine-precision nested list into a dense row-major `double`
buffer plus rank/dims, stored directly on the node instead of as a
`List[List[...]]` tree. Unlike Mathematica's packed arrays — an invisible
internal optimization a list may or may not have — `Matrix[...]` is always
visibly what it is: `Head`, `MatrixQ`/`VectorQ`/`ListQ`, and printing never
treat it as an ordinary `List`.

- **Core type** (`src/expr.h`, `src/expr.c`): `EXPR_MATRIX` added to
  `ExprType`, with a `MatrixData { int rank; int64_t* dims; double* data; }`
  union member. Extended `expr_new_matrix`, `expr_free`, `expr_eq`,
  `expr_hash`, `expr_unshare`, and `expr_compare` (`src/sort.c`) to handle it.
- **New module** `src/matrix.{c,h}`: `Matrix[nested_list]` constructor,
  `matrix_from_nested_list`/`matrix_to_nested_list` conversions (rejecting
  jagged, empty, or non-machine-precision inputs), and `matrix_init()`
  registering the symbol, `ATTR_PROTECTED`, and a docstring.
- **Introspection**: `Head[Matrix[...]]` is `Matrix`; `Dimensions` reads
  `rank`/`dims` directly (O(1)); `MatrixQ`/`VectorQ` answer per rank;
  `ListQ` is always `False` for a `Matrix` value; `Normal[Matrix[...]]`
  converts back to the equivalent nested `List`.
- **C-level fast paths**:
  - `Dot[Matrix[a], Matrix[b]]` (`src/linalg/dot.c`, `matrix_dot2` in
    `src/matrix.c`) contracts the trailing axis of `a` with the leading axis
    of `b` over raw doubles for rank <= 2 operands, skipping the
    symbolic-`Times`/`Plus`-per-element tensor path entirely. Falls back to
    the generic tensor path (via `Normal` conversion) for higher-rank
    operands; a genuine shape mismatch prints `Dot::dotsh` and leaves the
    call unevaluated, exactly as for nested-list tensors.
  - Elementwise `Plus`/`Times` (`matrix_elementwise` in `src/matrix.c`,
    wired into `builtin_plus`/`builtin_times`) add/multiply same-shape
    `Matrix` operands directly over their flat buffers. Falls through to the
    existing coefficient/base grouping algorithm — treating mismatched
    `Matrix` operands as opaque non-numeric terms — when shapes disagree.
- Machine precision only: any operation that would need a
  non-machine-precision entry (symbolic, exact/rational, BigInt, MPFR) is
  left unpacked/unevaluated rather than silently losing precision.
- New `tests/test_matrix.c` (construction/round-trip across ranks, rejection
  of jagged/symbolic/non-list input, `Head`/`Dimensions`/`MatrixQ`/`VectorQ`/
  `ListQ`, `expr_eq`/`===` structural comparison, and Dot/Plus/Times
  fast-path results checked against the equivalent nested-list path),
  registered in `tests/CMakeLists.txt`.
- `docs/spec/builtins/linear-algebra.md` gains a `## Matrix` section.
## Fix: working-directory-independent module resolution for a relocated/installed binary (2026-07-09)

Reported in [issue #15](https://github.com/stblake/mathilda/issues/15): moving
the `Mathilda` executable to a directory on `PATH` (e.g. `~/.local/bin`) and
running it from elsewhere produced `LoadModule::nofile: cannot locate
src/internal/simp/FullSimplify.m on disk.` and a non-functional kernel, because
the bundled `.m` files were located by paths relative to the *current working
directory*.

There were three independent loaders and only one honoured `MATHILDA_HOME`.
All three now share a single CWD-independent resolver, `mathilda_resolve_internal`
(`src/loadmodule.c`), which searches in order:

1. `$MATHILDA_HOME/<relpath>` (env var → `src/internal`);
2. `<exe_dir>/src/internal/<relpath>` — relative to the running binary, via
   `/proc/self/exe` (Linux), `_NSGetExecutablePath` (macOS), or
   `GetModuleFileNameA` (Windows), so a binary that carries its source tree
   works from **any** directory;
3. `<exe_dir>/../share/mathilda/internal/<relpath>` — installed layout;
4. `$(PREFIX)/share/mathilda/internal/<relpath>` when built with a compile-time
   `MATHILDA_PREFIX` (`make PREFIX=/usr/local`);
5. the previous CWD ladder (`src/internal/`, `../src/internal/`, …) as a
   last resort.

Call sites now routed through the resolver:
- `LoadModule[...]` / `FullSimplify`'s lazy per-family module loads
  (`src/loadmodule.c`);
- the startup bootstrap `init.m` (`src/repl.c`) — previously a hard-coded
  `Get["./src/internal/init.m"]` that failed silently, leaving a broken kernel;
  it now emits a clear `LoadModule::nofile` diagnostic when the tree genuinely
  cannot be found;
- the lazy CRC integral tables (`src/calculus/integrate.c`);
- `init.m` itself now loads `bessel.m` via `LoadModule["bessel.m"]` instead of a
  CWD-relative `Get`.

`makefile`: added an `EXTRA_LIBS` escape hatch (`make EXTRA_LIBS="-llapack
-lX11"`) so site-specific link libraries can be appended without editing the
makefile, and an optional `PREFIX` that defines `MATHILDA_PREFIX` for the
installed-layout search above.

Net effect: `./Mathilda` now works from any working directory; a binary copied
out of the tree works once `MATHILDA_HOME` points at `src/internal` (its
absence now produces a precise diagnostic rather than silent failure).

## Fix: Integrate no longer hangs on symbolic-exponent integrands (2026-07-09)

`Integrate` could hang indefinitely on integrands carrying a symbolic-exponent
power of the integration variable — e.g. `Integrate[x^(k-1) (1-x)^(l-1), {x, 0, 1}]`
(the Euler-Beta integrand), `Integrate[x^k + x^(k-1), x]`, or
`Integrate[x^(k-1) (1-x), x]`. The rational-integration probe (`is_rational_in`,
via `Together`) and the derivative-divides quotient normalisation both fed such
powers to `PolynomialGCD`, which treats `x^k`, `x^(k-1)` as independent
polynomial generators and drives the pseudo-remainder sequence into an unbounded
blow-up. Both paths now decline structurally up front (`has_symbolic_power_in`
in `src/calculus/integrate.c`, `has_symbolic_power_of` in
`src/calculus/integrate_derivdivides.c`): a `Power[b, e]` with `b` depending on
`x` and a non-numeric exponent `e` is provably not rational in `x` and is never
a productive `u`-substitution kernel. The Euler-Beta definite integral now
returns `ConditionalExpression[Beta[k, l], Re[k] > 0 && Re[l] > 0]` promptly, and
purely-rational integrals are unaffected (numeric exponents such as `x^2`,
`x^(1/2)` are never flagged).

## Feature: definite Integrate — Cauchy principal value (2026-07-09)

`Integrate[f, {x, a, b}, PrincipalValue -> True]` now computes the Cauchy
principal value across an interior pole (`src/calculus/integrate_newton_leibniz.c`,
`integrate_newton_leibniz_try_pv`). Valid only for **odd-order** poles (the
integrand changes sign across the pole — detected by a numeric sign-flip test —
so the one-sided divergences cancel), giving `PV = Re[F(b) − F(a)]` with the
real-branch antiderivative (the branch-cut crossing contributes a purely
imaginary `I Pi` residue that `Re` removes). An **even-order** interior pole has
no principal value and emits `Integrate::idiv`. The option defaults to `False`
(unchanged divergence behaviour) and is a no-op when there is no interior pole.
E.g. `∫₋₁¹ dx/x = 0`, `∫₀³ dx/(x−2) = −Log 2`, `∫₀² x/(x²−1) dx = ½ Log 3`, while
`∫₋₁¹ dx/x²` (even pole) still diverges. Covered by
`tests/test_integrate_principalvalue.c`.

## Feature: definite Integrate — half-line ssp / Fresnel / log*rat families (2026-07-09)

Three named half-line families in `src/calculus/integrate_ramanujan.c`, each a
pre-pass / path in the Ramanujan/Mellin engine's auto-dispatch and each with its
own `Method ->` selector and explicit builtin:

- **`Integrate\`SinPowerMonomial`** / `Method -> "SinPowerMonomial"` —
  `Sin[r x]^k / x^m` on `[0, Infinity)` (the ssp family). `TrigReduce` lowers the
  power to harmonics and the analytically-continued Mellin transforms of cos/sin
  (reflection form, so the Gamma pole cancels the trig zero; the scaleless
  constant term is dropped) are summed. E.g. `∫₀^∞ sin²x/x² = Pi/2`,
  `sin³x/x³ = 3Pi/8`, `sin⁴x/x⁴ = Pi/3`.
- **`Integrate\`RationalLog`** / `Method -> "RationalLog"` — `R(x) Log[x]^n` on
  `[0, Infinity)` for a proper rational `R` with negative-real-axis poles. The
  Mellin transform `M(s) = ∫ x^{s-1} R` is summed over the partial fractions of
  `R` (each simple pole `x = -a` contributing `a^{s-1} Pi/Sin[Pi s]`, higher
  order `a^{s-p} Beta[s, p-s]`) and the integral is `n! [coefficient of (s-1)^n
  in M]`. E.g. `∫₀^∞ Log x/((x+1)(x+2)) = (Log²2)/2`,
  `∫₀^∞ Log²x/((x+1)(x+2)) = ⅓Log2(Log²2 + Pi²)`. Positive-real / complex poles
  are declined (out of scope; PV only).
- **`Integrate\`OscillatoryPower`** / `Method -> "OscillatoryPower"` — the named
  entry point for the Fresnel-type `Cos[b x^n]` / `Sin[b x^n]`, which the
  monomial-substitution path of the Mellin engine already closes to
  `Gamma[1/n]/(n b^{1/n}) {Cos,Sin}[Pi/(2n)]`.

Covered by `tests/test_integrate_osclog.c` (unit / dispatch / stress + divergent
and positive-pole negative controls).

## Feature: definite Integrate — Euler-Beta reductions (2026-07-09)

Two new definite mechanisms (`src/calculus/integrate_beta.c`) closing the
non-elementary cases the FTC cannot reach (the antiderivative is an incomplete
Beta), tried after Newton-Leibniz (which owns the integer-power cases through an
elementary antiderivative) and before Ramanujan:

- **`Integrate\`Beta[f, {x, 0, 1}]`** / `Method -> "Beta"` —
  `x^(k-1) (1-x)^(l-1)` on `[0,1]` → `Beta[k, l]`; a `Log[x]^i Log[1-x]^j` weight
  → the mixed parameter derivative `∂^i_k ∂^j_l Beta[k,l]` (Beta already carries
  a PolyGamma derivative rule, so plain `D` suffices). E.g.
  `∫₀^1 √x √(1-x) dx = Pi/8`, `∫₀^1 √x √(1-x) Log[x] dx = (Pi/8)(1/2 - 2 Log 2)`.
- **`Integrate\`TrigPower[f, {x, 0, c}]`** / `Method -> "TrigPower"` —
  `Sin[x]^m Cos[x]^n` over `[0, Pi/2]` → `Beta[(m+1)/2, (n+1)/2]/2`, over `[0, Pi]`
  → `0` when `n` is an odd integer else twice the quarter-period value, over
  `[0, 2 Pi]` → `0` unless both `m, n` are even integers else four times it. E.g.
  `∫₀^{Pi/2} √(Sin x) √(Cos x) dx = Gamma[3/4]²/√Pi`.

Both are gated on the Beta convergence strip; the gate is *proved* on the plain
positivity form (`k > 0`, a sound sufficient condition Simplify can discharge
under a `k > 0` assumption) and *displayed* in the WL `Re[k] > 0` form when the
strip stays undecided (→ `ConditionalExpression`), refuted → unevaluated.
Covered by `tests/test_integrate_beta.c`.

## Feature: definite Integrate — origin-symmetry reduction (2026-07-09)

New definite-integration mechanism `Integrate\`Symmetry[f, {x, -c, c}]`
(`src/calculus/integrate_symmetry.c`), also `Method -> "Symmetry"`, and tried
first-after-residue in the `Integrate[f, {x, a, b}]` auto-dispatch. Over an
interval symmetric about the origin (`c` finite or `Infinity`):

- **odd** `f(-x) == -f(x)` → `0` — no antiderivative needed;
- **even** `f(-x) == f(x)` → `2 Integrate[f, {x, 0, c}]` — often unlocking the
  half-line Ramanujan/Mellin path for a non-rational even integrand.

Parity is a *proof* (`Simplify[f ± f(-x)] === 0`, not the heuristic
`PossibleZeroQ`), and a value is claimed only once the half integral is confirmed
finite, so a divergent principal value (e.g. `∫ x/(1+x²)` on the whole line) is
never mistaken for `0`. Because it runs after the residue families and always
falls through on `NULL`, it only adds coverage. New results now closed by the
auto-dispatch include the **Gaussian** `∫_{-∞}^∞ e^{-x²} dx = Sqrt[Pi]` and
`∫_{-∞}^∞ 1/Sqrt[1+x⁴] dx = Gamma[1/4]²/(2 Sqrt[Pi])` (even → half-line), and
every odd integrand over a symmetric interval short-circuits to `0`. Covered by
`tests/test_integrate_symmetry.c` (unit / regression / stress, incl. the
divergent-odd negative controls).

## Feature + fix: half-line Integrate — symbolic fugacity Bose integral (2026-07-09)

`Integrate[x^(s-1)/(z^-1 Exp[x]-1), {x,0,Infinity}, Assumptions -> Re[s]>1 && 0<z<1]`
— the **general-fugacity Bose integral** `∫₀^∞ x^{s-1}/(z^{-1}e^x-1) dx =
Γ(s) PolyLog(s, z)` — previously **ran until interrupted** and, even when reached,
returned unevaluated. Now it returns `ConditionalExpression[Gamma[s] PolyLog[s, z],
s > 0]` in under a second. Two independent fixes:

- **Exponential-geometric kernel: symbolic fugacity via a sound interval-bound
  gate** (`integrate_ramanujan.c`). The kernel already produced `Γ(s)·PolyLog`
  for a *concrete* `γ'`, but its `-1 ≤ γ' ≤ 1` convergence gate was decided by
  `Simplify[pred, assumptions]`, which discharges only syntactic matches — it
  proves neither `1/z > 0` nor `-1 ≤ -z ≤ 1` from `0 < z < 1`. Two changes: the
  `assumptions` are now threaded down to the recognizer (`dispatch_term` →
  `dispatch_kernel` → `try_recognizers` → `rec_expgeom`), and a small **interval
  arithmetic** prover decides the `A_eff > 0` and `-1 < γ' ≤ 1` bounds over the
  parameter box read off the `Assumptions`. Interval arithmetic yields a
  guaranteed enclosure, so the gate never accepts an inadmissible fugacity — an
  unbounded `z` (no assumption) or a provably out-of-range one (`z > 2`) declines,
  never a wrong value. The lower bound is kept STRICT (`γ' > -1`) to exclude the
  Bose boundary `z = 1`, whose convergence strip differs.

  ```
  Integrate[x^(s-1)/(z^-1 Exp[x]-1), {x,0,Infinity}, Assumptions->Re[s]>1 && 0<z<1]
      -> ConditionalExpression[Gamma[s] PolyLog[s, z], s > 0]
  Integrate[x^(s-1)/(Exp[x]-z),      {x,0,Infinity}, Assumptions->Re[s]>1 && 0<z<1]
      -> ConditionalExpression[Gamma[s] PolyLog[s, z]/z, s > 0]
  Integrate[x^(s-1)/(Exp[x]-1/2),    {x,0,Infinity}, Assumptions->Re[s]>1]
      -> ConditionalExpression[2 Gamma[s] PolyLog[s, 1/2], s > 0]
  ```

- **DiffUnderInt: rational-in-x gate stops the hang** (`integrate_diffunderint.c`).
  The Automatic cascade reaches DiffUnderInt before Ramanujan; differentiating the
  two-parameter integrand w.r.t. `z` gives an inner integrand still carrying `e^x`
  and `x^{s-1}`, which the **general rational half-line** family handed to
  `Apart[g, x]` — a non-terminating rewrite on a non-rational input. A new
  `is_rational_in_x` predicate now guards both rational half-line families
  (`rational_halfline`, `rational_halfline_general`): a form that is not a rational
  function of `x` (a transcendental kernel of `x`, or an `x`-dependent / non-integer
  exponent) is declined up front — this is Mellin/Ramanujan territory. The legit
  rational-in-`x` inner integrals (e.g. `Log[1+a²x²]/(1+x²)`, the decaying sinc)
  are unaffected.

## Perf: whole-line divergence gate + Risch-Norman swell guards (2026-07-09)

`Integrate[Exp[a x]/(Exp[x]-1), {x, -Infinity, Infinity}, Assumptions -> 0<a<1]`
(a genuinely divergent integral — real axis pole at `x=0`, principal value only)
ran for **130s+** before finally returning unevaluated, stalling the residue
test suite past its watchdog.  Two fixes; the integral still returns unevaluated,
now in **<1s**:

- **DiffUnderInt whole-line divergence gate** (`integrate_diffunderint.c`).  A
  two-sided-infinite integrand with a real axis pole cannot be handled by
  differentiation under the integral sign, and pursuing it drove a
  non-terminating escalation `Integrate[x^k f]` (plus `Exp -> Cosh/Sinh` rewrites
  that swell the Risch-Norman linear system).  The method now declines up front
  when the x-only denominator has a real zero, detected by a numeric sign-change
  scan (`Solve[..., Reals]` is unreliable on transcendental denominators —
  `Exp[x]-1` vs `Exp[x]+1` both return the same complex family).  Half-line and
  finite integrals are untouched (a boundary singularity is not interior).

- **Risch-Norman (pmint) symbolic-RowReduce guards** (`intrischnorman.c`).  The
  Phase-4 symbolic linear solve suffers expression swell on parameter-dependent
  matrices; a size cap (>180 rows or >8000 leaf nodes — legitimate parametric
  closures top out ~132×37 / 5500) plus a per-call wall-clock budget abandon the
  ansatz (integral left unevaluated) rather than run for minutes.  This bounds
  the standalone `Integrate[x^2 Exp[a x]/(Exp[x]-1), x]` (was 45s) as well.

## Feature: half-line Integrate — Bose–Einstein / Fermi–Dirac, Frullani, log-weight (2026-07-09)

Three additions to the Mellin / Ramanujan Master Theorem half-line integrator
(`Integrate`RamanujanMasterTheorem`, `Method -> "RamanujanMasterTheorem"`, and
the Automatic cascade), plus a correctness fix to the residue keyhole family.

- **Exponential-geometric kernel `1/(e^{cx}+γ)` → `Γ·PolyLog`.** Expanding the
  geometric series and integrating term by term. Two headline specialisations:
  Bose–Einstein `∫₀^∞ x^{s-1}/(e^{cx}-1) = c^{-s} Γ(s) ζ(s)` (Planck/Debye) and
  Fermi–Dirac `∫₀^∞ x^{s-1}/(e^{cx}+1) = c^{-s} Γ(s) η(s)`. Gated on `c>0`,
  `-1≤γ≤1`, strip `Re s>0` (`Re s>1` for the Bose `γ=-1` case). The Fermi form is
  emitted as `-Γ(s) PolyLog(s,-1)` so it stays finite at `s=1` (where
  `(1-2^{1-s})ζ(s)` is `0·∞`); the Bose case is canonicalised to `Zeta`.

  ```
  Integrate[x^3/(Exp[x]-1), {x,0,Infinity}]                    -> Pi^4/15
  Integrate[x^3/(Exp[x]+1), {x,0,Infinity}]                    -> 7 Pi^4/120
  Integrate[1/(Exp[x]+1), {x,0,Infinity}]                      -> Log[2]
  Integrate[x^(s-1)/(Exp[x]-1), {x,0,Infinity}, Assumptions->s>1] -> Gamma[s] Zeta[s]
  Integrate[1/(Exp[Sqrt[x]]-1), {x,0,Infinity}]               -> Pi^2/3   (monomial sub)
  ```

- **Frullani pre-pass** `∫₀^∞ (f(ax)-f(bx))/x dx = (f(0⁺)-f(∞)) Log(b/a)`, run on
  the whole integrand before Expand (each half is individually divergent). The
  scale ratio is read structurally and the pairing verified by the exact identity
  `(t₁ /. x→ρx)+t₂ = 0`; `a,b>0` is proved as same-sign scales.

  ```
  Integrate[(Exp[-2 x]-Exp[-5 x])/x, {x,0,Infinity}]          -> Log[5/2]
  Integrate[(ArcTan[5 x]-ArcTan[2 x])/x, {x,0,Infinity}]      -> (Pi/2) Log[5/2]
  Integrate[(Exp[-a x]-Exp[-b x])/x, {x,0,Infinity}, Assumptions->a>0&&b>0] -> Log[b/a]
  ```

- **`Log[x]^k` weight** = the `k`-th `s`-derivative of the base transform
  (`∂_s x^{s-1} = x^{s-1} Log x`); the open convergence strip carries unchanged.

  ```
  Integrate[Log[x]/(1+x^2), {x,0,Infinity}]                   -> 0
  Integrate[x Log[x] Exp[-x], {x,0,Infinity}]                 -> 1 - EulerGamma
  ```

## Fix: residue keyhole/Mellin — higher-order poles (2026-07-09)

`Integrate[Sqrt[x]/(1+x)^2, {x,0,Infinity}]` returned **`0`** (should be `π/2`):
the keyhole family summed `z_k^{s-1} Res(R, z_k)`, valid only for simple poles —
a pure double pole has `Res(R)=0`, silently zeroing the answer. The `Method ->
"Residue"` family runs before Ramanujan in the cascade, so it shadowed the
correct value. Now the sum is over residues of the **full** integrand
`z^{s-1} R(z)` (shift `w=z−z_k`, expand the analytic `(1+w/z_k)^{s-1}`), correct
for any pole order; simple poles keep the fast closed form. Examples now correct:
`∫₀^∞ √x/(1+x)² = π/2`, `∫₀^∞ √x/(1+x)³ = π/8`, `∫₀^∞ x^{1/3}/(1+x)² = 2π/(3√3)`.

## Fix: Simplify — recover integrand from nested-radical antiderivatives (2026-07-09)

Follow-on to the sign-indefinite-radicand fix below. `Simplify` now folds the
derivative of a nested-radical antiderivative all the way back to the original
integrand for a broader family, and no longer hangs on a doubly-nested case:

```
D[Integrate[x / Sqrt[1 + Sqrt[1-x]], x], x] // Simplify
  -> x / Sqrt[1 + Sqrt[1 - x]]
D[Integrate[x Sqrt[1-x] / Sqrt[1 + Sqrt[1-x]], x], x] // Simplify
  -> (x Sqrt[1 - x]) / Sqrt[1 + Sqrt[1 - x]]
D[Integrate[1/x/Sqrt[1-x]/Sqrt[1 + Sqrt[1-x]], x], x] // Simplify   (no longer hangs)
```

Three changes to `simp_radical_rational` (`src/simp/radrat.c`):

- **Numerator rationalisation (the dual of Step 4b).** The reduced
  polynomial-in-generators normal form is one representative of its
  equivalence class; the class also contains forms with a generator in the
  *denominator*. A new pass multiplies `num`/`den` by `g_i^k` and reduces the
  numerator modulo `g_i^{q_i} - B_i`, trading a numerator radical for the
  (often simpler) base polynomial `B_i`. Both sides are multiplied by the same
  generically-nonzero `g_i^k`, so it is exact on the principal branch; the
  strict score gate keeps a variant only when it is genuinely simpler. This is
  what turns `(1 - Sqrt[1-x]) Sqrt[1+Sqrt[1-x]]` into `x / Sqrt[1+Sqrt[1-x]]`.

- **Ring symbol set gathered from the input, not just the substituted form.**
  A variable appearing *only inside* a radical base (e.g. `x` in `Sqrt[1-x]`
  when the numerator carries no bare `x`) vanished from the substituted
  expression, so the relation-usability guard dropped its relation
  `g^2 - (1-x)`. That left odd generator powers (`g^3`) unreducible, so the
  integrand came back expanded (`Sqrt[1-x] - (1-x)^(3/2)`) instead of `x
  Sqrt[1-x]`. Collecting the parameter set from the original `input` keeps the
  relation. (Also fixes one previously-failing `simplify_tests` case.)

- **Bail on Laurent relations.** A nested base holding an *inverse* inner
  radical (e.g. `1 + 1/Sqrt[1-x]` → `1 + g^(-2)`) yields a Laurent — not
  polynomial — relation, which radrat's per-generator univariate reduction
  cannot handle; the downstream `PolynomialExtendedGCD` blew up and hung.
  `rr_has_nonpoly_gen_power` now detects this and abandons the pass (the input
  is outside radrat's polynomial-ring model), so `Simplify` returns a finite
  result instead of hanging. Full recovery of such integrands needs a genuine
  multivariate algebraic-tower normaliser and remains future work.

- **Verification.** The three reported cases + prior nested/cube-root
  round-trips; no regressions across `simplify`/`simp`/`rat`/`radical_simplify`/
  `integrate_derivdivides`/`integrate_jeffrey`/`simp_algebraic_cuberoot`/
  `trigrat` suites (`simplify_tests` FAILs 4 → 3); valgrind-clean (radrat path
  identical to the `Sin[1.0]` baseline, zero `rr_*` frames in leak stacks).

## Fix: Simplify — multi-generator radical normalisation for sign-indefinite radicands (2026-07-09)

`Simplify` can now normalise rationals of two or more radical generators when a
radicand is not provably positive — most importantly **nested** radicals whose
inner base is `1 - x` and similar. For example the derivative of a radical
antiderivative now collapses back to its integrand:

```
Integrate[Sqrt[1 + Sqrt[1 - x]], x]
  -> 1/15 (-12 (1 + Sqrt[1 - x])^(5/2) + 20 (1 + Sqrt[1 - x])^(3/2))
D[%, x] // Simplify           -> Sqrt[1 + Sqrt[1 - x]]
Simplify[((1-x)^(3/2) - Sqrt[1-x]) / (x (1-x))]   -> -1/Sqrt[1 - x]
```

- **Root cause.** `simp_radical_rational` (`src/simp/radrat.c`) previously bailed
  the entire pass unless every radical base was *provably positive*
  (`rr_pos_base`). A base like `1 - x` fails that test because of the `-x` term,
  so nested radicands such as `1 + Sqrt[1 - x]` were never collected and the
  reduction never ran.

- **Fix.** Replaced the positive-base gate with a real-radicand check
  (`rr_real_base`): symbols are assumed real and only bases carrying an explicit
  complex constant (a `Complex[..]` node or the imaginary unit `I`) are rejected.
  This is branch-safe: radrat's reduction is an *exact principal-branch rewrite*
  — each `B^(p/q)` maps to a fresh generator via the identity `(B^(1/q))^q = B`,
  generators are never merged across distinct bases, and the radicals are
  reconstructed verbatim before the final `Cancel`/`Factor`. So the result is
  equal to the input on the principal branch for any real radicand regardless of
  sign, and radrat still refuses branch-unsafe merges like
  `Sqrt[a] Sqrt[b] -> Sqrt[a b]` (distinct generators are kept separate; the
  score gate returns the input unchanged).

- **Verification.** Reported case + `D[Integrate[...]]` round-trip; numeric
  cross-checks (including the `1 - x < 0` region); no regressions across
  `simplify`/`simp`/`rat`/`radical_simplify`/`integrate_derivdivides`/
  `integrate_jeffrey`/`simp_algebraic_cuberoot` suites (identical FAIL set before
  and after); valgrind-clean (leak totals identical to the `Sin[1.0]` baseline).

## Feature: DiffUnderInt — non-even rational + Gaussian families, cleaner output (2026-07-08)

Extended `Integrate`DiffUnderInt` (differentiation under the integral sign) with
two new inner-integral families, an output-cleanup pass, and a message-noise
fix (`src/calculus/integrate_diffunderint.c`):

- **General (non-even) rational half-line** `rational_halfline_general`:
  `∫₀^∞ R(s) ds` for a real rational `R` with no poles on `[0,∞)` and degree
  drop ≥ 2, without the evenness restriction of the existing family. Partial-
  fractions `R`; each simple factor `(s+r)^m` and irreducible quadratic
  `((s+β)²+γ²)` integrates to a real `ArcTan`/`Log` boundary value (the
  `Log(s→∞)` coefficients must cancel, else divergent → declined). This closes
  the **decaying sinc** whose Laplace image `M(s)=q/((s+p)²+q²)` is non-even,
  e.g. `Integrate[Exp[-c x](1-Cos[a x])/x^2, {x,0,∞}, Assumptions->{a>0,c>0}]`
  → `a ArcTan[a/c] − (c/2) Log[1+a²/c²]`. The `ArcTan` boundary value is emitted
  as `ArcTan[γ/β]` (not `π/2 − ArcTan[β/γ]`) when `β>0`, so the parameter
  back-integration — which chokes on `ArcTan` of a reciprocal argument — closes.
- **Gaussian moment family** `gaussian_halfline`:
  `∫₀^∞ c xⁿ e^{-p x²}{1|Cos[q x]} dx` → `(c/2)Γ((n+1)/2)p^{-(n+1)/2)` (no trig)
  or `(c/2)Sqrt[π/p]e^{-q²/4p}` (cosine, `n=0`). Paired with a direct **Gaussian
  parameter back-integration** `∫ c e^{-k p²} dp = c(1/2)Sqrt[π/k] Erf(Sqrt[k] p)`
  (the engine produces no `Erf` antiderivative). Closes
  `Integrate[Exp[-x^2] Sin[a x]/x, {x,0,∞}]` → `(π/2) Erf[a/2]`. The Sin-Gaussian
  (Dawson/Erfi) moment is deliberately declined. The blanket up-front Gaussian
  decline is removed; half-line inner integrals still go only to closed-form
  families (never the hanging general engine).
- **Output cleanup** `diui_finalize`: `Simplify` canonicalises `c Log[w]` into the
  contracted `Log[w^c]` form (e.g. `Log[1/Sqrt[u]] + Log[Sqrt[v]]`); a 1-arg
  `PowerExpand` pulls those back out, kept only when provably equal to the
  verified answer (`Simplify[clean − I] === 0`), so it can never corrupt a
  result. Cleans the Frullani/Laplace outputs, e.g.
  `Integrate[(Exp[-a x]−Exp[-b x])Cos[c x]/x, {x,0,∞}, …]`
  → `−(1/2)Log[a²+c²] + (1/2)Log[b²+c²]`.
- **Message-noise fix:** the method's divergent-form probes (`1/0` in a
  degenerate `M(s)`, `Limit` at a pole) emitted a flood of spurious
  `Power::infy`/`Infinity::indet` warnings. The search is now wrapped in
  `arith_warnings_mute_push/pop` (the idiom `Limit` already uses), so declining
  e.g. `Integrate[Sin[a x] Sin[b x]/x^2, {x,0,∞}, …]` (a genuinely piecewise
  result) is silent. Return values that gate the search are unchanged.

Tests: `tests/test_integrate_diffunderint.c` gains `test_rational_halfline_general`
and `test_gaussian`. All DiffUnderInt and integration suites pass; no new leaks
(the residual are the pre-existing Goursat/Factor evaluate-class leaks).

## Feature: Mellin / Ramanujan method — full half-line corpus (2026-07-08)

## Feature: Mellin / Ramanujan Master Theorem definite integration (2026-07-08)

- **Motivation.** Every prior definite-integration mechanism was
  antiderivative-based (Newton-Leibniz), contour-based (residue, line), or
  parametric (DiffUnderInt). None was series/transform-based — the approach that
  closes half-line `∫₀^∞ x^{s-1} f(x) dx` of a *transcendental* `f`. The residue
  "Mellin" family is rational-only, so Gaussian moments (`∫₀^∞ xⁿ e^{-x²} dx`),
  `Γ`/Bessel/sine transforms, etc. had no systematic path.
- **New method (`src/calculus/integrate_ramanujan.c`).** A Mellin-transform
  table plus operational rules realizing Ramanujan's Master Theorem: the
  integrand is Expanded and, term by term, decomposed into `C · x^ρ · f(λx)`;
  the kernel `f` is matched against base transforms (exponential, Gaussian,
  algebraic binomial `(p+q xᵐ)^{-a}`, `Cos`, `Sin`, `BesselJ`) and the power
  prefactor sets `s = ρ+1`. Each application is gated on its convergence strip
  (checked by `Simplify` — numerically for numeric `s`, or against `Assumptions`
  for symbolic `s`), so results are correct by construction with **no** numeric
  crosscheck. The trig transforms use the reflection-formula form
  `π/(2 Cos(πs/2) Γ(1-s))`, regular at `s=0`, so `∫₀^∞ Sin[x]/x dx = π/2` needs
  no limit. Sums integrate term by term; a product of two transcendental kernels
  (Mellin convolution / Meijer-G), finite intervals, and chained-assumption
  two-sided strips are declined (unevaluated, never wrong).
- **Routing.** `Method -> "RamanujanMasterTheorem"` (alias `"Mellin"`), the
  explicit `Integrate\`RamanujanMasterTheorem[f, {x, 0, Infinity}]` builtin
  (`ATTR_PROTECTED | ATTR_READPROTECTED`, docstring), and the Automatic cascade
  (after Newton-Leibniz, before DiffUnderInt) in `integrate_definite`
  (`src/calculus/integrate.c`).
- **Closes.** `Exp[-x²]`→`√π/2`, `x²Exp[-x²]`→`√π/4`, `x^{s-1}Exp[-x]`→`Γ[s]`
  (with `s>0`), `1/(1+x²)`→`π/2`, `Sqrt[x]/(1+x)²`→`π/2`, `Sin[x]/x`→`π/2`,
  `Cos[x]/Sqrt[x]`→`Sqrt[π/2]`, `BesselJ[0,x]`→`1`.
- **Tests.** `tests/test_integrate_ramanujan.c` (8 groups). Valgrind-clean
  (1× and 20× pinned runs equal the baseline leak exactly). Docs:
  `docs/spec/builtins/calculus.md` (**Mellin / Ramanujan Master Theorem**).
- **Note (pre-existing, out of scope).** The residue "Mellin" family returns a
  wrong `0` for branch-power `x^p/(1+x)ⁿ` under `Method -> "Residue"`; since it
  runs first under Automatic it can mask the correct RMT value. Untouched here
  (`integrate_residue.c` unchanged) — flagged for a follow-up.

## Feature: Asymptotic behaviour at Infinity — `Series`, function values, and `Limit` (2026-07-08)

- **Motivation.** `Limit[Erf[x], x -> Infinity]` returned unevaluated. The root
  cause was a `Series` deficiency: `Series[Erf[x], {x, Infinity, n}]` had no

  - `Erfc` is the same multiplier without the leading `1` (`Erfc = 1 - Erf`);

  Infinity through one mechanism — `Erf`, `Erfc`, `Erfi`, `Exp`, `Log`, the
  hyperbolics, `Gamma`, `Zeta`, `ProductLog`, the inverse trig/hyperbolics —
  while oscillatory heads (`Sin`, `Cos`) correctly stay `Indeterminate`. A bare

## Feature: `Integrate[…, Assumptions -> …]` + three new contour families (2026-07-07)

- **Summary.** `Integrate` now accepts an `Assumptions -> …` option on the
  definite (and indefinite) forms — previously any option other than `Method`
  was rejected with `Integrate::method`. Under assumptions constraining the
  integrand's free parameters, the residue-theorem method (`src/calculus/
  integrate_residue.c`) evaluates several classical **parametric** definite
  integrals, and three new correct-by-construction contour families are added.

- **New closed forms.**
  - Fourier/Jordan with symbolic parameters:
    `Integrate[Cos[k x]/(x²+a²), {x,-∞,∞}, Assumptions -> {a>0, k>0}] = (π/a) E^{-a k}`
    (and `Sin → 0`); rational too: `Integrate[1/(x²+a²), {x,-∞,∞}, a>0] = π/a`.
  - **Keyhole / Mellin** on `(0,∞)` for a branch power × rational,
    `x^p R(x)` (`p ∉ ℤ`): `Integrate[x^(1/3)/(x²+1), {x,0,∞}] = π/√3`,
    `Integrate[Sqrt[x]/(x²+1), {x,0,∞}] = π/√2`.
  - **Sector** on `(0,∞)` with a symbolic exponent `n`:
    `Integrate[1/(1+x^n), {x,0,∞}, n>1] = (π/n) csc(π/n)`; monomial numerators
    `x^m/(c+x^n)` supported.
  - **Rectangular / quasi-periodic** on `(-∞,∞)` for `Exp[c x] R(Exp[x])`,
    reduced to the keyhole core by `w = Exp[x]`:
    `Integrate[Exp[a x]/(Exp[x]+1), {x,-∞,∞}, 0<a<1] = π csc(π a)`.

- **Design.** Symbolic-parameter poles/frequencies are classified by reading the
  sign at a single generic point of the assumption-pinned region (a
  sign-consistent instantiation), while the residue arithmetic stays symbolic —
  so answers are **correct by construction, with no `NIntegrate` crosscheck**
  (bugs surface rather than being masked). Convergence/applicability gates
  (`n > m+1`, `0 < s < deg`-drop, `c > 0`) are checked against the
  assumption-**guaranteed** interval bounds, so an under-constrained problem
  (e.g. only `n > 0`) or a two-sided-unbounded parameter is **refused**
  (unevaluated), not guessed. Radical pole locations are `PowerExpand`-cleaned
  under all-positive parameters (`Sqrt[-4a²] → 2 I a`) and real-parameter
  conjugation uses `I → −I`, so answers close to clean forms (`π/a`, `π csc(π a)`).

- **Scope.** The log-keyhole `∫₀^∞ Log[x] R(x)` with symbolic on-circle poles
  (`Integrate[Log[x]/(x²−2x Cos[a]+1), {x,0,∞}, 0<a<π]`, value `0`) is **not**
  handled: it needs assumption-aware `Arg`/`Log` `(0,2π)`-branch reasoning that
  does not yet exist. (The `Exp[I k x]` / `Power[E,…]` kernel form, noted here as

## Feature: Fourier/Jordan family recognises the bare `Exp[I k x]` (`Power[E, …]`) kernel (2026-07-08)

- **Summary.** The residue-theorem Fourier/Jordan family (`src/calculus/
  integrate_residue.c`) previously matched only `Cos[k x]` / `Sin[k x]` kernels.
  The evaluator normalises `Exp[I k x]` to `Power[E, I k x]`, so a complex-
  exponential integrand fell through unrecognised and returned unevaluated. It
  is now handled directly, closing the whole `Cos`/`Sin`/`Exp` trio.

- **New closed forms.**
  `Integrate[Exp[I x]/(x²+1), {x,-∞,∞}] = π/E` (and the `E^(I x)` spelling);
  `Exp[2 I x]/(x²+1) = π/E²`; negative frequency `Exp[-I x]/(x²+1) = π/E`
  (lower-half-plane closure); higher-order pole `Exp[I x]/(x²+1)² = π/E`;
  complex-valued `x Exp[I x]/(x²+1) = I π/E`; and with symbolic parameters
  `Integrate[Exp[I k x]/(x²+a²), {x,-∞,∞}, {a>0,k>0}] = (π/a) E^{-a k}`.

- **Implementation.** A small `kernel_arg(e)` helper returns the phase argument
  `u`: `args[0]` for `Cos`/`Sin`/`Exp`, and the exponent `args[1]` for
  `Power[E, u]` (accepted **only** with base `E`, so ordinary powers and rational
  denominators like `(1+x²)^-1` are rejected). `match_kernel` classifies the
  `Power[E, ·]` form as `KERN_EXP`, and the family builds its Jordan integrand
  from `kernel_arg` rather than a hard-coded `args[0]`. A real-exponent
  `Exp[a x]` still fails the purely-imaginary-slope test, so it correctly stays
  with the rectangular / quasi-periodic family rather than being grabbed here.

- **Tests.** `test_integrate_residue.c` gains bare/`E^`-spelling, higher-order,
  negative-frequency, complex-valued, and symbolic-parameter `Exp` cases in
  `test_family_fourier` and `test_fourier_symbolic`. valgrind-clean.

## Fix: `Series`/`Residue` gave wrong Laurent coefficients at a higher-order *complex* pole (2026-07-08)

- **Symptom.** `Residue[Exp[I x]/(x²+1)², {x, I}]` returned `-I/(4E)` instead of
  the correct `-I/(2E)`, and `Series[Exp[I x]/(x²+1)², {x, I, 0}]` reported a
  wrong `(x-I)^(-1)` coefficient. Downstream, the Fourier/Jordan contour family
  mis-evaluated integrals with repeated poles:
  `Integrate[Cos[3x]/(x²+1)², {x,-∞,∞}]` gave `(3/2)π/E³` instead of `2π/E³`,
  and `Cos[x]/(x²+1)²` gave `π/(2E)` instead of `π/E`. Simple poles and
  *purely rational* integrands (no transcendental kernel) were unaffected.

- **Root cause.** The series engine expands to a padded internal order
  (`order + pad`) so that Laurent/Puiseux operands survive the loss of O-term
  accuracy through multiplication. The `pad = 12` (vs. tight `pad = 2`) choice
  was gated on `x0` being a *literal number*, but the test only accepted
  `EXPR_INTEGER/REAL/BIGINT`. A complex pole such as `I = Complex[0,1]` is an
  `EXPR_FUNCTION`, so it was misclassified as symbolic and demoted to `pad = 2`.
  For an order-≥2 pole with a non-monomial denominator (`(x²+1)²`, which the
  engine must expand and invert at `z=I`), that left the analytic Taylor factor
  one term short, so the residue's product-rule cross term
  (`A[t^(-2)]·B[t^(+1)]`) was dropped — halving the double-pole residue.

- **Fix.** Both pad-selection sites in `src/calculus/series.c` now use
  `expr_is_numeric_like(x0)`, which additionally accepts `Complex[num,num]` and
  `Rational` — concrete numbers whose `so_inv` convolution coefficients stay
  numeric and bounded (so they safely earn the full Laurent pad), while a
  free/symbolic `x0` still gets the tight pad that guards against O(N²)
  symbolic-coefficient blow-up. This repairs `Series`, `Residue`, and the
  Fourier/Jordan contour family together.

- **Tests.** `test_residue.c` gains a complex order-2 pole with a non-monomial
  denominator; `test_integrate_residue.c` gains double- and triple-pole Fourier
  integrals (`Cos[x]/(x²+1)² = π/E`, `Cos[3x]/(x²+1)² = 2π/E³`,
  `Cos[x]/(x²+1)³ = 7π/(8E)`). valgrind-clean (no `src/` frames in lost stacks).

## Fix: divergent integral `Integrate[1/(x^2-1), {x,-Infinity,Infinity}]` returned a spurious finite value (2026-07-07)

- **Symptom.** `Integrate[1/(x^2-1), {x, -Infinity, Infinity}]` (and the explicit
  `Method -> "NewtonLeibniz"` form) returned `-I Pi` instead of warning
  `Integrate::idiv` and returning unevaluated. The integrand has non-integrable
  poles at `x = ±1` inside the interval, so the integral genuinely diverges.

- **Root cause.** The antiderivative is `F = -ArcTanh[x]`, and the poles at `±1`
  *were* detected and split at. But `ArcTanh[±1]` did not evaluate — the
  `ArcTanh`/`ArcCoth` builtins only special-cased `0`, leaving `ArcTanh[1]`
  symbolic. The Newton–Leibniz one-sided boundary evaluations therefore produced
  finite-looking `±ArcTanh[1]` terms that cancelled algebraically across the
  telescoping sum, masking the divergence and leaving the branch residue `-I Pi`.

- **Fix (two parts).**
  1. `builtin_arctanh` / `builtin_arccoth` (`src/hyperbolic.c`) now evaluate the
     genuine poles `ArcTanh[1] = Infinity` and `ArcCoth[1] = Infinity`
     (negatives via the existing odd-function fold), matching Mathematica.
  2. `nl_eval_at` (`src/calculus/integrate_newton_leibniz.c`) now keeps a bare
     directed-infinity produced by direct substitution at a finite interior pole
     (rather than discarding it and falling back to a one-sided `Limit` the
     engine may leave unevaluated). The infinite boundary value propagates to the
     divergence path, so `Integrate::idiv` is emitted and the integral is left
     unevaluated.

- **Verification.** The divergent case now warns `Integrate::idiv` and returns
  unevaluated; convergent regressions (`Integrate[1/(x^2+1), {x,-Infinity,Infinity}]
  = Pi`, `Integrate[1/(2+Cos[x]), {x,0,2Pi}] = 2Pi/Sqrt[3]`, etc.) are unchanged.
  `hyperbolic_tests`, `integrate_newton_leibniz_tests`, and `limit_tests` pass.

### Follow-up: the residue method also reports the divergence

- **Symptom.** `Integrate[1/(x^2-1), {x,-Infinity,Infinity}, Method -> "Residue"]`
  returned unevaluated but emitted **no** diagnostic. The residue families
  detected the real-axis pole and bailed silently, and the explicit `"Residue"`
  method has no Newton–Leibniz fallback, so no `Integrate::idiv` was ever
  emitted.

- **Fix.** The residue path now distinguishes a *conclusive* divergence — a
  genuine (non-removable) pole on the integration contour: the real axis for the
  rational Family A, or the unit circle for the trig Family C — from a merely
  unrecognized integrand (an undecidable/symbolic root). `integrate_residue_try`
  gained a `bool* diverges` out-parameter set only in the conclusive case. The
  dispatcher (`integrate_definite`) emits `Integrate::idiv` once and stops when
  `diverges` is set, so both the explicit `Method -> "Residue"` path and the
  `Automatic` path report the divergence with a single warning (the
  Newton–Leibniz fallback is suppressed to avoid a duplicate). The
  `Integrate::idiv` emitter was promoted from a static Newton–Leibniz helper to
  a shared `integrate_emit_idiv`.

- **Verification.** `Method -> "Residue"` and `Automatic` both warn
  `Integrate::idiv` and stay unevaluated for `1/(x^2-1)` on ℝ and for a trig
  integrand whose denominator vanishes on the axis (`1/(1-Cos[x])` over
  `[0,2Pi]`). Convergent residue results are unchanged (`1/(1+x^4)` → `Pi/Sqrt[2]`,
  `Cos[x]/(1+x^2)` → `Pi/E`, `1/(2+Cos[x])` → `2Pi/Sqrt[3]`, `Sin[x]/x` → `Pi`,
  half-line `1/(1+x^4)` over `[0,Infinity)` → `Pi/(2Sqrt[2])`).
  `integrate_residue_tests`, `integrate_newton_leibniz_tests`, and `residue_tests`
  pass.

## Definite integration by the residue theorem (2026-07-07)

New `src/calculus/integrate_residue.c` adds a residue-theorem method for the
classical **improper** and **periodic** real integrals, engaged by
`Integrate[f, {x, a, b}]` **before** Newton-Leibniz (also `Method -> "Residue"`
and the explicit `Integrate`ContourResidue[f, {x, a, b}]`). Each family's value
is **correct-by-construction** — the residue theorem gives the exact value once
the structural gates hold, so there is **no numeric quadrature (`NIntegrate`)
crosscheck**. The only post-hoc gate is a self-consistency check that the sum
closed to a scalar (no surviving `x`/`Root`) and, for the real-valued families,
that its imaginary part vanishes (a residual `Im` betrays a mis-classified pole);
a failure returns `NULL` and Newton-Leibniz takes over unchanged.

- **Family A — rational on `(-∞, ∞)`.** `f = P/Q`, `deg Q ≥ deg P + 2`, **no real
  pole** → `2 π i · Σ Res` over upper-half-plane poles.
  `Integrate[1/(1+x^4), {x, -Infinity, Infinity}]` → `Pi/Sqrt[2]`;
  `1/(1+x^2)^2` → `Pi/2` (order-2 pole); `1/((x^2+1)(x^2+4))` → `Pi/6`. A real
  pole of a pure rational integrand is a genuine singularity → unevaluated.
- **Family B — Fourier / Jordan on `(-∞, ∞)`.** `f = R(x)·K`,
  `K ∈ {Cos[a x], Sin[a x], Exp[I a x]}`, `R` rational with deg-drop `≥ 1`:
  `J = 2 π i · Σ_UHP Res[R Exp[I a x]]`, then `∫R Cos = Re[J]`, `∫R Sin = Im[J]`
  (via the `Conjugate` identities). `Cos[x]/(1+x^2)` → `Pi/E`;
  `x Sin[x]/(1+x^2)` → `Pi/E`; `Cos[2x]/(x^2+9)` → `(Pi/3) E^-6`.
- **Family C — rational-in-`{Sin,Cos}` over a full period** `(0,2π)`/`(-π,π)`,
  via `z = Exp[I x]` on the unit circle → `2 π i · Σ Res` inside the unit disk.
  `1/(2+Cos[x])` → `2 Pi/Sqrt[3]`; `1/(5-4 Cos[x])` → `2 Pi/3`.
- **Removable axis singularity (Fourier).** A **simple** real-axis pole of `R`
  where the kernel supplies a matching zero (so `f = R·K` is analytic there)
  contributes a **half residue** `π i · Res`, the indented-contour value:
  `Integrate[Sin[x]/x, {x, -Infinity, Infinity}]` → `Pi`. A genuine axis pole
  (`Cos[x]/x`, kernel nonzero at `0`) diverges → unevaluated (plain `Integrate`
  computes no principal value).
- **Even half-line.** `Integrate[1/(1+x^4), {x, 0, Infinity}]` → `Pi/(2 Sqrt[2])`
  via `∫₀^∞ = ½ ∫₋∞^∞` when `PossibleZeroQ[f(x) - f(-x)]`.

Algebraic residue sums close through `RootReduce` (factoring `Pi` out of the
otherwise-transcendental-blocked reduction); a surviving `Root` or non-negligible
imaginary part is treated as a mis-fire → unevaluated. Dispatch: `METHOD_RESIDUE`
added to `integrate.c`; the definite `Method` parser now reports the chosen
mechanism so `Method -> "NewtonLeibniz"` still bypasses the residue path and
strict `Method -> "Residue"` takes no Newton-Leibniz fallback. Tests in
`tests/test_integrate_residue.c`; docs under *Contour / residue-theorem definite
integration* in `docs/spec/builtins/calculus.md`.

## Residue: algebraic (sum-of-radical) pole locations (2026-07-07)

`Residue[f, {z, z0}]` (and the newly-exported `residue_compute`) now detects a
pole whose location is a **sum of radicals** — e.g.
`Residue[1/(1+4z+z^2), {z, -2+Sqrt[3]}]` → `1/(2 Sqrt[3])`, previously a wrong

integrator above.

## Residue: symbolic residues (2026-07-07)

New `src/calculus/residue.c` adds the Wolfram-faithful **`Residue[f, {z, z0}]`**,
the coefficient of `(z - z0)^-1` in the Laurent expansion of `f` — the symbolic
counterpart to the existing numerical `NResidue`. Registered in `core_init`
(after `limit_init`), symbol interned in `sym_names.c`, docstring in `info.c`,
attribute `Protected`.

- **Method.** Reads the `-1` coefficient out of `Series[f, {z, z0, 0}]`. An
  order-0 expansion spans exponent `-1` for most integrands; when the pole's
  leading behaviour is carried by an unknown function (`f[z]/z^5`) the series
  engine truncates relative to that function's depth, so `Residue` adaptively
  raises the expansion order until the `-1` coefficient is explicit (bounded, to
  stay safe at essential singularities). A fractional-power (Puiseux) expansion
  (`den > 1`) is a **branch point** → left unevaluated (e.g.
  `Residue[1/Sqrt[z], {z, 0}]`).
- **Coverage.** Simple, higher-order, and transcendental poles
  (`1/z → 1`, `1/z^2 → 0`, `(z+1)/(z^2(z-2)) → -3/4`, `Cot[z] → 1`,
  `1/Sin[z]^5 → 3/8`, `1/Sin[z]^7 → 5/16`); complex and algebraic pole locations
  (`1/(z^2+1)` at `I → -I/2`, `x^3/(x^4-2)` at `2^(1/4) → 1/4`); unknown-function
  numerators (`f[z]/z^5 → 1/24 Derivative[4][f][0]`); a high-order `Zeta` pole
  (`Zeta[z]/(z-1)^10` at `1 → -StieltjesGamma[9]/362880`); analytic points → `0`.
  `Residue[]` / `Residue[x]` emit `Residue::argm` and stay unevaluated.
- **Verification.** New `tests/test_residue.c` (8 groups, incl. `NResidue`
  numeric cross-checks via Cauchy's theorem); valgrind clean (no Mathilda-source
  leaks — only the macOS dyld/objc baseline).

See [`builtins/calculus.md`](../builtins/calculus.md#residue).

## Log: exact pure-imaginary arguments (2026-07-07)

`Log[z]` now simplifies exact pure-imaginary arguments on the principal branch,
matching Mathematica (`src/logexp.c`, `builtin_log`).

- **New rule.** For `z = Complex[0, b]` with `b` an exact real numeric (integer,
  bigint, rational, or exact real), `Log[b I]` rewrites to
  `Log[Abs[b]] + Sign[b] (I Pi)/2`:
  - `Log[I]` → `(I Pi)/2`, `Log[-I]` → `-((I Pi)/2)`
  - `Log[2 I]` → `(I Pi)/2 + Log[2]`, `Log[-3 I]` → `-((I Pi)/2) + Log[3]`
  - `Log[I/2]` → `(I Pi)/2 + Log[1/2]`
- **Branch-correct.** The imaginary offset is `+Pi/2` for `b > 0` and `-Pi/2`
  for `b < 0` (arg of `b I`); the magnitude term reuses the evaluator (`Log[1]`
  collapses to `0`, so `Log[I]` gives just `(I Pi)/2`). Verified `Exp[Log[b I]]`
  round-trips numerically and the closed form matches `N[Log[b I]]` (`clog`).
- **Scope.** Only exact pure-imaginary (`Re = 0`) arguments are handled;
  inexact imaginaries (`Log[-2. I]`) fall through to the existing numeric path,
  and non-pure-imaginary complexes (`Log[2 + 3 I]`) stay symbolic — unchanged.
- Tests: existing `logexp_tests` pass. (A pre-existing unrelated FAIL in
  `logexp_simplify_tests` — `Simplify[(a^p)^q, a>0]` — is untouched.)

See [`builtins/elementary-functions.md`](../builtins/elementary-functions.md).

## Integrate: complex line / contour integration (2026-07-07)

New `src/calculus/integrate_line.c` adds contour integration in the complex
plane, wired into `Integrate`'s definite-integral dispatch (`integrate.c`) and
exposed explicitly as `Integrate\`LineIntegral` and `Integrate\`PathSingularPoints`.

- **New surface.** `Integrate[f, {x, a, b}]` with a non-real endpoint, and the
  polyline form `Integrate[f, {x, z0, z1, …, zn}]`, evaluate the contour integral
  of `f` along the straight segments `z0 → z1 → … → zn`.  Real-endpoint two-point
  specs are unchanged (still the real-axis Newton-Leibniz path).
  - `Integrate[1/x, {x, 1 - I, 2 + 3 I}]` → `Log[2 + 3 I] - Log[1 - I]`
  - `Integrate[z^2, {z, 0, 1 + I}]` → `-2/3 + 2/3 I`; `Integrate[Cos[z], {z, 0, I}]`
    → `I Sinh[1]`; `Integrate[1/(z^2 + 1), {z, 0, 1 + I}]` → `ArcTan[1 + I]`.
- **Real-parameter reduction.** Each segment `a → b` is parametrised by a real
  `t ∈ [0, 1]`, `γ(t) = a + t(b − a)`.  The antiderivative is computed in the
  friendly variable `x`; on-path singularities become real roots `t* ∈ (0, 1)` of
  `Denominator[Together[f(γ(t))]]`, and endpoint singularities are resolved by
  **real one-sided limits in `t`** (so a complex-ray approach is a limit the
  existing `Limit` engine can take, e.g. the integrable branch-point endpoint of
  `Integrate[1/Sqrt[z], {z, 0, 1 + I}]` → `2 Sqrt[1 + I]`).
- **On-path singularity detection / divergence.** `Integrate\`PathSingularPoints[
  1/z, {z, -1 - I, 1 + I}]` → `{0}`; a genuine singularity on the contour makes
  the integral divergent — `Integrate[1/z, {z, -1 - I, 1 + I}]` warns
  `Integrate::idiv` and is left unevaluated.
- **Branch-correct residues.** For rational integrands whose antiderivative is a
  sum of `Log`/`ArcTan`/`ArcTanh` of **affine** arguments, the continuous change
  along a segment is computed as a principal `Log` of a ratio `Log[(u(b))/(u(a))]`
  — exact because a straight segment subtends `< π` at any off-path point.  Closed
  contours therefore close: `Chop[N[Integrate[1/z, {z, 1, I, -1, -I, 1}]]]` →
  `0.0 + 6.28319 I` (`2 Pi I`), and closed contours of entire functions vanish.
- **Never-wrong.** Every segment value is numerically cross-checked (loose 1e-3
  tolerance) against a complex quadrature of `f(γ(t)) γ'(t)`; an uncorrectable
  branch crossing leaves the integral unevaluated rather than return a wrong
  branch.
- **`Method` consistency.** Following its real-axis sibling `"NewtonLeibniz"`,
  the complex mechanism is now also selectable by name:
  `Integrate[f, {x, z0, …, zn}, Method -> "LineIntegral"]` is accepted (and
  produces the same result as the automatic dispatch), while
  `Integrate[f, x, Method -> "LineIntegral"]` on the indefinite form stays
  unevaluated (both are definite-only mechanisms — the actual mechanism is still
  chosen from the spec type).  Added to the `IntegrateMethod` enum,
  `method_from_string`, `definite_parse_method`, and both docstrings in
  `integrate.c`.
- Attributes `Protected, ReadProtected` on both builtins; docstrings via
  `symtab_set_docstring`.  Tests in `tests/test_integrate_line.c` cover detection,
  segment values, complex-plane limits, divergence, polyline path independence,
  and the `2 Pi I` residue.  Memory: the only leaks under valgrind are the
  pre-existing `Solve` / Goursat (`gs_guarded`) leaks that any `Integrate`/`Solve`
  caller triggers; no allocation in `integrate_line.c` leaks.

## Integrate: definite / line integration review — three correctness fixes (2026-07-07)

An extensive review of definite integration (Newton-Leibniz FTC path and the new
complex line/contour path) against a corpus of classical and stress examples
surfaced three bugs, now fixed and covered by regression tests.

- **Risch-Norman stack-overflow crash (critical).** An inexact-base exponential
  integrand — e.g. `2.71828^(-x)`, exactly what `N[]` produces from an `Exp[]`
  integrand — drove `split_factor` (`intrischnorman.c`) into unbounded recursion
  and overflowed the C stack, killing the whole process.  Minimal repro:
  `Integrate[x*2.71828^(-x), x]` and `N[Integrate[x*Exp[-x], {x, 0, Infinity}]]`.
  `split_factor` now carries a recursion-depth guard (`PMINT_MAX_SPLIT_DEPTH`) and
  gives up gracefully (integral left unevaluated) instead of crashing.
- **Reversed integration limits gave a wrong value (high).** With `a > b` and an
  interior pole, the FTC pole classifier assumed `a < b`, classified the pole as
  exterior, and returned a spurious finite branch value:
  `Integrate[1/z, {z, 1, -1}]` → `I Pi` (and `Integrate[1/x, {x, 2, -2}]` → `I Pi`)
  instead of diverging.  `nl_classify` is now orientation-independent — a pole
  strictly between the bounds is detected for both `a < b` and `a > b`, so the
  divergence is caught (`Integrate::idiv`, unevaluated) while non-pole reversed
  integrals still negate correctly (`Integrate[x^2, {x, 1, 0}]` → `-1/3`).
- **Missed on-path singularity for quadratic denominators (line path).** A contour
  segment passing through a pole of e.g. `1/(z^2+1)` was not reported divergent:
  the parametrised denominator is a polynomial in real `t` with *complex*
  coefficients, and `Solve[…, Reals]` drops its real roots.  `line_segment_singularities`
  now solves over ℂ and keeps the real roots via its existing `im ≈ 0` filter, so
  `Integrate[1/(z^2+1), {z, I-1, I+1}]` correctly warns `Integrate::idiv`.

Review notes (unchanged behaviour, documented gaps — all *safe* failures that stay
unevaluated, never wrong): several classical improper integrals miss because the
`Limit` engine does not resolve `Limit[x Log[x], x→0]`, `Limit[poly·E^(-x), x→∞]`,
or `Limit[Erf[x], x→∞]`; `Integrate[Sin[x]^2, {x, 0, Pi}]` fails because the
*indefinite* integral returns an unsimplified rational-trig antiderivative whose
denominator vanishes at `x = π`; `Integrate[Exp[-x^2], x]` emits no `Erf`
antiderivative; and `Abs[x]` is not integrated.

## Integrate: continuous branch-form definite integrals + `Integrate::idiv` (2026-07-07)

Follow-up to the Newton-Leibniz definite integrator: a class of *continuous*
integrands whose antiderivative is a branch form now evaluates, and divergent
integrals warn like Mathematica.  All changes in
`src/calculus/integrate_newton_leibniz.c`.

- **Continuous Weierstrass branch forms now close.** `Integrate[1/(2+Cos[x]),
  {x,0,2Pi}]` → `(2 Pi)/Sqrt[3]`, `Integrate[1/(5+4Cos[x]),{x,0,2Pi}]` →
  `2/3 Pi`, `Integrate[1/(3+2Sin[x]),{x,0,2Pi}]` → `(2 Pi)/Sqrt[5]`. The
  Weierstrass integrator emits an *already-continuous*, `Floor`-corrected
  antiderivative, so `F(b)−F(a)` by substitution is exact.  Previously these
  bailed because pole detection ran on `F` too and tripped on the spurious
  `Tan[x/2]` denominator.  Pole detection now runs on the **integrand `f`
  only** (a continuous `f` has a pole-free antiderivative).
- **"Never-wrong" continuity guard.** A finite value from an antiderivative
  that could carry a genuine uncorrected branch jump — one containing a step
  head (`Floor`/`Ceiling`/`Round`/`Sign`/…) or an inverse-trig node over a
  pole-bearing trig head (`ArcTan[Tan[x/2]…]`) — is accepted only if a numeric
  `NIntegrate` cross-check agrees (loose 1e-3 tolerance); otherwise the integral
  is left unevaluated.  The exact symbolic value is returned, never the numeric.
- **Spurious-pole numeric resolution.** When `Solve` cannot place a trig
  denominator's roots (e.g. it returns complex `ArcCos[2]` conditional
  solutions for the nowhere-zero `2+Cos[x]`), a numeric sign-change scan of the
  denominator over `(a,b)` settles whether a real interior pole exists; if none,
  integration proceeds under the cross-check guard.
- **`Integrate::idiv` warning.** A genuinely divergent definite integral now
  emits `Integrate::idiv: Integral of <f> does not converge on {a, b}.` and is
  left **unevaluated** (Mathematica-faithful).  Previously `Integrate[1/x,
  {x,-1,1}]` returned `Indeterminate` and `Integrate[1/x^2,{x,-1,1}]` returned
  `Infinity`, both silently; these now warn and echo the unevaluated integral.
- A boundary substitution that merely *contains* a singular sentinel (e.g.
  `ArcTan[ComplexInfinity/Sqrt[3]]` from evaluating a Weierstrass form exactly
  on a `Tan[x/2]` pole) is no longer accepted as a closed form — such partial

Tests: `tests/test_integrate_newton_leibniz.c` gains `test_continuous_periodic`
(5 exact cases) and 3 NIntegrate stress rows; `test_divergent` updated for the

## Integrate: definite integration via Newton-Leibniz (2026-07-07)

`Integrate` now evaluates definite and iterated integrals by the fundamental
theorem of calculus.  New module `src/calculus/integrate_newton_leibniz.c`.

- **`Integrate[f, {x, xmin, xmax}]`** — antidifferentiates `f` with the existing
  indefinite cascade to get `F`, then forms `F(xmax) − F(xmin)` through the
  `Limit` engine.  Boundaries at infinite endpoints use a plain limit; interior
  poles use one-sided limits (`Direction -> "FromBelow"/"FromAbove"`) after
  splitting the interval; ordinary finite endpoints use direct substitution.
  Improper integrals converge (`Integrate[1/Sqrt[x], {x, 0, 1}] -> 2`,
  `Integrate[1/(1 + x^2), {x, 0, Infinity}] -> Pi/2`); divergent ones surface
  `Infinity` / `ComplexInfinity` / `Indeterminate`
  (`Integrate[1/x^2, {x, -1, 1}] -> Infinity`,
  `Integrate[1/x, {x, -1, 1}] -> Indeterminate`).  An unknown antiderivative or
  an undecidable pole position leaves the integral unevaluated — never a wrong
  value.
- **Iterated integrals** — `Integrate[f, {x,a,b}, {y,c,d}, …]` reduces
  innermost-first (last spec is the inner integral), so an inner bound may
  depend on an outer variable (`Integrate[1, {x,0,1}, {y,0,x}] -> 1/2`).
- **`Integrate`NewtonLeibniz[f, {x, a, b}]`** — explicit entry point;
  `Method -> "NewtonLeibniz"` selects the definite mechanism, and any other
  `Method` name on a definite integral is passed through to the inner
  indefinite integration.
- **`Integrate`SingularPoints[expr, {x, a, b}]`** — the interior-pole detector,
  exposed for inspection/reuse: sorted real roots of
  `Denominator[Together[expr]]` strictly inside `(a, b)`.
- **Attributes.** `Integrate`NewtonLeibniz` and `Integrate`SingularPoints` are
  `Protected | ReadProtected`.  `Integrate` stays non-`Listable`.
- **Scope.** The detector covers rational (denominator) poles.  Branch-cut
  discontinuities of `F` that are not denominator roots are out of scope; a
  transcendental denominator that `Solve` cannot place is treated as
  undecidable → unevaluated.
- **Tests.** `tests/test_integrate_newton_leibniz.c` — 8 groups: pole detector,
  proper integrals, improper/infinite-bound, divergent, iterated, explicit
  entry / method pass-through, unevaluated-when-unknown, and a 14-case numeric
  stress batch cross-checked against `NIntegrate`.
- **See** [`docs/spec/builtins/calculus.md`](../builtins/calculus.md) →
  *Definite integration (Newton-Leibniz)*.

## StringReplacePart (2026-07-06)

Added the `StringReplacePart` string builtin (`src/picostrings.c`).

- `StringReplacePart["string", "snew", {m, n}]` replaces the characters at
  positions m through n by `"snew"`. `StringReplacePart["string", "snew",
  {{m1, n1}, ...}]` applies `"snew"` at several ranges, and
  `StringReplacePart["string", {"snew1", ...}, {{m1, n1}, ...}]` pairs each
  range with its own new string (the two lists must be the same length).
  Position specs use the `StringPosition` form (pairs `{m, n}`); negative
  positions count from the end, and all positions refer to the ORIGINAL string
  before any replacement is done.
- The flat single-range form `{m, n}` and the nested multi-range form
  `{{m, n}, ...}` are distinguished by whether the first element is itself a
  list. Ranges are resolved and validated up front; overlap is rejected in the
  order given via a `covered` bitmap — the first claimant of a character wins,
  and a later overlapping range emits `StringReplacePart::ovlp` and is dropped.
  Accepted ranges are insertion-sorted by start, then the result is assembled
  in a single left-to-right pass (length computed first, then filled).
- `StringReplacePart["string", "", ...]` deletes the selected characters. An
  empty position list makes no replacements.
- `StringReplacePart[{s1, s2, ...}, snew, part]` maps over a list of strings.
- Operator form: `StringReplacePart[new, part]` (a two-argument call) returns
  the pure function `Function[StringReplacePart[#1, new, part]]`, so
  `StringReplacePart[new, part][old]` reduces to
  `StringReplacePart[old, new, part]` (mirroring how `Cases[patt][expr]` is
  implemented).
- A malformed or out-of-range position, a reversed range (m > n after
  resolution), a non-string argument, or a new-string/position length mismatch
  leaves the call unevaluated so symbolic arguments flow through. A call whose
  argument count is not two or three emits `StringReplacePart::argt` (matching
  Mathematica's diagnostic).
- The result buffer is freed after `expr_new_string` copies it (leak-checked
  clean with `leaks`: `0 leaks for 0 total leaked bytes` across the single/
  multi-range, list-of-strings, operator, overlap, delete, list-mapping, and
  error paths).
- **Attributes**: `Protected`. Symbol registered in `src/sym_names.c`;
  docstring in `src/info.c`.
- 28 unit tests in `tests/test_strings.c` (every spec form and example,
  broadcast vs per-range new strings, both operator forms, overlap rejection
  including a mid-list skip, deletion, single/nested range, first/last/full
  replacement, negative and adjacent ranges, empty position list, list
  mapping, special characters, `argt` for 0/1/4 args, and symbolic/
  out-of-range/reversed/length-mismatch flow-through).

See [`builtins/string-operations.md`](../builtins/string-operations.md#stringreplacepart).

## RegularExpression + regex string functions (2026-07-06)

Added regular-expression matching for strings, built on **PCRE2** (`libpcre2-8`,
the same engine the Wolfram Language uses). New builtins live under
`src/strings/regex/`: `RegularExpression`, `StringMatchQ`, `StringCases`,
`StringReplace`, and `StringSplit`.

- **Build wiring** — PCRE2 is an optional pkg-config dependency
  (`-DUSE_REGEX -DPCRE2_CODE_UNIT_WIDTH=8`), autodetected and graceful-degrading
  exactly like Raylib/FLINT: when absent, `make` still succeeds and the regex
  builtins warn (`Name::regavail`) and stay unevaluated. A single wrapper
  (`regex_engine.{c,h}`) is the only translation unit that includes `<pcre2.h>`;
  its body is under `#ifdef USE_REGEX` with link-compatible stubs otherwise.
  Best-effort JIT (`pcre2_jit_compile`) is enabled.
- **`RegularExpression["re"]`** is an inert data head (evaluates to itself) that
  carries a PCRE pattern for the string functions. It best-effort-validates the
  pattern and emits `RegularExpression::regex` on a syntax error.
- **`StringMatchQ[s, patt]`** — whole-string match (the pattern is anchored
  `\A(?:...)\z`), returning `True`/`False`.
- **`StringCases[s, patt]`** — list of non-overlapping matches, left to right;
  with a rule `patt -> rhs` returns the rhs per match with `$0`/`$1…` expanded.
- **`StringReplace[s, rule | {rules…}]`** — replaces each non-overlapping match
  by the (`$n`-expanded) rhs; with several rules the leftmost match wins, ties
  broken by rule order; zero-width matches (e.g. `\b`) are handled without
  dropping characters.
- **`StringSplit[s, patt]`** — substrings between delimiter matches, dropping
  empty pieces (Wolfram default); zero-width delimiters like `(?m)^` / `(?m)$`
  split at positions.
- All four accept `RegularExpression["re"]`, a **literal string** (matched
  literally — PCRE metacharacters are escaped), or a **List** of
  alternatives/rules; a **list of subjects** threads element-wise (the builtins
  are deliberately not `Listable`, so the pattern/rule argument is never
  threaded — matching the `StringReplacePart`/`Replace` precedent). Non-string
  subjects flow through unevaluated.
- The shared layer `regex_common.{c,h}` compiles a pattern argument into a rule
  set, expands `$n`/`$0` (and `$$` → `$`) replacement templates, and provides a
  small growable buffer; `regex_rules_free` releases every compiled program.
- **Attributes**: `Protected`. `SYM_RegularExpression` registered in
  `src/sym_names.c`; docstrings for all five in `src/info.c`.
- Tests: `tests/test_regex.c` (engine wrapper — compile/match/unset-group/bad-
  pattern — and `$n` expansion, plus the inert head) and
  `tests/test_stringfns.c` (the worked spec examples across all four functions:
  character classes/ranges, `\d \D \s \S \w \W`, named classes, quantifiers and
  non-greedy `*?`, anchors and `\b`, inline `(?i)`, alternation, `$1`/multi-rule
  replacement, `(?m)^`/`(?m)$` splitting with real newlines, literal patterns,
  list threading, and unevaluated flow-through). Leak-checked clean (`leaks`:
  `0 leaks for 0 total leaked bytes`) across every path.

Note: Mathilda's string lexer does not interpret C-style escapes (`\n`, `\t`)
in string literals, so a newline in a *subject* must be a real newline byte;
this is a pre-existing parser property, independent of the regex layer.

See [`builtins/string-operations.md`](../builtins/string-operations.md#regularexpression).

## Integrate `"GoursatAlgebraic"` — denominator-radical normalisation fix (2026-07-07)

Fixed a branch-cut normalisation bug in the Goursat V4 (square-root) descent
(`src/calculus/integrate_goursat.c`). On e.g.
`Integrate[Sqrt[1 + x^4]/(1 - x^4), x, Method -> "GoursatAlgebraic"]` the result
carried a spurious nested radical
`Sqrt[2 (1 + (1+x)^4/(-1+x)^4 + 6 (1+x)^2/(-1+x)^2)]` in the ArcTan/ArcTanh
arguments, which is really `4 Sqrt[1 + x^4]/(-1 + x)^2` over the integrand's own
generator.

- **Root cause.** The eigendescent emits the radical in *denominator* positions
  as `Power[g, -1/2]` (`1/Sqrt[g]`), but the radical-over-R normaliser
  (`reexpress_over_radical`) only recognised and rewrote the `+1/2` form:
  `collect_sqrt_radicands` matched the exponent exactly `1/2`, and the rewrite
  rule was `Sqrt[g] -> cand` alone. So `+1/2` occurrences (in the `Log` term)
  healed to `Sqrt[1 + x^4]` while the `-1/2` occurrences stayed nested.
- **Fix.** `collect_sqrt_radicands` now gathers radicands from any half-integer
  power `Power[g, m/2]` (via `is_half_power`), so it also sees `1/Sqrt[g]`; and
  `reexpress_over_radical` now rewrites **both** signs — `Sqrt[g] -> cand` and
  `1/Sqrt[g] -> 1/cand` — in one `ReplaceAll`. `cand` is the sign-fixed
  `+Sqrt[g]`, so its reciprocal is `1/Sqrt[g]` with the matching sign; the
  differentiate-back guard still gates adoption.
- The antiderivative is now written entirely over `Sqrt[1 + x^4]` and
  differentiates back to the integrand (verified to 25 digits at several
  rational points).
- Regression tests added to `tests/test_integrate_goursat.c` (`test_sqrt`):
  numeric differentiate-back plus a structural assertion that no t-dependent
  half-power radical other than `Sqrt[1 + t^4]` survives.

## Integrate threads over a list integrand

`Integrate[{f1, ..., fn}, spec...]` now threads element-wise, returning
`{Integrate[f1, spec...], ..., Integrate[fn, spec...]}`. Previously the whole
list was handed to the rational/method cascade, which could not process a
`List` head and returned the input unevaluated (e.g.
`Integrate[{1, x, x^5/5}, x]` came back verbatim instead of
`{x, x^2/2, x^6/30}`).

- **Fix** (`src/calculus/integrate.c`, top of `builtin_integrate`): when the
  first argument has head `List`, build the list of sub-`Integrate` calls
  (each element paired with the unchanged trailing spec/Method args) and let
  the evaluator recurse into each. Elements that cannot be integrated stay
  symbolic, matching Mathematica.
- Applies to both the indefinite `x` and the definite `{x, a, b}` / contour
  spec forms. `Integrate` remains non-`Listable` on purpose — a `Listable`
  attribute would also (wrongly) thread over the `{x, a, b}` range spec.
- Fixes the previously-failing "Listable threading" case in
  `tests/test_integrate.c` (`integrals_tests`).

## Integrate: real-branch repair of ArcTanh in radical antiderivatives

A substitution integrator could back-substitute into an antiderivative carrying
`ArcTanh[g(x)]` whose argument `g` is real with `|g| > 1` on the integrand's
real domain. `ArcTanh`'s real interval is `(-1, 1)`; for `|g| > 1` the value
sits on the branch cut, so the antiderivative of a *real* integrand came out
complex-valued (off by a constant `±I Pi/2`). For example

```
Integrate[Sqrt[x + Sqrt[x]], x]
```

evaluated at `x = 2` returned `2.68952 - 0.392699 I` (a spurious `-I Pi/8`)
because it carried `ArcTanh[Sqrt[1 + Sqrt[x]]/x^(1/4)]` with the argument `> 1`
for every `x > 0`.

- **Fix** (`src/calculus/intsimp.c`, `intsimp_finalize` transcendental step):
  rewrite each x-dependent `ArcTanh[g]` to the derivative-identical
  `ArcCoth[g]` when `g` probes real with `|g| > 1` across a spread of
  positive-real sample points. Since `d/dz ArcTanh[z] = d/dz ArcCoth[z] =
  1/(1 - z^2)`, the swap is differentiate-back-preserving; `ArcCoth` is real
  precisely where `|z| > 1`, so it selects the real branch. The probe is
  conservative (a definite `|g| <= 1` or a nonzero imaginary part at any probe
  vetoes the rewrite), and the pass is scoped, as before, to radical-bearing
  antiderivatives so pure-rational / Risch / trig output is untouched.
- Result: `Integrate[Sqrt[x + Sqrt[x]], x]` now yields a real antiderivative
  (`F(2) = 2.68952`) with the derivative still matching the integrand; several
  other radical integrals gain real forms too, e.g.
  `Integrate[1/(x Sqrt[x + 1]), x]` now returns `-2 ArcCoth[Sqrt[1 + x]]`.
- No corpus regressions (`integrals_tests`, `integrate_derivdivides_tests`,
  `integrate_quadrad_tests`, `integrate_linrad_tests`,
  `integrate_linratiorad_tests`, `integrate_jeffrey_tests`,
  `integrate_dispatch_tests` all green).

## Integrate: derivative-divides keeps the integrand's own radicals (root kernels)

`Integrate[Sqrt[x + Sqrt[x]], x]` (and the wider `x^k Sqrt[x + Sqrt[x]]`
family) returned an antiderivative in *foreign* radical generators —
`x^(1/4)`, `Sqrt[1 + Sqrt[x]]`, and a large nested-rational tower — instead of
the integrand's own radical `Sqrt[x + Sqrt[x]]`. The derivative was correct
(`D[F]//Simplify` equalled the integrand up to the branch identity
`x^(1/4) Sqrt[1 + Sqrt[x]] == Sqrt[x + Sqrt[x]]`), but the surd form was
needlessly unlike the input.

Root cause: the derivative-divides **direct-quotient** strategy tries the
smallest kernel first — here `u = Sqrt[x]` — but a bare `Sqrt[x] -> u`
substitution leaves a residual `x` behind (`2 Sqrt[x] Sqrt[x + Sqrt[x]]` becomes
`2 u Sqrt[u + x]`, still a function of `x`), so the kernel was rejected and the
integral fell through to the heavier Eliminate/Solve search, which closes it in
the `x^(1/4)` binomial-differential shape.

- **Fix** (`src/calculus/integrate_derivdivides.c`, `try_direct_kernel`): when
  the kernel is a unit root of the variable, `w = x^(1/n)` (`n >= 2`), close the
  substitution with the exact inverse `x -> u^n` and `PowerExpand` the result so
  the `(u^n)^(p/n)` powers this creates collapse to `u^p` (sound under the
  integration-positivity convention `u = x^(1/n) >= 0`; genuine radicals of
  polynomials in `u`, e.g. `Sqrt[u^2 + u]`, are left intact). The reduced
  integrand is then a clean function of the integrand's own radical, and because
  the direct strategy runs *before* Eliminate/Solve it pre-empts the foreign-
  generator form. `differentiate-back` against the original integrand still
  gates acceptance, so a branch-crossing `PowerExpand` is rejected rather than
  trusted.
- Result: `Integrate[Sqrt[x + Sqrt[x]], x]` and `Integrate[x^2 Sqrt[x + Sqrt[x]],
  x]` now return in terms of `Sqrt[Sqrt[x] + x]` and `Sqrt[x]`; `D[%]//Simplify`
  returns exactly the integrand. Nested-root integrals such as
  `Integrate[Sqrt[1 + Sqrt[x]], x]` and `Integrate[Sqrt[x]/(1 + x), x]` likewise
  stay in their input radicals.
- No corpus regressions (`integrals_tests` and all `integrate_*` suites green).

## Feature: Integrate — differentiation under the integral sign (2026-07-08)

- **Summary.** New definite-integration method `Integrate`DiffUnderInt` (Feynman's
  trick / Leibniz rule), also reachable via `Method -> "DiffUnderInt"` /
  `"DifferentiationUnderIntegral"` and tried last in the automatic definite
  cascade (after residue and Newton-Leibniz). For a parameter-dependent
  `Integrate[f(x,p), {x,a,b}]` it differentiates w.r.t. a free parameter,
  evaluates the simpler inner integral, integrates back over the parameter, and
  fixes the constant from an exact base value. Solves the standard
  parameter-dependent families, e.g.
  `Integrate[(x^a-1)/Log[x], {x,0,1}]` → `Log[1+a]`,
  `Integrate[Exp[-a x] Sin[b x]/x, {x,0,Infinity}, Assumptions->a>0]` → `ArcTan[b/a]`,
  `Integrate[Sin[a x]^2/x^2, {x,0,Infinity}, Assumptions->a>0]` → `Pi a/2`,
  `Integrate[Log[1+a^2 x^2]/(1+x^2), {x,0,Infinity}, Assumptions->a>0]` → `Pi Log[1+a]`.

- **Implementation.** New `src/calculus/integrate_diffunderint.{c,h}`; wired into
  the dispatcher (`src/calculus/integrate.c`: method enum/parse, `integrate_definite`
  cascade after Newton-Leibniz, `integrate_init`). Correct-by-construction and
  SYMBOLIC only — no NIntegrate crosscheck (project rule); the conditional-
  convergence pitfall is caught for free because a non-integrable `D[f,p]` fails
  to close. Because the general integrator hangs (uninterruptibly from inside a
  builtin — `TimeConstrained` does not bound a nested `evaluate`) on the inner
  Laplace/Fourier/Gaussian forms, the method evaluates the standard families
  itself with closed-form formulas: Laplace/Fourier half-line (via the Laplace
  master formula on `TrigToExp`-linearised terms), sinc/Frullani (`/x`), and
  even-rational half-line (via `v = x^2` + Apart + the Beta integral). Forms
  outside these (Gaussians, finite-period trig, piecewise) are declined fast and
  returned unevaluated — never a wrong value or a hang. Assumptions are threaded
  into the `Simplify` calls to produce clean closed forms.

- **Attributes.** `Integrate`DiffUnderInt`: `Protected`, `ReadProtected`, with a
  docstring.

- **Tests.** `tests/test_integrate_diffunderint.c` (added to `COMMON_SRC` and a
